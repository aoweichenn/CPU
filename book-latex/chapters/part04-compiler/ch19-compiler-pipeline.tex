\section{编译器流水线：从源码到 IR}

\subsection{本章要解决的问题}

从本章开始，我们进入编译器部分。对性能优化来说，编译器不是一个黑盒按钮，也不是一个只负责“把 C++ 变成汇编”的工具。它是一条逐层建立事实、逐层丢失信息、逐层引入假设、逐层重写程序的流水线。

你写下的 C++ 源码不会直接变成机器指令。它通常会经历：

\begin{lstlisting}
source file
    -> preprocessing
    -> tokens
    -> AST
    -> semantic analysis
    -> IR generation
    -> optimization passes
    -> instruction selection
    -> register allocation
    -> assembly
    -> object file
    -> link
    -> executable or shared library
\end{lstlisting}

如果你只看源码，会误以为代码结构就是执行结构。如果你只看汇编，又会很难判断编译器为什么生成这种结构。IR 的价值在这里：它比源码更接近执行，比汇编更保留高级语义，是理解优化器的桥。

本章目标是让你掌握：

\begin{itemize}
  \item 编译器流水线每一层的职责和不变量。
  \item token、AST、语义分析、IR、优化、后端之间的区别。
  \item AST 和 IR 为什么不是同一种东西。
  \item 编译器为什么需要中间表示，而不是直接从 AST 生成汇编。
  \item \code{-O0}、\code{-O2}、\code{-O3} 大致改变哪些事情。
  \item 如何用 Clang 生成 AST dump、LLVM IR、汇编和优化报告。
  \item 如何把 C++ 源码、IR、汇编和 benchmark 证据连起来。
\end{itemize}

\begin{keyidea}
性能优化不是“源码优化”或“汇编优化”单点工作，而是跨越源码语义、编译器假设、IR 变换、目标代码和硬件执行的证据链。你必须知道每一层保留了什么信息、丢失了什么信息、引入了什么假设。
\end{keyidea}

\subsection{一个贯穿例子}

本章使用一个很小的函数：

\begin{lstlisting}[language=C++]
#include <cstddef>
#include <cstdint>
#include <span>

std::int64_t sum_i32(std::span<const std::int32_t> values)
{
    std::int64_t sum = 0;
    for (std::int32_t value : values) {
        sum += value;
    }
    return sum;
}
\end{lstlisting}

这个函数足够简单，刚学完 C++ 的读者能看懂；也足够真实，因为它包含性能优化里最常见的结构：

\begin{itemize}
  \item 一个输入数组。
  \item 一个循环。
  \item 一个累加变量。
  \item 一个返回值。
  \item 一次从 32 位整数到 64 位整数的扩展。
\end{itemize}

我们会反复问：

\begin{lstlisting}
At source level:
    What does the code mean?

At AST level:
    What syntax and semantic structure did the frontend build?

At IR level:
    What values, memory operations, branches and loops exist?

At optimized IR level:
    What did the optimizer prove and rewrite?

At assembly level:
    What instructions actually run?

At benchmark level:
    What evidence supports the performance claim?
\end{lstlisting}

\subsection{流水线总览}

先看各阶段的职责。

\begin{center}
\begin{tabular}{p{0.18\linewidth}p{0.62\linewidth}}
\toprule
阶段 & 主要职责 \\
\midrule
预处理 & 展开 \code{\#include}、宏和条件编译，形成翻译单元。 \\
词法分析 & 把字符流切成 token，例如关键字、标识符、字面量、运算符。 \\
语法分析 & 根据 C++ 语法把 token 组织成 AST。 \\
语义分析 & 做名字查找、类型检查、重载决议、模板实例化、访问控制等。 \\
IR 生成 & 把已检查的语义结构 lowering 成便于优化的中间表示。 \\
优化 & 通过一系列 pass 改写 IR，例如 DCE、inline、LICM、向量化。 \\
后端 & 选择指令、分配寄存器、安排栈帧、生成目标汇编或目标文件。 \\
链接 & 解析符号、合并目标文件、处理重定位、生成最终二进制。 \\
\bottomrule
\end{tabular}
\end{center}

每一层都有不变量。所谓不变量，就是下一层可以依赖的事实。

\begin{lstlisting}
lexer output invariant:
    token sequence has source ranges and token kinds

parser output invariant:
    AST shape follows grammar

semantic output invariant:
    expressions have types, names are resolved, overloads are selected

IR invariant:
    control flow is explicit, values have definitions, operations are well-typed

optimized IR invariant:
    program behavior is preserved under language rules and optimization assumptions

backend invariant:
    generated machine code implements the target-level calling convention
\end{lstlisting}

如果某一层的不变量不清楚，后面的优化就会变成危险猜测。工业编译器通常会在 IR 层提供 verifier，用来检查基本块、类型、控制流、SSA use-def 等结构是否满足规则。

\subsection{预处理：真正进入编译器的源码不等于你看到的文件}

C++ 编译的第一个大变化通常发生在预处理阶段。看这段源码：

\begin{lstlisting}[language=C++]
#include <cstdint>

#define SCALE 4

std::int32_t f(std::int32_t x)
{
    return x * SCALE;
}
\end{lstlisting}

预处理后，\code{\#include <cstdint>} 可能展开成大量声明，\code{SCALE} 会被替换成 \code{4}。可以用：

\begin{lstlisting}[language=bash]
clang++ -std=c++20 -E example.cpp -o example.ii
\end{lstlisting}

\filepath{example.ii} 往往非常大，因为标准库头文件会被展开。性能学习中你不需要每天阅读完整预处理输出，但你必须知道：编译器前端真正看到的是翻译单元，不是你屏幕上那个小文件。

预处理会影响性能相关工作：

\begin{itemize}
  \item 宏可能隐藏分支、循环、函数调用和常量。
  \item 条件编译可能让不同平台编译出完全不同的代码。
  \item header-only 库会把大量实现暴露给编译器，增加 inline 和优化机会。
  \item 包含顺序和宏定义可能影响 ABI 或代码路径。
\end{itemize}

\begin{warningbox}
不要只看一个 \code{.cpp} 文件就断言编译器输入是什么。宏和头文件可能让真实输入比你看到的源码复杂得多。
\end{warningbox}

\subsection{词法分析：字符流变成 token}

词法分析器不理解“求和函数是否高效”。它的任务更基础：把字符切成 token。

例如：

\begin{lstlisting}[language=C++]
sum += value;
\end{lstlisting}

大致会形成：

\begin{lstlisting}
identifier     "sum"
operator       "+="
identifier     "value"
punctuator     ";"
\end{lstlisting}

token 需要携带位置信息：

\begin{lstlisting}
file
line
column
byte offset
token kind
spelling
\end{lstlisting}

没有准确位置，编译器就不能给出高质量诊断，也不能让 IDE、格式化工具、静态分析器、代码导航工具稳定工作。

词法阶段的性能意义在于：后面所有诊断和工具能力都依赖它的 source range。你在优化代码时看到的 warning、vectorization report、optimization remark，最终都要定位到源码位置。

\subsection{语法分析：token 变成 AST}

Parser 根据语言语法把 token 组织成 AST。AST 是 Abstract Syntax Tree，抽象语法树。

例如：

\begin{lstlisting}[language=C++]
return x * 4 + y;
\end{lstlisting}

可以想成下面的树：

\begin{lstlisting}
ReturnStmt
  BinaryOperator '+'
    BinaryOperator '*'
      DeclRefExpr x
      IntegerLiteral 4
    DeclRefExpr y
\end{lstlisting}

AST 保留源码结构。它适合表达：

\begin{itemize}
  \item 函数、类、命名空间、模板。
  \item 声明和语句层次。
  \item 表达式优先级和嵌套。
  \item 源码位置。
  \item 类型、名字和语义信息的附着点。
\end{itemize}

生成 AST dump：

\begin{lstlisting}[language=bash]
clang++ -std=c++20 \
  -Xclang -ast-dump \
  -fsyntax-only sum.cpp
\end{lstlisting}

对 \code{sum_i32}，你会看到很多标准库类型信息。为了学习，可以先用裸指针版本：

\begin{lstlisting}[language=C++]
#include <cstddef>
#include <cstdint>

std::int64_t sum_i32_ptr(const std::int32_t* values, std::size_t size)
{
    std::int64_t sum = 0;
    for (std::size_t index = 0; index < size; ++index) {
        sum += values[index];
    }
    return sum;
}
\end{lstlisting}

它的 AST 更容易读，因为少了 \code{std::span} 的模板展开噪声。

\subsection{语义分析：源码结构变成确定含义}

AST 只说明“长什么样”，语义分析说明“是什么意思”。C++ 语义分析非常复杂，至少包括：

\begin{itemize}
  \item 名字查找：\code{sum} 指的是哪个变量？
  \item 类型检查：\code{values[index]} 是什么类型？
  \item 隐式转换：\code{std::int32_t} 如何加到 \code{std::int64_t}？
  \item 重载决议：调用的是哪个函数或运算符？
  \item 模板实例化：\code{std::span<const std::int32_t>} 展开成什么？
  \item 生命周期：引用、临时对象、析构何时发生？
  \item 可见性和访问控制：名字是否可访问？
\end{itemize}

对 \code{sum_i32_ptr} 这句：

\begin{lstlisting}[language=C++]
sum += values[index];
\end{lstlisting}

语义分析需要知道：

\begin{lstlisting}
sum:
    local variable
    type: std::int64_t

values:
    parameter
    type: pointer to const std::int32_t

index:
    local variable
    type: std::size_t

values[index]:
    lvalue expression
    type: const std::int32_t

sum += values[index]:
    read sum
    load values[index]
    convert int32 to int64
    add
    store back to sum
\end{lstlisting}

注意最后两行已经接近 IR 思维：读、加载、转换、加、写回。优化器需要的不是“源码看起来像加法”，而是明确的值和副作用。

\subsection{为什么 AST 不适合直接优化}

初学者常问：既然 AST 已经表示程序结构，为什么不直接在 AST 上优化并生成汇编？

可以，但不好。原因是 AST 太贴近源码语法，很多执行细节不够显式。

\begin{center}
\begin{tabular}{p{0.16\linewidth}p{0.70\linewidth}}
\toprule
问题 & AST 和 IR 的差异 \\
\midrule
控制流 & AST 中 \code{for}、\code{if}、\code{while} 是不同语法节点；IR 通常统一成基本块和分支。 \\
表达式 & AST 保留嵌套表达式的源码形状；IR 把表达式拆成有定义、有使用的中间值。 \\
变量 & AST 要承载局部变量、成员和引用语义；IR 更倾向于表达 SSA 值以及显式的 load 和 store。 \\
优化 & AST 优化要处理大量语法特例；IR pass 可以在统一结构上工作。 \\
目标无关性 & AST 仍然带有强烈语言特征；IR 更适合跨语言、跨目标共享优化。 \\
\bottomrule
\end{tabular}
\end{center}

举例，源码里有很多循环写法：

\begin{lstlisting}[language=C++]
for (std::size_t i = 0; i < n; ++i) { ... }

while (i < n) { ...; ++i; }

do { ...; ++i; } while (i < n);
\end{lstlisting}

AST 中它们是不同节点；IR 中它们都会降低成 basic block、条件分支和回边。优化器更愿意在这种统一结构上工作。

\begin{mentalmodel}
AST 是给语言语义服务的，IR 是给程序变换服务的。AST 关心“源码如何表达”，IR 关心“执行效果如何表示”。
\end{mentalmodel}

\subsection{IR：优化器的工作语言}

IR 是 Intermediate Representation，中间表示。不同编译器有不同 IR，例如 LLVM IR、GCC GIMPLE 和 RTL、MLIR dialect、JVM bytecode、WebAssembly、Cranelift IR。

本书主要使用 LLVM IR，因为 Clang 能方便地产生它，工具链成熟，结构适合学习。

生成 IR：

\begin{lstlisting}[language=bash]
clang++ -std=c++20 -O0 -emit-llvm -S sum.cpp -o sum_O0.ll
clang++ -std=c++20 -O3 -emit-llvm -S sum.cpp -o sum_O3.ll
\end{lstlisting}

\code{-O0} 的 IR 更接近源码和调试需求，常有大量 \code{alloca}、\code{load}、\code{store}。

\code{-O3} 的 IR 已经过优化，局部变量可能消失，循环可能展开或向量化，函数可能 inline，死代码可能被删除。

一个简化后的 LLVM IR 片段可能像这样：

\begin{lstlisting}
define i64 @sum_i32_ptr(ptr %values, i64 %size) {
entry:
  br label %loop

loop:
  %index = phi i64 [ 0, %entry ], [ %next, %loop ]
  %sum = phi i64 [ 0, %entry ], [ %sum.next, %loop ]
  %ptr = getelementptr inbounds i32, ptr %values, i64 %index
  %v32 = load i32, ptr %ptr, align 4
  %v64 = sext i32 %v32 to i64
  %sum.next = add nsw i64 %sum, %v64
  %next = add nuw i64 %index, 1
  %cond = icmp eq i64 %next, %size
  br i1 %cond, label %exit, label %loop

exit:
  ret i64 %sum.next
}
\end{lstlisting}

这段 IR 已经揭示了很多源码里不直观的事实：

\begin{itemize}
  \item 循环被 basic block 和 branch 表达。
  \item \code{index} 和 \code{sum} 在循环里通过 \code{phi} 表达。
  \item 数组访问通过 \code{getelementptr} 计算地址。
  \item 读取内存是 \code{load}。
  \item 32 位到 64 位有符号扩展是 \code{sext}。
  \item 循环条件是 \code{icmp} 加条件分支。
\end{itemize}

本章先建立直觉，下一章会系统讲 SSA、\code{phi}、basic block、CFG 和 dominance。

\subsection{Basic block：控制流的基本单位}

Basic block 是一段顺序执行的指令序列。控制流只能从 block 开头进入，从 block 末尾离开。LLVM IR 中每个 basic block 通常以 terminator 结束，例如：

\begin{lstlisting}
br label %loop
br i1 %cond, label %exit, label %loop
ret i64 %value
\end{lstlisting}

对循环来说，常见结构是：

\begin{lstlisting}
entry:
    initialize
    jump to loop

loop:
    phi values
    body
    compute next index
    branch to loop or exit

exit:
    return result
\end{lstlisting}

这比源码的 \code{for} 更底层。硬件执行的也是分支和跳转，不是 \code{for} 这个语法概念。

\subsection{SSA：值为什么只定义一次}

SSA 是 Static Single Assignment。简化理解：每个 SSA value 只定义一次。

源码：

\begin{lstlisting}[language=C++]
sum = 0;
sum = sum + value;
sum = sum + value2;
\end{lstlisting}

SSA 思维：

\begin{lstlisting}
sum0 = 0
sum1 = sum0 + value
sum2 = sum1 + value2
\end{lstlisting}

这对优化非常有用，因为 use-def chain 清楚。优化器能问：

\begin{itemize}
  \item 这个值在哪里定义？
  \item 它被谁使用？
  \item 如果替换这个值，会影响哪些地方？
  \item 如果它没有使用者，是否可以删除？
\end{itemize}

循环里的变量会遇到“值从哪里来”的问题，所以需要 \code{phi}：

\begin{lstlisting}
%sum = phi i64 [ 0, %entry ], [ %sum.next, %loop ]
\end{lstlisting}

含义是：

\begin{itemize}
  \item 如果控制流从 \code{entry} 来，\code{sum} 是初始值 \code{0}。
  \item 如果控制流从 \code{loop} 回来，\code{sum} 是上一轮的 \code{sum.next}。
\end{itemize}

你不需要在本章完全掌握 \code{phi}，但要先知道：它不是普通函数调用，也不是运行时选择表；它是 SSA 在控制流汇合处表达值来源的方式。

\subsection{优化 pass：编译器不是一次性“变聪明”}

优化器通常不是一个巨大函数，而是一串 pass。每个 pass 做某类分析或变换。

常见 pass 包括：

\begin{itemize}
  \item Dead Code Elimination：删除没有可观察用途的代码。
  \item Constant Propagation：传播常量，把能算出的值提前算出。
  \item InstCombine：合并和简化局部指令模式。
  \item Inlining：把函数调用替换成函数体。
  \item Loop Invariant Code Motion：把循环不变量移出循环。
  \item Induction Variable Simplification：简化循环归纳变量。
  \item Loop Unrolling：展开循环，减少分支开销，增加 ILP。
  \item SLP Vectorization：在基本块内寻找可打包的标量操作。
  \item Loop Vectorization：把循环变成 SIMD 形式。
\end{itemize}

一个 pass 通常有前提：

\begin{lstlisting}
input:
    valid IR
    required analyses available

transform:
    rewrite some instructions, blocks, metadata, attributes

output:
    valid IR
    preserve or invalidate analyses
\end{lstlisting}

这就是为什么 IR verifier 很重要。每个 pass 改完后，IR 仍然必须满足规则。

\subsection{优化级别：\code{-O0}、\code{-O2}、\code{-O3} 的区别}

不同编译器细节不同，但大方向可以这样理解：

\begin{center}
\begin{tabular}{p{0.15\linewidth}p{0.68\linewidth}}
\toprule
级别 & 常见特征 \\
\midrule
\code{-O0} & 面向可调试性，保留更多源码结构，优化少，性能结论通常不能使用。 \\
\code{-O1} & 做基础优化，编译时间和调试体验相对保守。 \\
\code{-O2} & 常用 Release 优化级别，启用大量标量优化和部分循环优化。 \\
\code{-O3} & 更激进，增加更多循环优化、向量化、展开和内联，可能增加代码尺寸。 \\
\code{-Ofast} & 允许更激进语义放松，尤其是浮点规则；不等价于“总是更好”。 \\
\bottomrule
\end{tabular}
\end{center}

不要把 \code{-O3} 理解成“最大优化，所以一定最好”。它可能：

\begin{itemize}
  \item 增加代码尺寸，导致 instruction cache 或 frontend 压力。
  \item 过度展开，增加 register pressure。
  \item 改变浮点优化策略。
  \item 编译更慢。
  \item 对小函数更快，对大工作集无明显收益。
\end{itemize}

性能实验必须把优化级别写进报告。

\subsection{Dead Code Elimination 和 Lab 00 的关系}

第 18 章反复强调 \code{do_not_optimize}。现在从编译器角度看原因。

源码：

\begin{lstlisting}[language=C++]
void bad(const std::uint64_t* values, std::size_t size)
{
    std::uint64_t sum = 0;
    for (std::size_t i = 0; i < size; ++i) {
        sum += values[i];
    }
}
\end{lstlisting}

如果 \code{sum} 没有可观察用途，优化器可以认为循环没有必要。优化后函数可能接近：

\begin{lstlisting}
define void @bad(ptr %values, i64 %size) {
entry:
  ret void
}
\end{lstlisting}

这不是编译器作弊，而是语言规则允许它删除没有可观察效果的计算。benchmark 如果没有观察结果，就可能测到一个被删除的 loop。

\code{do_not_optimize(sum)} 的作用是人为制造一个优化器必须尊重的观察点，阻止这类删除。

\subsection{Constant propagation：编译器会提前算出能证明的值}

源码：

\begin{lstlisting}[language=C++]
int f()
{
    int x = 2;
    int y = 3;
    return x * y + 4;
}
\end{lstlisting}

优化后可能直接变成：

\begin{lstlisting}
define i32 @f() {
entry:
  ret i32 10
}
\end{lstlisting}

汇编可能是：

\begin{lstlisting}
mov eax, 10
ret
\end{lstlisting}

这对普通程序是好事；对 benchmark 可能是灾难。如果你的被测输入是编译期常量，编译器可能提前算完。性能实验应使用运行时输入、外部数据、参数化规模或适当的 observation barrier。

\subsection{Inlining：性能优化的入口放大器}

Inlining 把函数调用替换成函数体。

收益：

\begin{itemize}
  \item 消除 call 和 ret 开销。
  \item 暴露常量参数。
  \item 暴露别名和内存访问模式。
  \item 让后续 pass 能跨函数优化。
\end{itemize}

代价：

\begin{itemize}
  \item 增加代码尺寸。
  \item 可能增加 instruction cache 压力。
  \item 可能增加 register pressure。
  \item 可能让前端取指和解码成为瓶颈。
\end{itemize}

对性能工程来说，inline 不是“越多越好”。一个小函数 inline 后可能变快；一个大模板函数到处 inline 可能让热路径变胖，反而变慢。后面的 frontend 和 code size 章节会重新回到这个问题。

\subsection{Lowering：高级语义向低级操作下降}

Lowering 是把高级结构转换成更低级结构的过程。它不只发生一次，而是贯穿编译器很多层。

例子：

\begin{lstlisting}
C++ range-for
    -> begin/end iterator loop
    -> pointer/index loop
    -> IR basic blocks
    -> machine branch
\end{lstlisting}

再看 \code{std::span}：

\begin{lstlisting}[language=C++]
std::span<const std::int32_t> values
\end{lstlisting}

在 ABI 层，它常常可以理解成类似：

\begin{lstlisting}[language=C++]
struct span_like {
    const std::int32_t* data;
    std::size_t size;
};
\end{lstlisting}

但这是帮助理解的模型，不是标准规定的 ABI 承诺。实际布局和调用细节要看标准库实现、编译器和 ABI。

当你在汇编里看到 \register{rdi} 和 \register{rsi} 像指针和长度时，本质上是在观察 lowering 之后的结果。

\subsection{后端：从 IR 到机器}

LLVM 后端大致还会经历：

\begin{lstlisting}
LLVM IR
    -> target-independent optimizations
    -> instruction selection
    -> machine IR
    -> register allocation
    -> instruction scheduling
    -> assembly emission
    -> object emission
\end{lstlisting}

后端要处理源码和 IR 不关心的目标细节：

\begin{itemize}
  \item x86-64 有哪些指令可以表达这个操作？
  \item 哪些值放进寄存器，哪些 spill 到栈？
  \item 函数参数和返回值遵守哪套 calling convention？
  \item 栈帧如何布局？
  \item 是否使用 SIMD 指令？
  \item 分支、跳转、重定位如何编码？
\end{itemize}

生成汇编：

\begin{lstlisting}[language=bash]
clang++ -std=c++20 -O3 -S -masm=intel sum.cpp -o sum_O3.s
\end{lstlisting}

生成目标文件并反汇编：

\begin{lstlisting}[language=bash]
clang++ -std=c++20 -O3 -c sum.cpp -o sum_O3.o
objdump -d -Mintel -C sum_O3.o
\end{lstlisting}

你应该同时保存 IR 和汇编，因为它们回答不同问题：

\begin{itemize}
  \item IR 解释优化器看到了什么、证明了什么。
  \item 汇编解释后端实际选择了什么指令。
\end{itemize}

\subsection{源码、IR、汇编和性能报告如何连接}

一个合格的性能分析应该能形成这样的链：

\begin{lstlisting}
C++ source:
    loop over int32 array and accumulate into int64

IR evidence:
    load i32
    sext i32 to i64
    add i64
    phi for index and sum

assembly evidence:
    load from memory
    sign-extend
    add to accumulator
    branch back to loop

benchmark evidence:
    raw samples for several input sizes
    median and tail behavior
    environment and compiler flags recorded

claim:
    under this compiler, flags, input sizes and machine,
    the scalar baseline behaves as observed
\end{lstlisting}

这条链越完整，你的结论越可信。只说“我改了一行，快了 20\%”不够。你需要说明：

\begin{itemize}
  \item 编译器是否真的生成了你以为的代码？
  \item 优化前后 IR 有什么变化？
  \item 汇编是否支持你的解释？
  \item 测量是否可复现？
  \item 输入规模是否覆盖声称的场景？
\end{itemize}

\subsection{优化器不是数学真理机}

编译器优化基于语言规则和它能证明的事实。它不能读懂你的意图。

三个常见限制：

\begin{itemize}
  \item 别名：两个指针是否可能指向同一块内存？
  \item 副作用：函数调用是否可能读写未知内存、抛异常、访问 volatile？
  \item 浮点语义：重排浮点加法是否改变结果？
\end{itemize}

例如：

\begin{lstlisting}[language=C++]
void saxpy(float* y, const float* x, float a, std::size_t n)
{
    for (std::size_t i = 0; i < n; ++i) {
        y[i] += a * x[i];
    }
}
\end{lstlisting}

如果 \code{x} 和 \code{y} 可能重叠，编译器就要谨慎。它可能生成 runtime alias check，也可能拒绝某些向量化。你知道两个数组不重叠，不等于编译器能证明它们不重叠。

再看浮点求和：

\begin{lstlisting}[language=C++]
float sum_f32(const float* values, std::size_t n)
{
    float sum = 0.0F;
    for (std::size_t i = 0; i < n; ++i) {
        sum += values[i];
    }
    return sum;
}
\end{lstlisting}

严格浮点语义下，编译器不能随便把加法重排成树形规约，因为浮点加法不满足严格结合律。使用 \code{-ffast-math} 或相关 fast-math flags 会放松语义，但也会改变数值行为。性能优化必须知道自己付出了什么语义代价。

\subsection{优化报告：让编译器解释自己}

Clang 可以输出优化 remark。对向量化，常用：

\begin{lstlisting}[language=bash]
clang++ -std=c++20 -O3 -march=native \
  -Rpass=loop-vectorize \
  -Rpass-missed=loop-vectorize \
  -Rpass-analysis=loop-vectorize \
  sum.cpp -c -o /tmp/sum.o
\end{lstlisting}

GCC 常用：

\begin{lstlisting}[language=bash]
g++ -std=c++20 -O3 -march=native \
  -fopt-info-vec-optimized \
  -fopt-info-vec-missed \
  sum.cpp -c -o /tmp/sum.o
\end{lstlisting}

优化报告不是最终证据，但它很有用。它能告诉你：

\begin{itemize}
  \item 哪个 loop 被向量化。
  \item 哪个 loop 没有被向量化。
  \item 原因是 legality 问题还是 profitability 问题。
  \item 源码位置在哪里。
\end{itemize}

你要把优化报告和 IR、汇编对照起来。报告说 vectorized，不代表最终汇编一定符合你的预期；报告说 missed，也要看 missed 的原因是否可以通过源码重写、alias 信息或编译参数改善。

\subsection{本章总结}

本章建立了从源码到 IR 的主线：

\begin{lstlisting}
source:
    human-written C++ with syntax and intent

AST:
    parsed and semantically attached language structure

IR:
    explicit values, memory operations and control flow

optimized IR:
    transformed program under language rules and compiler assumptions

assembly:
    target-specific instructions and registers

benchmark:
    measured behavior under a concrete environment
\end{lstlisting}

接下来第 20 章会深入 LLVM IR、SSA、basic block、CFG 和 \code{phi}。本章你只需要牢牢记住：编译器优化不是魔法，而是一系列在明确中间表示上进行的程序变换。看懂这些变换，你才能解释为什么 C++ 写法会影响汇编，为什么 benchmark 会被优化器误导，为什么某些循环能向量化而另一些不能。
