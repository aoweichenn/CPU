\section{标量优化：DCE、常量传播、内联和 LICM}

\subsection{本节要解决的问题}

标量优化处理的是“单个值、表达式、分支、调用和循环不变量”这类结构。它不像 SIMD 那样直接改变一次处理的数据宽度，也不像线程并行那样改变执行资源数量，但它是所有高性能代码的前置条件。很多看似高级的优化，都依赖一批标量 pass 先把程序整理成可分析形态。

本节要回答：

\begin{itemize}
  \item 编译器如何证明一段计算没有用，从而删除它？
  \item 常量为什么能穿过函数、分支和循环传播？
  \item 两次相同计算什么时候能合并？
  \item 内联为什么既可能加速，也可能让代码变慢？
  \item 循环里不变的表达式为什么能移到循环外？
  \item benchmark 为什么特别容易被标量优化“优化没了”？
\end{itemize}

\begin{keyidea}
标量优化的核心不是“编译器猜到程序员想快一点”，而是“编译器在 IR 上证明某些变换保持可观察行为”。你读优化后的 IR 时，要追问每个变化背后的证明条件：这个值是否被使用？这个表达式是否无副作用？这个 load 是否可能被 store 改写？这个调用是否可能读写内存？
\end{keyidea}

\subsection{可观察行为：优化的边界}

优化器不是随意改写程序。它必须保留语言规则定义下的可观察行为。粗略说，下面这些通常会成为边界：

\begin{itemize}
  \item 函数返回值。
  \item 对 volatile 对象的访问。
  \item 可能逃逸到外部的内存写入。
  \item I/O、系统调用、原子操作、同步操作。
  \item 可能有副作用的不透明函数调用。
  \item 未定义行为边界带来的优化假设。
\end{itemize}

对 benchmark 来说，最危险的问题是：你以为循环在测量某个计算，但结果没有被观察，优化器可以证明计算没有影响可观察行为，于是整个循环消失。

\begin{lstlisting}[language=C++]
extern "C" void bad_bench(const int* values, int n)
{
    long long sum = 0;
    for (int i = 0; i < n; ++i) {
        sum += values[i];
    }
}
\end{lstlisting}

如果 \code{sum} 没有返回、没有写到外部、没有被 observation barrier 使用，这段循环很可能在 \code{-O3} 下被删除。

改成：

\begin{lstlisting}[language=C++]
extern "C" long long good_bench(const int* values, int n)
{
    long long sum = 0;
    for (int i = 0; i < n; ++i) {
        sum += values[i];
    }
    return sum;
}
\end{lstlisting}

返回值成为可观察结果，循环就不能被简单删除。

\subsection{Dead Code Elimination}

DCE 删除不会影响可观察行为的代码。最简单的例子：

\begin{lstlisting}[language=C++]
extern "C" int dce_example(int x)
{
    int a = x + 1;
    int b = x + 2;
    return a;
}
\end{lstlisting}

\code{b} 的计算结果没有被使用。优化后 IR 可能只剩：

\begin{lstlisting}
define dso_local i32 @dce_example(i32 noundef %0) {
entry:
  %a = add nsw i32 %0, 1
  ret i32 %a
}
\end{lstlisting}

DCE 在 SSA 上非常自然。如果一个 instruction 产生的值没有 users，并且指令本身没有副作用，就可以删除。删除之后可能让它的 operands 也变成无用值，于是 DCE 会继续传播。

\begin{lstlisting}
%a = add i32 %x, 1
%b = mul i32 %a, 7
; %b has no users

delete %b
; now %a may also have no users
delete %a
\end{lstlisting}

但不是所有“结果没用”的指令都能删。例如：

\begin{lstlisting}
call void @puts(ptr %s)
store i32 1, ptr %p
volatile load i32, ptr %q
\end{lstlisting}

这些操作可能有可观察副作用。DCE 必须尊重副作用边界。

\subsection{Constant Folding 和 Constant Propagation}

constant folding 是在编译期计算常量表达式：

\begin{lstlisting}[language=C++]
extern "C" int folded()
{
    return (3 + 4) * 8;
}
\end{lstlisting}

优化后：

\begin{lstlisting}
ret i32 56
\end{lstlisting}

constant propagation 是把已知常量沿 use-def 链传播：

\begin{lstlisting}[language=C++]
extern "C" int branch_fold(int x)
{
    const int scale = 4;
    int y = x * scale;
    if (scale == 4) {
        return y + 1;
    }
    return y - 1;
}
\end{lstlisting}

因为 \code{scale == 4} 总为真，分支可以被折叠。优化器可能得到：

\begin{lstlisting}
%y = shl nsw i32 %x, 2
%ret = add nsw i32 %y, 1
ret i32 %ret
\end{lstlisting}

这里还出现了 strength reduction：乘以 4 被替换为左移。实际后端是否真的发出 \code{shl}，还取决于目标机器、后端选择和后续优化。

\subsection{Sparse Conditional Constant Propagation}

真实编译器通常不会只做“文本替换式”的常量传播。控制流也会参与。SCCP 会同时追踪值是否为常量，以及 basic block 是否可达。

例子：

\begin{lstlisting}[language=C++]
extern "C" int sccp_example(int x)
{
    int mode = 1;
    if (mode == 0) {
        return x * 100;
    }
    return x + 7;
}
\end{lstlisting}

优化器知道 \code{mode} 为 1，于是 \code{mode == 0} 的分支不可达，乘法分支可以删除。

这类优化对模板和 \code{constexpr} 很重要。C++ 常常通过编译期常量生成多种路径，优化器再把不可达路径消掉。很多高性能库利用这个机制把通用模板代码变成针对特定类型、大小、布局的专门代码。

\subsection{CSE 和 GVN}

CSE 是 Common Subexpression Elimination，公共子表达式消除。GVN 是 Global Value Numbering，全局值编号。它们都在寻找“已经算过的等价值”。

简单例子：

\begin{lstlisting}[language=C++]
extern "C" int cse_example(int a, int b)
{
    int x = (a + b) * 3;
    int y = (a + b) * 5;
    return x + y;
}
\end{lstlisting}

\code{a + b} 出现两次。优化器可以只算一次：

\begin{lstlisting}
%sum = add nsw i32 %b, %a
%x = mul nsw i32 %sum, 3
%y = mul nsw i32 %sum, 5
%ret = add nsw i32 %y, %x
ret i32 %ret
\end{lstlisting}

GVN 处理的问题更广：不同写法是否产生同一个值。例如：

\begin{lstlisting}
(x + 1) + 1
x + 2
\end{lstlisting}

在某些语义条件下可以等价，但有符号溢出、浮点舍入、NaN、异常、内存副作用都可能阻止这种等价。整数优化和浮点优化的差异，后面会反复出现。

\subsection{内存表达式为什么更难}

纯 SSA 值容易做 CSE，内存 load 更难。

\begin{lstlisting}[language=C++]
extern "C" int load_twice(const int* p)
{
    int a = *p;
    int b = *p;
    return a + b;
}
\end{lstlisting}

如果中间没有可能写 \code{*p} 的操作，两次 load 可能合并。但如果加一个不透明调用：

\begin{lstlisting}[language=C++]
extern "C" void unknown();

extern "C" int load_call_load(const int* p)
{
    int a = *p;
    unknown();
    int b = *p;
    return a + b;
}
\end{lstlisting}

优化器不能随便把第二次 load 合并到第一次。因为 \code{unknown()} 可能修改 \code{p} 指向的内存，除非函数属性、别名分析或链接时信息能证明不会。

\begin{mentalmodel}
对优化器来说，普通 SSA 值像一条清楚的电线；内存像一个共享的大盒子。只要指针、调用或 store 可能碰到同一个盒子，很多看似明显的优化就不能做。
\end{mentalmodel}

\subsection{Inlining：优化的入口放大器}

内联把被调用函数的函数体复制到调用点。它最直接的收益是消除调用开销，但更重要的收益是暴露优化机会。

\begin{lstlisting}[language=C++]
static int scale4(int x)
{
    return x * 4;
}

extern "C" int use_scale(int x)
{
    return scale4(x) + 1;
}
\end{lstlisting}

内联后，调用边界消失：

\begin{lstlisting}
%mul = shl nsw i32 %x, 2
%ret = add nsw i32 %mul, 1
ret i32 %ret
\end{lstlisting}

如果不内联，优化器可能看不到被调用函数内部，不知道它只是乘以 4。内联之后，constant propagation、CSE、DCE、LICM、向量化都可能获得更多事实。

但内联不是越多越好。过度内联会造成：

\begin{itemize}
  \item 代码体积膨胀。
  \item instruction cache 压力增加。
  \item 前端 decode/uop cache 压力增加。
  \item 寄存器压力增加。
  \item 编译时间增加。
\end{itemize}

现代编译器有内联 cost model。它会根据函数大小、调用频率、参数常量、是否热路径、是否会暴露优化机会等因素决策。PGO 会进一步影响这些决策。

\subsection{LICM：Loop Invariant Code Motion}

LICM 把循环里每轮结果都相同的计算移到循环外。

\begin{lstlisting}[language=C++]
extern "C" int sum_scaled(const int* values, int n, int scale)
{
    int sum = 0;
    int bias = scale * 7 + 3;
    for (int i = 0; i < n; ++i) {
        sum += values[i] + bias;
    }
    return sum;
}
\end{lstlisting}

如果 \code{bias} 写在循环外，当然没有问题。即使写在循环内：

\begin{lstlisting}[language=C++]
for (int i = 0; i < n; ++i) {
    int bias = scale * 7 + 3;
    sum += values[i] + bias;
}
\end{lstlisting}

优化器也可能把 \code{scale * 7 + 3} 移到循环外，因为它不依赖 \code{i}，也不被循环内可能的副作用改变。

LICM 的证明条件包括：

\begin{itemize}
  \item 表达式 operands 在循环中不变。
  \item 指令没有不可移动副作用。
  \item 移动后不会改变异常、未定义行为或内存可观察行为。
  \item 如果涉及 load，必须证明循环内没有 store 或 call 会改变它读到的内存。
\end{itemize}

内存 load 的 LICM 很常见也很危险：

\begin{lstlisting}[language=C++]
extern "C" int sum_with_factor(const int* values,
                               const int* factor,
                               int n)
{
    int sum = 0;
    for (int i = 0; i < n; ++i) {
        sum += values[i] * (*factor);
    }
    return sum;
}
\end{lstlisting}

能不能把 \code{*factor} 移出循环，取决于优化器能否证明 \code{values[i]} 的访问不会写到 \code{factor}，且中间没有其他可能修改 \code{*factor} 的操作。因为 \code{values} 是 \code{const int*} 只限制通过该指针写入，不自动证明它和 \code{factor} 不 alias。第 5.6 会深入讨论。

\subsection{Strength Reduction}

strength reduction 把昂贵操作换成等价的便宜操作。常见例子：

\begin{lstlisting}
i * 4        -> i << 2
base + i*4   -> addressing mode or induction increment
x / 8        -> shift with signedness caveats
\end{lstlisting}

在 x86-64 上，数组地址计算经常不需要显式乘法，因为寻址模式可以表达：

\begin{lstlisting}
[base + index*1 + displacement]
[base + index*2 + displacement]
[base + index*4 + displacement]
[base + index*8 + displacement]
\end{lstlisting}

因此 C++ 中的 \code{values[i]} 可能变成汇编里的 scale addressing，而不是单独的乘法指令。读汇编时要把地址计算和数据访问放在一起看。

\subsection{InstCombine：小变换的大集合}

LLVM 中有一个非常重要的 pass 叫 InstCombine。它做大量局部代数化简和规范化，例如：

\begin{lstlisting}
x + 0          -> x
x * 1          -> x
x * 0          -> 0
(x << 2) << 1  -> x << 3
icmp eq x, x   -> true, if x is not poison/undef sensitive
\end{lstlisting}

InstCombine 的目标不只是“少几条指令”，还包括把 IR 变成后续 pass 更容易识别的标准形态。编译器优化经常是多轮协作：

\begin{lstlisting}
inline
    -> exposes constants
constant propagation
    -> makes branches unreachable
simplify CFG
    -> deletes blocks
instcombine
    -> canonicalizes expressions
loop passes
    -> recognize loops better
\end{lstlisting}

\subsection{benchmark 中的优化陷阱}

性能实验最常见的错误，不是计时 API 用错，而是被优化器改变了实验对象。

错误模式 1：结果没人用。

\begin{lstlisting}
for repeat:
    compute sum
; sum discarded
\end{lstlisting}

错误模式 2：输入是编译期常量。

\begin{lstlisting}
for repeat:
    compute f(known_constant)
; compiler precomputes or folds
\end{lstlisting}

错误模式 3：benchmark 函数被内联到调用点后，整个计算被证明可删除。

错误模式 4：测量的是空循环、分支框架或计时框架，而不是目标 kernel。

最低限度的防护：

\begin{itemize}
  \item 返回或校验计算结果。
  \item 使用运行时输入。
  \item 查看优化后 IR 和汇编。
  \item 对照不同输入规模的运行时间。
  \item 用 PMU 计数确认指令和访存确实发生。
\end{itemize}

\subsection{实验 21.1：让 DCE 删除你的 benchmark}

\begin{labbox}
目标：亲眼观察 benchmark 计算被删除。

写两个函数：

\begin{lstlisting}[language=C++]
#include <cstddef>
#include <cstdint>

extern "C" void bad_sum(const std::uint64_t* values, std::size_t size)
{
    std::uint64_t sum = 0;
    for (std::size_t i = 0; i < size; ++i) {
        sum += values[i];
    }
}

extern "C" std::uint64_t good_sum(const std::uint64_t* values,
                                  std::size_t size)
{
    std::uint64_t sum = 0;
    for (std::size_t i = 0; i < size; ++i) {
        sum += values[i];
    }
    return sum;
}
\end{lstlisting}

任务：

\begin{enumerate}
  \item 生成 \code{-O0} 和 \code{-O3} IR。
  \item 生成 \code{-O3} 汇编。
  \item 解释 \code{bad_sum} 的循环是否存在。
  \item 解释 \code{good_sum} 为什么不能被同样删除。
  \item 用 \code{perf stat} 比较两者运行时的指令数。
\end{enumerate}

交付物：

\begin{itemize}
  \item IR 摘录。
  \item 汇编摘录。
  \item \code{perf stat} 输出。
  \item 一段报告：这个实验如何改变你对 benchmark 的理解。
\end{itemize}
\end{labbox}

\subsection{实验 21.2：观察内联暴露常量}

\begin{labbox}
目标：理解内联如何放大其他优化。

写：

\begin{lstlisting}[language=C++]
static int affine(int x, int a, int b)
{
    return x * a + b;
}

extern "C" int call_affine(int x)
{
    return affine(x, 8, 3);
}
\end{lstlisting}

任务：

\begin{enumerate}
  \item 生成 \code{-O0}、\code{-O1}、\code{-O3} IR。
  \item 找到 \code{affine} 是否仍然作为独立函数存在。
  \item 记录乘以 8 是否变成 shift 或地址形式。
  \item 尝试给 \code{affine} 加 \code{__attribute__((noinline))}。
  \item 对比 \code{noinline} 前后的 IR 和汇编。
\end{enumerate}

报告必须回答：内联本身省掉了什么？内联之后又暴露了哪些新的优化机会？
\end{labbox}

\subsection{实验 21.3：LICM 和 alias 的边界}

\begin{labbox}
目标：观察循环不变量移动什么时候失败。

准备两个函数：

\begin{lstlisting}[language=C++]
extern "C" int sum_factor_plain(const int* values,
                                const int* factor,
                                int n);

extern "C" int sum_factor_restrict(const int* __restrict values,
                                   const int* __restrict factor,
                                   int n);
\end{lstlisting}

函数体都计算：

\begin{lstlisting}[language=C++]
int sum = 0;
for (int i = 0; i < n; ++i) {
    sum += values[i] * (*factor);
}
return sum;
\end{lstlisting}

任务：

\begin{enumerate}
  \item 对两个函数生成 \code{-O3} IR。
  \item 找 \code{*factor} 的 load 在循环内还是循环外。
  \item 查看函数参数是否带有 noalias 相关属性。
  \item 生成汇编并标注 load 位置。
  \item 解释 \code{restrict} 改变了什么证明条件。
\end{enumerate}
\end{labbox}

\subsection{本节总结}

标量优化的主线可以压缩成一句话：优化器在 SSA、CFG、属性和内存事实上证明哪些计算可以删除、合并、提前、常量化或跨函数传播。DCE 保护你不浪费指令，也会毁掉写错的 benchmark；内联省掉调用，更重要的是暴露上下文；LICM 减少循环体工作量，但遇到内存和 alias 就必须谨慎。

下一节进入循环优化。循环优化建立在本节之上：先有清楚的 induction variable、循环不变量、内存访问和控制流，才谈得上展开、重排、归约、向量化和并行。
