\chapter*{全书架构设计}
\addcontentsline{toc}{chapter}{全书架构设计}

本册是系列教材的基础卷，正文分为三个部分、十二章，并保留若干附录作为工具和阅读材料。它不再把尚未写成正文的主题放进目录占位；没有正文的内容进入后续卷规划。

\section*{写作目标}

第一册的写作目标是覆盖：

\begin{itemize}
  \item C/C++ 程序如何变成机器执行。
  \item CPU 执行指令所需要的寄存器、内存、控制流和硬件状态模型。
  \item 数据表示、对象布局、指针、栈和最基本的软件运行时概念。
  \item x86-64 汇编语法、整数指令、控制流和 System V AMD64 ABI。
  \item C++ 与汇编互操作的边界、风险和验证方法。
  \item 可信性能测量、benchmark 输入设计、热路径和报告规范。
\end{itemize}

现代 CPU 微架构、cache/TLB、编译器优化、SIMD、GEMM、生产库和 AI CPU 算子属于第二册。它们需要第一册建立的机器执行模型、汇编读写能力和实验方法作为前提。

\section*{三部分结构}

\begin{enumerate}
  \item 底层模型：从程序到机器执行。回答源码如何成为进程、CPU 如何执行指令、数据如何落在内存里、如何用 Linux 工具观察证据。
  \item 汇编原理：x86-64、ABI 与二进制接口。回答 C/C++ 控制流、整数运算、函数调用和对象传递如何映射到汇编与调用约定。
  \item 实验原理：可信性能测量。回答为什么很多 benchmark 不可信，以及如何设计输入、隔离热路径、记录环境并写出可复现实验报告。
\end{enumerate}

\section*{章节深度标准}

每章按三层写：

\begin{enumerate}
  \item 必须掌握：刚学完 C/C++ 的读者能跟上。
  \item 工程理解：能用于真实代码分析和优化。
  \item 深入方向：连接体系结构、OS、编译器或生产库更深内容。
\end{enumerate}

\section*{规模目标和统计口径}

第一册最终正文规模目标不少于 36 万单位。这里的“单位”只按可阅读教材内容统计：

\begin{itemize}
  \item 中文按汉字字符计数。
  \item 英文按单词计数。
  \item 代码、命令、反汇编、终端输出、LaTeX 命令和工程配置不计入正文规模。
  \item 附录可以计入书稿总规模，但不能替代 12 个正文章节的讲解深度。
\end{itemize}

工程中使用 \code{scripts/count_text_units.py} 统计该口径。扩写时必须同时满足字数目标和质量目标；不能通过堆代码块、重复命令、无解释清单或空泛段落凑数。

\section*{反凑数质量红线}

36 万单位是教材深度目标，不是机械长度目标。扩写时宁可放慢进度，也不能牺牲解释质量。新增正文必须至少承担下面一种教学功能：

\begin{itemize}
  \item 给出一个读者原本缺失的背景知识，并说明它为什么影响后续学习。
  \item 把一个机制从动机、约束、实现边界到工具观察讲成闭环。
  \item 纠正一个常见误区，并说明错误推理发生在哪一层。
  \item 解释一个实验应该观察什么、怎样解释结果、结论边界在哪里。
  \item 把当前章节与后续汇编、ABI、内存、测量或第二册性能主题建立必要接口。
\end{itemize}

以下内容不得作为扩写手段：

\begin{itemize}
  \item 为增加长度而重复前文已经讲清楚的定义。
  \item 没有解释目的的代码块、命令块、反汇编块或工具输出。
  \item 只罗列名词、不说明关系和边界的清单。
  \item 只说“很重要”“需要理解”但不给机制和证据的空泛段落。
  \item 为了追求覆盖面而把第二册主题提前讲成浅层摘要。
\end{itemize}

每次扩写后应做一次结构审查：章节顺序是否自然，是否存在同一概念重复讲解，新增段落是否服务本章目标，实验是否配套解释，统计增长是否来自可阅读正文而不是代码或标记。

\section*{经典教材写法}

本册不按“名词解释合集”来写。每章都应从一个具体问题进入，然后逐层展开：

\begin{enumerate}
  \item 起始问题：读者在真实编程或实验中会遇到什么困惑。
  \item 最小例子：用一段短 C/C++ 程序或短汇编建立观察对象。
  \item 背景知识：补齐理解本章所需的硬件、软件、算法或工具背景。
  \item 底层机制：解释这个机制为什么存在，没有它会出什么问题。
  \item 工具证据：给出编译命令、反汇编、GDB、readelf、nm 或 benchmark 数据。
  \item 反例和误区：指出初学者最容易得出的错误结论。
  \item 实验和作业：要求读者修改输入、改变编译参数、观察差异并写报告。
  \item 验收标准：明确读者学完后应该能独立完成什么任务。
\end{enumerate}

每章都必须讲清楚“语言层、二进制层、硬件层、实验层”的边界。比如讲数组访问，不能只讲 \code{a[i]} 是第 \code{i} 个元素；还要讲指针类型决定缩放、对象连续存储、地址表达式、load 指令、cache line 预告和 benchmark 可能如何受访问模式影响。

\section*{第一册和第二册的边界}

第一册负责建立可复用基础。以下内容可以预告，但不在第一册深挖：

\begin{itemize}
  \item 现代乱序核心的完整性能模型。
  \item cache/TLB 的定量实验和 Roofline 模型。
  \item LLVM IR、SSA、自动向量化和复杂编译器优化。
  \item AVX2/AVX-512/FMA/VNNI intrinsics 和 SIMD kernel。
  \item GEMM blocking、packing、microkernel 和 MiniBLAS。
  \item softmax、layernorm、attention、operator fusion 和 AI CPU 算子库。
  \item 多线程、NUMA、ISA dispatch 和生产库源码阅读。
\end{itemize}

这些主题放到第二册，是为了避免第一册失焦。第一册真正要打牢的是：读者能从源码追到汇编，能解释机器状态，能遵守 ABI，能设计可信测量。没有这些能力，高级优化只会变成经验口号。

\section*{交付物标准}

每章最终应包含：

\begin{itemize}
  \item 不少于一个贯穿例子。
  \item 不少于三个实验。
  \item 不少于三组习题。
  \item 一个报告型大作业。
  \item 一组常见误区。
  \item 一组验收标准。
\end{itemize}
