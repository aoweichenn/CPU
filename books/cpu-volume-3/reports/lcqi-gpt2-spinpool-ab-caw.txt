All compared runs produced identical generated token ids and text for each token count.

## Accepted candidate

The accepted code changes are:

- Cap automatic GPT-2 worker count at 8 on caw-class CPUs.
- Replace the cached-decoder worker pool's mutex/condition_variable hand-off with an atomic generation hand-off and bounded spin/yield wait.
- Keep the worker pool scoped to one cached decoder session. Model loading, tokenizer work, and public model objects do not gain a global busy-wait service.

This target matches the measured hotspot: each generated token runs many small row-parallel matvec regions across QKV, attention projection, MLP fc, MLP projection, and lm head. The old condition-variable hand-off was paid hundreds of times per short generation.

## Rejected 8-row AVX2 candidate

A follow-up candidate changed the AVX2 row-range kernel from a 4-row block to an 8-row block while keeping the accepted spin/yield worker pool. It did not show stable benefit on caw and was not retained.

Auto thread medians, spinpool row4 -> spinpool row8:

| max new tokens | row4 generate ms | row8 generate ms | speedup |
| ---: | ---: | ---: | ---: |
| 4 | 37.4937 | 37.7082 | -0.57% |
| 16 | 103.714 | 103.178 | 0.52% |
| 64 | 374.439 | 374.457 | -0.00% |

Fixed --threads 8 medians, spinpool row4 -> spinpool row8:

| max new tokens | row4 generate ms | row8 generate ms | speedup |
| ---: | ---: | ---: | ---: |
| 4 | 36.8123 | 36.9807 | -0.46% |
| 16 | 102.928 | 102.376 | 0.54% |
| 64 | 371.807 | 377.053 | -1.41% |

Conclusion: row8 is noise-level or worse for this CPU/model shape. Keeping the existing 4-row AVX2 row-range kernel is the better mainline choice.

## Related rejected packed F32 candidate

The earlier packed F32 path is documented in `books/cpu-volume-3/reports/lcqi-gpt2-packed-f32-ab-caw.txt`. It was not accepted because caw fixed-thread and microbenchmark evidence showed no stable end-to-end win, and row-major AVX2 was faster for the measured matvec shapes.
