\chapter{同步原语：锁、条件变量、信号量、阶段和结果}

第 9 章让线程池问题变得可见：ready 任务排队、worker 阻塞、CPU 空闲、关闭卡住。现在看一个更小但更本质的事故。线程池 worker 在空队列上等待，主线程调用 \code{shutdown}，设置 \code{closed = true} 后只通知了 \code{not_full}。生产者被唤醒并退出，消费者仍睡在 \code{not_empty} 上，析构时 \code{join} 永远不返回。代码里有 mutex，有 condition variable，也有 \code{notify_one}，但同步语义仍然错，因为等待谓词没有覆盖关闭状态，通知对象也没有覆盖所有等待者。

同步原语必须从共享状态的不变量和等待谓词推出来。多个核心同时修改同一组字段时，硬件只能保证单条 load/store 或特定原子操作的局部语义；它不知道 \code{head}、\code{tail}、\code{size}、\code{closed} 之间应该满足什么关系。Mutex 的作用，是给一组字段的状态转换建立独占区间，让一个线程能在不变量被临时打破时完成修改，再把一致状态交给其他线程。Condition variable 的作用，是在谓词不满足时不再浪费 CPU 自旋，而是释放锁、进入内核或运行库的等待队列，等状态可能变化后再醒来重新检查。Semaphore 表达的是可等待的资源额度，例如空槽数量、I/O in-flight 许可和连接池令牌。

有界队列足够小，却同时包含互斥、等待、背压、关闭、异常、安全析构和测试。它能把同步的底层来源讲清：共享字段需要锁，因为字段关系不能被并发修改拆碎；空队列需要等待，因为消费者继续自旋会浪费 CPU；满队列需要背压，因为生产者不能无限分配内存；关闭需要广播，因为等待者睡在不同谓词上，且所有等待者都要有退出路径。

\section{同步原语的统一坐标系}

同步原语不能按“哪个 API 新”“哪个 API 快”来学。系统化的学法是先问等待对象是什么，再问这份等待对象由谁维护、状态在哪里、谁改变状态、谁唤醒等待者、错误路径如何退出。Mutex、condition variable、semaphore、latch、barrier、future/promise、atomic wait、eventfd 和 futex 都可以落到同一套坐标系里。

\topic{六个审查维度}
每个同步设计先填写六个字段。第一，\term{状态位置}{state location}：业务事实放在受 mutex 保护的字段、原语内部计数、原子值、future shared state、barrier phase，还是内核 fd 里。第二，\term{状态转移}{transition}：哪个操作让状态改变，例如 \code{push}、\code{pop}、\code{close}、\code{release}、\code{count\_down}、\code{set\_value}。第三，\term{等待对象}{wait object}：等待的是不变量可修改、谓词变真、许可可用、阶段结束、一次结果 ready，还是 fd 可读。第四，\term{唤醒来源}{wake source}：哪个状态变化会唤醒哪个等待者。第五，生命周期和失败边界：关闭、取消、超时、异常、析构和 broken promise 怎么表达。第六，证据字段：运行时如何证明等待发生在哪里。

\begin{lstlisting}[numbers=none,caption={同步原语统一审查模型}]
SyncModel:
  state_location:
    protected_fields / primitive_counter / atomic_value
    phase_generation / future_shared_state / kernel_fd

  transitions:
    who changes the state
    linearization point
    whether rollback is possible

  wait_object:
    invariant / predicate / permit / phase / one_shot_result
    atomic_version / event_fd / kernel_wait_address

  wake_source:
    which transition may make the wait object ready
    notify_one / notify_all / release / count_down
    arrive / set_value / set_exception / write_fd / futex_wake

  failure_boundary:
    close / drain / cancel / timeout / exception
    broken_promise / missing_arrival / object_destruction

  evidence:
    wait_count, wait_ns, wake_count, timeout_count
    phase_wait_ns, result_ready_ns, permit_leak_check
\end{lstlisting}

这份模型的目的不是把所有原语变成同一个东西，而是防止“都能等一下”这种模糊认识。一个原语只表达一部分状态；真实系统经常组合多个原语，但每个原语必须只负责自己能表达的那层事实。

\topic{选择算法}
代码审查时可以按下面顺序走。只要某一步回答不清，就不要急着换成更底层的原语。

\begin{lstlisting}[numbers=none,caption={同步原语选择算法}]
1. List the business facts.
   Examples: size, capacity, closed, running_count, phase, result.

2. Group facts into invariants.
   If several fields must agree, start with a mutex.

3. Write each wait condition as a sentence.
   "Wait until size > 0 or closed" is a predicate.
   "Wait until one permit is available" is a permit.
   "Wait until all fixed participants reach phase p" is a phase.

4. Add exit edges before choosing an API.
   close, drain, cancel, timeout, exception, broken promise, destruction.

5. Choose the narrowest primitive that fully expresses the wait object.
   If no primitive fully expresses it, keep explicit state plus mutex/CV.

6. Add evidence fields.
   If you cannot measure wait reason, you cannot debug the design.
\end{lstlisting}

这个算法会导出一个重要结论：\code{mutex + condition\_variable} 不是落后选择，而是复杂状态的默认 reference。只有状态收缩到“许可”“阶段”“一次结果”“单版本号”这种窄对象时，semaphore、barrier、future 或 atomic wait 才是更准确的表达。

\topic{组合不是混乱}
真实系统往往同时用多种原语。连接池可以用 semaphore 限制最多借出连接数，用 mutex 保护 free list，用 condition variable 管理关闭 drain，用 future 返回一次后台健康检查结果。I/O 子系统可以用 bounded queue 保存请求，用 semaphore 限制 in-flight，用 eventfd 唤醒 epoll 线程，用 promise 把一次提交结果交给调用者。组合本身没有问题；问题是每个原语的责任边界是否写清。

\begin{lstlisting}[numbers=none,caption={一个系统里多个原语的责任分层}]
ConnectionPool:
  mutex:
    protects free_list, closing, broken_connections
  counting_semaphore:
    limits borrowed connections
  condition_variable:
    waits for drain during shutdown
  future/promise:
    reports one background health-check result

Invalid design:
  semaphore protects vector lifetime
  future represents a stream of connections
  barrier waits for dynamic borrowers
\end{lstlisting}

这就是“系统”的含义：不是把 API 名字列全，而是能把每个等待事实放到正确层级，并解释为什么别的原语不合适。

\section{从自旋到睡眠：为什么需要条件变量}

没有条件变量时，消费者可能写出这样的循环：反复拿锁，检查队列是否非空，释放锁，再继续检查。队列短时间为空时，这种自旋可以避免睡眠唤醒成本；队列长时间为空时，它会持续占用 CPU，干扰真正能运行的线程。操作系统调度的基本单位是 thread，如果一个 thread 只是在检查“还没有数据”，它仍然可能消耗时间片、污染 cache、增加互连流量。

条件变量把这件事改成三步。第一，等待者持锁检查共享状态。第二，谓词不满足时，\code{wait} 原子地释放锁并让线程睡眠。第三，状态变化后通知等待者，等待者醒来重新持锁检查谓词。数据仍然在共享状态里，通知只是提示“状态可能变了”。这就是为什么条件变量必须和 mutex 一起使用：没有同一把锁，检查谓词和进入等待之间会出现丢失唤醒窗口。

\section{互斥锁保护的是不变量}

很多并发错误来自一个误解：mutex 保护某个变量。更准确地说，mutex 保护一组状态之间的不变量。环形队列的 \code{head}、\code{tail}、\code{size}、\code{buffer} 和 \code{closed} 不是孤立字段，它们共同说明哪些槽位可读、哪些槽位可写、队列是否还接收新元素。只把 \code{size} 改成 atomic，不能让队列线程安全，因为 \code{size} 与 \code{head/tail/buffer} 的关系仍可能被并发修改破坏。

\topic{先写不变量表}
有界队列的不变量至少包括：容量固定且大于零；\code{0 <= size <= capacity}；\code{head} 和 \code{tail} 总在数组范围内；\code{size == 0} 表示没有可读槽；\code{size == capacity} 表示没有可写槽；\code{head} 指向下一个可读槽；\code{tail} 指向下一个可写槽；\code{closed} 后不再接受新 \code{push}。所有会改变这些事实的代码，都必须在同一互斥边界内完成状态转换。

\begin{lstlisting}[numbers=none,caption={有界队列的不变量不是注释装饰，而是代码审查依据}]
bounded queue invariant:
  capacity > 0
  0 <= size <= capacity
  head and tail are valid slots
  size == 0 means no readable slot
  size == capacity means no writable slot
  closed means future push must fail
\end{lstlisting}

不变量表把代码审查变得具体。若 \code{push} 写入槽位后忘记更新 \code{size}，违反容量关系；若 \code{pop} 先减 \code{size} 再移动元素而移动抛异常，状态可能提交一半；若 \code{close} 在锁外设置 \code{closed}，等待者可能看见旧状态；若 \code{head} 在锁内改、\code{tail} 在锁外改，环形区间关系被拆碎。

\topic{临界区越短越好，但不能拆散不变量}
临界区应该短，是因为持锁时间越长，其他线程等待越多，尾延迟越不稳定。但“短”不能优先于“完整”。把队列是否满的检查放在锁外，看似减少持锁时间，实际上让检查和写入之间出现竞态。正确原则是：锁内完成最小的、不可拆分的状态转换；锁外执行耗时且不改变受保护不变量的工作。

锁内尤其不能执行未知代码：用户回调、任务函数、磁盘 I/O、网络请求、复杂日志格式化、可能很慢的析构、会再进入运行时的函数。C++ 中最后一个 \code{std::shared_ptr} 在锁内析构，也可能触发复杂逻辑。可靠代码常在锁内把对象移动到局部变量，释放锁后再处理或析构。

\topic{锁的硬件路径：低竞争和高竞争不是一种东西}
低竞争 mutex 常常只是一段很短的用户态原子状态转换：线程尝试把锁状态从 unlocked 改成 locked，成功后进入临界区；释放时把状态改回 unlocked。高竞争时，事情完全不同：多个核心反复读改写同一锁变量所在 cache line，失败线程可能自旋一段时间，然后进入内核等待队列；释放锁的线程还要唤醒等待者。于是同一个 \code{std::mutex} 在低竞争和高竞争下，成本模型完全不同。

接上第一册的汇编读法，低竞争锁的快路径可以理解成“对一个锁字做原子状态转换”。不同 C++ 标准库、运行库、操作系统和架构实现不同，但机器形状常围绕 CAS、交换或带锁定语义的读改写写展开。例如一个抽象锁字从 0 表示 unlocked，1 表示 locked，快路径可能类似：

\begin{lstlisting}[numbers=none,caption={mutex fast path 的典型汇编形状}]
; rdi = address of lock word
mov   eax, 1
xor   ecx, ecx
lock cmpxchg dword ptr [rdi], eax
jne   .Lcontended
; acquired
\end{lstlisting}

第一册可以解释这几条指令的局部语义：\instruction{cmpxchg} 比较 \register{eax} 与内存旧值，匹配时写入新值，不匹配时把旧值带回 \register{eax}；\instruction{jne} 根据 flags 进入竞争路径。这里继续向下看：带 \instruction{lock} 的 CAS 需要让锁字所在 cache line 进入可修改状态；若锁空闲且 line 已在本核心附近，成本很低；若许多核心同时抢锁，所有核心都围绕同一条 line 反复读改写写，失败路径还会污染互连和前端。锁的快路径短，不代表高竞争锁便宜。

\begin{lstlisting}[numbers=none,caption={unlock 和等待路径的粗略机器形状}]
unlock fast path:
  store lock word = 0 with release semantics
  if no waiter is recorded, return

contended path:
  mark lock as contended
  spin or enter futex/runtime wait
  sleep until unlock wakes a waiter
  retry acquisition after scheduled
\end{lstlisting}

这段不是某个库的源码，而是成本地图。低竞争时，锁主要是一个原子状态转换；竞争出现后，问题扩展成 cache line 所有权、等待队列、系统调用、调度唤醒和重新抢锁。报告中看到 \instruction{lock cmpxchg} 或类似形状，只能说明存在原子快路径；是否慢，要看竞争率、持锁时间、等待路径和锁字共享范围。

\begin{lstlisting}[numbers=none,caption={mutex 获取的粗粒度事件链}]
fast path:
  atomic compare-exchange lock word from unlocked to locked
  enter critical section

contended path:
  fail to acquire lock word
  optionally spin while owner may release soon
  enter kernel or runtime wait path
  sleep until unlock wakes a waiter
  become runnable, get scheduled, retry acquisition
\end{lstlisting}

这条链解释了为什么“锁慢”不是可靠结论。保护低频、复杂、多字段不变量时，mutex 往往是最清楚也足够快的选择；把每条记录的计数更新放进同一把锁里，锁变量、等待队列和临界区都会变成热点。性能报告要区分锁获取次数、竞争次数、临界区耗时、等待时间和持锁期间是否执行未知代码。没有这些分项，无法判断应缩短临界区、分片、批量、换原子，还是重新设计数据流。

\topic{自旋、睡眠和混合锁}
自旋等待适合预计锁很快释放的极短临界区，因为它避免睡眠和唤醒成本；睡眠适合等待时间可能较长的场景，因为它释放 CPU 给其他线程。混合锁先自旋少量轮次，失败后睡眠。选择哪种等待方式不是 API 品味，而是由临界区时间、线程数、CPU 预算、优先级和是否可能被抢占决定。

在 compute worker 中长时间自旋会浪费核心；在持锁线程被抢占时，其他线程继续自旋只能烧 CPU；在单核或过度订阅环境中，自旋甚至会阻止持锁线程获得时间片。条件变量把长等待转成睡眠，信号量把资源额度变成可等待计数，运行时 managed blocking 则让 worker 等待时补充执行资源。同步结构必须把等待时长当作一等变量。

\topic{完整竞争账本：从一把锁看 cache line、futex 和调度}
现在把一把 mutex 的获取过程拆成可以写入报告的事件链。低竞争路径中，线程在用户态对锁字执行一次 CAS、exchange 或等价原子操作，成功后进入临界区。这个锁字通常只有几个字节，但硬件以 cache line 为单位维护所有权；锁字所在的整条 cache line 会被获取锁的核心拉到可修改状态。若同一条 cache line 上还放了别的频繁写字段，即使这些字段和锁语义无关，也可能被拖进同一条一致性流量里，这就是 false sharing 的一个来源。

竞争路径从 CAS 失败开始。失败说明另一个线程已经拥有锁，当前线程可以选择短暂自旋、让出处理器、进入运行库等待队列，或通过 Linux futex 等机制进入内核睡眠。futex 的直觉是“先在用户态检查一个整数，只有确实需要睡眠时才请求内核把线程挂到等待队列”。它不是锁本身，而是用户态同步原语在竞争路径上使用的睡眠和唤醒接口。释放锁时，如果运行库知道存在等待者，就需要把锁字改成未持有状态，并唤醒一个或多个等待线程；被唤醒的线程只是变成 runnable，随后还要等调度器把它放回 CPU，再重新竞争锁。

\begin{lstlisting}[numbers=none,caption={锁竞争的完整事件账本}]
acquire attempt:
  read or modify lock word
  if unlocked, publish ownership and enter critical section

contention:
  fail atomic transition
  record waiter or contended state
  spin for a bounded budget if owner is expected to release soon
  enter futex or runtime wait when spinning is no longer profitable

unlock:
  finish updates protected by the mutex
  release lock word
  wake a waiter when contended state says someone may sleep

resume:
  waiter becomes runnable
  scheduler chooses a CPU
  waiter retries acquisition and may still lose the race
\end{lstlisting}

这条账本能解释很多“偶发慢”。等待线程被唤醒，不等于它立刻获得锁；它可能在 runnable 队列里等待，也可能醒来后发现另一个线程已经抢先进入临界区。释放线程调用 wake，不等于等待者一定仍在睡眠；等待者可能刚好超时、被取消、被信号打断或已经重新检查状态。同步实现必须允许这些竞态，把正确性建立在共享状态和谓词上，而不是建立在通知次数上。

性能报告因此不能只写一个总耗时。至少要拆出以下字段：

\begin{lstlisting}[numbers=none,caption={mutex 竞争报告字段}]
lock.acquire.count
lock.contended.count
lock.spin.iterations
lock.futex.wait.count
lock.futex.wake.count
lock.hold_ns.p50
lock.hold_ns.p99
lock.wait_ns.p50
lock.wait_ns.p99
lock.owner_preempted.count
critical_section.calls_unknown_code
critical_section.bytes_touched
\end{lstlisting}

\code{lock.hold_ns} 说明持锁线程在临界区里待了多久，\code{lock.wait_ns} 说明其他线程等了多久。二者不是同一件事：临界区很短但竞争极高，等待时间仍可能很大；临界区很长但低频，吞吐可能仍可接受。若 \code{owner_preempted} 很多，说明持锁线程经常在临界区内被调度走，其他线程自旋只会浪费 CPU。若 \code{calls_unknown_code} 不为零，说明锁内执行了无法界定时长的逻辑，尾延迟没有可靠上界。

\topic{从竞争账本反推设计}
拿有界队列举例，\code{mutex} 保护 \code{head/tail/size/closed} 的不变量；\code{not_empty} 和 \code{not_full} 保护等待谓词；\code{notify} 只是提示谓词可能变化。若报告显示锁竞争高，第一反应不能是把所有字段改成 atomic，而要先问四个问题。第一，临界区里是否做了拷贝、分配、日志、回调或析构。第二，队列是否太集中，是否需要每 worker 本地队列和全局注入队列。第三，是否可以批量 push 或批量 pop，把多次状态转换合并成一次。第四，等待者是否真正需要被全部唤醒，还是只需要唤醒一个能推进状态的线程。

这四个问题分别对应四类改造。缩短临界区解决持锁时间；分片队列减少锁字 cache line 迁移；批量操作降低获取次数；精确通知减少无效唤醒。只有在不变量已经收缩到单个状态字、等待协议也能由单个原子值表达时，才进入第 11 章的原子状态机设计。锁和原子不是谁替代谁，而是合同粒度不同：锁适合保护多字段一致性，原子适合承载小而明确的线性化点。

\begin{lstlisting}[language=C++,caption={互斥锁保护队列状态转换}]
class QueueState {
public:
  void push(std::int32_t value) {
    std::lock_guard<std::mutex> lock(this->mutex_);
    this->values_.push_back(value);
  }

private:
  std::mutex mutex_;
  std::vector<std::int32_t> values_;
};
\end{lstlisting}

这个示例很小，但习惯要严格：成员访问使用 \code{this->}，锁对象用 RAII，锁的作用域就是临界区。后续加入 \code{head_}、\code{tail_}、\code{size_}、\code{closed_} 后，它们都应归入同一不变量边界，直到测量证明需要拆分。

\section{futex 路径：用户态原子怎样变成内核等待}

Linux 上许多高层同步原语的竞争慢路径会落到 futex 或类似机制。futex 可以理解为 fast userspace mutex 的底层接口：正常无竞争时，用户态原子操作完成同步；只有需要睡眠或唤醒时，才进入内核。这个设计来自一个简单事实：每次加锁都进内核太贵，但长期自旋也会浪费 CPU。futex 把“先检查用户态整数”和“必要时挂到等待队列”结合起来。

\topic{wait 必须带预期值}
futex wait 的关键不是“睡眠”，而是“只有当用户态地址仍等于预期值时才睡眠”。这能避免经典竞态：线程检查到锁被占用，准备睡眠；就在进入内核前，持锁线程释放并唤醒；如果等待方无条件睡眠，它可能错过唤醒。带预期值的 wait 让内核重新读取用户态整数，若值已经变了，就不睡眠，直接返回让用户态重试。

\begin{lstlisting}[numbers=none,caption={futex wait 的竞态防护直觉}]
user space:
  if lock_word == locked:
      futex_wait(address=&lock_word, expected=locked)

kernel side:
  recheck *address
  if *address != expected:
      return immediately
  enqueue current thread on wait queue for address
  schedule away
\end{lstlisting}

这条设计说明条件变量为什么必须等待谓词。无论是 futex 还是 condition variable，通知都不是状态；状态在用户态整数或受 mutex 保护的对象里。等待者醒来后必须重新检查状态，因为醒来只说明“可能有变化”，不说明自己已经获得资源。

\topic{wake 不是交出锁}
unlock 里的 wake 也常被误解。释放线程把锁字改成 unlocked，然后唤醒一个等待者。被唤醒线程只是变成 runnable；它还要等待调度器选择 CPU，还要重新执行获取锁的原子操作，还可能输给另一个正在运行的线程。wake 不保证公平，也不保证被唤醒者立刻进入临界区。

\begin{lstlisting}[numbers=none,caption={unlock 到 waiter 真正进入临界区之间的路径}]
owner thread:
  store lock_word = unlocked with release semantics
  if lock_word indicated waiters:
      futex_wake(address=&lock_word, count=1)

waiter thread:
  wakes from kernel wait queue
  becomes runnable
  scheduler runs it later
  retries compare-exchange on lock_word
  enters critical section only if CAS succeeds
\end{lstlisting}

这个路径解释了 tail latency。某个线程的 \code{wait_ns} 可能很长，不只是因为持锁时间长，也可能因为调度延迟、优先级、CPU 过度订阅、cgroup 限制或唤醒后重新竞争失败。锁报告必须区分 hold time、wait time 和 runnable delay。

\topic{用户态锁字和内核等待队列必须保持一致}
futex 等待队列按用户态地址组织。若程序在等待者仍可能睡眠时销毁或复用锁对象，旧等待者可能挂在已经复用的地址语义上。高层库会通过对象生命周期规则避免这种错误；业务代码也必须保证 mutex、condition variable 和等待谓词的对象活得足够久。关闭队列时，先改状态，再通知所有等待者，再等待使用者退出，不能边销毁边唤醒。

\begin{lstlisting}[numbers=none,caption={销毁同步对象前必须清空等待协议}]
shutdown queue:
  lock mutex
  closed = true
  unlock mutex
  notify all waiters
  join worker threads or prove no waiter remains
  destroy mutex and condition variables
\end{lstlisting}

这不是形式主义。futex 的内核队列只知道地址，不知道 C++ 对象语义；条件变量也不会替业务证明生命周期。同步对象的生命周期是协议的一部分。

\topic{futex 观测字段}
真实系统里，futex 问题常表现为 CPU 不高但延迟高，或者线程大量处于 sleep/runnable 之间。报告可以把应用字段和系统字段对齐：

\begin{lstlisting}[numbers=none,caption={futex 和锁慢路径的观测字段}]
mutex_name
lock_word_address_or_id
acquire_attempts
contended_attempts
futex_wait_count
futex_wake_count
spurious_or_timeout_wake_count
runnable_delay_ns

\end{lstlisting}

如果没有系统级 trace，也要在应用里记录等待开始、进入睡眠前、醒来后、获得锁、释放锁的时间点。这样至少能判断等待发生在临界区、内核睡眠、调度还是重新竞争。后续运行时章节会把这种时间线扩展到 worker、task 和 completion。

\topic{从 futex 回到设计选择}
看到 futex wait 很多，不等于要删掉 mutex。可能的修复包括：缩短临界区、批量操作、分片队列、减少无效唤醒、把阻塞 I/O 移出 compute worker、提高任务粒度、或把复杂多字段状态保留在锁下。只有当状态机已经缩小到单个原子值、等待条件清楚、生命周期可证明时，才考虑 \code{atomic::wait} 或无锁结构。慢路径证据用来选择设计，不是用来制造对某个同步原语的偏见。

\section{条件变量等待的是谓词}

条件变量常被误解为事件通知系统。正确模型是：线程持锁检查共享状态；若谓词不满足，\code{wait} 原子地释放锁并睡眠；被通知或虚假唤醒后，重新持锁检查谓词。通知本身不保存状态；如果通知发生时没有等待者，它不会留在条件变量里等后来者；被唤醒也不代表条件已经成立，因为别的线程可能先拿到锁并消费资源。

\topic{谓词必须覆盖退出条件}
有界队列中，生产者等待的谓词不是“收到 \code{not_full} 通知”，而是“队列未满，或者队列已关闭”。消费者等待的谓词不是“收到 \code{not_empty} 通知”，而是“队列非空，或者队列已关闭”。关闭状态必须进入所有相关谓词，否则关闭时即使 \code{notify_all}，线程醒来后发现队列仍满或仍空，又会睡回去。

\begin{lstlisting}[language=C++,caption={条件变量等待的是谓词，不是通知次数}]
std::optional<std::int32_t> pop_wait() {
  std::unique_lock<std::mutex> lock(this->mutex_);
  this->not_empty_.wait(lock, [this]() {
    return this->closed_ || this->size_ > 0;
  });

  if (this->size_ == 0) {
    return std::nullopt;
  }

  const std::int32_t value =
      this->buffer_[static_cast<std::size_t>(this->head_)];
  this->head_ = (this->head_ + 1) %
      static_cast<std::int32_t>(this->buffer_.size());
  --this->size_;
  this->not_full_.notify_one();
  return value;
}
\end{lstlisting}

这段代码的线性化点是 \code{head/size} 更新。消费者取出元素并释放一个槽位后，通知一个生产者重新检查 \code{not_full} 谓词。若此时有多个消费者被唤醒，只有持锁后看到 \code{size > 0} 的消费者能继续；其他消费者必须继续等待或在关闭状态下退出。

\topic{notify 的时机}
常见写法是先在锁内修改状态，再通知。通知可以发生在仍持锁时，也可以在释放锁后发生，两者有性能和竞争细节差异。初始阶段不要为了微优化打乱状态逻辑：先保证所有等待者的谓词正确，所有能改变谓词结果的状态修改都在锁保护下，关闭时通知所有相关等待者。

\topic{丢失唤醒不是“少 notify 一次”}
丢失唤醒的根因通常不是通知次数少，而是“检查谓词”和“进入等待”没有作为一个受锁保护的动作。错误写法常先在锁外检查 \code{size_ == 0}，再进入 \code{wait}；如果生产者在两个动作之间完成 \code{push} 和 \code{notify}，消费者随后睡下去，就可能永远等不到下一次通知。标准条件变量的 \code{wait(lock, predicate)} 把这个模式固定下来：先持锁检查，谓词不满足时原子释放锁并睡眠，醒来后重新持锁检查。

\begin{lstlisting}[language=C++,caption={错误模式：if 等待无法抵抗虚假唤醒和关闭竞态}]
std::optional<std::int32_t> broken_pop() {
  std::unique_lock<std::mutex> lock(this->mutex_);
  if (this->size_ == 0) {
    this->not_empty_.wait(lock);
  }
  if (this->size_ == 0) {
    return std::nullopt;
  }
  return this->pop_locked();
}
\end{lstlisting}

这段代码有两个问题。第一，\code{wait} 允许虚假唤醒，醒来后不能假设队列非空。第二，关闭语义缺失：若队列空且已关闭，消费者应退出，而不是继续等待。正确写法不是在 \code{wait} 后补更多 if，而是把完整谓词写进去：\code{closed_ || size_ > 0}。同步代码的可读性来自谓词完整，而不是通知语句多。

\topic{\code{notify_one} 和 \code{notify_all} 的边界}
\code{notify_one} 适合一次状态变化最多只能让一个等待者继续，例如 \code{push} 一个元素后唤醒一个消费者，或 \code{pop} 一个元素后释放一个槽位唤醒一个生产者。\code{notify_all} 适合全局状态变化，例如 \code{close}、配置切换、取消、阶段结束。关闭时只 \code{notify_one} 很危险，因为可能有多个生产者和消费者分别睡在不同条件变量上；只醒一个等待者，其他等待者可能没有下一次状态变化来唤醒。

\begin{center}
\begin{tabular}{p{0.24\linewidth}p{0.34\linewidth}p{0.30\linewidth}}
\toprule
状态变化 & 推荐通知 & 原因 \\
\midrule
插入一个元素 & \code{not_empty.notify_one()} & 最多释放一个消费者的工作 \\
弹出一个元素 & \code{not_full.notify_one()} & 最多释放一个生产者的槽位 \\

\section{等待协议的三件事：状态、谓词、通知}

每个等待点都要写成三件事。第一，状态在哪里，由哪把锁或哪个原子保护。第二，等待谓词是什么，退出条件是否包含关闭、取消、错误和超时。第三，哪些状态变化会让谓词从 false 变成 true，并由谁负责通知。只写“在条件变量上等待”没有任何同步含义；条件变量本身不保存业务事实，事实必须在共享状态中。

\topic{状态变化必须能解释通知}
生产者 \code{push} 成功后，\code{size_} 从 0 变成 1，消费者的 \code{size_ > 0 || closed_} 谓词可能变真，所以通知 \code{not_empty_}。消费者 \code{pop} 成功后，\code{size_} 从 capacity 变成 capacity-1，生产者的 \code{size_ < capacity || closed_} 谓词可能变真，所以通知 \code{not_full_}。关闭时，\code{closed_} 从 false 变成 true，所有生产者和消费者的退出谓词都可能变真，所以两个条件变量都广播。

这套推导比背通知规则可靠。若以后增加 \code{cancelled_}、\code{error_}、\code{draining_}，必须重新检查每个等待谓词和通知点。同步代码的 bug 常来自新增状态后没有把它加入所有等待谓词。

\begin{lstlisting}[numbers=none,caption={等待点审查表}]
waiter: producer
  predicate: closed || size < capacity
  state changes that may satisfy it:
    pop decreases size
    close sets closed
  notifications:
    not_full.notify_one after pop
    not_full.notify_all after close

waiter: consumer
  predicate: closed || size > 0
  state changes that may satisfy it:
    push increases size
    close sets closed
  notifications:
    not_empty.notify_one after push
    not_empty.notify_all after close
\end{lstlisting}

\topic{丢失唤醒的真实时间线}
丢失唤醒常被误解成“少调用了一次 \code{notify}”。更准确地说，它发生在等待者检查谓词和进入睡眠之间存在未受保护窗口，而通知者在这个窗口里改变状态并发出通知。通知本身不保存业务事实；若等待者睡下时没有重新检查状态，也没有被登记到等待集合，就可能永远睡眠。

\begin{lstlisting}[numbers=none,caption={错误等待：检查和睡眠之间有窗口}]
consumer locks mutex
consumer sees size == 0
consumer unlocks mutex

producer locks mutex
producer pushes one element
producer unlocks mutex
producer notifies not_empty

consumer parks after notify has already happened
consumer sleeps even though size == 1
\end{lstlisting}

条件变量的 \code{wait(lock, predicate)} 之所以重要，是因为它把三件事组合成一个库协议：持锁检查谓词；谓词不满足时，把线程加入等待路径并释放 mutex；被唤醒后重新获得 mutex，再次检查谓词。用户代码不能把这三件事拆开。若拆开，通知和睡眠之间就会出现无主窗口。

\begin{lstlisting}[numbers=none,caption={正确等待：谓词和 wait 绑定}]
consumer locks mutex
while size == 0 and not closed:
    condition_variable_wait(mutex)
    ; wait returns only after re-locking mutex
consumer rechecks predicate under the same mutex
\end{lstlisting}

这条时间线也解释了虚假唤醒。线程醒来不一定表示资源已经可用，可能因为实现、信号、广播、竞争者先拿走资源或关闭路径触发。正确代码不依赖“为什么醒”，只依赖“醒来后在锁内重新检查谓词”。因此 \code{if} 等待是错误形状，\code{while} 或带谓词重载才是正确形状。

\topic{capacity-one 队列能放大所有错误}
容量为一的队列是同步协议的放大镜。只有一个槽位，任何遗漏都会迅速变成可复现事故：生产者和消费者交替时能暴露丢失唤醒；两个生产者同时 push 时能暴露谓词错误；close 与 push 竞争时能暴露提交点不清；消费者被唤醒后另一个消费者先拿走元素时能暴露 \code{if} 等待；异常发生在写槽位之后、递增 size 之前时能暴露状态不一致。

\begin{lstlisting}[numbers=none,caption={容量一队列的故障注入序列}]
case lost_wakeup:
  consumer reaches wait boundary
  producer pushes element and notifies
  scheduler parks consumer at the wrong time
  expected: consumer must not sleep forever

case close_full_queue:
  producer fills the only slot
  another producer waits on not_full
  close is called
  expected: waiting producer returns Closed, consumer may drain one element

case spurious_wakeup:
  consumer wakes while size == 0 and closed == false
  expected: consumer waits again without changing state
\end{lstlisting}

容量小不是为了模拟真实吞吐，而是为了把状态边界压缩到最少变量。若容量一队列都不能解释每个等待者为什么醒、为什么继续睡、为什么退出，容量变大只会把错误藏进概率。

\topic{超时等待不是简单返回 false}
超时会改变协议。生产者等待空间超时后，是返回失败、重试、丢弃、记录背压，还是关闭上游？消费者等待数据超时后，是继续轮询、上报 idle、检查取消，还是触发心跳？超时不是条件变量的附加参数，而是状态机的一条边。尤其不能把超时当作“队列永久空”或“下游永久满”的证明，它只说明在这段时间内谓词没有满足。

\topic{取消和关闭不是同一个状态}
关闭通常表示不再接受新输入，但已接受元素可以 drain；取消表示当前工作不再需要，等待者应尽快退出，已入队元素可能被丢弃或标记失败。二者的等待谓词相似，都要唤醒等待者，但语义不同。I/O 队列可能 close 后继续写完已接收 block；用户按下取消后则应停止提交新写并尽快释放资源。把二者都塞进一个 \code{closed_} 布尔值，后续恢复和错误报告会变模糊。

\begin{center}
\begin{tabular}{p{0.20\linewidth}p{0.36\linewidth}p{0.32\linewidth}}
\toprule
状态 & 语义 & 等待者行为 \\
\midrule
Open & 接收新元素，允许 pop & 按容量和非空谓词等待 \\
Closing & 拒绝新 push，保留已入队元素 & producer 退出，consumer drain \\
Cancelled & 不再要求完成旧元素 & 所有等待者退出，未处理元素进入取消摘要 \\
Closed & 队列空且生命周期结束 & 后续操作失败或无效 \\
\bottomrule
\end{tabular}
\end{center}

\section{有界队列：同步语义的最小系统}

Compute Systems Engine 的线程池、I/O 写出队列、MessageBus、分布式模拟请求队列，都需要一个共同概念：容量有限、能背压、能关闭、能 drain 的队列。这个队列不是普通容器，而是生产者和消费者之间的同步合同。

\topic{接口先表达合同}
队列接口应分层：非阻塞尝试、阻塞等待、关闭和可选 drain/timeout。非阻塞 \code{try_push/try_pop} 适合事件循环或自定义调度；阻塞 \code{push/pop} 适合普通生产者消费者；\code{close} 负责唤醒等待者并改变生命周期状态。哨兵值不是好方案，因为泛型元素未必有安全哨兵，且哨兵无法表达多个等待者和关闭策略。

\begin{lstlisting}[language=C++,caption={有关闭语义的有界队列接口草图}]
class BoundedQueue {
public:
  explicit BoundedQueue(std::int32_t capacity);

  bool try_push(std::int32_t value);
  bool push(std::int32_t value);
  std::optional<std::int32_t> pop();
  void close();

private:
  std::vector<std::int32_t> buffer_;
  std::int32_t head_ = 0;
  std::int32_t tail_ = 0;
  std::int32_t size_ = 0;
  bool closed_ = false;
  std::mutex mutex_;
  std::condition_variable not_empty_;
  std::condition_variable not_full_;
};
\end{lstlisting}

\code{push} 返回 \code{false}，表示关闭后不再接受新元素；\code{pop} 返回 \code{std::nullopt}，表示队列已关闭且没有剩余元素。这比在内部偷偷抛异常或让线程永久阻塞更适合作为运行时基础。业务层可以选择把这种状态转换成错误码、异常或取消，但底层合同必须明确。

\topic{push 和 pop 的状态转换}
生产者进入 \code{push} 后，先等待“有空间或关闭”。若关闭，返回失败；若有空间，写入槽位，更新 \code{tail/size}，通知消费者。消费者进入 \code{pop} 后，先等待“有元素或关闭”。若没有元素且关闭，返回空；若有元素，取出槽位，更新 \code{head/size}，通知生产者。

\begin{lstlisting}[language=C++,caption={push 的谓词包含关闭状态}]
bool BoundedQueue::push(std::int32_t value) {
  std::unique_lock<std::mutex> lock(this->mutex_);
  this->not_full_.wait(lock, [this]() {
    return this->closed_ ||
        this->size_ < static_cast<std::int32_t>(this->buffer_.size());
  });
  if (this->closed_) {
    return false;
  }

  this->buffer_[static_cast<std::size_t>(this->tail_)] = value;
  this->tail_ = (this->tail_ + 1) %
      static_cast<std::int32_t>(this->buffer_.size());
  ++this->size_;
  this->not_empty_.notify_one();
  return true;
}
\end{lstlisting}

\begin{lstlisting}[language=C++,caption={close 必须唤醒所有可能等待的谓词}]
void BoundedQueue::close() {
  {
    std::lock_guard<std::mutex> lock(this->mutex_);
    this->closed_ = true;
  }
  this->not_empty_.notify_all();
  this->not_full_.notify_all();
}
\end{lstlisting}

\topic{close 的线性化点和幂等性}
\code{close} 的线性化点是 \code{closed_ = true}。从这一刻开始，新的 \code{push} 必须失败；已经进入队列的元素仍可被 \code{pop} 排空；空队列上的 \code{pop} 返回 \code{std::nullopt}。重复 \code{close} 应该是安全的，因为析构、错误路径和取消路径可能从不同层次同时尝试关闭同一个队列。重复通知通常只是性能浪费，不应破坏语义；若需要减少通知风暴，可以只在第一次关闭时通知。

\begin{lstlisting}[language=C++,caption={只在第一次关闭时广播，语义仍保持幂等}]
bool BoundedQueue::mark_closed() {
  std::lock_guard<std::mutex> lock(this->mutex_);
  if (this->closed_) {
    return false;
  }
  this->closed_ = true;
  return true;
}

void BoundedQueue::close_once() {
  const bool changed = this->mark_closed();
  if (!changed) {
    return;
  }
  this->not_empty_.notify_all();
  this->not_full_.notify_all();
}
\end{lstlisting}

\topic{drain、cancel、shutdown 和析构不是同义词}
很多工业事故来自把这些词混成一个布尔量。Drain 表示拒绝新输入，但处理已接收元素；cancel 表示未开始元素应尽快退出或标记取消；shutdown 表示系统进入停止流程，可能包含 close、cancel、等待 running 任务结束和释放资源；析构要求没有线程仍可能访问对象。一个队列可以支持 close + drain，却不支持取消已经弹出的任务；线程池可以 cancel 未开始任务，但运行中任务只能协作检查停止标志。

\begin{center}
\begin{tabular}{p{0.20\linewidth}p{0.34\linewidth}p{0.34\linewidth}}
\toprule
术语 & 状态承诺 & 常见错误 \\
\midrule
close & 拒绝新输入，唤醒等待者 & 只设置 flag，不通知所有条件变量 \\
drain & 已接收元素仍会被消费 & close 后直接丢弃队列内容却不记录 \\
cancel & 未开始工作被标记取消或移除 & 把运行中线程强行停止，破坏不变量 \\
shutdown & 停止接收、排空或取消、等待线程退出 & 只关队列，不等待 running 任务 \\
destructor & 没有并发访问者，资源可释放 & 析构里调用未知回调或在持锁时 join \\
\bottomrule
\end{tabular}
\end{center}

\topic{异常安全和提交点}
示例代码用 \code{std::int32_t}，元素赋值不会抛出复杂异常。泛型队列则必须考虑移动构造、拷贝、分配和析构。状态提交点要清楚：元素成功进入槽位并更新 \code{tail/size} 后，\code{push} 才对消费者可见；元素成功移出并更新 \code{head/size} 后，槽位才重新可用。若元素操作可能抛异常，应在不改变不变量的阶段完成可能失败的工作，或者要求元素满足 \code{noexcept} move，或者使用更复杂的槽位生命周期管理。

\topic{背压是正常控制流}
队列满了，生产者等待或失败，这不是程序坏了，而是背压在工作。背压防止下游慢时上游无限制造任务和占用内存。容量不是越大越好：容量过大能隐藏短期拥塞，但会放大排队延迟和内存峰值；容量过小会频繁阻塞，降低吞吐。项目报告应记录队列满次数、生产者等待时间、最大队列长度、拒绝数和关闭耗时。

\topic{从队列到线程池：unfinished count 是另一个不变量}
线程池不能只看队列是否为空。Worker 从队列弹出任务后，任务已经不在队列里，但系统仍不 idle。线程池至少需要一个未完成计数或等价状态：提交任务时增加，任务函数执行结束后减少，异常路径也必须减少。等待空闲的谓词应是“队列为空且未完成计数为零”，关闭等待的谓词还要包含“所有 worker 已退出”。这就是为什么同步设计要从对象不变量出发，而不是从 API 名字出发。

\begin{lstlisting}[language=C++,caption={任务异常也必须提交 unfinished 状态转换}]
void ThreadPoolState::finish_task() {
  std::lock_guard<std::mutex> lock(this->mutex_);
  --this->unfinished_count_;
  if (this->unfinished_count_ == 0 && this->ready_queue_empty_) {
    this->idle_.notify_all();
  }
}
\end{lstlisting}

真实代码还要保证 \code{finish_task} 被 RAII guard 或 \code{try/catch} 包住，用户任务抛异常时也执行。若异常跳过未完成计数递减，\code{wait_idle} 会永久阻塞。若在持锁时执行用户任务，则用户任务可能递归提交任务、等待 future、记录日志或抛异常，把线程池状态机拖进未知代码。

\section{信号量：资源额度，不是对象不变量}

信号量表达“还有多少许可”。它适合连接池、I/O in-flight 限额、同时写文件数量、令牌桶、空槽/元素数量等资源计数。Mutex 保护复杂状态关系，semaphore 控制有多少个并发使用权。拿到许可后，线程仍可能需要 mutex 修改共享队列；释放许可时，还要保证失败和取消路径归还资源。

\topic{先分清四种 semaphore}
工程里说“信号量”时，至少可能指四件不同的东西：抽象计数信号量、二值信号量、POSIX \code{sem\_t} 或 C++20 \code{std::counting\_semaphore}。它们共享“许可计数”这个核心模型，但生命周期、进程边界、API 错误处理、取消/信号语义和调试方式不同。把这四者混在一起，会让读者误以为“会 \code{acquire/release} 就掌握了 semaphore”。

\begin{center}
\small
\begin{tabular}{p{0.18\linewidth}p{0.24\linewidth}p{0.25\linewidth}p{0.21\linewidth}}
\toprule
类型 & 核心状态 & 典型接口 & 适合场景 \\
\midrule
计数信号量 & \code{permits >= 0} & \code{acquire/release(n)} & 连接池、I/O in-flight、容量预算 \\
二值信号量 & \code{permits} 为 0 或 1 & \code{acquire/release} & 一次事件许可、单个后台阶段完成 \\
POSIX semaphore & \code{sem\_t} 或 named semaphore & \code{sem\_wait/sem\_post} & C/POSIX 代码、跨进程共享许可 \\
C++20 semaphore & 模板化标准库对象 & \code{acquire/try\_acquire/release} & C++ 线程内资源预算和 RAII 封装 \\
\bottomrule
\end{tabular}
\end{center}

这张表不是 API 速查，而是状态边界。计数信号量和二值信号量是语义模型；POSIX 和 C++20 是具体接口族。具体接口不能改变底层事实：semaphore 只保存许可，不保存业务对象、不保存关闭原因、不保存 payload，也不证明等待者的生命周期。

\topic{计数信号量的形式化状态}
计数信号量的核心状态可以抽象成一个非负整数 \code{permits} 和一组等待者。构造时给定初始许可数；\code{acquire} 尝试消耗一个许可；\code{release} 增加一个或多个许可，并使等待者有机会继续。这个整数不是业务对象数量，而是预算。连接池里许可数可以等于“最多允许借出多少连接”，但真实连接对象还在 free list、closing list 或 broken list 中；I/O 系统里许可数可以等于“最多允许多少个请求同时提交”，但请求 buffer、completion record 和错误状态仍在其他结构里。

\begin{lstlisting}[numbers=none,caption={counting semaphore 的最小状态机}]
state:
  permits >= 0
  waiters = threads blocked because permits == 0

acquire:
  if permits > 0:
      permits -= 1
      return acquired
  else:
      block until a release may make permits > 0
      retry acquisition

try_acquire:
  if permits > 0:
      permits -= 1
      return true
  else:
      return false

release(n):
  permits += n
  wake up to n waiters, or make permits visible to future acquirers
\end{lstlisting}

这份状态机刻意没有 \code{closed}、\code{cancelled}、\code{owner}、\code{payload}、\code{free list}。这些事实不属于信号量本身。Semaphore 的强项正是窄：它只回答“预算还有没有”。它的弱点也正是窄：只靠它不能证明对象生命周期、关闭语义和业务错误路径。

\topic{acquire 和 release 的线性化点}
线性化点是并发操作在抽象历史中“真正生效”的瞬间。对计数信号量来说，成功 \code{acquire} 的线性化点是许可计数被消耗；\code{release} 的线性化点是许可被归还并对其他线程可见。等待线程从睡眠中醒来不是线性化点，因为醒来后它还可能输给另一个线程，或发现关闭状态要求退出。正确代码必须把“拿到许可”和“拿到业务对象”分成两个提交点。

\begin{lstlisting}[numbers=none,caption={许可提交点和对象提交点不能混在一起}]
borrow connection:
  acquire permit                 # permit commit point
  lock pool mutex
  if pool is closing:
      unlock
      release permit             # undo the budget claim
      return closed
  remove object from free list    # object ownership commit point
  unlock
  return connection handle
\end{lstlisting}

这段账本能解释两个常见 bug。第一，拿到许可后发现系统关闭，必须释放许可；否则许可泄漏，所有后来者最终睡眠。第二，从 free list 取出对象失败时，不能假装许可就是对象；许可只是预算，真正的对象状态仍要在 mutex 下检查。很多 semaphore 事故不是 \code{acquire/release} API 用错，而是没有分清两个线性化点。

\topic{计数信号量：资源预算的正确用法}
计数信号量最自然的用法是把外部资源转成一个明确预算。例如最多 64 个请求进入压缩阶段，最多 8 个文件同时执行 \code{fsync}，最多 16 个连接被借出。预算耗尽时，调用方要么等待，要么超时，要么拒绝请求。这个选择由业务 SLO 决定，不由 semaphore 自动决定。

\begin{lstlisting}[numbers=none,caption={计数信号量的资源预算账本}]
resource budget:
  capacity = 64
  permits = 64 - in_flight

request enters expensive stage:
  if try_acquire succeeds:
      in_flight += 1 logically
      run stage
      release exactly once
  else:
      wait, timeout, reject, or shed load according to SLO
\end{lstlisting}

计数信号量的核心审查问题是“许可守恒”。如果系统没有关闭、取消和超时，长期运行后应满足：

\begin{lstlisting}[numbers=none,caption={许可守恒检查}]
initial_permits + release_count - acquire_success_count == current_permits
logical_in_flight == acquire_success_count - release_count
0 <= logical_in_flight <= capacity
\end{lstlisting}

真实实现不一定能直接读取准确的 \code{current\_permits}，但可以在业务层维护调试计数。若 \code{logical\_in\_flight} 长期低于容量而请求仍卡在 \code{acquire}，优先怀疑许可泄漏、等待者被旧对象挂住、或异常路径没有归还许可。

\topic{二值信号量：一次许可，不是 mutex}
\code{std::binary_semaphore} 可以看成最大计数为 1 的计数信号量。它适合表达“一个事件许可”或“最多一个后台阶段运行”。例如主线程启动后台 warmup，后台线程完成后 \code{release}，主线程 \code{acquire} 等待一次完成。这种场景中，release 的线程和 acquire 的线程本来就不是同一个线程，二值信号量的无所有者语义正好合适。

\begin{lstlisting}[numbers=none,caption={binary semaphore 适合一次完成信号}]
initial:
  ready = binary_semaphore(0)

background:
  build immutable table
  publish table pointer
  ready.release()

foreground:
  ready.acquire()
  read published table
\end{lstlisting}

反过来，用二值信号量保护 \code{vector}、map、队列或线程池状态，通常是坏设计。Mutex 的关键语义是所有者和临界区：同一线程 lock，完成受保护状态转换，再 unlock。二值信号量没有 owner，不阻止另一个线程 release；工具也更难判断谁持有临界区。若代码需要保护多字段不变量，默认用 mutex；若代码只需要一次事件许可，二值信号量才合适。

\begin{lstlisting}[numbers=none,caption={binary semaphore 和 mutex 的边界}]
binary semaphore:
  no owner
  release may happen on a different thread
  good for event permit or single in-flight budget

mutex:
  owner enters and leaves critical section
  protects object invariant while fields temporarily disagree
  better for containers, queues, pools, and runtime state
\end{lstlisting}

\topic{POSIX semaphore：C 接口、进程边界和信号中断}
POSIX semaphore 是 C/POSIX 世界里的具体接口族，常见对象是 \code{sem\_t}。未命名 semaphore 通常由 \code{sem\_init} 初始化，由 \code{sem\_destroy} 销毁；命名 semaphore 通过 \code{sem\_open} 获取，通过 \code{sem\_close} 关闭句柄，通过 \code{sem\_unlink} 删除名字。它比 C++20 semaphore 更靠近操作系统边界，也因此有更多系统级错误条件。

\begin{lstlisting}[numbers=none,caption={POSIX unnamed semaphore 的基本形态}]
sem_t sem;

sem_init(&sem, pshared, initial_value);

producer or releaser:
  sem_post(&sem);

consumer or acquirer:
  while sem_wait(&sem) == -1:
      if errno == EINTR:
          continue or handle cancellation policy
      else:
          report fatal synchronization error

shutdown:
  prove no waiter can still use sem
  sem_destroy(&sem);
\end{lstlisting}

\code{pshared} 为 0 时，semaphore 用于同一进程内线程；非 0 时，semaphore 必须放在可被多个进程共享的内存中，用于进程间同步。这个边界很重要：把进程共享 semaphore 放在普通进程私有堆内存里，不能让另一个进程看到同一个对象。命名 semaphore 则通过系统名字定位，适合不方便共享内存中放置 \code{sem\_t} 的场景，但也引入名字清理、权限和崩溃后残留的问题。

POSIX \code{sem\_wait} 可能因为 signal 返回 \code{-1} 并设置 \code{errno = EINTR}。这与 C++20 \code{acquire} 不同；C++20 接口不会把 EINTR 暴露给业务。POSIX 代码必须显式决定：收到 signal 后继续等待、返回取消、还是进入关闭路径。另一个常见错误是用 \code{sem\_getvalue} 做并发决策。它最多适合诊断，不适合作为“如果值大于 0 就一定能继续”的正确性依据，因为读取后状态可能立刻被其他线程改变。

\begin{lstlisting}[numbers=none,caption={POSIX semaphore 的工程边界}]
must handle:
  sem_wait may return EINTR
  sem_trywait may return EAGAIN when no permit exists
  sem_timedwait needs a timeout policy
  sem_destroy requires no thread can still wait or post
  process-shared sem_t must live in shared memory
  named semaphore needs close/unlink lifecycle
\end{lstlisting}

\topic{C++20 counting\_semaphore：标准库接口和限制}
\code{std::counting_semaphore<LeastMaxValue>} 是 C++20 对计数信号量的标准化封装。\code{LeastMaxValue} 是实现必须至少支持的最大计数下界，不应被误读成“这个对象运行时一定只允许这么多业务资源”。构造函数给出初始计数；\code{release(update)} 增加许可；\code{acquire} 阻塞直到成功消耗许可；\code{try_acquire} 尝试获取；\code{try_acquire_for} 和 \code{try_acquire_until} 提供超时等待。\code{std::binary_semaphore} 是 \code{std::counting_semaphore<1>} 的别名。

\begin{lstlisting}[language=C++,caption={C++20 counting\_semaphore 的最小接口形状}]
class InFlightLimit {
public:
  explicit InFlightLimit(std::ptrdiff_t initial)
      : permits_(initial) {}

  bool try_enter() {
    return this->permits_.try_acquire();
  }

  void leave() {
    this->permits_.release();
  }

private:
  std::counting_semaphore<1024> permits_;
};
\end{lstlisting}

C++20 semaphore 对象不可复制，通常也不应被随意移动进容器；它的地址和生命周期应稳定。标准没有给它业务关闭语义，也不承诺严格 FIFO 公平。若要支持取消、关闭、超时错误码和 drain，必须在外层状态机实现。还要注意 \code{try_acquire} 允许失败，业务代码不能把一次失败解释成永久无资源；带超时接口也必须记录 timeout 是正常背压、用户取消还是系统关闭。

\begin{lstlisting}[numbers=none,caption={C++20 semaphore 审查清单}]
C++20 counting_semaphore checklist:
  initial count matches resource budget
  every successful acquire has exactly one release
  timeout path records whether work was accepted
  close/cancel state is stored outside the semaphore
  object outlives all waiting and releasing threads
  code does not depend on FIFO fairness
  try_acquire failure is not treated as a correctness proof
\end{lstlisting}

从内存可见性看，semaphore 也不能替代对象协议。一个线程在 release 前写入某个对象，另一个线程 acquire 后读取该对象，通常可以建立生产者到消费者的同步边界；但对象本身的所有权、生命周期和别名仍要由外层合同证明。不要因为用了 semaphore，就在没有互斥或单所有权协议的情况下并发修改同一个容器。

\topic{C++20 中的共同边界：没有 close，也没有 ownership}
\code{std::counting_semaphore} 和 \code{std::binary_semaphore} 都不绑定线程所有者：一个线程可以 \code{acquire}，另一个线程可以 \code{release}。这与 mutex 完全不同。Mutex 表达的是“这个线程进入临界区并负责退出”；semaphore 表达的是“系统预算增加或减少一份”。因此用 binary semaphore 当 mutex，通常会丢掉所有权、调试和不变量语义。

\begin{center}
\small
\begin{tabular}{p{0.23\linewidth}p{0.30\linewidth}p{0.35\linewidth}}
\toprule
能力 & semaphore 提供吗 & 工程后果 \\
\midrule
计数许可 & 提供 & 适合容量、in-flight 和令牌桶 \\
线程所有权 & 不提供 & 不能替代 mutex 的 owner 语义 \\
复杂谓词 & 不提供 & close/drain/cancel 仍要额外状态 \\
业务 payload & 不提供 & 数据必须在队列、pool 或 record 中 \\
公平性保证 & 标准不承诺强公平 & 不能把它当严格 FIFO 调度器 \\
对象生命周期证明 & 不提供 & 等待者退出前不能销毁关联状态 \\
\bottomrule
\end{tabular}
\end{center}

公平性尤其容易被误解。实现可能让等待者大体按某种队列醒来，也可能允许正在运行的线程在释放后很快再次获取许可。标准层面不应依赖“最早等待者一定最早获得许可”。若业务需要严格顺序，应在受 mutex 保护的队列或调度器里表达顺序，而不是把公平性寄托给 semaphore。

\topic{许可守卫：异常路径必须自动归还}
凡是 \code{acquire} 后有多个返回路径，就应把许可归还写进 RAII。否则异常、取消、早返回和错误分支会慢慢吞掉许可。许可泄漏不像崩溃那样明显：系统会先变慢，然后并发度越来越低，最后所有线程阻塞在 \code{acquire}，看起来像下游服务挂住。

\begin{lstlisting}[language=C++,caption={许可守卫把 release 绑定到作用域}]
class PermitGuard {
public:
  explicit PermitGuard(std::counting_semaphore<1024>& permits)
      : permits_(&permits), owns_(true) {
    this->permits_->acquire();
  }

  PermitGuard(const PermitGuard&) = delete;
  PermitGuard& operator=(const PermitGuard&) = delete;

  PermitGuard(PermitGuard&& other) noexcept
      : permits_(std::exchange(other.permits_, nullptr)),
        owns_(std::exchange(other.owns_, false)) {}

  ~PermitGuard() {
    if (this->owns_) {
      this->permits_->release();
    }
  }

  void dismiss() noexcept {
    this->owns_ = false;
  }

private:
  std::counting_semaphore<1024>* permits_;
  bool owns_;
};
\end{lstlisting}

\code{dismiss} 只能在许可所有权已经转移到另一个对象时调用，例如连接 handle 接管归还责任。不要为了“避免析构 release”随手调用 \code{dismiss}。这类 guard 的代码审查问题很明确：每个 \code{acquire} 是否有唯一归还路径；所有权转移后谁负责 release；失败路径是否归还；对象析构是否可能在等待者仍存在时发生。

\begin{lstlisting}[numbers=none,caption={许可泄漏的证据形状}]
symptom:
  request throughput gradually decreases
  active in-flight count is below configured limit
  many threads blocked in semaphore acquire
  no downstream saturation is visible

likely cause:
  acquired permits - released permits keeps growing
  early return, exception, timeout, or cancellation skipped release
\end{lstlisting}

\topic{双信号量 bounded buffer 的严格账本}
用两个信号量写有界队列时，\code{empty_slots} 代表可写槽预算，\code{filled_slots} 代表可读元素预算。它的好处是容量和元素数量的等待对象很清楚；代价是关闭语义更难，因为标准 semaphore 不知道 \code{closed}。先看不含关闭的正常路径：

\begin{lstlisting}[numbers=none,caption={双信号量 bounded buffer 的正常路径}]
initial:
  empty_slots = capacity
  filled_slots = 0
  head, tail, buffer protected by mutex

push:
  empty_slots.acquire()
  lock mutex
  write buffer[tail]
  tail = next(tail)
  unlock mutex
  filled_slots.release()

pop:
  filled_slots.acquire()
  lock mutex
  read buffer[head]
  head = next(head)
  unlock mutex
  empty_slots.release()
\end{lstlisting}

这条账本里有三个必须保持的关系。第一，\code{empty_slots + filled_slots == capacity}，除非某个线程已经拿到许可但尚未完成锁内提交；这段短窗口必须被代码审查标出来。第二，\code{filled_slots} 的 release 必须发生在元素写入并提交之后，否则消费者可能拿到“可读许可”却读到未初始化槽。第三，\code{empty_slots} 的 release 必须发生在元素读出并槽位重新可写之后，否则生产者可能覆盖尚未被安全移出的元素。

\topic{关闭语义：semaphore 不能广播一个不存在的许可}
条件变量可以在 \code{closed = true} 后 \code{notify_all}，等待者醒来检查 \code{size > 0 || closed}。Semaphore 没有“状态变化但不增加资源”的通知语义。若消费者睡在 \code{filled_slots.acquire()}，队列关闭且没有元素时，应该让它退出；但调用 \code{filled_slots.release()} 会制造一个看起来真实的元素许可。生产者睡在 \code{empty_slots.acquire()} 时也有同样问题：关闭后唤醒它们，不等于真的多了空槽。

\begin{lstlisting}[numbers=none,caption={semaphore 队列关闭时最容易写错的地方}]
bad close:
  closed = true
  filled_slots.release(number_of_consumers)

why bad:
  fake filled permits look like real elements
  consumers must add another check after acquire
  permit accounting becomes tied to waiter count
  a late consumer may consume a fake permit after close logic changes
\end{lstlisting}

这不是说 semaphore 不能用于队列，而是说关闭协议要显式设计。常见选择有三种。第一，队列核心仍用 \code{mutex + condition_variable}，把 semaphore 只用于外层 in-flight 限额。第二，使用支持停止令牌或超时的获取路径，等待者周期性检查 \code{closed}，但要接受超时轮询成本和复杂性。第三，向队列插入结束哨兵，但这要求元素类型和消费者数量能承载该协议，且必须证明哨兵不会与业务数据混淆。对通用线程池队列，默认选择仍应是 CV，因为 close/drain/cancel 是业务谓词，不是资源额度。

\topic{二值信号量、mutex 和 latch 不要混用}
二值信号量、mutex 和 latch 都可能让线程“等一下”，但等待对象不同。二值信号量等一个可重复的一份许可；mutex 等对象临界区；latch 等一次性倒计数归零。若要等 N 个 worker 初始化完成，用 latch；若要限制最多一个后台 flush in-flight，用 binary semaphore 或普通 mutex，取决于是否需要 owner 和对象不变量；若要保护 \code{head/tail/size/closed}，用 mutex。

\topic{令牌桶和背压：semaphore 是容量控制，不是排队策略}
在服务端，semaphore 常用于 admission control：同时最多允许 N 个请求进入某个昂贵阶段。拿不到许可时，可以等待、拒绝、降级或排队。选择哪一种不是 semaphore 决定的，而是 SLO 决定的。等待会保护下游但增加队列延迟；直接拒绝能保护尾延迟但牺牲成功率；按优先级排队需要额外调度结构。

\begin{lstlisting}[numbers=none,caption={用 semaphore 表达 in-flight 预算}]
request arrives:
  if permits.try_acquire():
      enter expensive stage
  else if latency_slo_is_strict:
      reject or shed load
  else:
      wait with timeout and record queue_wait_ns

request finishes:
  release permit exactly once
\end{lstlisting}

因此 semaphore 相关指标至少要包括 \code{permit_acquire_count}、\code{permit_wait_ns}、\code{permit_timeout_count}、\code{permit_reject_count}、\code{permit_leak_check} 和 \code{in_flight_high_watermark}。若只记录吞吐，无法区分“下游慢导致等待增加”和“许可泄漏导致并发度下降”。

\topic{令牌和队列容量不是一回事}
令牌可以限制并发 I/O 数量，但不等于队列容量。一个系统可能允许队列里积压 100 个待写 block，但同时只允许 4 个 fsync in-flight；也可能允许 32 个 RPC in-flight，但请求队列容量按内存字节控制。把所有限制都写成一个队列长度，会混淆内存预算、设备并发度和业务优先级。第 15 章的异步 I/O 会复用这个区分。

\section{一次性完成、阶段同步和一次性结果}

Mutex/CV 和 semaphore 解决的是对象状态与资源额度；另一类常见等待对象是“某个阶段到了没有”。这里要分三种：latch 表达一次性归零，barrier 表达多轮固定参与者阶段，future/promise 表达一次性值或异常。三者都常被误写成“等一下”，但状态模型完全不同。

\topic{latch：一次性倒计数}
\code{std::latch} 表达的是一次性完成门闩。它内部有一个计数；参与者完成某个一次性阶段后调用 \code{count\_down}；等待者调用 \code{wait}，直到计数归零后通过。归零之后，latch 永远保持打开，不能重置到下一轮。这一点决定了它适合“等待所有 worker 初始化完成”“等待 N 个分片加载完 metadata”“等待测试线程同时准备好后开始压测”这类一次性边界。

\begin{lstlisting}[numbers=none,caption={latch 的状态机}]
state:
  count = N
  opened = false

count_down:
  if count > 0:
      count -= 1
  if count == 0:
      opened = true
      wake all waiters

wait:
  if opened:
      return immediately
  else:
      block until opened
\end{lstlisting}

标准接口要按状态机读。\code{count\_down(n)} 表示有 \code{n} 个一次性参与份额完成；\code{try\_wait()} 只观察是否已经打开；\code{wait()} 等待打开；\code{arrive\_and\_wait(n)} 等价于先贡献 \code{n} 个倒计数再等待打开。所有这些操作都围绕同一个不可重置计数工作。构造时给的计数就是本次门闩的全部债务，后续不能补参与者，也不能进入第二轮。

\begin{center}
\small
\begin{tabular}{p{0.26\linewidth}p{0.36\linewidth}p{0.26\linewidth}}
\toprule
接口 & 语义 & 审查问题 \\
\midrule
\code{count\_down(n)} & 提交 \code{n} 个一次性完成份额 & 所有异常和早退路径是否都会执行 \\
\code{try\_wait()} & 非阻塞观察计数是否归零 & 不能用来轮询可重复事件 \\
\code{wait()} & 等待计数归零 & 等待者是否有失败状态可读 \\
\code{arrive\_and\_wait(n)} & 倒计数后等待同一个 latch 打开 & 调用线程是否本身也是参与者 \\
\bottomrule
\end{tabular}
\end{center}

Latch 的关键不是互斥，也不是资源许可，而是“一次性归零”。它不保护对象不变量；它不保存每个参与者的结果；它不处理某个参与者失败后如何取消其他参与者。若 worker 初始化可能失败，外层仍要有错误状态，例如 \code{init_error}、错误队列或异常聚合。Latch 只保证所有成功或失败路径都调用了倒计数时，等待者不会早于计数归零继续。

\begin{lstlisting}[language=C++,caption={latch 等待所有 worker 完成初始化}]
std::latch initialized(worker_count);
std::atomic<bool> failed{false};

worker:
  try {
    initialize_local_state();
  } catch (...) {
    failed.store(true, std::memory_order_release);
  }
  initialized.count_down();

controller:
  initialized.wait();
  if (failed.load(std::memory_order_acquire)) {
    enter_startup_failure_path();
  }
\end{lstlisting}

常见错误是把 latch 当 condition variable 使用。队列从空到非空会发生很多次，线程池 idle 也会在 busy 和 idle 之间反复切换；latch 一旦打开就不会再次关闭，无法表达这些可重复谓词。另一个错误是漏掉失败路径的 \code{count\_down}。如果某个 worker 初始化抛异常后直接退出，控制线程会永久等待。可靠代码应使用 scope guard 或统一的 worker 入口，保证所有路径都会倒计数。

\begin{lstlisting}[numbers=none,caption={latch 最常见的异常路径事故}]
bad_worker:
  initialize_part_a()
  if part_b_failed:
      return               # count_down is skipped
  initialized.count_down()

robust_worker:
  on_scope_exit initialized.count_down()
  try:
      initialize_part_a()
      initialize_part_b()
  catch:
      publish init_error
\end{lstlisting}

Latch 的错误用法通常有三个信号。第一，代码想把 latch “关回去”，说明等待对象不是一次性完成，而是可重复阶段或谓词。第二，等待者需要知道每个 worker 的结果，却只看到 latch 打开，说明结果通道缺失。第三，参与者数量来自动态队列长度，运行过程中还会变化，说明应该建任务计数、future 聚合或 barrier drop 协议，而不是在 latch 上猜测。

\topic{barrier：多轮阶段同步}
\code{std::barrier} 表达的是多轮阶段边界。它有参与者数量和当前阶段。每个参与者完成本阶段工作后调用 \code{arrive\_and\_wait}；最后一个到达者触发阶段完成函数，然后所有参与者进入下一阶段。与 latch 不同，barrier 会重复使用，每轮结束后进入下一代 phase。

\begin{lstlisting}[numbers=none,caption={barrier 的多轮状态机}]
state:
  expected = participant_count
  arrived = 0
  phase = 0

arrive_and_wait:
  local_phase = phase
  arrived += 1
  if arrived == expected:
      run completion step for local_phase
      arrived = 0
      phase += 1
      wake all waiters for local_phase
  else:
      wait until phase != local_phase
\end{lstlisting}

Barrier 的接口比 \code{arrive\_and\_wait} 更细。\code{arrive()} 表示“我对当前 phase 的贡献已经完成”，并返回一个 arrival token；调用者可以在不再读写本 phase 共享边界的前提下做一点独立工作，随后把 token 交给 \code{wait}。这个 token 不是资源许可，而是当前 phase 的等待凭证。调用 \code{arrive()} 之后还继续修改本 phase 会制造 phase skew：其他线程可能已经触发完成函数并进入下一轮。

\begin{center}
\small
\begin{tabular}{p{0.26\linewidth}p{0.36\linewidth}p{0.26\linewidth}}
\toprule
接口 & 语义 & 审查问题 \\
\midrule
\code{arrive()} & 到达当前 phase，返回等待 token & 到达后是否还会触碰本 phase 数据 \\
\code{wait(token)} & 等待 token 对应 phase 完成 & token 是否只被等待一次 \\
\code{arrive\_and\_wait()} & 到达并等待 phase 推进 & 每个继续参与者每轮是否恰好一次 \\
\code{arrive\_and\_drop()} & 到达本轮并永久退出后续 phase & 退出是生命周期事件还是异常漏调用 \\
completion function & 最后到达者执行阶段完成动作 & 是否短、确定、不阻塞、不抛异常 \\
\bottomrule
\end{tabular}
\end{center}

Barrier 适合 stencil、仿真、多阶段流水和分片批处理：所有分片先完成本轮计算，再统一交换边界或提交下一轮。它不适合动态任务队列，因为参与者数量和每轮到达次数必须被协议定义。若某个参与者提前退出，其他参与者可能永远卡在当前 phase；若一个线程在同一 phase 到达两次，状态机也会被破坏。C++20 \code{barrier} 支持 \code{arrive\_and\_drop}，允许参与者永久退出后减少后续 phase 的 expected count，但这必须是显式生命周期事件，不是异常路径随手漏掉一次等待。

\begin{lstlisting}[language=C++,caption={barrier 表达多轮分片阶段}]
std::barrier step_done(worker_count, [] {
  publish_phase_summary();
});

worker:
  for (std::size_t phase = 0; phase != phase_count; ++phase) {
    compute_local_partition(phase);
    step_done.arrive_and_wait();
    consume_published_summary(phase);
  }
\end{lstlisting}

完成函数不是任意回调仓库。它运行在阶段切换的关键路径上，应该短、确定、不能阻塞 I/O，不能提交新一轮需要同一个 barrier 的工作，也不能抛出让参与者状态不一致的异常。若完成函数需要聚合大量数据，应让每个 worker 在到达前写入本地结果，完成函数只做轻量发布或版本推进。

\begin{lstlisting}[numbers=none,caption={barrier 阶段同步的可审查账本}]
phase p, worker i:
  read input snapshot for phase p
  compute local partition
  write local boundary and local metrics for phase p
  arrive at barrier exactly once

completion for phase p:
  publish summary for phase p
  advance global phase to p + 1

after release:
  workers may read summary for phase p
  workers start phase p + 1
\end{lstlisting}

阶段同步最危险的不是 API 调错，而是 phase 语义没有写成合同。若 completion function 里等待网络、写日志、提交任务或拿另一个可能被 worker 持有的锁，所有参与者都会卡在阶段边界。若一个 worker 因过滤条件本轮没有工作却直接跳过 barrier，其他 worker 会永久等待。若参与者数量随任务队列变化，就要用任务图、计数器或 continuation 表达动态依赖；barrier 适合固定参与者的重复阶段，不适合“谁有任务谁来一下”的队列。

\toprule
错误用法 & 失败原因 & 正确方向 \\
\midrule
用 barrier 管动态任务队列 & 每轮 expected count 不稳定 & 任务队列加 unfinished 计数或 task graph \\

\topic{future/promise：一次性结果和异常通道}
\code{std::future} 和 \code{std::promise} 不是队列、锁或 barrier，而是一份一次性 shared state 的两个端口。\code{promise} 是生产端，负责把一个值或异常写入 shared state；\code{future} 是消费端，负责等待并取出结果。这个原语最适合“后台加载完成后把结果交给控制线程”“一次异步请求返回一个结果”“任务异常要跨线程传播”这类一对一结果通道。

\begin{lstlisting}[numbers=none,caption={future/promise 的一次性共享状态机}]
state:
  Empty(producer alive)
  ValueReady(value)
  ExceptionReady(exception)
  BrokenPromise
  Consumed                 # std::future::get has taken the result

promise.set_value(v):
  Empty -> ValueReady(v), wake waiters

promise.set_exception(e):
  Empty -> ExceptionReady(e), wake waiters

promise destroyed while Empty:
  Empty -> BrokenPromise, wake waiters

future.get():
  wait until ready
  if ValueReady: return value and consume state
  if ExceptionReady: rethrow and consume state
  if BrokenPromise: throw future_error
\end{lstlisting}

这里的“ready”只表示结果槽位已被填充，不表示某个队列关闭、不表示所有任务完成、不表示运行时 idle。\code{std::future::get} 对普通 future 只能调用一次，因为它消费结果；多个观察者需要 \code{std::shared\_future}。生产端也只能满足一次：重复 \code{set\_value} 或先 \code{set\_value} 后 \code{set\_exception} 都是协议错误，标准库会以 \code{future\_error} 报告。

\topic{shared state 不是 future 对象本身}
\code{std::future} 和 \code{std::promise} 都只是访问同一份 shared state 的句柄。Shared state 才保存 ready 标志、结果存储、异常对象、等待队列、引用关系和同步边界。\code{promise.get\_future()} 只是在这份状态上创建消费端；\code{future} 移动之后，旧 future 变成 invalid，shared state 仍然存在；\code{promise} 移动之后，生产端责任也随之移动。把 future 对象本身当成结果槽，会导致生命周期判断错误。

\begin{lstlisting}[numbers=none,caption={future shared state 的概念字段}]
FutureSharedState<T>:
  producer_present: bool
  consumer_present: bool
  ready: bool
  result_storage: optional<T>
  exception_storage: exception_ptr
  waiters: condition_variable/futex waiters
  continuations: callbacks or coroutine handles
  mutex_or_atomic_protocol: publishes ready and result

std::promise<T>:
  owns producer endpoint of the shared state

std::future<T>:
  owns single-consumer endpoint of the shared state

std::shared_future<T>:
  shares read-only access after ready
\end{lstlisting}

这解释了几个标准库现象。\code{future.valid()} 只说明这个句柄是否还关联 shared state，不说明结果是否 ready。\code{wait()} 只等待 ready，不消费结果；\code{get()} 等待 ready 后消费普通 future 的结果；\code{shared\_future::get()} 可以被多次调用，因为它不把共享状态从消费者侧移走。Shared state 的生命周期通常持续到最后一个相关句柄释放，而不是到任务函数返回那一刻。

\topic{端点生命周期：\code{promise} 和 \code{future} 不是两个对象互相等}
把 shared state 看成中心对象之后，\code{promise} 和 \code{future} 就只是两个端点。生产端负责把 Pending 推到某个终态；消费端负责观察终态并取出值或异常。端点可以移动，shared state 不会因为句柄移动而移动。端点可以提前析构，shared state 也不一定立刻销毁；它必须活到所有可能观察它的句柄和等待者都结束。

\begin{lstlisting}[numbers=none,caption={future shared state 的端点生命周期}]
create promise:
  allocate shared state
  producer endpoint is present

promise.get_future:
  create exactly one ordinary future endpoint
  repeated get_future is a protocol error

move promise:
  producer endpoint moves to the new promise object
  old promise no longer owns producer responsibility

move future:
  consumer endpoint moves to the new future object
  old future.valid() becomes false

destroy promise while Pending:
  Pending -> BrokenPromise
  wake waiters

destroy future before ready:
  consumer endpoint disappears
  producer may still run unless runtime has a cancellation contract

destroy last endpoint after terminal state:
  shared state storage may be reclaimed
\end{lstlisting}

这几个转移决定了很多工程边界。第一，\code{promise.get\_future()} 通常只能调用一次，因为普通 \code{future} 是单消费端；要广播结果，应先转成 \code{shared\_future} 或用发布快照。第二，consumer 提前析构不等于 producer 被取消；标准 future 没有取消语义。第三，producer 析构且还没满足结果，不是静默丢弃，而是 broken promise。运行时若把 rejected task、canceled task、abandoned future 都混成一个布尔值，就无法给调用者正确错误。

\topic{标准操作表：每个 API 都是状态转移}
下面这张表不是背 API，而是给代码审查用的状态账本。任何线程池封装 \code{std::promise}、\code{std::packaged\_task} 或自研 future-like 类型，都应逐项回答这些操作在 Pending、Ready、Canceled、Broken、Consumed 下如何表现。

\begin{center}
\small
\begin{tabular}{p{0.23\linewidth}p{0.34\linewidth}p{0.29\linewidth}}
\toprule
操作 & 状态影响 & 工程边界 \\
\midrule
\code{promise::get\_future} & 创建普通消费端 & 只能一次；重复调用是协议错误 \\
\code{promise::set\_value} & Pending -> ValueReady & 必须发布结果后再唤醒等待者 \\
\code{promise::set\_exception} & Pending -> ExceptionReady & 异常对象进入 shared state，不停在 worker 栈 \\
\code{promise} 析构 & Pending -> BrokenPromise & 等待者必须醒来并得到可诊断错误 \\
\code{future::wait} & 只等待 ready & 不消费结果，不执行任务，不取消 producer \\
\code{future::wait\_for} & 在期限内尝试观察 ready & timeout 不表示任务结束或取消 \\

\bottomrule
\end{tabular}
\end{center}

最容易错的是把 \code{wait\_for(timeout)} 的 timeout 当成取消成功。Timeout 只是调用者本轮停止等待。producer 可能还在执行，结果稍后仍然会进入 shared state。若 timeout 后上层释放了 producer 需要写入的 buffer，后续 completion 就会写悬垂内存。正确做法是把 timeout、stop request、producer acknowledgement 和 shared state 终态分成四个事实，不要让一个返回值承担所有语义。

\begin{lstlisting}[numbers=none,caption={timeout 不等于取消完成}]
caller:
  if future.wait_for(10 ms) == timeout:
      request_stop()
      keep shared state and captured storage alive
      decide whether to wait, detach, or observe cancellation later

producer:
  if stop requested:
      publish Canceled or ExceptionReady
  else:
      publish ValueReady or ExceptionReady
\end{lstlisting}

\topic{异常传播是 shared state 的终态}
跨线程异常不能靠 worker 日志完成。生产端必须把异常捕获成 \code{std::exception\_ptr} 写入 shared state；消费端 \code{get()} 看到异常终态时在调用线程重新抛出。这样调用者能用普通 \code{try/catch} 处理后台任务失败，错误也不会被 worker 线程吞掉。

\begin{lstlisting}[language=C++,caption={异常必须进入 promise，而不是停在 worker 日志}]
std::promise<LoadReport> promise;
std::future<LoadReport> future = promise.get_future();

worker:
  try {
    promise.set_value(load_metadata());
  } catch (...) {
    promise.set_exception(std::current_exception());
  }

caller:
  try {
    LoadReport report = future.get();
    use(report);
  } catch (const std::exception& e) {
    handle_background_failure(e);
  }
\end{lstlisting}

若 producer 在 shared state 仍为 Empty 时析构，consumer 不应永远等待；标准库会让 consumer 观察到 broken promise，并在 \code{get()} 时抛出 \code{std::future\_error}。这不是小角落，而是生命周期协议：每个被接纳的异步任务都必须以 value、exception、canceled 或 broken promise 之一结束。若运行时取消任务，也要把取消转成 shared state 中可观察的终态，而不是只从 ready queue 删除任务。

\topic{阻塞等待的语义边界}
\code{wait()}、\code{wait\_for()}、\code{wait\_until()} 和 \code{get()} 都围绕同一件事：等待 shared state 进入 ready 终态。它们不保证哪个线程执行生产者，不保证队列被排空，不保证运行时仍有 worker 可用，也不保证超时后任务已经取消。超时只说明本次等待期限内没有观察到 ready；任务可能稍后成功、失败或取消。

\begin{lstlisting}[numbers=none,caption={future 等待不能被误读成调度保证}]
future.wait:
  blocks current thread until shared_state.ready may be true
  does not execute queued tasks
  does not create worker threads
  does not cancel the producer on timeout

future.get:
  waits until ready
  returns value, rethrows exception, or reports broken promise
  consumes ordinary future result

wait_for timeout:
  means caller stopped waiting for now
  producer may still be running
  shared state must remain valid
\end{lstlisting}

因此，持有 mutex 时调用 \code{get()} 是危险形状：producer 可能需要同一把锁才能写结果或推进状态，consumer 却在锁内睡眠。在线程池 worker 内调用同池 future 的 \code{get()} 也危险：等待者占住了执行被等待任务的 worker。外部控制线程阻塞等待一次后台结果通常可以接受；compute worker 内长期阻塞等待必须被运行时显式管理。

\topic{阻塞等待的真实时间线}
阻塞等待不是一个魔法暂停点。普通 mutex/CV 实现通常可以按下面的时间线理解；futex 或平台事件对象只是把 sleep/wake 的慢路径换成内核设施，语义仍然是“状态先改变，通知后观察”。

\begin{lstlisting}[numbers=none,caption={future.get 的 park/unpark 时间线}]
consumer future.get:
  lock shared_state
  if state is terminal:
      consume or rethrow
      unlock
      return
  register waiter
  while state is Pending:
      atomically release lock and park current thread
      wake up and reacquire lock
  remove waiter
  consume or rethrow terminal state
  unlock

producer set_value/set_exception:
  lock shared_state
  write result or exception storage
  state = ValueReady or ExceptionReady
  detach waiters and continuations
  unlock
  notify waiters
  schedule continuations
\end{lstlisting}

这条线有三个审查点。第一，consumer 必须在检查 Pending 和进入等待队列之间保持原子性；否则 producer 可能已经通知，而 consumer 还没登记，形成丢失唤醒。第二，producer 必须先写结果、再改 ready、再通知；否则 consumer 可能醒来读到未构造完成的结果。第三，通知和调度 continuation 应在释放 shared state 锁之后发生，避免在内部锁下执行未知代码。

\midrule

\topic{线程池集成：submit 返回 future 不是自动安全}
线程池常见接口是 \code{submit(callable) -> future<R>}。这个接口只在语义上承诺“一次被接纳的 callable 会满足 shared state”。它没有自动解决容量、关闭、取消、同池等待和异常传播。正确集成至少要保证四件事：提交被拒绝时不给调用者一个永远不 ready 的 future；任务函数正常返回时写 value；任务函数抛异常时写 exception；运行时取消已接纳任务时写取消终态或异常。

\begin{lstlisting}[numbers=none,caption={submit 返回 future 的最低合同}]
submit:
  if runtime is not Open:
      return Rejected, no pending future
  create shared state
  enqueue task wrapper that owns producer endpoint
  return future endpoint to caller

task wrapper:
  try:
      result = callable()
      shared_state.set_value(result)
  catch:
      shared_state.set_exception(current_exception)

cancel accepted task before running:
  shared_state.set_exception(task_canceled)
  decrement unfinished count
  wake waiters
\end{lstlisting}

\code{std::packaged\_task} 可以把 callable 和 future shared state 包在一起，适合写最小线程池。但它仍然只是“callable 执行一次后满足结果”的工具，不是运行时。任务队列容量、worker 是否能阻塞等待、取消如何传播、shutdown drain 是否执行已接纳任务，都必须由线程池自己的状态机定义。第 14 章会把这条线推进到 task node、continuation 和 coroutine。

\topic{coroutine：把等待端变成 continuation 注册}
C++ coroutine 不是另一种线程，也不是自动并行。它把函数的局部状态放进 coroutine frame，在 \code{co\_await} 处可以暂停当前函数，把恢复动作注册到某个 awaitable 上；生产者完成时再恢复这个 coroutine。换句话说，coroutine 把“线程阻塞在 \code{get()}”改成“shared state ready 后调度 continuation”。这正好对应 future 的 shared state 模型。

\begin{lstlisting}[numbers=none,caption={co\_await future-like 状态的语义}]
await_ready:
  if shared_state.ready:
      continue without suspension

await_suspend(coroutine_handle h):
  register h as continuation in shared_state
  return to scheduler without occupying worker thread

producer completion:
  store value or exception
  mark shared_state ready
  schedule registered coroutine handles

await_resume:
  return value or rethrow exception
\end{lstlisting}

这里的关键边界是调度位置。恢复 coroutine 不应在持有 shared state 锁时直接执行用户代码；更稳的做法是把 coroutine handle 作为 continuation 放回 executor 或 runtime 队列。否则一个完成路径可能在锁内重入用户代码，制造锁顺序反转、长临界区和难以追踪的递归调用。Coroutine 购买的是非阻塞等待表达能力，不是替代运行时调度。

\topic{coroutine lowering：顺序写法背后是 frame 和状态机}
C++ coroutine 的难点不在 \code{co\_await} 这个关键字，而在编译器把函数拆成了可暂停状态机。一个 coroutine 调用不会简单创建线程；它通常分配或嵌入一个 coroutine frame，frame 里保存参数、局部变量、当前暂停点、返回对象的 \code{promise\_type} 和恢复入口。每次 \code{co\_await} 都是在问 awaitable 三个问题：现在能不能继续，若不能继续把我挂到哪里，恢复时给我什么结果或异常。

\begin{lstlisting}[numbers=none,caption={coroutine frame 的概念字段}]
CoroutineFrame:
  promise_object
  parameters and live locals
  current_suspend_point
  continuation_handle
  exception_slot
  result_slot or reference to FutureState
  allocator and destroy protocol

promise_type:
  get_return_object
  initial_suspend
  final_suspend
  return_value or return_void
  unhandled_exception
\end{lstlisting}

\code{initial\_suspend} 决定 coroutine 创建后是立刻运行，还是先把 handle 交给调用者或 executor。\code{final\_suspend} 决定 coroutine 结束时如何通知等待者或恢复 continuation。\code{return\_value} 把普通返回值写入 promise 或 FutureState。\code{unhandled\_exception} 把逃逸异常保存起来，让 \code{await\_resume} 或最终 \code{get} 能在正确线程重新抛出。若这些函数没有定义清楚，coroutine 代码看起来像同步函数，错误传播却比回调更难诊断。

\begin{lstlisting}[numbers=none,caption={co\_await 的三段合同}]
co_await expression:
  awaiter = get awaiter from expression
  if awaiter.await_ready():
      result = awaiter.await_resume()
      continue in the same call path
  else:
      suspend current coroutine frame
      decision = awaiter.await_suspend(current_handle)
      if decision means suspended:
          return to caller or scheduler
      later, executor resumes current_handle
      result = awaiter.await_resume()
\end{lstlisting}

\code{await\_suspend} 是最危险的点。它运行在 coroutine 已经暂停、但还没有完全交出控制权的边界上。若 awaitable 在这里把 handle 注册到 shared state 后，producer 同时完成，就会出现竞态：completion 可能试图恢复一个刚注册的 handle。健壮实现需要在同一把锁或同一个原子状态机下处理“注册 continuation”和“发现已经 ready”两条路径。

\begin{lstlisting}[numbers=none,caption={await\_suspend 的竞态处理}]
await_suspend(handle h):
  lock shared_state
  if state is terminal:
      unlock
      enqueue h to executor
      return suspended_or_transfer
  append h to continuation list
  unlock
  return suspended

completion:
  lock shared_state
  state = terminal
  continuations = move continuation list
  unlock
  enqueue continuations to executor
\end{lstlisting}

\topic{final suspend：结束点不能把 frame 过早销毁}
很多 coroutine 事故发生在结束路径。用户代码执行完并不等于 frame 可以立刻销毁，因为还有等待者可能需要读取结果、异常或恢复 continuation。可靠的 \code{final\_suspend} 往往返回一个 awaiter：它在 coroutine 终态发布后，把 continuation 放回 executor，然后让外部 owner 在安全时机销毁 frame。

\begin{lstlisting}[numbers=none,caption={coroutine 结束路径的状态账本}]
body returns value:
  promise_type.return_value stores result
  enter final_suspend
  publish terminal state
  schedule continuation
  frame remains until owner destroys it

body throws:
  promise_type.unhandled_exception stores exception_ptr
  enter final_suspend
  publish ExceptionReady
  schedule continuation
  await_resume later rethrows

consumer destroys task object before ready:
  according to API:
    request cancellation, detach, or destroy only if coroutine is not running
\end{lstlisting}

因此，coroutine 类型的析构合同必须非常明确。若 task 对象析构就直接 \code{destroy()} frame，而 producer 线程或 executor 还可能恢复它，就是 use-after-free。若 task 析构只 detach，却没有任何 owner 最终销毁 frame，就是泄漏。若取消只是丢掉 continuation，不把 FutureState 推到终态，等待者会永久 pending。协程不是免生命周期的语法糖，它把生命周期从调用栈搬到了 heap 或运行时管理的 frame。

\begin{lstlisting}[language=C++,caption={promise 必须在成功和失败路径都满足共享状态}]
std::promise<LoadReport> promise;
std::future<LoadReport> result = promise.get_future();

loader_thread:
  try {
    LoadReport report = load_metadata();
    promise.set_value(std::move(report));
  } catch (...) {
    promise.set_exception(std::current_exception());
  }

controller:
  try {
    LoadReport report = result.get();
    publish_loaded_manifest(report);
  } catch (const std::exception& e) {
    enter_startup_failure_path(e);
  }
\end{lstlisting}

\code{std::async} 也返回 \code{future}，但它不是生产级 task runtime。未显式指定策略时，实现可以选择异步执行，也可以选择 deferred：任务直到 \code{get} 或 \code{wait} 才在等待线程上运行。指定 \code{std::launch::async} 可以要求异步执行，但仍然缺少队列容量、任务分类、worker 可观测性、取消、背压和调度策略。\code{std::packaged\_task} 可以把 callable 包成 future 生产端；它仍然只解决“一次 callable 产生一次结果”，不解决运行时调度。

\topic{latch、barrier、future、semaphore、CV 的选择边界}
这些原语都能让线程等待，但等待对象完全不同。Latch 等一次性归零；barrier 等每一轮所有参与者到达；future 等一次性结果或异常；semaphore 等资源许可；condition variable 等受 mutex 保护的业务谓词。把它们互换，代码通常还能编译，状态机却会错。

\begin{tabular}{p{0.20\linewidth}p{0.30\linewidth}p{0.36\linewidth}}

\topic{C++20 和标准库同步原语的适用边界}
\code{std::latch} 适合一次性等待 N 个参与者完成，例如主线程等待所有 worker 初始化；\code{std::barrier} 适合多轮阶段同步，例如每轮模拟都要等所有分片完成再进入下一轮；\code{std::future}/\code{std::promise} 适合一次性结果和异常传播；\code{std::jthread} 和 \code{std::stop_token} 适合把协作停止变成显式参数；\code{std::atomic::wait} 适合围绕单个原子状态等待。它们都不能替代队列合同：队列仍要定义容量、所有权、关闭、drain 和异常路径。

\section{同步原语选择族谱：把坐标系统一回工程报告}

前面已经分别展开了复杂对象谓词、资源许可、一次性完成、多轮阶段和一次性结果。现在把它们收束成工程报告格式。只会说“mutex 慢，atomic 快”不是并发能力；真正要能说清的是：等待事实是什么，状态在哪里，哪个转移会让等待完成，失败和关闭如何退出，运行时证据如何证明这个选择是对的。

\topic{按统一模型填写报告}
一份同步设计报告不应该只写“使用 mutex”或“使用 semaphore”。它应该把等待事实压成固定字段。下面是同一套字段在不同场景里的填写方式：

\begin{tabular}{p{0.18\linewidth}p{0.22\linewidth}p{0.22\linewidth}p{0.24\linewidth}}
\toprule

\midrule
bounded queue pop & \code{size > 0 || closed} & mutex 下的 \code{size/closed} & 多字段谓词，选 CV \\
连接池借用 & 一个借出许可 & semaphore 计数 + pool mutex & 许可和对象列表分层 \\
worker 初始化 & N 个一次性完成份额 & latch 计数 + 错误状态 & 只开一次，不能重置 \\

\end{tabular}

这张表比 API 速查更重要。它把“为什么选这个原语”落到等待对象上，也把“为什么不选别的原语”变成可审查结论。比如 bounded queue 不选 semaphore，不是因为 semaphore 慢，而是因为 close/drain/cancel 是多字段业务谓词；连接池可以用 semaphore，但只能限制借出预算，不能替代 pool mutex；future 可以返回一次后台加载结果，但不能表达一条长期任务流。

\topic{反向排除比正向选择更能暴露错误}
很多同步事故不是选不出原语，而是排除理由没写。报告中必须有 \code{rejected\_alternatives}。如果写不出为什么不用另一个原语，通常说明等待对象还没定义清楚。

\begin{lstlisting}[numbers=none,caption={反向排除理由示例}]
bounded_queue:
  chosen: mutex + condition_variable
  rejected:
    counting_semaphore:
      permits cannot broadcast close without fake slots
    atomic_wait:
      queue state spans size, head, tail, closed, cancelled
    future:
      queue produces a stream, not one value
    barrier:
      producers and consumers are dynamic, not fixed participants

phase_stencil:
  chosen: barrier
  rejected:
    latch:
      phase repeats; latch cannot close again
    condition_variable:
      possible, but barrier directly encodes fixed participants and phase
    semaphore:
      permits do not encode all participants reached the same generation
\end{lstlisting}

\topic{从报告反推测试}
报告字段必须能导出故障注入。等待对象是谓词，就测试 close 期间等待者是否退出；等待对象是许可，就测试异常路径是否泄漏许可；等待对象是阶段，就测试参与者跳过一轮是否卡住；等待对象是 future shared state，就测试 value、exception、broken promise 和同池等待饥饿。没有这种从模型到测试的映射，章节仍然只是概念说明。

\begin{tabular}{p{0.16\linewidth}p{0.23\linewidth}p{0.21\linewidth}p{0.24\linewidth}}

这些类别不是互斥的工程模块，而是不同层次。一个连接池可能同时使用 semaphore 限制借出许可，用 mutex 保护空闲连接链表，用 condition variable 管理关闭时的 drain，用 future 把一次后台健康检查结果交给控制线程，用 eventfd 唤醒事件循环。错误不在于“用了多种原语”，而在于没有写清每种原语表达哪一层事实。后面几段不再重新讲 API，而是按这张矩阵检查每一类原语的边界。

\topic{semaphore 只表达许可，不表达所有权和生命周期}
信号量的计数是抽象许可，不是对象本身。拿到许可说明容量预算允许继续，不说明你已经拥有某个 C++ 对象，也不说明系统没有关闭。比如连接池：

\begin{lstlisting}[numbers=none,caption={semaphore 许可和连接对象是两份状态}]
borrow:
  permits.acquire()          # budget says one borrower may proceed
  lock pool mutex            # object list still needs protection
  if closing:
      unlock
      permits.release()
      return Closed
  take one connection from free list
  unlock

return:
  lock pool mutex
  put connection back, or mark it broken
  unlock
  permits.release()
\end{lstlisting}

这个例子暴露两个常见事故。第一，\code{borrow} 拿到许可后发现池已关闭，必须归还许可；否则系统会泄漏许可，最终所有线程都睡在 \code{acquire}。第二，连接坏了不能简单 \code{release} 后塞回池里；许可计数和对象健康状态是两层。信号量适合表达“最多 N 个 in-flight”，不适合单独表达“这些对象的列表、状态和生命周期都正确”。

\topic{condition variable 只保存等待队列，不保存业务事实}
条件变量不记住业务事件。队列里有没有元素、线程池是否 idle、运行时是否关闭，都在受 mutex 保护的字段里。条件变量只让等待者在谓词暂时不满足时睡眠，并在状态可能变化后回来重查。

\begin{lstlisting}[numbers=none,caption={CV 谓词必须写完整}]
bad:
  wait until queue_size > 0

better:
  wait until queue_size > 0 or closed or cancelled or error

why:
  empty + closed should exit
  empty + cancelled should exit
  empty + error should report failure
  wakeup without data should continue waiting
\end{lstlisting}

因此，新增一个状态字段就必须重新审查所有 CV 谓词。给队列加 \code{cancelled}，却只在 \code{push} 路径检查取消，消费者仍可能睡在 \code{not\_empty}；给线程池加 \code{failed}，却不唤醒 \code{wait\_idle}，控制线程会永远等待。CV 的正确性不来自 \code{notify} 次数，而来自完整谓词和所有状态变化的通知覆盖。

\topic{\code{atomic::wait} 适合版本，不适合复杂谓词}
\code{atomic::wait} 的强项是单个原子值。它等待的是“这个值不再等于我看到的旧值”。这非常适合版本号、单状态枚举、一次性发布标志和窄 ready bit。它不适合跨多个字段的队列状态。

\begin{lstlisting}[numbers=none,caption={atomic wait 的合适形态}]
reader:
  old = version.load(acquire)
  version.wait(old)
  new = version.load(acquire)
  reload immutable snapshot

writer:
  publish immutable snapshot
  version.fetch_add(1, acq_rel)
  version.notify_all()
\end{lstlisting}

反例是把线程池 idle 压成一个原子布尔：

\begin{lstlisting}[numbers=none,caption={错误：idle 不是单个布尔事实}]
wrong_idle_flag:
  ready_queue_empty = true
  running_count may still be nonzero
  submitter may be between increment and enqueue
  a completion callback may enqueue more work
  shutdown may be draining accepted tasks
\end{lstlisting}

这里等待谓词跨 ready queue、running count、submission in progress、shutdown state 和 failure state。把它压成一个 atomic bool 会制造提前返回。正确路径是先用 mutex 写清多字段谓词；若后续发现某个窄版本发布路径是热点，再把那个路径单独改成 atomic wait。

\topic{\code{eventfd} 唤醒事件循环，不承载 payload}
\code{eventfd} 是 Linux fd 形式的计数器，适合把“有 completion 了”通知给 epoll/io\_uring 所在的线程。它不保存业务数据，不表达队列容量，也不替代 mutex。正确形态通常是：业务数据写入 completion queue；再写 eventfd 唤醒事件循环；事件循环读 eventfd 后 drain completion queue。

\begin{lstlisting}[numbers=none,caption={eventfd 的正确分层}]
worker thread:
  lock completion queue
  push completion record
  unlock
  write eventfd counter by 1

event loop:
  epoll_wait returns eventfd readable
  read eventfd counter
  lock completion queue
  drain completion records
  unlock
\end{lstlisting}

错误形态是把 eventfd 当业务消息队列。eventfd 只有一个 64-bit counter；多次写入可以合并成一个可读计数，它不会保存每个请求的 id、错误码、buffer 所有权或完成顺序。payload 必须在另一个受保护的数据结构中，eventfd 只负责把事件循环从睡眠中叫醒。

\topic{futex 是等待慢路径，不是业务抽象}
futex 的核心是“用户态地址上的整数状态 + 内核等待队列”。它解决的是高层同步原语的睡眠慢路径：用户态先检查值，若值仍等于 expected，再让内核把线程挂到该地址对应的等待队列上。它不懂 C++ 对象生命周期、不懂队列 close、不懂元素所有权。

\begin{lstlisting}[numbers=none,caption={futex 语义边界}]
what futex knows:
  user address
  expected integer value
  wait queue keyed by address
  wake count

what futex does not know:
  C++ object lifetime
  queue closed/draining/cancelled state
  whether a task was accepted
  whether a permit leaked
  whether a wake is business-complete
\end{lstlisting}

因此，直接拿 futex 写业务队列时，最难的不是系统调用语法，而是证明地址生命周期、等待谓词、关闭唤醒和对象复用都正确。高层代码通常应先用标准库 mutex/CV/semaphore 建立合同，再通过性能证据决定是否需要更底层的等待协议。

\topic{六个错误案例：为什么“能跑”不等于选对原语}
下面这些案例适合写进实验报告：

\begin{lstlisting}[numbers=none,caption={同步原语错误选择案例}]
case semaphore_as_mutex:
  use a binary semaphore to protect a vector
  forget owner/lifetime rules
  destructor runs while a waiter still holds or waits
  result: race, permit leak, or use-after-free

case cv_without_close_predicate:
  consumer waits for size > 0
  controller sets closed = true and notifies
  consumer wakes, sees size == 0, waits again
  result: shutdown hangs

case atomic_wait_for_thread_pool_idle:
  worker pops last ready task and sets queue_empty = true
  controller observes idle flag
  running task has not finished
  result: wait_idle returns before work is complete

case barrier_for_dynamic_task_queue:
  worker with no local tasks skips phase p
  other workers call arrive_and_wait for phase p
  expected count is never reached
  result: phase p never completes

case future_wait_inside_same_pool:
  each worker submits a child task to the same pool
  each worker then calls future.get()
  child tasks stay ready but no worker is runnable
  result: starvation without a mutex deadlock

case eventfd_as_payload_queue:
  worker writes eventfd for each completion
  event loop reads only aggregate counter
  no completion record exists elsewhere
  result: request identity and error state are lost
\end{lstlisting}

这些案例的共同点是：原语表面上都能“等”，但等待对象错了。同步设计要先写业务事实，再选原语；不能先选一个看起来高级的 API，再倒推业务语义。

\section{锁顺序、性能路径和工业现实}

单个锁保护单个队列容易证明；多个锁组合后，死锁和尾延迟变得真实。死锁的必要条件包括互斥、持有并等待、不可抢占和循环等待。工程上最常见的破坏方式是规定锁顺序，或者把组合操作改成消息传递/两阶段提交，避免同时持有多把锁。

\topic{锁顺序表}
项目应维护锁顺序表。例如：全局配置锁先于任务图锁，任务图锁先于队列锁，队列锁先于指标锁。任何反向获取都应被重构或严格证明不会并发。锁顺序表还要写明哪些锁持有时禁止调用用户回调、禁止 I/O、禁止等待 future。持锁进入未知代码，比持有两把锁更危险，因为未知代码可能再进入运行时并获取另一把锁。

\topic{优先级反转和 convoy}
优先级反转发生在低优先级线程持有锁，高优先级线程等待它，而中等优先级线程持续占用 CPU，使低优先级线程迟迟不能释放锁。普通服务端也有类似现象：前台请求等待后台 checkpoint 持有的锁，尾延迟被放大。Lock convoy 则是大量线程排队等待同一锁，释放后唤醒、抢占和再次竞争降低吞吐。

公平锁可以减少饥饿，但可能降低吞吐；非公平锁吞吐可能更高，但某些线程等待时间更长。同步原语的选择要服务业务目标：批处理更关注总吞吐，交互服务更关注尾延迟和饥饿边界。

\topic{mutex 的 fast path 和 futex 慢路径}
Linux 上常见 mutex 在无竞争时尽量只做用户态原子状态改变；竞争严重时，线程通过 futex 进入内核睡眠，释放方再唤醒等待者。条件变量等待也会释放 mutex 并进入等待，唤醒后重新竞争 mutex。理解这点后，“锁慢”就不再是单一结论：低竞争短临界区很便宜，高竞争睡眠唤醒会涉及内核和调度，过度订阅下自旋会浪费 CPU。

\topic{同步性能要按等待原因归因}
只记录吞吐不够。同步性能问题要拆成：锁等待时间、临界区执行时间、条件变量睡眠时间、被唤醒后重新竞争锁的时间、队列满导致的背压时间、队列空导致的 idle 时间、关闭广播后的退出时间。若没有这些指标，团队很容易把“下游 I/O 慢”误判成“mutex 慢”，或者把“用户任务太长”误判成“condition variable 唤醒慢”。

\section{完整执行账本：从 wait 到 futex 再到关闭}

条件变量最容易被写成“通知机制”，但底层路径更接近“围绕共享谓词的睡眠协议”。下面把消费者等待空队列、生产者插入任务、关闭队列、线程池等待 idle 四条路径写成账本。账本的目标不是模拟某个标准库实现，而是固定每个状态变化的语义边界：谁持有锁，哪个谓词被检查，线程是否还在 run queue，通知之后为什么还要重新检查。

\topic{空队列等待的事件链}
消费者进入 \code{pop}，持有 mutex 后看到 \code{size == 0} 且 \code{closed == false}。此时继续持锁睡眠会死锁，因为生产者无法进入临界区插入元素；先释放锁再睡眠又可能丢通知。因此条件变量的 \code{wait} 必须把“把自己加入等待集合、释放 mutex、进入睡眠”做成一个不会被普通用户代码拆开的协议。

\begin{lstlisting}[numbers=none,caption={消费者等待空队列的语义事件链}]
consumer locks queue mutex
consumer observes size == 0 and closed == false
consumer enters condition-variable wait
wait operation releases mutex and parks the thread atomically
kernel/runtime removes the thread from runnable execution
producer can now acquire mutex and change queue state
\end{lstlisting}

这里的“atomically”不是说 CPU 执行一条神奇指令，而是说标准库和内核配合，避免在“已经决定睡眠但尚未真正等待”这个窗口里错过通知。Linux futex 路径的常见思想是：用户态先检查某个内存状态，只有仍然不满足时才让内核挂起线程；唤醒方改变内存状态后唤醒等待者。状态仍是权威，futex 只是等待队列。

\topic{生产者插入后的唤醒链}
生产者持锁写入槽位，更新 \code{tail} 和 \code{size}，这时谓词 \code{size > 0 || closed} 从 false 变成 true。通知可以在锁内或锁外发生，但必须发生在状态变化之后。被唤醒的消费者不是立刻运行：它先从睡眠变成 runnable，等待调度器分配 CPU，随后重新竞争 mutex，最后再次检查谓词。

\begin{lstlisting}[numbers=none,caption={notify 到真正执行之间仍有调度路径}]
producer locks queue mutex
producer writes element into tail slot
producer increments size
producer unlocks mutex
producer notifies not_empty
waiting consumer becomes runnable
scheduler eventually runs consumer
consumer re-locks queue mutex
consumer rechecks size > 0 or closed
\end{lstlisting}

这条链解释了两个现象。第一，\code{notify_one} 之后消费者的延迟不只由条件变量决定，还包含调度延迟和重新拿锁的时间。第二，醒来的消费者可能发现队列又空了，因为另一个消费者先拿到锁并取走元素。谓词循环不是保守写法，而是条件变量正确性的中心。

\topic{关闭广播的事件链}
关闭是全局状态变化，不能只唤醒一个等待者。生产者可能睡在 \code{not_full}，消费者可能睡在 \code{not_empty}，等待 idle 的线程可能睡在 \code{idle}。关闭必须改变共享状态，再广播所有受影响的等待点，让每个等待者回到锁内重新判断退出路径。

\begin{lstlisting}[numbers=none,caption={队列关闭的语义事件链}]
close locks queue mutex
close sets closed = true
close unlocks queue mutex
close notifies not_empty waiters
close notifies not_full waiters
each waiter wakes, re-locks, rechecks predicate
push waiters return Closed
pop waiters drain remaining elements or return empty
\end{lstlisting}

关闭时如果只通知 \code{not_full}，空队列上的消费者不会醒；如果只通知 \code{not_empty}，满队列上的生产者不会醒；如果谓词不包含 \code{closed}，醒来的线程可能重新睡回去。可靠关闭是“状态 + 所有等待点 + 完整谓词”的组合。

\topic{wait idle 的不变量}
线程池的 \code{wait_idle} 不是等待队列为空。队列为空只能说明没有尚未领取的 ready task；worker 已经领取并正在执行的 task 不在队列里。正确谓词通常是：ready 队列为空，unfinished count 为零，并且没有正在提交的任务跨过计数边界。若任务执行中抛异常，仍必须递减 unfinished count；若 completion 回调递归提交任务，必须先增加 unfinished count 再把任务放入队列，防止 \code{wait_idle} 看到瞬时空状态提前返回。

\begin{lstlisting}[numbers=none,caption={wait idle 的谓词账本}]
submit task:
  lock runtime
  increment unfinished_count
  push task to ready queue
  notify worker

worker finish task:
  run task outside runtime lock
  lock runtime
  decrement unfinished_count
  if unfinished_count == 0 and ready queue is empty:
      notify idle waiters

wait idle:
  lock runtime
  wait until unfinished_count == 0 and ready queue is empty
\end{lstlisting}

这条账本比“任务队列为空”更强，也更接近运行时真实语义。第 14 章的任务图会把 ready、running、finished、failed 和 canceled 状态写得更细；这里先固定同步底座：等待者必须等待系统状态，而不是等待某次通知。

\topic{惊群和 convoy}
\code{notify_all} 能保证等待者有机会重新检查状态，但也可能制造惊群：很多线程同时醒来，只有少数线程能取得资源，其余线程重新睡眠。关闭、取消和阶段完成必须广播；普通 push 一个元素时通常只需要 \code{notify_one}。如果队列容量很小、生产者频繁 \code{notify_all}，系统会在唤醒、调度、抢锁和再睡眠之间浪费大量时间，表现为吞吐下降和 p99 上升。

Lock convoy 则常出现在高竞争单锁路径：一个线程释放锁后，多个等待者被唤醒或排队，锁在核心间迁移，下一位持锁者又执行较长临界区。队列、指标、日志、内存分配若都塞在同一锁下，convoy 会把短操作拖成尾延迟。拆锁前要先写不变量边界，不能把一个完整状态机拆成数据竞争。

\begin{casebox}
\textbf{Linux futex。}
futex 的核心工程思想是：同步状态放在用户态地址，内核只在需要睡眠和唤醒时介入。它购买的能力是低争用 fast path 和高争用 blocking path 的分离；代价是用户态必须正确维护等待条件，避免丢失唤醒。Compute Systems Engine 不实现 futex，但要吸收“状态检查和睡眠必须成对”这条原则。

\textbf{pthread mutex/condition variable。}
pthread 把互斥、等待、唤醒和调度交给成熟库和内核配合处理。它购买的是可移植、可靠和经过大量边界打磨的同步语义；代价是抽象层隐藏了一些性能细节。示例代码应使用标准库 RAII 包装，而不是手写锁原语。

\textbf{Linux wait queue 和 workqueue。}
内核 wait queue 强调等待条件和唤醒来源；workqueue 强调工作项与执行线程分离。两者共同提醒我们：等待者要围绕状态睡眠，工作项不等于线程。Compute Systems Engine 的 \code{QueueContract} 和 worker 退出协议应吸收这个原则。

\textbf{生产级 executor。}
oneTBB、Folly executor、Java executor 和 Go scheduler 都把“提交工作”“执行工作”“等待完成”“关闭运行时”拆成不同协议。成熟设计通常不会让队列析构隐式杀死 worker，也不会把任务函数放在队列锁内执行。它们购买的是清楚生命周期和可观测性；代价是状态更多，测试矩阵更大。Compute Systems Engine 应先实现清楚的锁版 executor，再把热点迁移到更细的队列或 work stealing。

\textbf{数据库后台线程。}
RocksDB compaction、PostgreSQL background writer、Kafka log cleaner 这类后台任务通常有队列、条件变量、停止标志、in-flight 工作和错误传播。它们最重视的不是“锁最少”，而是关闭时不丢已提交事实、故障路径可恢复、等待状态可诊断。可迁移原则是：后台线程必须能解释自己在等什么，停止请求必须能唤醒所有等待点。

\textbf{RCU。}
RCU 通过“读取方读旧版本，写者发布新版本，延迟回收旧对象”优化读多写少路径。它购买的是极轻读路径；代价是对象生命周期和回收协议复杂。Compute Systems Engine 可在配置快照或路由表章节借鉴“不可变版本发布”，但不要用 RCU 替代需要复杂关闭语义的队列。
\end{casebox}

\section{读写锁、RCU 和原子的边界}

读写锁允许多个读取方并发、写者独占，适合读临界区相对较长、写入低频、读路径不修改共享状态的对象，例如配置表或路由快照。它不适合每次请求都要更新统计、LRU、引用计数或访问时间的“伪读操作”。读临界区极短时，读写锁维护读取方计数和公平策略的成本可能超过普通 mutex。

RCU 或 copy-on-write 更适合不可变快照：读取方读取稳定版本，写者复制并发布新版本，旧版本延迟回收。它让读路径很轻，却把复杂度转移到写路径和内存回收。第 12 章会深入生命周期和回收问题。

Atomic 适合更窄的协议：统计计数、一次性发布、小状态机、槽位 ready 标志、单生产者单消费者 ring 的索引。Atomic 不是锁的替代品。若一个状态需要维护多字段不变量、复杂关闭和等待队列，先用 mutex/reference 写清语义，再考虑是否有足够窄的原子协议。

\section{QueueContract：项目中的同步合同}

\code{QueueContract} 不是一个必须存在的 C++ 类，而是每个队列必须填写的工程合同。线程池队列、I/O 队列、MessageBus 队列、completion 队列可以使用不同实现，但必须回答同一组问题：容量按任务数还是字节数计算；满时阻塞、失败还是丢弃；空时等待还是返回；关闭后是否允许 drain；是否支持 timeout；元素所有权何时转移；等待谓词是什么；关闭时通知哪些等待者。

\topic{从队列迁移到线程池}
线程池比队列多两个状态：正在运行的任务和未完成计数。队列为空不代表线程池 idle，因为 worker 可能已经取出任务正在执行；队列关闭不代表任务完成，因为已接收任务可能还在 running。线程池的 \code{wait_idle} 谓词应围绕 unfinished count，而不是队列长度。任务抛异常时，也必须减少 unfinished count 并通知等待者。

\topic{从队列迁移到 I/O 背压}
I/O 队列还要考虑字节容量和 in-flight 许可。一个 \code{ShuffleBlock} 可能大小差异很大，只按任务数限制会导致内存峰值失控。可靠写出还会引入 temp file、checksum、fsync、rename、manifest commit，队列只负责交付 block 描述，不能把长 I/O 放进队列锁。第 15 章会把这里的背压合同扩展成完整 I/O 管线。

可关闭有界队列是本章最小综合路径。实现一个 \code{BoundedQueue<int32_t>} 或等价示例队列，覆盖容量为 1、容量为 2、多个 producer/consumer、空队列 close、满队列 close、重复 close、运行中 close、消费者慢、生产者慢。每个元素包含 producer id 和 sequence，最终检查不丢、不重、每个 producer 内部顺序符合接口声明。证据保留 push 等待次数、pop 等待次数、最大队列长度、close 到所有线程退出的时间。

\topic{测试要攻击等待路径}
并发测试应故意制造极端条件：队列容量为一，worker 数为一或二，随机 \code{yield}，随机短 sleep，固定 seed 重放失败，任务抛异常，close 与 push/pop 并发。大容量和大线程数有时只会掩盖状态机错误。ThreadSanitizer 可以发现数据竞争，但会改变性能，不能用于性能结论。

\topic{反例驱动的验收}
实验不应只提交一个“正确版本”。报告要包含至少三个反例：等待谓词不含 \code{closed_} 导致空队列 close 卡住；任务异常跳过 \code{unfinished_count_} 递减导致 \code{wait_idle} 卡住；持队列锁执行任务导致任务递归提交时死锁。每个反例都要有可复现 seed、线程数量、容量和事件日志。反例能够被复现和修复，才说明同步原理进入了工程判断。

\begin{sourcereadingbox}
源码阅读按“等待为何能正确醒来”这条线组织。
\begin{enumerate}
  \item Linux futex：阅读 \filepath{kernel/futex/core.c} 相关等待和唤醒路径，重点看用户态地址、等待队列和 wake 的关系，不需要追完整优先级继承实现。
  \item Linux mutex/wait queue：阅读 \filepath{kernel/locking/mutex.c} 和 \filepath{include/linux/wait.h}，说明 fast path、阻塞路径和唤醒条件如何分层。
  \item pthread 或 libc++ 同步原语：观察 \code{std::condition_variable} 如何映射到底层 pthread，理解标准库为什么要求配合 \code{std::unique_lock}。
  \item 工业队列或 executor：找一个有 shutdown/drain 语义的队列，分析它怎样定义关闭、取消和等待者唤醒。
\end{enumerate}
报告必须从一个具体失败开始，例如“close 后消费者不醒”或“任务异常后 wait 永久阻塞”，然后把源码机制和项目 \code{QueueContract} 对上。
\end{sourcereadingbox}

\section{完整案例：容量为一的队列如何暴露所有同步错误}

容量为一的有界队列是最好的同步显微镜。容量大时，许多错误会被缓冲掩盖；容量为一时，生产者和消费者几乎每一步都要交接状态，满、空、关闭、异常和通知都会被频繁触发。若一个队列在容量为一、多生产者、多消费者、随机调度和并发关闭下仍能解释所有状态，扩大容量只是工程问题；若容量为一就说不清，复杂系统只会把错误埋得更深。

\topic{状态变量必须能解释每一种等待}
队列至少有四类状态：缓冲区内容、\code{size} 或 head/tail、\code{closed} 标志、等待者关心的容量和空非空谓词。生产者等待的是“有空间或关闭”；消费者等待的是“有元素或关闭”；关闭者改变的是“未来不再接纳新元素，并唤醒所有可能等待的人”。这些谓词不是注释，而是条件变量等待的核心。

\begin{lstlisting}[numbers=none,caption={容量为一队列的谓词}]
producer may push when:
  size < capacity and not closed

producer must stop when:
  closed

consumer may pop when:
  size > 0

consumer may finish when:
  size == 0 and closed
\end{lstlisting}

若 \code{pop} 只等待 \code{size > 0}，空队列 close 时消费者会永久睡眠；若 \code{push} 只等待 \code{size < capacity}，满队列 close 时生产者也可能永久睡眠。关闭必须进入所有等待谓词。

\topic{线性化点要和通知配套}
\code{push} 的线性化点是元素进入队列并且 \code{size} 增加的状态变化。通知消费者应发生在这个状态变化之后。若先通知再写元素，消费者醒来后可能仍看不到元素；若写元素后没有通知，消费者可能一直睡到下一次偶然通知。由于等待者会在锁下重新检查谓词，通知在锁内还是锁外不是最核心问题；核心是状态变化和通知不能脱节。

\begin{lstlisting}[numbers=none,caption={push 的状态和通知顺序}]
push:
  lock
  wait until size < capacity or closed
  if closed:
      return Closed
  write element into slot
  size += 1
  unlock
  notify not_empty
\end{lstlisting}

\code{pop} 对称：取走元素，\code{size} 减少，通知 \code{not_full}。若元素类型移动构造可能抛异常，必须决定异常发生在线性化点之前还是之后。更稳的工程做法是把可能抛异常的构造尽量放在进入临界区之前，临界区内只做不会失败的移动或状态交换；若无法保证，就要写异常安全状态表。

\topic{Close 不是销毁}
\code{close} 表示不再接收新元素，并唤醒等待者。它不是析构，也不是清空队列。Drain 语义下，close 后消费者仍可取走已经入队的元素；队列为空且 closed 时，消费者返回结束。Cancel 语义则可以丢弃未处理元素，但必须把丢弃写成可观察事实。两者不能混用。

\begin{lstlisting}[numbers=none,caption={drain close 的事件链}]
initial:
  size = 1, closed = false

producer calls close:
  lock
  closed = true
  unlock
  notify all not_empty and not_full

consumer pop:
  sees size = 1, pops last element
  later pop sees size = 0 and closed = true
  returns Closed
\end{lstlisting}

重复 close 应幂等。第一个 close 改变状态并通知；后续 close 可以返回 already closed，但不能破坏队列。析构则要求更强：不能在仍有生产者或消费者访问队列时销毁内部 mutex 和 condition variable。对象生命周期必须由外层 join 或 shared ownership 管理。

\topic{容量为一的失败日志}
一次典型 bug 是生产者在满队列等待，另一个线程 close，但 close 只通知 \code{not_empty}，没有通知 \code{not_full}。消费者已经退出，生产者永远睡在 \code{not_full} 上。容量为一时很容易复现：

\begin{lstlisting}[numbers=none,caption={关闭没有唤醒生产者的失败日志}]
T0 consumer pops first item and exits early
T1 producer pushes item 7, queue becomes full
T2 producer tries to push item 8, waits on not_full
T3 controller closes queue, notify not_empty only
T2 remains asleep although closed == true
\end{lstlisting}

修复不是“多 notify 几次”这么粗。正确修复是：所有等待谓词包含 \code{closed}；close 状态变化后广播所有相关条件变量；醒来的生产者看到 closed，返回 Closed；醒来的消费者要么 drain 剩余元素，要么看到空且 closed 后退出。测试应验证 close 后所有等待线程能在有限时间内退出。

\topic{从同步状态推出报告字段}
队列报告至少记录 push 成功、push closed、push 等待次数、pop 成功、pop closed、pop 等待次数、close 次数、最大 size、等待总时间和最长等待。这样当系统卡住时，可以知道谁在等空间、谁在等元素、close 是否发生、等待者是否被唤醒。同步原语本身很底层，但工程排障需要这些业务化指标。

\begin{lstlisting}[numbers=none,caption={QueueReport 的语义字段}]
push_success
push_closed
push_wait_count
push_wait_ns
pop_success
pop_closed
pop_wait_count
pop_wait_ns
close_count
max_size
\end{lstlisting}

这些字段不能都在同一热锁内高频更新，否则报告本身会成为竞争源。可以在每个 worker 本地累积统计，周期性汇总；但控制面状态，例如 \code{closed} 和队列 size，仍必须由队列锁或严格原子协议保护。

\section{接口延伸}

锁、条件变量、信号量、latch、barrier 和 future 不是一组可以随便替换的“等待 API”，而是不同状态模型的工具。Mutex 保护不变量；condition variable 等待谓词；semaphore 表达资源额度；latch 表达一次性归零；barrier 表达多轮阶段边界；future/promise 表达一次性结果或异常。同步设计的质量取决于状态机是否完整、等待者是否有退出路径、关闭和异常是否覆盖、锁内是否只做可控状态转换、测试是否故意攻击边界。

这里留下的项目对象是 \code{QueueContract}。第 9 章的 \code{ThreadPoolPlan} 和 \code{WorkerSnapshot} 需要队列合同才能可靠 shutdown；第 14 章的任务运行时会在队列之上增加 ready count 和 unfinished count；第 15 章的 I/O 管线会把容量从任务数扩展到字节和 in-flight；第 16 章以后的 MessageBus 会复用同样的 close/drain/backpressure 语义。
