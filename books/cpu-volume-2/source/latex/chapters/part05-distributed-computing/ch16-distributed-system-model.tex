\chapter{分布式计算模型}

分布式计算把计算扩展到多台机器，但同时引入网络通信和故障。共享内存消失后，线程之间的读写变成消息；锁和原子不再直接可用；时间、顺序和失败都变得不可靠。

\section{从共享内存到消息系统}

单机内存访问通常以纳秒计，网络往返通常以微秒到毫秒计。单机 cache miss 已经很贵，跨网络请求更贵。分布式系统要尽量减少同步往返，使用批处理、流水线、数据本地性和异步通信。

网络还可能丢包、重排、延迟抖动或断开。应用层必须处理超时、重试、重复请求和部分失败。一个请求超时，不代表对方没有执行；重试可能造成重复副作用，因此幂等性非常重要。

\section{RPC、延迟和 in-flight 请求}

RPC 把远程调用包装成类似本地函数调用的形式，但它不是本地函数。参数要序列化，消息要进入内核或用户态网络栈，经过网卡和交换机，到达对端后再反序列化和调度执行。一次 RPC 的固定成本远高于一次函数调用，因此分布式系统要避免细粒度频繁 RPC。

批处理、流水线和异步调用是常见优化。批处理减少每条消息固定开销，流水线隐藏往返延迟，异步调用避免线程阻塞。但它们都会增加错误处理复杂度：部分成功、乱序返回、超时重试和取消都要定义清楚。

RPC 还会隐藏背压问题。本地函数调用如果慢，调用者自然等在当前线程；远程调用如果被异步提交，调用者可能继续制造更多请求，直到队列、连接池或对端服务被压垮。因此 RPC 客户端必须有并发上限、超时、重试预算和排队策略。没有这些边界，异步 RPC 只是把阻塞变成不可控积压。

\topic{时间和顺序}
不同机器没有完美同步时钟。日志时间戳可以帮助观察，但不能单独作为一致性依据。分布式系统常用逻辑时钟、版本号、任期号和日志索引表达顺序关系。

顺序要区分因果顺序和物理时间顺序。请求 A 触发请求 B，B 因果上发生在 A 之后；但两个机器的墙钟时间可能显示 B 的时间戳更早。协议通常不依赖墙钟证明因果，而是使用请求 ID、版本号、日志索引、任期或向量时钟等逻辑信息。日志时间戳主要用于观测和排查，不应被误用成强一致证据。

\section{超时、重试和幂等}

幂等操作执行一次和执行多次效果相同。分布式系统中，客户端超时后常常不知道服务器是否执行了请求。若请求不是幂等的，重试可能造成重复扣款、重复创建或重复写入。常见做法是给请求分配唯一 ID，服务器记录已处理 ID，并在重复请求时返回同一结果。

幂等设计要落到存储结构。只在客户端生成请求 ID 不够，服务器还要把请求 ID、执行结果和业务状态放在同一个原子提交边界里。否则服务器可能已经修改业务状态，但还没记录请求 ID 就崩溃，重试后仍然重复执行。工程上常用去重表、条件写、事务日志或状态机日志解决这个问题。

\section{故障、降级和背压}

机器可能宕机，进程可能重启，网络可能分区，磁盘可能慢，消息可能重复。多数工程系统采用 crash-stop 或 crash-recovery 模型，不处理任意恶意错误。明确故障模型，是协议设计的第一步。

\topic{数据本地性}
分布式计算中，移动计算到数据附近通常比移动大量数据更划算。MapReduce、Spark、分布式数据库和对象存储都围绕数据分片、任务调度和网络 shuffle 做权衡。

数据本地性是 NUMA 原则在集群尺度上的版本。单机里远端 NUMA 访问慢，集群里跨机拉取数据更慢；单机里先在线程本地 partial 再合并，集群里先在 map 端 combine 再 shuffle；单机里热点 cache line 会拖慢所有核心，集群里热点 key 会拖慢整个 stage。前面章节的思想在这里不是类比游戏，而是同一原则换了成本单位。

\topic{CAP 的正确理解}
CAP 不是说系统只能在一致性、可用性、分区容忍性里任选两个。网络分区是分布式系统必须面对的现实；在分区发生时，系统需要在继续响应请求和保持强一致之间做选择。正常情况下，系统仍然要同时追求低延迟、高可用和足够一致。工程讨论要具体到故障场景，而不是只背缩写。

\topic{超时、重试和重试风暴}
超时不是失败证明，只是本地等待时间超过阈值。请求可能在网络上，可能在对端队列里，可能已经执行但响应丢失，也可能确实失败。重试可以提高成功率，也可能放大过载。如果一个服务已经慢了，所有客户端同时重试，会制造重试风暴，让系统更慢。

健康的重试策略通常包括指数退避、抖动、最大重试次数、总超时预算、幂等保证和熔断。重试预算尤其重要：一个上游请求如果调用十个下游，每个下游都重试三次，总请求量可能迅速膨胀。分布式系统的性能问题常常来自这些控制路径，而不是用户函数本身。

\begin{keyidea}
分布式系统的性能问题往往是正确性问题的影子。重试、复制、超时、共识和恢复都会改变吞吐和延迟。
\end{keyidea}

\topic{从共享内存到消息}
单机多线程里，线程通过共享内存通信。即使共享内存有 cache line、一致性协议和内存序成本，程序仍然可以用一个地址表达共同状态。分布式系统里，这个前提消失了。机器 A 不能直接读取机器 B 的内存；它只能发送消息，请求 B 执行某个操作或返回某份数据。共享变量变成消息，锁变成协议，原子操作变成带版本号或日志索引的提交。

这个变化非常根本。单机中一个 mutex 临界区通常在纳秒到微秒级，跨网络的一次协调可能是微秒到毫秒级，还会遇到排队、丢包、重试和对端故障。单机中线程崩溃通常意味着进程整体失败，分布式中一个节点失败后其他节点可能继续运行。单机中内存序由语言和硬件定义，分布式中消息顺序、重复和超时都要由协议处理。

因此分布式系统不能把远程调用当成本地函数。RPC 库可以隐藏序列化细节，但不能隐藏失败语义。调用者必须问：请求是否到达，对端是否执行，响应是否丢失，重试是否安全，重复请求如何处理，超时后资源如何回收。若这些问题没有答案，系统在正常路径看起来能跑，在故障路径会变得不可控。

\topic{消息的生命周期}
一条消息从客户端到服务器，至少经过序列化、排队、发送、网络传输、接收、反序列化、调度、执行、生成响应、响应传输和客户端处理。每一步都可能排队，每一步都可能失败。性能报告如果只写“RPC 延迟 3 ms”，还不够；要拆出客户端排队、网络 RTT、服务端队列、执行时间和响应排队。

消息还需要身份。没有 request id，客户端无法把响应和请求可靠对应；没有 attempt id，服务端无法区分第一次请求和重试；没有 trace id，观测系统无法串起跨节点路径；没有 partition id，shuffle 系统无法知道数据应写到哪里。系统工程中，消息头不是形式，它承载正确性和观测性。

\begin{lstlisting}[language=C++,caption={本机模拟用的消息元数据}]
struct MessageHeader {
 std::uint64_t request_id = 0;
 std::uint32_t source_worker = 0;
 std::uint32_t target_partition = 0;
 std::uint32_t attempt = 0;
};

struct ShuffleMessage {
 MessageHeader header;
 std::vector<std::int32_t> keys;
 std::vector<std::int64_t> values;
};
\end{lstlisting}

这段代码不是网络协议实现，但它展示了分布式消息需要的基本边界。request id 用于幂等和去重，source worker 用于观测和调试，target partition 用于路由，attempt 用于重试分析。真实系统还会有校验、版本、压缩格式、认证信息和时间戳。

\topic{延迟、吞吐和 in-flight 请求}
分布式系统里，吞吐往往依赖足够的 in-flight 请求。一个请求的往返延迟可能很高，如果客户端每次只发一个请求，吞吐会被 RTT 限制。流水线允许多个请求同时在网络中飞行，提高吞吐。但 in-flight 太多会导致队列积压、内存占用上升、对端过载和尾延迟恶化。

这和 CPU 的内存级并行度很像：核心可以同时挂起多个 cache miss 来隐藏延迟，但 miss 太多会占满 load buffer；客户端可以同时发多个 RPC 来隐藏 RTT，但请求太多会占满队列和连接。系统要有并发窗口，窗口大小要由延迟、吞吐、错误率和对端能力共同决定。

一个简单模型是 Little's Law：在稳定系统中，平均并发数约等于吞吐乘以平均延迟。若希望每秒处理 10000 个请求，平均延迟 10 ms，则系统中大约有 100 个请求在飞行。这个模型不能替代测量，但能帮助你判断一个并发上限是否荒谬。

\topic{超时不是取消}
客户端超时只表示客户端不再等待，不表示服务器停止执行。服务器可能已经完成业务状态修改，只是响应丢失；也可能还在队列中等待；也可能稍后才执行。若客户端超时后立即重试，而操作不是幂等的，就会重复副作用。若客户端释放请求上下文，但底层完成回调稍后到达，就可能访问已释放对象。

因此超时要配合取消和幂等。取消是尽力而为，不一定成功；幂等保证重复执行的外部效果可控；请求上下文保证完成路径无论成功、失败、超时还是取消，都能安全回收。异步 I/O 章节里的生命周期问题，在分布式 RPC 中更严重，因为远端是否执行不是本地进程能完全控制的。

\topic{幂等提交点}
幂等性不是“重试时先查一下”这么简单。真正的幂等需要把请求 ID、业务状态修改和响应结果放在同一个提交边界里。考虑转账、创建订单或写检查点：如果服务器先修改业务状态，随后记录 request id 前崩溃，重试时看不到旧 request id，就可能重复执行；如果先记录 request id，但业务状态没修改就崩溃，重试时又可能误以为已经成功。

一种常见设计是把请求处理当作状态机事务：检查 request id 是否已处理；若已处理，返回保存的结果；若未处理，执行业务修改，同时写入 request id 和结果；提交后再返回。这个过程可以用数据库事务、共识日志或本地 WAL 实现。关键是提交点必须清楚。

\topic{消息顺序和重复}
网络可能重排消息。客户端发送 A 再发送 B，不代表服务器一定先处理 A。即使 TCP 在单连接内保持字节顺序，应用层多连接、多队列、多线程处理仍可能改变完成顺序。协议如果依赖顺序，就必须显式携带序号、版本或日志索引。

消息还可能重复。客户端重试会重复发送，网络代理可能重放，服务端超时后客户端又发下一次 attempt。服务端必须决定重复消息如何处理：直接丢弃、返回旧结果、合并、覆盖还是报错。没有 request id 和幂等表，重复消息通常无法安全处理。

\topic{故障检测的模糊性}
分布式系统无法可靠地区分“节点死了”和“节点很慢”。心跳超时可能因为节点崩溃、网络拥塞、GC 暂停、磁盘卡顿、CPU 被抢占或调度延迟。把慢节点误判为死节点可能导致重复调度和资源浪费；把死节点长期当成活节点会拖慢恢复。故障检测本质上是基于超时的怀疑，不是数学证明。

工程上通常使用多次心跳、租约、任期、健康检查、慢节点隔离和人工可观测指标。对计算任务而言，可以把慢任务 speculative retry 到其他节点，但要处理重复输出；对存储副本而言，不能轻易让旧 leader 继续写，需要任期和共识保证。故障检测和一致性协议紧密相连。

\topic{部分失败和降级}
单机程序常常整体成功或整体失败，分布式系统经常部分失败。一个查询请求可能需要访问十个分片，其中九个成功、一个超时。系统要定义结果：失败整个请求，返回部分结果，使用缓存旧值，还是降级为近似结果。这个选择取决于业务语义。

搜索结果可以返回部分分片并标记结果不完整；计费系统不能忽略一个失败分片；监控系统可以延迟补齐；推荐系统可以使用缓存。分布式教材不能只讲协议，还要让读者看到业务语义如何决定失败处理。

\topic{背压和重试预算}
分布式背压要跨节点传播。服务端队列满时，应让客户端降低发送速率、拒绝低优先级请求或返回明确错误。若服务端已经过载，客户端仍按固定策略重试，只会放大故障。健康的客户端要有最大并发、总超时、重试次数、指数退避、抖动和熔断。

重试预算可以按上游请求计算。例如一个用户请求最多允许产生 20 次下游调用尝试，超过就停止重试并返回错误。这样可以防止多层服务各自重试导致请求量指数膨胀。预算还应被观测：报告中要记录原始请求数、重试请求数、超时数、最终失败数和重试原因。

\topic{数据本地性和调度}
分布式计算里，调度任务时要考虑数据位置。若输入分片在节点 A，本地执行可以避免网络读取；若节点 A 忙或失败，可以远程读取或复制数据，但要付出网络成本。数据本地性不是绝对规则，而是调度决策的一项成本。

这和 NUMA 类似。单机里，线程靠近内存节点更快；集群里，任务靠近数据分片更快。若本地节点排队很长，远程执行可能更快；若远程网络拥塞，本地等待更好。调度器需要把数据位置、队列长度、节点健康、资源类型和任务优先级一起考虑。

\topic{本机多进程模拟}
本册不要求读者立刻搭建真实集群。可以先用本机多进程或多线程模拟分布式系统：每个 worker 有自己的输入分片和消息队列，driver 负责分配任务，网络用有界队列和随机延迟模拟，故障用随机丢弃、超时和重复消息模拟。这样可以在一台机器上训练协议思维。

模拟不能证明真实网络性能，但能验证语义：请求 ID 是否去重，重试是否幂等，队列是否背压，检查点是否能恢复，重复输出是否被提交协议挡住。先在本机把语义做清楚，再迁移到真实网络环境，风险更低。

\topic{RPC 不是函数调用}
远程调用看起来像函数调用，但语义完全不同。本地函数调用要么进入函数体，要么在调用前就失败；调用栈、对象生命周期和异常传播都在同一进程内。RPC 经过序列化、发送队列、内核网络栈、网卡、交换机、对端内核、服务端队列、业务线程、响应路径。任何一段都可能排队、失败、重复、超时或部分完成。把 RPC 当成本地函数，是分布式系统设计里最常见的错误之一。

RPC 的调用者必须接受一个事实：超时后不知道远端到底发生了什么。远端可能没有收到请求，可能收到但还没执行，可能已经执行并提交，可能响应丢失，可能响应正在返回。这个不确定性决定了幂等、请求 ID、重试预算和提交点不是附加功能，而是 RPC 的基本语义。只要系统允许超时重试，就必须回答重复请求如何处理。

服务端也不能把 RPC 当成本地函数。请求进入服务端后，首先进入接收队列，再由某个 worker 处理。若业务处理很慢，接收队列会增长；若队列无限，内存会增长；若队列有限，客户端会遇到拒绝或超时。服务端处理完后，响应也可能在发送队列中等待。一次 RPC 的延迟不是业务函数耗时，而是排队、执行、序列化和网络共同组成的端到端路径。

\topic{消息边界和协议版本}
分布式系统中，消息必须有明确边界。TCP 是字节流，不保留应用层消息边界；一次 \code{write} 不等于对端一次 \code{read}。应用协议需要 framing，例如固定长度头部加变长 body，或者长度前缀，或者分隔符。没有 framing，服务端无法可靠知道一条消息在哪里结束，下一条消息在哪里开始。

消息还需要版本。系统升级时，客户端和服务端可能不是同一版本。若消息结构只按当前代码隐式解析，一次滚动升级就可能造成字段错读。稳健协议会在 header 中放 protocol version、message type、flags、payload length 和 checksum。新增字段要有默认值，删除字段要经历兼容期。分布式系统的接口不是函数签名，而是长期存在的协议合同。

\begin{lstlisting}[language=C++,caption={消息头部的语义字段}]
struct WireHeader {
 std::uint16_t protocol_version = 1;
 std::uint16_t message_type = 0;
 std::uint32_t flags = 0;
 std::uint64_t request_id = 0;
 std::uint32_t payload_bytes = 0;
 std::uint32_t checksum = 0;
};
\end{lstlisting}

这段代码只是表达字段，不是跨平台二进制协议的完整实现。真实协议还要定义字节序、对齐、字段编码、字符串格式和兼容规则。不要直接把 C++ 结构体内存写到网络里，因为 padding、大小端、编译器布局和 ABI 都可能不同。协议格式要按字节定义，而不是按当前进程的内存布局定义。

\topic{序列化的成本和边界}
序列化不是简单的“把对象转成字节”。它会消耗 CPU、分配内存、访问字段、复制数据、计算校验，可能还要压缩。对象布局如果适合本地计算，不一定适合网络传输；网络格式如果紧凑，也不一定适合 CPU 直接计算。高性能系统常在内部布局和 wire format 之间做显式转换。

零拷贝也不是免费。要真正避免拷贝，需要缓冲区生命周期跨越发送完成，不能在发送返回后立即复用；需要处理分片、对齐和 scatter-gather；还要保证应用不能在内核或网卡尚未发送完成时修改内容。许多系统先用清晰的复制式协议做正确，再对热点路径引入缓冲池和批量发送。不要在协议语义没清楚时追求零拷贝。

序列化还影响故障恢复。检查点文件、shuffle manifest、任务提交表都需要稳定格式。若格式依赖当前类定义，代码升级后旧检查点可能无法读取。一个工程化系统应为持久化数据定义版本和迁移策略。分布式计算不是只在网络上传消息，还要把状态写到磁盘，并在失败后读回来。

\topic{逻辑时间和因果关系}
分布式系统不能依赖全局精确时钟来判断所有事件顺序。机器之间时钟会漂移，NTP 会校正，进程会暂停，网络延迟不稳定。物理时间适合做超时、租约和观测，但协议正确性常需要逻辑时间。最简单的逻辑时间是单调递增版本号、epoch、term 或日志索引。它们不表示真实时间，而表示协议中的先后和所有权。

例如 driver 每次发布新的路由表版本，就把 epoch 加一。Worker 收到旧 epoch 的任务，可以拒绝并要求刷新。Raft 的 term 也是类似思想：旧任期的 leader 即使还活着，也不能继续以旧身份提交新写。逻辑时间让系统能区分“我看到的状态”和“当前合法状态”。

向量时钟能表达更细的因果关系，但成本更高。很多系统不需要完整向量时钟，只需要针对某个资源的版本号。教材阶段要让读者理解：当两个事件分布在不同机器上时，肉眼时间先后不一定可靠；协议需要自己的顺序标记。

\topic{请求 ID 的设计}
请求 ID 不是随机数装饰，而是去重、追踪、幂等和调试的核心。一个请求 ID 至少应在一次逻辑操作内稳定，重试 attempt 不应改变逻辑 request id。否则服务端无法知道两次到达的请求其实是同一个操作的重复。Attempt id 可以单独增加，用于观测和超时分析。

请求 ID 的生成要考虑冲突范围。单机自增 ID 在单个进程内简单，但跨进程和重启后需要节点 ID、时间段或持久化计数。随机 ID 冲突概率低，但调试时不如结构化 ID 直观。雪花类 ID 把时间、节点和序列号组合起来，但依赖时钟行为。教学项目可以使用 driver 分配的 \code{std::uint64_t} 递增 ID，因为它在本机模拟里足够清楚；真实系统要写明冲突域和重启策略。

\begin{lstlisting}[language=C++,caption={逻辑请求和 attempt 分离}]
struct RequestKey {
 std::uint64_t request_id = 0;
 std::uint32_t attempt = 0;
};
\end{lstlisting}

服务端去重表通常按 \code{request_id} 存储逻辑结果，而不是按 attempt 存储。新 attempt 到达时，如果 \code{request_id} 已提交，就返回旧结果或忽略重复副作用。Attempt 只用于记录“这是第几次尝试”，不改变业务语义。

\topic{幂等表的容量和回收}
幂等表不能无限增长。服务端记录所有 request id，内存迟早耗尽。系统必须定义 request id 的有效期、客户端重试窗口和服务端保留时间。若客户端可能在一天后重试，服务端至少要保留一天；若服务端只保留一分钟，客户端一分钟后重试就可能重复执行。幂等语义是客户端和服务端共同的合同。

回收幂等表时要小心。按时间删除最简单，但依赖时钟和请求延迟分布。按提交日志截断更稳：如果所有客户端都保证不会重试某个 checkpoint 之前的请求，服务端可以删除旧记录。对于批处理任务，幂等表可以和 task attempt manifest 绑定：一个 stage 完成并 checkpoint 后，旧 attempt 去重记录可以随 checkpoint 压缩。

幂等表里保存什么也要权衡。只保存 request id 可以防止重复执行，但重复请求无法返回原响应；保存 request id 和响应结果，可以让客户端收到同样结果，但占用更多空间；保存结果摘要可以减少空间，但可能不能满足业务。设计时要从调用者语义出发。

\topic{错误分类}
分布式错误不能只返回“失败”。至少要区分可重试错误、不可重试错误、过载错误、版本错误、权限错误和未知错误。可重试错误包括临时网络失败、服务端 busy、leader 切换中。不可重试错误包括请求格式错误、权限不足、资源不存在。版本错误说明客户端路由表或协议版本过旧，需要刷新而不是盲目重试。

错误分类直接影响重试策略。对不可重试错误反复重试，只会浪费资源；对 busy 错误立即高频重试，会加剧过载；对版本错误不刷新路由，会一直打到错误节点。返回码是背压和控制面的接口，不只是调试信息。

\begin{lstlisting}[language=C++,caption={错误分类的教学形态}]
enum class RpcStatus : std::int32_t {
 Ok,
 Retryable,
 Busy,
 StaleRoute,
 InvalidRequest,
 PermissionDenied,
 Unknown,
};
\end{lstlisting}

真实协议还会有更细错误码和错误详情，但分类必须稳定。客户端重试库通常只看分类，日志和排查再看详细原因。把所有错误都归为 Unknown，会让客户端无法做正确行为。

\topic{超时预算的分解}
一个用户请求的总超时预算应被分解到各个下游调用，而不是每一层都重新给自己一个完整超时。假设用户请求总预算 1000 ms，服务 A 调服务 B，B 调 C。如果每层都设置 1000 ms，最坏情况下总延迟会远超用户预算。更好的做法是把剩余 deadline 随请求传递，下游根据剩余时间决定是否执行、降级或立即失败。

Deadline 比 timeout 更适合跨层传播。Timeout 是相对时间，每一层都要重新计算；deadline 是绝对截止点，传递后各层可以看当前剩余时间。时钟漂移会影响跨机器 deadline，因此通常还要保守处理，但它仍比无预算重试更清楚。批处理系统也有类似概念：一个 stage 的时间预算应来自整作业 SLA，而不是每个 task 自己无限重试。

\topic{重试、退避和抖动}
重试要慢下来，而不是立刻把同一个请求再打一遍。指数退避让连续失败后的等待时间增长，抖动让大量客户端不要在同一时刻重试。没有抖动时，所有客户端可能在服务恢复的一瞬间同时重试，造成第二波过载。重试预算限制一次逻辑请求最多产生多少 attempt，避免多层服务重试相乘。

重试还要和幂等绑定。读请求通常更容易重试，写请求必须有 request id 和提交语义。若写请求不是幂等的，客户端在超时后可能只能查询状态，而不是直接重试。教材要让读者看到：重试是正确性功能，不是简单可靠性增强。

\topic{慢节点和尾延迟}
分布式系统的整体延迟常被最慢分片决定。一个查询要访问十个分片，九个分片很快，一个分片慢，整体就慢。慢节点可能没有完全失败，只是磁盘抖动、CPU 被抢占、GC、网络拥塞或热点 key。故障检测如果只判断死活，就看不见慢节点。

处理慢节点的方法包括 speculative execution、请求 hedging、负载隔离、热点迁移和降级。Speculation 会启动重复 attempt，最先完成者获胜；hedging 会在等待一段时间后给另一个副本发同样读请求。这些方法能降低尾延迟，但增加资源消耗和重复副作用风险。没有幂等提交和重试预算，speculation 会让系统更危险。

\topic{队列和排队论直觉}
分布式系统里很多延迟来自排队。服务时间接近满载时，队列等待会快速上升。即使平均处理能力略高于平均到达率，突发流量也会造成队列。背压和限流的目的，就是在队列失控前让上游看到压力。

Little's Law 给出一个粗略关系：系统内平均在途请求数等于吞吐乘以平均延迟。若 in-flight 请求数持续增加而吞吐没有增加，说明延迟正在上升。若队列长度上升但 CPU 很空，可能是 I/O 或锁等待；若 CPU 满且队列上升，可能是计算能力不足；若网络出站队列上升，可能是下游慢或带宽瓶颈。队列是分布式系统最重要的压力表。

\topic{消息总线的本机实现}
本机模拟可以用一个 MessageBus 对象表示网络。它不需要真实 socket，先用多个有界队列表达节点之间的通道。每条消息带 source、target、request id、attempt、payload bytes、deadline 和 status。MessageBus 可以注入延迟、丢弃、重复和乱序。这样一台手机或一台开发机就能训练分布式协议。

\begin{lstlisting}[language=C++,caption={本机消息的核心字段}]
struct LocalMessage {
 std::uint32_t source_worker = 0;
 std::uint32_t target_worker = 0;
 std::uint64_t request_id = 0;
 std::uint32_t attempt = 0;
 std::uint32_t payload_bytes = 0;
 RpcStatus status = RpcStatus::Ok;
};
\end{lstlisting}

模拟器要把故障做成可重复。使用固定 seed，记录每次丢包、重复、延迟和重排事件。测试失败后，可以用同一个 seed 重放。没有可重复性，分布式 bug 会非常难查。

\topic{实验报告的故障矩阵}
分布式实验报告不能只写正常路径。至少要列出故障矩阵：请求丢失、响应丢失、服务端忙、服务端执行后崩溃、客户端超时后重试、重复消息到达、旧路由请求、慢节点、队列满、driver 重启。每一项都要写预期行为：重试、去重、返回旧结果、刷新路由、背压、失败还是恢复。

这张矩阵会迫使设计暴露缺口。例如如果响应丢失后客户端重试，而服务端没有保存响应结果，客户端可能无法知道第一次是否成功；如果执行后崩溃但提交点没持久化，恢复后可能重复执行；如果旧路由请求没有版本检查，迁移期间可能写到旧分片。故障矩阵不是测试附录，而是分布式设计本身。

\topic{连接管理}
真实 RPC 系统还要管理连接。每个请求新建 TCP 连接成本高，通常会复用连接池；连接池又要处理最大连接数、空闲超时、健康检查、连接重置和负载均衡。一个连接上的请求可能按顺序发送，也可能多路复用。若一个连接出现队头阻塞，后续请求会被拖慢；若连接太多，内核和远端都会有压力。

连接池的背压语义也要定义。连接都忙时，新请求是等待、失败、还是新建连接？等待是否消耗用户 deadline？连接错误时，正在飞行的请求如何标记？这些问题和线程池、有界队列非常相似。连接池不是网络库细节，而是分布式系统的执行资源管理。

\topic{服务发现和负载均衡}
客户端需要知道请求发给哪些服务实例。服务发现可以来自静态配置、DNS、注册中心或控制面。发现结果通常会缓存，因此要处理实例上下线、健康状态和版本变化。负载均衡策略可以是轮询、随机、最少连接、按延迟加权、按 locality 选择。策略不同，故障时行为也不同。

负载均衡不能只看请求数量。一个实例可能处理轻请求很多，另一个处理重请求少；一个实例延迟高可能是过载，也可能是网络远；一个实例刚启动缓存冷，直接大量流量打过去会抖动。工程系统常使用慢启动、健康检查、熔断和 outlier detection。教材项目可以简化为 driver 分配 worker，但应让读者知道真实服务发现是控制面的一部分。

\topic{熔断和隔离}
熔断器在下游持续失败或延迟过高时，暂时停止向它发送请求，快速失败或降级。这样可以避免线程、连接和队列被坏下游拖住，也给下游恢复时间。熔断不是为了隐藏错误，而是保护系统。熔断状态通常包括 closed、open、half-open：正常发送、快速失败、试探恢复。

隔离则是把不同下游或不同优先级请求放进不同资源池。若所有下游共享一个线程池，某个慢下游可能耗尽所有 worker，影响无关请求。隔离可以用独立队列、独立连接池、独立线程池或配额。代价是资源利用率可能下降，但故障影响范围更可控。

\topic{幂等性和业务键}
请求 ID 可以由系统生成，也可以来自业务键。创建订单时，客户端可以提供 client request id；写日志时，可以使用文件分片和 offset；提交 map output 时，可以使用 job id、stage id、task id。业务键更容易跨重启恢复，因为它和逻辑操作绑定。系统生成的随机 ID 如果只在内存里保存，重启后可能丢失去重语义。

幂等键还要定义作用域。一个 request id 在单个用户内唯一，还是全系统唯一？一个 task id 在一个 job 内唯一，还是跨 job 唯一？作用域不清会导致误去重或漏去重。教学项目应使用结构化 key，例如 job id、stage id、task id、attempt。这样日志和 manifest 更容易读。

\topic{时钟和单调时间}
超时测量应使用单调时钟，而不是 wall clock。系统时间可能被 NTP 调整，向前或向后跳；单调时钟适合计算 elapsed time。分布式系统传 deadline 时，跨机器 wall clock 差异仍要考虑，但本机队列和 I/O 超时应使用单调时间。日志时间戳和超时计时是不同用途。

租约和跨机器 deadline 更复杂。若依赖物理时间做安全判断，就要给时钟漂移和暂停留余量。很多协议避免直接依赖绝对时间来保证一致性，而使用 term、epoch 和 quorum。时间可以优化性能，不应轻易成为唯一正确性依据。

\topic{有序、因果和重放}
消息重放是测试分布式系统的重要方法。记录一段消息历史，包括发送、延迟、丢弃、重复和处理结果，然后在本机重放。若系统确定性足够，重放能复现 bug。为了支持重放，消息处理应尽量由输入消息和当前状态决定，避免隐藏随机数和真实时间直接影响核心语义。随机故障应使用记录的 seed。

因果关系也可以在 trace 中表达。一个 retry attempt 由哪次 timeout 触发，一个 reduce commit 由哪些 shuffle block 组成，一个 checkpoint 覆盖到哪个 stage，这些都应有 ID 关联。没有因果关联的日志，只是一堆时间戳，出了问题很难追。

\topic{分布式观测性}
分布式观测性至少包含 metrics、logs 和 traces。Metrics 聚合数量和延迟，适合看趋势；logs 记录离散事件和错误，适合排查细节；traces 串起一次请求或一个 task attempt 的路径，适合分析长尾。三者需要共同的 ID：request id、job id、stage id、task id、attempt、partition。没有 ID，无法把 driver、worker、shuffle 和 reduce 的事件连起来。

指标也要按维度分解。只看全局平均延迟，会掩盖某个 partition 长尾；只看总重试数，无法知道哪个 worker 或哪类错误导致；只看总吞吐，无法发现队列越来越长。分布式系统的指标设计要服务故障定位，而不是只服务漂亮仪表盘。

\topic{本章新增实验}
本章可以增加三个小实验。第一，连接池模拟：限制最大 in-flight 连接数，观察过载时等待和失败。第二，熔断模拟：让一个 worker 持续超时，客户端在多次失败后暂时停止发送，观察重试量是否下降。第三，trace 关联：为每条消息记录 request id 和 parent event id，重放一次响应丢失导致的重试。

这些实验不需要真实网络，但能训练分布式语义。它们也会给第 18 章的 driver 和 message bus 提供基础。

\topic{多租户和公平性}
分布式计算系统常同时运行多个 job。若所有 job 共用 worker 和 shuffle 队列，大 job 可能占满资源，小 job 长时间等待。公平调度需要定义资源份额、优先级和抢占策略。最简单的策略是每个 job 有独立队列，driver 按轮询或权重取任务；更复杂策略会考虑输入大小、deadline、用户配额和历史用量。

公平性和吞吐有时冲突。让大 job 连续占用资源，吞吐可能最高；穿插小 job，平均响应更好但缓存和调度成本增加。教学项目可以先支持单 job，但设计字段时保留 job id，就是为了未来多租户和公平调度。没有 job id，所有指标和去重都会混在一起。

\topic{背压和公平的交汇}
多个上游共享一个下游时，背压还要公平。若 reduce partition 队列满，是否阻塞所有 map worker，还是只阻塞发送该 partition 的 worker？若某个 job 产生大量热点 key，是否影响其他 job 的正常 partition？系统可以为每个 job 或 partition 设置独立队列和配额，防止一个热点拖垮全局。

这和单机线程池里的优先级队列类似。资源隔离降低故障传播，但可能降低资源利用率。工程设计要写清目标：批处理吞吐、交互延迟、公平性还是隔离。不同目标会得到不同调度器。

\topic{数据本地性和复制选择}
若输入分片有多个副本，调度器可以选择负载较低且距离较近的副本。只看本地性可能把任务都打到一个繁忙节点，只看负载可能产生大量远程读取。调度器需要一个成本函数，把数据位置、队列长度、节点健康、网络成本和优先级结合起来。教学项目可以用简单规则：优先本地，若本地队列超过阈值则远程。

这个规则和 NUMA 调度完全相似：优先本地内存，过载时跨节点。第二册反复建立这种跨尺度类比，是为了让读者面对新系统时能迁移思维。

\topic{协议演进和兼容测试}
分布式协议一旦持久化或跨进程，就需要兼容测试。新增字段要有默认值，旧节点收到新字段能忽略，新节点收到旧消息能补默认值。删除字段要经历过渡期。消息版本、feature flags 和最小兼容版本都要有策略。否则滚动升级时，集群可能出现部分新旧节点互相不能通信。

教学项目可以简单要求“同一版本运行”，但如果写成教材，就要说明这是简化。第三册 AI 推理引擎如果加入模型格式和算子版本，也会遇到同样问题。协议演进是长期系统的基本能力。

\topic{混沌测试和演练}
故障注入可以从单元测试扩展到混沌测试：在长时间运行中随机让 worker 变慢、丢消息、重启 driver、损坏临时 block、让队列变满。混沌测试的目标不是制造混乱，而是验证系统在预期故障集合内保持可控。每次注入都应记录事件，失败后能回放。

混沌测试必须有安全边界。不要在没有隔离的真实数据上随意注入；不要注入系统无法承受的未知破坏；不要只看系统没崩就算通过，还要检查结果正确、重试预算、队列长度和恢复时间。可靠性测试同样需要 oracle。

\topic{分布式模型的知识地图}
到本章结束，读者应能把分布式系统拆成：消息、时间、故障、身份、路由、幂等、背压、调度和观测。消息回答数据如何传；时间回答何时放弃；故障回答什么可能坏；身份回答这是哪个请求；路由回答发给谁；幂等回答重复怎么办；背压回答过载怎么办；调度回答谁执行；观测回答如何知道发生了什么。后两章会把这些部件组合成分片复制和计算引擎。

\topic{分布式模型章节总结}
分布式系统的困难不是网络 API，而是不确定性。消息可能丢、重复、乱序；节点可能慢、停、恢复；客户端超时不代表服务端停止；重试可能放大故障；旧路由和旧 leader 可能继续发送请求。解决这些问题需要请求身份、版本、deadline、幂等表、背压、故障矩阵和 trace。把这些基础模型建好，后面的复制和计算工程才有可靠语义。

\topic{分布式模型决策表}
设计一个 RPC 或消息协议时，先填决策表。请求是否幂等？若不是，超时后不能盲目重试。是否有 request id？若没有，服务端无法去重。是否有 deadline？若没有，多层调用会放大延迟。是否区分 busy 和 fatal？若没有，客户端无法正确退避。是否有路由版本？若没有，迁移期间旧请求难以拒绝。是否有 trace id？若没有，故障后无法串联路径。

这张表适用于任何分布式调用，也适用于本机异步消息。只要执行和完成分离，就需要身份、时间、状态和错误语义。

\topic{消息协议审查表}
每个消息类型都应审查：谁发送，谁接收，是否允许重复，是否有响应，超时后发送方做什么，接收方是否幂等，payload 是否有版本和长度，错误码是否分类，是否进入 trace，是否影响提交状态。审查后会发现很多消息只是指标，可以丢；有些消息是提交，必须持久化；有些消息是控制面，必须带 epoch。消息不是字节包，而是系统状态转换。

分布式模型章的回归阅读应从本地调用栈失效开始。调用本地函数时，返回值和异常沿栈同步传播；发送远端消息后，请求可能排队、丢失、重复、乱序、执行成功但响应丢失，调用者无法仅凭超时知道远端发生了什么。因此 request id、attempt id、deadline、重试预算和幂等表不是附加字段，而是远端调用的最低语义。

响应丢失案例要完整走一遍。driver 发送 attempt A，worker 执行并写出结果，响应在返回途中丢失；driver 超时后发送 attempt B；稍后 A 的迟到响应到达。系统必须决定谁能提交，旧响应如何拒绝，重复输出如何清理，最终 manifest 是否只引用一次结果。若这里解释不清，就不能进入复制和共识。

本章还要防止重试风暴。客户端重试必须有指数退避、抖动、总预算和并发窗口；服务端过载时要把背压传回上游；监控要区分原始请求、重试请求、超时和最终失败。分布式性能经常是正确性控制路径的后果，不能把 RPC 当作稍慢的函数调用。

消息状态表是本章的核心工具。每个 request 至少经历 Created、Sent、InFlight、TimedOut、Retried、Completed、Committed、Rejected 或 Cancelled 中的一部分状态。attempt 是物理尝试，request 是逻辑请求，二者不能混用。响应回来时，driver 先按 request 找逻辑状态，再按 attempt 判断是否仍然有效，最后才决定是否提交。没有这张表，超时和迟到响应只能靠 if 语句临时处理。

本机模拟也要承认边界。线程队列可以模拟丢消息、重复消息、延迟和乱序，但不能完全模拟真实网络拥塞、内核协议栈、跨机时钟和进程崩溃。模拟的价值是让语义先可测：身份、超时、重试、幂等和背压。等语义稳定后，再迁移到进程间或真实网络，性能数字必须重新测，协议边界则可以复用。

超时配置要从延迟分布推导，而不是随便写一个数字。超时太短会制造无意义重试，超时太长会拖慢恢复；不同阶段的超时也不同，map task、shuffle fetch、manifest commit 和 heartbeat 不能共用同一个值。报告应记录正常延迟、尾延迟、重试次数和最终失败数，再调整 deadline。分布式系统里，时间参数本身就是协议的一部分。

幂等表要和业务状态同提交。若先写输出文件，再记录 request id，中途崩溃会导致重试重复写；若先记录 request id，再写输出失败，又可能误认为已完成。更稳的做法是让 attempt 写临时输出，commit 阶段原子更新 manifest 和去重记录。这样幂等性不是靠记忆，而是靠提交边界。

消息格式要有版本。第一版消息可能只有 request、attempt 和 payload，第二版加入 deadline，第三版加入 epoch 和 trace。如果没有 version，旧 worker 和新 driver 混跑时就无法判断字段含义。版本不是为了兼容所有未来，而是为了让系统在升级和回滚时能拒绝不认识的消息，或者按旧语义解析。教材项目可以用简单整数版本和必填字段检查，先建立意识。

Deadline 还要和取消传播连接。上游请求超时后，下游 task 可能仍在运行；如果它最终完成并提交，就会产生用户已经放弃但系统仍改变状态的问题。稳妥设计是 deadline 进入 MessageEnvelope，worker 在开始前和关键检查点判断是否过期，driver 对过期 attempt 的响应拒绝提交。取消不保证远端立刻停止，但提交点必须尊重逻辑时限。

分布式章还应讲观察字段。每条消息记录 send time、receive time、start execute、finish execute、response send 和 response receive，才能区分网络慢、对端排队慢、执行慢还是响应丢失。只记录一个“RPC latency”会把所有成本混在一起。后面的最终项目 runbook 也应要求保留这些字段，方便读者做故障演练。

\section{把本机模拟写成真正的分布式语义}

本机 MessageBus 很容易被写成函数调用包装：driver 调一个方法，worker 立即执行，响应直接返回。这样虽然代码跑在一个进程里，却没有训练分布式思维。正确的教学模拟应强制隔离：driver 和 worker 不共享业务对象，交互只能通过消息；消息可延迟、重复、乱序、丢失；worker 输出必须先写临时位置，再由 driver commit；driver 重启后只能依靠持久状态恢复。

这套限制不是为了复杂，而是为了显式补回本地调用栈失去的保证。本地调用里，返回值天然对应本次调用；远端消息里，响应只是带着 request id 的字节包。本地异常能沿栈传播；远端错误必须编码成状态。本地对象生命周期由进程统一管理；远端输出生命周期要靠 attempt、manifest 和清理协议。模拟如果不体现这些差异，读者学到的只是多线程队列，不是分布式模型。

\topic{消息身份和状态表是控制面}

消息字段要回答“这是谁的一次尝试”。Job、stage、task、attempt、request 的作用域不同。Task 是逻辑工作单元，attempt 是物理执行尝试，request 是一次消息交互。响应丢失后重试会产生新 request 和新 attempt，但逻辑 task 不变；提交表应按逻辑 task 去重，而不是按 request 去重。若这个边界混乱，旧响应覆盖新状态、重复 attempt 重复提交都会发生。

Driver 状态表是远端调用栈的替代物。它记录当前 attempt、目标 worker、deadline、状态、重试次数和提交结果。响应处理必须先查状态，再验证身份，再检查输出，最后提交。已经 Committed 的 task 收到迟到响应，只能记录 late reply 和清理临时文件，不能改变结果。

状态表也服务观测性。每个 task 的发送时间、完成时间、超时次数、late reply、stale reply、输出字节和 checksum 都应记录。没有这些指标，分布式作业慢时只能看到“某些任务慢”，无法区分排队、执行、网络、重试和提交。

\topic{超时不是远端失败证明}

Timeout 只说明本地等待预算耗尽，不说明远端没有执行。请求可能没到，worker 可能正在排队，worker 可能执行完但响应丢失，响应可能迟到。每一种情况的处理不同。若超时后直接认为任务失败并允许新 attempt 提交，同时旧 attempt 仍能提交，就会重复计数或覆盖结果。

Deadline 比层层 timeout 更适合复杂路径。Driver 给一次 task attempt 剩余预算，MessageBus 和 worker 都按同一 deadline 判断是否继续。若每层重新设置 timeout，整体请求可能无限拖长。Retry budget 也要全局控制，避免重试风暴。队列满、worker busy、网络抖动时，盲目重试会放大压力。

幂等提交保护外部效果。Worker 写临时输出可以重复，driver commit 只能接受一个 attempt。Commit table 记录 task id、accepted attempt、checksum、manifest entry。重复响应、迟到响应和旧 attempt 都不能改变已提交结果。Exactly once 不是“执行只发生一次”，而是“外部可见提交只生效一次”。这句话必须在分布式章讲清楚。

\topic{背压和流控防止重试风暴}

分布式系统里的队列也要有界。Driver 向 worker 发送任务，要限制 per-worker in-flight；worker 接收队列满时，应返回 busy 或让 driver 等待；shuffle block 拉取也要控制并发。没有流控，慢 worker 会积压请求，driver 超时后又重试到更多 worker，最终把系统拖垮。

流控要区分任务数和字节数。一个大 split 和一个小 split 都是一项任务，但内存、I/O 和网络成本不同。Window 可以同时按 in-flight task 数、in-flight bytes 和 deadline 控制。响应包含 output bytes 和处理耗时，driver 根据反馈调整分配。这个机制和本机 I/O 背压、线程池队列容量同源。

Trace id 串联三端日志。Driver 发请求、MessageBus 投递、worker 接收、worker 写输出、driver commit，都要携带同一组 id。否则响应丢失和乱序时无法复盘。分布式系统的可调试性来自控制面字段的一致使用，而不是事后拼文本。

\topic{故障注入和恢复演练}

本章实验必须包含故障注入：丢请求、丢响应、重复请求、乱序响应、worker 崩溃、driver 重启、队列满、commit 失败。每次只开一种故障，先看 reference 是否保持，再组合故障。不要一开始就做复杂混沌测试，否则无法归因。

恢复演练要从 checkpoint 开始。Driver 重启后读取 task table 和 commit table，重新发送未完成 task，拒绝旧 epoch 响应，清理未提交临时文件。若所有状态只在内存里，重启演练必然失败。教学项目可以用本地文件保存简化状态，但必须有版本、checksum 和崩溃点测试。

本章和真实系统仍有距离：没有真实网络拥塞、没有多进程隔离、没有节点时钟漂移、没有滚动升级。但它已经覆盖分布式模型的核心：身份、状态、超时、重试、幂等、背压和恢复。把这些在本机讲透，比一开始列 CAP 和 Paxos 更适合第二册。

\topic{案例走查：响应丢失后的重复提交}

Worker 收到 request 8001，完成 task 17 attempt 0，写出临时 block，并发送 reply。MessageBus 丢弃 reply。Driver 超时，启动 attempt 1，worker 又写出一个 block，reply 8002 到达并提交。稍后旧 reply 8001 迟到。若 driver 只按 \code{task_id} 更新结果，旧 reply 可能覆盖新提交或重复计数；若 commit table 按 \code{task_id} 去重并检查 attempt，旧 reply 只能进入 late reply 指标。

这个案例要写出状态表每一步。Pending、Sent、Running、Retrying、Committed 如何变化；\code{current_attempt} 何时递增；commit table 何时写入；旧 response 如何被拒绝；临时文件如何清理。读者看到状态表，才能理解幂等提交不是口号。

实验中固定随机 seed，注入一次响应丢失和一次迟到。期望最终输出和串行 reference 一致，commit table 只有一条 task 17，late replies 为 1，临时目录没有未引用 block。这个验收比“作业成功”更具体。

\topic{MessageBus 的故障规则文件}

故障注入要可复现，不能靠随机心情。MessageBus 可以读取一个规则列表：第几条消息、匹配哪个 request id、执行 drop、delay、duplicate、reorder 或 corrupt。每条规则有 seed 和触发次数。这样报告能写“丢弃 request 8001 的第一次 reply”，别人可以复现同一路径。

规则文件还要记录注入点。丢请求和丢响应不同；worker 执行前崩溃和写出后崩溃不同；driver commit 前崩溃和 commit 后崩溃不同。单点问题单点分析，不要一次开一堆故障。只有单点路径都清楚后，组合故障才有意义。

项目中的 MessageBus 不要偷偷共享对象引用。消息进入 bus 时复制或移动成独立对象，worker 修改本地副本，不影响 driver 内存。这样本机模拟才接近跨进程语义。若为了性能共享指针，就必须在教学文本中说明这是模拟层优化，不能改变协议边界。

\topic{分布式章节的源码阅读边界}

本章不是网络库教程。源码阅读可以看 Linux socket buffer、epoll ready、TCP 重传的高层资料，但主线仍是协议状态。读者先把 request/reply、deadline、retry、commit table 写清，再去看网络实现。否则很容易陷入 API 细节，忘记超时不能证明远端失败。

也可以阅读成熟系统的任务状态设计，例如 MapReduce/Spark 的 stage、task、attempt 概念，但要避免照搬。阅读时只摘取身份、attempt、speculation、shuffle 文件和 commit protocol 这些与本章项目对应的点。每读一个概念，都要写“它解决本项目哪个失败路径”。

本章作业要求写一个故障矩阵，而不是写更多代码。行是故障点，列是 driver 状态、worker 状态、临时输出、commit table、最终结果。填完矩阵，代码实现就有了目标。

\topic{综合练习：填写响应丢失故障矩阵}

本题给定一个 task 的时间线：driver 发送 request，worker 收到，worker 写临时输出，reply 丢失，driver 超时重试，新 attempt 成功提交，旧 reply 迟到。读者要填写矩阵：每个时间点 driver task state、current attempt、commit table、临时文件、可接受响应集合、指标如何变化。矩阵中任何一格写不出，就说明状态机不完整。

第二步改故障点：丢请求而不是丢响应。此时 worker 没有执行，旧输出不存在，driver 重试是安全的。再改成请求重复投递，worker 可能执行两次同一 attempt。读者要比较三种情况，不能用同一个“超时重试”句子糊弄。单点问题单点分析，是分布式章的基本纪律。

第三步加入 driver 重启。内存中的 current attempt 丢失后，持久 task table 和 commit table 如何恢复？旧 worker 回来的响应带旧 epoch，应如何处理？若没有持久状态，矩阵会空掉。这个题能让读者理解为什么分布式系统必须把调用栈变成状态表。

验收不是口头回答，而是项目测试。MessageBus 按规则注入故障，程序输出矩阵摘要和最终 checksum。测试通过时，late reply、stale reply、retry count 等指标也要符合期望。

\topic{容易出错的边界：远端调用不能用本地直觉修补}

第一个反例是“超时等于失败”。超时只说明本地没等到，不说明远端没执行。第二个反例是“重试总能提高可靠性”。没有幂等和预算的重试会重复提交并放大故障。第三个反例是“响应到了就成功”。响应可能来自旧 attempt、旧 epoch 或旧 request，必须先检查身份和状态。

第四个反例是“本机模拟共享内存没关系”。如果 driver 和 worker 共享业务对象，消息丢失、序列化、生命周期和旧响应都被掩盖。第五个反例是“exactly once 表示只执行一次”。在故障下执行可能多次，系统能保证的是外部提交只生效一次。第六个反例是“日志里有 request id 就够”。状态表、指标、manifest 和临时文件都要使用同一身份体系。

本章源码索引要求读者检查所有跨边界函数调用：是否实际上绕过了 MessageBus，是否传了指针或引用，是否共享了 mutable 状态，是否没有 request id。任何绕过协议的捷径都要删除或标明只用于测试。教学模拟要逼迫读者面对分布式语义，而不是用本机便利逃避问题。

故障注入也要写索引。每个规则有名字、触发点、期望指标和恢复动作。比如 \code{drop_reply_once} 期望 retry count 加一、late reply 可能出现、commit 仍唯一。规则名比随机代码更能形成学习材料。

\topic{本章核心接口：MessageEnvelope}

分布式模型章给项目留下 \code{MessageEnvelope}。所有消息都有 protocol version、message type、job id、stage id、task id、attempt、request id、epoch、deadline、payload bytes 和 checksum。业务 payload 可以变化，envelope 字段必须稳定。没有 envelope，就无法统一处理重试、迟到响应、版本兼容和 trace。

MessageEnvelope 也让本机模拟更真实。Driver 和 worker 只能看到 envelope 和 payload，不能直接共享对象。故障注入规则匹配 envelope 字段，指标也按 envelope 聚合。这个接口是从单机运行时走向分布式系统的关键边界。

\topic{从消息模型进入分区和复制}

消息模型解决单个 task attempt 的身份、超时、重试和提交。下一章的问题更大：当很多 task、很多 key、很多 partition 同时存在时，谁拥有哪份状态，状态如何复制，旧 owner 的消息如何拒绝。没有第十六章的 request、attempt 和 commit table，就无法理解分区 epoch；没有分区 epoch，迟到响应和迁移会再次失控。

项目中，MessageEnvelope 的 epoch 字段在本章可能只是 driver 重启版本，下一章会扩展成 OwnershipEpoch。这个扩展说明协议字段不是随便加的，而是从单点故障逐步推广到状态归属。读者应看到概念如何长大，而不是突然进入“共识”大词。

\topic{深入补充：单点故障要按位置拆开}

分布式故障不能只写“worker 失败”。Worker 在收到请求前失败，driver 只会超时；worker 收到请求后、执行前失败，可能没有输出；执行后、写临时文件前失败，没有可提交结果；写临时文件后、发送响应前失败，输出存在但 driver 未知；发送响应后失败，driver 可能已经提交。每个位置对恢复的要求不同。把它们都叫 worker failed，会掩盖协议边界。

Driver 失败也要拆。发送请求前失败，重启后任务仍 pending；发送请求后、持久化状态前失败，可能出现远端执行但 driver 不知道；commit table 写入前失败，临时输出未发布；manifest 写入后失败，结果已可见。教学项目可以简化持久化，但必须写清简化假设。否则读者会误以为内存状态足以恢复分布式作业。

网络故障同样拆位置。请求丢失、请求重复、响应丢失、响应重复、响应乱序、响应损坏，处理方式都不同。请求重复要求 worker 或 commit 幂等，响应乱序要求 attempt 检查，响应损坏要求 checksum 或协议校验。单点问题单点分析，就是把这些位置逐个写进测试，而不是用一个随机故障开关覆盖一切。

\topic{深入补充：消息格式和版本演进}

教学项目即使用内存对象模拟消息，也要设计稳定格式。字段要有版本，未知字段如何处理要有策略，必需字段缺失要拒绝。真实系统滚动升级时，新旧 worker 可能同时存在；即使本书项目不实现滚动升级，也要让读者知道协议版本为什么存在。没有版本，消息结构一改，旧状态和旧日志都可能无法读取。

版本演进有三条保守规则。第一，新增字段要有默认值，旧消息读入时能得到合理语义。第二，改变字段含义要增加版本，不要复用旧名字。第三，删除字段要经过兼容期，至少恢复程序能读旧 checkpoint。Manifest、task table 和 message envelope 都属于长期状态，不应随意改变。

这部分会影响第三册模型文件管理。模型 artifact、tokenizer、量化参数和算子版本都需要稳定元数据。第二册在 MessageEnvelope 中训练版本意识，第三册就能自然迁移到 ModelArtifact。

\topic{深入补充：本机模拟的边界声明}

本机模拟无法覆盖真实网络的全部问题。它没有跨机器时钟漂移，没有真实 TCP 拥塞控制，没有进程崩溃后的资源泄漏，没有操作系统级网络分区，也没有多版本节点长期共存。教材必须把这些边界写清楚。写清边界不是降低质量，而是防止读者把模拟结论过度外推。

但是本机模拟仍然非常有价值。它可以稳定复现请求丢失、响应丢失、重复、乱序、队列满、driver 重启和 commit 幂等。它让读者在一台手机或笔记本上训练协议思维。等将来进入真实多机环境，新增的是网络和运维复杂度，核心状态机已经有基础。

因此本章验收分两层。第一层是模拟内语义正确：所有故障注入下 checksum 和状态表符合预期。第二层是边界声明完整：列出哪些真实分布式问题尚未覆盖。两个层次都完成，才算合格。

\topic{故障矩阵细化：请求、执行、输出、响应四段分开}

为了让分布式模型不再模糊，本章可以把每个 task attempt 拆成四段。第一段是请求投递：driver 生成 request，MessageBus 接收，worker 队列入队。这里的失败包括请求没有进入 bus、进入 bus 但被丢弃、worker 队列满、请求重复入队。第二段是远端执行：worker 取出 request，检查 epoch 和 deadline，读取 input split，运行 map。这里的失败包括 worker busy、deadline 已过、输入读失败、解析错误、执行中崩溃。第三段是输出生成：worker 写临时 block，计算 checksum，记录 output bytes。这里的失败包括短写、校验不一致、临时目录不可写、写出后崩溃。第四段是响应返回：worker 发送 reply，driver 接收，验证身份和输出，写 commit table。这里的失败包括响应丢失、重复、乱序、旧 attempt、旧 epoch、commit 前崩溃。

这四段对应四类状态，不能混成一个 Running。请求投递阶段失败，通常可以换 worker 重发；远端执行阶段失败，要看是否产生外部效果；输出生成阶段失败，可能需要清理临时文件；响应返回阶段失败，最容易出现“远端已经做完，本地不知道”。不同阶段的重试语义不同。若教材只写“超时后重试”，读者就会忽略这些差异。

故障矩阵可以用文本列表示。行是故障点，列是是否可能已有临时输出、是否可能已有外部提交、是否允许同 attempt 重发、是否必须增加 attempt、是否需要清理、是否影响最终 checksum。比如响应丢失时，可能已有临时输出但未提交，允许新 attempt，旧 reply 迟到要拒绝；请求丢失时，没有临时输出，重发较简单；commit 后 driver 崩溃时，最终输出已可见，恢复不能重复提交。这样的矩阵能把分布式故障从故事变成决策表。

\topic{本机模拟中的持久化最小集}

即使是本机模拟，也要明确哪些状态必须持久化。最小集合包括 job id、epoch、task table、commit table、partition assignment 和 manifest。Task table 让 driver 知道哪些任务未完成；commit table 防止重复提交；epoch 拒绝旧响应；manifest 告诉 reduce 哪些输出可见。若这些只存在内存里，driver 重启测试没有意义。

持久化可以很简单。教学项目可以用文本或 JSON 行，每条记录带 version、checksum 和换行结束标记。恢复时读到半行就丢弃或报错，读到 checksum 不匹配就拒绝。这个实现不追求数据库级能力，但能训练两个关键概念：状态要有稳定格式，恢复要验证完整性。不要用“以后再持久化”跳过这个边界，因为一跳过，很多分布式语义都无法测试。

持久化状态还要和临时文件目录对账。恢复时扫描 manifest 引用的 block，确认存在且 checksum 正确；扫描临时目录中未引用 block，按 epoch 和 attempt 判断可清理或保留诊断。这个对账过程和 I/O 章的崩溃点矩阵衔接。单机可靠写入和分布式 commit 在这里合并成同一条恢复路径。

\topic{故障注入报告的固定格式}

每个分布式故障实验都应输出固定格式。第一段写规则：例如 drop reply for request 8001 once。第二段写期望：retry count 加一，commit table 最终一条，late reply 一条，checksum 与 reference 一致。第三段写实际状态：task table 状态变化，worker 输出文件，driver 指标。第四段写结论：是否通过，若失败，失败发生在哪个阶段。第五段写边界：本实验没有覆盖真实网络拥塞、时钟漂移或多进程崩溃。

这种格式能避免故障实验变成日志截图。读者不需要翻几千行日志找答案，而是先看摘要，再按 id 追踪细节。项目工具可以自动生成一部分摘要，书中展示文本样例。EPUB 中保持文本形式，也避免图片解析问题。

本章最后要强调，分布式模型的学习成果不是会说 RPC、CAP 或一致性，而是能面对一个具体故障，写出状态表、提交边界和恢复动作。只要做到这一点，后续分区、复制和最终工程章才有坚实基础。

\section{项目落地}

Compute Systems Engine 的分布式模拟从三个对象开始：driver、worker、message bus。driver 生成任务和 request id；worker 读取本地分片并执行 map/reduce；message bus 用有界队列模拟网络，支持随机延迟、超时、重复和丢弃。每条消息携带 request id、attempt、source、target partition 和 payload size。

第一版只做日志计数：worker 本地统计 key，再把 partial 发给 reduce partition。第二版加入重试：message bus 随机丢响应，driver 重试同一 request id，reduce 端必须去重。第三版加入背压：reduce 队列满时返回 busy 或延迟，map 端降低发送速率。第四版加入检查点：每个 partition 记录已提交 request id 和结果。

\topic{消息身份}
request id 表示一次逻辑操作，attempt 表示第几次尝试。重试不能换 request id，否则服务端无法去重。


\topic{超时}
超时不是失败证明。远端可能没收到，可能排队，可能执行完成但响应丢失。客户端必须把未知状态写进协议。


\topic{幂等}
幂等要把 request id、业务状态和结果放在同一提交边界里。只在客户端生成 ID，不在服务端持久记录，不能防重复执行。


\topic{deadline}
跨层调用应传递截止时间，而不是每层重新给自己完整 timeout。deadline 让下游知道剩余预算，避免请求超过用户期望。


\topic{重试预算}
重试提高成功率，也可能制造风暴。预算要按一次上游请求计算，限制 attempt 总数、退避和抖动。


\topic{消息版本}
协议升级要求版本、类型、长度和兼容规则。C++ 结构体内存布局不能直接当 wire format。


\topic{背压协议}
服务端 busy 应进入响应分类，让客户端降低并发或停止低优先级请求。把 busy 当普通失败重试，会放大过载。


\topic{本机模拟}
本册可以用 MessageBus 在本机模拟丢包、重复、延迟和 busy。模拟不能证明网络性能，却能验证语义。


\topic{案例：响应丢失}
worker 已提交结果但响应被丢弃，driver 用同一 request id 重试。reduce 端应返回旧结果，不重复执行。


\topic{案例：慢节点和重试风暴}
让一个 worker 变慢，客户端无预算重试，再加入 retry budget 和 jitter，比较请求放大倍数。


\topic{案例：协议版本}
构造旧版本 worker 收到新字段的消息，要求拒绝或按兼容规则处理。


\topic{源码阅读路线}
Linux 源码阅读从 \filepath{net/core/dev.c}、\filepath{fs/eventpoll.c}、\filepath{kernel/time/timer.c} 附近进入即可。阅读目标不是把整套子系统抄进书里，而是追一条和本章实验相关的路径：用户态动作怎样变成内核对象，内核对象怎样排队、等待、唤醒、计数、提交或回收。

阅读报告要写证据等级。能直接测到的数字放在第一层；从源码控制流推导出的解释放在第二层；当前设备无法验证、只能作为合理猜测的内容放在第三层。这样读者会知道 消息语义报告 中哪些结论可复现，哪些结论需要换机器继续确认。

源码阅读还要克制。看到一个复杂函数，不要把每个分支都展开；先回到本章的问题，问它能否解释一个等待点、一次状态转移或一条性能曲线。解释不了的内容可以留到索引，不要打断正文主线。

\topic{响应丢失}
worker 已提交结果但响应被丢弃，driver 用同一 request id 重试。reduce 端应返回旧结果，不重复执行。

\topic{慢节点和重试风暴}
让一个 worker 变慢，客户端无预算重试，再加入 retry budget 和 jitter，比较请求放大倍数。

\topic{协议版本}
构造旧版本 worker 收到新字段的消息，要求拒绝或按兼容规则处理。

\topic{RPC 不是函数}

远程调用经过序列化、排队、网络、对端调度和响应路径。超时后不知道远端是否执行。

实验设计：让服务端执行成功但响应丢失，观察客户端重试。

\topic{消息身份}

request id、attempt id、trace id 和 partition id 分别解决不同问题。一个 id 不能承担全部语义。

实验设计：构造同一 request 多次 attempt，检查去重表。

\topic{幂等提交}

幂等不是客户端生成 id 就够了。服务器要把 request id、业务修改和结果放进同一提交边界。

实验设计：模拟修改状态后崩溃但未记录 id 的错误路径。

\topic{重试预算}

重试能提高成功率，也能放大过载。预算要按上游请求计算，不能每层服务各自无限重试。

实验设计：统计原始请求数、尝试数、超时数和最终失败数。

\topic{退避和抖动}

所有客户端同时重试会形成重试风暴。指数退避和抖动让压力分散，但会增加尾延迟。

实验设计：让多个客户端同时遇到慢服务，比较固定间隔与抖动。

\topic{逻辑时间}

墙钟适合观测，不适合证明顺序。epoch、term、version 和 log index 表达协议中的先后。

实验设计：让两个节点时钟错开，说明不能用时间戳决定提交顺序。

\topic{故障检测}

超时只能表示怀疑，不能证明节点已死。慢节点、网络拥塞和进程暂停都可能表现为心跳丢失。

实验设计：在模拟器中注入慢 worker 和断连，比较恢复动作。

\topic{部分失败}

十个分片九个成功一个失败，业务要决定返回部分结果、失败整体还是降级。没有统一答案，只有语义选择。

实验设计：为搜索、计费和统计三种业务分别写处理策略。

\topic{本机模拟}

没有多机也能训练协议思维。多进程或多线程队列可以模拟乱序、重复、丢失、延迟和崩溃。

实验设计：实现有界 MessageBus，并可配置错误率和延迟。

\topic{项目接口}

MessageEnvelope 记录协议版本、请求身份、attempt、deadline、payload、错误分类和幂等键。

实验设计：后续分片、复制和 shuffle 都使用同一消息头。

\section{分布式不是远程函数调用，而是消息、身份和不确定性}

分布式模型章节要先拆掉本地函数调用直觉。本地函数调用要么进入函数体，要么调用前失败；调用栈、对象寿命和异常都在一个进程里。RPC 看起来像函数，实际经过序列化、队列、网络、对端调度、执行、响应和客户端完成路径。任何一段都可能慢、丢、重复或部分完成。超时后，客户端不知道远端是否执行，这就是幂等和提交点出现的根源。

消息身份不能只放一个 id。request id 表示逻辑请求，attempt id 表示一次尝试，trace id 串起跨节点路径，partition id 指示归属，epoch 或 version 表示路由和协议状态。缺少 attempt，服务端无法区分第一次请求和重试；缺少 request id，重复请求无法去重；缺少 epoch，旧路由可能继续写旧 owner。身份字段不是日志装饰，而是协议语义。

幂等要落到提交边界。客户端生成 request id 只是开始。服务端必须把 request id、业务状态修改和响应结果放进同一个原子提交边界。若先修改业务状态，崩溃前没记录 request id，重试会重复执行；若先记录 request id，业务状态没改就崩溃，重试又可能误以为成功。数据库事务、WAL、共识日志或状态机日志都是围绕这个边界服务。

重试预算是性能和正确性的交界。一个请求调用十个下游，每个下游重试三次，系统负载可能迅速放大。服务已经慢时，所有客户端同时重试，会形成重试风暴。指数退避和抖动能分散压力，但也会增加尾延迟。报告要记录原始请求数、尝试数、超时数、成功数和最终失败数，而不是只写“加了重试”。

时间也要重新理解。墙钟适合日志排序和观测，不适合证明因果。两个机器时钟可能漂移，进程可能暂停，网络延迟可能抖动。逻辑时间如 epoch、term、version、log index 才是协议里的顺序。请求 A 触发请求 B，B 的墙钟时间可能更早，但因果上仍在 A 之后。教材要让读者看到为什么分布式系统常用版本号而不是只看时间戳。

本章可以在本机模拟，不必等待真实集群。用多线程或多进程建立 MessageBus，每个 worker 有输入队列，网络由有界队列、随机延迟、重复、丢弃和乱序模拟。这个模拟不能证明真实网络性能，却能验证语义：request id 是否去重，attempt 是否可追踪，deadline 是否传播，背压是否控制队列，迟到响应是否被安全处理。先在一台机器上把协议状态讲清，迁移到真实网络时才不会把不确定性放大成混乱。

故障模型要写明。多数教学系统处理 crash-stop 或 crash-recovery，不处理恶意拜占庭错误。机器慢、网络分区、响应丢失、消息重复和进程重启分别需要不同策略。CAP 也要具体到场景：分区发生时，是继续响应可能陈旧的请求，还是拒绝以保持强一致。只背 CAP 缩写没有工程价值。

\topic{案例推导：一次响应丢失如何逼出 request id、attempt 和幂等表}

客户端向 worker 发送“处理 split 7”的请求，worker 完成处理并提交输出，但响应在返回途中丢失。客户端超时后重试。同一个逻辑请求现在有两次 attempt：第一次可能已经成功，第二次正在发送。如果系统只按当前连接状态判断，就会把同一 split 再处理一遍，产生重复输出。这个小案例足以推导出 request id、attempt id 和幂等提交表。

request id 表示逻辑请求，重试不能改变它；attempt id 表示一次尝试，便于观察和诊断；输出提交表记录 request id 对应的结果和状态。worker 收到重复 request id 时，不应再次执行业务副作用，而应返回之前保存的结果。这里的关键是保存结果必须和业务状态修改处于同一提交边界。否则修改完成但结果未保存时崩溃，重试仍会重复执行。

超时也要重新解释。客户端超时只说明本地等待超过预算，不说明服务器没执行。取消请求也是尽力而为，不保证远端停止。客户端可以停止等待用户结果，但系统仍要消费迟到响应，清理 attempt 状态，并防止重复回调。这个逻辑和 I/O completion 迟到完全一致，只是完成事件来自网络。

重试预算防止系统自我放大。若每个 split 请求最多重试三次，一个作业有一万个 split，最坏尝试数就会显著增加。若下游已经过载，固定间隔重试会形成风暴。指数退避和抖动可以分散压力，但 deadline 剩余不足时不应继续重试。报告要记录每个 request 的 attempt 数和最终原因，而不是只记录成功率。

本机模拟可以复现全部语义。MessageBus 随机丢弃响应、重复请求、延迟消息；worker 维护幂等表；driver 维护 deadline 和 attempt 状态。输入不需要大，三个 split 就够：一个正常，一个响应丢失，一个 worker 慢。观察 manifest 是否只提交一次，迟到响应是否被识别，重试预算是否生效。这个实验比搭建真实集群更适合教学，因为每个不确定性都可控。

源码或工具阅读可以补充 socket 的 ready 语义和网络缓冲，但不要把本章变成网络协议大全。读者要带走的是：分布式正确性从“我调用了函数”转成“我发送了带身份的消息，并定义了重复、超时和提交”。

\topic{分布式从本地调用失效开始}
本地函数调用有调用栈、返回值和异常；远程调用没有这些天然保证。请求可能丢，响应可能丢，远端可能执行成功但本地不知道，远端可能执行两次，本地时钟可能和远端不一致，网络延迟可能突然变大。分布式系统模型不是概念列表，而是解释为什么本地调用语义在跨进程后失效。只有从这个断裂开始，超时、重试、幂等和 trace 才有必要。

贯穿项目中，driver 发送 task 给 worker。若 driver 超时没有收到响应，有三种可能：worker 没收到，worker 收到但还没执行完，worker 执行完但响应丢了。driver 若直接重发，就可能让同一个 task 执行两次。若 task 写输出没有 attempt 身份和幂等提交，重复执行会污染结果。这个例子应贯穿本章，而不是只在最后出现。

\topic{消息身份是恢复的起点}
每条请求至少需要 request id、task id、attempt id、driver epoch 和 deadline。Request id 用于去重，task id 表示逻辑任务，attempt id 表示一次执行尝试，epoch 区分 driver 重启或领导权变化，deadline 限制过期工作。响应也要带回这些身份，否则 driver 无法判断它对应哪个请求、是否过期、是否来自旧 attempt。

幂等结果表记录已经接受的结果。若同一个 request id 重复到达，可以返回同一摘要；若旧 attempt 晚到，要检查它是否仍是有效 attempt；若新 attempt 已提交，旧响应不能覆盖。这里的核心不是某个 RPC 框架，而是身份和状态机。没有身份，重试无法安全。

\topic{超时不是失败证明}
超时只说明在本地预算内没有等到结果，不证明远端没有执行。把超时当失败证明，是分布式程序的常见错误。正确做法是把超时视为不确定状态，进入查询、重试、取消或降级流程。重试必须和幂等配合，取消也不一定能终止远端正在运行的任务，只能请求它在安全点停止或忽略其结果。

Deadline 比单次 timeout 更完整。Deadline 表示整个请求的截止时间，传递给下游后，每一层都能知道剩余预算。否则每层都用自己的 timeout，会让总耗时无限放大。项目里的 task message 应携带 deadline，worker 在开始前检查，运行中在 batch 边界检查，写出前再次检查。过期任务不应继续消耗大量资源。

\topic{Trace 替代调用栈}
跨进程后，本地调用栈断了，trace id 需要承担连接责任。一条任务从 driver 发出，到 worker 接收，到执行 parse，到写 block，到返回结果，每个事件都带同一个 trace id 和 task identity。故障时，读者可以沿 trace 看到请求在哪里等待、在哪里重试、哪个 attempt 写了输出、哪个响应被丢弃。没有 trace，分布式调试只能靠猜。

Trace 也能用纯文本输出：时间、组件、事件、request id、attempt、state、摘要。EPUB 中不需要图片。读者可以把文本 trace 和单机任务 trace 对照，看到第二册从线程运行时到分布式 driver 的连续性。

\topic{实验：丢响应后的正确行为}
构造一个本机 MessageBus，支持注入丢请求、丢响应、延迟、重复和乱序。场景是 worker 实际完成并写出结果，但成功响应被丢。Driver 超时后重试，worker 收到重复 request。正确行为是：若原 request 已完成且结果可验证，返回同一结果摘要；若新 attempt 被创建，提交表只能接受一个有效 attempt；旧 attempt 晚到不能覆盖新提交。

报告要列状态表，而不是只说“支持重试”。状态包括 Sent、Received、Running、Completed、ResponseLost、Retrying、DuplicateObserved、Committed。每个状态允许哪些转移，哪些消息被忽略，哪些消息触发告警。分布式章节若没有状态表，就容易变成概念堆砌。

\topic{Linux 和系统工具连接点}
本章不需要深入内核网络栈，但要会用基本工具观察连接和延迟。\cmd{ss} 查看连接状态，\cmd{tcpdump} 或日志 trace 观察消息，\cmd{tc netem} 可以注入延迟和丢包，进程日志记录 request id。源码阅读可以关注 RPC 框架或消息队列如何处理 request id、deadline、retry 和 connection state。重点仍是语义，不是工具列表。

源码阅读只服务这个问题。为什么 busy 和 fatal 要区分，不是为了多记一个术语，而是为了处理 worker 成功执行但响应丢失，driver 超时后重试导致重复执行 这样的具体场景。如果仍沿用 RPC 返回失败就重新调用，没有 request id 和结果表，小输入可能看不出问题，可一旦出现并发、失败、倾斜或资源等待，状态就会散落在调用栈、临时文件、日志文本或线程局部变量里。本章要求读者把这些状态写回 message envelope、request id、attempt id、deadline、idempotent result、fault rule，让后续观察和恢复有共同语言。


C++ 代码里的表达要跟这个推导一致。身份、长度、版本、计数和偏移使用位宽明确的整数；只读数据用视图；拥有者和观察者分开；状态用枚举或结构化记录。测试不能只走正常路径，还要覆盖旧输入、压力输入、取消、重试、关闭或恢复。若这个点不能进入测试和报告，它就还只是文字解释。

\topic{消息故障实验判读}
分布式模拟的判读核心是同一逻辑任务是否只有一个有效结果。丢请求、丢响应、重复响应、乱序响应和 deadline 过期要分别记录。超时后看到迟到成功，不应覆盖已提交的新 attempt；重复 request 应返回同一摘要或明确拒绝。trace 中缺少 request id，就无法做这种判读。


这一段也服务于复习。读者合上书后，应该能凭本章的判读口径重新审查自己的代码：有没有最小正确版本，有没有能击穿旧方案的输入，有没有状态记录，有没有资源指标，有没有失败后的恢复动作。若五个问题缺两个以上，就说明实现还停在演示层，而不是工程层。


\topic{丢响应场景的逐步状态表}
丢响应不是一句“重试即可”。Driver 发送 request 17，worker 收到并执行，写出 attempt 3 的临时结果，提交表接受 attempt 3，成功响应在网络中丢失。Driver 到 deadline 后只知道没有收到响应，不知道远端是否执行。此时重试必须携带同一 task id 和新的或相同 attempt 策略。

若服务端保存 request id 的结果摘要，重复 request 可以返回同一摘要；若 driver 创建新 attempt，提交表必须保证旧 attempt 和新 attempt 只有一个进入 manifest。迟到响应到达时，也要根据 attempt 状态判断是否忽略。这个状态表能让读者理解：超时后的世界是不确定的，不是简单失败。

本机 MessageBus 实验应把这些状态打印出来。只看最终输出一致还不够，要看到哪个消息被丢，哪个 request 被重试，哪个 attempt 被接受，哪个响应被忽略。分布式系统的可调试性来自这种身份链。

\topic{实验课：本机 MessageBus 注入真实分布式语义}
MessageBus 不需要真实网络，也能模拟分布式关键问题。每条消息带 request id、task id、attempt id、deadline 和 trace id。故障规则可以按 request id 丢请求、丢响应、延迟、重复、乱序。Driver 和 worker 跑在同机进程或线程里，但只通过消息交互，不共享调用栈。

丢响应实验最重要。Worker 已经执行并写出结果，响应被丢。Driver 超时后重试。若 worker 保存 request id 到结果摘要的映射，重复请求可以返回同一摘要；若 driver 生成新 attempt，提交表必须保证只有一个 attempt 进入 manifest。迟到响应到达时，要根据 attempt 状态决定忽略或确认。

Deadline 实验也要做。每层都用自己的 timeout 会放大总时间；携带 deadline 后，worker 在开始前、批次边界和写出前检查剩余预算。过期任务不应继续消耗大量资源，除非业务明确要求必须完成。报告要比较局部 timeout 和全链路 deadline 的行为差异。

最终 trace 应能串起一次请求：send、receive、start、write、commit、respond、retry、ignore。没有 trace id 的系统，遇到分布式故障只能猜。

\topic{消息系统练习验收重点}
合格答案不能把 timeout 当失败证明。必须写 request id、attempt id、deadline、重试预算和幂等结果表。丢响应题要给状态表：远端可能已执行，本地未知，重试如何不重复提交，迟到响应如何处理。

答案还要写“为什么不采用更复杂方案”。例如没有证据证明队列是瓶颈，就不应把锁队列换成无锁；没有证明输出顺序可变，就不应随意重排；没有证明错误路径能恢复，就不应只提交正常路径。能解释不做什么，说明读者开始具备工程判断。

最后，答案必须包含可复查材料。至少有输入摘要、关键状态、期望输出、实际输出、运行命令和未验证边界。没有这些材料，老师或同伴无法判断结论是否成立。第二册的作业不是让读者写漂亮段落，而是训练读者把系统问题变成可验证证据。

\section{综合案例：把本机函数调用改造成消息系统}

分布式系统不能从 CAP、共识、微服务这些大词开始。读者最需要先看见一个变化：本机函数调用有调用栈、共享地址空间和确定返回；消息系统只有请求、响应、超时、重复、乱序和部分失败。只要把 Compute Systems Engine 的 worker 从本地函数改成消息，就会自然推出 request id、deadline、幂等、重试预算和背压。

\topic{一、函数调用丢失了哪些保证}
本机调用 \code{run_task(task)} 时，调用者知道自己把参数交给了哪个函数；函数返回后，调用者知道它已经完成或抛出异常；如果进程没有崩溃，调用栈给出了清楚的因果关系。RPC 改变了这些假设。请求可能没发出，可能发出但丢失，可能被远端执行但响应丢失，可能响应迟到，可能执行两次，可能远端已经完成但本地超时。

因此，分布式语义的第一步不是选协议，而是补回本机调用丢掉的身份。每个请求需要 request id，每次任务尝试需要 attempt id，每个响应要带回对应身份，每个输出要说明由哪个 attempt 产生。没有这些身份，本地超时后无法判断迟到响应是否仍应采纳，也无法区分重试产生的重复输出。

\topic{二、超时不是取消}
初学者常把 timeout 理解成远端停止执行。实际 timeout 只说明本地等待时间耗尽，不说明远端发生了什么。远端可能从未收到请求，也可能正在执行，也可能已经写出输出，只是响应丢了。若本地超时后立即重试，而远端旧 attempt 后来成功，系统就可能重复提交。

正确做法是把超时和提交分开。超时触发本地状态从 Waiting 进入 Suspect 或 Retrying，但外部结果是否有效由 commit protocol 决定。旧 attempt 的响应迟到时，driver 根据 attempt id、route epoch 和 commit table 判断接纳、忽略还是记录为迟到。这样 timeout 只是故障检测信号，不是最终事实。

\topic{三、幂等性来自提交点，不来自祈祷}
幂等不是“我觉得这个操作重复执行也没事”。它要落到状态表。以 shuffle block 写出为例，worker 可以重复计算同一个 input split，但只有一个 attempt 的 block 被 manifest 接纳。未接纳的临时文件可以清理，迟到响应只能报告 already committed。这样重复执行不会重复计数，因为 reduce 只读取 manifest。

请求处理也一样。若操作有外部效果，例如写输出、扣库存、发送通知，就必须有稳定操作 id 和提交记录。收到重复请求时，系统返回第一次提交的结果，或明确说明仍在处理中。没有提交点，重试只是把临时故障变成重复效果。分布式章节要让读者把幂等看成状态设计，而不是接口注释。

\topic{四、消息信封是分布式系统的最小类型}
贯穿项目可以先在本机模拟消息，不急着上真实网络。关键是消息信封必须包含足够语义：job id、stage id、task id、attempt id、source、destination、deadline、route epoch、payload checksum。payload 是数据面，信封是控制面。分开之后，测试可以注入丢包、乱序、重复和延迟，而不破坏业务数据结构。

\begin{lstlisting}[language=C++,caption={消息信封保存控制面身份}]
struct MessageEnvelope {
    std::uint64_t job_id = 0;
    std::uint32_t stage_id = 0;
    std::uint64_t task_id = 0;
    std::uint32_t attempt_id = 0;
    std::uint64_t route_epoch = 0;
    std::uint64_t deadline_ns = 0;
    std::uint64_t payload_checksum = 0;
};
\end{lstlisting}

这个结构不关心 payload 类型，因此可以用于 map 请求、reduce 请求、心跳、取消和提交响应。它把分布式章节的抽象问题压成可测试对象：重复消息是否有同一个 identity，旧 epoch 消息是否被拒绝，deadline 过期后是否仍可能收到响应，checksum 不一致是否能被发现。

\topic{五、重试预算防止故障放大}
重试能掩盖短暂故障，也能放大故障。若所有客户端在同一超时时间后立即重试，故障节点刚恢复就会收到更大的流量；若 driver 对每个慢 task 无限重试，队列会被旧 attempt 和新 attempt 塞满；若没有退避和抖动，系统会形成同步风暴。重试策略必须有预算：最大次数、最大总等待、指数退避、随机抖动和按错误分类的处理。

错误分类很重要。连接失败、远端忙、校验失败、权限错误、输入不存在、版本过旧，不应使用同一种重试策略。可恢复错误可以重试，不可恢复错误应快速失败，疑似慢节点可以 speculative retry，但提交点仍然只允许一个 attempt 生效。工程报告要记录每类错误的次数和最终状态，否则“重试成功率”会掩盖系统压力。

\topic{六、背压跨越进程边界}
单机有界队列满了，生产者可以阻塞或失败；分布式系统中，队列满可能表现为远端拒绝、响应变慢、连接堆积或消息超时。背压必须在协议中显式表达。Worker 可以返回 Busy 并带建议重试时间，driver 可以限制每个 worker 的 in-flight 请求，shuffle 可以按 partition 维持水位，写出端可以把磁盘慢反馈给 map 端。

没有背压，系统会把压力转成内存、重试和尾延迟。更糟的是，局部优化可能放大全局故障：map 端变快后生成更多 shuffle block，reduce 端和磁盘来不及消费，driver 又因为超时重试制造更多 block。第 16 章要把背压从队列概念扩展成端到端的流量合同。

\topic{七、本机模拟要模拟失败，而不是只模拟 API}
很多项目用本机多线程模拟分布式，但仍然共享对象指针，函数调用直接返回，错误路径很少。这种模拟只能验证业务算法，不能验证分布式语义。真正有价值的本机模拟要强制消息通过队列，payload 序列化或复制，响应可能乱序，消息可能重复，driver 和 worker 只能通过信封识别请求。

故障注入至少包括五类：请求丢失、响应丢失、重复投递、延迟超过 deadline、旧 epoch 消息迟到。每类故障都要检查 commit table 和 manifest 是否保持一致。若模拟器没有这些故障，后面写真实网络只会更难。

\topic{八、观察指标要从平均值扩展到分布}
分布式系统的平均延迟经常骗人。少量慢请求会拖住 stage barrier，少量热点 partition 会拖住 reduce，少量重试会让尾部时间膨胀。指标至少要包含 in-flight 请求数、队列水位、超时次数、重试次数、迟到响应次数、重复请求次数、每个 worker 的成功率、每个 partition 的字节数和耗时分位数。

这些指标要和身份关联。只知道有十次 timeout 不够，还要知道它们属于哪个 stage、哪个 worker、哪个 partition、哪个 attempt。否则无法判断是网络抖动、热点 key、磁盘慢还是调度错误。第一章建立的 trace id，在分布式章正式发挥作用。

\section{案例深化：消息乱序和重复如何改变程序结构}

本机调用的返回值天然对应一次调用；消息系统里，响应可能乱序、重复、迟到。程序结构必须从“等待一个返回值”改成“维护请求状态表”。这个改变是分布式系统的入门门槛。

\topic{一、请求状态表替代调用栈}
Driver 发送请求后，把 request id、attempt id、deadline、状态和重试次数写入表。收到响应时，根据 id 查表，而不是依赖当前调用栈。若响应对应已提交 attempt，忽略或返回已提交结果；若对应已取消 attempt，记录迟到；若未知，可能是旧消息或 bug。状态表让乱序响应变成可处理事件。

没有状态表，代码常写成发送后阻塞等待，这会把消息系统退化成本地调用，并隐藏迟到和重复问题。教学模拟器应强制异步回调或事件循环，让读者面对状态表。

\topic{二、重复请求必须返回同一语义}
重复请求可能来自客户端重试，也可能来自消息总线重复投递。服务端看到同一个 request id，应返回第一次处理结果、当前处理中状态或明确拒绝，而不是再执行一次外部效果。若任务是纯计算且输出未提交，重复执行可能可接受；若写 manifest 或外部系统，必须去重。

去重表也有生命周期。保存太久耗内存，保存太短会让迟到重复逃逸。通常按 job、epoch 或 checkpoint 清理。项目可以在 job 完成后一段时间保留 request outcome，用于处理迟到响应。清理策略也是协议的一部分。

\topic{三、乱序测试}
模拟器应把响应顺序随机打乱。发送 task 1、2、3，可能先收到 3，再收到 1，最后 2 超时。Driver 不能假设响应顺序等于发送顺序。测试检查最终 manifest 是否按 task id 和 attempt id 提交，而不是按响应到达顺序错误覆盖。

\topic{四、事件循环报告}
分布式模型实验应保存事件日志：send、deliver、drop、timeout、retry、response、commit、ignore stale。每个事件带 logical time 和 id。读者能从日志重放状态机，就说明协议可解释。没有事件日志，分布式 bug 很难复盘。

\section{补强案例：时钟只能辅助，不能替代协议}

分布式系统里很容易滥用时间。超时、租约、重试退避、指标窗口都需要时钟，但时钟有漂移、暂停和调度延迟。协议不能仅凭“我觉得已经过了足够久”判断远端一定停止或旧 leader 一定失效。

\topic{一、超时是怀疑，不是事实}
请求超时只说明本地等待超过阈值。远端可能仍在执行。把超时当成取消，会导致重复提交。正确做法是超时后进入怀疑状态，发起重试或探测，但最终提交仍靠 attempt id 和 commit table 判断。

\topic{二、租约需要时钟假设}
Leader lease 或 cache lease 可以减少读协调，但依赖时钟误差有界和进程不长时间暂停。若虚拟机暂停、系统休眠或 GC 停顿超过租约假设，旧 leader 可能继续认为自己有效。使用租约必须写出时钟假设和安全边界。

\topic{三、逻辑时间用于排序事件}
Route epoch、term、log index、attempt id 都是逻辑时间。它们不代表物理时间，却能表达版本先后。分布式协议更常依赖逻辑时间而不是墙钟。项目中的 route epoch 和 manifest version 就是逻辑时间，用来拒绝旧请求和旧输出。

\topic{四、实验}
在消息模拟器中加入延迟和本地时钟跳变。检查 timeout 后旧响应是否被正确处理，旧 epoch 请求是否被拒绝，重试退避是否避免风暴。报告明确哪些地方使用物理时间，哪些地方使用逻辑版本。这样读者不会把时间戳当作协议本身。

\section{协议案例：从一次重复提交推导幂等表}

分布式系统里，重复提交比请求失败更危险。失败通常能重试，重复提交会让结果悄悄变错。日志统计作业中，如果同一个 map attempt 的 block 被 reduce 读两次，最终计数可能翻倍。这个事故能让读者理解幂等表、attempt id 和 manifest 的必要性。

\topic{一、重复提交的具体时间线}
Driver 发送 map attempt 5 给 worker。Worker 计算完成，写出临时 block，并发送成功响应。响应在网络中延迟，driver 超时后启动 attempt 6。Attempt 6 也成功，driver 把 attempt 6 的 block 写入 manifest。随后 attempt 5 的迟到响应到达。如果 driver 只看 task id，不看 attempt 和 commit 状态，可能把 attempt 5 也写入 manifest。Reduce 读取两个 block，结果重复计数。

注意这里每个局部动作都“成功”：worker 没错，网络只是延迟，重试也合理。错误来自控制面没有定义唯一提交。这个例子比抽象讲 exactly once 更适合入门，因为它把重复效果压到一个可见输出上。

\topic{二、幂等表保存请求结果}
幂等表按 task id 或 operation id 记录提交状态。状态可以是 Empty、Running、Committed、Failed。Committed 记录被接纳的 attempt id、block id、checksum 和结果摘要。新响应到达时，driver 先查表：若 Empty，可以尝试提交；若 Committed 且 attempt 相同，返回已提交；若 Committed 但 attempt 不同，忽略迟到；若 Running，按策略等待或比较 attempt。

这张表必须持久化或进入 checkpoint，否则 driver 崩溃后会忘记已提交操作。幂等不是内存 map 这么简单，它是恢复状态的一部分。保存多久也要定义：job 完成后可以清理，跨 job 的 request id 不应冲突。

\topic{三、提交要原子地更新表和 manifest}
如果 driver 先写 manifest，再更新幂等表，中间崩溃后恢复可能不知道 manifest 中的条目对应哪个提交状态；如果先更新表，再写 manifest，恢复后可能认为已提交但 reduce 看不到 block。需要一个明确的提交协议。单机项目可以让 manifest entry 本身就是提交记录，或用临时 manifest 加原子 rename；多机系统可能需要复制状态机。

教学项目可简化为：构造 ManifestEntry，校验 block，写入 manifest log，fsync 或按承诺级别同步，内存表从 log 重建。这样崩溃后以 manifest log 为准。关键是只有一个权威提交源，不能让目录扫描、内存表和日志各说各话。

\topic{四、幂等表和重试预算配合}
幂等表防重复，重试预算防风暴。一个任务超时后，可以启动新 attempt，但同时记录旧 attempt 可能迟到。达到重试上限后，task 进入 Failed，不再接受新 attempt 的成功响应，除非恢复策略明确允许。否则系统可能在用户已经看到失败后，又迟到提交成功结果。

状态机要写清最终状态。Committed 和 Failed 都是终态吗？Failed 后是否允许人工 retry？人工 retry 是否使用新 task id 或新 epoch？这些问题如果不写，迟到消息会不断挑战边界。分布式系统的难点不在 happy path，而在终态和旧消息。

\topic{五、实验：重复响应矩阵}
模拟四种情况：响应丢失后重试成功，旧响应迟到；旧响应先到，新响应后到；两个 attempt 都写出 block 但只有一个 commit；driver commit 后崩溃再恢复。每种情况检查 manifest 只有一个条目，reduce 只读一个 block，迟到响应被记录但不改变结果。事件日志要能重放整个状态变化。

\topic{六、和锁、原子线性化的对应}
单机原子 CAS 从 Empty 到 Committed，只有一个线程成功；分布式幂等表从 Empty 到 Committed，只有一个 attempt 成功。前者线性化点是 CAS 成功，后者线性化点是 manifest commit 或复制日志提交。尺度不同，结构相同。把这个对应讲清楚，读者会发现并发和分布式不是割裂的两门课。

\section{补充案例：流量控制和重试预算的联动}

重试和流量控制必须一起设计。没有流量控制的重试会放大故障；没有重试的流量控制会让短暂错误直接失败。系统需要同时限制 in-flight 请求、重试次数和重试节奏。

\topic{一、每个 worker 的窗口}
Driver 对每个 worker 维护 in-flight 窗口。窗口满了，不再发送新请求。成功响应释放窗口；超时可能暂时保留或转入 retry 状态；Busy 响应可以降低窗口。窗口防止单个慢 worker 被更多请求压垮。

\topic{二、全局重试预算}
除了单 worker 窗口，还要有 job 级重试预算。某个阶段大量失败时，继续重试可能浪费资源。预算耗尽后，job 进入 Failed 或 Degraded。预算要按错误类型区分，校验失败和 Busy 不应同等处理。

\topic{三、退避和抖动}
多个请求同时超时，如果同时重试，会形成峰值。指数退避和随机抖动能打散流量。退避不是延迟越长越好，太长会增加恢复时间。报告要记录重试时间分布。

\topic{四、实验}
模拟 worker 变慢和 worker 短暂不可用。比较无预算立即重试、固定重试、退避加窗口三种策略。观察队列水位、成功时间、重复请求和失败放大。这个实验让重试从口号变成控制系统。

\section{补充专题：消息大小和批处理}

分布式性能不仅由延迟决定，也由消息大小和批处理决定。小消息太多，协议头、系统调用和调度成本高；大消息太大，尾延迟、内存峰值和重传成本高。批处理是吞吐和延迟的取舍。

\topic{一、控制消息和数据消息分开}
心跳、状态更新、commit ack 是控制消息，应小而及时；shuffle block 是数据消息，可以大而批量。把控制消息塞进大数据队列，会让控制延迟受数据传输影响；把数据拆成过小控制消息，会浪费吞吐。

\topic{二、批处理的边界}
批处理应有最大字节数、最大记录数和最大等待时间。只按字节可能让低流量请求等待太久；只按时间可能产生小包；只按记录数可能忽略记录大小。三个边界共同控制。

\topic{三、失败成本}
批越大，失败重传越贵；批越小，协议开销越高。对于可重算 block，大批可以接受；对于交互请求，小批更合适。报告要写重做字节数和尾延迟。

\topic{四、实验}
模拟不同 message batch size，记录吞吐、p99、重试时重做字节和队列水位。让读者看到批处理不是单纯越大越好。

\section{补充专题：服务降级和部分结果}

分布式系统不总是在成功和失败之间二选一。有些计算可以返回部分结果或降级结果，有些必须失败。服务降级要有语义合同，否则会把错误伪装成成功。

\topic{一、部分结果的条件}
日志统计如果用于审计，不能丢 partition；如果用于实时监控，可能允许返回已完成 partition 并标注缺失比例。合同必须写清输出是否完整。调用者看到结果时，应知道它是 complete、partial 还是 stale。

\topic{二、降级和重试}
降级不等于停止重试。系统可以先返回部分结果，同时后台继续补齐；也可以在预算耗尽后返回 partial。状态机要表达 Degraded，而不是把它当 Completed。Manifest 也要标注缺失。

\topic{三、指标}
降级结果要记录缺失 task、缺失字节、缺失 key range 和错误原因。没有这些指标，用户无法判断结果能否使用。部分结果比失败更需要清楚元数据。

\topic{四、实验}
让一个 reduce partition 永久失败。分别按严格模式返回 job failure，按监控模式返回 partial。比较输出元数据。这样读者会理解可用性提升需要语义支持。

\section{消息系统实现细节：本机 MessageBus 的最小语义}

在真正网络之前，本书项目可以实现本机 MessageBus。它不追求吞吐，而是提供分布式语义：消息复制、身份、延迟、丢失、重复、乱序和关闭。只有模拟器具备这些能力，后续协议测试才有价值。

\topic{一、消息必须复制或拥有 payload}
如果 MessageBus 只是保存指向调用者栈上对象的指针，就不是消息系统。发送后，发送者可能修改或释放对象，接收者看到的不是发送时内容。Bus 应复制 payload，或接管引用计数块所有权。这个边界强迫模块放弃共享内存假设。

\topic{二、故障注入是核心功能}
Bus 应能按规则丢弃请求、丢弃响应、重复投递、延迟投递、乱序投递。规则可按 message type、task id、attempt id 或随机 seed 配置。每次注入都写事件日志。没有故障注入，MessageBus 只是函数调用包装。

\topic{三、关闭语义}
Bus 关闭时，未投递消息如何处理？可以 drain，也可以 cancel。Driver shutdown 和 worker shutdown 要看到明确状态。关闭期间发送新消息应失败。这个语义和本地队列关闭一致，只是对象变成消息。

\topic{四、事件日志}
Bus 记录 send、drop、duplicate、deliver、timeout、close。每条事件带 envelope id 和 logical time。协议测试失败时，事件日志能重放发生了什么。分布式调试高度依赖这种日志。

\topic{五、实验}
用 MessageBus 跑 map attempt。开启响应丢失和重复投递，验证 driver 通过幂等表只提交一次。再开启乱序，验证响应顺序不影响 manifest。这个模拟器是第 18 章项目的核心测试工具。

\section{从一次超时推导分布式系统模型}

分布式系统不应该从 CAP、RPC、租约、逻辑时钟这些概念名词开始。更好的入口是一条普通请求超时：客户端发送任务给 worker，等待超过预算，没有收到响应。此时客户端知道的事实很少：请求可能没有发出，可能已经到达，可能正在队列中，可能已经执行完成但响应丢失，可能 worker 崩溃，可能只是网络慢。分布式系统模型就是为这种“不知道”建立可执行的处理规则。

\topic{一、超时只表达本地等待结束}
超时不是远端失败证明。若客户端把超时直接解释成失败并重新提交，可能重复执行；若把超时解释成成功，又可能丢任务。正确做法是把 Unknown 当成真实状态写进协议。请求状态可以从 Created、Sent、TimedOut、Retrying、Succeeded、Failed、Abandoned 转换，但 TimedOut 不等于 Failed。这个区别会逼出 request id、attempt id、幂等表和迟到响应处理。概念不是凭空加的，而是由超时后的信息缺口逼出来的。

\topic{二、消息身份替代跨进程调用栈}
本地函数调用有调用栈，远程消息没有。消息必须携带自己的身份：request id 表示逻辑请求，attempt id 表示一次尝试，trace id 串起跨服务路径，deadline 表示剩余时间，epoch 表示发送方或路由版本。响应也必须带回这些字段。否则 driver 无法判断响应属于哪个任务、是否已经过期、是否来自旧尝试。身份字段不是日志装饰，而是恢复和去重的最低条件。

\topic{三、幂等来自服务端提交点}
客户端生成 request id 只解决一半问题。服务端必须把 request id、业务状态修改和响应结果放进同一提交边界。若先修改业务状态，再崩溃于记录 request id 前，重试会重复执行；若先记录 request id，再崩溃于业务修改前，重试可能误判成功。文件系统、数据库事务、WAL、状态机日志或 manifest 都是在帮助建立这个提交边界。教学中要用时间线把这些错误讲清，不要把幂等说成“重复调用也没事”这么空泛。

\topic{四、重试预算防止系统自我攻击}
重试可以提高成功率，也能在下游变慢时制造风暴。一个上游请求调用十个下游，每个下游重试三次，尝试数会迅速放大；若所有客户端固定间隔重试，压力会在同一时刻打回故障节点。重试预算应随原始请求传播，deadline 不足时停止重试，退避和抖动用于分散压力，busy 响应用于让上游降速。报告要记录原始请求数、尝试数、超时数、busy 数、最终失败数和队列长度。没有这些数字，重试策略无法评价。

\topic{五、本机模拟要模拟不确定性}
没有真实集群也能训练分布式思维。MessageBus 可以在本机模拟延迟、丢请求、丢响应、重复、乱序、busy 和 worker 崩溃。重点不是追求网络真实带宽，而是让协议状态可控。三个 split 就能覆盖基本语义：一个正常完成，一个响应丢失后重试，一个 worker 慢到 deadline 过期。验收时检查 manifest 是否只提交一次、迟到响应是否被识别、幂等表是否返回同一结果、重试预算是否阻止放大。这种小实验比直接搭复杂集群更适合讲原理。

\section{消息系统的最小实现剖面}

分布式模型要避免只讲概念，也要给出本机 MessageBus 的实现剖面。这个剖面至少包含 Envelope、Endpoint、Inbox、NetworkRule、Timer、RetryPolicy、IdempotencyTable 和 TraceLog。Envelope 保存身份和 payload；Endpoint 表示 driver 或 worker；Inbox 是有界队列；NetworkRule 注入丢弃、重复、延迟和乱序；Timer 触发 deadline 和重试；RetryPolicy 决定 attempt；IdempotencyTable 记录提交结果；TraceLog 让跨端调试可复盘。

\topic{一、发送不是直接调用接收函数}
如果 send 直接调用 receiver 的处理函数，本机模拟会重新获得本地调用栈，失去分布式不确定性。更好的做法是 send 把 Envelope 复制或移动到网络层队列，网络层按规则决定何时投递、是否丢弃、是否重复。这样即使在一个进程里，driver 也不能知道 worker 何时处理，worker 也不能依赖 driver 的调用栈。模拟的价值就在于打破本地直觉。

\topic{二、有界 Inbox 让 busy 成为协议}
每个 endpoint 的 Inbox 应有容量。满了以后，网络层可以返回 busy、丢弃低优先级消息或延迟投递。busy 必须进入响应分类，让发送方减少并发或停止重试。若 Inbox 无限增长，模拟永远不会暴露过载，读者会误以为重试没有成本。有界队列把资源限制带进分布式协议，是背压章节在消息系统里的延续。

\topic{三、Timer 驱动状态而不是睡眠}
超时不应靠业务线程 sleep 后猜测，而应由 Timer 根据 deadline 或 \code{retry_at} 产生事件。事件到来时，状态机检查 request 是否已经完成、是否还有预算、是否应发送新 attempt。若迟到响应先到，Timer 事件应发现请求已完成并忽略。这样超时和响应成为两个竞争事件，正好训练读者理解分布式不确定性。

\topic{四、幂等表保存结果而不只是保存见过的 ID}
服务端收到重复 request id 时，应返回同一语义结果。若幂等表只记录“见过”，却不保存结果，客户端重试后无法得到之前响应。表项应包含 request id、状态、结果摘要、错误分类、提交时间和过期策略。过期太早会让迟到重试重复执行，过期太晚会占用内存。过期策略要和业务 deadline、重试窗口和恢复日志一起设计。

\topic{五、TraceLog 替代跨进程调试栈}
每个事件记录 trace id、request id、attempt id、endpoint、状态、时间和原因。调试响应丢失时，TraceLog 应能串起发送、投递、执行、提交、响应丢弃、超时、重试、重复命中幂等表、最终完成。没有 TraceLog，分布式 bug 只剩“偶尔超时”。有了 TraceLog，读者能把不确定性拆成事件序列。

\section{故障矩阵：从事件组合推导处理规则}

分布式章节最后还要训练读者写故障矩阵。单个概念讲清楚还不够，因为真实错误往往是事件组合：请求丢失加上客户端重试，响应丢失加上服务端已提交，worker 慢加上 deadline 过期，driver 重启加上旧 attempt 迟到。故障矩阵的作用是把这些组合列出来，逐格写处理规则，而不是等 bug 出现后临时解释。

\topic{一、请求未到达}
客户端发送后，消息在网络层被丢弃，worker 没有任何记录。客户端超时后用同一 request id 新建 attempt。服务端看到的是第一次到达的请求，正常执行。这个格子的关键是：客户端不能因为“上一 attempt 没响应”就换 request id，否则后续去重会失效。TraceLog 应记录第一次 attempt 发送和丢弃，第二次 attempt 投递和执行。

\topic{二、请求到达但未执行}
消息进入 worker inbox，但 worker 忙或进程崩溃，业务逻辑未运行。客户端仍然只能看到超时。重试到另一个 worker 时，如果分区归属允许，新 worker 可以执行；如果同一分区只能由原 owner 执行，就要等待归属切换。这个格子把分布式模型和分区归属连接起来：能不能换 worker，不由客户端心情决定，而由 ownership epoch 决定。

\topic{三、执行完成但响应丢失}
这是幂等表的核心场景。worker 已经提交输出和结果记录，响应被丢弃。客户端重试后，worker 或新 owner 应返回旧结果，而不是再次执行副作用。若幂等表只在内存中，worker 崩溃后会丢失去重信息；若幂等表和输出提交不在同一边界，也可能重复。这个格子要求读者检查提交点，而不是只检查重试代码。

\topic{四、响应迟到且客户端已放弃}
deadline 过期后，客户端可能已经向用户返回失败或启动新的计划。迟到响应不能随便写入最终状态。状态机要检查 request 是否仍然活跃、attempt 是否仍然属于当前 epoch、结果是否已经提交。若请求已 abandoned，迟到响应只能记录并释放资源。这个格子帮助读者理解取消不是远端事实，只是本地不再等待。

\topic{五、driver 重启后的恢复}
driver 崩溃重启后，内存中的 in-flight 表消失，但 manifest、幂等表或任务日志仍在。恢复时要重建 request 状态：已提交的不再执行，未提交的重新排队，状态未知的按 request id 查询或重试。若没有持久化任务身份，driver 只能重新全量执行，可能产生重复输出。故障矩阵把“持久化最小集”逼出来：输入 split、request id、attempt、输出路径、提交状态和 checksum。

习题四：用本机多线程模拟消息乱序和重复。要求每条消息携带 request id 和 attempt，服务端能识别重复请求并返回同一结果。

习题五：比较数据本地调度和远程调度。给出本地排队长、远程网络快以及本地网络快、远程排队短两种场景，说明调度器应如何取舍。
