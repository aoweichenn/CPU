\chapter{分布式计算模型}

上一章之前，我们一直在单机内部讨论计算。线程共享同一进程地址空间，任务队列里放的是一个 C++ 对象，worker 处理完以后可以直接修改同一块内存，主线程等待 future 或 join 就能拿到结果。现在把同一个日志统计作业拆到多台 worker 上，最先消失的不是某个 API，而是“调用栈会回来”这个假设。Driver 不能把一个 lambda 指针发给另一台机器，也不能把本地引用交给远端。它只能发一条消息，然后等待另一条消息。消息可能晚到、重复、丢失，也可能在远端已经执行完以后只丢失响应。

本章不从 CAP、RPC、Quorum 这些词开始。它们当然重要，但如果读者还没有亲眼看到一个普通任务怎样在网络和故障下变形，这些词只会变成概念清单。我们从一个固定案例出发：一个 driver，三个 worker，一个本机模拟的 \code{MessageBus}，输入日志被切成多个 split。Driver 发送 \code{MapTaskRequest}，worker 读取 split，按 event type 做本地计数，把临时输出写到自己的工作目录，再返回 \code{MapTaskReply}。第一版完全可以在同一个进程里跑，但必须按分布式语义设计：driver 和 worker 不共享业务状态，所有交互都通过消息完成，MessageBus 可以延迟、丢弃、重复和乱序投递消息。

这个小系统已经足够暴露分布式计算的核心困难。只要响应可能丢失，超时就不能证明 worker 没有执行；只要请求可能重复，输出提交就必须幂等；只要响应可能乱序，driver 就必须区分旧 attempt 和新 attempt；只要 worker 队列可能满，异步发送就必须有背压；只要 driver 可能重启，task table 和 commit table 就不能只留在内存里。分布式系统不是把单机程序搬到网络上，而是把原来隐含在调用栈、共享内存和进程生命周期里的边界全部写成协议。

\begin{keyidea}
分布式系统的第一原则不是“使用网络”，而是承认本地调用栈不再能表达远端执行的完成、失败、重复和恢复。协议字段、状态表、提交点和指标，都是为了把这些失去的保证显式补回来。
\end{keyidea}

\section{从一次日志统计作业出发}

先把任务说具体。输入是一批日志文件，每行包含时间戳、用户 ID 和事件类型。目标是统计每种事件出现次数。串行 reference 很简单：按顺序读所有日志，解析 event type，用一个 map 或数组累加，最后输出按 event type 排序的计数。这个 reference 不追求性能，它只定义语义：哪些输入算有效，解析失败如何计数，每条记录应该贡献几次，最终结果怎样比较。

在单机线程池版本里，driver 可以把 split 描述和函数对象放入队列。Worker 线程执行函数，直接把 partial 写进共享数组，最后主线程合并。这里的很多保证来自同一进程：对象地址有效，异常能沿调用栈传播，内存生命周期由同一个运行时控制，进程退出意味着所有 worker 一起消失。即使并发代码有锁、原子和队列成本，通信对象仍然是同一台机器上的地址。

把 worker 看成另一台机器后，这些前提全部失效。Driver 不能发送本地指针，因为对端没有相同地址空间；不能发送捕获了引用的 lambda，因为捕获对象只在本进程存在；不能让 worker 直接写 driver 的 vector，因为那块内存不在 worker 上。Driver 只能发送一份可序列化的请求：哪个 job，哪个 stage，哪个 task，哪个 input split，当前是第几次 attempt，deadline 是什么时候，输出应该提交到哪里。Worker 收到请求后，重新定位输入，执行任务，写临时输出，再用响应报告结果摘要。

\begin{lstlisting}[language=C++,caption={日志统计作业的输入和请求语义}]
struct InputSplit {
    std::uint64_t split_id = 0;
    std::uint64_t file_id = 0;
    std::uint64_t begin_offset = 0;
    std::uint64_t end_offset = 0;
};

struct MapTaskRequest {
    std::uint64_t job_id = 0;
    std::uint64_t stage_id = 0;
    std::uint64_t task_id = 0;
    std::uint32_t attempt = 0;
    std::uint64_t request_id = 0;
    InputSplit split;
    std::uint32_t target_partition_count = 0;
    std::uint64_t deadline_milliseconds = 0;
};

struct MapTaskReply {
    std::uint64_t request_id = 0;
    std::uint64_t task_id = 0;
    std::uint32_t attempt = 0;
    std::uint32_t status = 0;
    std::uint64_t output_records = 0;
    std::uint64_t output_bytes = 0;
    std::uint64_t output_checksum = 0;
};
\end{lstlisting}

这段代码不是为了展示网络库，而是为了把语义字段摆到桌面上。\code{task_id} 是逻辑任务身份，同一个 input split 重试时不变；\code{attempt} 是物理执行身份，每次重试递增；\code{request_id} 是一次消息交互身份，用于匹配请求和响应；\code{deadline_milliseconds} 是这次请求的预算；\code{output_checksum} 让 driver 不必只相信一句“我完成了”。如果这些字段缺失，后面的失败路径都会变成猜谜。

\topic{为什么不能只发函数名}

初学者常说：driver 发一个 \code{runMap(splitId)} 不就行了吗？这个想法的问题不在于函数名错误，而在于函数名只描述“做什么”，没有描述“这次执行是谁”。当响应回来时，driver 需要判断它属于哪个 job、哪个 stage、哪个 task、哪个 attempt，是否已经过期，是否来自旧路由，是否能提交结果。函数名无法回答这些问题。

本地函数调用把身份藏在调用栈里。调用者进入函数体，函数体返回到调用者，返回值天然对应这一次调用。远程调用没有这条栈。请求和响应在不同线程、不同队列、不同机器上流动，中间可能经过重试、超时、排队和连接复用。响应到达时，driver 看到的只是一包字节。没有身份字段，它无法知道这包字节应该改变哪个状态。

所以消息头不是装饰，它是分布式系统的控制面。它携带的不是“日志上好看”的信息，而是协议判断需要的信息：这条响应是否重复，是否属于当前 attempt，是否还能提交，是否应该触发重试，是否应该刷新路由。很多分布式 bug 不是用户函数算错，而是消息身份不清导致旧响应覆盖新状态、重复 attempt 重复提交、超时请求在后台继续写入。

\topic{从内存对象到 wire format}

上面的 C++ 结构体是教学模型，不等于真实 wire format。真实网络协议要定义字节序、字段长度、版本、校验、payload 长度和兼容策略。不能把一个 C++ struct 的内存直接写到 socket，因为 padding、对齐、大小端和 ABI 都可能不同。同一段代码在不同编译器、不同平台或不同编译选项下，结构体布局可能变化。协议格式必须按字节定义，而不是按当前进程的内存布局定义。

教学项目可以先用进程内对象模拟消息，但文字和接口要保留清楚边界：\code{MapTaskRequest} 表示逻辑消息，MessageBus 内部可以直接移动对象；如果未来改成 socket，必须加入编码层，把逻辑消息转成稳定字节流。这样做能避免一个常见误解：本机模拟不是因为分布式系统“本质上就是函数调用”，而是为了在单机上先验证协议语义。

\begin{lstlisting}[language=C++,caption={协议头部应表达的稳定字段}]
struct WireHeader {
    std::uint16_t protocol_version = 1;
    std::uint16_t message_type = 0;
    std::uint32_t flags = 0;
    std::uint64_t request_id = 0;
    std::uint32_t payload_bytes = 0;
    std::uint32_t checksum = 0;
};
\end{lstlisting}

协议版本看似琐碎，实际很关键。分布式系统升级通常不是所有节点同时重启。某些 worker 可能已经运行新版本，某些 worker 还在旧版本。新增字段要有默认值，旧节点要能忽略未知字段，新节点要能读旧消息。教学项目可以暂时要求同版本运行，但教材必须说明这是简化，不是工程事实。

\topic{输入 split 的边界不是随便切}

日志统计作业里，input split 也不能写得太抽象。如果按文本文件的字节范围切分，split 的 begin offset 可能落在一行中间。非零 begin 应先跳到下一行起点，end 可以读到当前行结束。否则两个 worker 可能重复读一行，或者都跳过同一行。分布式系统中很多“结果偶尔不对”的 bug，就来自这些被当作小事的输入边界。

二进制格式也类似。固定长度记录可以按记录大小对齐；变长记录需要块目录、索引或同步标记。Split 元数据要持久化，因为失败后重试必须读取同一个逻辑范围。若重试时重新按当前文件大小切分，输入发生变化就会破坏 reference 语义。分布式计算的正确性从输入合同开始，不是从 RPC 开始。

\topic{身份字段的作用域}

\code{job_id}、\code{stage_id}、\code{task_id}、\code{attempt}、\code{request_id} 不是越多越好，而是各自回答不同问题。Job 表示一次完整作业；stage 表示一组依赖相同输入的任务；task 表示一个逻辑输入 split 或 reduce partition；attempt 表示某个 task 的一次物理执行；request 表示一次消息发送。若把它们混成一个 ID，代码短一些，但恢复、去重和观测会变得混乱。

作用域也要说清。\code{task_id} 可以只在一个 stage 内唯一，也可以在整个 job 内唯一；\code{request_id} 可以由 driver 全局递增，也可以按 worker 分配范围；\code{attempt} 通常只在 task 内有意义。日志和提交表必须能从字段组合唯一定位对象。例如提交表按 \code{job_id, stage_id, task_id} 去重，而不是按 \code{request_id} 去重。因为响应丢失后重试会产生新的 \code{request_id}，但逻辑任务还是同一个。

\topic{本章固定拓扑}

本章后面所有机制都围绕同一个拓扑展开。Driver 维护 task table 和 commit table。Worker 0、Worker 1、Worker 2 各有接收队列、执行线程和临时输出目录。MessageBus 负责把 driver 请求放到目标 worker 队列，把 worker 响应放回 driver 队列。为了训练协议思维，MessageBus 支持四个故障开关：延迟某条消息，丢弃某条消息，重复投递某条消息，改变响应到达顺序。

\begin{lstlisting}[numbers=none]
Driver
  task table: task_id -> state, attempt, request_id, deadline
  commit table: task_id -> committed attempt, output checksum
  send MapTaskRequest to MessageBus

MessageBus
  delay / drop / duplicate / reorder
  enforce bounded queues and in-flight windows

Worker
  receive request
  execute split
  write temporary output
  send MapTaskReply
\end{lstlisting}

这个拓扑很小，但它已经覆盖分布式模型最重要的失败：请求可能没有到达，响应可能没有回来，worker 可能执行完却被 driver 当作超时，多个 attempt 可能同时存在，队列可能积压，driver 可能崩溃后丢失内存状态。后面所有概念都从这些路径里推出来。

\section{Driver 状态表和正常路径}

单机函数调用依赖调用栈保存状态。函数还没返回时，调用者的栈帧还在；函数返回后，返回值进入调用者；异常沿栈传播。分布式系统没有一条跨机器的调用栈，所以 driver 必须把“我发了什么、等什么、当前能接受什么响应”写成显式状态表。没有状态表，正常路径看起来能跑，故障路径一来就会失控。

状态表不能由一堆 bool 拼起来。比如 \code{sent=true}、\code{done=false}、\code{retrying=true}、\code{failed=false} 看似灵活，实际会出现很多非法组合。一个 task 不应该同时是 committed 和 retrying，也不应该在没有 request id 的情况下处于 running。用单一枚举表达主状态，再配合 attempt、deadline 和 retry count，代码更容易审查。

\begin{lstlisting}[language=C++,caption={Driver 端任务状态}]
enum class DriverTaskState : std::int32_t {
    Pending,
    Sent,
    Running,
    Retrying,
    Committed,
    Failed,
};

struct DriverTaskEntry {
    std::uint64_t task_id = 0;
    std::uint32_t current_attempt = 0;
    std::uint64_t current_request_id = 0;
    DriverTaskState state = DriverTaskState::Pending;
    std::uint64_t deadline_milliseconds = 0;
    std::uint32_t retry_count = 0;
    std::uint32_t assigned_worker = 0;
};
\end{lstlisting}

\code{Pending} 表示还没发出请求；\code{Sent} 表示请求已经进入 MessageBus，但未必到达 worker；\code{Running} 表示 worker 已确认接收，或者 driver 根据发送记录认为它可能在执行；\code{Retrying} 表示当前 attempt 已经超过本地等待预算，但逻辑 task 尚未失败；\code{Committed} 表示某个 attempt 的输出已经被接受；\code{Failed} 表示预算耗尽或遇到不可重试错误。每个状态都要有允许的进入和退出路径，不能靠注释猜。

\topic{正常路径先画清楚}

正常路径不是为了浪费篇幅。只有正常路径的边界画清楚，故障路径才知道在哪里插入。下面是一条没有丢包、没有慢节点、没有重试的时间线。

\begin{lstlisting}[numbers=none]
t0 driver: task 17 is Pending
t1 driver -> worker1: request_id 8001, task 17, attempt 0
t2 driver: task 17 is Sent
t3 worker1: receive request 8001 and start split 17
t4 worker1: write temp output task17_attempt0
t5 worker1 -> driver: reply request 8001, checksum C
t6 driver: verify reply and commit task17_attempt0
t7 driver: task 17 is Committed
\end{lstlisting}

这条时间线暴露出三个提交边界。第一，driver 发出请求不等于 worker 收到请求；第二，worker 写出临时输出不等于结果对作业可见；第三，driver commit 之后，迟到的其他 attempt 不能再改变结果。朴素程序常把这三件事混在一起：发出请求就认为任务在跑，worker 写文件就认为结果有效，响应回来就覆盖状态。正常路径里这些偷懒不一定出错，故障路径里一定会出错。

\topic{状态转换要由事件驱动}

Driver 不是每隔一会儿“猜一猜”任务状态，而是根据事件转换状态。事件包括：请求入队成功，worker 接收确认，deadline 到达，响应到达，commit 成功，commit 失败，worker busy，driver 恢复 checkpoint。每个事件都要检查当前状态是否允许处理。若一个已经 committed 的 task 又收到响应，它不是“再成功一次”，而是 late reply。

\begin{lstlisting}[numbers=none]
event: send accepted by MessageBus
  Pending -> Sent

event: worker accepted request
  Sent -> Running

event: deadline reached
  Sent or Running -> Retrying

event: valid reply committed
  Sent or Running or Retrying -> Committed

event: retry budget exhausted
  Retrying -> Failed
\end{lstlisting}

这个状态机不是形式主义。它能让代码审查有抓手。看到一个响应处理分支时，可以问：当前 task 若已 committed 会怎样？若 attempt 不匹配会怎样？若 response 的 request id 不是当前 request 会怎样？若 commit table 已有记录会怎样？没有状态机，这些问题会散落在许多 if 里，最后变成线上偶发错误。

\topic{响应处理的检查顺序}

响应到达时，driver 不能先处理输出，再更新状态。正确顺序应该先确认身份和状态，再验证输出，再提交。第一步查 \code{task_id} 是否存在；第二步查 task 是否已经 \code{Committed}；第三步查响应 attempt 是否还能被接受；第四步查 request id 是否匹配当前 attempt 或可接受的历史记录；第五步验证输出摘要；第六步进入 commit table。

\begin{lstlisting}[numbers=none]
on reply:
  task = task_table.find(reply.task_id)
  if task does not exist:
      record unknown_reply
      return
  if task.state == Committed:
      record late_reply
      return
  if reply.attempt != task.current_attempt:
      record stale_attempt
      return
  if reply.request_id != task.current_request_id:
      record stale_request
      return
  verify output checksum
  commit output for task_id
\end{lstlisting}

这个顺序的意义是把旧消息挡在业务效果之外。很多系统出错，是因为先把输出加入结果，再发现状态不对。提交前的检查和提交后的审计要分开。迟到响应可以记录、可以触发清理临时文件、可以用于诊断慢节点，但不能再改变业务结果。

\topic{指标也是状态表的一部分}

状态表不是只服务正确性，也服务观测性。Driver 至少要记录每个 task 的发送时间、开始时间、完成时间、attempt 数、最后错误、所在 worker、输出字节和 checksum。否则 stage 变慢时，只能知道“某些任务慢”，不知道是排队慢、执行慢、响应慢、重试多、还是 commit 慢。

\begin{lstlisting}[language=C++,caption={教学项目的任务指标}]
struct TaskMetrics {
    std::uint64_t task_id = 0;
    std::uint32_t attempts_started = 0;
    std::uint32_t timeouts = 0;
    std::uint32_t late_replies = 0;
    std::uint32_t stale_replies = 0;
    std::uint64_t first_send_milliseconds = 0;
    std::uint64_t commit_milliseconds = 0;
    std::uint64_t output_bytes = 0;
};
\end{lstlisting}

分布式系统的可维护性来自 ID 和指标的统一。日志里出现 \code{task_id=17 attempt=1 request_id=8002}，指标里也要能按这些字段聚合。没有共同 ID，driver、worker、MessageBus 的日志只是三堆时间戳。出了问题以后，读者无法还原因果关系。

\topic{本地时间和单调时间}

超时测量应使用单调时间，而不是 wall clock。系统时间可能被 NTP 调整，向前或向后跳；单调时间适合计算 elapsed time。跨机器传递 deadline 时，物理时钟差异还要保守处理，但在本机模拟里，至少要把“日志时间戳”和“超时计时”分开。日志时间用于排查，单调时间用于预算。

教学项目可以把 \code{deadline_milliseconds} 定义为模拟时钟上的绝对时间点。Driver 发送请求时记录当前模拟时间加预算；MessageBus 和 worker 在处理前检查是否已经过期。这样做比每一层重新设置 timeout 更清楚。Timeout 是相对等待时长，deadline 是绝对截止点。多层系统里，deadline 更容易表达“整个操作还剩多少预算”。

\topic{正常路径的第一组测试}

第一组测试不注入故障，只验证协议骨架。给三个 input split，三个 worker，MessageBus 不丢包不乱序。期望每个 split 生成一个 \code{MapTaskReply}，commit table 有三条记录，最终结果和串行 reference 一致。测试还要检查指标：\code{attempts_started} 全部为 1，\code{timeouts} 为 0，\code{late_replies} 为 0，最大 in-flight 不超过配置。

这组测试看似简单，但它防止后续故障逻辑污染正常路径。很多系统为了处理异常路径，正常路径被写得难以理解。高质量教材应该让读者看到：先有清晰正常路径，再逐个加入失败条件，每次只新增一个机制。

\section{响应丢失与幂等提交}

第一个故障是响应丢失。Worker 已经执行完 split，写出了临时输出，也发送了响应，但响应在 MessageBus 中被丢弃。Driver 等到 deadline 还没有收到响应，只能看到一个事实：本地等待时间到了。它不能证明 worker 没收到请求，也不能证明 worker 没执行，更不能证明 worker 的临时输出不存在。

\begin{lstlisting}[numbers=none]
t0 driver -> worker1: request 8001, task 17, attempt 0
t1 worker1: execute split 17
t2 worker1: write temp output task17_attempt0
t3 worker1 -> driver: reply 8001 is dropped
t4 driver: deadline reached, task 17 enters Retrying
t5 driver -> worker2: request 8002, task 17, attempt 1
t6 worker2: execute split 17
t7 worker2 -> driver: reply 8002 arrives
t8 driver: commit attempt 1
t9 worker1 cleanup or late reply appears
\end{lstlisting}

这条时间线说明一个核心事实：\important{超时是本地观察，不是全局事实}。Driver 在 t4 只能说“我没有等到可接受响应”，不能说“attempt 0 没有执行”。如果 reduce 端已经读取了 attempt 0 的输出，或者 worker1 的迟到响应又触发一次提交，最终计数就会重复。日志统计看似只是加法，一旦同一个 split 被加两次，结果就错了。

\topic{朴素修复为什么不够}

朴素修复一是“超时后删掉旧输出”。问题是旧输出在 worker1 上，driver 未必能访问；worker1 可能仍在执行；删除请求也可能丢失；甚至删除本身可能和旧 attempt 写文件并发。朴素修复二是“worker 收到取消就停”。取消只能尽力而为，不能保证已经执行的副作用消失。朴素修复三是“把 timeout 调大”。这只能减少触发概率，不能改变语义。

正确方向是承认重复 attempt 可能存在，然后让外部效果只发生一次。对本章任务来说，外部效果是“某个 \code{task_id} 的输出进入最终 manifest 或 reduce 输入集合”。Worker 写临时文件不是外部效果，driver 把某个 attempt 记入 commit table 才是外部效果。只要 commit table 按逻辑 task 去重，attempt 0 和 attempt 1 可以都执行，但最终只能接受其中一个。

\topic{提交表的最小形态}

\begin{lstlisting}[language=C++,caption={按逻辑任务去重的提交记录}]
struct CommitRecord {
    std::uint64_t job_id = 0;
    std::uint64_t stage_id = 0;
    std::uint64_t task_id = 0;
    std::uint32_t committed_attempt = 0;
    std::uint64_t output_bytes = 0;
    std::uint64_t output_checksum = 0;
    bool committed = false;
};
\end{lstlisting}

提交表按 \code{job_id, stage_id, task_id} 去重，而不是按 \code{request_id} 去重。因为响应丢失后，attempt 1 会有新的 request id；按 request id 去重只能防止同一消息重复处理，不能防止同一逻辑任务被不同 attempt 重复提交。提交表也不应该只保存一个 bool。它要保存选择了哪个 attempt、输出大小和 checksum，这些信息用于恢复和审计。

\topic{提交点必须原子}

幂等提交不是“先查一下表”这么简单。假设 driver 先把 attempt 1 的输出加入 manifest，然后在写 commit record 前崩溃。恢复后，commit table 看不到 task 17 已提交，可能再次接受 attempt 0 或 attempt 2，导致重复结果。反过来，若先写 commit record，再写 manifest 前崩溃，恢复后可能认为 task 已完成，却找不到有效输出。

所以幂等记录和业务结果必须处在同一个提交边界里。工程上可以使用数据库事务、WAL、原子 rename、两阶段 manifest 或共识日志。教学项目可以用本地文件模拟：先把新 manifest 写到临时文件，包含 commit record 和输出条目，fsync 后原子 rename 为当前 manifest。只要恢复时只读取完整 manifest，就不会把半提交状态当成成功。

\begin{lstlisting}[numbers=none]
safe commit outline:
  read current committed manifest
  if task_id already committed:
      reject duplicate attempt
  verify temporary output checksum
  append commit record and output metadata to new manifest file
  flush new manifest file
  atomically publish new manifest
  mark task as Committed in memory
\end{lstlisting}

这里的关键不是文件 API 细节，而是顺序。内存状态最后更新，因为内存状态崩溃后会丢。可恢复事实必须先写到持久化提交边界里。很多教材讲“幂等”时只说重复执行结果相同，却没有讲提交边界，这会让读者在真正写系统时踩坑。

\topic{临时输出和最终输出}

Worker 不应直接写最终输出路径。每个 attempt 写自己的临时路径，例如 \code{job7/stage2/task17/attempt0/part0.tmp}。Driver 提交后，manifest 指向被接受的 attempt。其他 attempt 即使文件存在，也不是有效结果。Reduce 端不应该扫描目录猜哪些文件可读，而应该只读取 manifest 中列出的 block。

这种设计把“文件存在”和“结果有效”分开。文件存在只是数据面事实，manifest 才是控制面事实。分布式计算系统里，控制面决定哪些数据属于最终结果，数据面负责搬运和保存字节。把这两件事混在一起，旧 attempt 残留文件、半写文件、损坏文件都可能被误读。

\topic{worker 端是否也要去重}

Driver 端 commit table 保证最终结果只提交一次，但 worker 端仍然可能重复执行同一 request。是否要在 worker 端去重，要看副作用在哪里。如果 worker 只写 attempt 临时文件，重复执行成本高但不破坏最终结果；如果 worker 会扣减配额、写外部系统、发送下游消息，就必须在 worker 端也做幂等。对于本章教学项目，worker 端可以先不跨重启保存去重表，但要记录重复 request 指标。

更成熟的设计会把 request id、执行结果和响应摘要保存在 worker 的本地去重表里。重复 request 到达时，worker 不重新执行，而是返回旧响应或报告 duplicate。这个表同样需要回收。若客户端可能在一分钟内重试，worker 至少要保留一分钟；若去重记录太早删除，迟到重试仍可能重复执行。幂等语义是客户端和服务端共同的合同。

\topic{响应丢失实验}

响应丢失实验要有朴素版和修复版。朴素版让 MessageBus 固定丢弃 request 8001 的响应。Driver timeout 后向 worker2 派发 attempt 1。若系统没有 commit table，最终可能把两个 attempt 的输出都纳入结果，reference 对比会失败。修复版加入 commit table 和 manifest，只接受第一个通过校验并成功提交的 attempt，迟到响应只增加 \code{late_reply_count}。

验收条件不能只写“结果正确”。至少要检查：最终每个 \code{task_id} 只有一个 commit record；重复或迟到响应不会改变 commit record；manifest 中没有半写临时文件；最终计数与串行 reference 一致；指标中能看到一次 timeout、一次 retry、一次 late 或 duplicate 事件。这样读者才能知道机制真的被触发，而不是测试没有覆盖故障。

\section{超时、重试和乱序响应}

第二组故障来自超时和重试。响应丢失会触发重试，worker 慢也会触发重试，MessageBus 延迟也会触发重试。重试本身不是坏事，它让系统能从临时失败中恢复。但没有预算、退避和幂等的重试，会把小故障放大成系统过载。

先看最朴素策略：driver 等到 timeout，立刻再发一次；再次 timeout，再发一次；直到成功。这种策略在单元测试里很容易通过，因为测试规模小、worker 很快、消息不拥塞。但真实系统里，timeout 往往正是因为下游慢或队列长。此时所有 driver 都立刻重试，会让下游队列更长，响应更慢，更多请求 timeout，形成重试风暴。

\topic{deadline 比每层 timeout 更清楚}

一个用户操作或一个 stage 应该有总预算。Driver 发送 request 时，把剩余预算作为 deadline 传下去，而不是让每一层重新设置完整 timeout。假设 stage 希望在 30 秒内完成，某个 task 已经等待 28 秒，重试 attempt 不应再拿到完整 30 秒预算。否则局部 timeout 会让整体时间失控。

Deadline 也能让 worker 做更合理的判断。Worker 收到请求时，发现 deadline 只剩很短时间，可以拒绝或快速返回 busy，而不是开始一个注定来不及完成的计算。这样可以减少无效工作。注意，deadline 不等于取消。Worker 看到 deadline 过期后可以停止新工作，但已经完成的提交仍要按幂等规则处理。

\begin{lstlisting}[language=C++,caption={重试预算和 deadline 的教学结构}]
struct RetryPolicy {
    std::uint32_t max_attempts = 3;
    std::uint64_t initial_backoff_milliseconds = 20;
    std::uint64_t max_backoff_milliseconds = 1000;
    std::uint64_t jitter_milliseconds = 10;
};

struct TaskDeadline {
    std::uint64_t absolute_deadline_milliseconds = 0;
};
\end{lstlisting}

这里没有写随机数实现，是因为书中重点是语义。指数退避让连续失败后的等待增长，抖动让大量 driver 不在同一时刻重试。最大 attempt 数限制重复执行规模，绝对 deadline 限制整体等待。重试策略不是“可靠性增强开关”，而是控制系统压力的机制。

\topic{错误分类决定能否重试}

不是所有失败都能重试。Worker 返回 input split 不存在，说明元数据或输入损坏，换个 worker 也不一定能解决；返回 busy，说明容量不足，应该退避或换 worker；返回 stale epoch，说明路由版本过旧，应该刷新控制面；返回 temporary I/O error，可以在预算内重试；返回 invalid request，应该停止并暴露 bug。

\begin{lstlisting}[language=C++,caption={面向行为的任务状态码}]
enum class TaskStatus : std::int32_t {
    Ok,
    Busy,
    RetryableIoError,
    InvalidSplit,
    StaleEpoch,
    DuplicateRequest,
    DeadlineExceeded,
    InternalError,
};
\end{lstlisting}

错误码不是日志装饰，而是协议的一部分。Driver 根据错误码决定退避、刷新路由、换 worker、停止重试还是失败 task。把所有失败都写成 \code{InternalError}，等于让上层只能猜。教材里的错误分类不用追求工业系统那么细，但必须能表达行为差异。

\topic{重试是新 attempt，不是删除旧 attempt}

当 driver 决定重试时，它启动的是另一个 attempt。它不能宣布旧 attempt 不存在。旧 attempt 可能还在 worker 队列里，可能正在执行，可能已经写完临时输出，只是响应还没到。新 attempt 和旧 attempt 可以同时存在，所以状态表必须允许“逻辑 task 未提交，但多个物理 attempt 曾经启动过”。

这也是 \code{task_id} 和 \code{attempt} 分离的原因。\code{task_id} 保护逻辑结果只提交一次；\code{attempt} 帮助区分执行历史。若每次重试都生成新的 task id，commit table 就无法知道这些 attempt 来自同一个 input split。若不递增 attempt，driver 又无法判断哪个响应是旧执行。

\topic{乱序响应}

现在看第二条时间线。Attempt 0 先发出，超时后 attempt 1 发出。Attempt 1 的响应先到并成功提交，随后 attempt 0 的响应迟到。

\begin{lstlisting}[numbers=none]
t0 request 8001, task 17, attempt 0 -> worker1
t1 deadline reached, driver starts attempt 1
t2 request 8002, task 17, attempt 1 -> worker2
t3 reply 8002 arrives, driver commits attempt 1
t4 reply 8001 arrives late
\end{lstlisting}

朴素 driver 若按“最后到达覆盖状态”更新，t4 的旧响应会污染已提交结果。正确规则是：\important{已提交任务不能被迟到 attempt 改写}。迟到响应可以记录为 late reply，可以用于清理 attempt 0 的临时输出，但不能再进入 commit table。

乱序不仅发生在不同 worker 之间。即使使用 TCP，单连接内字节顺序是有序的，应用层多连接、多队列、多线程处理仍可能让完成顺序变化。一个服务端线程先处理后到请求，另一个线程后处理先到请求，响应完成顺序就变了。协议如果依赖顺序，就必须显式携带序号、版本或 attempt，而不是相信到达顺序。

\topic{旧路由和 epoch}

重试和乱序在后续分片迁移中还会遇到 epoch。假设 task 17 原本分配给 worker1，路由 epoch 为 5。Driver 超时后控制面把该 partition 迁到 worker2，epoch 变成 6。Worker1 恢复后带着 epoch 5 的旧响应回来，driver 必须拒绝它对新状态的修改。Epoch 是 fencing token，用来阻止旧持有者继续写。

本章不展开完整再分片，那是下一章的主题。但第16章需要先建立直觉：分布式系统里的“旧”不等于“已经死”。旧 worker、旧 leader、旧 attempt、旧路由都可能在网络恢复后重新出现。协议必须让旧消息没有破坏力，而不是希望它们永远不回来。

\topic{重试风暴的具体数字}

设想 1000 个 task 同时发往 worker 集群。Worker 因磁盘抖动，平均响应从 50 ms 变成 800 ms。Driver timeout 设置为 100 ms，且超时后立即重试。100 ms 时，第一批 1000 个请求大多还在执行；driver 又发 1000 个；200 ms 再发 1000 个。几轮以后，系统里同时存在数千个 attempt，worker 队列更长，磁盘更忙，真正能完成的请求反而更少。

加入预算后，行为应该不同。每个 task 最多 3 个 attempt；每次 retry 使用退避和抖动；每个 worker 有 in-flight 上限；收到 busy 后不会立即重试同一个 worker。这样做可能让某些 task 更早失败，但系统不会被无界重复请求拖垮。分布式系统里，明确失败常常比无界等待更健康。

\begin{lstlisting}[numbers=none]
retry allowed only if:
  task is not committed
  error is retryable or local deadline expired
  retry_count < max_attempts
  job deadline still has remaining budget
  target worker or alternative worker has capacity
  retry budget for the job is not exhausted
\end{lstlisting}

\topic{请求取消的边界}

Driver 超时后可以向 worker 发送 cancel，但 cancel 不能作为正确性基础。Cancel 消息可能丢失，worker 可能已经执行完，worker 可能在不可中断的 I/O 中，或者 cancel 到达时旧 attempt 已经写出临时输出。正确系统应该把 cancel 当作资源回收优化，而不是当作避免重复提交的机制。

这点和单机异步 I/O 很像。调用者超时后释放上下文，但底层完成回调可能稍后到达。若没有安全生命周期管理，就会访问已释放对象。分布式系统中，迟到完成不仅可能访问旧对象，还可能写入旧结果。所以超时路径必须和幂等提交、迟到响应处理、临时文件清理一起设计。

\topic{driver 重启}

再加入一个实际故障：driver 在 task \code{Sent} 或 \code{Retrying} 状态崩溃。若 task table 和 commit table 只在内存里，重启后 driver 不知道哪些 task 已经提交，哪些 attempt 只是写了临时文件，哪些响应还可能回来。最简单的恢复策略是：持久化 commit table 和已发布 manifest；重启时把未提交的 running task 全部转回 pending；已提交 task 根据 manifest 验证输出存在和 checksum。

\begin{lstlisting}[numbers=none]
driver recovery:
  load last valid checkpoint
  verify committed manifest entries and checksums
  for each task:
      if committed in manifest:
          state = Committed
      else:
          state = Pending
          current_attempt = next attempt after checkpoint
  ignore replies with request_id not present in recovered state
\end{lstlisting}

这个策略保守但清楚。它可能重复执行未提交 task，但不会重复提交已提交 task。恢复设计的核心不是“尽量少重跑”，而是先保证不漏算、不重复算。性能优化可以后加，例如恢复 worker 本地 attempt 状态、复用完整临时输出、增量扫描 manifest。教材阶段先把可恢复边界写对。

\topic{超时和重试实验}

超时实验让 worker 固定 sleep 500 ms，driver timeout 为 100 ms。朴素版本立即重试，记录请求总数、最大 in-flight、worker 队列峰值和最终失败数。修复版加入 max attempts、退避、抖动和 worker 窗口。对比不应只看最终是否成功，还要看重复 attempt 是否受控、队列水位是否回落、busy 是否被正确处理。

乱序实验让 MessageBus 固定让 attempt 1 响应先到，attempt 0 后到。验收条件是 commit record 保持 attempt 1，attempt 0 只增加 stale 或 late 指标。Driver 日志要能串起两条响应的 request id 和 attempt。没有这条日志，读者看见结果正确，也不知道系统是否真的走过乱序路径。

\section{背压、慢节点和 in-flight 窗口}

分布式系统里，异步发送容易掩盖等待。单机函数慢，调用者卡在当前线程；异步消息慢，driver 可能继续发更多请求，直到 MessageBus、worker 队列、连接池或内存被塞满。背压不是性能优化，而是系统不崩溃的条件。

本章模型里，背压至少有三层。第一，每个 worker 的接收队列有容量。第二，driver 对每个 worker 有 in-flight 上限。第三，整个 job 有总 in-flight 和总重试预算。发送请求前，driver 要检查这些窗口。若 worker 队列满，MessageBus 返回 \code{Busy} 或拒绝入队；若 driver 在本地无限排队，只是把问题从 worker 内存转移到 driver 内存。

\topic{in-flight 窗口从哪里来}

窗口大小不是凭感觉写一个大数字。Little's Law 给出一个基本直觉：稳定系统中，平均在途请求数约等于吞吐乘以平均延迟。若 worker 平均 20 ms 完成一个 split，希望每秒完成 500 个 split，系统大约需要 10 个 in-flight split 才能填满流水线。若只允许 1 个，吞吐被延迟限制；若允许 10000 个，队列和内存会膨胀。

这个模型不是精确预测工具，但能帮助判断配置是否荒谬。窗口大小要随延迟、吞吐、worker 数量、输出大小和错误率调整。短任务需要更小批量和更低排队；长任务可能需要较大 in-flight 来隐藏调度延迟；输出很大的任务受网络和磁盘限制，不能只按任务数控流，还要按字节数控流。

\begin{lstlisting}[language=C++,caption={按任务数和字节数控制窗口}]
struct WorkerWindow {
    std::uint32_t worker_id = 0;
    std::uint32_t max_in_flight_tasks = 0;
    std::uint32_t current_in_flight_tasks = 0;
    std::uint64_t max_in_flight_bytes = 0;
    std::uint64_t current_in_flight_bytes = 0;
};
\end{lstlisting}

\topic{慢节点不是死节点}

故障检测的难点是无法可靠区分“死了”和“很慢”。Worker 可能在磁盘等待、CPU 被抢占、内存回收、网络拥塞、运行时暂停或处理热点 split。Driver 等不到响应，只能怀疑它慢或不可达，不能证明它没有执行。把慢节点误判为死节点，会启动重复 attempt；把死节点长期当成慢节点，会拖慢恢复。

处理慢节点时，可以降低发往它的窗口，也可以启动 speculative attempt。Speculation 的规则是：某个 task 运行时间明显超过同类任务分位数后，在另一个 worker 启动新的 attempt，先完成且通过校验的 attempt 可以提交，其他 attempt 的迟到结果必须拒绝。注意顺序：先有 commit table 和 attempt 检查，再开启 speculation。没有幂等提交，speculation 会直接制造重复结果。

\topic{慢节点指标}

慢节点不能靠感觉判断。Driver 应记录每个 worker 的队列长度、接收速率、完成速率、平均延迟、尾延迟、timeout 数、busy 数、重试数和输出字节。一个 worker 响应慢，可能是它处理的 split 更大，也可能是节点本身慢。若不记录输入字节和输出字节，就会把数据倾斜误判为节点故障。

Stage 级别也要看分布，而不是只看平均值。平均 task 100 ms，P99 task 5 s，说明长尾严重；每个 worker 平均差不多，但某个 partition 队列很长，说明热点 key 或 reduce 背压；CPU 很空但队列长，可能是 I/O 或锁等待；CPU 满且队列长，可能是计算能力不足。队列是分布式系统最重要的压力表之一。

\topic{reduce 背压}

日志统计作业不只有 map 请求。Map worker 本地 combine 后，还要把 partial 按 event type 发送到 reduce partition。若某个 event type 特别热，负责它的 reduce partition 会变慢。Map worker 如果继续发送，reduce 队列会越来越长，内存最终失控。正确行为是 reduce 端返回 \code{Busy} 或降低 credit，map worker 暂停向该 partition 发送，继续处理其他 partition 或在本地做更多 combine。

\begin{lstlisting}[numbers=none]
reduce partition backpressure:
  queue below low watermark  -> accept new messages
  queue above high watermark -> return Busy with retry_after
  map worker receives Busy   -> pause that partition
  map worker may combine more local records before retry
  retry uses backoff and jitter
\end{lstlisting}

这里有两个水位，而不是一个阈值。高水位触发 busy，低水位恢复接收，中间区域保持当前状态。这可以避免队列在阈值附近反复抖动。Map 端收到 busy 后，也不能所有 worker 在同一时间重试，否则会形成同步尖峰。Busy 响应最好带 \code{retry_after} 或 credit 信息，让上游有节奏地恢复。

\topic{热点 key 的层次化处理}

热点 key 是分布式版的共享写热点。单机里，全局原子计数器会让多个核心争用同一 cache line；分布式里，所有 map worker 把同一个 event type 发到同一个 reduce partition，会让该 partition 成为长尾。解决思路也类似：先分散，再合并。

Map-side combine 已经把同一 worker 内的重复 key 合并成 partial。若某个 key 仍然太热，可以把它拆成多个子 key，例如 \code{event_type, local_shard}，让多个 reduce partition 分别合并，最后再做二级 reduce。这个方案会增加最终合并阶段，但能降低第一阶段长尾。它不是第16章的主线，但它说明背压和数据倾斜不是纯网络问题，而是数据分布、调度和算法共同造成的。

\begin{lstlisting}[numbers=none]
normal key:
  partition = hash(event_type) % reduce_count

hot key:
  shard = hash(map_task_id) % hot_shard_count
  partition = hot_partition_base + shard
  final stage merges all shards for the same event_type
\end{lstlisting}

\topic{背压和公平性}

如果只有一个 job，背压主要保护系统稳定。如果多个 job 共用 worker，背压还要考虑公平性。一个大 job 或热点 key 不应该占满所有 reduce 队列，让小 job 无法前进。可以按 job、stage 或 partition 设置独立队列和配额，也可以用加权轮询调度。代价是资源利用率可能下降，但故障半径更可控。

教学项目可以先只支持单 job，但字段里保留 \code{job_id} 是有意义的。未来扩展多租户时，指标、去重、manifest 和配额都要按 job 区分。没有 job id，所有请求混在一起，系统很难解释“是谁压垮了谁”。

\topic{连接池也是背压点}

本机 MessageBus 可以先不用真实 socket，但真实 RPC 系统还有连接池。连接池有最大连接数、每连接最大 in-flight、空闲超时、健康检查和错误恢复。连接都忙时，新请求是等待、失败还是新建连接？等待是否消耗 deadline？连接 reset 时，正在飞行的请求算失败还是未知？这些问题和 worker 队列一样属于背压语义。

这也是为什么本章强调模型而不是具体网络库。换成 gRPC、HTTP、TCP 自定义协议或共享内存队列，问题形状仍然存在：发送方能否无限提交，接收方忙时如何反馈，已提交但未完成的请求如何计数，失败后是否重试。库可以帮你传字节，不能替你定义业务语义。

\topic{背压实验}

背压实验构造一个热点 event type，让 90\% 的记录都落到同一个 reduce partition。朴素版本中，map worker 不看 reduce 队列水位，持续发送 partial，reduce 队列峰值持续增长。修复版本加入 high watermark、low watermark、busy 和 retry after。验收条件是：队列水位能回落，map 端等待时间可观测，最终结果与 reference 一致，重试请求量不超过预算。

另一个实验是慢 worker。让 worker2 每个 split sleep 2 秒，其他 worker 50 ms 完成。Driver 开始时平均分配任务，随后根据延迟和队列水位降低 worker2 窗口。验收条件是：worker2 不再积压大量新任务；stage 尾延迟下降；如果开启 speculation，重复 attempt 不会重复提交。报告要同时写收益和成本：speculation 降低尾延迟，但增加 CPU 和 I/O 消耗。

\section{MessageBus 模拟和故障注入}

本章不要求读者立刻搭真实集群。更好的训练方式，是先用本机 \code{MessageBus} 模拟网络。MessageBus 不需要真实 socket，先用有界队列表达节点之间的通道。每条消息携带 source、target、request id、task id、attempt、payload size、deadline 和 message type。MessageBus 可以按规则注入延迟、丢弃、重复和乱序。

本机模拟不能证明真实网络性能，但能验证协议语义。请求 ID 是否去重，重试是否有预算，队列是否背压，迟到响应是否被拒绝，driver 重启后是否恢复，重复输出是否被提交表挡住，这些都可以在一台机器上验证。先把语义做清楚，再迁移到真实网络，风险更低。

\begin{lstlisting}[language=C++,caption={本机消息的核心字段}]
enum class MessageType : std::int32_t {
    MapTaskRequest,
    MapTaskReply,
    Busy,
    Cancel,
};

struct LocalMessage {
    MessageType type = MessageType::MapTaskRequest;
    std::uint32_t source_node = 0;
    std::uint32_t target_node = 0;
    std::uint64_t request_id = 0;
    std::uint64_t task_id = 0;
    std::uint32_t attempt = 0;
    std::uint64_t payload_bytes = 0;
    std::uint64_t deadline_milliseconds = 0;
};
\end{lstlisting}

这段结构体仍然是逻辑模型。真实 payload 可以放在单独对象或缓冲区中。重要的是每条消息都有足够身份信息，MessageBus 的日志也能记录这些字段。故障注入失败后，读者要能看到“request 8001 被丢弃，task 17 attempt 0 timeout，request 8002 被发送，attempt 1 提交，attempt 0 迟到响应被拒绝”。

\topic{故障规则要可复现}

随机故障有价值，但单元测试需要确定性。MessageBus 应支持按 request id 注入故障，例如丢弃 request 8001 的响应，延迟 task 17 attempt 0，重复投递 request 9002，或者让某个 worker 队列在第 N 次发送时返回 busy。也可以使用固定随机 seed 生成故障序列，并把序列写入日志，失败后用同一个 seed 重放。

\begin{lstlisting}[language=C++,caption={可复现故障注入规则}]
enum class FailureAction : std::int32_t {
    None,
    Drop,
    Delay,
    Duplicate,
    Reorder,
};

struct FailureRule {
    std::uint64_t request_id = 0;
    MessageType message_type = MessageType::MapTaskReply;
    FailureAction action = FailureAction::None;
    std::uint64_t delay_milliseconds = 0;
};
\end{lstlisting}

可复现性是分布式测试的生命线。偶现 bug 如果无法重放，修复很容易变成猜测。教学项目里的故障注入不必覆盖所有真实网络行为，但至少要覆盖本章讨论的四类：丢失、重复、延迟、乱序。每类故障都要有一个最小测试。

\topic{MessageBus 不应该隐藏背压}

很多教学模拟会把 MessageBus 写成无限 vector 或无限 queue。这样做会掩盖最重要的系统问题：队列无界增长。MessageBus 的每个通道都应有容量，可以按消息数限制，也可以按 payload 字节限制。容量满时，发送方必须得到明确结果：accepted、busy、deadline exceeded 或 invalid target。

\begin{lstlisting}[numbers=none]
send result:
  Accepted         message is queued
  Busy             target queue is above high watermark
  DeadlineExpired  message deadline has already passed
  InvalidTarget    route is unknown or stale
\end{lstlisting}

这个返回值会影响 driver 行为。Accepted 后 task 进入 Sent；Busy 后进入退避或换 worker；DeadlineExpired 后不应再重试同一个 attempt；InvalidTarget 后需要刷新路由或失败。MessageBus 不是简单管道，它是本机训练背压和错误分类的工具。

\topic{Trace 关联}

每次事件都要写 trace。Trace 不只是日志文本，而是带结构字段的事件流。Driver 发送、MessageBus 入队、MessageBus 丢弃、worker 接收、worker 完成、driver 提交、driver 拒绝迟到响应，都应该有 event type、timestamp、job id、task id、attempt、request id 和 node id。这样一次故障能被完整串起来。

\begin{lstlisting}[numbers=none]
trace event fields:
  time_milliseconds
  event_type
  job_id
  stage_id
  task_id
  attempt
  request_id
  source_node
  target_node
  status
\end{lstlisting}

没有 trace，读者只能看最终结果。最终结果正确不代表系统语义正确，可能只是故障没有触发。Trace 让实验报告能回答：哪条消息丢了，哪次 timeout 触发重试，哪个 attempt 提交，哪个旧响应被拒绝。分布式系统的学习不能只看结果，还要看路径。

\topic{指标清单}

本章最低指标清单如下。

\begin{lstlisting}[numbers=none]
minimum distributed metrics:
  requests_sent
  replies_received
  timeouts
  retries_started
  duplicate_messages
  duplicate_replies
  stale_replies
  late_replies
  busy_responses
  max_in_flight_tasks
  max_worker_queue_depth
  max_reduce_queue_depth
  committed_tasks
  failed_tasks
\end{lstlisting}

每个指标都要服务一个问题。请求数和重试数看重试是否失控；late 和 stale 看乱序响应是否被挡住；busy 和队列水位看背压是否生效；committed 和 failed 看任务最终状态；in-flight 看窗口配置是否合理。指标不是为了漂亮仪表盘，而是为了验证协议假设。

\topic{实验矩阵}

实验矩阵应覆盖正常输入、边界输入、压力输入和错误输入。正常输入验证功能；边界输入验证 split 边界、空 split、单条记录、多个 split 同一 key；压力输入验证队列、水位和窗口；错误输入验证丢响应、重复响应、乱序响应、busy、worker 慢、driver 重启。每一项都要写预期行为。

\begin{lstlisting}[numbers=none]
fault matrix:
  request dropped        -> retry within budget
  reply dropped          -> retry; old attempt may finish but cannot double commit
  reply duplicated       -> second reply is ignored or returns old result
  replies reordered      -> stale attempt cannot overwrite committed result
  worker busy            -> backoff or choose another worker
  worker slow            -> reduce window or start speculative attempt
  driver restart         -> recover manifest; rerun only uncommitted tasks
  reduce queue full      -> map side pauses that partition
\end{lstlisting}

这张矩阵不是附录装饰，而是设计本身。每一行都逼迫系统回答一个具体问题。若某一行没有答案，说明协议还有洞。分布式系统最危险的不是某个名词没背，而是故障路径没有被写成可执行行为。

\topic{Linux 视角：不要神化网络}

真实网络路径经过用户态缓冲、系统调用或用户态网络栈、socket buffer、调度、网卡队列、交换机、对端队列和对端线程。Linux 源码阅读可以从窄路径开始：socket 发送和接收、定时器、epoll 唤醒、网络设备队列。阅读目标不是一次读懂网络栈，而是把用户态现象和内核对象对应起来：发送为什么会阻塞，接收为什么会等待，超时由谁触发，队列满时错误如何返回。

对本章来说，最重要的 Linux 启发是：网络不是魔法函数调用。一次 \code{write} 不等于对端一次 \code{read}；TCP 是字节流，不保留应用消息边界；socket buffer 满会导致排队或返回 busy；定时器触发只说明本地等待时间到了；线程调度会影响延迟。理解这些事实，能帮助读者相信本章的模型不是凭空复杂，而是对真实系统边界的抽象。

\section{落到贯穿项目}

Compute Systems Engine 在本章要实现的不是完整集群，而是一个语义正确的本机分布式模拟。最低模块包括 \code{Driver}、\code{Worker}、\code{MessageBus}、\code{CommitTable}、\code{TraceLog} 和 \code{Metrics}。Driver 维护 task table、attempt、deadline 和调度窗口；Worker 只通过消息接收任务并返回结果；MessageBus 支持延迟、丢弃、重复、乱序和有界队列；CommitTable 保证每个逻辑 task 只提交一次；TraceLog 让故障能被复盘。

实现顺序要克制。不要一开始搭一个看似完整的框架。第一步写串行 reference，固定输入格式和输出比较。第二步让 driver 发送一个 map task 给一个 worker，跑正常路径。第三步加入三个 worker 和 task table。第四步让 MessageBus 支持丢响应，加入 timeout 和 retry budget。第五步实现 commit table 和 manifest，保证重复 attempt 不重复提交。第六步加入乱序和重复消息测试。第七步加入 worker busy、reduce 背压和窗口。第八步加入 checkpoint 和 driver 重启恢复。

\begin{lstlisting}[numbers=none]
implementation order:
  1. serial reference for log counting
  2. Driver sends one MapTaskRequest to one Worker
  3. three Workers and a Driver task table
  4. MessageBus supports delay and drop
  5. Driver adds timeout, retry budget and backoff
  6. CommitTable rejects duplicate task commits
  7. MessageBus supports duplicate and reorder
  8. reduce queues return Busy with high and low watermarks
  9. TraceLog and metrics CSV record protocol behavior
  10. checkpoint allows Driver restart recovery
\end{lstlisting}

\topic{reference 不能省}

Reference 是整个项目的锚点。没有 reference，最终输出是否正确只能靠感觉。Reference 可以很慢，可以是单线程，可以不做任何分布式优化，但必须精确定义语义。分布式版本无论如何重试、乱序、背压、speculation，最终输出都要和 reference 一致。若输入包含坏行，reference 要定义坏行如何处理；若 event type 越界，reference 要定义是丢弃、报错还是进入错误计数。

Reference 还可以帮助定位错误。若分布式结果比 reference 多，优先怀疑重复提交或 split 重叠；若少，怀疑漏 split、commit 丢失或恢复跳过；若只有热点 key 错，怀疑热点拆分或 reduce 去重；若每次运行输出顺序不同但计数相同，可能是排序或输出格式问题。好的 reference 不只是测试 oracle，也是调试地图。

\topic{CommitTable 接口}

CommitTable 的接口应该围绕不变量设计，而不是暴露随意写 map 的能力。调用者提交一个 attempt，CommitTable 返回 accepted、already committed、stale attempt 或 checksum mismatch。CommitTable 内部负责检查 task 是否已提交，验证输出摘要，更新 manifest。外部不能绕过它直接修改 committed 状态。

\begin{lstlisting}[language=C++,caption={提交决策的接口形态}]
enum class CommitDecision : std::int32_t {
    Accepted,
    AlreadyCommitted,
    StaleAttempt,
    ChecksumMismatch,
};

struct CommitRequest {
    std::uint64_t task_id = 0;
    std::uint32_t attempt = 0;
    std::uint64_t output_bytes = 0;
    std::uint64_t output_checksum = 0;
};
\end{lstlisting}

这个接口没有展示文件写入细节，但它把行为边界说清。Driver 处理响应时不应该自己写一堆 if 再改 manifest，而应该把提交请求交给 CommitTable。这样测试可以集中覆盖重复提交、迟到 attempt、checksum mismatch 和恢复后的提交。

\topic{Worker 的责任边界}

Worker 的职责是读取 split、执行 map、写 attempt 临时输出、返回摘要。Worker 不决定最终结果是否有效，也不直接修改全局 manifest。这样做让 worker 可以重复执行而不破坏外部效果。Worker 可以清理自己未被接受的临时文件，但清理失败不影响最终语义，只影响磁盘空间。

Worker 还要报告可观测信息：读取字节、解析记录、错误记录、本地 combine 后 key 数、输出字节、执行耗时。若 worker 只返回 \code{Ok}，driver 无法判断慢在哪里。一个 split 处理慢，可能是输入大、坏行多、key 多、磁盘慢、CPU 忙或队列等待。指标越早设计，后面的性能章节越容易承接。

\topic{MessageBus 的责任边界}

MessageBus 负责消息投递和故障注入，不负责业务提交。它可以根据规则丢弃、延迟、重复和乱序消息，可以根据队列容量返回 busy，可以记录 trace。但它不应该知道 task 是否 committed，也不应该替 driver 决定重试。这样模块边界清楚：MessageBus 模拟不可靠通信，Driver 处理协议语义，CommitTable 处理提交不变量。

这个边界也有助于未来替换真实网络。若 MessageBus 里混入大量业务逻辑，换成 socket 后就很难迁移。若 MessageBus 只表达通信语义，真实网络层只需要提供类似的 send result、receive event 和 trace hook。

\topic{Checkpoint 的最低内容}

Driver checkpoint 至少包含 job 配置、input split 列表、task 状态摘要、commit table、manifest、下一批 request id 或其生成策略、路由 epoch 和指标摘要。缺少 job 配置，恢复后可能用不同 reduce 数解释旧输出；缺少 split 列表，无法判断哪些 task 还要跑；缺少 commit table，重复响应可能再次提交；缺少 request id 策略，日志关联会混乱。

教学项目可以在 stage 边界 checkpoint，也可以每次 commit 后 checkpoint。前者正常路径成本低，但崩溃后重跑更多；后者恢复更快，但持久化成本高。教材要让读者看到这是权衡，而不是有一个永远正确的频率。高质量系统设计总是把成本写出来。

\topic{和第 17、18 章的衔接}

本章只解决“一个逻辑 task 如何在消息系统里安全执行和重试”。第17章再讨论分片、复制和共识：partition 在哪里，路由版本怎么变，哪些控制面状态需要强一致，旧 leader 如何被 fencing。第18章把这些机制放进完整 shuffle：map 输出如何进入 manifest，reduce 如何只读已提交 block，checkpoint 如何恢复，热点 key 如何做二级聚合。

这个顺序很重要。若读者还说不清 \code{task_id}、\code{attempt}、\code{request_id} 的区别，就不该急着看共识；若还说不清超时和取消的区别，就不该急着写 speculative execution；若还没有 commit table，就不该允许重复 attempt 写最终结果。分布式系统不是概念越多越高级，而是每个失败路径都有明确边界。

\topic{事故复盘：把概念压成可执行流程}

如果只说“要有 request id、要能重试、要做幂等”，这章仍然会显得模糊。真正理解分布式系统，要能把一次事故从输入、状态、消息、指标和修复全部走通。下面用同一个日志统计作业复盘一次具体事故：最终 event 7 的计数比串行 reference 多了 1280。输入没有变化，单机 reference 稳定，只有分布式模拟偶尔多算。这个现象足够具体，后面才能讨论原因。

事故发生时的系统配置是：一个 driver，三个 worker，十二个 input split，四个 reduce partition。Driver timeout 为 100 ms，worker1 被故障注入延迟 300 ms，MessageBus 对 request 4107 的响应执行 drop。朴素版本里，worker 写完 map 临时输出后，reduce 端会直接读取 worker 目录下所有完成文件。Driver 收不到响应后启动 attempt 1，attempt 1 也写出一份同一 split 的 partial。最后 reduce 扫目录时读到了两份 partial，于是 event 7 多了 1280。

这不是抽象的“幂等问题”，而是一条可复现的路径。复盘时先把时间线写出来。

\begin{lstlisting}[numbers=none]
t0  driver sends request 4107: task 5, attempt 0, worker1
t1  worker1 reads split 5 and writes temp block A
t2  worker1 sends reply 4107, MessageBus drops it
t3  driver timeout for request 4107
t4  driver sends request 4119: task 5, attempt 1, worker2
t5  worker2 reads split 5 and writes temp block B
t6  worker2 reply 4119 arrives
t7  driver marks task 5 completed
t8  reduce scans worker temp directories and reads block A and block B
\end{lstlisting}

这条线一写出来，修复方向就变得清楚。问题不是 worker1 写出了 block A，重复执行在分布式系统中可以发生；问题是 reduce 不应该扫描临时目录。Reduce 应该只读取 driver commit manifest 中被接受的 block。临时目录是数据面暂存，manifest 是控制面提交边界。把二者分开后，attempt 0 和 attempt 1 可以都产生文件，但只有被 commit table 接受的那个 attempt 会进入 manifest。

\topic{复盘第一步：确认 reference}

事故复盘不能从猜测开始。第一步是确认串行 reference。拿同一批十二个 split，单线程读取，按同样解析规则统计 event type，输出稳定 checksum。若 reference 本身每次不一样，分布式层无从判断。若 reference 和分布式版本解析规则不同，后面所有 trace 都会偏。Reference 要记录输入文件列表、split 边界、总记录数、解析失败数、每个 event 的计数摘要和最终 checksum。

这里的关键是“同一语义”。如果分布式 worker 遇到坏行会跳过，而 reference 遇到坏行会中止，结果不一致不是分布式协议造成的。如果 reference 按 event type 排序输出，而分布式版本按哈希表遍历顺序输出，字节级比较可能失败但计数正确。教材项目必须先把这些差异写清，否则读者会把数据合同问题误判为 RPC 问题。

\topic{复盘第二步：查 trace 而不是猜}

有了 reference，再看 trace。Trace 不应该只有一行“task 5 success”。它要能回答 task 5 到底启动了几次 attempt，哪个 request 被丢弃，哪个 attempt 写了 block，哪个 block 被 commit，reduce 读了哪些 block。一个有用的 trace 片段应该像下面这样。

\begin{lstlisting}[numbers=none]
000120 driver_send       task=5 attempt=0 request=4107 target=worker1
000128 worker_start      task=5 attempt=0 request=4107 worker=worker1
000311 worker_write      task=5 attempt=0 block=A checksum=991
000315 bus_drop_reply    task=5 attempt=0 request=4107
000420 driver_timeout    task=5 attempt=0 request=4107
000436 driver_send       task=5 attempt=1 request=4119 target=worker2
000491 worker_write      task=5 attempt=1 block=B checksum=991
000500 driver_commit     task=5 attempt=1 block=B checksum=991
000700 reduce_read       task=5 attempt=0 block=A
000701 reduce_read       task=5 attempt=1 block=B
\end{lstlisting}

最后两行就是 bug。Reduce 读 block 时没有经过 manifest 过滤。注意，这个结论不是靠背“exactly once”得到的，而是 trace 把错误路径暴露出来。高质量分布式教材要训练读者看路径：一条消息、一个状态、一个提交点。只给名词，不给路径，读者遇到真实问题仍然不会修。

\topic{复盘第三步：收紧不变量}

修复时不要只加一个 if。要把不变量写出来：每个 \code{job_id, stage_id, task_id} 最多有一个 committed attempt；reduce 只能读取 manifest 中的 committed block；worker 临时目录中的文件没有业务可见性；迟到响应不能修改 committed record；checkpoint 恢复后仍然遵守这些规则。代码和测试都要围绕这些不变量展开。

不变量写清后，模块职责也清楚。Worker 可以写临时 block，但不能发布最终结果；Driver 可以接收 reply，但必须通过 CommitTable；CommitTable 可以修改 manifest，但必须保证原子发布；Reduce 可以读取 block，但只能从 manifest 获取有效 block 列表；MessageBus 可以丢消息，但不能改变业务状态。职责一分开，很多模糊设计会自然消失。

\topic{修复后的时间线}

修复后，同样事故应该变成下面的路径。

\begin{lstlisting}[numbers=none]
t0  attempt 0 writes temp block A
t1  reply for attempt 0 is dropped
t2  driver starts attempt 1
t3  attempt 1 writes temp block B
t4  CommitTable accepts task 5 attempt 1
t5  manifest contains task 5 -> block B
t6  attempt 0 late reply arrives or block A remains on disk
t7  driver records late reply and schedules cleanup for block A
t8  reduce reads manifest and sees only block B
\end{lstlisting}

这条路径允许旧 attempt 留下痕迹，但不允许旧 attempt 产生业务效果。这个区别很重要。分布式系统很难保证“坏事完全不发生”，更现实的目标是让坏事发生后越不过提交边界。临时文件残留是资源问题，重复计数是正确性问题。先保护正确性，再处理资源清理。

\topic{事故报告应该怎样写}

事故报告不应只写“加入幂等表修复”。至少要包括五段。第一段写现象：哪个输入、哪个 event、差多少、是否可复现。第二段写触发条件：响应丢失、timeout、重试、reduce 扫目录。第三段写根因：reduce 的有效输入来源错误，把临时目录当成提交事实。第四段写修复：manifest 成为唯一有效输入，CommitTable 原子发布。第五段写验证：同一故障种子下 reference 一致，late reply 增加，manifest 只含一个 block，临时残留不影响结果。

这种报告会让读者形成正确习惯：先描述事实，再定位路径，再提出机制，再验证机制。它比“分布式系统要幂等”这种句子慢，但更接近工程。一个人只有能写出这种复盘，才算真正理解本章。

\topic{协议细节：从 driver 主循环到提交崩溃点}

本章前面讲了状态和失败，现在把它落到更接近代码的流程。Driver 主循环可以看成四类工作：调度新 task，处理消息，检查 deadline，提交 checkpoint。真实实现可以是事件循环、多线程或 actor，但语义顺序基本一致。重要的是每一步都要先看状态，再改变状态，不能把网络回调写成随意修改全局 map 的函数。

\begin{lstlisting}[numbers=none]
driver loop:
  poll messages from MessageBus
  for each message:
      dispatch by message type
  check deadlines for sent or running tasks
  start retries if policy allows
  schedule pending tasks if worker windows allow
  flush commit manifest or checkpoint when needed
  emit metrics and trace events
\end{lstlisting}

这个循环让读者看到，分布式系统不是一个大 if。消息处理、deadline 检查、调度和 checkpoint 都会竞争同一批 task 状态。若 response handler 正在 commit，deadline checker 又认为同一 task timeout，代码必须有清晰的状态转换规则。最简单的教学实现可以把 driver loop 放在单线程中，先避免数据竞争，把协议语义写对。等语义稳定后，再讨论多线程 driver 和锁。

\topic{发送请求前的检查}

Driver 准备发送 task 时，至少检查五件事。第一，task 是否仍为 \code{Pending} 或允许 retry；第二，job deadline 是否还有预算；第三，目标 worker 的窗口是否有容量；第四，split 元数据是否有效；第五，新 request id 是否已经分配并写入 task entry。发送前先写状态还是发送后再写状态，也要明确。

教学实现可以采用保守顺序：先在内存中生成 request id 和 attempt，调用 MessageBus send；若 send 返回 accepted，再把 task 标记为 \code{Sent} 并记录发送时间；若 send 返回 busy，则 task 保持 pending 或进入退避队列。若先把 task 标记为 sent，再发现 MessageBus 拒绝，就会留下一个没有真实消息的 sent 状态。真实多线程实现还要考虑 send accepted 后 driver 立刻崩溃，这就进入 checkpoint 设计。

\topic{处理 Busy 不是处理失败}

Busy 表示容量信号，不等于业务失败。Worker busy 时，driver 不应该马上把 task 标成 failed，也不应该立刻对同一 worker 高速重试。更合理的行为是记录 busy，更新 worker 窗口，设置 retry after，把 task 放回 pending 或 delayed queue。若有其他健康 worker，可以换 worker；若所有 worker 都 busy，driver 应降低调度速率。

Busy 的语义要和 deadline 结合。若 task deadline 还有很多预算，可以等待；若 deadline 接近耗尽，继续排队可能没有意义。此时应返回 deadline exceeded 或 stage timeout，而不是把系统压到更满。过载时快速失败虽然看起来不友好，但比无限排队更可诊断。

\topic{处理响应的细化流程}

响应处理可以拆成身份检查、状态检查、输出检查和提交检查。身份检查确认 \code{task_id}、\code{attempt}、\code{request_id} 是否属于可接受范围；状态检查确认 task 未提交且未失败；输出检查确认 worker 报告的字节数、记录数、checksum 和 manifest 中临时 block 一致；提交检查由 CommitTable 决定是否接受。

\begin{lstlisting}[numbers=none]
reply handling:
  1. find task entry
  2. reject unknown task
  3. reject already committed task as late reply
  4. reject stale attempt
  5. reject stale request id
  6. verify status and output checksum
  7. call CommitTable.try_commit
  8. update task state from commit decision
  9. emit trace and metrics
\end{lstlisting}

顺序不能随便调。若先 verify output 再检查 stale attempt，旧 attempt 可能触发昂贵 I/O；若先更新 task state 再 commit manifest，崩溃后内存状态丢失；若 CommitTable 返回 already committed 却仍覆盖 task entry，会破坏迟到响应语义。书里把这个流程写细，是为了让读者知道哪些 if 是协议边界，不是代码风格。

\topic{提交崩溃点矩阵}

Commit 是最容易模糊的地方。下面不用表格，而用四个崩溃点逐个分析。第一，验证输出前崩溃。恢复后 manifest 没有新记录，task 仍未提交，可以重跑。临时文件可能存在，但 reduce 不会读。第二，验证输出后、写新 manifest 前崩溃。恢复行为相同，因为没有发布提交事实。第三，写完新 manifest 临时文件、rename 前崩溃。恢复时只读取已发布 manifest，临时 manifest 被清理或忽略。第四，rename 成功后、内存 task state 更新前崩溃。恢复时读取新 manifest，task 变为 committed，不会重跑。

这四个点说明为什么原子发布重要。恢复逻辑只相信一个事实来源：已发布 manifest。内存状态、临时文件、未发布 manifest 都不能单独证明 task 已提交。这样设计后，崩溃点再多，也会收敛到两类：发布前当作未提交，发布后当作已提交。

\topic{manifest 的内容}

Manifest 不是一个简单路径列表。它至少要包含 job id、stage id、task id、attempt、reduce partition、文件位置、字节数、记录数、checksum 和 schema version。若只保存文件路径，恢复时不知道这个文件属于哪个逻辑 task，也不知道是否来自旧 attempt。若不保存 checksum，磁盘残缺或写入错误可能被当作有效数据。若不保存 schema version，后续格式变化会破坏旧 checkpoint。

\begin{lstlisting}[language=C++,caption={manifest 条目的教学形态}]
struct ManifestEntry {
    std::uint64_t job_id = 0;
    std::uint64_t stage_id = 0;
    std::uint64_t task_id = 0;
    std::uint32_t attempt = 0;
    std::uint32_t reduce_partition = 0;
    std::uint64_t block_id = 0;
    std::uint64_t byte_size = 0;
    std::uint64_t record_count = 0;
    std::uint64_t checksum = 0;
    std::uint32_t schema_version = 1;
};
\end{lstlisting}

这段结构同样不是二进制格式，只是说明 manifest 的语义字段。真实系统还要保存路径、压缩格式、副本位置、创建时间和加密信息。教学项目先保留核心字段，就能把“哪些输出有效”这件事说清。

\topic{清理不是提交路径}

未被接受的 attempt 临时文件要清理，但清理不应影响正确性。清理请求可能失败，worker 可能离线，文件可能已经被手动删除。系统应该允许垃圾文件暂时存在，并通过后台清理按 manifest 反向判断哪些文件不再需要。若把清理放进提交关键路径，worker1 清理失败可能阻塞 attempt 1 提交，反而降低可用性。

这又是控制面和数据面的分离。Manifest 控制有效数据集合；后台清理优化磁盘空间。正确性依赖 manifest，不依赖清理成功。读者写系统时要保持这种分离，否则会把资源回收错误升级成业务失败。

\topic{driver 多线程化的边界}

教学实现可以先单线程 driver，但真实 driver 可能多线程处理消息、调度和 checkpoint。多线程化时，task entry 和 commit table 是共享状态，必须有清晰同步。最保守的方法是让所有状态转换回到一个 driver event loop，其他线程只把事件放入队列。这样业务状态仍然单线程修改，网络和 I/O 可以并行。

如果直接让多个线程同时修改 task map，很容易出现竞态：deadline 线程启动 attempt 1 的同时，response 线程提交 attempt 0；checkpoint 线程正在序列化 manifest 时，commit 线程追加新记录；metrics 线程遍历 map 时，调度线程删除 entry。分布式系统已经有消息乱序，不要再让本地共享状态增加不必要的不确定性。

\topic{从推送 shuffle 到拉取 manifest}

前面的事故暴露了一个核心问题：reduce 端怎样知道哪些 map 输出有效。这里需要比较两种设计。第一种是推送式：map worker 完成后主动把 partial 推给 reduce worker。第二种是拉取式：map worker 先写 shuffle block，driver commit manifest，reduce worker 根据 manifest 拉取已提交 block。两种都能工作，但故障语义不同。

推送式看起来更直接，map 端有数据就发，reduce 端收到就合并。问题是重复 attempt 和响应丢失时，reduce 端可能收到同一逻辑 task 的多份 partial。若 reduce 端边收边加到最终 hash map，就必须维护去重表，且要处理“attempt 0 发了一半失败，attempt 1 完整成功”的边界。去重粒度按 request id 不够，按 attempt 也不够，最终要回到 logical task 或 manifest。

拉取式把有效性判断集中到 driver manifest。Map attempt 写完整 block 后报告摘要，driver 选择一个 attempt commit，reduce 只拉取 manifest 中列出的 block。这样 reduce 端不用猜目录，也不会读旧 attempt。代价是多一个 manifest 控制面，reduce 要等 map commit 后才能拉取，可能增加一点调度延迟。但对教学项目来说，拉取式更容易把 exactly once 外部效果讲清楚。

\topic{reduce 端合并的去重边界}

即使使用 manifest，reduce 端也应该防御重复拉取。网络重试可能让同一个 block 被拉两次，reduce worker 重启可能重放一段输入，MessageBus 可能重复投递 block 消息。Reduce 可以维护 \code{block_id} 或 \code{task_id, reduce_partition} 的已处理集合。区别在于，\code{block_id} 防止同一 block 重复处理，\code{task_id, reduce_partition} 防止同一逻辑 map task 的多个 block 进入同一 partition。

如果 manifest 已经保证每个 task 每个 partition 只有一个 block，reduce 端按 \code{block_id} 去重足够防重复拉取；若 reduce 端允许直接接收推送，则必须按逻辑 task 去重。教材要明确前提，不能笼统说“reduce 要去重”。去重键取错，会让系统在某些故障下仍然重复计数。

\topic{局部聚合和输出确定性}

Map-side combine 会改变数据发送形态，但不应改变结果。Worker 可以用哈希表做本地聚合，输出时要按 partition 组织。如果最终结果需要稳定比较，reduce 输出应按 event type 排序，或者输出 checksum 时使用确定顺序。否则多线程和哈希表遍历顺序会让每次输出字节不同，虽然数学计数相同。

输出确定性对测试和恢复很重要。若每次运行输出顺序不同，reference 对比要么变复杂，要么误报失败。一个教学项目应先追求确定输出：相同输入、相同故障规则、相同配置，输出字节和 trace 关键路径稳定。性能优化可以在确定性基础上做，而不是让随机顺序掩盖协议问题。

\topic{热点 key 的实际处理流程}

热点 key 处理也要从现象出发。假设 reduce partition 1 的队列水位持续高，其他 partition 很低。Trace 显示 90\% 的 records 都是 event type 7，普通哈希分区把它们都路由到 partition 1。第一步不是立刻上复杂算法，而是确认 map-side combine 是否已经开启。如果 map 端还在逐记录发送，先做本地聚合，网络量可能立刻下降。

若 combine 后仍然热点，就引入 hot key split。Driver 或采样器把 event type 7 标记为 hot，map worker 对它输出多个 shard，reduce 第一阶段分别合并 shard，第二阶段再把 shard 合成 event type 7 的最终计数。这个改动必须进入 manifest：每个 hot shard 都有自己的 block 和 checksum，最终合并 stage 也有自己的 task id。否则恢复时不知道哪些 shard 已经处理。

\topic{背压下的本地 combine}

Reduce 返回 busy 后，map worker 不一定只能睡眠。它可以继续读取输入并在本地 combine 更多记录，前提是本地内存水位允许。这样上游把压力转成更大的 batch，而不是立刻重试网络发送。若本地 combine 内存也达到上限，worker 才应该暂停读取或把 partial spill 到本地磁盘。

这个流程把单机和分布式原则串起来：有界队列给出背压，本地聚合减少数据量，spill 防止内存爆，manifest 管理中间文件，retry budget 防止 busy 放大。它不是一堆概念并列，而是一条压力从 reduce 端传回 map 端的实际路径。

\topic{实验报告：必须能复现、能解释、能回归}

本章的实验报告要比“跑通截图”严格得多。每个实验都应包含输入描述、配置、故障规则、朴素版本行为、修复版本行为、关键指标、trace 摘要、reference 对比和未验证边界。这样写看起来麻烦，但它能防止教材内容变成口号。读者必须知道每个机制到底解决了什么。

\topic{实验一：响应丢失}

输入为十二个 split，每个 split 至少包含一个共同 event type，方便发现重复计数。故障规则固定为丢弃 task 5 attempt 0 的 reply。朴素版本让 reduce 扫描临时目录，预期结果是 event 计数高于 reference。修复版本让 reduce 只读 manifest，预期结果和 reference 一致。

报告中的指标至少应包含：\code{requests_sent=13}，因为一个 task 重试一次；\code{timeouts=1}；\code{retries_started=1}；\code{committed_tasks=12}；\code{late_replies} 可能为 0 或 1，取决于是否让旧 reply 延迟到达；manifest 中 task 5 只有一个 entry。若这些指标不满足，说明实验没有真正覆盖目标故障。

\topic{实验二：乱序响应}

故障规则固定为让 task 6 attempt 1 的 reply 先于 attempt 0 到达。朴素版本若采用“最后响应覆盖”，可能表现为 committed attempt 被改回旧 attempt，或者 metrics 中没有 stale reply。修复版本应在 attempt 1 commit 后拒绝 attempt 0。报告要贴出两条 trace：attempt 1 commit 的事件和 attempt 0 stale reply 的事件。

这个实验的重点不是结果是否正确，因为两个 attempt 计算同一 split 时 checksum 可能相同，最终计数未必错。重点是状态不能被旧响应覆盖。如果实验只看最终计数，就会漏掉一个潜在恢复 bug。高质量测试要验证不变量，不只验证输出。

\topic{实验三：重试风暴}

配置为 100 个 task，worker 固定 sleep 300 ms，driver timeout 50 ms。朴素版本立即重试且无 max attempts，预期 in-flight 和队列水位持续上升。修复版本设置 max attempts 为 3，退避从 50 ms 开始，加入 jitter，并限制每个 worker in-flight。预期请求总数不会超过 300 加上少量 busy 探测，队列峰值受窗口限制。

报告要写出两组数字：朴素版本的最大 in-flight、最大 worker 队列、总请求数；修复版本的同样指标。若修复版本只是最终成功，但请求数仍然爆炸，说明预算没有生效。若队列水位下降但失败数上升，也要如实写出，这是稳定性和完成率的权衡。

\topic{实验四：driver 重启}

这个实验要覆盖四个崩溃点：request sent 后崩溃、worker 写临时 block 后崩溃、manifest 临时文件写完但未发布时崩溃、manifest 发布后内存状态更新前崩溃。每个崩溃点都用固定钩子触发。恢复后运行到结束，最终结果必须和 reference 一致。

报告要说明每个点的恢复状态。发布前崩溃，task 回到 pending 或 retrying，可能重跑；发布后崩溃，task 从 manifest 恢复为 committed，不重跑。若某个崩溃点需要人工清理才能通过，说明恢复协议不完整。分布式系统不能只在干净关闭后恢复，真正要处理的是中途崩溃。

\topic{实验五：reduce 背压}

输入让 90\% 记录落到 event type 7。普通分区下，partition 1 队列变长。开启 high watermark 和 low watermark 后，map 端收到 busy，暂停该 partition，并继续处理其他 partition 或本地 combine。报告要记录 partition 级别队列水位、busy 次数、map 等待时间、最终 checksum。若只看总吞吐，会看不出背压是否按 partition 生效。

进一步可以比较三种版本：无 combine、combine、combine 加 hot key split。无 combine 网络消息最多；combine 降低消息数；hot key split 降低 partition 长尾但增加最终合并阶段。这个对比能让读者看到优化不是单调“更高级更好”，而是在不同成本之间移动。

\topic{未验证边界也要写}

实验报告最后要写未验证边界。例如本机 MessageBus 没有真实网络拥塞，不能证明真实 TCP 下的吞吐；单进程模拟没有跨机器时钟漂移，不能证明租约安全；本地文件 rename 的语义不能直接代表对象存储；固定输入规模不能证明更大数据下内存上限合理。写出这些边界不是降低质量，而是防止把局部证据夸大成普遍结论。

这也是本章写法和概念清单的区别。概念清单容易给人一种“知道名词就掌握系统”的错觉。实验边界会提醒读者：原理可以迁移，数字要重测；不变量可以保留，实现要适配；本机模拟能证明语义，不能替代真实部署压测。

\topic{常见错误：为什么很多分布式章节读起来空}

最后把几个常见写法问题直接指出来。第一种错误是先讲 CAP、Paxos、RPC，再补一个例子。读者会记住术语，却不知道这些术语从哪个失败路径里长出来。更好的顺序是先让一个普通任务失败：响应丢了、重复执行了、旧响应回来了、队列满了、driver 重启了。然后再引出机制。

第二种错误是把所有问题都叫“一致性”。重复 attempt 导致重复计数，是幂等提交问题；旧路由写入，是 epoch 和 fencing 问题；多个副本选择写入顺序，是共识或复制问题；reduce 读半写文件，是 manifest 提交边界问题。都叫一致性，听起来高级，实际会模糊修复点。教材应该训练读者把问题切细。

第三种错误是只讲正常路径。正常路径里，一条请求、一条响应、一个结果，几乎所有设计都看起来合理。分布式系统的设计质量要看故障路径：响应丢失后是否重复提交，旧 attempt 回来后是否覆盖状态，worker busy 后是否重试风暴，driver 崩溃后是否恢复，manifest 半写后是否被误读。没有故障路径，章节就会空。

\topic{不要把 exactly once 写成魔法}

Exactly once 在工程中通常不是“函数真的只执行一次”。函数可能执行多次，消息可能投递多次，attempt 可能跑多次。系统追求的是外部效果等价于一次：重复执行被 commit table、manifest、去重表或事务输出吸收。若教材把 exactly once 写成网络保证，读者会误以为只要换一个消息队列就能解决问题。

本章的日志统计正好能说明这点。Task 5 的 attempt 0 和 attempt 1 都可以执行，这是 at least once 的执行现实；manifest 中 task 5 只有一个有效 block，这是 exactly once 的外部效果。把执行次数和外部效果分开，很多概念会清楚很多。

\topic{不要把 timeout 写成失败证明}

Timeout 只是本地等待到期。它不能证明请求没到，不能证明 worker 没执行，不能证明响应不会回来。若教材说“超时后任务失败”，读者就会写出危险代码。正确说法是：超时后 driver 可以怀疑当前 attempt 不再值得等待，并在预算内启动新 attempt；旧 attempt 的后续响应仍要按迟到消息处理。

这句话看似冗长，却是分布式系统的根。只要理解它，request id、attempt、commit table、late reply、cancel、retry budget 都会变得自然。若不理解它，所有机制都像强行加上的复杂度。

\topic{不要把背压写成性能章节的附属品}

背压不是“优化一下吞吐”。没有背压，系统在过载时会用队列和重试把自己压垮。Driver 的发送窗口、worker 队列容量、reduce partition 水位、连接池 in-flight、retry budget 都是背压的一部分。它们共同回答一个问题：下游处理不过来时，上游如何减速。

如果只在性能章节讲背压，读者会忽略它的正确性意义。重试风暴会导致 deadline 大量过期，旧 attempt 堆积会增加重复提交压力，队列爆炸会触发 OOM，OOM 后恢复又会考验 checkpoint。背压、可靠性和恢复不是三件孤立的事。

\topic{不要让项目代码偏离书的主线}

贯穿项目最容易失控的地方，是先写一堆框架：抽象节点、抽象协议、抽象调度器、抽象插件。这样代码看起来工程化，实际没有验证任何不变量。第16章项目必须反过来：先让一个 split 正常执行，再让一个响应丢失，再让重试不重复提交，再让乱序响应被拒绝，再让 busy 产生背压，再让 driver 重启恢复。每一步都有一个失败路径和一个验收指标。

代码结构也要服务这个顺序。\code{MessageBus} 不需要一开始支持所有协议，但必须能确定性丢一条 reply；\code{CommitTable} 不需要一开始支持分布式存储，但必须能拒绝重复 task；\code{TraceLog} 不需要接入大型观测系统，但必须能串起 request id；\code{Metrics} 不需要漂亮 UI，但必须输出 CSV。先把证据链做出来，再扩展规模。

\topic{本章最终判断力}

读完本章，读者应能拿到一个分布式调用场景，先问七个具体问题。第一，这个请求的逻辑身份是什么？第二，重试时哪些字段不变，哪些字段变化？第三，超时后旧执行可能留下什么副作用？第四，外部效果的提交点在哪里？第五，重复响应和迟到响应如何处理？第六，下游 busy 时上游如何减速？第七，崩溃恢复时相信哪个持久化事实？

如果这七个问题都有答案，系统设计至少有了骨架。若答案是“应该不会发生”“网络库会处理”“超时就失败”“扫目录就行”，说明设计还停留在单机直觉。分布式系统的难点不在于代码行数，而在于每条失败路径都要有明确边界。

\topic{一条记录从输入到输出的完整旅程}

为了让本章不悬在概念层，再把一条日志记录的旅程完整走一遍。输入文件里有一行记录，落在 split 5 的字节范围内。Driver 不直接读取这行记录，它只知道 split 5 的元数据。Driver 选择 worker1，发送 task 5 attempt 0。Worker1 收到请求后，先打开输入文件，调整 begin offset 到行边界，顺序读取到 end offset 后的当前行结束。解析器把这一行转换成 event type 7。Map-side combine 把 event type 7 的局部计数加一。等 split 处理完，worker1 按 reduce partition 把 partial 写成临时 block，计算 checksum，然后向 driver 返回 reply。

这条旅程里有许多容易被省略的边界。第一，driver 调度的是 split，不是单条记录；所以 split 边界决定这条记录会不会被处理。第二，worker 的局部计数不是最终结果，它只是 attempt 临时输出。第三，临时 block 写完不代表 reduce 可以读取，因为 attempt 可能是旧的，也可能永远不被 commit。第四，reply 返回 driver 只是提交申请，不是提交完成。第五，driver 把 block 写入 manifest 后，reduce 才能把这条记录贡献的 partial 纳入最终结果。

若读者能沿着这条旅程指出每个状态，就会发现分布式系统的很多术语都不是凭空来的。Split 边界解决输入恰好一次的问题；attempt 解决重复执行的问题；checksum 解决输出是否完整的问题；manifest 解决哪些 block 有效的问题；deadline 解决等待多久的问题；backpressure 解决下游处理不过来时怎么办的问题。每个概念都对应这条旅程中的一个断点。

\topic{Map block 的状态机}

临时输出也需要状态机。很多重复计算 bug 不是 task 状态错，而是输出文件状态含混。一个 map block 可以有五个状态：building、sealed、reported、committed、abandoned。Building 表示 worker 正在写，reduce 绝不能读取；sealed 表示 worker 写完并计算 checksum，但 driver 还不知道；reported 表示 reply 已发出，但可能丢失；committed 表示 driver manifest 接受；abandoned 表示这个 attempt 没被接受，可以后台清理。

\begin{lstlisting}[numbers=none]
map block lifecycle:
  building   worker is writing bytes
  sealed     bytes and checksum are complete
  reported   reply has been sent to driver
  committed  manifest includes this block
  abandoned  another attempt won or task failed
\end{lstlisting}

这个状态机能解释为什么 reduce 扫目录是错的。目录里可能有 building 文件，读它会读到半截；可能有 sealed 文件，worker 写完但 driver 未提交；可能有 reported 文件，响应在路上或已经丢失；可能有 abandoned 文件，旧 attempt 输给了新 attempt。只有 committed 文件具有业务可见性。文件系统只告诉你字节存在，不告诉你协议是否接受。

状态机还影响清理策略。Building 文件在 worker 崩溃后可能残留，恢复时可以按临时命名规则清理；sealed 但未 reported 的文件可以在 worker 本地恢复后重新报告，也可以丢弃重跑；reported 但未 committed 的文件要等待 driver 决定；abandoned 文件可以延迟清理。若清理程序不知道这些状态，可能删掉仍有机会被 commit 的输出，也可能保留大量无用文件。

\topic{请求丢失和响应丢失不是同一种故障}

请求丢失时，worker 没有收到 task，通常没有业务副作用。Driver timeout 后重试，主要成本是等待时间。响应丢失时，worker 可能已经执行并写出临时输出，业务副作用已经部分发生。两者在 driver 眼里都表现为 timeout，但恢复动作不同。若教材把它们都写成“网络失败”，读者会错过关键差别。

请求丢失实验中，trace 应显示 driver send、bus drop request、无 worker start、driver timeout、retry。响应丢失实验中，trace 应显示 worker start、worker write、bus drop reply、driver timeout、retry。后者必须验证 commit table；前者主要验证 retry budget 和 deadline。两个实验的最终结果都应该正确，但触发的机制不同。

同样，worker 执行前崩溃、执行中崩溃、执行后响应前崩溃也不同。执行前崩溃没有输出，重跑即可；执行中崩溃可能留下 building 临时文件，恢复要清理；执行后响应前崩溃可能留下 sealed block，driver 不知道它存在。若 worker 本地有恢复日志，可以重新报告 sealed block；若没有，就让 task 重跑，并让旧 block 作为垃圾清理。设计要把这些位置说清，而不是统称“worker fail”。

\topic{Worker 崩溃的三个位置}

先看 worker 在读取 split 前崩溃。Driver 只会看到 timeout 或连接断开。因为 worker 没写输出，commit table 不会变化。恢复策略是 retry。这个场景最简单，但仍要受 retry budget 控制，不能无限重试同一个坏 split 或坏 worker。

再看 worker 写了一半 block 后崩溃。磁盘上可能留下一个临时文件。这个文件没有 sealed 标记，没有 checksum，没有 manifest entry。恢复时不能读它，也不能拿它补结果。后台清理可以删除它。若 worker 使用临时文件名加原子 rename，那么半写文件不会伪装成 sealed block；若直接写最终临时名，恢复程序就要用额外元数据判断是否完整。

最后看 worker 写完 block、计算 checksum、但 reply 还没到 driver 就崩溃。这个位置最微妙。数据其实完整，但 driver 不知道。系统可以选择简单策略：不恢复 worker 本地 sealed block，直接重跑 task。也可以选择优化策略：worker 重启后扫描 sealed block，重新向 driver report。优化策略能省计算，但必须保证 report 仍按原 task id 和 attempt 进入 CommitTable，不能绕过 manifest。教学项目先用简单策略也可以，但要明确代价是多重跑。

\topic{Reduce 读取 manifest 的具体步骤}

Reduce worker 不应该拿到一个目录路径就开读。它应该拿到一个 manifest snapshot。第一步，driver 发布 stage manifest，包含所有已提交 map block。第二步，reduce worker 选择自己负责的 partition，只过滤 manifest 中 \code{reduce_partition} 匹配的条目。第三步，对每个条目检查 task id、attempt、block id、byte size 和 checksum。第四步，拉取或读取 block。第五步，合并前再次记录 \code{block_id} 已处理，防止重放。第六步，合并完成后输出 reduce attempt 临时结果，再由 driver 提交最终 output manifest。

这个流程看起来比扫目录麻烦，但每一步都有故障意义。Manifest snapshot 保证 reduce 看到的是同一个提交视图；partition 过滤防止读到别人的数据；checksum 防止半写或损坏；已处理集合防止重复拉取；reduce output 也走 attempt 提交，防止 reduce 自己重试时覆盖结果。Map 和 reduce 的提交模型是一致的，只是输入输出角色不同。

\begin{lstlisting}[numbers=none]
reduce read path:
  load manifest snapshot for stage
  select entries for my reduce partition
  for each entry:
      skip if block_id was already processed
      read block bytes
      verify byte size and checksum
      merge partial counts
      record block_id as processed
  write reduce attempt output
  ask driver to commit reduce output
\end{lstlisting}

这里还有一个隐藏点：manifest snapshot 要稳定。若 reduce 一边读 manifest，一边 driver 继续追加条目，reduce 可能看到半个 stage。批处理系统通常在 map stage 全部完成后再启动 reduce，或者给 reduce 一个明确的 snapshot version。流式系统可以边产边消费，但要有 offset 和 checkpoint 语义。本章先按批处理讲，边界更清楚。

\topic{把背压接入调度器}

背压不是 worker 返回 busy 就结束。Driver 调度器要根据 busy 改变后续决策。若 worker1 连续 busy，driver 应降低它的窗口；若 partition 2 busy，map worker 应暂停发送 partition 2 的数据，但可以继续发送其他 partition；若所有 partition 都 busy，map worker 应停止读取新 split 或把 partial spill 到本地。背压信号必须能沿数据流向上游传播。

一个实用的教学策略是维护三类窗口。Worker window 控制发往某个 worker 的 task 数；partition window 控制某个 reduce partition 的 in-flight 字节；job window 控制整个 job 的总 in-flight 和总 retry。调度时先检查 job window，再检查 worker 或 partition window。任何一层满了，都不继续发送。这样能避免一个局部热点吃掉全局资源。

\begin{lstlisting}[numbers=none]
scheduling gates:
  job_in_flight_tasks < job_limit
  worker_in_flight_tasks[target] < worker_limit[target]
  partition_in_flight_bytes[p] < partition_byte_limit[p]
  retry_budget[job] has remaining attempts
  current_time < job_deadline
\end{lstlisting}

这些 gate 不是为了让程序变复杂，而是把资源边界写成代码。没有 gate，系统默认资源无限；真实机器会用内存耗尽、连接耗尽、磁盘队列爆掉来提醒你这个假设错误。高质量系统设计要主动写上限，而不是等操作系统替你失败。

\topic{调度选择要解释成本}

Driver 选择 worker 时，不能只轮询。至少要考虑数据本地性、worker 队列长度、历史延迟、失败率和当前窗口。若 split 在 worker1 本地，worker1 队列很短，就本地执行；若 worker1 队列很长而 worker2 空闲，可以远程读；若 worker2 最近频繁 timeout，就不要把新任务发过去。调度器不是追求某个绝对最优，而是在可观测成本之间做保守选择。

教学项目可以从简单评分开始。每个候选 worker 有一个 cost：本地性惩罚、队列惩罚、延迟惩罚、失败惩罚。选择 cost 最低且窗口可用的 worker。这个策略不需要复杂数学，却能让读者看到调度不是“随便找一个 worker”。更重要的是，每个惩罚都能对应指标，实验报告可以解释为什么某个 task 被调度到远端。

\topic{端到端 trace 样例}

一条成功但经历 retry 的 task trace 应该能连成一条因果链。下面的格式不是唯一的，但字段要足够稳定。

\begin{lstlisting}[numbers=none]
driver_schedule task=5 attempt=0 worker=1 reason=locality
bus_accept      request=4107 queue=worker1 depth=3
worker_start    request=4107 task=5 attempt=0
worker_seal     request=4107 block=770 bytes=4096 checksum=991
bus_drop_reply  request=4107 rule=drop_once
driver_timeout  request=4107 task=5 attempt=0
driver_retry    task=5 attempt=1 worker=2 reason=timeout
worker_seal     request=4119 block=884 bytes=4096 checksum=991
driver_commit   task=5 attempt=1 block=884 decision=accepted
reduce_read     partition=3 block=884 checksum=991
\end{lstlisting}

读者可以用这条 trace 检查本章所有概念。Request 4107 和 4119 是不同消息；task 5 是同一个逻辑任务；attempt 0 超时但不等于未执行；attempt 1 被 commit；block 770 存在但没有进入 reduce；reduce 只读 block 884。若 trace 缺少这些字段，系统出了错就无法复盘。

\topic{为什么要强调单点问题单点分析}

用户看到“分布式”三个字，很容易把所有机制混在一起讲：RPC、共识、复制、缓存、负载均衡、容错、观测性。这样的章节读起来会散，因为每个概念都只说了半页。更好的写法是单点问题单点分析：响应丢失只分析响应丢失；乱序响应只分析乱序响应；背压只分析队列水位和上游减速；driver 重启只分析持久化事实。分析完一个点，再说明它怎样接到主线。

本章所有补充都遵守这个原则。响应丢失推出 commit table，不顺便讲共识；乱序响应推出 attempt 检查，不顺便讲复制；reduce 背压推出 high watermark，不顺便讲全局调度论文；driver 重启推出 manifest 和 checkpoint，不顺便讲所有备份策略。这样读者不会在一个段落里同时接收五个新概念，也更容易把知识转成代码。

\topic{本章和真实系统的距离}

本章模型仍然简化。真实系统可能有多个 driver 或 driver 高可用，可能有对象存储而不是本地文件，可能有跨机时钟漂移，可能有认证和权限，可能有压缩、加密、版本兼容和多租户隔离。简化不是问题，问题是不能忘记简化在哪里。本章只承诺训练核心语义：身份、attempt、deadline、幂等提交、manifest、背压、故障注入和恢复。

当读者把项目迁移到真实 Linux 网络环境时，需要重新测量网络 RTT、socket buffer、磁盘 fsync、rename 语义、线程调度、CPU 占用和内存峰值。语义不变量可以沿用：一个 task 只提交一次，reduce 只读 manifest，迟到响应不能覆盖状态。性能数字不能照搬：本机 MessageBus 的 1 ms 延迟不代表真实网络，单机文件 rename 的成本也不代表对象存储 commit。区分不变量和数字，是系统工程判断力的一部分。

\topic{单点故障场景库}

为了避免本章变成“知道一堆原则但不会排查”，下面把常见故障按单点拆开。每个场景都只分析一个问题：现象是什么，朴素误判是什么，正确边界在哪里，应该看哪些指标，测试如何复现。实际系统的事故常常由多个场景叠加，但学习时必须先能把单点说清楚。若一个单点都说不清，叠加场景只会更混乱。

\topic{场景一：请求丢失}

请求丢失的现象是 driver 发出了 request，但 worker 从未出现 start trace。Driver 等到 deadline，看到 task 仍未完成。朴素误判是“worker 慢”，于是把 worker 标成坏节点。正确判断要看 MessageBus trace 或 worker 接收日志：如果没有 worker start，说明请求可能在发送路径丢失、路由错误、队列拒绝或目标不可达。此时重试是合理的，但不能无限重试。

这个场景主要验证 retry budget 和路由检查，不主要验证 commit table。因为 worker 没执行，就没有输出副作用。测试可以固定丢弃 request 9001，期望 driver 在预算内发出 request 9002。指标应显示 \code{timeouts=1}、\code{retries_started=1}、worker 端没有 attempt 0 的 start 事件。若 worker 端出现 start，却 driver 认为是请求丢失，说明 trace 关联有问题。

请求丢失还要区分目标队列 busy。若 MessageBus 返回 busy，driver 明确知道请求未入队；若 MessageBus accepted 后内部丢弃，driver 只会通过 timeout 观察到失败。两种情况都可以重试，但日志语义不同。Accepted 后丢失更像网络不可靠；busy 更像背压。把两者都写成 timeout，会让排查方向混乱。

\topic{场景二：重复请求}

重复请求的现象是 worker 收到两条相同 \code{request_id} 的消息，或者收到同一 \code{task_id} 同一 \code{attempt} 的重复投递。重复可能来自 MessageBus duplicate，也可能来自客户端在未确认 accepted 时重新发送。朴素处理是再执行一次。若 worker 只写 attempt 临时文件，重复执行主要浪费资源；若 worker 会向下游发送 partial 或写外部系统，就可能制造重复副作用。

正确边界取决于副作用位置。对本章 map worker，最少要保证 attempt 临时文件名包含 task 和 attempt，重复写不会混到别的 task。更稳的做法是在 worker 端维护短期 request table：同一 request id 已经完成，就返回同一摘要；正在执行，就拒绝或合并等待；已经过期，就按策略重新执行但不能绕过 driver commit。这个表不替代 driver commit table，它只是减少重复执行和重复响应。

测试方法是让 MessageBus 对 request 9010 执行 duplicate。Worker trace 应显示 duplicate received。若采用 worker 去重，执行次数应为一，响应次数可以为二，两个响应摘要相同。Driver 最终只 commit 一次。若不采用 worker 去重，执行次数可以为二，但两个输出都必须是 attempt 临时输出，最终 manifest 仍只接受一个。这两种设计都可以，但报告必须写清成本和边界。

\topic{场景三：重复响应}

重复响应比重复请求更常见。Worker 只执行一次，但 reply 被 MessageBus duplicate，driver 收到两次相同 request id。朴素 driver 如果每次响应都累加结果，会重复提交。正确处理是响应进入 CommitTable，第二次返回 already committed 或 late reply。这里去重键可以是 \code{request_id}，也可以是 \code{task_id} 加 attempt，最终仍要以 \code{task_id} 的 committed 状态为准。

重复响应测试应该让同一 reply 连续到达两次。期望第一条 accepted，第二条 rejected 或 ignored。指标应显示 \code{duplicate_replies=1}，\code{committed_tasks} 不变，manifest 条目不增加。若最终计数正确但 duplicate 指标为零，说明测试没有覆盖重复响应，或者系统把重复响应静默吞掉而不可观测。静默吞掉看似没错，但排查线上问题时会丢失证据。

\topic{场景四：校验失败}

校验失败的现象是 worker reply 中的 checksum 与 driver 读取到的 block checksum 不一致，或者 manifest 中记录的 byte size 与实际文件不一致。朴素处理是忽略 checksum，继续 reduce。这样会把损坏数据带入最终结果。正确处理是拒绝该 attempt 的提交，把 task 标记为 retryable failure，并记录损坏 block。若多次校验失败，可能是 worker 磁盘、编码逻辑或传输路径有问题，应隔离 worker。

校验失败不是普通 timeout。Timeout 说明本地没等到响应；checksum mismatch 说明响应到了，但输出不可信。Retry 可以解决临时损坏，但必须让坏 block 不进入 manifest。测试可以在 worker seal 后篡改 block 字节，保持 reply checksum 不变。CommitTable 应返回 checksum mismatch，task 进入 retry 或 failed，reduce 不应读取该 block。

这个场景还说明 checksum 的位置。只在网络消息上做 checksum，不足以发现磁盘临时文件损坏；只在 manifest 写 checksum，不足以发现 worker 报告前已经算错。教学项目可以先用简单 checksum，但要说明真实系统会使用更强校验和、长度校验、编码版本和可能的端到端数据摘要。目标不是追求复杂算法，而是让“输出完整性”成为提交条件。

\topic{场景五：旧 epoch 请求}

旧 epoch 的现象是请求携带的路由版本或 partition epoch 低于服务端当前版本。朴素 worker 可能仍然执行请求，导致旧路由写入新系统。正确处理是拒绝旧 epoch，并返回 stale epoch，让 driver 刷新路由或重新调度。这个场景在第17章会展开，但本章需要先知道 epoch 是防旧写的边界。

在日志统计项目里，可以模拟 reduce partition 迁移。Partition 3 原本由 worker1 负责，epoch 为 5；控制面把它迁到 worker2，epoch 变成 6；worker1 收到 epoch 5 的 shuffle 消息。Worker1 不应继续接受写入，也不应悄悄转发后仍返回 success。它应返回 stale epoch，driver 或 map worker 根据新路由重发。否则旧 worker 的迟到写可能进入旧目录，被后续错误扫描读到。

测试方法是先发送 epoch 5 消息，再切换到 epoch 6，让旧消息迟到。期望旧消息被拒绝，trace 中出现 stale epoch，manifest 只包含 epoch 6 的 block。若旧消息被接受但最终结果仍然正确，仍然要算风险，因为下一次清理或恢复可能把旧 block 引入结果。旧写被拒绝是协议不变量，不应依赖后续运气。

\topic{场景六：checkpoint 损坏}

Checkpoint 损坏的现象是 driver 重启后无法读取 checkpoint，或者 checkpoint 中 manifest checksum 不匹配。朴素处理是忽略损坏 checkpoint，从头开始或按部分内容恢复。部分恢复最危险，因为它可能相信半截 commit table，导致漏算或重复算。正确处理是 checkpoint 必须有版本、长度、checksum 和原子发布；恢复时只接受完整且校验通过的 checkpoint。若当前 checkpoint 损坏，可以回退到上一个完整 checkpoint，或者明确从头重跑。

测试可以在 checkpoint 写入一半时 kill driver，或者篡改 checkpoint 中一条 manifest entry。恢复程序应拒绝当前 checkpoint，并报告原因。若系统选择从上一个 checkpoint 恢复，必须验证上一个 checkpoint 覆盖的 stage 边界清楚。若没有上一个 checkpoint，就从 reference 输入重新开始。不能把半损坏文件“尽量读出来”，这在计算系统中比失败更危险。

这个场景还连接到文件系统语义。写 checkpoint 文件后，只调用 close 不一定代表目录项和数据都已经按预期持久化。教学项目可以不深入所有文件系统细节，但要让读者知道可靠发布通常使用临时文件、flush、rename 和必要的目录同步。不同环境语义不同，所以实验报告要写清运行文件系统和简化假设。

\topic{场景七：队列满但 CPU 很空}

队列满并不一定表示 CPU 不够。Reduce 队列满而 CPU 很空，可能是 reduce 在等待磁盘，可能是锁竞争，可能是单个热点 key 让一个 partition 串行处理，可能是下游输出提交慢。朴素判断是“加 worker”。如果瓶颈是单个 partition 或单个磁盘，加 worker 未必有用。正确做法是看分层指标：队列长度、处理速率、CPU 时间、I/O 等待、锁等待、partition 分布和 key 分布。

测试可以让 reduce 在写输出时 sleep，同时 CPU 不忙。此时 map 端收到 busy，队列水位高，但 CPU 指标低。报告应解释瓶颈是输出路径，不是计算路径。修复可以是降低 map 发送速率、增加 reduce 输出缓冲、批量写、改变输出提交频率，或者把热点 partition 拆分。若只根据 CPU 空闲继续加请求，队列会更长。

这个场景提醒读者：背压信号告诉你“下游承受不了”，但不直接告诉你为什么承受不了。原因要靠指标拆分。系统书如果只写“队列满就限流”，仍然不够；还要教读者限流后如何定位瓶颈。

\topic{场景八：平均延迟正常但尾延迟很高}

Stage 平均延迟 100 ms 看起来正常，但 P99 是 5 s。朴素报告只写平均值，会掩盖长尾。分布式计算的作业完成时间常由最慢 task 决定，尤其在 barrier stage 中，所有 map 完成后 reduce 才能开始。一个慢 task 就能拖住整批任务。正确指标要看分位数、慢 task 列表、worker 分布、split 大小和重试历史。

慢 task 分析要先区分输入大还是执行慢。若 split 17 输入字节是其他 split 的十倍，它慢是数据倾斜；若输入一样大但只在 worker2 上慢，是节点问题；若所有包含 event 7 的 split 都慢，是热点 key 或解析路径问题；若慢 task 都经历 retry，是网络或 timeout 配置问题。不同原因对应不同修复：重新切分、隔离 worker、热点拆分、调整 deadline 或 speculation。

测试可以构造一个大 split 和一个慢 worker。报告必须说明调度器如何区分两者。若只看 duration，可能把大 split 当成坏节点；若只看 worker，可能把数据倾斜当成机器慢。高质量观测要把输入规模和执行时间放在一起看。

\topic{场景九：结果正确但资源泄漏}

有些故障不会立刻让结果错误，却会慢慢拖垮系统。响应丢失后旧 attempt 的临时 block 没有进入 manifest，结果正确，但文件没有清理；重复请求被 commit table 拦住，结果正确，但 worker 重复执行消耗大量 CPU；busy 被退避处理，结果正确，但 map 端缓存 partial 越积越多。若只看最终 checksum，这些问题都会被漏掉。

因此实验报告要记录资源指标：临时文件数量和字节数、abandoned block 数、清理成功数、worker 执行次数、map 端缓存字节、spill 次数、队列峰值。正确性测试保证不算错，资源测试保证系统能长期运行。分布式系统的很多线上事故不是第一分钟算错，而是运行数小时后磁盘满、内存满或队列满。

这个场景也说明 cleanup 需要自己的指标和重试策略。清理失败不应阻塞提交，但不能永久无人管。后台清理可以扫描 manifest 之外的旧 attempt 文件，根据 age 和状态删除。清理动作也要保守，不能删除仍可能被 commit 的 sealed block。最简单规则是只清理 checkpoint 已经确认不需要的旧 attempt。

\topic{场景十：指标互相矛盾}

有时指标会看起来互相矛盾：requests sent 不高，但 worker queue 很长；timeouts 很多，但 worker 完成率也很高；busy 很少，但内存持续上涨；committed tasks 正常，但 late replies 很多。遇到这种情况，不要立刻改代码，要先检查指标定义。Requests sent 是逻辑请求还是 attempt？Queue depth 是消息数还是字节数？Timeout 是 task deadline 还是 RPC timeout？Late reply 是已提交后的响应，还是任意迟到消息？

指标定义不清，会让排查偏离。教学项目要在 README 或实验报告里定义每个指标。比如 \code{requests_sent} 统计每次 MessageBus accepted 的请求，不统计 busy 拒绝；\code{retries_started} 统计新 attempt，不统计同一 attempt 的重复发送；\code{late_replies} 统计 task 已 committed 后到达的 reply；\code{stale_replies} 统计 attempt 或 request id 不匹配的 reply。定义清楚后，指标才能比较。

这个场景的测试可以故意制造重复响应和旧 attempt 响应，检查 late 与 stale 是否按定义增加。若两个指标混在一起，虽然结果不受影响，但后续排查会困难。分布式系统的观测性不是附加功能，它是协议的一部分。

\topic{场景库的使用方式}

这十个场景不是让读者死记，而是提供排查顺序。遇到结果错误，先查 reference 和 manifest；遇到重复计数，查 commit table、reduce 输入来源和 late reply；遇到作业慢，查队列、分位数、worker 和 split 分布；遇到恢复异常，查 checkpoint 和崩溃点；遇到资源上涨，查 abandoned block、缓存和清理。每次只改变一个变量，保留 trace 和指标，再比较 reference。

真正的工程能力来自这种收敛过程。先缩小现象，再定位单点，再绑定不变量，再写测试。第17章会讲分片、复制和共识，但那些机制也要按同样方法学习。不要因为概念更高级，就放弃单点分析。系统越大，越需要把问题拆回一条消息、一个状态、一条日志和一个提交边界。

\topic{为什么本章不直接上共识}

读者可能会问：既然 commit table 和 manifest 这么重要，为什么不直接用共识协议把它们复制起来？原因不是共识不重要，而是学习顺序不能倒。共识解决的是多个节点在故障中如何对一串状态变更达成同一顺序；本章先解决的是单个 driver 如何定义状态变更本身。如果你还没说清“task 5 attempt 1 commit”到底意味着什么，把它放进 Raft 日志也只是把模糊状态复制到更多节点。

共识日志里不能放含混操作。好的日志条目应像这样清楚：在 job 7 stage 2 中，task 5 的 attempt 1 被接受，block 884 的 byte size 是 4096，checksum 是 991，reduce partition 是 3，前提是 task 5 尚未提交。这个条目有明确前置条件和结果。坏的日志条目是“worker2 完成了任务”，它没有说明任务身份、attempt、输出、提交条件和重复处理。第17章讲共识时，会继续沿用本章的 commit 语义。

\topic{为什么本章不展开复制}

复制也一样。复制 manifest 之前，先要知道 manifest 是什么、何时发布、损坏后如何恢复、旧 attempt 如何被排除。否则复制三份目录扫描结果，只会把错误放大三倍。复制解决可用性和容错，不能替你定义有效数据集合。若 reduce 仍然扫描临时目录，即使目录有三副本，重复 attempt 仍然会被读入。

本章的边界是单 driver、单提交事实来源。这个模型简单，但它让读者看清提交点。下一章把这个提交事实来源复制起来，才有意义。学习系统时经常要先把一个节点上的语义写对，再讨论多节点复制。否则读者会误以为复杂协议能弥补本地状态设计不清。

\topic{为什么本章仍然提前引入 epoch}

虽然本章不展开分片迁移，但提前提 epoch 是必要的。因为旧 attempt 和旧路由有同一个问题：旧的执行者可能在你认为它失效之后重新出现。Attempt 通过 attempt id 和 commit table 被挡住；旧路由通过 epoch 和 fencing 被挡住。读者先在 task attempt 上理解“旧消息不能改写新状态”，下一章学旧 leader、旧副本、旧路由时就不会陌生。

Epoch 不需要一开始很复杂。教学项目可以让 driver 持有一个单调递增的 partition epoch，消息携带 epoch，接收方发现旧 epoch 就拒绝。关键是让读者看到 epoch 不是时间戳，也不是日志装饰，而是权限版本。谁持有旧 epoch，谁就没有资格对新状态产生写入效果。

\topic{控制面和数据面的再次分离}

本章反复出现控制面和数据面的分离：task table、commit table、manifest、epoch、deadline 属于控制面；日志记录、partial block、shuffle 字节、临时文件属于数据面。控制面决定哪些数据有效，数据面负责产生和搬运数据。这个分离会贯穿后续章节。第17章的路由表和副本组是控制面，实际 key-value 或 shuffle block 是数据面；第18章的 stage manifest 是控制面，map 和 reduce 的大批量中间数据是数据面。

很多系统变慢或变脆，是因为控制面和数据面混在一起。把每条日志记录都写入强一致控制面，吞吐会被控制面拖死；把提交决策藏在数据目录里，恢复时又不知道哪些数据有效。正确做法是让控制面小而强，让数据面大而快。Manifest 正是这个思想在本章的具体形态：它不保存所有数据，只保存哪些数据有效。

\topic{本章给后续项目留下的接口}

为了让后续章节能继续扩展，本章项目应留下几个稳定接口。第一，\code{MessageBus} 提供 accepted、busy、deadline expired 和 invalid target 四类发送结果。第二，\code{CommitTable} 提供 accepted、already committed、stale attempt 和 checksum mismatch 四类提交结果。第三，\code{TraceLog} 至少能按 job、task、attempt 和 request id 过滤事件。第四，\code{Metrics} 能按 worker、partition 和 stage 聚合。第五，manifest 条目包含 task、attempt、partition、字节数和 checksum。

这些接口不追求一次到位覆盖真实集群，但要足够稳定。第17章加入路由 epoch 时，不应推翻 request id 和 attempt；第18章加入完整 shuffle 时，不应推翻 manifest；后续如果把 MessageBus 换成真实 socket，也不应改变 CommitTable 的不变量。好的章节之间不是互相重写，而是在前一章稳定边界上增加能力。

\topic{读者自查清单}

本章结束前，读者可以用一份自查清单判断自己是否真的理解。给定一个 task，能否说清它的 logical id 和 attempt id？给定一次 timeout，能否列出请求未到、worker 执行中、响应丢失、响应迟到四种可能？给定一个临时 block，能否判断它是 building、sealed、reported、committed 还是 abandoned？给定一次重复响应，能否说明哪个模块拒绝它？给定一次 driver 崩溃，能否说明恢复时相信 manifest 还是内存状态？给定一个 reduce 长队列，能否判断是 CPU、I/O、热点 key 还是背压配置？

如果这些问题能回答，本章主线就连起来了。如果只能背“分布式系统需要幂等和重试”，说明还停留在口号层。系统学习的关键不是记住更多词，而是把每个词放回具体路径：哪条消息、哪个状态、哪个文件、哪个指标、哪个提交点。这样进入第17章时，分片、复制和共识才会变成自然扩展，而不是又一组孤立名词。

\topic{一次完整排查演练}

最后用一次完整排查把本章收住。假设实验报告显示：最终 checksum 和 reference 不一致，event type 7 多了 1280，其他 event 大多正确。第一反应不应该是“分布式系统不一致”，而是先缩小差异。Event type 7 多，说明某些记录被重复合并，或者 reference 与分布式解析规则不同。先跑串行 reference 两次，确认输入、split 列表和输出 checksum 稳定；再让分布式系统关闭故障注入，只跑正常路径。如果正常路径一致，说明解析和 split 大体没问题，错误来自故障路径。

第二步看 manifest，而不是看 worker 目录。找到所有包含 event type 7 的 block，按 \code{task_id} 分组。如果 task 5 出现两个 committed block，说明 CommitTable 没有按逻辑 task 去重；如果 manifest 只有一个 block，但 reduce trace 读了两个 block，说明 reduce 绕过 manifest 读了临时目录；如果 manifest 和 reduce 都只有一个 block，但计数仍然多，说明 block 内部内容可能重复，回到 worker 的 split 边界和 combine 逻辑。这个顺序能把问题从“结果错了”压缩成三个明确方向。

第三步检查 task 5 的 trace。理想 trace 应能看到 attempt 0、attempt 1、两个 request id、一次 timeout、一次 commit 和一次 late 或 stale 响应。若 trace 里没有 attempt 0 的 worker start，说明不是重复执行，而可能是请求丢失；若 attempt 0 写了 block 但没有 commit，且 reduce 读了它，说明是 manifest 边界被绕过；若 attempt 0 和 attempt 1 都被 commit，说明 CommitTable 不变量被破坏；若两个 attempt 都没有被 commit，却 reduce 有输入，说明 reduce 输入来源完全错误。每一种结论都对应不同修复。

第四步看指标，验证 trace 结论。若怀疑响应丢失，应看到 \code{timeouts} 和 \code{retries_started} 增加；若怀疑乱序响应，应看到 \code{stale_replies} 或 \code{late_replies}；若怀疑背压导致重试风暴，应看到 \code{max_in_flight_tasks} 和 worker queue depth 上升；若怀疑 reduce 绕过 manifest，应看到 committed block 数和 reduce read block 数不一致。指标不是替代 trace，而是帮助你判断问题是否只有一个点，还是多个机制同时失效。

第五步写最小复现。不要直接用完整大输入调试。构造两个 split，其中 split 5 只包含 event type 7 的 1280 条记录，另一个 split 包含少量其他 event。固定故障规则：丢弃 split 5 attempt 0 的 reply。这样如果系统错误，event type 7 正好多 1280，现象和线上一致。修复后，保留这个测试作为回归。好的故障测试应该小、确定、解释力强，而不是依赖大输入随机触发。

第六步修复并复查边界。若根因是 reduce 扫目录，修复不是“过滤一下 attempt 名字”，而是让 reduce 只能从 manifest snapshot 获取输入。然后新增测试：旧 attempt 文件存在但不在 manifest，reduce 不读取；manifest 中 block checksum 错，reduce 拒绝；manifest snapshot 固定后，后续新提交不影响当前 reduce。这样修复的是边界，而不是当前一个例子。

第七步写报告。报告应包含：现象、reference 结果、故障规则、task 5 时间线、manifest 对比、reduce 读取列表、根因、不变量、修复、回归测试和未验证边界。未验证边界也要写，例如本机文件系统的 rename 语义与对象存储不同，真实网络可能有更复杂的连接断开，当前测试只覆盖单 driver。这样的报告会让后来者知道你修的是哪个失败路径，也知道哪些问题还没有证明。

这次演练展示了本章真正想教的东西：不是把“幂等、重试、背压、manifest”分别背下来，而是在一个实际错误里按顺序使用它们。Reference 定义正确结果；trace 还原因果；manifest 定义有效数据；CommitTable 保护提交不变量；metrics 检查压力和异常；故障注入把事故变成可重复测试。只要这个闭环能跑通，分布式章节就不再是空泛概念，而是一套能落地的工程方法。

\topic{质量验收：怎样判断这一章写实}

评价这一章不能只看有没有覆盖名词，而要看读者能否拿它写出一个小系统，并在小系统出错时定位问题。第一条标准是主线是否稳定。所有机制都应回到同一个日志统计作业，而不是上一段讲 RPC，下一段突然讲 CAP，再下一段跳到缓存。读者应始终知道 driver、worker、MessageBus、CommitTable 和 manifest 分别在做什么。若一个概念无法放回这五个对象的交互中，它就不应该在本章展开。

第二条标准是失败路径是否具体。只写“网络可能失败”不合格；要写请求丢失、响应丢失、重复响应、乱序响应、worker 崩溃、driver 崩溃、队列满、checkpoint 损坏分别怎样表现。每个失败都要有可观察信号：trace 中缺哪一行，metrics 中哪个计数增加，manifest 中哪个条目变化，reference 对比会怎样失败。没有观察信号，概念就无法转成工程判断。

第三条标准是不变量是否能落到代码。比如“每个 task 只提交一次”必须落到 CommitTable 的 key 和返回值；“reduce 只读有效数据”必须落到 manifest snapshot；“迟到响应不能覆盖状态”必须落到 reply handler 的检查顺序；“下游过载要让上游减速”必须落到窗口、水位和 busy 处理。若不变量只存在于文字中，代码仍然可以绕过它，这章就没有真正讲透。

第四条标准是实验能否复现。每个关键机制都要能用固定 seed 或固定 request id 触发。响应丢失不是“有时会发生”，而是 request 4107 的 reply 被丢弃；乱序不是“网络可能乱序”，而是 attempt 1 先到、attempt 0 后到；driver 崩溃不是“模拟故障”，而是在 manifest rename 前后两个位置分别 kill。可复现的实验才能形成回归测试，不可复现的实验只能形成经验故事。

第五条标准是取舍是否写清。重试提高成功概率，但会增加重复执行和过载风险；speculation 降低尾延迟，但会消耗额外 CPU 和 I/O；每次 commit 都 checkpoint 恢复快，但正常路径变慢；manifest 控制有效数据，但增加控制面发布成本；hot key split 降低单 partition 长尾，但增加最终合并阶段。高质量教材不能只说机制好，还要说机制把成本移到了哪里。

第六条标准是和后续章节的边界是否清楚。本章不需要把 Raft、再分片和完整 shuffle 全讲完，但必须把它们需要的地基打好：消息身份、attempt、deadline、commit record、manifest、epoch、trace 和 metrics。下一章讲复制时，复制的是这些清楚的状态；第18章讲工程实践时，组合的是这些已经验证过的边界。若本章把所有高级主题都浅讲一遍，后面就没有承接；若本章把一个 task 的生命周期讲透，后面扩展才有章法。

按这六条标准复查，本章的写作目标就很明确：让读者从一个真实计算任务出发，先看到朴素方案为什么会失败，再理解机制为什么出现，最后能用实验和指标证明修复有效。这样的章节才不像概念机器，也不会靠字数堆密度。它把分布式系统还原成可观察、可测试、可恢复的一组状态转换。

\topic{本章最小可实现定义}

把本章压到最小实现，可以这样定义：Driver 拥有唯一的 task table 和 commit table；Worker 只能写 attempt 临时输出，不能发布最终结果；MessageBus 可以接受、拒绝、延迟、丢弃、重复和乱序消息，但不能改变业务状态；CommitTable 是唯一能把 attempt 输出写入 manifest 的模块；Reduce 只能读取 manifest snapshot，不能扫描临时目录；Trace 和 Metrics 必须覆盖每一次发送、接收、超时、重试、提交、拒绝和 busy。

只要这七条成立，一个本机模拟系统就已经具备分布式计算的基本骨架。它还不是高可用集群，也没有完整复制和共识，但它已经能回答最关键的问题：同一个逻辑任务重复执行时，哪个结果有效；消息晚到时，谁有权改变状态；下游过载时，上游怎样减速；driver 重启时，恢复相信什么。第17章和第18章要扩展的是规模和容错层次，不应推翻这些最小定义。

因此，本章验收不看概念数量，而看这些对象能否在故障注入下守住边界。只要边界清楚，后续扩展才是在稳固地基上增加能力；边界含混时，增加节点只会放大错误。

这也是分布式学习最核心的起点。

后续所有复杂系统都从这里继续生长。

\section{本章习题}

\begin{exercisebox}
习题一：画出 task 17 的三条时间线：正常响应、响应丢失后重试、attempt 1 先返回而 attempt 0 后返回。每条时间线标出 driver task state、request id、attempt、deadline 和 commit table 的变化。

习题二：设计 \code{CommitTable}。说明它按 \code{task_id} 还是按 \code{request_id} 去重，为什么。给出 attempt 0 成功但响应丢失、attempt 1 后成功、attempt 0 迟到响应三个场景下的处理结果。

习题三：实现一个本机 \code{MessageBus} 的故障注入接口。至少支持 delay、drop、duplicate、reorder。要求每个故障能用固定 seed 或固定 request id 复现，并把注入事件写入 trace。

习题四：构造重试风暴实验。让 worker 固定 sleep，driver timeout 很短。记录请求总数、最大 in-flight、worker 队列峰值和最终失败数。然后加入重试预算、退避和抖动，比较两次结果。

习题五：为 reduce 背压设计 high watermark 和 low watermark。说明 map worker 收到 \code{Busy} 后应该等待多久，是否可以继续处理其他 partition，以及如何避免所有 map worker 同时重试。

习题六：设计 driver 重启恢复流程。Driver 在 task \code{Sent}、\code{Retrying}、commit manifest 写入前、commit manifest 写入后四个位置崩溃。分别说明恢复后 task 状态、临时输出和重复响应如何处理。
\end{exercisebox}

\begin{labbox}
实验：在本机实现 driver、三个 worker 和 MessageBus。输入是一组日志 split，输出是 event 计数。先跑正常路径，再分别注入响应丢失、重复响应、乱序响应、慢 worker、reduce busy 和 driver 重启。最终结果必须与串行 reference 一致，同时输出 trace 和 metrics CSV。实验报告必须包含朴素版本、失败现象、修复机制、指标对照和未验证边界。
\end{labbox}
