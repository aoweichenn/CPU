\chapter{计算系统全景}

第一册已经说明程序如何变成进程、虚拟地址、调用约定和机器指令。第二册继续追问：当一个程序需要利用多个核心、多个内存层级、多个线程甚至多台机器时，计算能力从哪里来，又会在哪里损失。这个问题不能只从代码出发，也不能只从硬件手册出发。真正的计算系统由硬件、操作系统、运行时、算法和通信协议共同构成。

\section{从一个计算任务到完整系统}

第一册的学习对象是一段程序如何成为一个可观察的执行实例。读者已经知道源文件、目标文件、链接、ELF、进程、虚拟地址、ABI、汇编指令和基础测量之间的关系。第二册的学习对象不是再把这条路径复述一遍，而是把这个执行实例放进更复杂的资源环境：多个核心同时执行，多个缓存层级同时保存数据，多个线程共享状态，多个任务争用队列，多个节点通过网络交换消息。

这意味着本册每次遇到第一册知识时，都要换一个问题问。看到汇编，不再只问“这条指令做什么”，还要问“它处在依赖链上还是可以并行发射”；看到虚拟地址，不再只问“它属于哪个映射”，还要问“这个访问是否会击穿 TLB、是否发生首次触页、页面是否放在远端 NUMA 节点”；看到 \cmd{perf stat}，不再只问“数字如何记录”，还要问“这些事件能否区分前端、后端、内存带宽、锁等待和调度噪声”。

\begin{keyidea}
第一册把程序执行讲清楚，第二册把可扩展计算讲清楚。前者关心“程序如何被机器执行”，后者关心“多个执行实体怎样在硬件、操作系统和网络边界内协同前进”。
\end{keyidea}

\topic{从一条指令到一个系统}
一条加法指令看起来很简单，但它并不是孤立执行的。核心要取指、译码、重命名寄存器、等待操作数、选择执行端口、写回结果并提交状态。如果操作数在寄存器里，成本很低；如果来自 L1 cache，延迟增加；如果来自 L3、远端 NUMA 节点或磁盘，程序看到的时间会跨越几个数量级。并发程序还要考虑另一个核心是否正在写同一条 cache line，分布式程序还要考虑消息是否跨越网络。

因此本册把计算系统分成五层。第一层是硬件执行，包括核心、缓存、TLB、内存控制器和互连。第二层是操作系统，包括调度、页表、系统调用、I/O 和计数器。第三层是运行时，包括线程池、任务队列、同步原语和内存分配器。第四层是算法，包括归约、扫描、分区、图遍历和负载均衡。第五层是分布式协议，包括 RPC、复制、共识、shuffle 和检查点。

\begin{keyidea}
高性能不是某一层单独决定的。硬件给出能力边界，操作系统分配资源，运行时组织执行，算法决定数据依赖，分布式协议处理通信和故障。
\end{keyidea}

\topic{四种尺度}
计算系统至少有四种尺度。第一种是单条指令尺度，讨论延迟、吞吐、依赖、端口和分支预测。第二种是单个核心尺度，讨论流水线、乱序窗口、寄存器重命名、load/store queue 和 SIMD。第三种是单台机器尺度，讨论缓存层级、TLB、NUMA、线程调度、锁、原子、队列和 I/O。第四种是集群尺度，讨论 RPC、分片、复制、共识、shuffle、检查点和故障恢复。

这四种尺度不能混用。单条指令尺度上，减少一条指令可能有效；单机多线程尺度上，减少一条指令可能完全被锁等待淹没；集群尺度上，减少一百万条指令也可能抵不过一次跨机重试。工程判断必须先确认当前瓶颈属于哪个尺度。第二册的核心训练之一，就是把“慢”定位到正确尺度，再选择对应证据。

\section{成本、证据与瓶颈迁移}

分析性能时，不能只看“执行了多少行代码”。更稳定的单位是指令数、周期数、访存字节数、cache miss、TLB miss、分支误预测、同步等待时间、上下文切换、系统调用次数和网络往返延迟。不同单位之间不能随便相加，但它们共同解释程序为什么慢。

例如，一个数组求和循环的算术量很小，主要成本是把数据从内存送到核心。一个矩阵乘如果分块得当，会重复使用缓存中的数据，瓶颈可能转向浮点执行端口。一个共享队列如果每次 push 都抢同一把锁，瓶颈不是算术，也不是内存带宽，而是临界区串行化和 cache line 所有权迁移。

\topic{把成本转成证据}
第二册不鼓励凭经验猜瓶颈。每个成本单位都应该有可观察证据。依赖链问题可以用汇编、优化报告、IPC 和不同累加器版本比较；分支问题可以用输入分布、branch-misses 和 branchless 版本比较；cache 问题可以用工作集大小、访问步长和 cache 事件比较；TLB 问题可以用页数、大页和 dTLB 事件比较；锁问题可以用等待时间、futex、context switches 和吞吐曲线比较；NUMA 问题可以用 first-touch、绑定策略和远端访问计数比较；分布式问题可以用 trace、队列长度、重试次数和超时分布比较。

证据的作用不是一次性证明真理，而是排除错误方向。若一个归约程序在线程数增加后不再加速，可能原因很多：内存带宽到顶、false sharing、锁竞争、线程迁移、任务粒度太小、NUMA 放置错误、CPU 频率下降或 benchmark 把初始化混入热路径。没有证据时，这些解释都像对；有了证据，才能逐步缩小范围。

\topic{工作负载决定系统形状}
同一台机器在不同工作负载下会呈现完全不同的系统形状。一个大数组顺序求和主要考验内存带宽、预取和 TLB 覆盖；一个哈希表查询主要考验随机访问、分支、缓存 miss 和分配器布局；一个日志写入服务主要考验队列、批处理、I/O 延迟和背压；一个并行图遍历主要考验负载均衡、前沿大小、随机邻接访问和同步开销。若不先描述工作负载，就谈不上性能分析。

工作负载至少要写清五件事。第一，输入规模和数据分布。例如数组长度、key 的倾斜程度、图的顶点度分布、请求大小分布。第二，读写比例。例如只读查询、读多写少、写多读少、读写混合。第三，局部性形状。例如顺序扫描、固定步长、随机访问、热点访问、分层访问。第四，依赖形状。例如每个元素独立、前缀依赖、图依赖、任务依赖、跨节点依赖。第五，失败和等待形状。例如是否会阻塞 I/O，是否会超时重试，是否会遇到队列满，是否会有节点失败。

经典教材式写法不应把“程序”当成孤立对象，而应把程序放在 workload 里看。一个函数在小输入上很快，不代表在大输入上快；在均匀 key 上扩展良好，不代表在热点 key 上可靠；在单线程上 cache 命中高，不代表多线程不会互相驱逐；在本机多进程模拟正常，不代表真实网络环境下重试和乱序没有影响。第二册后续所有实验都必须先写 workload，再写实现，再写测量。

\topic{瓶颈会迁移}
优化不是把一个程序永久变快，而是把瓶颈从一个位置推到另一个位置。一个标量循环最初可能受长依赖链限制；使用多累加器后，依赖链缓解，瓶颈可能变成加载端口；再使用 SIMD 后，指令数下降，瓶颈可能变成内存带宽；再多线程后，瓶颈可能变成 NUMA、LLC 争用或内存控制器；再把任务分布到多机后，瓶颈可能变成 shuffle、热点 key 或检查点。每次优化都应该重新判断瓶颈，而不是沿用上一次结论。

瓶颈迁移也是很多性能争论的根源。一个人说“这个优化没用”，另一个人说“这个优化提升很大”，可能他们测的是不同阶段的程序。在内存带宽已经打满的版本上减少几条整数指令，确实不会明显加速；在还受依赖链限制的版本上，多累加器可能非常有效。没有上下文的优化建议通常没有意义。

因此本册要求性能报告写出“优化前瓶颈假设”和“优化后瓶颈是否迁移”。例如优化并行归约时，可以先写：单线程版本 IPC 较低，怀疑依赖链；四累加器版本 IPC 提高但带宽接近上限；多线程版本 8 线程后不再加速，怀疑内存带宽或 NUMA；按 NUMA 节点初始化后带宽改善，说明原先包含远端访问因素。这样的叙述才像工程分析，而不是只列快了多少。

\topic{系统边界和责任边界}
计算系统由多个层次组成，但工程修改必须有责任边界。硬件给出指令、缓存、TLB、互连和计数器；操作系统负责调度、地址空间、页表、I/O 和权限；运行时负责任务、队列、线程池、内存分配和同步封装；算法负责依赖、切分、合并和数据结构；业务协议负责语义、重试、幂等和恢复。性能问题经常跨层出现，但修复不能没有边界。

例如，一个服务尾延迟高，可能是锁竞争、页错误、磁盘抖动、网络重试、GC 或业务热点。排查时可以跨层收集证据，但修复时要明确到底改哪一层。如果把业务幂等问题硬塞进线程池，线程池会变得难以复用；如果把 NUMA 亲和性写死在算法库里，库就难以适配不同机器；如果把所有指标都放在业务日志里，底层运行时就无法单独诊断。好的系统设计会让层次之间有清晰接口，同时允许证据跨层关联。

本册后面讲线程池时，会强调 submit、shutdown、wait 的语义；讲无锁结构时，会强调线性化点和内存回收；讲分布式系统时，会强调请求 ID、重试预算和提交点。这些内容看起来分散，其实都在回答同一个问题：系统的责任边界在哪里，失败时谁负责恢复，性能结论应归因到哪一层。

\topic{延迟、吞吐和并发度}
系统分析经常混淆三个词：延迟、吞吐和并发度。延迟是单个操作从开始到结束的时间，吞吐是单位时间完成多少操作，并发度是同时在系统中飞行的操作数量。一个内存 load 的延迟可能很高，但如果核心能同时挂起多个 miss，整体吞吐仍然可以接受；一个网络请求延迟很高，但如果客户端保持足够多的 in-flight 请求，吞吐可以接近带宽上限。

这三个量之间有一个朴素但非常有用的关系：要填满一个延迟很高的管道，需要足够并发度。磁盘、网络、内存和 CPU 后端都可以用这个心智模型理解。区别在于，CPU 的并发度来自乱序窗口、load buffer、SIMD lane 和多核心；网络系统的并发度来自连接数、请求队列和异步 I/O；分布式计算的并发度来自分片、任务和流水线。

\begin{mentalmodel}
延迟回答“一个操作要等多久”，吞吐回答“系统每秒能做多少”，并发度回答“为了达到吞吐，需要同时容纳多少未完成工作”。
\end{mentalmodel}

\topic{Amdahl 和 Gustafson}
多核性能讨论离不开可并行比例。Amdahl 定律说明，如果程序有一部分必须串行执行，那么线程数增加到一定程度后，加速会被串行部分限制。Gustafson 定律从另一个角度看问题：如果随着核心数增加而扩大问题规模，可并行部分也随之增加，系统仍然能获得更高总吞吐。

这两个定律不是考试公式，而是工程判断工具。一个服务端请求路径中，如果日志锁、全局分配器、单分片元数据或单线程事件循环占比很高，增加 worker 不会线性加速。一个离线计算任务如果可以把输入切成更多分片，让每个节点处理独立数据，再做树形合并，扩大规模时就更接近 Gustafson 的场景。

\topic{观察系统的三张图}
学习计算系统时，建议每遇到一个程序都画三张图。第一张是数据流图：输入从哪里来，经过哪些结构，输出到哪里。第二张是执行图：哪些线程、任务或节点执行哪些阶段，哪里并行，哪里等待。第三张是成本图：每个阶段主要消耗 CPU、内存、I/O、锁等待还是网络。

这三张图可以避免把问题混成一团。一个队列积压，可能是消费者计算慢，也可能是生产者太快，也可能是下游 I/O 阻塞，也可能是队列锁竞争。只有先画清数据和执行边界，测量才知道该放在哪里。

数据流图应标出数据的所有权和复制点。数组是只读输入，还是会被多个线程写回；任务队列里保存的是值、指针还是共享对象；网络请求里发送的是完整数据、增量还是引用；文件映射里的页面是否会被多个进程共享。所有权不清，优化很容易破坏正确性。比如把对象从共享指针改成裸指针可能减少引用计数开销，但如果生命周期没有重新证明，就可能引入悬垂访问。

执行图应标出并行区、串行区和等待区。很多程序源码看上去“用了线程”，执行图却显示所有线程都在等待一个队列、一把锁或一个单线程提交阶段。执行图还要标出调度关系：任务由谁提交，由谁执行，是否允许嵌套，是否可能在 worker 内等待 future，是否可能被取消。没有执行图，线程池和异步代码很容易变成看不见的控制流。

成本图应标出主成本和次成本。一个阶段可以同时消耗 CPU 和内存，但报告要判断主导因素。若主成本是内存带宽，减少日志字符串格式化不一定有用；若主成本是锁等待，改 SIMD 不会解决；若主成本是网络重试，优化本地解析函数意义有限。成本图不是精确模型，而是让团队讨论同一个对象。

\section{语义边界、状态和失败}

并发和分布式计算最容易出现的错误，是把速度放在正确性之前。数据竞争可能让程序偶尔错；错误的内存序可能只在某些 CPU 上触发；无锁结构如果没有处理对象回收，可能在压力测试里才出现悬垂访问；分布式系统如果没有处理超时和重试，可能在网络抖动时重复提交请求。

本册每次讨论优化，都要先给出语义边界。共享变量由谁写，谁读，什么时候发布，什么时候回收。任务是否可以重排，结果是否依赖顺序。消息是否幂等，故障后是否可以重试。只有这些问题说清楚，性能数字才有意义。

\topic{从单机到分布式的一条线}
很多读者会把并发和分布式分成两门课：前者讲线程，后者讲网络。第二册刻意把它们放在一条线上，是因为它们面对的是同一种基本矛盾：多个执行实体共享某些状态，但共享状态越多，协调成本越高。在单机上，协调成本表现为锁、原子、cache line 迁移、调度和内存序；在集群中，协调成本表现为 RPC、复制、共识、重试、超时和数据迁移。

因此，一个好的单机设计原则常常能迁移到分布式系统。线程本地计数再合并，对应 map-side combine；分片锁对应按 key 分区；树形归约对应分层聚合；有界队列和背压对应限流和排队控制；线性化点对应协议中的提交点；对象生命周期对应消息幂等和状态恢复。反过来，分布式系统也会提醒单机程序：只要协调成本足够高，就必须减少共享、批量通信、明确失败语义。

\topic{贯穿项目：Compute Systems Engine}
本册需要一个贯穿全书的大项目，否则知识会散成很多孤立技巧。这个项目命名为 Compute Systems Engine。它的目标不是做一个商业框架，而是用足够小、可测试、可测量的工程，把现代计算系统的核心问题串起来：数据怎样放置，线程怎样执行，任务怎样调度，同步怎样证明，队列怎样背压，失败怎样恢复，分布式阶段怎样保持幂等。

项目的第一条主线是计算内核。最开始只有顺序归约和并行归约，但它们不是为了演示“多线程加速”这么简单。顺序归约用于建立 reference 和测量基线；多累加器版本用于解释依赖链和乱序执行；线程本地 partial 用于解释 false sharing 和共享写；NUMA 分块初始化用于解释 first-touch；树形合并用于连接后面的分布式聚合。一个小小的归约，在第二册里要承担从核心流水线到集群聚合的教学任务。

项目的第二条主线是运行时。先实现一个 mutex 和 condition variable 保护的 bounded queue，用它讲清锁保护的不变量、等待谓词、关闭语义和背压。然后实现固定线程数线程池，用它讲清 submit、wait、shutdown、异常传播和任务粒度。再加入任务图和工作窃取，用它讲清依赖、ready 队列、本地性和负载均衡。最后把队列和线程池用于分布式模拟，让读者看到单机运行时和分布式执行器之间的相似性。

项目的第三条主线是并发数据结构。先比较 mutex counter、atomic counter 和 sharded counter，解释原子热点和 cache line 所有权迁移。再实现 SPSC ring buffer，解释 head/tail 单写者、acquire/release 和环形容量边界。之后再讨论 MPMC 队列、ABA、hazard pointer、epoch 和 RCU，不让读者一上来就陷入复杂无锁代码。每一步都要有 reference、有压力测试、有错误注入。

项目的第四条主线是分布式计算模拟。它可以先在本机多进程或多线程中完成，不依赖真实集群。输入按分片读取，worker 本地聚合，shuffle 按 key 重分布，reduce 合并输出。随后加入重试、超时、重复请求、幂等提交、检查点和故障注入。这个阶段把前面所有原则重新组合：局部性变成数据本地调度，锁热点变成热点 key，背压变成限流，线性化点变成提交点，内存回收变成状态恢复。

\begin{mentalmodel}
贯穿项目的每一次扩展，都必须同时回答三个问题：语义是否正确，性能瓶颈在哪里，失败时怎样恢复。只回答其中一个，都不是系统工程。
\end{mentalmodel}

\topic{数据移动守恒}
计算系统里最容易被低估的成本，是数据移动。一个整数加法可能只需要很少周期，但把数据从内存、磁盘或网络移动到执行单元，往往比加法昂贵得多。很多优化本质上不是“少算”，而是“少搬”“近处搬”“成批搬”“只搬热数据”。缓存分块减少从内存反复搬同一块数据；map-side combine 减少跨网络发送重复 key；压缩用更多 CPU 换更少 I/O；冷热分离避免把调试字段搬进热循环。这些看似不同的技巧，其实都遵守同一条原则：昂贵移动必须服务足够多的有效工作。

这条原则也能解释为什么某些算法复杂度相同，实际性能差很多。两个 \complexity{O(n)} 算法，一个顺序扫描数组，一个随机追踪指针，它们搬动的数据量和等待形状完全不同。两个 \complexity{O(n \log n)} 算法，一个在连续数组上排序，一个不断分配小节点，它们对 cache、TLB 和分配器的压力也不同。第二册不是替代算法复杂度，而是把复杂度放回真实机器里理解。

数据移动还有尺度差异。L1 到寄存器很快，远端 NUMA 内存慢很多，NVMe 更慢，跨机网络又是另一套延迟和带宽模型。一个设计如果在 L1 尺度上节省几次加法，却在网络尺度上多发一次请求，通常得不偿失。反过来，分布式系统为了减少网络字节，常愿意在 map 端多做本地聚合。成本尺度决定优化优先级。

\topic{延迟阶梯}
学习现代计算系统时，建议把常见操作放到延迟阶梯里。寄存器访问、L1 命中、L2 命中、LLC 命中、DRAM、本地 NVMe、网络 RTT、远端存储，它们相差可能是数量级。具体数字会随硬件变化，但阶梯关系长期稳定。高性能代码的很多设计，就是尽量让热路径停留在更低阶梯，把高阶梯操作批量化、异步化或移出关键路径。

延迟阶梯不能单独看，还要看吞吐。一次 DRAM 访问延迟高，但内存系统能同时处理多个请求；一次网络请求延迟高，但多个 in-flight 请求可以填满带宽；一次磁盘写延迟高，但顺序大批写能获得高吞吐。延迟回答单个操作要等多久，吞吐回答系统单位时间能做多少，二者需要并发度连接。这个心智模型会在内存级并行度、异步 I/O 和分布式 RPC 里反复出现。

读者要避免一个常见错误：看到某个操作延迟高，就认为它一定是瓶颈。若高延迟操作很少发生，或者被流水线隐藏，它可能不是主成本；若低延迟操作发生在所有元素上，并且阻塞关键路径，它也可能成为瓶颈。性能分析的对象不是单次操作，而是整个 workload 下的频率、依赖和并发度。

\topic{状态、数据和控制}
系统设计可以拆成状态、数据和控制三类对象。状态描述系统当前处于什么阶段，例如队列 open 还是 closed，任务 pending 还是 committed，分片属于哪个 epoch。数据是被计算处理的主体，例如数组元素、日志记录、shuffle block。控制是让状态和数据按规则流动的机制，例如调度器、锁、路由表、提交协议。

很多混乱来自把三者混在一起。把每条大数据记录都写入共识日志，是把数据面塞进控制面；把线程池的关闭语义隐藏在任务函数里，是把控制状态塞进业务数据；把请求去重只放在日志文本里，是把协议状态变成不可查询的副作用。清晰系统会让控制面小而可靠，让数据面高吞吐，让状态有明确持久化和恢复边界。

单机系统同样如此。环形队列的 head、tail、size 是控制状态，buffer 里的任务是数据，mutex 和条件变量是控制机制。分布式 shuffle 中，manifest 是控制状态，中间文件是数据，driver 的 commit protocol 是控制机制。把这种对应关系看清，后面学习锁、无锁、任务图和分布式提交时会更容易。

\topic{不确定性从哪里来}
系统越往右侧尺度推进，不确定性越多。单线程程序的执行顺序大体由代码决定；多线程程序会受到调度、缓存一致性、内存序和锁竞争影响；异步 I/O 会受到设备完成时间、队列长度和取消路径影响；分布式系统会受到网络延迟、消息重复、节点故障和时钟漂移影响。工程设计不能假设这些不确定性不存在，而要把它们纳入语义。

不确定性有两种处理方式。第一种是消除，例如用锁建立互斥，用固定合并树保证浮点归约顺序，用 route epoch 拒绝旧请求。第二种是容忍，例如用幂等处理重复请求，用重试预算处理临时失败，用误差范围处理浮点差异，用背压处理流量波动。优秀系统会明确哪些不确定性必须消除，哪些可以容忍，哪些只能观测和降级。

这也是为什么正确性先于性能。一个不确定性没有被定义的系统，性能数字越漂亮越危险。并发程序偶尔丢任务，分布式作业偶尔重复计数，I/O 管线偶尔在关闭时 use-after-free，这些问题都不能用平均吞吐掩盖。第二册每个章节都会先写语义边界，再写性能优化。

\topic{软件接口如何影响硬件}
软件接口会决定硬件能看到什么模式。一个函数如果每次只接受单个对象，硬件只能看到零散访问；一个函数接受连续 span，编译器和预取器才有机会处理批量数据。一个 API 如果允许输入输出重叠，编译器必须保守；一个 API 明确只读输入和独占输出，就更容易向量化和并行化。一个运行时如果把所有任务都塞进全局队列，硬件会看到共享 cache line 争用；一个运行时使用本地队列和分层合并，硬件会看到更多本地访问。

因此，性能不是最后用几行特殊指令补上的。类型、容器、所有权、错误返回、任务粒度、消息格式和提交协议都会影响底层行为。高质量系统教材必须把接口设计和硬件成本放在同一张图里。第一册讲过 C++ 对象和内存，本册继续追问：这些对象如何穿过核心、缓存、线程、队列和网络。

\topic{从小实验到大系统}
本册很多实验看起来很小：数组求和、计数器、环形队列、bounded queue、parallel scan。它们之所以重要，是因为每个小实验都代表一个大系统中的核心矛盾。数组求和代表数据移动和依赖链；计数器代表共享写热点；环形队列代表发布和消费；bounded queue 代表背压；parallel scan 代表无冲突输出位置；shuffle 代表跨节点数据重排。

读者做实验时，不要只得到“这个版本快”或“那个版本慢”的结论，而要把实验抽象成系统原则。计数器分片成功后，要想到热点 key；bounded queue 阻塞成功后，要想到服务过载；SPSC ring buffer 正确后，要想到生产者消费者模型的适用范围；manifest 去重正确后，要想到外部效果 exactly-once。小实验是可控环境，大系统是同一原则的组合。

\topic{本册的阅读方式}
第二册不适合只顺着文字看。每章至少要做三件事。第一，画图：数据流、执行图和成本图。第二，写 reference：无论是算法还是协议，都先写最简单正确版本。第三，跑对照：改变一个变量，观察结果是否支持假设。只看概念不做实验，容易把系统知识学成口号；只写代码不画图，容易在复杂并发和分布式路径里迷路；只跑 benchmark 不写语义，容易优化错误程序。

本册也不要求读者一开始就拥有大服务器。手机 Termux、普通 Linux 笔记本或云主机都可以完成大部分语义实验。没有 NUMA 设备时，就写清不能验证远端内存；没有硬件计数器权限时，就用输入规模、时间和源码结构做替代；没有多机时，就用本机多进程模拟消息乱序和重试。证据边界要诚实，但学习不必等待完美环境。

\topic{一个请求的全链路视角}
为了把本册的层次放在一起，可以跟踪一个日志统计请求。用户提交一个作业，driver 读取作业配置，切分输入文件，创建 map task。Worker 线程从队列取到 task，读取文件页，解析文本，写入列式 batch，做本地 histogram，把 partial 放入 shuffle 队列。Reduce worker 读取 partial，合并 key，写输出临时文件，driver 提交 manifest，最后返回作业完成。

这条链路里，每一步都对应本册一类知识。读取文件涉及 page cache、短读、I/O 背压；解析文本涉及分支、SIMD 字节扫描和数据布局；本地 histogram 涉及 cache line、线程本地 partial 和合并；队列涉及 mutex、条件变量、关闭和背压；线程池涉及任务粒度和调度；shuffle 涉及分片、热点 key 和队列容量；manifest 涉及提交点、幂等和恢复。一本系统书的价值，就在于让读者能把这些层次连成一条链。

当这条链路变慢时，也要沿链路分析，而不是随便挑一个熟悉点优化。若输入读取慢，改 SIMD 没用；若 reduce partition 热点，改 parser 没用；若 manifest fsync 慢，增加 worker 可能让等待更严重；若队列容量太大，吞吐看似高但尾延迟失控。全链路视角能防止局部优化伤害整体。

\topic{成本账本}
系统工程可以像记账一样记录成本。每处理一条记录，需要读多少字节，解析多少字符，写多少热字段，更新多少本地桶，发送多少 shuffle 字节，最终写多少输出。每个阶段还要记录固定成本：任务调度、队列入出、批次头部、manifest 条目、checkpoint fsync。可变成本和固定成本共同决定最佳 batch 大小和任务粒度。

成本账本不需要一开始完全精确。先做粗估：每条记录 40 字节文本，解析后热字段 24 字节，本地聚合后每个 key 16 字节，shuffle 压缩后多少字节。再用实际指标校正。若估算和测量差异巨大，说明有隐藏成本，例如字符串分配、写放大、cache miss 或重试。成本账本是把直觉变成可检查假设的方法。

\topic{语义账本}
性能账本之外，还要有语义账本。每个数据项要知道来源、所有者、可变性、提交点和恢复方式。输入 split 来自文件 offset，只读；ParsedLogHotBatch 由 parse stage 生成，由 compute stage 读取，batch 生命周期由队列管理；ShuffleBlock 由 map attempt 生成，只有 driver commit 后有效；OutputBlock 由 reduce attempt 生成，manifest 发布后对外可见。语义账本能防止重复处理和悬垂访问。

很多 bug 都能归因到语义账本缺失。一个 batch 被下游异步使用时，上游复用了缓冲；一个 attempt 输出已经失败，却被 reduce 读到；一个队列关闭后 producer 仍然写入；一个配置对象发布后旧版本被释放。把所有权和提交点写清，是性能系统的安全网。

\topic{全书复用的六个问题}
遇到任何新系统，都可以问六个问题。第一，数据在哪里，按什么布局存放。第二，谁执行，执行实体是线程、任务、进程还是节点。第三，谁共享，哪些状态被多个执行实体读写。第四，谁等待，等待发生在锁、I/O、队列、网络还是依赖。第五，谁提交，什么时候外部可以认为结果有效。第六，谁恢复，失败后从哪里继续。

这六个问题会在后文反复使用。数组内核回答数据和执行；锁章回答共享和等待；I/O 章回答等待和背压；分布式章回答提交和恢复；贯穿项目把六个问题同时回答。读者如果能把任何系统拆成这六类，就已经掌握第二册的主线。

\topic{学习路线和阶段目标}
第一阶段，读者应能解释一个单线程循环为什么慢。这里的关键词是依赖链、分支、cache line、TLB、预取和编译器优化报告。不要急着上线程，先把单核执行模型看清。第二阶段，读者应能把一个循环扩展到多线程，并说明任务切分、partial、false sharing、线程绑定和扩展曲线。第三阶段，读者应能写出 bounded queue、线程池和任务图，说明关闭、等待、异常和取消。第四阶段，读者应能把单机 pipeline 扩展到本机分布式模拟，处理 request id、重试、manifest 和 checkpoint。

每个阶段都有明确交付物。单线程阶段交付一个 reference 和性能报告；多线程阶段交付扩展曲线和 partial 布局说明；运行时阶段交付线程池状态机和测试；分布式阶段交付故障矩阵和恢复演练。这样学习不会变成泛泛读书，而是形成可运行、可测量、可解释的项目。

\topic{常见误区总览}
第一，看到 CPU 使用率低就认为要开更多线程，但真实瓶颈可能是 I/O、锁等待或背压。第二，看到 atomic 就认为比 mutex 快，但热点 atomic 仍会造成 cache line 迁移。第三，看到 SIMD 就认为一定更快，但内存带宽和数据布局可能才是瓶颈。第四，看到队列积压就扩大队列，但这可能只是延迟和内存风险后移。第五，看到重试能提高成功率就无限重试，结果放大故障。第六，看到复制就以为安全，却没有备份和恢复演练。

这些误区贯穿全书。每当读者想套用一个简单答案，都应回到证据：当前瓶颈是什么，语义要求是什么，失败路径是什么，成本迁移到哪里。系统工程很少有永远正确的局部技巧，只有在上下文中成立的设计。

\topic{从面试知识到工程知识}
很多读者熟悉算法复杂度、操作系统概念和 C++ 语法，但进入工程系统后会发现问题变得模糊。一个 \complexity{O(n)} 扫描可能慢在 TLB，一个线程安全队列可能慢在 false sharing，一个正确 RPC 可能因为重试风暴拖垮服务，一个分布式作业可能因为响应丢失重复计数。第二册要做的事情，就是把这些看似分散的工程问题放回统一框架：数据移动、共享状态、等待关系和提交语义。

面试知识常把问题边界给好：输入在内存里，函数一次调用，失败不存在，线程不存在。工程知识需要自己定义边界：输入来自哪里，能否失败，谁拥有内存，是否并发，结果何时提交，重复执行怎么办。不是面试知识没用，而是它只覆盖系统的一部分。本册希望读者能把算法题里的思维扩展到真实计算系统。

\topic{系统设计审查表}
读完本册后，面对一个新模块，可以按审查表提问。数据层面：输入规模多大，字段冷热如何，是否需要列式布局，是否有稳定 ID，是否需要序列化。执行层面：任务粒度多大，线程数如何配置，是否可能阻塞，是否存在嵌套并行。同步层面：哪些状态共享，谁写谁读，锁保护什么不变量，原子是否发布其他数据，是否需要回收协议。I/O 层面：队列是否有界，写入何时持久化，超时后底层完成如何处理。分布式层面：请求如何去重，提交点在哪里，旧路由如何拒绝，恢复从哪个 checkpoint 开始。

这个审查表不是流程负担，而是防止遗漏。系统 bug 常常不是因为工程师不知道某个概念，而是设计时忘了问某个问题。比如写了线程池但没有关闭语义，写了 manifest 但没有校验，写了重试但没有幂等，写了 SIMD 内核但没有 scalar reference。审查表把这些经验变成可重复动作。

\topic{模块成熟度}
一个模块可以按成熟度分层。第一层是能跑通 happy path。第二层是有 reference 和边界测试。第三层是有性能指标和对照实验。第四层是有失败注入和恢复。第五层是有文档、审查表和可运维指标。很多教学代码停在第一层，本册要求至少把核心模块推进到第四层。只有这样，项目才像计算系统，而不是演示程序。

成熟度也能帮助安排工作顺序。不要在第一层还不稳定时做复杂优化；不要在没有 reference 时引入多线程；不要在没有幂等提交时开启 speculation；不要在没有指标时调调度器。工程推进需要台阶，每个台阶都要能验证。

\section{从朴素程序推导系统结构}

先不要急着给读者一串概念。我们从一个最朴素的问题开始：写一个程序，读取一批日志，按事件类型统计总数，最后输出结果。第一反应很自然：打开文件，逐行读取，解析字段，更新一个 map，最后把 map 写出去。这个程序也许几十行就能跑通。它为什么后来会长成“计算系统”？不是因为工程师喜欢把事情复杂化，而是因为朴素程序在真实约束下会一步步暴露问题。

第一个问题是结果怎么证明正确。小文件可以人工看输出，大文件不行；单线程一次跑完可以相信过程，多线程和重试之后就不能只看“程序没崩”。于是我们需要 reference：最简单、最慢也可以接受，但语义必须清楚的版本。Reference 的出现不是教材规定，而是从“我怎么知道优化后没算错”这个问题自然推出来的。没有 reference，后面讲并行、SIMD、分布式恢复都没有锚点。

第二个问题是输入并不总是干净。某行少字段怎么办，事件类型越界怎么办，数值溢出怎么办，没有换行结尾怎么办，文本编码坏了怎么办。朴素程序常把这些情况藏在 if 里，写到后来调用者不知道错误行是跳过、计入错误、还是终止作业。于是“数据合同”出现了：它不是抽象术语，而是为了回答“什么输入算合法，什么输入算错误，错误怎样进入结果”而产生的。数据合同一旦明确，后面的布局和算法选择也会随之改变。若事件类型保证是小范围整数，可以用数组 histogram；若 key 是任意字符串，就要考虑哈希、排序或字典编码。

第三个问题是程序变慢。逐行读取、逐行解析、逐次更新 map 在小文件上没问题，大文件上会暴露 I/O、分配、哈希、cache miss 和同步成本。我们尝试批量读取，批量解析，批量聚合。此时又出现一个新边界：哪部分负责控制任务，哪部分负责处理数据。负责切分输入、记录 task 状态、决定重试和提交的逻辑，和真正扫描字节、解析字段、累加计数的逻辑，不应该混在一条热循环里。于是“控制面”和“数据面”不是凭空来的，而是从“每条记录都背着调度和提交逻辑跑太慢、太难恢复”这个失败中推出来的。

第四个问题是资源会被撑爆。读得太快，解析队列越来越长；解析太快，写出队列堵住；队列无限增长，内存迟早爆。最简单的程序没有“背压”这个词，它只是一路 push。只有当下游跟不上，上游又继续生产，系统才迫使我们提出背压：队列必须有界，满了以后生产者要等待、失败、丢弃低优先级数据或降级处理。背压不是概念清单里的项目，而是从“无限队列把问题推迟到内存崩溃”中生长出来的。

第五个问题是失败。单机顺序程序中途失败，通常重新跑；但输入很大时，全部重跑成本高。多线程程序中一个任务失败，其他任务可能已经写了 partial；分布式程序中一个 attempt 超时，远端可能其实已经执行完，只是响应丢了。于是我们需要提交点、manifest、checkpoint 和幂等。它们共同回答一个问题：哪些结果已经对外生效，哪些只是临时产物，失败后从哪里继续，重复执行会不会重复计数。没有失败压力，就不会自然需要这些概念；有了失败压力，它们就是系统能否可信的核心。

这样，一条主线就清楚了：从“读取日志并统计”出发，先需要 reference 证明结果，再需要数据合同处理脏输入，再需要控制面和数据面拆分提高吞吐和可恢复性，再需要背压防止资源失控，最后需要提交和恢复语义处理失败。第二册后面的章节不是并列概念，而是沿着这条主线展开：硬件章节解释为什么数据面会慢；同步章节解释控制状态怎样安全变化；运行时章节解释任务怎样调度；I/O 章节解释背压和完成；分布式章节解释提交、重试和恢复。

\topic{从模糊目标推出可观测指标}
继续沿着这个日志作业看。用户说“要快一点”，这句话没有工程意义。快是总吞吐高，还是第一条结果快，还是尾部任务少抖动，还是失败后恢复快？如果目标不清，优化方向会互相打架。把 batch 做大，吞吐可能提高，但尾延迟变差；checkpoint 更频繁，恢复更快，但写 I/O 增加；输出排序固定，可测试性更好，但合并自由度下降。于是我们需要把模糊目标拆成可观测指标。

吞吐目标对应处理记录数、字节数和各阶段耗时；延迟目标对应排队时间、执行时间、写出时间和分位数；资源目标对应内存峰值、队列长度、in-flight 请求数；恢复目标对应失败注入后是否重复、是否丢失、从哪个 checkpoint 恢复。指标不是最后贴到系统上的监控装饰，而是从“快一点到底是什么意思”这个问题推出的度量语言。

有了指标，实验才有章法。若 parse queue 持续增长，说明解析或下游计算跟不上读取；若 write queue 持续增长，说明输出慢；若 CPU 很空但队列很多，说明 worker 可能阻塞在 I/O 或锁；若吞吐上升但 p99 变坏，说明批处理或排队牺牲了尾延迟。每个指标都对应一个下一步问题，而不是孤立数字。后续章节讲 perf、线程池 trace、I/O completion 和分布式 driver 指标，都是这套推导的延伸。

\topic{为什么先把单机讲透}
现在再问：既然日志很大，为什么不一上来就分布式？因为分布式不是把线程换成机器那么简单。单机里一次函数调用变成分布式里的 RPC，一次锁等待变成远端协调，一次内存状态更新变成多副本状态转移。延迟被放大，失败形态被放大，状态管理也被放大。如果单机里还说不清 batch 的所有权、队列关闭、输出提交点和失败恢复，搬到多机只会把混乱放大。

因此本册先把单机计算讲透。数组求和不是为了求和本身，而是为了看到依赖链和内存带宽；bounded queue 不是为了写一个容器，而是为了理解背压和关闭；parallel scan 不是为了刷算法题，而是为了给并行输出分配唯一位置；manifest 不是为了多写一个文件，而是为了定义提交点。单机阶段把这些问题讲清，分布式阶段才是在扩大尺度，而不是换一套玄学。

这也解释了本册和第三册 AI 的关系。AI 推理引擎最终也会面对同样问题：权重如何布局，算子如何批处理，线程如何调度，输出何时有效，模型文件如何加载，错误如何恢复。第二册先讲通用计算系统，不急着讲 AI，是为了让读者知道 AI 算子开发不是孤立魔法，而是数据移动、并行执行、内存层级和运行时调度的特殊组合。

\section{学习路径、实验与贯穿项目}

全书实验可以按能力排列。硬件阶段做单累加器、多累加器、分支预测和链表对照。内存阶段做 AoS/SoA、冷热分离、TLB、mmap 和 page fault。单机高性能阶段做自动向量化、histogram、top-k、stream compaction。同步阶段做 bounded queue、atomic counter、SPSC ring 和 hazard/epoch 设计。运行时阶段做线程池、任务图、work stealing 和 trace。I/O 阶段做批处理、背压、fsync 和 completion。分布式阶段做 request id、重试、manifest、checkpoint 和热点 key。

这张总表能防止学习碎片化。每个实验都不是孤立练习，而是下一阶段的前置。比如 stream compaction 的 scan，会在 shuffle offset 中复用；bounded queue 的 close，会在线程池 shutdown 中复用；request id 的幂等，会在 task attempt commit 中复用。

\topic{如何判断自己学会了}
判断是否真正学会第二册，不是看能否复述术语，而是看能否面对一个慢系统提出可验证假设。比如一个日志作业慢，你能否画出数据流和执行图；能否说出是读取、解析、聚合、shuffle、reduce 还是提交慢；能否设计对照实验；能否说明失败后如何恢复；能否写出哪些指标缺失。若能做到这些，就说明你已经从“知道知识点”进入“能做系统分析”。

另一个判断标准是能否拒绝不合适的优化。别人建议“换无锁队列”，你能问生产者消费者模型是什么、关闭语义是什么、瓶颈是否真在队列；别人建议“上 SIMD”，你能问数据是否连续、是否受内存带宽限制、是否有 reference；别人建议“无限重试”，你能指出幂等和重试预算。能拒绝错误优化，是成熟工程能力的一部分。

这一章的回归阅读应改成一条记录的系统账本。选一行输入日志，给它分配稳定的 \code{record_id}，再沿着 parser、batch、kernel、queue、writer、manifest 逐段追踪。每一段都写清三个问题：谁拥有这份数据，谁可以修改它，失败后从哪里恢复。这样全景章不是概念地图，而是读者第一次看到“身份、所有权、成本、提交”怎样同时出现。

账本还要把控制面和数据面分开。数据面关心字节如何移动、字段如何进入热路径、partial 如何合并；控制面关心任务身份、错误分类、重试预算和提交记录。两者不能互相替代。只看数据面，系统容易跑得快但恢复不可信；只看控制面，系统会有很多状态表却没有吞吐。全景章的任务就是让读者从一开始知道两个面都必须存在。

本章验收不靠大段总结，而靠三张图。第一张是数据身份图，从 input split 到 output manifest；第二张是执行等待图，标出线程、队列和阻塞点；第三张是成本证据图，标出 CPU、内存、I/O 和重试。若读者画不出这三张图，后面章节的优化就会失去坐标。若三张图能互相对应，第二册的主线就已经立住了。

全景章还要确定全书的最小命名系统。输入侧使用 \code{record_id} 和 \code{split_id}，执行侧使用 \code{stage_id}、\code{task_id} 和 \code{attempt_id}，输出侧使用 \code{block_id} 和 \code{manifest_id}，观测侧使用 \code{trace_id}。这些名字不是为了显得正式，而是为了防止后续章节各说各话。缓存章讨论 batch 时要能引用同一条记录，线程章观察 worker 时要能关联同一次 attempt，分布式章处理迟到响应时要能判断它对应哪个提交边界。如果这些身份在第一章没有定下来，后面每章都会临时造词，书稿就会散。

读者做完本章项目时，应能手写一份最小账本：一条日志从哪个文件偏移进入，经过哪个 batch，在哪个 worker 上被处理，生成哪个 partial，写成哪个 block，被哪个 manifest 接纳。如果某一步只能写“某个函数处理了它”，说明边界仍然停留在调用栈，而不是系统对象。后面所有优化都必须回到这份账本：优化可以改变布局、线程和执行顺序，但不能让身份和提交语义消失。

这份账本还要说明哪些对象属于数据面，哪些对象属于控制面。数据面对象包括输入字节、热字段列、partial、shuffle block 和最终输出；控制面对象包括任务状态、队列水位、重试预算、错误摘要和提交表。很多系统设计混乱，根源就是把控制面信息塞进数据结构，或者让数据面函数偷偷修改调度状态。第一章先把这条边界画出来，后面章节讨论锁、I/O、重试和 manifest 时才不会互相污染。

全景章最后可以安排一个很小的手工实验：用同一份十行日志，手动写出每行的 \code{record_id}、解析结果、错误状态和最终计数。然后人为插入一行坏数据和一次重复提交，观察账本中哪些字段能阻止错误扩散。这个实验不追求性能，却能让读者提前看到系统书写的基本姿态：先定义对象，后讨论优化；先保证可追踪，再追求可扩展。

\section{把全景图变成可执行的学习项目}

全景章节最容易写成名词地图：核心、缓存、线程、队列、网络、协议，每个词都看起来重要，却没有告诉读者为什么这些词必须同时出现。更稳的写法，是从贯穿项目的一条日志记录出发，把它经过的每个边界都写清楚。记录先以字节形式停在文件或内存映射里，随后被解析成热字段，进入批次，进入本地统计，进入线程间队列，进入分区，进入临时输出，最后进入提交表。每过一个边界，系统都要回答一次相同问题：数据还归谁所有，状态是否已经发布，失败后能否重做，成本主要来自计算还是移动。

这条路径能把第二册的层次自然串起来。核心流水线解释解析循环为什么受分支和依赖影响；缓存和 TLB 解释为什么同样是 \complexity{O(n)} 扫描，连续数组和指针链表差距巨大；NUMA 解释为什么多线程初始化和执行线程位置不能随便换；测量章节解释为什么只看总耗时无法定位瓶颈；布局章节解释为什么日志对象不能把热字段、原始文本、调试字符串和临时状态混在一起；同步章节解释为什么队列关闭和任务完成必须写成状态机；分布式章节解释为什么本机函数调用变成消息以后，超时、重试和提交都要变成显式协议。

\topic{第一张图：数据的身份和所有权}

学习系统时，第一张图不是 CPU 结构图，而是数据身份图。以日志统计为例，输入字节有文件偏移和长度，解析后的记录有 batch id 和行号，本地聚合结果有 worker id 和 partition id，shuffle block 有 attempt id 和 checksum，最终输出有 manifest entry。每个身份都服务一个工程问题。文件偏移让失败后能重新读取同一段输入；batch id 让 trace 能把解析耗时和后续计算关联起来；attempt id 让重复执行不覆盖已提交结果；checksum 让 driver 不只相信一条成功消息。

没有身份，系统会退回到“这个对象现在在内存里，所以我知道它是谁”的单进程直觉。这个直觉在简单程序里没问题，但进入线程池、异步 I/O 和分布式模拟后就会失效。一个 batch 可能在队列里等待，一个 task 可能被取消，一个 worker 可能返回迟到响应，一个输出文件可能已经写出但还没有提交。身份图把这些状态分开，后面的章节才有地方放锁、原子、背压和提交协议。

身份还决定测试方式。串行 reference 可以只比较最终结果；并行和分布式项目必须比较中间身份是否稳定。例如同一个 input split 失败后重试，逻辑 task id 不应改变，attempt id 应递增，旧 attempt 的临时文件不能进入 manifest。若测试只看最终计数，许多重复提交、迟到响应和清理泄漏不会被发现。全景章要提前建立这个观念：系统正确性不是最后一行输出偶然相等，而是每个边界上的身份、状态和提交都能解释。

\topic{第二张图：执行实体和等待点}

第二张图是执行图。记录由哪个线程解析，由哪个 worker 聚合，由哪个队列传递，由哪个 reduce 任务合并，哪里可能阻塞，哪里允许取消，哪里必须等待所有前驱。执行图不是把源码函数列出来，而是把正在消耗资源或等待资源的实体画出来。一个函数调用可能只是把任务入队，真正执行在另一个线程；一个 future 看起来像值，实际可能让 worker 阻塞；一次异步写提交很快，真正完成事件稍后才到。

执行图的价值在排查。若吞吐不增长，执行图能区分是所有 worker 都在跑但内存带宽到顶，还是大部分 worker 在等队列锁，还是 reduce 端被一个热点 partition 拖住。若尾延迟很高，执行图能区分是任务粒度太大、I/O completion 抖动、后台 flush 影响，还是 driver 提交阶段串行。没有执行图，优化往往会落到熟悉的点上：会 SIMD 的人改 SIMD，会锁的人换锁，会网络的人调重试。但系统慢不一定慢在你熟悉的层。

本册后续每章都要复用执行图。讲流水线时，执行实体是一个核心里的指令；讲 NUMA 时，执行实体是不同节点上的线程和内存控制器；讲任务运行时时，执行实体是 worker、ready queue、deque 和 continuation；讲分布式时，执行实体是 driver、worker、MessageBus 和持久化目录。名称变了，问题不变：谁在跑，谁在等，谁拥有下一步，谁能取消等待。

\topic{第三张图：成本账本和证据链}

第三张图是成本账本。它把“慢”拆成可测的项：读取多少字节，解析多少字符，触发多少分支，访问多少 cache line，写入多少队列节点，发生多少锁等待，发送多少 block，提交多少 manifest entry。成本账本不要求一开始精确到周期，但要求每一项都有来源。若一条记录平均 80 字节，解析后只需要 event type 和用户桶，那么热路径就不应搬动整条原始字符串。若本地 combine 能把 100 万条记录压成几千个 key 计数，那么网络或 shuffle 前的聚合价值就能被估算。

证据链是成本账本的验证方式。源码推理告诉我们可能减少字节移动，perf 事件告诉我们 cache miss 是否变化，trace 告诉我们等待是否迁移，端到端结果告诉我们语义是否保持。任何单一证据都不够。只看源码，容易忽视编译器和硬件；只看 perf，容易丢掉业务语义；只看最终耗时，无法解释为什么变快或变慢。全景章要让读者从第一天就知道：高性能工程不是一条 benchmark 曲线，而是“假设、改动、指标、反例、边界”的链条。

贯穿项目的第一份报告可以很简单：实现串行日志统计 reference，记录输入字节数、记录数、key 数、输出条数和总耗时；再实现一个多线程版本，但只允许改变任务切分方式，不允许同时改变解析和布局。这样读者会看到，实验变量一旦失控，结论就会失控。后续章节再逐步加入多累加器、列式 batch、线程本地 partial、有界队列和分布式提交，每一步都能回到这份账本上比较。

\topic{从学习顺序到工程顺序}

全书的学习顺序不是“先硬件、再软件、再算法”这么机械。真实项目里，这几层互相反馈。数据布局决定硬件能否顺序访问；硬件缓存行为决定线程数增加后是否仍然扩展；线程池语义决定算法能否安全拆成任务；算法切分决定分布式 shuffle 的数据量；分布式提交又反过来要求本地输出有稳定身份和校验。因此每章虽然有主角，但都要回到同一条链路，而不是讲完一个概念就换下一个。

工程顺序也要克制。第一步先写正确 reference。第二步写最小可测版本。第三步只改变一个结构，例如数据布局、partial 合并、队列容量或重试策略。第四步收集指标和反例。第五步决定是否保留复杂结构。很多教材直接给最终方案，读者看不到为什么需要它；本册必须保留朴素版本和失败路径，让概念从问题里长出来。

本章最后给读者一个自查标准：能否拿一个陌生系统，画出数据身份图、执行等待图和成本账本；能否说明哪些结论来自第一册，哪些是第二册新增；能否指出哪个边界发生失败后由谁恢复；能否写出最小 reference 和第一组对照实验。能做到这些，再进入后续硬件和并发细节，知识才不会散。

\topic{案例走查：一次作业为什么必须跨层解释}

假设 Compute Systems Engine 处理 8 GiB 日志，用户看到吞吐只有预期一半。全景章要求读者不要立刻改代码，而是沿链路提出候选假设。输入阶段可能慢在 page cache 未预热或 split 边界处理；解析阶段可能慢在分支预测和字符扫描；统计阶段可能慢在热点 key 或线程本地 partial 合并；运行时可能慢在任务粒度过细；I/O 阶段可能慢在写出队列背压；分布式模拟可能慢在重试和 manifest 提交。

这不是把所有原因都列一遍，而是给出排查顺序。先确认 reference 输出和输入规模，防止测错数据；再看阶段耗时，找最大项；再在最大项内部看硬件事件或队列状态；最后只改一个变量。若阶段耗时显示 70% 时间在写出等待，优化解析 SIMD 没意义。若写出等待是因为 block 太小导致系统调用多，增加线程数也没意义。跨层解释的价值，是让每个优化动作有位置。

这类案例也适合作为每章开头复习。后续任何章节讲完后，都回到这条作业链问：本章新增了哪个可观测指标，能排除哪类错误解释，给项目增加了哪个可验证能力。这样全书不是 18 章并列知识，而是一条逐步增强的工程路径。

\topic{全景章的源码阅读路线}

全景章的源码阅读不追求一次读深，而是建立映射表。用户态先看项目自己的 driver、batch、queue、task、manifest 这些对象；系统态再看它们对应到 Linux 的哪些边界。进程和线程对应 task，虚拟地址和 mmap 对应 VMA，文件读写对应 page cache 和 inode，等待队列对应 futex 或 I/O wait，网络消息对应 socket buffer。读者只要能把一个用户态现象映射到一个内核对象，就已经跨过了“只会背概念”的门槛。

建议先做四栏笔记。第一栏写用户态动作：提交任务、读文件、入队、等待、写临时文件、提交 manifest。第二栏写项目对象：TaskNode、BoundedQueue、ParsedLogBatch、ShuffleBlock、CommitEntry。第三栏写可能的 Linux 对象或工具入口：task、VMA、page cache、futex、block I/O、socket。第四栏写可观测指标：队列水位、page fault、context switch、writeback、retry count。这个表会在后续章节反复填充，越往后越细。

全景章的实验也要简单但完整。第一天只写串行日志统计和一份报告，不急着多线程。报告里写输入生成方式、reference checksum、总字节数、记录数、key 分布、错误行数量、运行时间和未验证边界。第二天把同一程序拆成 parser、compute、writer 三个阶段，但仍然串行运行，只增加阶段耗时。第三天才引入队列或线程。这样读者能看到系统结构如何逐步出现，而不是一开始就掉进复杂并发代码。

\topic{全书项目的目录和命名约定}

项目仓库建议按能力分层：\filepath{engine/core} 放 id、status、error 这类基础类型，\filepath{engine/io} 放 split、reader、writer，\filepath{engine/runtime} 放 queue、thread pool、task graph，\filepath{engine/kernels} 放 parse、histogram、partition，\filepath{engine/driver} 放 task table、retry、manifest。书中每章新增的能力都要落到这些目录中的一个，不要散落到单个大文件。

命名也要服务教学。逻辑身份叫 \code{task_id}，物理尝试叫 \code{attempt}，消息交互叫 \code{request_id}，提交记录叫 \code{commit_entry}。不要今天叫 job，明天叫 work，后天叫 unit。系统书的连贯性来自稳定词汇。若一个词在单机和分布式里含义不同，就要换名或加限定，例如 worker thread 和 worker node 不要混用。

本章最后的作业可以要求读者写一页设计草案：画出三张图，列出目录结构，给出第一份 reference 报告。代码可以很少，但边界要清楚。没有这一步，后续章节再深入硬件或并发时，读者会缺少承载知识的工程对象。

\topic{综合练习：把一条记录写成系统账本}

这一章的综合题不要求读者优化任何代码，而是要求读者把一条日志记录从输入到输出的全部状态写成账本。账本第一列写数据形态：原始字节、解析字段、热字段 batch、本地 partial、partition block、manifest entry、最终输出。第二列写所有者：文件系统、parser、compute worker、partitioner、I/O worker、driver、reduce worker。第三列写可见性：只是本地临时状态，已经进入队列，已经写入临时文件，还是已经提交到 manifest。第四列写失败后动作：重读、重算、丢弃、清理、重试或拒绝旧响应。

这个题的价值在于逼迫读者把“系统”从抽象词变成可检查对象。很多初学者以为系统设计就是画框图，但框图无法回答 batch 何时能释放、旧 attempt 的输出能不能被 reduce 读、driver 重启后从哪里恢复。账本可以回答这些问题。若某一行写不出所有者，就说明生命周期不清；若写不出可见性，就说明提交点不清；若写不出失败动作，就说明恢复语义不清。

完成账本后，再要求读者写三条端到端不变量。第一，最终输出只来自 manifest 中已提交的 block。第二，同一个逻辑 task 最多有一个 attempt 对外可见。第三，任何队列关闭后都不会再接受新业务对象，但已经进入 drain 范围的对象要么完成，要么被明确取消。三条不变量看似简单，却会贯穿锁、原子、I/O 和分布式章节。

最后让读者对账本做一次“故障插针”。任选一行插入失败：parser 遇到坏行，compute worker 抛异常，I/O worker 短写，driver 响应丢失，reduce 校验失败。每个失败都要说明上游看到什么、下游是否继续、状态表如何变化、最终输出是否受影响。这个练习能让全景章从目录式介绍变成系统思维训练。

\topic{容易出错的边界：哪些系统问题不能只靠一层解释}

全景章最后还要建立反例索引。第一个反例是“CPU 使用率低所以加线程”。如果低使用率来自 I/O 背压、队列满、下游提交慢或线程阻塞在 future，增加线程只会增加排队和内存。第二个反例是“单线程热循环快所以系统会快”。热循环快不代表队列、写出、合并和恢复路径快。第三个反例是“本机模拟成功所以分布式语义正确”。本机共享对象如果没有消息身份、attempt、deadline 和 commit table，就没有覆盖远端调用失去的保证。

第四个反例是“输出正确一次所以并发正确”。并发错误往往依赖交错，单次运行不能证明没有数据竞争、迟到响应、重复提交和关闭竞态。第五个反例是“平均吞吐高所以体验好”。任务系统和 I/O 系统常常死在尾部，p95 和 p99 可能暴露队列积压、慢 worker、热点 key 或 checkpoint 抖动。第六个反例是“源码清楚所以机器执行清楚”。编译器可能内联、展开、向量化或保守拒绝优化，硬件可能慢在分支、TLB、cache line 和互连，而不是源码看起来最多的语句。

这些反例不是额外知识点，而是全书的护栏。每当后续章节引入一个技术，读者都要问它修补了哪个反例，又可能制造哪个新反例。比如列式布局修补无效字节搬运，但可能制造生命周期和调试复杂度；线程本地 partial 修补共享写热点，但可能制造合并阶段长尾；异步 I/O 修补线程阻塞，但可能制造 buffer 生命周期和完成错误传播；重试修补临时失败，但可能制造重复提交和重试风暴。能同时看到收益和新风险，才算进入系统层。

因此全景章的最终输出可以是一张“问题到层次”的文本表。慢在热循环，先看指令、分支、缓存；慢在多线程扩展，先看共享写、NUMA、调度；慢在管线，先看队列、水位、背压；慢在分布式阶段，先看身份、重试、提交和恢复。文本表不需要画成图片，放在 EPUB 里也能稳定阅读。它会成为读者学习后续章节的导航。

\topic{本章核心接口：统一 ID、统一 trace、统一报告}

全景章给项目留下三个接口。第一个是统一 ID。所有阶段都使用 job、stage、task、attempt、partition、block 这些稳定身份，不允许某个模块另起一套临时名字。第二个是统一 trace。每个阶段都能输出 begin、end、status、bytes、records、checksum 和 error summary。第三个是统一报告。无论是硬件实验、队列实验还是分布式实验，都要写输入、环境、版本、指标、结论和边界。

这三个接口会让后续章节连起来。流水线章把热循环指标写进报告，缓存章把字节移动写进报告，线程章把任务状态写进 trace，分布式章把 request id 和 attempt 写进同一条 trace。若第一章不统一这些对象，后面每章都会变成单独 demo。

\topic{从全景进入硬件细节的过渡}

全景章结束时，读者应带着一个具体问题进入下一章：同一段代码为什么在机器上表现出不同速度。不要把下一章理解成硬件百科，而要把它看成全景账本中的第一段成本解释。指令、分支、依赖、缓存和内存并不是为了让读者背参数，而是为了回答项目里第一批热循环为什么快或慢。只要记住这一点，后面的硬件章节就不会和工程项目脱节。

本章也给后续章节设了一个质量门槛。任何新增机制都必须能回到数据身份、执行等待和成本账本。若某段文字只介绍概念，却不能说明它改变了哪条数据路径、哪个等待点或哪项成本，就应该重写。这个门槛能帮助整本书保持连贯，避免变成系统名词堆叠。

\topic{全书阅读检查点：每章都要回答的八个问题}

为了防止后续学习散掉，读者可以在每章结束时回答八个问题。第一，本章处理的是哪一种具体失败或成本，不允许只写概念名。第二，本章最朴素的实现是什么，它为什么在小输入或 happy path 下合理。第三，压力输入、并发交错、硬件边界或故障路径怎样击穿这个朴素实现。第四，本章引入的机制修补了哪个缺口。第五，它引入了什么新成本，例如空间、延迟、复杂性、调试难度或可移植性。第六，项目中哪个对象承载这个机制。第七，用什么实验、指标或 trace 验证它。第八，哪些结论在当前设备上没有验证。

这八个问题会让每章形成同一种思考方法，但不会变成同一种文字模板。流水线章回答的是依赖和分支，缓存章回答的是地址序列，锁章回答的是不变量，I/O 章回答的是完成事件，分布式章回答的是消息身份。方法统一，内容必须各自具体。读者写学习笔记时，也可以按这八个问题组织，而不是摘抄定义。

全景章还要求读者把每章接口放进项目主线。没有 HotKernelReport，硬件优化没有证据；没有 MemoryShape，布局优化没有对象；没有 QueueContract，背压语义不统一；没有 TaskStateRecord，任务运行时不可诊断；没有 MessageEnvelope，分布式模拟不可信。接口不是为了形式化，而是为了让知识在代码里留下抓手。

如果某章读完后只能回答“本章讲了 cache/TLB/锁/原子/分布式”，说明学习还停在标题层。如果能回答“本章让我把 ParsedLogBatch 的热字段拆成 span，并用 MemoryShape 记录工作集页数，再用 BenchmarkCase 验证随机访问和连续访问差异”，说明知识已经落到工程对象。整本书要追求后者。

\topic{从一条指令到一个系统}

日志统计作业从输入文件走到 manifest 的全过程中，从一条指令到一个系统首先表现为对象边界不清：记录身份、阶段状态、执行实体、成本账本和提交边界里至少有一个对象的身份、所有者或状态没有写明。 \topic{瓶颈会迁移}

日志统计作业从输入文件走到 manifest 的全过程中，瓶颈会迁移首先表现为对象边界不清：记录身份、阶段状态、执行实体、成本账本和提交边界里至少有一个对象的身份、所有者或状态没有写明。 \topic{观察系统的三张图}

日志统计作业从输入文件走到 manifest 的全过程里，观察系统的三张图的出发点是串行语义：输入怎样变成输出，输出位置由谁决定，合并顺序是否影响结果。 \topic{作业账本的第二轮细读}

本节继续把 日志统计作业 拆开。前面已经建立了基本概念，现在要进一步追问：写出后崩溃导致重复 partial 被合并 时，旧程序为什么失去解释能力，哪些对象必须独立出来，哪些实验能证明改进有效。这里不新增目录层级，只把本章最容易含糊的点逐个压实。

\topic{为什么一条记录必须有稳定身份}

先把问题落到一个具体动作上。为什么一条记录必须有稳定身份，不是为了多记一个术语，而是为了处理 写出后崩溃导致重复 partial 被合并 这样的具体场景。如果仍沿用 逐行读取、直接更新一个 map、最后扫描目录输出，小输入可能看不出问题，可一旦出现并发、失败、倾斜或资源等待，状态就会散落在调用栈、临时文件、日志文本或线程局部变量里。本章要求读者把这些状态写回 split、record、attempt、manifest、trace，让后续观察和恢复有共同语言。


C++ 代码里的表达要跟这个推导一致。身份、长度、版本、计数和偏移使用位宽明确的整数；只读数据用视图；拥有者和观察者分开；状态用枚举或结构化记录。测试不能只走正常路径，还要覆盖旧输入、压力输入、取消、重试、关闭或恢复。若这个点不能进入测试和报告，它就还只是文字解释。

实验报告要能独立阅读。输入摘要、命令、reference、关键 trace、资源指标、错误注入和未验证边界都要保留。源码或工具阅读则围绕 perf 事件入口、调度状态和页缓存提交路径 展开，只解释与当前现象相关的路径。完成后，本章留下的材料应能自然交给下一章的 热循环证据包，而不是在章末断掉。

\topic{一次作业事故的完整复盘写法}
复盘一开始不要写“系统发生异常”，而要写清作业的外部合同：输入文件包含多少条记录，串行 reference 的输出是什么，最终 manifest 应该包含哪些结果。然后按时间线记录：driver 创建 split，worker 领取 attempt，parser 产出 batch，aggregator 生成 partial，writer 写临时文件，driver 发布 manifest。每个动作都要对应一个可以查询的状态，而不是只出现在日志文本中。

真正有价值的是把错误路径写出来。假设 worker 在临时文件写完后、manifest 发布前退出，恢复程序应该看到一个未提交的文件；它可以保留诊断，也可以清理，但不能让 reduce 自动读取。假设响应在 manifest 发布后丢失，重试程序应该返回已提交摘要，而不是再写一份结果。这样复盘后，读者会自然理解为什么提交点、attempt 和输出身份是系统基本对象。

这类复盘也能约束后续优化。如果某次优化把 partial 直接写到最终目录，却没有 manifest 状态，就算吞吐提高，也破坏了恢复语义；如果某次优化扩大 batch，却让单条记录无法追踪错误位置，也要退回重新设计。第一章的系统账本不是文档任务，而是后续所有性能修改的闸门。

\topic{实验课：把一次作业拆成可审计状态}
这节实验只处理一个很小的输入文件，但要把状态写完整。输入包含三十条日志，字段包括时间、用户、事件、数值和一段可选文本。先写串行 reference：遇到格式错误的行，错误计数加一；遇到未知事件，归入 unknown；所有有效记录按事件计数；最终输出按事件名排序。reference 不追求快，只负责定义答案和错误语义。

朴素系统随后把输入分成三个 split，每个 split 一个 worker，worker 直接写 partial 文件，driver 最后扫描目录合并。第一次运行不注入故障，输出应和 reference 一致。第二次在某个 worker 写完 partial 后立即终止进程，再重启作业。若 reduce 扫描目录，它会把旧 partial 和新 partial 一起读到，重复计数就会出现。这个失败信号非常清楚：算法没有错，错误来自“文件存在”被误当成“结果已提交”。

改进版引入最小账本。每个 split 有 split id；每次执行有 attempt id；每个 partial 写入临时路径；只有 driver 验证 checksum 后，manifest 中才出现 committed 条目。Reduce 只读 manifest，不扫描目录。重启后，未提交临时文件只进入诊断列表，不进入结果。实验验收时，报告要同时展示 reference 输出、旧系统重复输出、新系统 manifest、旧临时文件清单和最终输出 checksum。

这节课的关键不是 manifest 这个名字，而是把“外部可见”从“中间产物已经出现”里拆出来。读者完成后应能回答：哪一步是数据生成，哪一步是控制提交，哪一步允许重做，哪一步必须幂等。这个能力会直接迁移到后面的 I/O、分布式消息和最终毕业项目。

\topic{系统账本练习验收重点}
合格答案要先写串行 reference，再写作业身份表。身份表至少包含输入 split、record id、stage、attempt、输出状态和错误状态。答案不能只给一段流程描述，必须用一个失败输入说明旧程序为什么不能恢复。最后给出恢复后 manifest 的文本摘要，证明 reduce 只读已提交结果。

答案还要写“为什么不采用更复杂方案”。例如没有证据证明队列是瓶颈，就不应把锁队列换成无锁；没有证明输出顺序可变，就不应随意重排；没有证明错误路径能恢复，就不应只提交正常路径。能解释不做什么，说明读者开始具备工程判断。

最后，答案必须包含可复查材料。至少有输入摘要、关键状态、期望输出、实际输出、运行命令和未验证边界。没有这些材料，老师或同伴无法判断结论是否成立。第二册的作业不是让读者写漂亮段落，而是训练读者把系统问题变成可验证证据。

\topic{项目实现：计算系统全景}
本章对应的贯穿项目实现不要求一次写成完整框架，但必须有最小可运行切片。这个切片包含三个部分：一个 reference，一个带本章机制的实现，一个能击穿 reference 之外隐含假设的测试。Reference 用来固定语义，机制实现用来展示本章知识，测试用来证明新增复杂度确实解决了问题。

代码组织上，不要把所有内容放进一个大文件。基础类型、输入生成、核心机制、报告输出和测试应分开。即使当前只是教学项目，也要让目录能表达责任边界。这样后续章节接入时，不会因为找不到状态归属而重新发明一套对象。

每个实现都要输出一段机器可读摘要。摘要至少包含输入规模、关键状态、错误数、输出校验、耗时和未验证边界。这里的机器可读不要求复杂格式，简单的 key value 文本也可以。关键是让脚本和读者都能复查，而不是只能看屏幕截图。

测试应包含正常路径、边界路径和失败路径。正常路径证明功能可用；边界路径证明长度、容量、尾部、空输入、关闭或超时没有遗漏；失败路径证明本章机制不是装饰。若失败路径写不出来，往往说明本章概念还没有落到工程对象上。

报告最后写一个取舍段。新增机制带来了什么好处，增加了什么成本，哪些场景不应该使用它，下一章会复用哪个字段或对象。这个取舍段非常重要，因为系统工程不是把所有高级机制都加进去，而是在证据和约束下选择合适的边界。

\topic{调试笔记：作业结果不一致时先查提交边界}
当最终计数和 reference 不一致时，第一步不要改聚合代码，而是列出所有对外可见的输出。临时文件、局部 partial、manifest 条目和最终结果要分开。若 reduce 读取了没有 manifest 的文件，说明提交边界失效。若 manifest 中有两个 attempt 指向同一 split，说明去重表失效。若错误记录没有 record id，说明输入身份失效。这个排查顺序能防止读者把系统状态错误误判成算法错误。


\topic{复查清单：计算系统全景}
复查本章时，先检查 reference 是否仍然存在。没有 reference，后面的并发、优化、异步或分布式改造都无法证明语义没有变。再检查压力输入是否尖锐：它应当攻击本章最核心的边界，而不是只把数据量放大。第三，检查状态是否可观察：关键对象的身份、状态、错误和提交结果应能在报告中找到。

第四，检查代码示例是否说明禁止行为。调用者不能做什么，往往比能做什么更能体现工程质量。第五，检查实验是否写出未验证边界；当前设备、权限或输入无法证明的结论，必须标注。第六，检查本章对象是否能被下一章复用，不能复用就说明边界还不稳。

完成这六项后，再看文字是否连贯。每个概念都应从问题推出，每个案例都应从朴素方案走到改进方案，每个实验都应能解释一个判断。若某段话放到其他章节也成立，就把它改成本章的具体对象、具体输入和具体失败。

\section{综合案例：一条日志记录怎样迫使系统分层}

如果只看函数调用，日志统计作业很像一个普通程序：读文件，解析一行，更新计数，写结果。可是只要把输入规模、失败、并发和恢复放进问题里，这个普通程序就会被迫长出系统结构。高质量的系统书不应该把这种结构直接列成名词表，而应该让读者看见它从哪里来。这里用一条日志记录贯穿整本书的第一轮推导。

\topic{一、身份先于优化}
一条记录进入系统时，最先需要的不是线程，也不是 SIMD，而是身份。文件名、偏移、长度、行号、split 编号和作业编号共同说明这条记录从哪里来。没有身份，程序只能在内存里看到一个字符串；一旦发生重试、切分、异步写出或分布式执行，这个字符串就很难再和原始输入对应。身份的作用不是形式化，而是让失败后还能回答“这条结果是否已经处理过”“这次处理对应哪个输入范围”“旧 attempt 的输出是否应该被采纳”。

身份设计还会反过来约束性能路径。若每条记录都携带完整文件名和大段诊断文本，热循环会搬动大量冷数据；若热路径只保存 \code{split_id}、偏移和长度，诊断信息放到旁路表里，解析和聚合就能更接近连续数组扫描。因此第一章就要说明：系统分层不是为了抽象好看，而是为了让热路径和控制路径各自承担合适的责任。身份在控制面，字段值在数据面，二者通过稳定编号连接。

贯穿项目可以把记录身份写成一个很小的结构。代码里不必一开始就实现完整分布式系统，但要提前养成固定身份的习惯。

\begin{lstlisting}[language=C++,caption={记录身份只保存热路径需要的稳定编号}]
struct RecordId {
    std::uint64_t job_id = 0;
    std::uint32_t split_id = 0;
    std::uint64_t byte_offset = 0;
    std::uint32_t line_in_split = 0;
};
\end{lstlisting}

这个结构没有保存字符串路径，不保存错误文本，也不保存解析后的全部字段。它回答的是“这条记录是谁”。后续 trace、错误报告和 manifest 可以用它关联更多冷信息。这样的边界会在缓存、线程池、I/O 和分布式章节反复出现：热路径保存最少可判定身份，冷路径保存可诊断材料。

\topic{二、reference 不是慢版本，而是语义锚点}
朴素串行版本常被误解成“以后会删掉的慢代码”。在系统教材里，reference 更像法律文本：它定义输入如何解析，错误如何处理，输出如何排序，计数是否允许溢出，遇到重复输入时是否重复计数。没有 reference，并行版本、SIMD 版本和分布式版本都失去比较对象。快但语义不明的程序，在系统工程里没有价值。

日志统计的 reference 应该故意写得简单。它顺序读取，顺序解析，遇到坏行按合同记录错误，最后按 key 排序输出。它不追求吞吐，却要覆盖边界：空文件、最后一行没有换行、字段缺失、字段过长、数值超范围、重复 key、极端倾斜 key。后续任何优化版本都必须先和它比较结果，再比较成本。这样读者会明白：优化不是改写问题定义，而是在保留语义合同的前提下改变执行形状。

reference 还决定测试粒度。若输出合同要求稳定排序，测试就不能只比较总数；若错误合同要求保留第一处错误，测试就不能只比较错误数量；若计数溢出应报告失败，测试就不能把溢出后的值当作正常结果。第一章提前讲这些细节，是为了防止后面并行算法章变成“会用线程就行”。

\topic{三、控制面和数据面从失败中分开}
最小日志程序常把读取、解析、调度、错误处理和输出提交放在同一个调用栈里。小输入时这样最清楚；输入变大后，它会同时承担热路径和控制路径，既慢又难恢复。数据面负责搬动字节、解析字段、聚合计数和写 block；控制面负责任务身份、状态转换、重试预算、错误摘要和 manifest。二者分开后，热路径才能保持短、连续、可测，控制路径才能处理失败和恢复。

这个拆分可以从一个故障自然推出。假设 map 任务已经写出一个临时 block，但 driver 在提交 manifest 前崩溃。重启后，如果 reduce 只是扫描目录里所有文件，就可能把未提交的临时 block 当成有效输出；如果 control plane 有 manifest，只有被提交表接纳的 block 才对 reduce 可见。这里不是性能技巧，而是语义边界：文件存在不等于结果提交，消息到达不等于动作成功，函数返回不等于外部效果已经可见。

后续 I/O 和分布式章节会继续使用这条边界。临时文件、校验、manifest、attempt 编号和提交点都属于控制面；block 的字节内容属于数据面。把二者混在一起，最终会让恢复逻辑依赖目录扫描、日志文本或时间戳，系统就很难证明正确。

\topic{四、三张图必须能互相校验}
第一张图是数据身份图。它从 input split 画到 parsed batch、local partial、shuffle block 和 output manifest。每个节点回答“这个对象是谁，由谁拥有，是否可重做”。第二张图是执行等待图。它画出 reader、parser、worker、writer、driver 和队列，标出哪里运行、哪里阻塞、哪里可以取消。第三张图是成本账本。它记录每阶段读取多少字节、写多少字节、分配多少对象、发生多少等待、产生多少重试。

三张图不能各画各的。数据身份图中的 block，必须能在执行图里找到生产它的 task 和提交它的 driver；执行图中的等待，必须能在成本账本里看到队列水位、I/O 时间或锁等待；成本账本里的高写放大，必须能回到数据图里解释是哪一个阶段材料化了太多中间结果。若三张图无法互相校验，说明系统观测还没有闭合。

这也是后续章节的阅读方法。流水线章解释解析热循环的局部成本，但不能解释 manifest 重复提交；锁章解释队列状态保护，但不能解释远端消息丢失；分布式章解释重试和幂等，但不能替代单机热路径测量。第一章把三张图立住，读者才知道每一章在整条链路里负责哪一段。

\topic{五、学习顺序和工程顺序不同但互相约束}
书的顺序从硬件开始，因为读者需要知道一段代码如何在核心、缓存、TLB 和内存中表现。工程落地却常从语义开始，因为没有 reference、身份和提交点，任何性能数字都可能是在测错误程序。二者并不矛盾。正确的学习节奏是：先写 reference 和账本，再用硬件章节解释热路径，再用并发章节解释共享状态，再用运行时章节组织任务，再用分布式章节扩大失败模型。

例如日志解析慢，硬件章节会让你看分支、依赖和地址序列；但是否能把错误路径拆出去，要看语义合同是否允许延迟报告错误。线程池能提高吞吐，但队列关闭、任务取消和异常传播必须先定义。分布式重试能提高成功率，但如果输出没有幂等提交，重试会制造重复计数。每一次优化都要同时问三个问题：语义还在不在，瓶颈移动到哪里，失败后还能否解释。

本章给后续所有章节留下一个最低要求：任何新概念都要能落到对象、状态、成本或恢复中的至少一个位置。若只能背定义，不能说明它改变了哪条数据路径或哪个等待点，就说明还没学透。

\topic{六、第一章项目交付物}
第一章不要求写完整计算引擎，但要交付四件小东西。第一，串行 reference 和固定输入集。第二，记录身份结构和最小 trace。第三，数据身份图、执行等待图和成本账本的模板。第四，一个故障演练：临时 block 已写出但未提交，重启后系统如何判断它无效。

这个故障演练很重要。它让读者从一开始就知道，系统正确性不是最后输出碰巧相等，而是每个中间状态都有解释。后续第 18 章做完整 Compute Systems Engine 时，仍然会回到这四件东西。规模会变大，工具会变多，但问题没有变：数据是谁，谁在执行，哪里等待，什么结果已经提交，失败后从哪里恢复。

\section{贯穿项目的第一张设计评审}

第一章最后应让读者做一次小型设计评审，而不是只带着概念进入下一章。评审对象仍然是日志统计作业，但评审问题要覆盖全册：输入如何定义，计算如何切分，状态如何提交，失败如何恢复，性能如何归因。这个评审不是正式流程，而是把“系统思维”变成可重复动作。

\topic{一、先问输入边界}
输入不是“一个文件”这么简单。它有编码、行分隔、最大行长、字段数量、字段类型、错误策略和切分方式。若 split 可以从任意字节开始，就要处理跨 split 的半行；若 split 只能从行边界开始，就需要预扫描或让 reader 找边界；若字段允许转义和引号，解析器状态会跨字符传播。输入边界决定后面能否并行，也决定错误是否可局部恢复。

评审时要求写出三类输入。第一类是正常输入，用于吞吐基线。第二类是边界输入，例如空文件、超长行、最后一行无换行、字段缺失。第三类是破坏朴素假设的输入，例如一条记录跨 split、一个 key 占据大部分记录、错误行集中在某个块。没有这三类输入，后续优化容易只服务 happy path。

\topic{二、再问输出何时有效}
输出可以是内存里的计数 map，也可以是磁盘文件，也可以是 manifest 中的一组 block。不同输出有不同有效条件。内存 map 在函数返回时有效；临时文件存在不代表有效；manifest 条目提交后才对 reduce 可见；最终结果写出后还要考虑是否持久化。第一章要让读者从一开始区分“产生了数据”和“数据已经对外有效”。

这个区分会贯穿所有章节。并行 filter 写入输出数组时，scan 分配的位置才是有效位置；I/O 写临时文件时，completion 和校验决定 block 是否完整；分布式 attempt 返回成功时，driver 仍要判断是否已经有别的 attempt 提交。若输出有效性不清楚，失败恢复无法证明。

\topic{三、系统里的时间线不是源码顺序}
源码顺序给人一种线性幻觉：调用 parse，调用 aggregate，调用 write，调用 commit。多线程和分布式之后，真实时间线会分叉。Reader 可能已经读取下一批，parser 正在处理当前批，writer 还在写上一批，driver 正在等待某个迟到响应。错误也会沿不同路径返回。用源码调用栈描述系统，就会漏掉排队、等待和迟到。

评审要求画时间线。每个对象从 created、queued、running、completed、committed 到 reclaimed 的时间点要能说明。尤其要标出“外部可见”时刻和“内存可释放”时刻。一个 buffer 写入 I/O 队列后，对上游不可再修改，但对下游还不可见；一个任务完成计算后，结果还未提交；一个旧配置被新配置替换后，旧读者仍可能持有 snapshot。时间线清楚，很多并发和分布式概念会自然出现。

\topic{四、把失败当成正常路径设计}
失败不是额外章节才处理的内容。输入会坏，线程会取消，I/O 会短写，消息会丢失，进程会崩溃，计数器会溢出。第一章就应要求每个状态写失败出口。失败出口不是简单返回 false，而是要说明谁观察到失败，哪些资源释放，哪些结果撤销，哪些结果保留，是否可重试。

例如 map worker 写出 block 时失败，应该返回 attempt failed，临时文件标记无效或清理，driver 根据重试预算决定是否重试。若 failure 只写日志，上游无法恢复；若临时文件未清理，reduce 扫目录可能误读；若没有 attempt id，迟到成功和失败响应无法归属。失败路径设计会逼迫系统身份、所有权和提交点变清楚。

\topic{五、性能评审先找主路径}
系统里到处都有可优化点，但第一轮评审只找主路径。主路径是大部分数据经过的路径，不是代码最复杂的路径。日志统计的主路径可能是读取字节、解析热字段、本地聚合和写 shuffle；错误诊断、配置热更新和恢复路径很重要，但不应污染主路径。把主路径画出来，才能决定哪些字段热、哪些锁会频繁竞争、哪些 I/O 要批处理。

主路径之外的代码要保证正确和可观测，不一定追求极致速度。错误路径可以多保存诊断，恢复路径可以多做校验，控制面可以多写 trace。高质量系统不是所有代码都一样快，而是热路径短而清楚，冷路径完整而可靠。这个判断会影响数据布局、锁边界、任务运行时和分布式协议。

\topic{六、第一轮评审输出}
完成第一章后，读者应提交一页设计评审。它包括输入合同、输出有效条件、对象生命周期、失败矩阵、主路径和当前不可验证边界。不可验证边界也要写，例如没有 NUMA 机器、没有 PMU 权限、没有真实多机网络。这样后续章节每学一个机制，都能回到这页评审更新，而不是重新发明一套说法。

\section{把全书主线压成一个可执行检查表}

一本系统书如果只在每章讲局部知识，读者很容易学完后仍然不知道面对真实项目从哪里下手。第一章最后需要一张可执行检查表。它不是行政流程，而是帮助读者把硬件、运行时、算法和分布式语义放在同一张工作台上。

\topic{一、先写语义，再写资源}
每个新模块先回答语义问题：输入是什么，输出是什么，错误怎么表示，重复执行是否允许，结果何时对外有效。只有这些答案清楚，资源问题才有意义。否则你可能把一个会重复计数的作业优化得很快，或者把一个会丢错误的解析器写得很漂亮。

语义问题要写到接口里，而不是只写在脑子里。输入 split 要有边界，输出 block 要有身份，任务状态要有枚举，错误要有分类，提交要有 manifest。这样后续章节引入 cache、SIMD、线程、锁、原子和 RPC 时，都知道自己不能破坏哪些合同。

\topic{二、再写数据路径}
数据路径回答字节如何移动。输入字节进入 page cache 或 buffer，解析成热字段，聚合成 partial，分区成 block，写入 shuffle store，再被 reduce 读取。每一段都要写出是否复制、是否压缩、是否材料化、是否可重算。很多性能问题不是算得慢，而是搬了太多、搬错地方、搬完还无法解释。

数据路径还决定布局。热字段进入连续数组，冷诊断进入旁路存储，控制状态进入 manifest 或任务表。不要让一个对象同时承担热循环、错误诊断、持久化和调度状态。对象一旦过胖，缓存、并发和恢复都会变差。

\topic{三、再写执行路径}
执行路径回答谁来做。单线程 reference、固定线程池、任务图、I/O 线程、driver 事件循环和 worker 消息处理，都是不同执行实体。每个实体要知道它是否可能阻塞，是否拥有状态，是否可以取消，是否能重试。一个函数是否异步，不由名字决定，而由它返回后工作是否仍在进行决定。

执行路径最容易隐藏等待。等待锁、等待条件变量、等待 I/O、等待 future、等待远端响应、等待重试预算，表现都不同。检查表要求把等待点画出来。看不到等待，就无法解释尾延迟；看不到尾延迟，就会误把平均吞吐当成系统能力。

\topic{四、最后写观测路径}
观测路径回答如何证明。Reference 证明语义，checksum 证明数据未损坏，trace 证明状态顺序，benchmark 证明成本变化，故障注入证明恢复。观测不是最后加日志，而是设计的一部分。一个没有 task id 的系统，后面补再多日志也很难关联事件。

观测还要保留边界。某些设备没有 PMU 权限，某些环境没有 NUMA，某些文件系统不能模拟掉电，某些网络故障只能用本机消息总线近似。诚实写边界，比把所有结论说满更专业。读者在自己的机器上得到不同数字时，应能根据观测路径解释差异。

\topic{五、检查表的最终形状}
面对任何系统模块，依次写六行：语义合同、数据路径、执行路径、共享状态、提交与恢复、证据与边界。每一行都必须落到具体对象。例如语义合同落到 \code{RecordId} 和 reference，数据路径落到 HotBatch 和 ShuffleBlock，执行路径落到 TaskStateRecord，共享状态落到 QueueContract，提交恢复落到 ManifestEntry，证据落到 BenchmarkCase。

后续章节都可以回到这张表。流水线章填热循环证据，缓存章填地址序列，锁章填共享状态，I/O 章填完成事件，分布式章填消息身份。这样全书不会散成概念集，而是一套可执行工程方法。

\section{补充案例：系统边界如何防止后续章节散乱}

第二册覆盖硬件、并发、并行算法和分布式，很容易让读者觉得每章都在讲不同东西。防止散乱的办法，是每一章都回到同一批系统边界：数据身份、状态归属、执行实体、等待点、提交点和恢复点。概念可以不同，边界必须一致。

\topic{一、数据身份贯穿所有层}
在缓存章节，数据身份帮助我们知道哪些字段是热字段；在线程章节，数据身份帮助我们把任务和输入 split 关联；在 I/O 章节，数据身份变成 block id 和 checksum；在分布式章节，数据身份变成 request id、attempt id 和 manifest entry。名字变了，但作用不变：让系统能回答“这份数据是谁，从哪里来，现在是否有效”。

若没有数据身份，优化会破坏可追踪性。比如把多个记录合并成一个 partial 后，如果 partial 不记录输入范围，失败后就不知道该重做哪段；把 block 写到文件后，如果文件名不含 attempt 或 manifest 不记录 checksum，迟到响应和重复文件就难以处理。身份不是控制面负担，而是高性能系统仍可恢复的前提。

\topic{二、状态归属决定谁能修改}
共享状态不是看谁拿得到指针，而是看谁拥有修改权。队列 head/tail 由队列合同管理，任务状态由运行时或 driver 管理，manifest 由提交协议管理，配置版本由发布者管理。若多个模块都能随意修改同一状态，锁和原子也救不了设计。

状态归属还影响分布式设计。某个 partition 的 owner 决定谁接受写请求；route epoch 决定旧 owner 何时退休；manifest 决定 reduce 读哪些 block。把这些归属写清楚，后面的锁顺序、内存序、RPC 重试都会有依据。

\topic{三、等待点决定系统是否扩展}
等待点可以在硬件里是 load miss，在单机里是锁和队列，在 I/O 里是完成事件，在分布式里是响应和 quorum。它们都消耗时间，但处理方式不同。硬件等待靠独立工作和局部性隐藏；锁等待靠缩小临界区或分片；I/O 等待靠异步和背压；分布式等待靠超时、重试和幂等。

如果只说“慢”，不说等待点，就无法选择方法。第一章要求读者把等待点列出来，是为了后面不把 SIMD 用到锁等待上，不把加线程用到 I/O 背压上，不把重试用到不可恢复错误上。

\topic{四、提交点把正确性和性能连起来}
提交点是结果从临时变成有效的瞬间。并行算法里，写入分配好的输出区间是局部提交；I/O 里，block 通过校验并进入 manifest 才有效；分布式里，commit table 或复制日志决定外部承诺。提交点越强，恢复越容易，但可能增加延迟；提交点越弱，吞吐可能高，但失败后难解释。

系统工程的难点就在这里：性能和正确性不是两个阶段，而是在提交点上同时出现。批量提交提高吞吐，但扩大重做范围；即时提交降低恢复成本，但增加 I/O。读者要学会把这种取舍写出来。

\section{从学习目录变成系统推导方法}

第一章最后还要明确一种读法：本册不是把硬件、线程、锁、算法、I/O、分布式逐个背完，而是训练同一种推导习惯。遇到一个现象，先问它属于语义问题、资源问题、等待问题、生命周期问题还是观测问题；再沿着这五条线往下追证据。这样读后面的章节时，读者不会把每章看成孤立知识点，而会看到它们都在回答同一个问题：一个计算任务怎样在真实机器上正确、快速、可恢复地完成。

\topic{一、语义先于优化}
任何优化前都要写出“什么结果算正确”。日志处理要定义重复行、错误行、排序和输出格式；并行归约要定义浮点误差和合并顺序；分布式执行要定义重复请求和迟到响应。语义没有写清时，性能越高，错误越难发现。第一章要求读者每做一个模块都先写 reference，不是为了保守，而是为了给后续优化建立可比较的锚点。

\topic{二、资源必须有单位}
“内存很多”“队列有点长”“线程数差不多”都不能指导工程。资源要落到单位：字节、cache line、页面、buffer 数、in-flight 请求数、线程数、文件描述符数、重试次数、manifest 条目数。单位清楚以后，容量才可推导。比如一个 buffer 多大、队列允许多少个、每个 worker 同时处理几个任务，都会变成可算的预算，而不是凭感觉写常量。

\topic{三、等待要有可见状态}
系统变慢时，等待常常比计算更难解释。线程在等锁，任务在等依赖，I/O 在等完成，RPC 在等响应，分布式副本在等多数派。每一种等待都需要状态字段和时间戳。若等待只藏在调用栈里，报告无法判断是上游太快、下游太慢、容量太小还是死锁。后续所有章节都会反复把等待从隐式行为变成显式状态，这是贯穿全书的主线。

\topic{四、生命周期决定释放和恢复}
对象什么时候创建、谁拥有、谁能观察、什么时候提交、什么时候释放，是系统正确性的另一条主线。无锁节点不能随便释放，异步 buffer 不能提交后马上复用，分布式输出不能在 manifest 提交前被读取。生命周期写不清，错误通常不会立刻出现，而会在高并发、取消、崩溃和迟到完成时出现。第一章要求读者画所有权图，就是为了让这些迟发错误提前暴露。

\topic{五、观测材料要能复盘}
学习系统不是靠感觉进步，而是靠报告复盘。每个实验至少保存输入摘要、版本信息、指标、状态转移、失败注入和未验证边界。这样后续发现问题时，可以回到证据链，而不是重新猜。读完整本第二册后，读者应该能把任何性能或正确性问题写成一份小报告：现象是什么，语义是否明确，资源是否到顶，等待在哪里，生命周期是否闭合，下一步验证什么。
