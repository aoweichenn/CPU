\chapter{I/O、异步与背压}

先看一个写出队列事故。解析和计算阶段很快，reduce worker 很快生成大量 \code{OutputBlock}；磁盘偶尔变慢，写出线程跟不上。第一版用无界队列，短时间 benchmark 很漂亮，因为计算线程从不等待；几分钟后内存持续上涨，尾延迟越来越长，最终进程被系统杀掉。第二版改成异步写，\code{submit_write} 返回成功后，上游立刻复用 buffer；几毫秒后 I/O 线程才真正写文件，落盘内容偶尔变成下一批数据。第三版又修了 buffer 生命周期，却把 \code{submit} 成功当作业务成功，manifest 过早发布，短写和失败 completion 被日志吞掉。

这三个事故逼出 I/O 章节的主线：I/O 不是“换一个系统调用”这么简单，而是必须分清 \term{提交}{submit}、\term{完成}{completion}、\term{提交业务结果}{commit}。提交只表示请求被接纳；完成才说明底层读写有结果；业务提交才说明外部消费者可以相信这个输出。异步把等待从调用线程上移走，同时也把所有权、短读短写、错误、取消、背压、可靠写入和崩溃恢复全部暴露出来。

后面的结构都围绕同一条因果链展开：阻塞调用为什么占住 worker；非阻塞 ready 为什么不等于消息完整；异步 completion 为什么要有请求身份；buffer 为什么要有唯一所有者；队列为什么要按 item 和 byte 双重限流；可靠写入为什么要把临时文件、checksum、fsync、rename 和 manifest 分层处理；这些对象之后会扩展成分布式 reply、shuffle block、checkpoint 和流控窗口。

\section{从阻塞 I/O 到完成事件}

阻塞 I/O 的语义最接近普通函数调用：调用线程发起 \code{read} 或 \code{write}，线程等待，返回值告诉本次调用处理了多少字节或出了什么错误。它适合配置读取、低并发工具、简单批处理和语义优先的 reference。它的问题是等待设备时占住线程。若计算 worker 在 \code{fsync}、网络发送或慢磁盘上等待，任务运行时再精巧也无法让这些 worker 执行 ready 的 CPU 任务。

异步 I/O 的目标不是让设备变快，而是把等待从执行线程上移走。提交请求后，CPU 可以继续处理其他工作；完成事件稍后返回，告诉系统这次请求成功、失败、短写、取消还是超时。这个分离提升并发度，也引入新的状态机。任何异步设计都要先回答：请求提交后谁拥有 buffer，completion 由谁消费，失败如何传播，取消后迟到 completion 怎么处理。

\topic{文件 I/O 的第一层对象：path、fd、open file description、inode}
文件 I/O 不能只记住 \code{read(fd,...)} 和 \code{write(fd,...)}。应用传入的是 path，内核打开后返回 file descriptor，fd 指向一个 open file description，open file description 再引用文件对象、当前 offset、打开标志和访问模式，底层文件系统对象通常可理解为 inode 或类似结构。多个 fd 可能指向同一个 open file description，也可能分别打开同一个 path 得到不同 description。

\begin{lstlisting}[numbers=none,caption={文件 I/O 对象关系}]
path string
  -> pathname lookup
  -> inode or filesystem object
  -> open file description: flags, current offset, status
  -> file descriptor number in this process
\end{lstlisting}

这条链解释了很多并发 I/O bug。复制 fd 的 \code{dup}、\code{fork} 继承或把同一个 fd 传给多个线程，可能共享同一个 current offset；两个线程同时用 \code{read} 读取同一个 fd，读到的位置由共享 offset 推进顺序决定。若每个线程单独 \code{open} 同一个文件，通常得到各自的 open file description，offset 不共享。若要并行读取固定区间，应优先使用带显式 offset 的 \code{pread}，避免把文件位置当成隐藏共享状态。

\small
\begin{tabular}{p{0.22\linewidth}p{0.35\linewidth}p{0.29\linewidth}}
\toprule
操作 & offset 行为 & 并发后果 \\
\midrule
\code{read(fd)} & 使用并推进 open file description 的 current offset & 多线程共享 fd 时顺序变成隐藏状态 \\

\code{O_APPEND} 写 & 每次写追加到文件末尾 & 避免手动 seek 竞争，但仍要处理短写和错误 \\

\end{center}

报告里如果只写“多线程读同一个文件”，信息不够。要写清：是共享一个 fd，还是每线程独立 open；是 \code{read}，还是 \code{pread}；是否使用 \code{O_APPEND}；是否有 \code{dup}/\code{fork}；是否把 fd 交给了库函数。否则很多“随机错位”“重复读取”“漏读”的 bug 会被误判成 parser 问题。

\topic{基础闭环：fd、VFS、page cache 和业务字节}
这一章后面会讲异步、completion 和背压，但在那之前，必须先把同步文件 I/O 的基础闭环钉住。应用代码里的 \code{int fd} 只是一个进程局部编号；编号通过进程 fd table 找到 open file description；description 再指向 VFS 文件对象；VFS 根据底层文件系统、设备和打开标志决定读写路径。业务看到的是“读写字节”，内核维护的是 offset、权限、引用计数、page cache、dirty page、writeback 和错误延迟。

\begin{lstlisting}[numbers=none,caption={一次普通文件 I/O 的最小对象链}]
process-local fd table
  fd 3
    -> open file description
       flags, current offset, file status, refcount
       -> VFS file object
          file operations: read_iter, write_iter, fsync, mmap
          -> filesystem object: inode, address_space, metadata
             -> page cache pages or direct I/O mapping
                -> block layer and device
\end{lstlisting}

这条链把几个基础问题连在一起。第一，\code{fd} 的生命周期和底层文件对象不同：\code{close(fd)} 只是释放当前进程里的一个引用，别的 fd、子进程或库对象仍可能持有同一个 open file description。第二，\code{read/write} 的 current offset 属于 open file description，不属于 fd 号码本身。第三，普通文件的 buffered I/O 会经过 page cache，\code{write} 成功经常只说明字节进入内核缓存并变脏，不说明设备完成。第四，direct I/O 试图绕过 page cache 的数据路径，但不会绕过权限检查、VFS、文件系统、错误处理和持久化协议。

\toprule
对象或状态 & 它保存什么 & 常见误判 \\
\midrule
fd number & 当前进程 fd table 的索引 & 以为它就是文件本身 \\

最小闭环可以按状态机记住：

\begin{lstlisting}[numbers=none,caption={fd 和普通文件数据路径状态机}]
Closed:
  no process-local handle exists

Opened:
  fd table entry references an open file description

BufferedRead:
  read checks page cache; cache miss may allocate page and wait for device

BufferedWrite:
  write copies user bytes into page cache and marks pages dirty

WritebackPending:
  dirty pages wait for background writeback or fsync

  fsync/fdatasync waits for filesystem/device durability boundary

  close drops this reference; delayed write error may be reported
\end{lstlisting}

读路径和写路径都要按这个状态机报告。读路径要写清是 cold page cache 还是 warm page cache，是 \code{read} 复制到用户 buffer，还是 \code{mmap} 后由 page fault 填页。写路径要写清是否 buffered，是否调用 \code{fsync/fdatasync}，是否把 \code{close} 返回值纳入错误处理，是否有 manifest 或 rename 作为业务提交边界。没有这些字段，I/O benchmark 的数字很容易把内存 copy、缺页、设备吞吐和持久化等待混在一起。

\topic{\code{open}、\code{read}、\code{write}、\code{close} 的最小合同}
\code{open} 的返回值不是文件内容，而是进程 fd 表中的一个小整数句柄。它可能失败，失败原因包括路径不存在、权限不足、fd 数量耗尽、目录不存在、文件系统只读、路径太长等。成功后，fd 必须在所有路径上关闭；但 \code{close} 本身也可能返回错误，尤其是写出路径中，某些错误可能延迟到 \code{close} 或 \code{fsync} 才暴露。写可靠输出时，不能只检查 \code{write} 返回值而忽略 \code{fsync/close}。

\begin{lstlisting}[language=C++,caption={同步文件读取的基础合同，不是高性能版本}]
int fd = ::open(path, O_RDONLY | O_CLOEXEC);
if (fd < 0) {
    return error_from_errno(errno);
}

std::uint64_t done = 0;
while (done < size) {
    ssize_t n = ::read(fd, buffer + done, size - done);
    if (n > 0) {
        done += static_cast<std::uint64_t>(n);
        continue;
    }
    if (n == 0) {
        break; // EOF
    }
    if (errno == EINTR) {
        continue;
    }
    return error_from_errno(errno);
}

if (::close(fd) != 0) {
    return error_from_errno(errno);
}
\end{lstlisting}

这段代码故意保留循环，因为短读是正常路径。它也区分 \code{n == 0} 的 EOF、\code{EINTR} 的可重试中断、和真实错误。写路径同理：\code{write} 返回正数但小于请求长度时必须继续写；返回 \code{-1} 且 \code{errno == EINTR} 可重试；非阻塞 fd 的 \code{EAGAIN} 表示当前不能推进，应该进入等待机制，而不是当成 fatal error。

\topic{stdio 缓冲和系统调用不是同一层}
\code{FILE*}、\code{fread}、\code{fprintf}、\code{std::ofstream} 这类接口通常在用户态有缓冲。用户调用 \code{fprintf} 成功，可能只是写入了 libc 或 C++ iostream 缓冲；真正的 \code{write} 系统调用稍后才发生。\code{fflush} 把用户态缓冲推给内核，不等于数据持久化；\code{fsync} 针对 fd 的内核侧脏数据和 metadata 才讨论持久边界。混用 \code{FILE*} 缓冲和底层 fd 系统调用时，如果不 \code{fflush} 或不理解 offset，同一文件的读写顺序很容易错乱。

\begin{center}
\small
\begin{tabular}{p{0.23\linewidth}p{0.37\linewidth}p{0.28\linewidth}}
\toprule
层次 & 成功说明 & 不说明什么 \\
\midrule
\code{fprintf}/\code{ofstream} & 用户态库接受了数据 & 不说明已进入内核 page cache \\
\code{fflush} & 用户态缓冲尝试写给内核 & 不说明已经落盘 \\
\code{write} & 内核接受了部分或全部字节 & 不说明 durable \\
\code{fsync} & 推进文件持久化边界 & 不说明业务 manifest 已提交 \\
\code{close} & 释放 fd，可能报告延迟错误 & 不等同于完整恢复协议 \\
\bottomrule
\end{tabular}
\end{center}

工程建议很直接：热路径和可靠写路径尽量不要混用多层 I/O 抽象。同一个 fd 上如果既用 \code{FILE*} 又用 \code{read/write/lseek}，必须明确缓冲刷新、offset 和错误传播。性能报告也要写清使用的是 raw syscall、stdio、iostream、mmap 还是库封装；否则“文件 I/O 慢”这个结论不可比较。

\topic{文件 offset 是隐藏共享状态}
线程安全不等于语义正确。两个线程同时调用 \code{read(fd)}，内核可以让每次读取按某种顺序推进共享 offset，数据竞争未必发生在 C++ 对象上，但业务分片可能已经错了。更隐蔽的是一个线程 \code{lseek(fd, off)}，另一个线程随后 \code{read(fd)}；如果中间有第三个线程改变 offset，读取位置就不是调用者以为的位置。并行扫描文件时，显式 offset 比共享 offset 更可靠。

\begin{lstlisting}[numbers=none,caption={共享 offset 导致分片语义不稳定}]
thread 0: lseek(fd, 0)
thread 1: lseek(fd, 4096)
thread 0: read(fd, buf0, 4096)   // may read from 4096, not 0
thread 1: read(fd, buf1, 4096)   // offset already advanced
\end{lstlisting}

正确的并行读设计通常让分片计划先产生 \code{(offset, length)}，worker 使用 \code{pread} 或独立 fd 读取自己的范围。若文件是日志或变长记录，还要处理边界：每个 split 从某个 offset 附近开始，但第一条完整记录可能要向前或向后找换行。I/O 基础和第 13 章 partition 的联系就在这里：分片不仅是计算问题，也是文件位置和记录边界问题。

\topic{阻塞、非阻塞和异步}
阻塞 I/O 调用可能让当前线程睡眠，直到有结果。非阻塞 I/O 调用立即返回，若当前不能读写，返回 \code{EAGAIN} 或类似状态；应用需要等待 ready 事件后再试。异步 I/O 提交请求，完成队列稍后返回结果。三者不是“低级到高级”的线性关系，而是控制流和资源模型不同。

\code{epoll} 这类 ready 机制只说明 fd 可能可读或可写，不说明业务消息已经完整。Socket 可读可能只有半个 header；可写也不保证整个响应一次写完。\code{io_uring} 更接近提交队列和完成队列模型，但仍然要处理 buffer 生命周期、短读短写、错误码、取消和完成顺序。必须避免把具体 API 神化，核心是状态机。

\topic{eventfd：把用户态状态变化变成 fd 可读}
\code{eventfd} 常被用来唤醒 \code{epoll} 线程。它的语义不是传输业务消息，而是维护一个内核计数器：写入一个 64 位值会增加计数；读取会取走计数并让它回到零，或者在 semaphore 模式下一次读走一个单位。于是 eventfd 适合表达“有事情需要 event loop 回来检查”，不适合把每个任务 payload 塞进 fd。

\begin{lstlisting}[numbers=none,caption={eventfd 唤醒的正确分层}]
producer thread:
  push request into user-space queue
  if queue was previously empty:
      write(eventfd, 1)        // wake event loop

event loop:
  epoll_wait returns eventfd readable
  read(eventfd)                // drain wake counter
  pop all available requests from user-space queue
  process bounded amount of control work
\end{lstlisting}

这条链里，业务事实在用户态队列中；eventfd 只是把“队列状态可能变了”转成 fd readiness。若生产者每 push 一个请求都无条件写 eventfd，event loop 可能收到大量重复唤醒；若 event loop 只读 eventfd 却不 drain 用户态队列，计数器清零后仍有工作积压；若把 request id 当作 eventfd 计数写进去，多个写会被合并，身份信息会丢失。

对象 & 保存什么状态 & 常见误用 \\
\midrule

\topic{epoll：ready 事件不是 completion}
\code{epoll} 管的是 fd readiness。它告诉事件循环“这个 fd 现在做某类 I/O 操作可能推进”，不是“某个业务请求完成”。TCP socket readable 可能只够读半个 header；writable 可能只够写响应的一部分；eventfd readable 只说明计数器非零；管道 readable 只说明至少有字节可读。应用必须在 fd ready 后推进自己的状态机，直到遇到 \code{EAGAIN}、完成一个业务消息、达到本轮预算或进入错误路径。

\begin{lstlisting}[numbers=none,caption={epoll reactor 的基本状态循环}]
event loop:
  events = epoll_wait(epoll_fd)
  for each event:
    if event.fd is listen socket:
        accept until EAGAIN
    if event.fd is data socket and readable:
        read into connection buffer until EAGAIN or budget exhausted
        parse zero or more complete messages
        enqueue CPU work to compute executor
    if event.fd is data socket and writable:
        flush output buffer until EAGAIN or buffer empty
    if event.fd is eventfd:
        read eventfd counter
        drain user-space control queue
\end{lstlisting}

这个循环里有两个预算。I/O 预算防止单个热 fd 独占 event loop；CPU 预算防止事件回调直接变成业务执行线程。解析出完整消息后，重 CPU 工作应该作为 task 进入 compute executor；event loop 只保留连接状态、buffer offset、订阅的读写兴趣和必要的轻量协议推进。

\topic{水平触发、边沿触发和 drain 规则}
水平触发下，只要 fd 仍 ready，\code{epoll\_wait} 之后还可能继续报告它。边沿触发下，内核只在 ready 状态变化时提示；应用若没有一直读写到 \code{EAGAIN}，可能把剩余数据留在 fd 里，却再也等不到下一次边沿通知。边沿触发常见于高性能 reactor，但它购买的是更严格的 drain 责任，不是自动更快。

\begin{lstlisting}[numbers=none,caption={边沿触发必须 drain 到 EAGAIN}]
on EPOLLIN for edge-triggered socket:
  loop:
    n = read(fd, buffer)
    if n > 0:
        append to connection input buffer
        parse complete messages
        continue
    if n == -1 and errno == EAGAIN:
        stop reading; wait for next edge
    if n == 0:
        peer closed
    else:
        handle error
\end{lstlisting}

这条规则同样适用于 eventfd：event loop 看到 eventfd readable 后应读取计数器，并检查对应用户态队列。若多个 producer 写入 eventfd，读到的计数可能是合并值；它只能告诉你“至少发生过这些唤醒写”，不能替代队列长度。真正的 drain 条件是用户态队列空、连接 buffer 没有完整消息、输出 buffer 暂时写不动，或者本轮处理预算耗尽。

\topic{eventfd/epoll 和条件变量的相似点}
eventfd/epoll 和 condition variable 看起来一个是 fd，一个是线程同步原语，但它们共享同一条原则：通知不是状态。条件变量的通知要求等待者回到锁内检查谓词；epoll 的 ready 要求 event loop 回到 fd 和用户态连接状态检查进度；eventfd 的可读要求 event loop 回到用户态队列检查真实工作。把通知当事实，是两类系统都会犯的错误。

\begin{center}
\small
\begin{tabular}{p{0.22\linewidth}p{0.34\linewidth}p{0.30\linewidth}}
\toprule
机制 & 通知含义 & 必须重新检查的状态 \\
\midrule
condition variable & 谓词可能变化 & mutex 保护的队列/关闭/error \\
eventfd & 有线程写过唤醒计数 & 用户态请求队列和 shutdown flag \\
epoll readable & fd 可能可读 & connection buffer 和 message framing \\
epoll writable & fd 可能可写 & output buffer、short write offset、backpressure \\
io completion & 请求返回 bytes/status & request record、buffer owner、commit state \\
\bottomrule
\end{tabular}
\end{center}

\topic{event loop 唤醒报告字段}
如果一个 I/O runtime 只记录 \code{epoll\_wait} 次数，很难解释延迟。报告应把 eventfd、epoll、用户态队列和业务 completion 对齐。最小字段如下：

\begin{lstlisting}[numbers=none,caption={eventfd/epoll runtime 的观测字段}]
epoll_wait_count
epoll_wait_timeout_count
epoll_events_returned
eventfd_write_count
eventfd_read_count
eventfd_counter_sum
control_queue_depth_max
control_queue_drain_items
socket_read_ready_count
socket_write_ready_count
read_until_eagain_count
write_until_eagain_count
messages_parsed_count
cpu_tasks_enqueued_from_event_loop
event_loop_callback_ns_p99
\end{lstlisting}

这些字段能区分几种不同瓶颈：eventfd 写很多但队列很浅，说明唤醒合并策略差；epoll 返回很多 writable 但写不完，说明下游慢或输出 buffer 过大；read ready 很多但 messages parsed 少，说明消息分片或 framing 成本高；callback p99 很高，说明 event loop 执行了不该由它执行的 CPU 工作。

\topic{eventfd 的读写合同}
eventfd 的基础合同很小，但细节足以制造 bug。创建时通常选择 \code{EFD\_NONBLOCK} 和 \code{EFD\_CLOEXEC}：前者避免 event loop 或生产者在 fd 操作上意外阻塞，后者避免子进程继承控制 fd。普通模式下，读一次返回当前计数并把计数清零；semaphore 模式下，读一次返回 1 并把计数减 1。写入会增加计数，但计数有上限；非阻塞写在无法增加时返回 \code{EAGAIN}。

\begin{lstlisting}[numbers=none,caption={eventfd read/write 的基本语义}]
create:
  fd = eventfd(0, EFD_NONBLOCK | EFD_CLOEXEC)

write side:
  write uint64 value > 0
  counter += value
  fd becomes readable
  nonblocking write may fail with EAGAIN if counter would overflow

read side, normal mode:
  if counter > 0:
      read returns counter
      counter = 0
  else if nonblocking:
      read fails with EAGAIN

read side, semaphore mode:
  if counter > 0:
      read returns 1
      counter -= 1
\end{lstlisting}

因此 eventfd wake 的正确用法通常是“合并唤醒”。用户态队列保存真实工作；eventfd 只负责把 event loop 从 \code{epoll\_wait} 里拉出来。若需要每个请求都有身份，就在队列元素里保存 request id；若需要每个完成都有状态，就用 CompletionRecord；不要把 eventfd counter 当 request id、字节数或完成次数的权威来源。

\topic{实测证据：eventfd 合并计数和 semaphore 模式}
下面的最小实验在 x86-64 Linux 环境中运行，系统内核为 \code{Linux 6.6.87.2-microsoft-standard-WSL2 x86\_64 GNU/Linux}，glibc 为 \code{2.42}。实验只验证用户可见的 \code{eventfd} 读写合同和 \code{epoll} readiness 行为，不声称分析内核内部实现。

\begin{lstlisting}[numbers=none,caption={eventfd/epoll 合同实验命令}]
gcc -std=c11 -O2 -Wall -Wextra eventfd_epoll_probe.c \
  -o eventfd_epoll_probe
./eventfd_epoll_probe
perf stat -e cycles,instructions ./eventfd_epoll_probe
\end{lstlisting}

\begin{lstlisting}[numbers=none,caption={eventfd 读写实验输出摘要}]
case=eventfd_normal_writes write1=8 write2=8
case=eventfd_normal_read_merged read_n=8 errno=OK value=5
case=eventfd_normal_read_empty read_n=-1 errno=EAGAIN value=0
case=eventfd_semaphore_write write_n=8
case=eventfd_semaphore_read1 read_n=8 errno=OK value=1
case=eventfd_semaphore_read2 read_n=8 errno=OK value=1
case=eventfd_semaphore_read3 read_n=8 errno=OK value=1
case=eventfd_semaphore_empty read_n=-1 errno=EAGAIN value=0
\end{lstlisting}

两次普通写入 \code{2} 和 \code{3} 后，一次 read 返回 \code{5}，证明普通模式读的是合并后的 counter，并把 counter 清空；第二次读返回 \code{EAGAIN}，证明非阻塞空读不会睡眠。semaphore 模式下，写入 \code{3} 后连续三次 read 都返回 \code{1}，第四次空读返回 \code{EAGAIN}。这就是为什么 eventfd 适合作为“有工作了”的唤醒 fd，而不适合作为工作身份或完成记录本身。

\topic{eventfd 的常见竞态：队列先还是写 fd 先}
生产者必须先发布用户态队列中的工作，再写 eventfd。若顺序反过来，event loop 可能醒来、读掉 eventfd 计数、检查队列为空，然后回到 \code{epoll\_wait}；生产者随后才把工作放入队列，这个工作就可能一直等到下一次唤醒。和 condition variable 一样，通知必须跟在状态变化之后。

\begin{lstlisting}[numbers=none,caption={eventfd 唤醒顺序的对错}]
wrong:
  write(eventfd, 1)
  push request into queue

right:
  push request into queue with release semantics or under queue mutex
  write(eventfd, 1)

event loop:
  read(eventfd)
  drain queue with acquire semantics or under queue mutex
\end{lstlisting}

如果队列本身用 mutex 保护，push 和 drain 通过锁建立可见性；如果用无锁 MPSC 队列，发布节点和消费节点必须有 acquire/release 关系。eventfd 不替队列建立内存序，它只让等待线程变成 runnable 或让 epoll 返回。

\topic{epoll 的 interest、ready 和 user data}
epoll 有三层状态。第一层是 interest list：应用通过 \code{epoll\_ctl} 注册 fd 和关注的事件掩码，例如 readable、writable、hangup、edge-triggered、one-shot。第二层是内核观察到的 ready 状态。第三层是应用自己的 connection/request 状态。事件返回时携带的 \code{data} 字段应该能定位稳定的连接对象或 id，而不是指向可能已经释放的栈对象。

\begin{lstlisting}[numbers=none,caption={epoll fd 注册要能回到稳定状态对象}]
connection table:
  conn_id -> ConnectionState {
    fd
    input_buffer
    output_buffer
    parse_state
    subscribed_events
    generation
  }

epoll_ctl add:
  event.events = EPOLLIN | EPOLLET
  event.data.u64 = pack(conn_id, generation)

event loop:
  unpack conn_id and generation
  find ConnectionState
  reject stale event if generation changed
  advance fd and connection state
\end{lstlisting}

generation 很重要。fd number 会被内核复用；连接关闭后，同一个整数 fd 可能很快属于另一条连接。若 event loop 中还有旧事件或业务队列里还有旧任务，裸 fd 可能把旧事件投给新连接。稳定的 id/generation 句柄能把迟到事件识别为 stale。

\topic{EPOLLIN、EPOLLOUT、HUP 和 ERR 的边界}
readable 和 writable 不是唯一事件。对端关闭、错误、半关闭、连接失败都可能通过 hangup 或 error 事件进入循环。健壮的 reactor 不应只在 \code{EPOLLIN} 时读、只在 \code{EPOLLOUT} 时写；它要在每次事件后检查错误状态，并让 ConnectionState 进入 Closing、Draining 或 Failed。

\begin{lstlisting}[numbers=none,caption={epoll 事件处理的边界检查}]
for event in epoll_wait results:
  if event has EPOLLERR:
      read socket error with getsockopt(SO_ERROR)
      mark connection Failed
  if event has EPOLLHUP or EPOLLRDHUP:
      mark peer close or half-close
      drain remaining readable bytes if protocol allows
  if event has EPOLLIN:
      read until EAGAIN or budget
  if event has EPOLLOUT:
      write until EAGAIN or output empty
  if output empty:
      remove EPOLLOUT interest to avoid writable wake storms
\end{lstlisting}

可写事件尤其容易误用。大多数 socket 大部分时间都 writable；若一直订阅 \code{EPOLLOUT}，event loop 会被“可以写但没有东西写”的事件淹没。常见策略是：只有 output buffer 非空且上次写到 \code{EAGAIN} 后，才订阅 writable；buffer 清空后取消 writable interest。

\topic{实测证据：epoll readiness、one-shot 和半关闭}
同一个实验还验证了 \code{epoll} 的几条用户可见边界：

\begin{lstlisting}[numbers=none,caption={epoll 合同实验输出摘要}]
case=epoll_ctl_eventfd_add rc=0
case=epoll_eventfd_ready epoll_n=1 errno=OK events=0x1 data=0xfeed0011
case=epoll_eventfd_drain read_n=8 errno=OK value=1
case=epoll_eventfd_after_drain epoll_n=0 errno=OK events=0x0 data=0x0
case=epoll_ctl_pipe_oneshot_add rc=0
case=oneshot_first_ready epoll_n=1 errno=OK events=0x1 data=0xabc00001
case=oneshot_without_rearm epoll_n=0 errno=OK events=0x0 data=0x0
case=epoll_ctl_pipe_oneshot_rearm rc=0
case=oneshot_after_rearm epoll_n=1 errno=OK events=0x1 data=0xabc00002
case=oneshot_drain read_n=2 errno=OK
case=epoll_ctl_socket_add rc=0
case=socket_peer_half_close epoll_n=1 errno=OK events=0x2001 data=0xbeef0001
\end{lstlisting}

第一段证明 eventfd 进入 readable 后会让 \code{epoll\_wait} 返回 \code{EPOLLIN}，读掉 counter 后再次等待返回 0。第二段证明 \code{EPOLLONESHOT} 触发一次后会暂时禁用该 fd：即使管道又写入了字节，不 rearm 的 \code{epoll\_wait} 仍返回 0；执行 \code{EPOLL\_CTL\_MOD} 重新 armed 后，同一个 fd 再次返回 ready。第三段用 \code{socketpair} 和 \code{shutdown(SHUT\_WR)} 观察半关闭，事件位 \code{0x2001} 表示 \code{EPOLLRDHUP | EPOLLIN}。这说明 reactor 不能只处理读写，还要把 half-close、hangup 和 error 统一纳入连接状态机。

本次环境不能采集 \code{cycles,instructions}：

\begin{lstlisting}[numbers=none,caption={perf 计数器边界}]
Error:
No supported events found.
The cycles:u event is not supported.
\end{lstlisting}

因此这些结果只支持 I/O 合同和 readiness 状态分析，不能支持 event loop 开销、IPC 或 cache 行为结论。若要讨论性能，需要另补可用 PMU、调度 trace、callback p99 和队列水位。

\topic{EPOLLONESHOT 和跨线程处理}
如果多个线程同时消费同一个 epoll fd，或者 event loop 把 fd 事件交给 worker 处理，就要防止同一个 fd 的事件被并发处理。一个常见模式是 \code{EPOLLONESHOT}：事件触发后，内核暂时禁用这个 fd 的后续通知，应用处理完并更新 ConnectionState 后，再用 \code{epoll\_ctl MOD} 重新 armed。它购买的是“同一连接同一时间只有一个处理者”的边界，代价是每次处理后都要正确 rearm。

\begin{lstlisting}[numbers=none,caption={one-shot reactor 的 rearm 协议}]
on event:
  mark connection Handling
  process bounded read/write work
  compute next interest mask from ConnectionState
  if connection still open:
      epoll_ctl MOD fd with EPOLLONESHOT | next mask
      mark connection Armed
  else:
      epoll_ctl DEL fd
      close fd after no owner remains
\end{lstlisting}

若忘记 rearm，连接会永久沉默；若在处理尚未结束时 rearm，另一个线程可能并发处理同一 ConnectionState；若 close 后没有删除或识别 generation，迟到事件可能打到新连接。epoll 编程的难点不在 API 名字，而在 fd 生命周期、事件订阅和用户态状态三者必须一致。

\topic{从 epoll 到任务运行时的边界}
event loop 不应该直接执行重 CPU work。更稳的结构是：event loop 只推进 fd 状态和消息 framing；完整消息变成 MessageEnvelope 或 TaskInput；compute executor 处理 CPU 逻辑；结果再以 OutputBuffer 或 WriteRequest 回到 I/O 层。这样可观测性也清楚：ready 延迟、parse 延迟、compute 排队、write backpressure 是不同阶段。

\begin{lstlisting}[numbers=none,caption={reactor 和 compute executor 的分层}]
event loop:
  fd readiness -> ConnectionState progress
  complete message -> enqueue TaskInput
  task output available -> enqueue OutputBuffer and arm EPOLLOUT

compute executor:
  TaskInput -> CPU work -> TaskOutput
  never blocks in epoll_wait
  never owns fd readiness state

I/O state:
  owns fd, input buffer, output buffer, parse offset
  owns event subscription mask
  owns connection close/drain/fail state
\end{lstlisting}

这条边界和第 14 章 scheduler/awaiter 边界一致：ready 事件只是把某个状态推进了一步，不能直接变成任意用户代码。否则 event loop 的 tail latency 会污染所有连接，系统看起来像 I/O 慢，实际是事件线程被 CPU continuation 占住。

\topic{异步 I/O 和 coroutine：completion 才能恢复 awaiter}
把 I/O 接到 coroutine 时，关键不是把 \code{read} 包在 \code{co\_await} 后面，而是让 awaiter 等真正的 I/O completion。若 \code{await\_suspend} 里调用阻塞 \code{read}，这个 coroutine 只是换了一种写法，compute worker 仍被设备等待占住。正确模型是：提交请求，保存 buffer 所有权和 coroutine handle，worker 返回调度器；completion 到达后，I/O 层更新请求状态，再把 handle 入恢复 executor。

\begin{lstlisting}[numbers=none,caption={异步 I/O awaiter 的事件链}]
co_await async_read(request):
  await_ready:
    return request state is already completed

  await_suspend(handle h):
    allocate or find IoRequestState
    record buffer_id, offset, remaining, attempt
    record h and resume_executor
    submit request to I/O backend
    return to scheduler without blocking worker

I/O completion loop:
  receive CompletionRecord
  update bytes_done, status, error_code
  if request needs more bytes:
      resubmit next slice or enqueue retry
  else:
      mark request terminal
      detach coroutine handle
      enqueue handle to resume_executor

await_resume:
  return completed byte count
  or throw/read error/canceled according to request state
\end{lstlisting}

这条链把第 14 章的 awaiter 合同和本章的 CompletionRecord 接起来。awaiter 的 observed state 是 \code{IoRequestState}；completion 来源是 I/O backend；恢复动作是向 executor 投递 \code{CoroutineResumeItem}。其中任何一步直接执行用户代码，都会把 I/O completion loop 变成业务执行线程，背压和尾延迟都会失真。

\topic{ready、completion 和 await resume 的区别}
非阻塞 fd 的 ready、异步 I/O 的 completion、coroutine 的 resume 是三件事。Ready 表示“现在重试系统调用可能推进”；completion 表示“某个已经提交的请求有结果”；resume 表示“用户态 coroutine 可以继续执行”。把三者混起来，会导致半包、短写、重复提交和错执行域恢复。

\begin{center}
\small
\begin{tabular}{p{0.20\linewidth}p{0.35\linewidth}p{0.31\linewidth}}
\toprule
事件 & 它证明什么 & 不证明什么 \\
\midrule
epoll readable & fd 可能读到一些字节 & 不证明完整消息或请求完成 \\
epoll writable & fd 可能接受一些字节 & 不证明整个 buffer 已发送 \\
io completion & 某个 request 有 bytes/status & 不证明业务 manifest 可见 \\
awaiter ready & \code{await\_resume} 可立即完成 & 不证明一定 inline 恢复安全 \\
coroutine resume & executor 开始执行 frame & 不证明 I/O 层还有 buffer 所有权 \\
manifest commit & 业务结果可见 & 不证明底层临时文件都已清理 \\
\bottomrule
\end{tabular}
\end{center}

因此，I/O coroutine 的 trace 至少要记录三组时间：request submit 到 completion 的 I/O 时间，completion 到 coroutine resume 的调度时间，resume 到 manifest commit 的业务时间。只看总耗时，无法判断慢在设备、completion loop、executor 队列还是业务提交。

\topic{IoRequestState：把 awaiter 和 CompletionRecord 串起来}
\code{CompletionRecord} 是一次完成事件，\code{IoRequestState} 是请求跨多次 completion 的状态。短读短写、重试、取消和迟到 completion 都要求这个状态对象稳定存在。Coroutine awaiter 不应直接等一个裸 fd，也不应只等一个布尔 ready，而应等这条请求状态进入终态。

\begin{lstlisting}[language=C++,caption={I/O awaitable 的请求状态字段}]
enum class IoRequestStateKind : std::int32_t {
  Created,
  Submitted,
  WaitingCompletion,
  Partial,
  RetryScheduled,
  Completed,
  Failed,
  Canceled,
  Late,
};

struct IoRequestState {
  std::uint64_t request_id = 0;
  std::uint64_t buffer_id = 0;
  std::uint64_t bytes_total = 0;
  std::uint64_t bytes_done = 0;
  IoRequestStateKind state = IoRequestStateKind::Created;
  ExecutorId resume_executor;
  CoroutineHandleId awaiting_coroutine = 0;
};
\end{lstlisting}

真实 C++ 实现会用平台的 \code{std::coroutine\_handle<>}、类型擦除 handle 或自定义 handle id；书稿里用 \code{CoroutineHandleId} 是为了强调状态边界。关键点是：completion handler 更新 \code{bytes\_done/state}；awaiter 注册 \code{awaiting\_coroutine/resume\_executor}；scheduler 只恢复 terminal request 对应的 coroutine。若 \code{bytes\_done < bytes\_total}，awaiter 不能恢复为成功，只能继续提交剩余字节或返回 partial 语义。

\topic{Blocking pool、event loop 和 compute executor 的分工}
不是所有平台都直接用 \code{io\_uring}。有些实现会用 dedicated blocking I/O thread pool 包装同步系统调用；有些用 epoll event loop；有些用真正的异步提交/完成队列。无论后端怎样，compute executor 都不应该执行长期阻塞 I/O。

\begin{center}
\small
\begin{tabular}{p{0.23\linewidth}p{0.34\linewidth}p{0.29\linewidth}}
\toprule
执行域 & 负责什么 & 不应该做什么 \\
\midrule
compute executor & CPU task、短 continuation、coroutine resume & 阻塞磁盘、网络、长 fsync \\
blocking I/O pool & 包装可能阻塞的 syscall & 执行大量 CPU continuation \\
event loop & epoll ready、timer、轻量状态推进 & 在回调里跑重业务代码 \\
completion loop & 消费 I/O completion，更新请求表 & 持锁恢复 coroutine 或提交 manifest 重活 \\
commit executor & checksum、manifest gate、批量提交 & 无限积压 completion buffer \\
\bottomrule
\end{tabular}
\end{center}

这张表对应第 14 章的 resume executor 选择。I/O completion loop 可以把 coroutine 恢复项投到 compute executor，也可以投到专用 continuation executor；但它不应在 completion 回调里直接运行用户 coroutine。否则一个慢 continuation 会阻塞后续 completion，buffer 归还变慢，背压链失真。

\topic{异步 I/O awaiter 的关闭和取消}
关闭时最危险的是“请求已经提交，coroutine 等着，buffer 也被 I/O 层持有”。Cancel request 可能成功，也可能请求已经完成，还可能稍后返回 late completion。awaiter 必须把这些路径都推到终态。

\begin{lstlisting}[numbers=none,caption={I/O awaiter 取消和关闭路径}]
shutdown drain:
  reject new IoRequestState
  keep submitted requests alive
  process completions until request table empty
  resume awaiting coroutines with value or error

shutdown cancel:
  reject new IoRequestState
  request cancellation for submitted entries
  mark not-yet-submitted entries Canceled
  keep buffers and states until completion or reap
  resume awaiting coroutines with canceled or final result

late completion:
  find request_id
  if request already terminal:
      classify as Late
      return or retire buffer exactly once
      do not resume coroutine twice
\end{lstlisting}

这和 future 的取消语义一致：取消请求不是终态，终态必须能被等待者观察。若 cancel 后直接销毁 \code{IoRequestState}，迟到 completion 会找不到归属；若直接销毁 coroutine frame，completion 可能恢复悬垂 handle；若不恢复 coroutine，调用者会永久挂起。I/O awaiter 的正确性就是请求表、buffer 所有权、completion 和 coroutine frame 四者同时闭合。

\topic{短读短写是正常路径}
\code{read} 和 \code{write} 返回的字节数可能小于请求字节数。网络、管道、非阻塞 fd、信号中断、设备队列和某些文件系统路径都可能出现 partial completion。正确代码必须维护 \code{offset} 和 \code{remaining}，直到完成、出错、超时或进入等待。把一次返回当作完整消息，是 I/O 程序最常见的边界错误。

\begin{lstlisting}[language=C++,caption={同步可靠写循环的示例形态}]
enum class IoStepResult : std::int32_t {
  Complete,
  RetryLater,
  Failed,
};

struct WriteProgress {
  std::uint64_t bytes_total = 0;
  std::uint64_t bytes_done = 0;
};

bool is_write_complete(const WriteProgress& progress) {
  return progress.bytes_done == progress.bytes_total;
}
\end{lstlisting}

这个片段没有包装系统调用，而是先固定语义：写出路径必须有进度对象。同步版本在循环里推进进度；异步版本在 completion handler 里推进进度。异步化不改变短写语义，只是把“下一步继续写”从调用栈搬到请求状态表。

\topic{CompletionRecord}
Compute Systems Engine 使用 \code{CompletionRecord} 保存 I/O completion 的结构化事实，作用类似第 14 章的 \code{TaskStateRecord}。业务层只能根据 completion 改变提交状态，不能在 submit 后提前宣布成功。

\begin{lstlisting}[language=C++,caption={CompletionRecord 的示例形态}]
enum class IoStatus : std::int32_t {
  Succeeded,
  Partial,
  RetryableError,
  FatalError,
  Canceled,
  TimedOut,
};

struct CompletionRecord {
  std::uint64_t request_id = 0;
  std::uint64_t block_id = 0;
  std::uint32_t attempt = 0;
  IoStatus status = IoStatus::Succeeded;
  std::uint64_t bytes_requested = 0;
  std::uint64_t bytes_transferred = 0;
  std::uint64_t checksum = 0;
  std::uint64_t submit_ns = 0;
  std::uint64_t complete_ns = 0;
  std::uint32_t error_code = 0;
};
\end{lstlisting}

\code{request_id} 区分一次 I/O 请求，\code{block_id} 区分业务输出，\code{attempt} 区分重试或迟到完成。若 completion 不带这些身份，完成顺序一乱，driver 就无法判断哪个结果仍然有效。第 16 章的 RPC reply 会复用同样思想，只是 completion 从本机设备事件变成网络响应。

\topic{迟到 completion}
请求超时后，底层操作可能仍然完成。上层“不等了”不等于设备“不会再回来”。Completion 到达时，状态机要先检查 request 是否仍然有效。若新 attempt 已经提交，旧 completion 只能释放资源、记录 late completion 或清理临时文件，不能重复提交业务结果。

这就是为什么写出路径不能直接写最终文件。旧 attempt 即使迟到成功，也应只写自己的临时文件；最终 manifest 由 driver 决定哪个 attempt 生效。没有临时文件和 manifest，迟到 completion 就可能覆盖新结果。

\section{从系统调用到设备完成：底层路径账本}

理解 I/O 不能只看 C++ 函数返回值。一次文件写入可能穿过用户态 buffer、系统调用入口、VFS、文件系统、page cache、writeback、块层队列、设备缓存和完成中断。不同路径暴露给应用的“成功”含义不同：\code{write} 返回成功，可能只说明字节进入 page cache；\code{fsync} 返回成功，才把承诺推进到更强的持久化边界；manifest 提交成功，才说明业务上这份输出可见。

\topic{普通文件写入的事件链}
最常见的 \code{write(fd, buf, n)} 并不是直接把 \code{buf} 写到盘片或闪存。典型路径可以粗略写成：

\begin{lstlisting}[numbers=none,caption={普通文件写入的底层事件链}]
user buffer contains OutputBlock bytes
enter kernel through write syscall
copy bytes from user memory into page cache pages
mark pages dirty and update inode/file metadata in memory
return bytes_written to user space
writeback thread later submits dirty pages to block layer
storage device completes writeback requests
fsync waits for selected dirty data and metadata to reach durable boundary
manifest commit records which block is externally visible
\end{lstlisting}

这条链解释了两个常见误判。第一，\code{write} 很快不代表设备很快，它可能只是内存 copy 快。第二，进程崩溃和机器掉电不是同一类故障：进程崩溃后 page cache 仍在内核中，掉电后易失缓存可能消失。可靠性承诺必须说明恢复边界：只承诺进程崩溃恢复，还是承诺掉电后也能恢复；只承认 \code{fsync} 后的数据，还是允许未同步中间结果重算。

\topic{案例：一次 \code{write} 穿过 VFS 和 page cache}
现在把普通文件写入固定成一个可分析案例。Compute Systems Engine 要写出一个 64 MiB 的 \code{ShuffleBlock} 到本地文件系统。业务代码看到的是 \code{write_all(fd, buffer)}；内核看到的是 fd 对应的 open file description、VFS 文件对象、文件系统方法、page cache 页、脏页队列、writeback 和块层请求。这个案例的目标不是背 Linux 源码函数名，而是训练读者分清：哪一步证明用户态请求被接受，哪一步证明数据进入内核缓存，哪一步证明文件系统推进持久边界，哪一步才是业务提交。

\begin{lstlisting}[language=C++,caption={write VFS probe：固定一个写入案例}]
int fd = ::open("shuffle-block.tmp",
                O_CREAT | O_TRUNC | O_WRONLY | O_CLOEXEC,
                0644);
if (fd < 0) {
    return error_from_errno(errno);
}

std::vector<std::byte> block(64ull * 1024 * 1024);
fill_shuffle_block(block);

std::uint64_t off = 0;
while (off < block.size()) {
    ssize_t n = ::write(fd, block.data() + off, block.size() - off);
    if (n > 0) {
        off += static_cast<std::uint64_t>(n);
        continue;
    }
    if (n < 0 && errno == EINTR) {
        continue;
    }
    return error_from_errno(errno);
}

if (require_durable) {
    if (::fsync(fd) != 0) {
        return error_from_errno(errno);
    }
}
if (::close(fd) != 0) {
    return error_from_errno(errno);
}
\end{lstlisting}

这段代码故意使用普通 \code{write}，不使用 direct I/O、\code{io_uring} 或 mmap。原因是普通路径最容易误判：\code{write} 返回很快时，读者容易以为“磁盘很快”；实际可能只是用户态字节被复制到 page cache 并标成 dirty。若不调用 \code{fsync}，短时间 benchmark 很可能测的是内存带宽、页分配、VFS 锁和 dirty throttling 前的提交速度，而不是设备持久化吞吐。

\begin{lstlisting}[numbers=none,caption={本案例的 VFS/page cache 逻辑路径}]
fd number in process
  -> open file description: current offset and status flags
  -> VFS file object and file operations
  -> filesystem write path
  -> address_space page cache pages
  -> dirty page accounting
  -> background writeback
  -> block layer request queue
  -> device completion
  -> fsync/fdatasync waits for the promised durability boundary
\end{lstlisting}

VFS 的作用是把“对 fd 写字节”转成“对某个文件系统对象执行写方法”。应用不应假设 ext4、xfs、tmpfs、FUSE、NFS 和 overlayfs 的细节相同。VFS 层提供统一入口，具体文件系统决定 page cache、metadata、日志、权限、copy、direct I/O 和同步语义如何落地。因此报告必须写明文件系统、挂载选项、是否 tmpfs、是否容器 overlay、是否网络文件系统。没有这些字段，\code{write} 案例就不能迁移。

\begin{tabular}{p{0.20\linewidth}p{0.38\linewidth}p{0.30\linewidth}}
\toprule
层级 & 观察到的事实 & 不能推出的结论 \\
\midrule

\topic{案例证据包：命令要证明哪一层}
这个案例至少要收集四类证据。第一类证明应用确实调用了 \code{open/write/fsync/close}。第二类证明写入不是一次魔法设备操作，而是经过 page cache 和 dirty accounting。第三类观察 writeback 或同步等待。第四类记录业务层是否已经发布 manifest。每类证据都有边界，不能互相替代。

\begin{lstlisting}[language=bash,caption={write/VFS 案例的观测命令}]
# 系统调用边界：证明用户态请求形状。
strace -tt -e trace=openat,write,fsync,fdatasync,close ./write_vfs_probe

# 文件系统和挂载事实：证明案例跑在哪个语义环境。
findmnt -T ./shuffle-block.tmp
stat -f .

# page cache / dirty 债务：观察内核是否积累脏页。
grep -E 'Dirty|Writeback|Cached' /proc/meminfo
cat /proc/vmstat | grep -E 'nr_dirty|nr_writeback|pgpgout'

# 进程级 I/O 账本：观察写入字节和系统调用数量。
cat /proc/$PID/io

# 调度和阻塞：观察 fsync/write 是否让线程睡眠。
perf trace -e sys_enter_write,sys_exit_write,sys_enter_fsync,sys_exit_fsync ./write_vfs_probe
\end{lstlisting}

\code{strace} 能证明 \code{write} 调用次数、返回值和 \code{fsync} 是否发生，但它不能证明数据已经到设备。\code{/proc/meminfo} 和 \code{/proc/vmstat} 能看到系统级 dirty/writeback 变化，却不能把每个脏页精确归因到某个业务 block。\code{/proc/<pid>/io} 能说明进程发起的 I/O 账本，但字段含义受内核路径和缓存影响。若有权限，可以再用 ftrace、eBPF 或文件系统 tracepoint 细化路径；没有权限时，报告必须写明证据边界。

\topic{两个对照：无 \code{fsync} 和有 \code{fsync}}
同一段写入代码要至少跑两个模式。无 \code{fsync} 模式回答“应用把字节交给内核有多快”；有 \code{fsync} 模式回答“文件系统同步承诺有多贵”。二者不是同一个指标。

\begin{lstlisting}[numbers=none,caption={write/VFS 对照报告字段}]
write_vfs_report:
  filesystem
  mount_options
  file_size_bytes
  block_size_bytes
  write_call_count
  write_returned_bytes
  write_elapsed_ms
  fsync_enabled
  fsync_elapsed_ms
  close_elapsed_ms
  dirty_before_bytes
  dirty_after_write_bytes
  dirty_after_fsync_bytes
  proc_io_write_bytes
  manifest_committed
  evidence_limitations
\end{lstlisting}

若无 \code{fsync} 模式吞吐很高，而有 \code{fsync} 模式 p99 很差，结论不是“\code{fsync} 可删”。正确结论是：同步承诺成本高，需要决定哪些输出必须 durable、哪些中间结果可重算，是否使用 group commit，是否把 fsync 从 compute worker 隔离到 I/O worker。若两个模式都很慢，瓶颈可能是内存 copy、文件系统锁、dirty throttling、设备队列或容器存储层。只有把对照字段写全，才知道要优化哪一层。

\topic{VFS 案例的错误结论清单}
这个案例最重要的训练价值，是防止把不同层级证据混用。看到 \code{write} 返回 64 MiB，不得写“64 MiB 已经落盘”；看到文件大小变大，不得写“崩溃后一定可恢复”；看到 \code{fsync} 慢，不得写“线程池调度器慢”；看到 dirty bytes 增长，不得写“进程内存泄漏”；看到 \code{write} 偶尔变慢，不得只怀疑磁盘，也要考虑 dirty page 阈值、回写限速、文件系统日志和同步写策略。

\topic{读路径和 page fault}
读取也不一定真的触碰设备。热 page cache 中的 \code{read} 可以只做内存 copy；冷 page cache 的 \code{read} 会触发设备 I/O；\code{mmap} 顺序访问表面上是普通 load，底层可能在 page fault 中等待磁盘。若 compute worker 直接访问冷 mmap 区域，线程可能在缺页路径上睡眠，表现为 CPU 空闲但 ready task 堆积。

\topic{Ready 事件和 completion 事件的根本差别}
Ready 事件说明“现在尝试可能不会立刻阻塞”。它是机会通知，不是完成事实。Socket 可读可能只有半个消息头，管道可写可能只能写一部分，边沿触发还要求应用一直读到 \code{EAGAIN}。Completion 事件说明“某个已提交请求已经得到结果”。它带 request id、字节数和错误码，可以推进请求状态机。

这两种模型会导出不同代码结构。Ready 模型需要应用自己维护每个连接的读写进度：还有多少 header 没读、body 还差多少、send buffer 还剩多少。Completion 模型需要应用维护请求表：哪个 request 已提交，buffer 归谁，completion 回来时是否仍有效。二者都不能省掉业务状态机，只是把等待位置放在不同地方。

\topic{Direct I/O 和 registered buffer 的边界}
Direct I/O、registered buffer、零拷贝发送看起来能减少 copy，但它们会收紧所有权和对齐要求。Buffer 可能要求按页对齐，大小按块对齐，生命周期必须覆盖内核或设备访问窗口。减少一次 copy 的代价，是业务层更难提前复用内存，也更难处理取消和迟到 completion。

Direct I/O 的基础合同必须讲得更窄：它不是“更快的 \code{write}”，而是“尽量不通过 page cache 保存文件数据页”的 I/O 形态。Linux 上常见入口是 \code{O_DIRECT}，但具体支持、对齐要求、和 page cache 的一致性细节受文件系统、块设备、内核版本和挂载选项影响。通常要同时满足用户 buffer 地址对齐、文件 offset 对齐、I/O size 对齐；不满足时可能失败，或者在某些路径上退回到不同实现。它仍然要经过 VFS、权限检查、文件系统方法、块层队列和 completion；它也不自动提供异步语义，更不自动提供掉电持久化。

buffered I/O & 简单，利用 page cache、readahead、writeback 合并 & 可能把设备等待隐藏成 dirty debt，热缓存测不到设备吞吐 \\
direct I/O & 减少 page cache copy 和缓存污染，更接近设备队列 & 对齐和生命周期更硬，随机小 I/O 可能更差，仍需处理 completion 和 fsync 语义 \\
\code{mmap} & load/store 形式访问文件，适合部分随机读 & page fault 混入计算路径，SIGBUS 和映射生命周期要处理 \\
registered buffer & 减少重复 pin/register 成本 & buffer pool 必须和请求状态机绑定，取消和迟到 completion 更复杂 \\
\bottomrule
\end{tabular}
\end{center}

因此选择 buffered 还是 direct，必须先写 workload。顺序读大文件时，buffered I/O 的 readahead 可能已经很好；大量一次性读写大 block、且不希望污染 page cache 时，direct I/O 才可能有意义；可靠写出时，不管选择哪条路径，都要说明 durable 边界和业务 commit 边界。若报告只写“用了 \code{O_DIRECT} 所以绕过内核缓存”，仍然不合格：它还要列出对齐、文件系统、设备、是否混用 buffered I/O、以及如何证明实际路径。

因此 Compute Systems Engine 的第一版不应追求零拷贝。先用普通 buffer 和 I/O worker 把状态机做对，再用报告证明 copy 成为瓶颈。若 copy 只占总时间很小，零拷贝会增加复杂度却没有明显收益。若 block 很大、CPU copy 确实占主导，再把 buffer pool、对齐、pinning、registered buffer 和 completion ownership 加入设计。

\topic{写入路径的五个成功层级}
“写成功”必须拆成层级，否则可靠性讨论会混乱。

\begin{center}
\begin{tabular}{p{0.18\linewidth}p{0.38\linewidth}p{0.32\linewidth}}
\toprule
层级 & 含义 & 仍未解决的问题 \\
\midrule
Accepted & 请求进入用户态或内核队列 & 可能尚未写任何字节 \\
Copied & 字节进入内核 page cache 或发送缓冲 & 设备可能未完成，掉电可能丢失 \\
Completed & 底层请求返回成功或失败 & 业务是否接受该 attempt 尚未确定 \\
Durable & 数据和必要 metadata 到达承诺边界 & 是否被 manifest 引用尚未确定 \\
Committed & manifest 接受该 block & 后续消费者可以读取 \\
\bottomrule
\end{tabular}
\end{center}

异步 I/O 编程的核心就是不把这些层级混成一个布尔值。Submit 返回只能说明 Accepted；completion 只能说明 Completed；\code{fsync} 推进 Durable；manifest 推进 Committed。不同层级的失败处理不同：Accepted 前失败可以直接把 buffer 还给 producer；Completed 失败要记录错误和释放 in-flight；Durable 前崩溃可能重写；Committed 后崩溃必须承认提交事实。

\topic{Dirty page 是隐藏队列}
普通文件写入返回后，脏页可能留在 page cache 中等待 writeback。这个等待队列不在用户态线程池里，却消耗内存、内核回写带宽、块层队列和设备吞吐。若写入速度长期高于设备完成速度，dirty pages 会积累；达到系统阈值后，普通 \code{write} 可能突然变慢，甚至调用线程被迫参与回写或等待脏页下降。应用如果只看自己的提交队列长度，会误以为没有积压；实际上债务已经转移到内核。

\begin{lstlisting}[numbers=none,caption={写入债务从用户态转移到内核的过程}]
producer generates output blocks faster than storage can persist
writer calls write and receives success quickly
page cache dirty bytes grow
background writeback submits requests to storage
storage queue latency grows
kernel throttles dirtying tasks or fsync waits longer
application sees sudden write or fsync latency spikes
\end{lstlisting}

这就是背压必须看多层指标的原因。用户态队列空，不代表系统没有债务；\code{write} 平均延迟低，不代表可靠提交快；吞吐看起来高，可能只是把数据堆进易失缓存。稳态判断要同时观察 output 生成速率、write 返回速率、fsync 或 durable 速率、dirty bytes、completion 延迟和 manifest commit 速率。任何一层长期低于上游，都需要把背压向上游传播。

\topic{fsync 不是单纯慢函数}
\code{fsync} 慢，是因为它把之前隐藏的持久化债务拉到当前线程面前。它可能等待文件数据、metadata、日志提交、块层请求和设备缓存 flush；具体语义受文件系统、挂载选项、设备和内核影响。不能因为 \code{fsync} 慢就删掉它，也不能因为 benchmark 中 \code{fsync} 很快就认为可靠写没有成本。正确做法是说明承诺边界：需要掉电恢复时，必须把必要数据和 metadata 推到文件系统承诺的持久边界；只需进程崩溃恢复时，可以把部分中间结果作为可重算缓存处理。

\begin{center}
\small
\begin{tabular}{@{}p{0.20\linewidth}p{0.34\linewidth}p{0.34\linewidth}@{}}
\toprule
承诺 & 需要等待的事实 & 可接受的恢复动作 \\
\midrule
进程内完成 & request 状态转为 completed & 进程崩溃后可丢弃并重算 \\
进程崩溃恢复 & 内核仍可见的文件或临时状态 & 重启扫描并校验 checksum \\
掉电恢复 & fsync 后的数据和必要 metadata & 依赖文件系统承诺，缺失则重算 \\
业务提交 & manifest 接受某个 attempt & 后续消费者以 manifest 为准 \\
\bottomrule
\end{tabular}
\end{center}

这张表把可靠性和性能拆开。系统可以选择某些中间 block 不做强持久化，只要 manifest 不把它们宣布为已提交；也可以选择 group commit，把多个 block 的持久化成本合并。无论选择哪种策略，都要写入恢复协议，而不是让恢复逻辑从文件名猜状态。

\topic{rename 只是名字切换，不是完整提交协议}
可靠写常用模式是：写临时文件，校验，\code{fsync} 文件，\code{rename} 到目标名，再更新 manifest。这里每一步都有不同含义。\code{rename} 在许多本地文件系统上能提供同目录下名字切换的原子性：恢复时通常看到旧名字或新名字，不会看到半个名字。但名字原子性不等于数据已经持久，也不等于目录项一定经受住掉电。若承诺掉电恢复，通常还要考虑对文件和目录的同步边界；若只承诺进程崩溃恢复，page cache 中的目录和文件状态仍可能足够，但恢复协议要明确。

\begin{lstlisting}[numbers=none,caption={可靠 block 写入的提交层次}]
write temp file:
  bytes enter page cache or direct I/O path

fsync temp file:
  block bytes and required file metadata reach durable boundary

rename temp -> final:
  directory name now points to the final block name

fsync parent directory:
  directory entry update reaches durable boundary when required

append or rewrite manifest:
  business layer accepts this attempt as visible output
\end{lstlisting}

这个顺序不表示所有系统都必须对每个中间 shuffle block 做最强同步。批处理系统可以把中间 block 视为可重算缓存，只对最终 manifest 做更强承诺；也可以用 group commit 把多个文件和 manifest 的同步成本合并。关键是不能把 \code{rename}、\code{fsync} 和 manifest commit 混成一个“保存成功”。\code{rename} 解决名字切换，\code{fsync} 解决持久边界，manifest 解决业务可见性和 attempt 选择。

\topic{Manifest 为什么比最终文件名更强}
只靠最终文件名判断成功，会在重试和迟到 completion 中出错。假设 split 7 的 attempt 0 写出了 \filepath{part-7.tmp.0}，超时后 attempt 1 写出了 \filepath{part-7.tmp.1} 并提交 manifest。之后 attempt 0 迟到完成并被 rename 成 \filepath{part-7.final}。如果下游只按文件名读取，它可能读到旧 attempt；如果下游只读 manifest，它会知道 accepted attempt 是 1，attempt 0 即使文件完整也不能生效。

\begin{lstlisting}[numbers=none,caption={manifest 条目表达的业务事实}]
logical_partition = 7
accepted_attempt = 1
file_path = part-7-attempt-1.block
records = 842193
bytes = 67128320
checksum = 0x8f21...
partition_plan_version = 3
\end{lstlisting}

Manifest 的作用不是重复文件系统目录，而是把业务身份、attempt、校验、分区规则和可见性绑在一起。文件存在是物理事实；manifest 接受是业务事实。I/O 层必须把这两个事实分开，否则恢复程序无法判断哪些文件是 orphan，哪些是 late attempt，哪些才是后续 stage 的输入。

\topic{系统调用边界和 C++ 内存边界不同}
一个常见混淆是把线程间可见性和持久化可见性放在一起。C++ 的 release/acquire 或 mutex 可以保证另一个线程看到 buffer 中已经写好的字节；系统调用可以把这些字节交给内核；\code{fsync} 可以把文件推进到文件系统承诺的持久边界；manifest commit 可以让业务层承认输出。它们是不同层级的顺序关系。

\begin{center}
\small
\begin{tabular}{@{}p{0.24\linewidth}p{0.34\linewidth}p{0.30\linewidth}@{}}
\toprule
边界 & 解决的问题 & 不能替代的层级 \\
\midrule
C++ release/acquire & 线程看到已构造请求和 buffer 字段 & 不能保证内核接受或磁盘持久 \\
\code{write} 返回 & 内核接受部分或全部字节 & 不能保证 durable 或业务提交 \\
\code{fsync} 返回 & 文件系统承诺的持久边界推进 & 不能决定哪个 attempt 生效 \\
\code{rename} 返回 & 目录命名状态改变 & 不能单独表达业务校验和分区版本 \\
manifest commit & 业务可见输出被接受 & 依赖底层文件和恢复协议的承诺 \\
\bottomrule
\end{tabular}
\end{center}

这张表能解释为什么 I/O 代码看起来“步骤很多”。每一步解决一个不同故障模型。删掉其中一步，系统未必立刻变错，但承诺边界已经改变。报告必须写明这种改变，而不是只写“优化掉 fsync 后变快”。

\topic{Completion 线程不能成为新瓶颈}
异步 I/O 常把等待从提交线程移到 completion loop。Completion loop 的职责应尽量短：取回结果、更新 request 状态、释放或转交 buffer、记录错误、推进 manifest 或通知上层。若在 completion 回调里做压缩、解析、复杂合并、同步日志格式化或长时间持锁，完成队列会被阻塞，设备已经完成的请求无法及时回收 buffer，上游看到的背压会被放大。

Completion 线程本质上也是运行时资源。它需要自己的队列、优先级、关闭协议和异常边界。若 completion loop 和 compute worker 共享同一个线程池，I/O 完成可能被数据任务淹没；若 completion loop 过多，又会争抢 CPU 和 cache。通常更稳的结构是：completion loop 只做轻量状态推进，重 CPU 后处理重新投递到计算队列，可靠提交和 manifest 更新有独立、小而可审计的路径。

\begin{lstlisting}[numbers=none,caption={completion loop 的短路径原则}]
on completion:
  locate request by request_id
  validate attempt and current state
  record bytes, error, and latency
  release or transfer buffer ownership
  enqueue follow-up CPU work if needed
  perform only bounded manifest/commit work
\end{lstlisting}

这条原则不是追求代码整洁，而是防止完成事件本身形成新的隐藏队列。异步系统真正的吞吐上限取决于 submit、device、completion、commit 和 recovery 这些阶段的共同稳态。

\section{完整案例：从阻塞 write 改造成有背压的完成流水线}

现在把 I/O 路径写成一次完整改造。初始版本由 compute worker 直接调用 \code{write} 写 shuffle block。小输入时它很简单；输入变大后，worker 有时卡在 \code{write}、\code{fsync} 或冷 \code{mmap} 缺页里，线程池看起来还有线程，实际可运行计算任务越来越少。第 9 章把这种现象归为 blocked worker；本章要把它改成可观测的 I/O 状态机。

\topic{第一版：阻塞写把等待混进计算预算}
朴素代码通常长这样：

\begin{lstlisting}[numbers=none,caption={阻塞写版本把设备等待放进 worker}]
for each batch assigned to worker:
    partial = aggregate(batch)
    block = serialize(partial)
    write_all(output_fd, block.bytes)
    fsync_if_required(output_fd)
    append_manifest(block.metadata)
\end{lstlisting}

这个版本的问题不是“同步 I/O 一定错”，而是等待边界不清。Worker 既做 CPU 聚合，又做序列化，又等待存储，又提交 manifest。Profile 可能显示 worker 在系统调用或内核路径里，任务运行时却只看到 worker 忙。上游继续提交任务，dirty page 和输出队列继续增长，背压来得太晚。

\topic{第二版：把请求、buffer 和 attempt 分开}
改造第一步是定义 \code{WriteRequest}。它携带逻辑身份、attempt、buffer、目标路径、可靠性策略和状态。提交请求后，buffer 所有权转移给 I/O 子系统；compute worker 只拿到一个 request id 或 future。

\begin{lstlisting}[language=C++,caption={WriteRequest 表达 I/O 状态机输入}]
struct WriteRequest {
  std::uint64_t logical_partition = 0;
  std::uint32_t attempt = 0;
  std::uint64_t request_id = 0;
  BufferHandle buffer;
  std::string temp_path;
  bool requires_fsync = false;
  std::uint64_t checksum = 0;
};
\end{lstlisting}

\code{logical_partition} 和 \code{attempt} 防止迟到 completion 越权提交；\code{request_id} 定位 in-flight 表；\code{BufferHandle} 保护生命周期；\code{requires_fsync} 明确持久化承诺；\code{checksum} 让 completion 和 manifest 能验证数据身份。没有这些字段，异步只是把错误延迟到以后发生。

\topic{第三版：completion 只推进状态}
I/O worker 或 completion loop 负责处理 \code{WriteRequest}。它可以内部使用阻塞写线程、\code{epoll}、\code{io\_uring} 或模拟完成队列；原理卷不绑定某个接口。关键是 completion 回来时只做有界工作：校验 request 当前仍有效，记录结果，释放或转交 buffer，必要时把 manifest 候选投递给提交线程。

\begin{lstlisting}[numbers=none,caption={完成流水线的状态转移}]
compute worker:
  build block in owned buffer
  submit WriteRequest
  continue CPU work or respect backpressure

io subsystem:
  write temp bytes, handle short writes
  fsync if requested by policy
  emit CompletionRecord(request_id, bytes, error, durable_state)

completion loop:
  validate request still current
  recycle or transfer buffer
  enqueue ManifestCandidate only if completion is acceptable
\end{lstlisting}

这条流水线把等待从 compute worker 移走，但没有让等待消失。设备仍然慢，fsync 仍然慢，buffer 仍然有限。区别是等待变成了可测队列，而不是混在任意 worker 的调用栈里。

\topic{第四版：背压从最低吞吐阶段向上游传播}
异步系统最危险的错觉是“提交很快，所以系统很快”。如果 submit 速率长期高于 durable 或 commit 速率，in-flight 请求、buffer、dirty pages 或临时文件会堆积。背压要从最慢的稳定阶段向上游传播。可以设多个水位线：free buffer 数量、in-flight bytes、dirty estimate、completion latency、manifest queue 长度。

\begin{lstlisting}[numbers=none,caption={多层背压水位线}]
if free_buffers < low_watermark:
    stop accepting new serialization work

if inflight_write_bytes > write_budget:
    slow down map-side combine output

if fsync_latency_p99 exceeds budget:
    increase group commit window or reduce output concurrency

if manifest_queue grows:
    stop admitting new attempts for the same stage
\end{lstlisting}

背压不是错误路径，而是正常控制流。它的目标不是让队列永远为空，而是让每层债务有上限：buffer 不被复用过早，dirty pages 不无限增长，completion 不被 CPU 后处理堵住，manifest 不被重复 attempt 淹没。

\topic{第五版：关闭和取消}
关闭路径要比正常路径更严格。停止接收新请求，不代表已有请求不存在；取消 attempt，不代表底层 I/O 不会完成；释放 buffer，不代表设备不再访问。关闭状态机应按阶段推进：

\begin{lstlisting}[numbers=none,caption={I/O 流水线关闭顺序}]
stop admission:
  reject new WriteRequest or mark stage closing

drain or cancel:
  choose whether in-flight requests may finish
  mark canceled attempts so late completions cannot commit

reap completions:
  process every completion until ownership is resolved
  recycle buffers only after completion or proven cancellation boundary

commit or discard:
  append manifest for accepted attempts
  remove orphan temp files according to recovery policy
\end{lstlisting}

这个顺序和线程池关闭、条件变量谓词、原子状态机是一条主线：状态先改变，等待者或 completion 再观察状态，资源最后释放。若顺序反过来，就会出现 use-after-free、重复提交、孤儿文件或丢失唤醒。

\topic{第六版：状态观测点}
状态观测点要能解释系统是 CPU 慢、设备慢、completion 慢还是 commit 慢：

\begin{lstlisting}[numbers=none,caption={异步写流水线观测点}]
io.submit.count
io.submit.rejected_by_backpressure
io.inflight.requests
io.inflight.bytes
io.short_write.count
io.completion.latency_p50
io.completion.latency_p99
io.fsync.latency_p99
io.buffer.free_count_min
io.dirty_bytes_if_available
manifest.queue_depth_max
manifest.commit.latency_p99
late_completion.count
orphan_temp_files_reclaimed
\end{lstlisting}

有了这些观测点，调优才不会盲目。若 free buffer 见底而设备延迟高，瓶颈在持久化；若 completion 队列增长但设备延迟不高，completion loop 太重；若 manifest 队列增长，提交路径串行化；若 late completion 多，attempt timeout 或取消策略需要复查。

\section{所有权、Buffer 池和取消}

异步 I/O 最常见的正确性 bug 不是系统调用用错，而是 buffer 生命周期用错。同步调用返回后，调用者通常可以复用 buffer；异步提交后，设备、内核、I/O 线程或发送队列可能仍在读取这个 buffer。若上游提前复用，磁盘或网络看到的就是后续数据。高性能系统常用 buffer pool 减少分配，但 buffer pool 只有配套状态机才安全。

\topic{所有权转移}
Producer 从 pool 取出 free buffer，填充数据后提交给 writer。提交成功后，buffer 所有权转移给 I/O 子系统；producer 不能再写。Completion 处理完以后，buffer 才能进入 recycling 或 free。取消、失败、短写都必须走 completion 或 reap 路径，不能绕开状态机。

\topic{Buffer 池也是背压}
Buffer 池大小决定最多 in-flight 字节。若每个 buffer 1 MiB，允许 512 个 in-flight，就已经占用 512 MiB。池耗尽时，上游必须等待、降低读取速率、减小 batch 或返回 backpressure。这比只限制队列条目数更直接，因为少量超大 block 也会击穿内存预算。

项目应同时记录 item count 和 byte count。队列里 100 个 4 KiB 请求和 100 个 64 MiB 请求不是同一种压力。第 18 章 shuffle 的 in-flight bytes 限制，正是从这里来的。

\topic{取消是尽力而为}
取消请求后，底层操作可能已经完成、正在内核或设备中、还在队列里、或者尚未提交。取消只能改变上层接受结果的策略，不能保证物理写入没有发生。状态机至少要区分 \code{Canceled} 和 \code{Reaped}：前者表示用户不再接受业务结果，后者表示底层完成事件已经被消费、资源可以释放。

若取消后直接销毁上下文，迟到 completion 就可能访问已释放对象。正确方式是让 request context 活到 reaped；completion 到达时检查状态，若已取消，则释放 buffer、清理临时文件、记录 late completion，不提交 manifest。

\topic{关闭时 drain 还是 cancel}
I/O 子系统关闭有两种常见语义。Drain 表示拒绝新请求，但继续处理已经接受的请求，直到它们完成或失败；cancel 表示未开始请求取消，in-flight 请求尽力取消或等待 completion 后清理。日志调试数据可能允许 cancel，checkpoint、manifest、必须完成的 shuffle block 通常需要 drain 或明确失败。

关闭不是析构时设置一个布尔值。它要停止接收新请求、唤醒等待者、处理队列中请求、消费 completion queue、归还 buffer、报告未完成请求并 join 线程。若 completion queue 没有 drain，底层稍后完成可能访问已经销毁的上下文。

\topic{WriteRequest 和 completion loop}
异步 I/O 的中心不是 submit，而是 completion loop。Submit 只是把请求交出去；completion loop 接收完成事件，更新请求状态，推进短写 offset，归还 buffer，触发后续 task，汇总错误。若 completion loop 设计不清，异步代码就会散成回调、锁和共享变量，最后很难证明同一个请求只完成一次。

\begin{lstlisting}[language=C++,caption={WriteRequest 的示例状态}]
enum class WriteRequestState : std::int32_t {
  Created,
  Queued,
  Submitted,
  PartialWritten,
  Completed,
  Failed,
  Canceled,
  Reaped,
};

struct WriteRequest {
  std::uint64_t request_id = 0;
  std::uint64_t block_id = 0;
  std::uint32_t attempt = 0;
  std::uint64_t buffer_id = 0;
  std::uint64_t bytes_total = 0;
  std::uint64_t bytes_done = 0;
  WriteRequestState state = WriteRequestState::Created;
};
\end{lstlisting}

Completion loop 收到事件后，第一步不是执行业务回调，而是查找 \code{request_id}，验证 attempt，检查当前状态。若状态已经是 \code{Canceled}，completion 只能清理底层资源并进入 \code{Reaped}；若是 \code{Submitted} 且只写了一部分，就更新 \code{bytes_done} 并提交剩余部分；若全部完成且 checksum 通过，才触发 commit task。这个顺序能防止重复回调、迟到完成污染结果和失败路径泄漏 buffer。

\topic{完成回调不要做重 CPU 工作}
Completion loop 线程不应执行重 CPU 计算，例如解析大文件、压缩、排序或大规模 checksum 聚合。它可以做必要的状态验证和轻量 checksum 检查，然后把后续 CPU work 放回任务运行时。否则底层 I/O 已经完成，completion queue 却因为回调线程忙于计算而积压，表现为“设备不慢，但用户态迟迟看不到完成”。

这个边界和第 14 章一致：worker 只执行 ready 的 CPU 任务，I/O completion 只把等待完成的事实带回状态表。若 completion handler 直接提交 manifest、执行 reduce 或递归触发大量回调，系统会把控制面和数据面混在一起，调试难度急剧上升。

\section{背压：容量合同和传播链}

背压是系统告诉上游“下游处理不过来”的机制。无界队列把这个事实藏进内存，直到 OOM、尾延迟爆炸或恢复窗口无法清理。有界队列把容量变成合同：队列满时，生产者必须等待、失败、降级、丢弃低优先级数据或把压力继续传回源头。背压不是性能失败，而是系统保护自己。

\topic{队列长度不是唯一指标}
队列长度、队列等待时间、入队速率、出队速率、in-flight bytes、completion 延迟和错误率要一起看。长度高但短暂，可能只是 burst；长度持续增长，说明生产速度长期超过消费速度；长度不高但等待时间长，可能是每个请求很大；completion 延迟高，可能是设备慢；completed 到 committed 时间高，可能是提交路径慢。

I/O 报告应拆分延迟：queued 到 submitted，submitted 到 completed，completed 到 committed。只看端到端总时间，无法判断背压该加在 reader、compute、writer 还是 manifest。

\topic{高低水位线}
单一容量会让系统在“满”和“不满”之间频繁抖动。高低水位线更稳定：超过高水位时上游暂停或减速；低于低水位时恢复。高水位不能太接近容量，否则反馈太晚；低水位不能太接近高水位，否则频繁开关。水位线要用实验调，不是随手写一个数字。

\topic{策略有业务语义}
队列满时怎么处理，取决于数据价值。准确计算的 shuffle block 不能静默丢弃；调试日志可以在过载时丢弃并增加 drop counter；交互请求可能返回 busy 或 deadline exceeded；离线批处理可以等待更久。一个 bool 返回值不够，接口应区分 accepted、would block、timed out、closed、canceled 和 failed。

\begin{lstlisting}[language=C++,caption={背压提交结果的示例枚举}]
enum class BackpressureResult : std::int32_t {
  Accepted,
  WouldBlock,
  TimedOut,
  Closed,
  DroppedByPolicy,
};

struct QueueCapacity {
  std::uint64_t max_items = 0;
  std::uint64_t max_bytes = 0;
  std::uint64_t high_water_bytes = 0;
  std::uint64_t low_water_bytes = 0;
};
\end{lstlisting}

策略要可观测。若丢弃低优先级日志，必须记录丢弃数量、优先级和原因；若阻塞生产者，必须记录阻塞时间；若返回 busy，必须记录上游是否尊重了 busy。静默背压和静默丢弃都会让系统不可诊断。

\topic{deadline 和 timeout}
Timeout 是某个操作愿意等待多久；deadline 是整个请求最晚可以完成的时间点。流水线更适合传 deadline，因为请求经过 read、parse、compute、write、commit 多个阶段，每个阶段都应知道剩余预算。若每段都重新获得完整 timeout，端到端延迟可能无限膨胀。

Deadline 也约束重试。剩余 5 ms 的请求，不应启动预期 50 ms 的 fsync 或 RPC 重试。分布式章节还会遇到时钟和跨节点传播问题；这里先用单调时钟建立“剩余预算”这个思维。

\topic{延迟分解比总耗时更有解释力}
I/O pipeline 的端到端时间应拆成阶段：read service、parse queue wait、parse service、compute queue wait、compute service、write queue wait、write service、commit wait。若总耗时上升但 write service 没变，可能是队列等待增加；若 write service 上升，可能是设备、fsync 或文件系统路径变慢；若 commit wait 上升，可能是 group commit 等批或 manifest 锁。

\section{可靠写入：从临时文件到 Manifest}

写文件成功不等于结果可靠。普通 \code{write} 可能只是把数据放进 page cache；rename 可能只更新目录项；manifest 可能已写但未持久化；进程可能在任意步骤崩溃。可靠写入的目标不是让所有中间状态都消失，而是让每个中间状态都能被恢复程序解释。

\topic{先写临时文件}
输出不应直接覆盖最终文件。Writer 先写临时路径，临时名包含 stage、partition、task、attempt 和随机后缀。数据完整、checksum 通过、必要同步完成后，再通过 rename 或 manifest commit 让它成为可见结果。未提交临时文件不是有效输出，reduce 不应扫描临时目录取数据。

\topic{崩溃点矩阵}
可靠 I/O 不是一句“使用 fsync”，而是一张崩溃点矩阵。每一步都要说明崩溃后恢复怎么处理。

\begin{center}
\begin{tabular}{p{0.24\linewidth}p{0.34\linewidth}p{0.30\linewidth}}
\toprule
崩溃位置 & 恢复时看到什么 & 恢复动作 \\
\midrule
创建临时文件前 & 没有候选输出 & 重新执行 task 或保持未完成 \\
写临时文件中 & 可能有残缺临时文件 & 校验失败，删除或保留诊断 \\
写完但未校验 & 文件存在但身份未确认 & 重新校验，失败则重做 \\
校验通过但未提交 manifest & 候选文件存在，外部不可见 & 按策略补提交或清理孤儿 \\
manifest 已提交 & 权威记录引用该 block & 接受为有效输出 \\
清理旧文件前 & 旧临时文件仍在 & 根据 manifest 再次清理 \\
\bottomrule
\end{tabular}
\end{center}

项目可以先承诺进程崩溃恢复，再把掉电持久化作为扩展。若要求掉电后恢复，需要考虑文件 fsync、目录 fsync、manifest fsync、文件系统和设备缓存语义。不同平台的细节不可能完全一致，但可靠性承诺必须分层说明。

\topic{fsync、rename 和 manifest 的边界}
常见本地文件模式是：写临时文件，确保内容完整，必要时 \code{fsync} 文件，\code{rename} 到最终名，必要时 \code{fsync} 父目录，最后发布 manifest 或把 manifest 作为唯一权威入口。不同文件系统、对象存储、NFS 和移动平台语义会变化，所以项目报告必须写清文件系统、测试方式和未验证边界。

Manifest 的意义是把“哪些输出有效”集中成权威事实。外部消费者只信 manifest，不扫描目录。这样崩溃后，孤儿临时文件可清理，重复 attempt 可去重，迟到 completion 不会直接污染最终结果。第 18 章的 shuffle manifest 会直接复用这个模型。

\topic{批处理和 group commit}
每个 block 都 fsync，可靠但慢；多个 block 共享一次持久化，吞吐高但每个请求等待更久，崩溃后可能重做更多未提交工作。数量阈值和时间阈值要一起设置：达到 N 个 block 或等待 T 时间就提交。只按数量低流量时延迟高；只按时间高流量时批太小。

\begin{center}
\begin{tabular}{p{0.22\linewidth}p{0.32\linewidth}p{0.34\linewidth}}
\toprule
策略 & 适用场景 & 风险 \\
\midrule
每 block 同步 & 审计强、数据少、语义优先 & 吞吐低，尾延迟高 \\
按数量批量提交 & 高吞吐离线写出 & 低流量等待时间长 \\
按时间窗口提交 & 延迟预算明确 & 高流量批次可能偏小 \\

\bottomrule
\end{tabular}
\end{center}

\topic{可靠写入也是性能取舍}
可靠写入会增加系统调用、flush、metadata 和恢复扫描成本。它不是无条件越强越好。离线中间结果可能允许重算，最终 checkpoint 需要更强提交；调试日志可以丢，manifest 不可以丢。可靠性设计要把可靠性承诺、重做成本和性能成本放在同一个表里比较。

\topic{Page cache 会误导 benchmark}
文件 I/O benchmark 很容易被 page cache 误导。第一次读大文件可能来自设备，第二次读同一文件可能来自内存；写文件返回快，可能只是进入 page cache，真正 writeback 稍后发生。若报告没有说明冷 cache、热 cache、是否 fsync、文件大小是否超过内存、是否使用 tmpfs，结果就无法解释。

实验应区分三类目标。研究背压时，可以使用模拟 I/O 稳定控制延迟和错误；研究 page cache 时，要明确预热、文件大小和系统限制；研究可靠提交时，要关注 fsync、rename、manifest 和崩溃点。把三类目标混在同一张性能图里，缓存效果很容易被误判为 I/O 设计效果。

Page cache 还会影响内存账本。大量未落盘 dirty page 可能不计入进程 RSS，却消耗系统内存并触发回写压力。报告可以记录进程内存、系统可用内存、输出目录大小和 completion 延迟。即使指标粗略，也比完全忽略系统缓存更诚实。

\section{工业界设计案例}

工业 I/O 设计的共同点，是把等待、完成、容量和提交边界写成可审查状态。读源码时不要只列 API 名称，要说明它购买了什么能力、付出什么成本、适合什么负载。

\begin{casebox}
\topic{Linux page cache 和 writeback}
普通文件写入常先进入 page cache，dirty page 之后由 writeback 写到设备。于是 \code{write} 返回快，不代表掉电后可恢复。fsync 的价值是推动数据和相关元数据到更强持久化边界。这个设计购买了吞吐和合并写入，也带来 benchmark 误判：只测 write 返回时间，可能完全测的是内存路径。

项目报告应区分应用层背压实验、page cache 实验和可靠提交实验。研究背压时可以用模拟 I/O；研究可靠性时要关注 fsync 和崩溃点；研究真实设备吞吐时要记录 cache 热冷、文件大小、同步策略和系统权限。
\end{casebox}

\begin{casebox}
\topic{\texorpdfstring{\code{epoll}}{epoll} 的 ready 语义}
\code{epoll} 通知 fd ready，不通知业务消息完整。它购买的是少量线程等待大量 fd 的能力，不替应用处理 framing、短读、短写和业务队列。边沿触发减少重复通知，但要求应用读写到 \code{EAGAIN}，否则容易遗漏机会。

Compute Systems Engine 不必实现复杂网络 reactor，但要吸收 ready 语义的边界：事件循环线程不能做重 CPU 工作；业务消息要有状态机；完成后要把 CPU work 交给任务运行时；背压要通过停止读取、减少发送或应用层 busy 传出去。
\end{casebox}

\begin{casebox}
\topic{\texorpdfstring{\code{io_uring}}{io-uring} 的 SQ/CQ 模型}
\code{io_uring} 把提交队列和完成队列显式化。用户态提交请求，内核推进，完成队列返回结果。它购买了更低系统调用开销和更丰富的异步操作，但没有取消设备延迟、错误、容量和生命周期问题。Buffer 注册和零拷贝还会让 buffer 生命周期更严格。

项目可以先用 I/O worker 模拟异步，引入 CompletionRecord、buffer ownership、backpressure 和 reliable commit；之后再把底层替换成 \code{io_uring}。这样 API 变化不会改变主线。
\end{casebox}

\topic{SQE、CQE 和业务状态不是同一层}
\code{io_uring} 的提交队列项 SQE 描述“希望内核执行什么操作”，完成队列项 CQE 描述“内核观察到这个操作怎样结束”。业务层的 block 状态还要再往上走一层：buffer 是否仍被内核持有，短写是否已经补齐，临时文件是否完整，manifest 是否接受，迟到 completion 是否需要清理。把 CQE 直接当作业务提交，是异步 I/O 中最常见的层级错误。

\begin{lstlisting}[numbers=none,caption={从 SQE 到业务提交的状态层次}]
application:
  allocate buffer and build WriteRequest

submit:
  fill SQE with fd, offset, buffer, length, user_data
  publish request as InFlight

complete:
  read CQE result
  if result is short write:
      submit next SQE for remaining range
  if result is error:
      classify retryable or terminal
  if full byte range is written:
      verify temp block state
      request manifest commit

commit:
  manifest accepts exactly one attempt
  buffer may return to pool after no late user owns it
\end{lstlisting}

这里的 \code{user_data} 很关键。它不能只是一个裸指针，更不能指向可能已经释放的栈对象。它应该能定位稳定的 request record，例如 request id、attempt id 或对象池中的 generation handle。否则取消、超时和迟到 completion 会把已经复用的 buffer 或 request 错认成当前请求。底层 API 越异步，身份和生命周期越不能靠调用栈保存。

\topic{注册 buffer 购买少拷贝，也购买更严格的所有权}
Registered buffer 可以减少页固定、映射和检查开销，也更接近零拷贝路径。但注册 buffer 在 I/O 完成前不能被上层复用，取消也不保证设备或内核立刻停止访问。Buffer pool 因此必须有状态：Free、Filling、Submitted、Completed、Committing、Reclaimable。若只有一个 \code{std::vector<char>} 被多个请求轮流写，异步完成顺序稍一变化，就会把 block 内容、checksum 和 manifest 身份混在一起。

\begin{center}
\small
\begin{tabular}{@{}p{0.22\linewidth}p{0.36\linewidth}p{0.31\linewidth}@{}}
\toprule
状态 & 谁拥有 buffer & 允许动作 \\
\midrule
Free & buffer pool & 分配给新的写请求 \\
Filling & compute task & 写入 block 内容，尚未提交 I/O \\
Submitted & kernel/device path & 上层不能复用，等待 CQE 或取消结果 \\
Completed & completion loop & 检查短写、错误和字节范围 \\
Committing & manifest task & 等待业务提交或拒绝 \\
Reclaimable & buffer pool & 清理后回收，generation 前进 \\
\bottomrule
\end{tabular}
\end{center}

这张状态表比“使用 \code{io_uring} 会更快”重要得多。没有它，性能优化会把生命周期边界压扁；有了它，底层从 worker thread、\code{epoll}、\code{io_uring} 到直接 I/O 的替换，都只是完成来源改变，业务语义不变。

\begin{casebox}
\topic{Kafka、RocksDB 和 group commit}
Kafka、RocksDB 等系统都大量使用批处理、顺序写、后台 flush、日志或 manifest 记录提交事实。它们的设计说明：高吞吐存储系统不是每条记录都立即同步到最终结构，而是把数据面和控制面分开，用日志、manifest、sequence number、flush 和 compaction 管理可见性。

Compute Systems Engine 借鉴的是原则：大数据块可以提前流式写入，真正需要强一致提交的是小的 manifest 或 commit record；后台清理和恢复扫描必须以 commit record 为权威，不以目录扫描为权威。
\end{casebox}

\begin{casebox}
\topic{TCP 背压和应用层背压}
TCP 接收端慢会让接收窗口缩小，本端发送最终阻塞或返回 \code{EAGAIN}。这是字节流层面的背压。应用层仍要有自己的背压，因为 TCP 不知道业务优先级、deadline、幂等、内存预算和队列语义。Reduce 端忙时，应用可以返回 busy 或降低 shuffle 拉取窗口；只依赖 TCP buffer，反馈太晚且语义太粗。

第 18 章分布式 shuffle 会把这里的本机 write queue 扩展成远端 in-flight bytes、per-worker window 和 busy reply。概念相同，尺度变大。
\end{casebox}

\section{深度案例：异步写出 Shuffle Block}

把 partition manifest、任务运行时和 I/O 放到一起，得到一个可恢复写出管线。Compute task 生成 \code{OutputBlock}，I/O stage 写临时文件，completion 成功后触发 commit task，commit task 更新 manifest。Reduce 只读取 manifest 中已提交的 block。这个结构看起来比“直接写文件”复杂，但它能解释短写、失败、迟到 completion、重复 attempt、崩溃恢复和背压。

\topic{第一版：同步 reliable write reference}
先写同步 reference：创建临时文件，可靠写循环推进 offset，完成后计算 checksum，执行项目承诺的同步策略，然后返回候选 block。这个版本可能慢，但它固定短写、错误和文件状态语义。异步版本必须和它输出同样的 manifest 结果。

\topic{第二版：I/O worker 和 completion queue}
把 reliable write 放到 I/O worker，计算线程提交 \code{WriteRequest} 后继续处理 CPU work。Completion queue 返回 \code{CompletionRecord}。Completion loop 只更新状态、归还 buffer、触发后续 task，不在 completion 线程里做重 CPU 计算。若 completion loop 自己被 CPU 工作拖慢，已经完成的 I/O 也会迟迟不被业务层看到。

\topic{第三版：背压闭环}
为 write queue 和 buffer pool 设置 item、byte、高低水位。高水位时，reduce 暂停生成新 block 或降低 batch；reduce 队列满时，partition 减速；partition 减速后，map 输出缓冲减少；最终 reader 停止读取更多输入。报告要画出这条背压传播链，不能只说“队列满了会等”。

\topic{第四版：迟到 completion 和 attempt}
请求超时后，driver 可以创建新 attempt。旧 attempt 后来完成，只能写自己的临时文件并进入 late/reaped 状态。Commit task 检查当前 task attempt 和 manifest，只有获胜 attempt 能提交。这个结构和第 16 章“成功响应丢失后重试”的问题完全同构。

\section{实现流程：把 I/O 代码拆成可分析状态}

I/O 章节不需要堆很多外部实验。真正关键的是把一段看似普通的写文件代码，拆成内核路径、缓冲区所有权、完成事件和业务提交四层。第一版实现可以很小：同步可靠写作为 reference，一个有界队列，一个 writer 线程，一个 completion 处理函数，一个 commit gate。只要这四层清楚，后续把 writer 线程替换成 \code{io_uring} 或直接 I/O，都不会改变语义。

\topic{从同步写循环推导异步状态}
同步版本的核心是 offset 推进。它把系统调用返回值转成请求状态，而不是把正返回值直接当成功。

\begin{lstlisting}[language=C++,caption={可靠写循环的最小语义}]
WriteResult write_all(int fd, std::span<const std::byte> bytes) {
  std::uint64_t offset = 0;
  while (offset < bytes.size()) {
    const ssize_t n = ::write(fd, bytes.data() + offset,
                              bytes.size() - offset);
    if (n > 0) {
      offset += static_cast<std::uint64_t>(n);
      continue;
    }
    if (n < 0 && errno == EINTR) {
      continue;
    }
    return WriteResult::failed(offset, errno);
  }
  return WriteResult::completed(offset);
}
\end{lstlisting}

这段代码里已经包含异步版本需要的状态：\code{offset}、剩余字节、可重试错误、终态错误和完成字节数。异步化不是把这段循环扔给线程池，而是把调用栈里的局部变量搬到 \code{WriteRequestRecord} 中。每次 completion 回来，都相当于同步循环执行了一次系统调用，然后根据返回值决定重新提交、失败还是进入校验。

\topic{从代码走到内核路径}
\code{write} 进入内核后，普通文件常先走 page cache。用户态字节被复制或引用到内核维护的页中，页被标记为 dirty，真正设备写回可能稍后发生。若使用 direct I/O 或注册 buffer，复制路径和页缓存路径会改变，但“提交请求、等待完成、确认业务生效”这三层仍然存在。代码分析时要按顺序问四个问题：系统调用是否接受请求，接受了多少字节；这些字节现在归谁所有；completion 或同步返回证明了哪一层完成；业务是否已经允许 reduce 读取这份输出。

\topic{只保留三个验证点}
验证不需要铺成大矩阵，保留三个点即可。第一，短写：每次只写一部分字节，确认 offset 推进和最终 checksum。第二，迟到 completion：旧 attempt 完成后不能覆盖已经提交的新 attempt。第三，背压：writer 停住时，上游不能无限分配 buffer。这三个点分别覆盖字节状态、业务状态和资源状态，足够支撑本章主线。

\section{完整案例：一次 ShuffleBlock 写出怎样变成提交事实}

把 I/O 讲清楚，不能只讲 \code{write}、\code{epoll} 或 \code{io_uring} 的 API。必须沿着一块 \code{ShuffleBlock} 的生命周期走完：计算线程生成字节，buffer pool 转移所有权，I/O 层把请求提交给内核，内核可能只接受一部分字节，page cache 或设备稍后完成，completion 回到用户态，校验通过后才生成 manifest 候选，最后提交逻辑决定这个 attempt 是否生效。每一步都可能成功、失败、迟到或被取消。

\topic{T0：计算完成不等于可以释放 buffer}
Reduce 或 map-side combine 生成 block 后，buffer 里有业务字节，但所有权仍在计算阶段。提交写请求时，buffer 所有权转给 I/O 管线；从这个时刻开始，计算线程不能复用或修改 buffer，直到 completion 明确归还。若计算线程为了省内存提前复用 buffer，底层写出可能把后续内容写进当前 block，checksum 会不稳定，故障通常很难复现。

\begin{lstlisting}[numbers=none,caption={buffer 所有权转移账本}]
compute owns buffer:
  fill bytes
  compute logical checksum

submit write:
  request owns buffer
  compute must not mutate bytes

completion:
  success -> commit candidate owns metadata, pool may reclaim buffer
  failure -> retry owns buffer or retire buffer with error
  cancel  -> ownership remains pending until completion or explicit reap
\end{lstlisting}

所有权账本是异步 I/O 的第一条线。没有它，后面的短写、取消和迟到 completion 都无法解释。

\topic{T1：系统调用返回值只是提交边界的一层}
普通文件 \code{write} 返回正数时，只说明内核接受了这么多字节进入某条写路径；它可能少于请求字节数，也可能只是进入 page cache，并不等于设备已经持久化。异步接口更明显：submit 成功只表示请求进入提交队列或内核队列；completion 才表示内核对这次请求给出结果；业务 commit 仍要等待 checksum、attempt 和 manifest 判断。

\begin{lstlisting}[numbers=none,caption={写出成功的层级}]
submit accepted:
  kernel or I/O worker accepted request

bytes copied or referenced:
  some bytes entered kernel/page-cache/device path

completion success:
  request finished according to API result

durability boundary:
  fsync/fdatasync or project-defined weaker policy completed

manifest commit:
  logical task attempt is accepted as business fact
\end{lstlisting}

这五层经常被混成一个 \code{ok=true}。混淆后，系统会在响应丢失、崩溃恢复、短写和重复 attempt 中产生重复输出或坏 block。

\topic{T2：短写把循环拆成状态机}
同步 reference 可以在一个循环里处理短写：写了多少推进多少 offset，剩余继续写。异步版本不能把循环藏在调用栈里，因为每次 completion 才知道推进了多少。于是 \code{WriteRequest} 必须保存 offset、remaining、attempt id、buffer id 和错误状态。一次 partial completion 后，请求重新进入 Submitted 或 Queued；fatal error 后进入 Failed；完全写完后进入 Completed。

\begin{lstlisting}[numbers=none,caption={异步短写状态推进}]
completion(result = 4096 bytes):
  request.offset += 4096
  request.remaining -= 4096
  if request.remaining > 0:
      resubmit request
  else:
      verify checksum and move to Completed
\end{lstlisting}

如果 completion 只返回“成功”而不记录写入字节数，短写会被误提交。测试必须故意让每次只写少量字节，并断言最终 block checksum 与 reference 一致。

\topic{T3：Page cache 会把快假象放大}
写入普通文件时，内核可能先把数据放入 page cache，标记 dirty，稍后 writeback。短 benchmark 只测到用户态到 page cache 的速度，看起来非常快；长时间运行或内存压力上来后，dirty page 达到阈值，写回开始限速，\code{write}、\code{fsync} 或内核后台线程都可能成为等待点。若中间队列无界，前半段吞吐会掩盖后半段压力，最终表现为内存暴涨和长尾。

因此写出报告必须分开：用户态排队时间、提交系统调用时间、completion 时间、fsync 或 commit 时间、队列水位、dirty/writeback 现象的可观察指标。没有这些字段时，“磁盘慢”只是猜测。

\topic{T4：Manifest 才能拒绝迟到成功}
假设 attempt 0 超时，driver 重试 attempt 1。attempt 1 先完成并提交 manifest；attempt 0 后来 completion 成功。此时 attempt 0 的字节可能完整，checksum 也正确，但它已经不能成为业务事实。Commit 逻辑必须按 logical task 查询 manifest：若已有 accepted attempt，迟到 attempt 只能进入 LateAttempt，临时文件可稍后清理。

\begin{lstlisting}[numbers=none,caption={迟到 completion 的提交判定}]
completion for attempt 0:
  block checksum is valid
  logical task = split 7
  manifest already accepts attempt 1 for split 7
  decision = LateAttempt
  do not append manifest
  release or mark temp block for cleanup
\end{lstlisting}

这个例子说明 completion success 不等于 commit success。底层 I/O 只知道请求完成；业务层必须知道 logical task、attempt、epoch 和 manifest 状态。

\topic{T5：背压从 commit 失败向上游传播}
如果 manifest commit 变慢，completion queue 会堆积；completion queue 堆积后 buffer 归还变慢；buffer pool 变少后 write queue 不能接受新请求；write queue 满后 reduce 暂停生成新 block；reduce 暂停后 partition 和 map 也应减速。背压不是某个队列的局部行为，而是一条从下游提交点传到输入源头的控制链。

\begin{lstlisting}[numbers=none,caption={背压传播链}]
manifest commit slow
  -> completion loop cannot retire requests fast enough
  -> buffer pool free bytes drop below low watermark
  -> write queue rejects or blocks new ShuffleBlock
  -> reduce tasks wait on output capacity
  -> upstream partition stops producing more bytes
  -> reader eventually slows down
\end{lstlisting}

若其中某一段使用无界队列，背压链就断了。系统短时间看似吞吐更高，实际上只是把压力存成内存、临时文件或未提交 completion。

\topic{T6：关闭时选择 drain 还是 cancel}
I/O 管线关闭不能只设置一个 bool。Drain 表示不再接收新 block，但已经接收的请求继续写完、完成校验和提交；Cancel 表示尽力取消尚未提交的请求，正在内核或设备中的请求可能仍会返回 completion。两种模式都必须处理迟到 completion 和 buffer 归还。若析构直接销毁队列和 buffer pool，未完成请求回来时就可能访问已释放内存。

\begin{lstlisting}[numbers=none,caption={I/O 关闭模式}]
drain:
  reject new writes
  wait queued/submitted requests reach terminal state
  commit or fail each request
  release all buffers

cancel:
  reject new writes
  request cancellation for queued/submitted requests
  accept late completions as cleanup events
  release buffers only after ownership is known
\end{lstlisting}

关闭报告应记录 close mode、queued at close、submitted at close、completed after close、canceled、late completion 和 unreaped requests。这样 shutdown 卡住或丢数据时，能知道是等设备、等 manifest、等 buffer，还是有请求状态丢失。

\topic{T7：可靠写入的最小恢复扫描}
进程重启后，恢复程序会看到临时目录、manifest 和可能的 checkpoint。正确顺序仍然是先读 manifest，得到已提交 block；再扫描临时目录，把文件分为 accepted、candidate、orphan、corrupt。Candidate 不能直接进入结果，必须回到 attempt/commit 表验证。Corrupt 文件可以删除或隔离；orphan 文件要等审计策略允许再清理。

\begin{lstlisting}[numbers=none,caption={I/O 恢复扫描分类}]
for each temp file:
  if referenced by manifest and checksum matches:
      classify AcceptedData
  elif has valid block metadata but no accepted attempt:
      classify CandidateOrLateAttempt
  elif checksum mismatch:
      classify Corrupt
  else:
      classify Orphan
\end{lstlisting}

这条恢复路径把第 15 章和第 18 章连接起来。I/O 层负责不把坏字节伪装成完成；分布式提交层负责决定哪个 attempt 生效。两层都不能用“文件存在”替代 manifest。

\topic{T8：I/O 到 manifest 的状态表必须能重放}
前面的生命周期沿一个 \code{ShuffleBlock} 讲清了 submit、completion 和 commit 的边界，但实现时还缺一张能落到代码里的状态表。异步 I/O 的危险点恰好在这里：事件从多个方向进入系统，计算线程提交请求，writer 线程发起系统调用，completion loop 收到结果，commit 线程更新 manifest，关闭线程要求 cancel 或 drain，恢复程序又从磁盘扫描出旧文件。若每条路径各自改几个布尔字段，状态很快就无法重放。

最小实现应把 \code{WriteRequest} 写成一条可持久化或可审计的记录。它不一定每次状态变化都同步落盘，但日志、trace 和恢复表必须能重建同一条路径。请求身份至少包含 \code{request_id}、\code{logical_task}、\code{attempt}、\code{block_id} 和 \code{buffer_id}；进度包含 \code{bytes_done}、\code{remaining}、\code{checksum} 和最后一次错误；业务身份包含 stage、partition、route epoch、manifest generation 候选和 deadline。这样 completion 回来时，系统先查请求表，而不是根据文件名猜它属于谁。

\begin{lstlisting}[language=C++,caption={可重放写请求记录的示例形态}]
enum class WriteRequestState : std::uint8_t {
  Created,
  Queued,
  Submitted,
  PartiallyCompleted,
  Completed,
  Verified,
  Durable,
  Candidate,
  Committed,
  Failed,
  Canceled,
  Reaped,
};

struct WriteRequestRecord {
  std::uint64_t request_id = 0;
  std::uint64_t logical_task = 0;
  std::uint32_t attempt = 0;
  std::uint64_t block_id = 0;
  std::uint64_t buffer_id = 0;
  std::uint32_t stage_id = 0;
  std::uint32_t partition_id = 0;
  std::uint64_t route_epoch = 0;
  std::uint64_t manifest_generation = 0;
  std::uint64_t bytes_total = 0;
  std::uint64_t bytes_done = 0;
  std::uint64_t checksum = 0;
  std::uint64_t deadline_ns = 0;
  WriteRequestState state = WriteRequestState::Created;
};
\end{lstlisting}

状态转移要写成有限集合，而不是任意赋值。\code{Created} 只能进入 \code{Queued} 或 \code{Failed}；\code{Queued} 可进入 \code{Submitted}、\code{Canceled} 或 \code{Failed}；\code{Submitted} 收到短写进入 \code{PartiallyCompleted} 后重新提交，写完进入 \code{Completed}，错误进入 \code{Failed}，取消请求则进入 \code{Canceled} 但等待 reaped；\code{Completed} 通过 checksum 后进入 \code{Verified}；满足项目承诺的持久化边界后进入 \code{Durable}；只有 commit 线程检查 attempt、epoch 和 manifest 后，才从 \code{Candidate} 进入 \code{Committed}。所有未被接受的完成结果最后都要进入 \code{Reaped}，释放 buffer、容量预算和临时文件引用。

\begin{lstlisting}[numbers=none,caption={状态转移不是布尔成功}]
Created -> Queued -> Submitted
Submitted -> PartiallyCompleted -> Submitted
Submitted -> Completed -> Verified -> Durable -> Candidate
Candidate -> Committed
Candidate -> Reaped        # late attempt or stale route
Submitted -> Failed        # fatal I/O error
Submitted -> Canceled -> Reaped
\end{lstlisting}

这张表的价值在崩溃恢复时才完全显现。恢复不能从临时目录开始，因为目录只告诉系统“某些字节存在”。恢复顺序应先读 manifest，得到业务上已经接受的 block；再读 completion log、request table 或 checkpoint，恢复每个 request 的最后已知状态；最后扫描临时目录和最终目录，把文件分类为 accepted、candidate、late、orphan、corrupt。分类规则必须能解释所有中间状态：\code{Committed} 且 checksum 匹配的是 accepted；\code{Durable} 或 \code{Candidate} 但 manifest 未接受的是候选，需重新走 commit 判定；旧 attempt 完整但已有新 attempt accepted，是 late；无请求记录或元数据不完整是 orphan；checksum 不匹配是 corrupt。

\begin{lstlisting}[numbers=none,caption={恢复重放的顺序}]
recover:
  load manifest and accepted attempt table
  load request table or completion log
  scan temp and final directories
  for each discovered file:
      join by block_id, attempt, checksum
      classify accepted, candidate, late, orphan, or corrupt
  replay candidate commit only through manifest gate
  reap stale and orphan files after cleanup policy allows
\end{lstlisting}

注意最后一行：候选文件不能直接变成结果。恢复程序和在线 commit 线程必须调用同一个 manifest gate。若在线路径检查 attempt 和 route epoch，恢复路径却按“文件完整且较新”接受，系统仍会在重启后分叉。项目中可以把这个 gate 做成小函数，例如 \code{try_commit_manifest_candidate(record, ownership, manifest)}，让在线 completion、重试和恢复扫描都走同一条判断。

背压也应使用同一张状态表计算债务。粗略的队列长度只能看到 \code{Queued}；真正的债务还包括 \code{Submitted} 的 in-flight bytes、\code{Completed} 到 \code{Candidate} 之间的未提交 bytes、\code{late/orphan} 临时文件和最老未提交时间。推荐维护一张 \code{IoDebtLedger}：\code{queued_bytes} 限制生产速度，\code{inflight_bytes} 限制 buffer pool 和取消成本，\code{completed_uncommitted_bytes} 暴露 manifest 瓶颈，\code{orphan_temp_bytes} 暴露清理和恢复窗口，\code{oldest_uncommitted_age} 暴露尾部债务。

\begin{lstlisting}[language=C++,caption={I/O 债务账本的示例形态}]
struct IoDebtLedger {
  std::uint64_t queued_bytes = 0;
  std::uint64_t inflight_bytes = 0;
  std::uint64_t completed_uncommitted_bytes = 0;
  std::uint64_t orphan_temp_bytes = 0;
  std::uint64_t retry_backlog = 0;
  std::uint64_t oldest_uncommitted_age_ns = 0;
};
\end{lstlisting}

这几个字段直接决定背压打到哪里。\code{queued_bytes} 超过高水位，compute 暂停生成新 block；\code{inflight_bytes} 长期不降，writer 或设备路径需要降速、拆小 block 或增加 completion 能力；\code{completed_uncommitted_bytes} 增长，说明 manifest gate 或 group commit 是瓶颈，继续增加 writer 只会制造更多候选；\code{orphan_temp_bytes} 增长，说明恢复窗口正在扩大，清理任务要获得预算；\code{oldest_uncommitted_age} 超过 deadline，driver 应该停止制造新 attempt，先处理提交或失败传播。

至此，异步 I/O 的核心不再是某个 API，而是一张能被在线路径、关闭路径和恢复路径共同解释的账本。只要账本能重放，短写、迟到 completion、崩溃扫描和背压都能落到同一个模型；账本不能重放，系统迟早会退化成“日志里说成功、目录里也有文件、但 manifest 不知道该信谁”的状态。

\topic{I/O 指标要同时看吞吐和债务}
异步 I/O 最容易制造“吞吐看起来高，债务不断增加”的假象。报告除了 bytes/s，还要记录 queued bytes、dirty bytes 近似指标、in-flight requests、oldest request age、oldest uncommitted block age、buffer pool free bytes、retry backlog 和 orphan temp bytes。吞吐表示当前流量，债务表示未来必须偿还的工作。若吞吐高但 oldest uncommitted block age 持续增长，系统实际上已经失去稳态。

\begin{lstlisting}[numbers=none,caption={I/O 债务指标}]
write_queue.bytes_current
write_queue.bytes_high_water
inflight.requests
oldest_request_age_ms
oldest_uncommitted_block_age_ms
buffer_pool.free_bytes
retry.backlog
temp.orphan_bytes
\end{lstlisting}

这些字段能解释背压是否真的闭环。没有债务指标时，无界队列和无限重试会伪装成高吞吐。

\topic{背压策略要写业务语义}
当下游慢时，上游可以阻塞、降级、丢弃、合并、采样、失败或扩容。批处理 shuffle 通常选择阻塞和限速，因为丢弃会破坏结果；监控 telemetry 可能选择采样或丢弃；在线请求可能选择失败并返回 overloaded。背压不是单一算法，而是业务语义在资源不足时的表达。Compute Systems Engine 的默认策略是：计算结果不丢弃，任务可等待，重试有预算，错误要结构化报告。

这条默认策略会降低某些压力场景下的瞬时吞吐，但它换来结果不丢、内存有界和恢复可解释。若未来某条 telemetry 支路允许丢弃，也必须单独声明，不能把可丢弃策略混进 shuffle 输出路径。

背压还要进入用户可见报告。一次运行如果因为写出慢而主动限速，应明确写出限速次数、限速持续时间、最高水位、最低可用 buffer 和最终恢复时间。这样吞吐下降时，结论不是“程序变慢”，而是“系统按合同保护了内存和提交事实”，并能继续定位下游真正的慢点。没有这些字段，限速会被误判成随机抖动，甚至被后续优化错误地删除。

关键观测点本身就是 I/O 合同的一部分：没有被记录的压力，迟早会被当作不存在。

\section{稳态判断：吞吐、债务和恢复窗口}

异步 I/O 最容易制造一种假稳态：每秒处理的输入看起来很高，但未提交 block、dirty page、in-flight request、orphan candidate 和 retry backlog 都在增长。真正的稳态不是“当前吞吐高”，而是“在给定输入速率下，队列、水位、债务和恢复窗口不会持续增长”。如果系统只能靠不断增加内存、临时文件和未完成请求维持吞吐，它已经失去稳态。

\topic{吞吐和债务要同时画}
吞吐曲线告诉当前处理速度，债务曲线告诉未来还欠多少工作。写出系统至少有四类债务：已生成但未提交的 block，已提交给 I/O 但未 completion 的 request，已 completion 但未 manifest 的候选，已过期或迟到但未清理的临时文件。任何一类债务持续上升，最终都会变成内存、磁盘、恢复时间或尾延迟问题。

\begin{center}
\begin{tabular}{p{0.25\linewidth}p{0.34\linewidth}p{0.29\linewidth}}
\toprule
债务类型 & 含义 & 失控后果 \\
\midrule
queued bytes & 上游已生产，下游尚未接收 & 内存上涨，背压延迟变长 \\
in-flight requests & 已提交 I/O，尚未 completion & buffer pool 枯竭，取消复杂 \\
uncommitted blocks & 字节完成，业务未提交 & 恢复扫描变慢，重复 attempt 增多 \\
orphan temp bytes & 未被 manifest 引用的候选文件 & 磁盘占用，恢复审计变重 \\
retry backlog & 等待重试或退避的请求 & 重试风暴，deadline 过期 \\
\bottomrule
\end{tabular}
\end{center}

\topic{恢复窗口也是资源}
崩溃恢复要扫描 manifest、checkpoint、临时目录和候选 block。未提交候选越多，恢复窗口越长；orphan 文件越多，恢复时越难区分 late attempt、corrupt block 和清理失败。一个系统即使运行时内存稳定，如果每小时留下数百万个 orphan temp file，恢复时间也会不可接受。RunReport 应记录 recovery scan count、orphan bytes、candidate bytes 和 cleanup lag。恢复窗口和内存一样，是资源预算。

\begin{lstlisting}[numbers=none,caption={恢复窗口指标}]
recovery.manifest_entries
recovery.temp_files_scanned
recovery.accepted_blocks
recovery.candidate_blocks
recovery.orphan_blocks
recovery.corrupt_blocks
recovery.scan_elapsed_ms
cleanup.oldest_orphan_age_ms
\end{lstlisting}

\topic{稳态判读}
稳态分析不必变成复杂实验表。只要沿着状态表看债务是否回落：writer 变慢时 queued bytes 上升，buffer pool 下降，上游开始等待；writer 恢复后 queued bytes 和 completed-uncommitted bytes 应下降，oldest uncommitted age 不应无限增长。若吞吐曲线好看但债务不回落，系统只是把压力藏到了内存、临时文件或恢复窗口里。

\begin{lstlisting}[numbers=none,caption={稳态判读的最小链条}]
writer slow
  -> queued bytes rises
  -> buffer pool free bytes drops
  -> upstream waits instead of allocating more buffers

writer recovers
  -> queued bytes drains
  -> completed-uncommitted bytes drains
  -> oldest uncommitted age returns below budget
\end{lstlisting}

\topic{债务无法回落时的修复方向}
如果 queued bytes 不回落，说明下游平均服务能力低于输入速率，需要降低输入、增加 writer、减少输出字节或改变提交策略。若 in-flight request 不回落，可能是 completion loop 慢、设备慢或请求太大。若 uncommitted blocks 不回落，manifest commit 是瓶颈。若 orphan bytes 不回落，清理和恢复审计路径不足。不同债务对应不同修复，不能都归因于“磁盘慢”。

\topic{稳态报告要写保护动作}
背压真正生效时，吞吐曲线可能主动下降。报告必须把保护动作写出来：暂停读取多少次，reduce 等待 output capacity 多久，buffer pool 低水位持续多久，manifest group commit 批了多少条。否则后续优化可能把这些保护动作误删，以为自己提高了吞吐。系统的目标不是始终满速，而是在预算内持续正确地前进。

\section{复查清单}

\topic{一、是否分清 submit、complete、commit}
Submit 成功只表示请求被接受；completion 成功只表示底层 I/O 有结果；commit 才表示业务结果可见。任何把三者混成一个布尔值的设计都不合格。

\topic{二、buffer 生命周期是否有唯一所有者}
Filling、submitted、completed、free 每个状态都要有唯一所有者。失败、取消、迟到 completion 都要归还或 retire buffer，不能泄漏或双重归还。

\topic{三、背压是否传到源头}
write queue 高水位必须影响 compute、parse 或 read。若中间某段仍无限生产，背压链断了。

\topic{四、可靠写入是否有崩溃点矩阵}
只写“使用临时文件和 fsync”不够。每个崩溃位置都要有恢复动作，manifest 必须是权威入口。

\topic{五、错误是否结构化分类}
Retryable、fatal、canceled、timed out、corrupt、backpressure 不应都变成字符串日志。driver 要能根据错误类型做不同决策。

\topic{六、是否连接分布式边界}
CompletionRecord 对应 RPC reply；attempt 对应远端 task attempt；manifest 对应 checkpoint 和 shuffle 元数据；in-flight bytes 对应分布式流控窗口。若这些对象不能复用，说明边界还不稳。
