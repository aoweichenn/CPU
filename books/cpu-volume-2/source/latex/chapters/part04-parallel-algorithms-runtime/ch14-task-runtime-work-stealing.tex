\chapter{任务运行时与工作窃取}

先看一个很小、但足够刺痛工程师的事故：线程池有四个 worker，四个父任务都已经开始执行。每个父任务提交一个子任务后立刻调用 \code{future.get()}。子任务已经排进同一个线程池的队列，队列里有可运行工作；可是四个 worker 都被父任务占住，正在等待这些还没有机会运行的子任务。程序没有数据竞争，锁也没有形成循环等待，CPU 甚至可能还在忙等，但作业不再前进。

操作系统调度器看不见“父任务”和“子任务”。它只调度内核线程。线程池里的 task 只是用户态内存中的一段函数对象和状态记录，除非某个 worker 线程取出它并开始运行，否则它不会自动获得 CPU 时间。父任务在 worker 内调用 \code{future.get()} 时，可能进入条件变量或 futex 等待，也可能忙等；无论是哪一种，结果都是一个原本可以执行 ready task 的 worker 执行槽被等待占住。队列里有 ready task，不等于系统有可用执行资源。

这个事故说明：线程池只知道“有函数就找线程执行”，并不自动理解任务之间的依赖。真正的任务运行时要把依赖、就绪、等待、取消、异常和关闭写成显式状态，让 worker 只执行已经 ready 的工作，而不是把宝贵的执行资源拿去等待同一个资源池里的未来工作。工作窃取也不是“更高级的线程池”这个标签，而是在任务已经 ready 之后，解决本地性、负载均衡和长尾的调度策略。

\section{从线程池事故到运行时语义}

线程池的基本工作很简单：创建一组 worker 线程，外部提交函数对象，worker 从队列中取出并执行。这个模型适合互相独立的任务，例如并行压缩多个文件、处理多个互不依赖的请求、把输入数组切块后分别计算。它的危险在于接口太容易让人误以为“能提交函数”就等于“能表达并行计算”。

任务运行时比线程池多回答几类问题：某个任务依赖哪些前驱，什么时候 ready，结果由谁拥有，worker 能否阻塞等待，失败怎样传播，取消怎样收束，关闭时已入队和未入队任务分别如何处理。只要这些问题没有写清，后续加入 work stealing、priority、future、I/O completion 或分布式 attempt 都会变成难以复现的隐式控制流。

\topic{Thread、worker、task 和 runtime}
Thread 是操作系统调度的执行实体；worker 是运行时长期持有、用于消费任务的线程；task 是可执行工作单位；runtime 是管理 task 状态、ready 队列、worker、结果、取消和关闭协议的系统。把这些词混用，会导致错误设计。

一个 task 不应等同于一个 thread。创建一百万个 task 可以合理，创建一百万个 OS thread 通常不合理。一个 future 也不应等同于一个运行中的线程，它只是“将来会有结果或错误”的共享状态。运行时的核心价值，就是让大量 task 在有限 worker 上推进，并且让等待不必占住 worker。

这一区分还解释了为什么“多开几个线程”不是根本修复。线程变多会增加内核调度对象、栈内存、上下文切换和 cache/TLB 扰动；若任务依赖仍然藏在 worker 栈上的同步等待里，线程数只是把饥饿推迟到更大规模。运行时要做的是把“谁等谁”从调用栈里拿出来，变成 task node、ready count、continuation 和队列状态。这样 OS 仍然只调度 worker 线程，但用户态运行时能决定哪个 task 应该占用这些线程。

\topic{最小线程池也要有状态机}
很多入门线程池只有一个 \code{closed} 布尔值，这会把“拒绝新任务”“排空旧任务”“取消未开始任务”“worker 已经退出”混在一起。更清楚的模型至少有三个状态：

\begin{lstlisting}[language=C++,caption={运行时生命周期的最小枚举}]
enum class RuntimeState : std::int32_t {
  Open,
  Draining,
  Stopped,
};

enum class SubmitResult : std::int32_t {
  Accepted,
  RejectedDraining,
  RejectedStopped,
  QueueFull,
};
\end{lstlisting}

\code{Open} 允许提交新任务。\code{Draining} 拒绝新任务，但继续执行已经接纳的任务。\code{Stopped} 表示 worker 已退出，所有等待者都必须被唤醒。若支持立即取消，还要把 \code{shutdown drain} 和 \code{shutdown cancel} 分成两条路径。状态机不是形式主义，它直接决定条件变量谓词、析构安全、future 等待者能看到什么结果。

\topic{队列谓词和唤醒不是装饰}
线程池最小循环通常是：worker 拿锁，检查队列是否非空，空则等待条件变量，被唤醒后再检查。这里的“再检查”不是保守写法，而是正确性要求。条件变量允许虚假唤醒；多个 worker 被唤醒时，先拿到锁的 worker 可能已经取走任务；shutdown 也可能在等待期间发生。可靠等待谓词至少要表达“队列非空或运行时状态不再允许继续等待”。若谓词只写成“被通知了”，就可能出现空队列取任务、shutdown 后睡死或遗漏取消。

futex 的思想也类似：用户态先检查内存中的条件，不满足时才让内核把线程挂到等待队列；唤醒方先改变内存状态，再通知等待者。条件变量、futex 和无锁队列的细节不同，但原则一样：状态是事实，通知只是加速观察事实。运行时设计应先定义状态转移，再定义谁负责通知，而不是把 \code{notify_one()} 当成语义本身。

\topic{Future 的本质是共享结果状态}
Future 的重要部分不是 \code{get()} 这个阻塞调用，而是 shared state。任务函数运行在 worker 栈上，提交者可能在另一个线程等待结果；值、异常、取消状态、完成通知和 continuation 必须放在一个拥有明确生命周期的对象里。若异常只打印到 worker 日志，调用者无法知道任务失败；若取消只设置标志但不唤醒等待者，系统会悬挂；若结果保存在已销毁的 lambda 捕获里，就会出现悬垂引用。

\begin{lstlisting}[language=C++,caption={Future 共享状态需要表达的语义}]
enum class CompletionState : std::int32_t {
  Pending,
  Succeeded,
  Failed,
  Canceled,
};

struct CompletionRecord {
  std::uint64_t task_id = 0;
  CompletionState state = CompletionState::Pending;
  std::uint32_t error_code = 0;
  std::uint64_t result_bytes = 0;
};
\end{lstlisting}

这段代码不是完整 future 实现，而是在固定语义。一个可靠运行时必须能区分成功、失败和取消；也必须能让等待者在所有终态下醒来。生产实现还要保存异常对象、结果存储、等待队列、引用计数和内存同步关系。

\topic{运行时 FutureState：结果、异常、等待者和 continuation}
线程池如果提供 \code{submit -> future}，就必须有一份比 \code{CompletionRecord} 更具体的结果状态。它不一定要复刻标准库实现，但要表达同一组事实：生产端是否还存在，结果是否 ready，结果或异常存在哪里，谁在等待，哪些 continuation 要在 ready 后调度，以及这份状态何时可以释放。

\begin{lstlisting}[language=C++,caption={运行时 FutureState 的语义字段}]
enum class FutureStateKind : std::int32_t {
  Pending,
  ValueReady,
  ExceptionReady,
  Canceled,
  BrokenProducer,
  Consumed,
};

struct FutureStateHeader {
  std::uint64_t task_id = 0;
  FutureStateKind state = FutureStateKind::Pending;
  std::uint32_t waiter_count = 0;
  std::uint32_t continuation_count = 0;
  bool producer_alive = true;
  bool consumer_alive = true;
};
\end{lstlisting}

真实的结果存储可以是类型擦除、模板化 storage、\code{std::variant} 或标准库 \code{std::promise} 包装；本章先关心状态边界。任务被接纳后，FutureState 必须活到 producer 进入终态，并且活到 consumer 取走结果或放弃句柄。若任务完成后立即释放任务对象，而 future 还没 \code{get()}，调用者会读到悬垂状态。若 consumer 放弃 future，任务仍可能继续运行；运行时可以选择继续计算、协作取消，或在背压策略里丢弃后继，但必须写成合同。

\topic{FutureState 的完整状态机}
最小运行时不能只保存一个 \code{ready} 布尔值。它至少要区分“还没有结果”“有值”“有异常”“被取消”“生产者消失”“普通 future 已消费”。这些状态不是为了日志好看，而是为了让 \code{get}、\code{wait\_idle}、取消传播和 coroutine 恢复都能做出确定动作。

\begin{lstlisting}[numbers=none,caption={FutureState 状态转移表}]
Pending:
  complete_value      -> ValueReady
  complete_exception  -> ExceptionReady
  cancel_before_run   -> Canceled
  producer_destroyed  -> BrokenProducer
  consumer_destroyed  -> Pending or producer policy decides cancel

ValueReady:
  ordinary_get        -> Consumed
  shared_get          -> ValueReady
  destroy_last_handle -> reclaim storage

ExceptionReady:
  ordinary_get        -> Consumed after rethrow path is prepared
  shared_get          -> ExceptionReady
  destroy_last_handle -> reclaim storage

Canceled:
  get/await_resume    -> throw task_canceled or return canceled result
  destroy_last_handle -> reclaim storage

BrokenProducer:
  get/await_resume    -> throw broken_task or future_error
  destroy_last_handle -> reclaim storage

Consumed:
  second ordinary_get -> diagnostic or future_error
  destroy_last_handle -> reclaim storage
\end{lstlisting}

这张表必须和接口一致。如果运行时返回普通 single-consumer future，\code{ValueReady -> Consumed} 是合法消费；如果返回 shared future-like handle，ready 状态不能被第一个 reader 移走。如果 consumer 提前丢弃句柄，运行时也不能默认杀掉任务，除非 API 写明“future destructor requests cancellation”。标准库 \code{std::future} 没有取消语义；项目运行时可以有，但必须显式。

\topic{FutureState 对象边界：热字段、冷字段和等待者}
FutureState 同时被 worker、外部 caller、continuation 和 coroutine completion 访问。为了让后续优化有边界，先把字段按用途拆开：

\begin{lstlisting}[language=C++,caption={FutureState 的语义布局}]
struct FutureStateHot {
  std::atomic<FutureStateKind> state{FutureStateKind::Pending};
  std::uint64_t task_id = 0;
  std::uint32_t flags = 0;
};

struct FutureStateCold {
  std::uint64_t create_ns = 0;
  std::uint64_t ready_ns = 0;
  std::uint64_t first_wait_ns = 0;
  std::uint32_t waiter_count = 0;
  std::uint32_t continuation_count = 0;
  FutureErrorKind error_kind = FutureErrorKind::None;
};

struct FutureStateSync {
  std::mutex mutex;
  std::condition_variable cv;
  ContinuationList continuations;
};
\end{lstlisting}

热字段服务快速检查和 trace 关联；冷字段服务诊断；同步字段服务等待和 continuation 注册。真实实现可以合并或重排，但不能丢掉语义。特别是 continuation list 不能只是一组裸回调：每个 continuation 还要知道恢复到哪个 executor、是否携带取消令牌、是否只能恢复一次。

\topic{完成路径的原子提交}
FutureState completion 必须是单点提交。不要让 worker loop 写一部分，future 写一部分，task graph 又写一部分。下面的账本展示了一个可靠完成函数要做的事：

\begin{lstlisting}[numbers=none,caption={complete\_future 的提交边界}]
complete_future(task_id, terminal_result):
  lock future_state
  if state is not Pending:
      record duplicate completion bug
      unlock
      return
  store value, exception, canceled marker, or broken marker
  state = terminal state
  ready_ns = now
  waiters = waiter_count
  continuations = move continuation list
  unlock

  notify waiters
  enqueue continuations to their executors

  complete_task_state(task_id, terminal_result)
  decrement unfinished_tasks exactly once
  notify wait_idle if unfinished_tasks == 0
\end{lstlisting}

这里故意把“发布 FutureState 终态”和“运行 continuation”分开。终态要在锁内一次提交；唤醒和入队要在锁外做。若在锁内调用 continuation，用户代码可能再次调用 runtime、提交任务、等待另一个 future 或拿用户锁，最终把完成路径变成不可控递归。若 FutureState ready 以后没有同步更新 TaskState，调用者能拿到结果，但任务图还以为节点未完成；若只更新 TaskState，不发布 FutureState，外部 \code{get} 会卡住。

\topic{等待路径：外部线程、worker 和 coroutine 是三种不同入口}
同一个 FutureState 可以被三种入口观察，但它们的合同不同。

\begin{lstlisting}[numbers=none,caption={三种等待入口的伪代码}]
external_get:
  lock FutureState
  while state is Pending:
      register external waiter
      cv.wait(lock)
  consume or rethrow
  unlock

worker_get_same_runtime:
  if runtime policy forbids blocking:
      return diagnostic: BlockingWaitInWorker
  if policy allows helping:
      leave user locks and runtime locks
      execute allowed ready work while target is Pending
  else if policy allows blocking pool:
      mark worker blocked and compensate

future_await:
  lock FutureState
  if state is terminal:
      unlock
      enqueue coroutine handle or continue fast path
  else:
      append continuation with resume executor
      unlock
      suspend without occupying worker
\end{lstlisting}

把这三条入口混成一个 \code{wait()} 函数，是很多运行时事故的根源。外部线程阻塞只是调用方延迟；worker 阻塞会减少执行槽；coroutine 等待应该登记 continuation 而不是 park 线程。接口层应能区分调用者是否位于 runtime worker 内。不能区分时，宁可默认禁止 worker 同池阻塞等待。

\topic{FutureState 和 TaskState 的一致性矩阵}
FutureState 面向结果消费者；TaskState 面向调度器。二者不完全相同，但终态必须一致。

\begin{center}
\small
\begin{tabular}{p{0.20\linewidth}p{0.24\linewidth}p{0.34\linewidth}}
\toprule
FutureState & TaskState & 含义 \\
\midrule
ValueReady & Succeeded & 用户函数成功，后继可按成功边推进 \\
ExceptionReady & Failed & 用户函数或 runtime wrapper 捕获异常，后继按失败策略处理 \\
Canceled & Canceled & 任务已被接纳但未执行或协作停止，等待者必须醒来 \\
BrokenProducer & Failed & producer 端丢失，说明运行时合同破坏或 wrapper 未完成 \\
Consumed & Succeeded/Failed/Canceled & 结果已被普通 future 取走，不改变调度终态 \\
Pending & Waiting/Ready/Running & 任务还未进入可观察结果终态 \\
\bottomrule
\end{tabular}
\end{center}

不允许出现的组合也要写进断言。例如 TaskState 已经 Succeeded 而 FutureState 仍 Pending，说明完成函数漏写结果；TaskState 已经 Canceled 而 FutureState 仍 Pending，说明取消没有唤醒等待者；FutureState ExceptionReady 但 TaskState 仍 Running，说明 worker 异常路径没有收束。运行时 trace 应把这些组合当成 invariant violation，而不是普通错误日志。

\topic{submit 到 future ready 的完整链路}
\code{submit} 返回 future 时，接纳、入队、执行、完成和读取必须连成一条线。下面的账本比“把 callable 放进队列”更严格：

\begin{lstlisting}[numbers=none,caption={submit -> FutureState -> worker -> get 的账本}]
submit(callable):
  lock runtime
  if state != Open:
      return RejectedDraining or RejectedStopped
  allocate FutureState and TaskNode
  producer endpoint is owned by the task wrapper
  consumer endpoint is returned to caller
  unfinished_tasks += 1
  enqueue ReadyItem
  unlock runtime
  notify worker

worker runs task wrapper:
  try:
      value = callable()
      complete_future_with_value(future_state, value)
  catch:
      complete_future_with_exception(future_state, current_exception)
  complete_task and decrement unfinished_tasks exactly once

future.get from caller:
  wait until FutureState is terminal
  ValueReady -> move result and mark Consumed
  ExceptionReady -> rethrow in caller thread
  Canceled/BrokenProducer -> report a defined error
\end{lstlisting}

这条链有三个容易错的提交点。第一，\code{submit} 只有在任务和 FutureState 都被接纳后才能返回成功；否则调用者会拿到一个无人满足的 future。第二，worker 必须捕获所有逃逸异常并写入 FutureState；否则异常停在 worker 线程，调用者只会永远等待或得到静默失败。第三，未完成计数和 FutureState 终态要一起提交；不能 future 已 ready 但 \code{wait\_idle} 还永远等，也不能 \code{wait\_idle} 返回但 future 仍 Pending。

\topic{提交失败也必须是可观察语义}
线程池 \code{submit} 最危险的返回值不是 \code{Rejected}，而是“返回了一个永远 Pending 的 future”。拒绝提交有两种合理形态：要么返回 \code{SubmitResult::Rejected} 且不创建消费端；要么返回一个已经 ready 的 future，里面保存明确的 rejection 异常。两者都可以审查；不能审查的是“分配了 FutureState，入队失败，然后忘了满足它”。

\begin{lstlisting}[numbers=none,caption={submit 失败的两种合法合同}]
contract A:
  submit returns expected<Future<T>, SubmitError>
  if QueueFull or Draining:
      return unexpected(error)
  no FutureState is exposed to caller

contract B:
  submit returns Future<T>
  if QueueFull or Draining:
      FutureState = ExceptionReady(submit_rejected)
      return future already ready

invalid:
  create FutureState
  fail to enqueue task
  return Future<T> whose state remains Pending forever
\end{lstlisting}

合同 A 更适合系统库，因为调用者必须处理拒绝。合同 B 更像异步 API，把拒绝也当成异步结果。无论选择哪种，trace 都应记录 reject reason、queue depth、runtime state 和是否创建 FutureState。没有这些字段，线上看到等待超时，很难区分任务还在排队、任务被拒绝、worker 死掉，还是 future 合同本身破坏。

\topic{阻塞等待、helping 等待和非阻塞 continuation}
等待 FutureState 有三种工程形态。外部线程可以阻塞等待，因为它不是运行时的执行资源；compute worker 阻塞等待同一 runtime 的 future 会吃掉 worker；coroutine 或 continuation 则把等待者保存为后续任务，不占线程。它们不是性能微调，而是三种不同合同。

\begin{center}
\small
\begin{tabular}{p{0.22\linewidth}p{0.34\linewidth}p{0.32\linewidth}}
\toprule
等待形态 & 语义 & 主要风险 \\
\midrule
外部 \code{get()} & 调用线程睡眠，直到 future ready & 调用方延迟增加，但不耗 compute worker \\
worker \code{get()} & worker 睡眠或忙等同池结果 & ready 任务无人执行，形成饥饿 \\
worker helping & 等待时执行允许范围内 ready 任务 & 重入、锁顺序、栈增长、范围选择复杂 \\
continuation & future ready 后把后继放入队列 & 控制流拆分，异常和取消要沿边传播 \\
coroutine \code{co\_await} & suspend frame，ready 后调度恢复 & coroutine frame 生命周期和恢复 executor 要清楚 \\
\bottomrule
\end{tabular}
\end{center}

一个严格的最小运行时可以先禁止 worker 内部阻塞 \code{get()}，只允许外部 \code{get()} 和 continuation。这样错误更早暴露：若用户在 worker 里等待同池 future，接口直接失败或诊断，而不是偶发卡死。等任务图、trace 和执行域稳定后，再决定是否购买 helping 或 compensation worker 的复杂度。

\topic{异常传播不能绕开任务图}
FutureState 保存异常只是第一步。任务图还要决定后继如何处理异常。若节点 A 失败，依赖 A 的 B 不能永远保持 Waiting；它必须变成 Canceled、Failed、RetryPending，或由上层策略改接到补偿节点。FutureState 面向调用者传播一次结果；TaskState 面向运行时传播调度事实。两者要一致。

\begin{lstlisting}[numbers=none,caption={异常从 callable 传播到 future 和后继}]
callable throws:
  FutureState: Pending -> ExceptionReady(exception_ptr)
  TaskState: Running -> Failed
  successors:
    if policy == fail_fast:
        Waiting/Ready successors -> Canceled
    if policy == retry:
        create retry attempt or RetryPending
  waiters:
    external future.get rethrows
    wait_idle wakes when all affected nodes are terminal
\end{lstlisting}

若只设置 FutureState 异常，不更新 TaskState，任务图可能还在等后继 ready count；若只把 TaskState 标为 Failed，不写 FutureState，调用者会拿不到异常。运行时完成路径必须同时处理两边，而且只能处理一次。

\topic{worker 内等待为什么危险}
外部线程等待 future 通常只是让调用者停住；worker 等待同一个运行时里的 future 则可能消耗执行资源。若被等待的任务还需要 worker 执行，等待者就挡住了被等待者。这不是“future 有问题”，而是“等待发生在错误的位置”。

可选修复有三类。第一，禁止 worker 内部同步等待同池任务，把后续逻辑注册成 continuation。第二，允许 work helping：worker 等待某个 task group 时，主动执行同一组的 ready 任务。第三，把可能阻塞的任务隔离到专用执行域，例如 I/O blocking pool，不占用 compute worker。三者都可以正确，但语义和复杂度不同，必须写进接口。

从底层看，等待有两种常见形态。阻塞等待会让线程睡到内核对象上，Linux 上常见路径最终会落到 futex 或类似机制；线程不烧 CPU，但 worker 数量减少了。忙等会不断读某个状态位，可能还用 \code{pause} 降低流水线压力；线程仍占用 CPU，状态位所在 cache line 还会被反复读取。二者都不适合长期发生在 compute worker 内。compute worker 的职责是推进 ready work，而不是保存等待上下文。

\topic{Continuation 的直觉}
Continuation 把“等 B 完成后继续做 A 的后半段”变成一个新的 task。B 完成时，运行时更新 shared state，递减后继的依赖计数，计数归零后把 continuation 放入 ready 队列。这样等待不占线程，worker 可以去执行其他 ready task。

Continuation 也有代价。控制流不如同步代码直观，异常和取消需要沿依赖边传播，深层 continuation 可能造成调试困难。可靠运行时不是盲目禁止同步，而是明确区分：外部同步等待可以简化调用者；worker 内同步等待必须被运行时理解，或者被接口禁止。

\topic{Coroutine：把 future 等待编译成 continuation}
C++ coroutine 可以把“等待结果后继续执行”的代码写成顺序形态，但运行时看到的仍然是 continuation。Coroutine frame 保存局部变量、暂停点和返回对象；\code{co\_await} 某个 future-like 对象时，awaiter 决定是否立即继续、是否挂起、挂起后把 coroutine handle 注册到哪里、恢复时如何返回值或重抛异常。

\begin{lstlisting}[numbers=none,caption={future-like awaiter 的运行时语义}]
await_ready:
  if FutureState is terminal:
      return true
  return false

await_suspend(coroutine_handle h):
  lock FutureState
  if state became terminal:
      enqueue h to current executor
  else:
      append h to continuations
  unlock
  return to worker without blocking the worker thread

future completion:
  store value, exception, or cancellation
  detach continuation handles
  enqueue handles onto the selected runtime queue

await_resume:
  if ValueReady: return value
  if ExceptionReady: rethrow exception
  if Canceled: throw or return canceled result according to API
\end{lstlisting}

这段语义有两个硬边界。第一，\code{await\_suspend} 不能在持有 FutureState 锁时直接恢复 coroutine，因为恢复会执行用户代码，可能重新进入运行时或拿用户锁。第二，恢复必须有执行域：恢复到 compute runtime、I/O completion loop、调用线程，还是专用 executor。若没有恢复 executor，coroutine 只是把阻塞等待换成了不可控回调。

\begin{lstlisting}[numbers=none,caption={coroutine 和线程池集成的错误形态}]
bad completion:
  lock future_state
  mark ready
  for each coroutine handle:
      handle.resume()     # runs user code under future_state lock
  unlock

robust completion:
  lock future_state
  mark ready
  move continuation handles to local list
  unlock
  enqueue handles as ReadyItem into the chosen executor
\end{lstlisting}

Coroutine 购买的是可读的非阻塞控制流：业务代码看起来像 \code{Load -> Parse -> Compute}，运行时实际看到的是多个 ready continuation。它不自动解决背压、取消、异常传播、执行域和生命周期。若 \code{co\_await} 的对象没有把取消和异常写进 FutureState，coroutine 只会把错误藏得更深。

\topic{CoroutineTask：运行时看到的协程任务}
为了把协程放进任务运行时，需要把 coroutine frame 看成一种可恢复任务。它至少有四类状态：frame 是否已创建，是否在 executor 队列里，是否正在某个 worker 上恢复，是否已经完成并等待销毁。

\begin{lstlisting}[numbers=none,caption={CoroutineTask 状态机}]
Created:
  initial_suspend returns suspend_always -> Suspended
  initial_suspend returns suspend_never  -> Runnable

Runnable:
  enqueue to executor -> Queued

Queued:
  worker dequeues handle -> Running

Running:
  co_await pending future -> Suspended(waiting on FutureState)
  returns value          -> FinalSuspended(ValueReady)
  throws                 -> FinalSuspended(ExceptionReady)

Suspended:
  awaited FutureState completes -> Runnable
  cancellation requested        -> CancelRequested

FinalSuspended:
  schedule continuation
  owner observes result
  destroy frame -> Destroyed
\end{lstlisting}

这张状态机比 \code{co\_await} 语法更重要。若 Running 状态下直接销毁 frame，就是正在执行的栈被释放。若 Suspended 状态没有记录它挂在哪个 FutureState 上，取消和诊断都找不到等待边。若 FinalSuspended 直接恢复 continuation 而不是入 executor，completion 线程可能执行大量用户代码。若 Destroyed 之后 FutureState completion 仍握着 handle，下一次 resume 就是 use-after-free。

\topic{协程恢复执行域}
协程等待完成后由谁恢复，是运行时合同的一部分。恢复到 producer 线程，延迟最低但容易把 I/O completion 或锁内路径变成用户代码执行路径；恢复到原 compute executor，语义稳定但需要一次入队；恢复到 caller 线程通常不可行，因为 caller 可能早已返回；恢复到 dedicated continuation executor 可以隔离长回调，但增加调度成本。

\begin{center}
\small
\begin{tabular}{p{0.24\linewidth}p{0.34\linewidth}p{0.28\linewidth}}
\toprule
恢复位置 & 购买能力 & 主要风险 \\
\midrule
producer completion 线程 & 少一次入队，延迟低 & 锁内或 I/O 线程执行用户代码 \\
原 compute executor & 执行域清楚，trace 连续 & 多一次调度，可能排队 \\
blocking/I/O executor & 隔离慢 continuation & 需要跨域 backpressure \\
inline fast path & ready 时避免 suspend & 双 resume 和递归深度要审查 \\
\bottomrule
\end{tabular}
\end{center}

本书建议最小运行时只允许两条路径：ready fast path 直接继续；pending path 注册 continuation，并在 completion 后把 handle 入原 executor。这样 trace 能把每次恢复都记成 ReadyItem，锁顺序也更容易审查。后续若要 inline resume，必须先证明不会在 FutureState 锁内、runtime 锁内或用户不可重入区域内执行。

\topic{协程和取消：取消请求不是终态}
取消协程有三层事实：有人请求取消，协程观察到取消，FutureState 进入可观察终态。只设置 stop flag 不会唤醒等待者；只丢弃 handle 会泄漏或悬挂；只把 FutureState 标 Canceled 但 frame 还会被 executor 恢复，也会形成重复终态。

\begin{lstlisting}[numbers=none,caption={coroutine cancellation 的审查账本}]
request_cancel(coroutine_task):
  mark cancel_requested
  if task is queued:
      remove or let worker observe canceled before running
      complete FutureState as Canceled
  if task is suspended on FutureState X:
      unregister continuation from X if possible
      complete outer FutureState as Canceled
  if task is running:
      set cooperative stop token
      running code must reach a cancellation point

await_resume after cancellation:
  throw task_canceled or return canceled result
  never silently continue with default value
\end{lstlisting}

这里没有免费的强制杀线程。C++ coroutine 取消通常是协作式的：运行中的用户代码必须在某些点观察 stop token 或被等待对象返回取消结果。运行时能做的是不再调度尚未运行的 continuation，唤醒等待者，并确保所有状态进入终态。

\topic{Awaiter 合同：它是调度器边界，不是语法装饰}
一个 awaiter 同时面对三方：当前 coroutine frame、被等待的状态对象、恢复它的 executor。只写 \code{await\_ready/await\_suspend/await\_resume} 三个函数名还不够，运行时必须写清它们分别持有什么锁、是否允许 inline resume、completion 发生在哪个线程、恢复后异常如何传播。下面是 Compute Systems Engine 可以采用的最小 awaiter 合同。

\begin{lstlisting}[numbers=none,caption={运行时 awaiter 的最低合同}]
RuntimeAwaiter:
  observed_state:
    FutureState, IoRequestState, TimerState, or TaskDependency
  await_ready:
    read terminal state with acquire semantics
    return true only if await_resume can complete immediately
  await_suspend:
    register coroutine_handle exactly once
    record resume_executor
    never block the current worker
    never call user code under observed_state lock
  completion path:
    publish terminal state with release semantics
    detach continuation handles
    enqueue handles to resume_executor
  await_resume:
    read terminal result
    return value, return cancellation result, or rethrow exception
\end{lstlisting}

这个合同把 awaiter 和普通回调区分开。回调常常由 completion 线程直接调用，容易在不可控线程和锁内执行用户代码；awaiter 应该把恢复动作交给 executor。这样运行时 trace 能看到“哪个请求完成，哪个 coroutine 被入队，哪个 worker 恢复了它”。没有这条 trace，\code{co\_await} 后面的延迟很难归因：可能是 I/O 慢、completion loop 慢、executor 队列堵，也可能是 coroutine 自己执行慢。

\topic{Awaitable 的四种来源}
运行时会遇到多种 awaitable，但它们都应被降低成“状态对象 + continuation 注册 + executor 恢复”。

\begin{center}
\small
\begin{tabular}{p{0.22\linewidth}p{0.30\linewidth}p{0.34\linewidth}}
\toprule
awaitable 来源 & 状态对象 & completion 由谁产生 \\
\midrule
任务 future & FutureState & worker 完成 task wrapper \\
任务依赖 & TaskState/ready count & 最后一个前驱完成线程 \\
timer/sleep & TimerState & timer wheel 或 event loop \\
异步 I/O & IoRequestState 或 CompletionRecord & I/O completion loop 或 blocking I/O domain \\
远端 RPC & RpcCallState & network event loop 或 RPC runtime \\
\bottomrule
\end{tabular}
\end{center}

表里每一行都能用同一个调度动作结束：把 coroutine handle 包成 ReadyItem，进入某个 executor 队列。不同的只是状态对象和 completion 来源。这个统一模型非常重要，因为它让 future、timer、I/O、RPC 都走同一套取消、trace、背压和 shutdown 规则。

\topic{Scheduler 看到的是 ReadyItem，不是 C++ 语法}
协程恢复时，scheduler 不需要知道源代码写的是 \code{co\_await read()} 还是 \code{co\_await sleep()}。它看到的应是一条 ReadyItem：恢复哪个 coroutine、属于哪个 request/task、恢复到哪个执行域、是否带取消或异常摘要。

\begin{lstlisting}[language=C++,caption={协程恢复 ReadyItem 的语义字段}]
enum class ReadyItemKind : std::int32_t {
  TaskFunction,
  CoroutineResume,
  IoCompletionContinuation,
  TimerContinuation,
};

struct CoroutineResumeItem {
  std::uint64_t coroutine_id = 0;
  std::uint64_t source_state_id = 0;
  ReadyItemKind kind = ReadyItemKind::CoroutineResume;
  ExecutorId resume_executor;
  FutureErrorKind error_kind = FutureErrorKind::None;
};
\end{lstlisting}

这段结构不是为了限制实现，而是为了固定观测字段。若恢复项没有 \code{source\_state\_id}，trace 只能看到 coroutine 突然运行，无法知道它等的是哪个 I/O 或 future。若没有 \code{resume\_executor}，跨执行域恢复就会变成隐式回调。若没有 \code{error\_kind}，异常路径只能靠用户日志追。

\topic{调度器和操作系统调度器的边界}
项目里的 scheduler 调度的是用户态工作项；Linux scheduler 调度的是内核线程。Coroutine suspend 只让用户态工作项离开 ready 队列，不会让操作系统知道“这个业务请求在等 I/O”。只有当某个 worker 队列为空、线程进入 CV/futex sleep，或者执行阻塞系统调用时，内核才看到线程睡眠。

\begin{lstlisting}[numbers=none,caption={两层调度器的边界}]
user-level scheduler:
  owns ready task queue
  owns coroutine ready handles
  decides which worker executes which ReadyItem
  records task/coroutine/request trace

OS scheduler:
  owns native threads
  runs or parks worker threads
  observes futex sleep, blocking syscall, runnable delay
  does not know task dependency graph
\end{lstlisting}

这解释了为什么“CPU 线程数很多”不等于“任务运行时有执行资源”。所有 worker 都在 futex 上睡，说明 ready 队列可能为空；所有 worker 都在阻塞 I/O，说明 runtime 把设备等待放错执行域；所有 worker 都在 \code{future.get} 等同池任务，说明用户态调度器还有 ready work，但执行槽被隐藏等待占住。报告必须同时记录 OS 线程状态和用户态 worker/task 状态。

\topic{Awaiter 的失败模式清单}
协程相关 bug 经常不是语法错误，而是运行时合同破坏。最小失败模式清单如下：

\begin{center}
\small
\begin{tabular}{p{0.27\linewidth}p{0.34\linewidth}p{0.25\linewidth}}
\toprule
失败模式 & 具体错误 & 证据 \\
\midrule
双恢复 & ready fast path 和 completion path 都 resume & coroutine id resume count 大于 1 \\
丢恢复 & completion 发生在注册窗口，handle 未入队 & FutureState terminal 但 coroutine Suspended \\
锁内恢复 & completion 持锁直接执行用户代码 & lock hold ns 包含用户函数 \\
错执行域恢复 & I/O loop 直接跑 CPU continuation & worker trace 显示 event loop 被占住 \\

这些失败模式应该进入测试矩阵：并发注册和 completion、ready fast path、late completion、取消、shutdown、跨 executor 恢复、锁内恢复检测。只测试 \code{co\_await} 成功返回值，没有覆盖协程运行时真正危险的边界。

\section{任务图：把等待变成状态}

第 13 章的 reduce、scan、partition 已经把算法拆成阶段。count 必须先完成，scan 才能计算 offsets；scan 完成后，write 才能写入独占区间；所有 write 完成后，manifest 才能提交。这种结构天然是任务图。若运行时只提供 \code{submit}，程序很容易写成 barrier 或 worker 内等待；若运行时支持任务图，依赖就能成为调度事实。

\topic{ready count}
任务图的节点表示可执行工作，边表示依赖。每个节点记录还有多少前驱未完成。初始计数为零的节点进入 ready 队列；某个节点完成后，遍历后继并递减它们的计数；减到零的那个线程负责把后继入队。这个计数通常叫 ready count 或 remaining predecessors。

ready count 的本质是把“等待所有前驱”变成一个原子状态变化。假设节点 X 有三个前驱，计数初值为 3。每个前驱完成时执行一次递减，只有观察到递减前值为 1 的那个完成线程，才拥有把 X 发布到 ready 队列的权利。这个点就是 X 从 Waiting 到 Ready 的线性化点。若两个线程都能把 X 入队，就会重复执行；若没有线程负责入队，X 会永远等待。任务图运行时的正确性，往往就藏在这种看似小的原子边界里。

\begin{lstlisting}[language=C++,caption={任务图节点的核心调度字段}]
enum class TaskState : std::int32_t {
  Waiting,
  Ready,
  Running,
  Succeeded,
  Failed,
  Canceled,
};

struct TaskNode {
  std::uint64_t task_id = 0;
  std::uint32_t first_successor = 0;
  std::uint32_t successor_count = 0;
  std::atomic<std::uint32_t> remaining_predecessors{0};
  std::atomic<TaskState> state{TaskState::Waiting};
};
\end{lstlisting}

这里直接把状态写成枚举，是为了让示例代码先表达清楚生命周期。大型运行时可能把状态压缩成整数编码，或者把热调度字段和冷诊断字段分开存放；但无论编码方式怎样，状态都必须是程序可判断的事实，而不是日志字符串。

\topic{任务状态不是日志字符串}
状态要能被程序判断，而不是只写在日志里。一个任务从 \code{Waiting} 进入 \code{Ready}，必须有唯一线性化点；从 \code{Running} 进入 \code{Succeeded} 或 \code{Failed}，必须释放结果并唤醒后继；从 \code{Ready} 进入 \code{Canceled}，worker 取出时必须能跳过它并完成计数。若状态只是“打印了一行 ready”，运行时无法防止重复入队、重复完成和取消后仍执行。

\begin{center}
\begin{tabular}{p{0.18\linewidth}p{0.32\linewidth}p{0.38\linewidth}}
\toprule
状态 & 进入条件 & 允许的后续状态 \\
\midrule
Waiting & 依赖尚未满足 & Ready、Canceled、Failed \\
Ready & ready count 归零，任务已入队 & Running、Canceled \\
Running & worker 成功取得任务并开始执行 & Succeeded、Failed、Canceled \\
Succeeded & 用户函数正常完成，结果可见 & 终态 \\
Failed & 用户函数或运行时错误不可忽略 & 终态，触发后继策略 \\
Canceled & 上游失败、用户取消或 shutdown cancel & 终态 \\
\bottomrule
\end{tabular}
\end{center}

\topic{完成路径是运行时最关键的代码}
任务完成时，运行时要保存结果或异常、转换任务状态、更新 trace、递减后继 ready count、把变 ready 的任务入队、通知等待者，并在失败时触发取消传播。这些动作应集中在一个完成函数中，不能让 worker loop、future、取消逻辑各自修改状态。状态转换分散，是任务运行时最常见的错误来源。

完成路径还要避免在锁内执行未知用户代码。若一个 continuation 是用户函数，完成线程不应在持有 shared state 锁时直接调用它；应先收集需要唤醒或入队的任务，释放锁，再发布到 ready 队列。这个原则会在 I/O completion 和分布式回调中反复出现。

完成路径还承担内存发布。用户函数写入结果对象后，运行时把状态改成 \code{Succeeded}；等待者或后继看到 \code{Succeeded} 后必须能看到结果内容。mutex 版本可以依赖锁的 release/acquire；无锁版本需要显式 release 存储和 acquire 读取。ready 队列也是同一个问题：生产者必须先完整构造 task node，再发布队列指针或索引；worker 取出 task 后才能读到正确字段。若发布顺序错了，bug 可能表现为偶发空函数、旧参数、丢失 successor 或错误 trace。

\topic{结果所有权和任务所有权要分开}
任务对象完成了，不表示结果对象可以释放。future 可能还没被调用者读取，trace 可能还要汇总，后继任务可能只需要错误摘要而不是完整任务对象。高性能运行时通常把热调度节点、用户结果、诊断记录和队列节点分开管理。示例实现可以简单一些，但必须说明为什么“任务执行完就 delete”在 future 和 continuation 存在时非常危险。

\topic{关键路径决定任务图上限}
任务图性能由总工作量和关键路径共同决定。总工作量是所有任务耗时之和；关键路径是依赖图中必须串行经过的最长路径。线程数再多，也不能突破关键路径。若实际时间远高于关键路径，说明调度、同步、I/O 或负载倾斜有问题；若实际时间接近关键路径，继续增加 worker 收益有限，应优化依赖结构或关键任务本身。

实验必须输出 trace，而不是只输出总耗时。Trace 至少要能估计每个任务的 ready 等待时间、运行时间和前驱关系。没有这些数据，无法判断慢在关键路径、队列等待、窃取失败还是任务粒度。

\section{从 worker loop 到任务图调度：完整执行路径}

运行时的最小内核可以拆成六条路径：提交任务、发布 ready 任务、worker 取任务、任务完成、等待 idle、关闭运行时。每条路径都要有线性化点。线性化点是一次并发操作在语义上“生效”的瞬间：\code{submit} 何时算被接纳，任务何时算 ready，完成何时算对后继可见，关闭何时开始拒绝新任务。没有这些边界，偶发重复执行、丢任务、等待不醒和关闭卡住都很难定位。

\topic{运行时表：把热调度字段和冷诊断字段分开}
一个小型实现可以把任务存入 task table。调度热路径只需要状态、ready count、函数入口和 successor 范围；trace、错误摘要、输入范围和调试字符串可以放在冷表。这样 worker loop 不会为了每个任务搬运大量诊断字段。

\begin{lstlisting}[language=C++,caption={任务运行时表的最小形态}]
struct ReadyItem {
  std::uint64_t task_id = 0;
  std::uint32_t attempt = 0;
};

struct TaskHotFields {
  std::atomic<TaskState> state{TaskState::Waiting};
  std::atomic<std::uint32_t> remaining_predecessors{0};
  std::uint32_t first_successor = 0;
  std::uint32_t successor_count = 0;
  TaskFunction function;
};

struct TaskColdFields {
  std::uint64_t submit_ns = 0;
  std::uint64_t ready_ns = 0;
  std::uint64_t start_ns = 0;
  std::uint64_t finish_ns = 0;
  TaskErrorSummary error;
};

struct RuntimeCore {
  std::mutex mutex;
  std::condition_variable cv;
  RuntimeState state = RuntimeState::Open;
  std::deque<ReadyItem> global_ready;
  std::uint64_t unfinished_tasks = 0;
  std::vector<TaskHotFields> hot;
  std::vector<TaskColdFields> cold;
};
\end{lstlisting}

这个版本故意使用 mutex 和全局 ready 队列。它不是最终高性能结构，却适合作为语义 reference。全局队列让关闭、异常、未完成计数和 trace 更容易验证；等这些语义稳定后，再把队列换成本地 deque 和 stealing。若一开始就写无锁 deque，调度策略错误和内存同步错误会混在一起。

\topic{submit：接纳和拒绝必须有返回值}
\code{submit} 的线性化点应在持锁状态下完成：检查 runtime 状态、分配 task id、写入 task 表、增加 unfinished 计数、如果无依赖则入 ready 队列。只有这些动作完成后，\code{submit} 才能返回 \code{Accepted}。若 runtime 已经进入 \code{Draining} 或 \code{Stopped}，它必须返回明确拒绝，而不是静默丢弃函数对象。

\begin{lstlisting}[language=C++,caption={submit 的语义伪代码}]
SubmitResult submit(RuntimeCore& rt, TaskSpec spec) {
  std::unique_lock lock(rt.mutex);
  if (rt.state == RuntimeState::Draining) {
    return SubmitResult::RejectedDraining;
  }
  if (rt.state == RuntimeState::Stopped) {
    return SubmitResult::RejectedStopped;
  }

  const std::uint64_t task_id = allocate_task_id(rt);
  rt.hot[task_id] = build_hot_fields(spec);
  rt.cold[task_id].submit_ns = now_ns();
  ++rt.unfinished_tasks;

  if (spec.predecessor_count == 0) {
    mark_ready_locked(rt, task_id);
  }

  lock.unlock();
  rt.cv.notify_one();
  return SubmitResult::Accepted;
}
\end{lstlisting}

这里有两个容易忽略的点。第一，\code{unfinished_tasks} 只在接纳成功后增加；被拒绝的任务不能计入，否则 \code{wait_idle} 会永远等不到归零。第二，ready 入队必须发生在通知之前；通知只是让 worker 更快观察队列非空，不能替代队列状态本身。

\topic{mark ready：只能发布一次}
任务从 \code{Waiting} 到 \code{Ready} 必须只能发生一次。对于无依赖任务，\code{submit} 负责发布；对于有依赖任务，最后一个前驱的完成线程负责发布。发布前要写好任务字段和 trace 时间，发布后 worker 才可能取出执行。

\begin{lstlisting}[language=C++,caption={发布 ready 任务的伪代码}]
void mark_ready_locked(RuntimeCore& rt, std::uint64_t task_id) {
  TaskState expected = TaskState::Waiting;
  const bool changed = rt.hot[task_id].state.compare_exchange_strong(
      expected,
      TaskState::Ready,
      std::memory_order_release,
      std::memory_order_relaxed);
  if (!changed) {
    return;
  }

  rt.cold[task_id].ready_ns = now_ns();
  rt.global_ready.push_back(ReadyItem{task_id, 0});
}
\end{lstlisting}

mutex 版本通常不需要显式写这么细的 memory order，因为锁已经建立 happens-before。伪代码仍然写出 release，是为了强调发布语义：生产者必须先构造完整任务，再让 worker 通过队列看到它。后续换成无锁 ready queue 时，这个边界不能丢。

\topic{worker loop：只长期执行 ready work}
worker loop 的职责很窄：等待 ready item，取出任务，尝试把状态从 \code{Ready} 改成 \code{Running}，执行用户函数，把结果交给 \code{complete_task}。它不应在持有 runtime 锁时执行用户函数，也不应在用户函数内部同步等待同一 runtime 的 future。

\begin{lstlisting}[language=C++,caption={全局队列版 worker loop 伪代码}]
void worker_loop(RuntimeCore& rt, std::uint32_t worker_id) {
  for (;;) {
    ReadyItem item;
    {
      std::unique_lock lock(rt.mutex);
      rt.cv.wait(lock, [&] {
        return !rt.global_ready.empty() ||
               rt.state == RuntimeState::Stopped ||
               (rt.state == RuntimeState::Draining &&
                rt.unfinished_tasks == 0);
      });

      if (rt.global_ready.empty()) {
        if (rt.state != RuntimeState::Open) {
          return;
        }
        continue;
      }

      item = rt.global_ready.front();
      rt.global_ready.pop_front();
    }

    TaskHotFields& task = rt.hot[item.task_id];
    TaskState expected = TaskState::Ready;
    if (!task.state.compare_exchange_strong(expected, TaskState::Running)) {
      continue;
    }

    rt.cold[item.task_id].start_ns = now_ns();
    TaskErrorSummary error = run_user_function(task.function);
    complete_task(rt, item.task_id, error);
  }
}
\end{lstlisting}

这个循环展示了三个原则。第一，等待谓词检查队列和生命周期状态，而不是检查“是否收到通知”。第二，取任务后释放锁，再执行用户函数。第三，从 \code{Ready} 到 \code{Running} 用状态转换保护，即使取消或关闭把 ready 任务改成终态，worker 取到旧 item 也能跳过。

\topic{complete task：运行时的关键提交点}
任务完成时，要把用户函数结果发布到 shared state，记录 trace，转换终态，递减后继依赖，减少 unfinished 计数，并唤醒等待者。这个路径应集中在一个函数里，避免 worker loop、future、取消传播各自修改计数。

\begin{lstlisting}[language=C++,caption={complete task 的语义伪代码}]
void complete_task(RuntimeCore& rt,
                   std::uint64_t task_id,
                   TaskErrorSummary error) {
  std::vector<std::uint64_t> ready_successors;
  bool notify_all = false;

  {
    std::unique_lock lock(rt.mutex);
    TaskState terminal =
        error.kind == TaskErrorKind::None ? TaskState::Succeeded
                                          : TaskState::Failed;

    rt.cold[task_id].error = error;
    rt.cold[task_id].finish_ns = now_ns();
    rt.hot[task_id].state.store(terminal, std::memory_order_release);

    for_each_successor(rt, task_id, [&](std::uint64_t succ) {
      if (error.kind != TaskErrorKind::None) {
        cancel_successor_locked(rt, succ, error);
        return;
      }
      if (try_make_ready_locked(rt, succ)) {
        ready_successors.push_back(succ);
      }
    });

    --rt.unfinished_tasks;
    if (rt.unfinished_tasks == 0) {
      notify_all = true;
    }
  }

  if (!ready_successors.empty()) {
    rt.cv.notify_all();
  } else if (notify_all) {
    rt.cv.notify_all();
  }
}
\end{lstlisting}

这里 \code{ready_successors} 只是说明完成路径会产生新 ready 任务；实际实现可以在锁内入队后统一通知。失败路径不能遗漏 unfinished 计数，也不能让后继永远停在 \code{Waiting}。若策略是“前驱失败则后继取消”，取消本身也要进入终态并减少 unfinished 计数；若策略是“前驱失败后重试”，则 driver 要创建新的 attempt，并让旧 attempt 的后继等待新的结果。

\topic{try make ready：ready count 的线性化点}
后继有多个前驱时，只有最后一个完成的前驱能把它入队。这个线性化点通常是 \code{fetch_sub} 观察到旧值为 1。旧值不是 1 的完成线程，只是把计数减一，没有发布权。

\begin{lstlisting}[language=C++,caption={后继进入 ready 的原子边界}]
bool try_make_ready_locked(RuntimeCore& rt, std::uint64_t task_id) {
  const std::uint32_t old =
      rt.hot[task_id].remaining_predecessors.fetch_sub(
          1, std::memory_order_acq_rel);
  if (old != 1) {
    return false;
  }
  mark_ready_locked(rt, task_id);
  return true;
}
\end{lstlisting}

若 \code{remaining_predecessors} 初值错了，运行时会出现两类相反事故：初值过大，后继永远不 ready；初值过小，后继提前执行，可能读到未产生的 counts、offsets 或 manifest。任务图构建阶段必须检查边数、重复边和自环；调度阶段不能靠运气弥补图构建错误。

\topic{wait idle：不能只看队列为空}
\code{wait_idle} 的含义是所有已接纳任务都进入终态。它不能只看 ready 队列为空，因为 worker 可能已经取出任务正在运行；也不能只看 worker 空闲，因为某个提交者可能已经接纳任务但尚未入队。unfinished 计数把这些状态统一起来：接纳成功加一，进入终态减一。

\begin{lstlisting}[language=C++,caption={wait idle 的语义伪代码}]
void wait_idle(RuntimeCore& rt) {
  std::unique_lock lock(rt.mutex);
  rt.cv.wait(lock, [&] {
    return rt.unfinished_tasks == 0 ||
           rt.state == RuntimeState::Stopped;
  });
}
\end{lstlisting}

异常、取消和关闭路径都要审查 unfinished 是否平衡。最常见的 bug 是用户函数抛异常后 worker 提前退出，没有减少计数；或者 shutdown cancel 标记 ready 任务为 \code{Canceled}，却没有为每个被取消任务减少计数。这样的运行时在成功路径上看似正常，一到失败路径就卡死。

\topic{shutdown：drain 和 cancel 是两种协议}
\code{shutdown drain} 停止接收新任务，继续执行已接纳任务，直到 unfinished 归零后 worker 退出。\code{shutdown cancel} 停止接收新任务，把未开始任务标记为 \code{Canceled}，运行中任务通过 cancel token 协作结束。二者返回给调用方的语义不同，不能混成一个含糊的 \code{shutdown()}。

\begin{lstlisting}[language=C++,caption={shutdown drain 的语义伪代码}]
void shutdown_drain(RuntimeCore& rt) {
  {
    std::unique_lock lock(rt.mutex);
    if (rt.state == RuntimeState::Stopped) {
      return;
    }
    rt.state = RuntimeState::Draining;
  }
  rt.cv.notify_all();

  wait_idle(rt);

  {
    std::unique_lock lock(rt.mutex);
    rt.state = RuntimeState::Stopped;
  }
  rt.cv.notify_all();
  join_all_workers();
}
\end{lstlisting}

与 \code{submit} 并发时，\code{shutdown_drain} 的状态转换是拒绝新任务的线性化点。若 \code{submit} 先在锁内观察到 \code{Open} 并完成接纳，它就必须被 drain 执行完；若 drain 先把状态改成 \code{Draining}，后续 \code{submit} 必须返回拒绝。不能出现 \code{submit} 返回成功但任务既不执行也不取消。

\topic{三条事故账本}
把完整路径写清后，经典事故会变成可解释的状态账本。

\begin{lstlisting}[numbers=none,caption={future 饥饿的事件序列}]
t0: worker 0..3 start parent tasks
t1: each parent submits one child task
t2: child tasks are Ready in the same runtime
t3: each parent calls future.get inside worker
t4: all workers block or spin waiting for children
t5: ready queue has children, but no worker can execute them
fix: parent continuation becomes a task; worker never waits on same pool
\end{lstlisting}

\begin{lstlisting}[numbers=none,caption={关闭竞态的事件序列}]
t0: submit A observes Open under runtime mutex
t1: submit A increments unfinished and enqueues A
t2: shutdown_drain changes state to Draining
t3: submit B observes Draining and returns RejectedDraining
t4: workers finish A, unfinished becomes zero
t5: runtime becomes Stopped and workers join
\end{lstlisting}

\begin{lstlisting}[numbers=none,caption={前驱失败传播的事件序列}]
t0: count 2 starts Running
t1: count 2 detects corrupted input and returns UserException
t2: complete_task marks count 2 Failed
t3: scan node is marked Canceled, not left Waiting
t4: write nodes and commit are canceled or skipped by policy
t5: wait_idle observes all affected nodes reached terminal states
\end{lstlisting}

这三条账本分别约束 worker 内等待、关闭线性化点和失败传播。运行时实现完成后，测试应能复现这些序列，并在 trace 中看到相同状态转移。若 trace 无法表达这些事件，说明观测字段还不够。

\section{运行时证据链：ready、running、blocked 和 helping}

任务运行时的错误常藏在“线程还活着，但系统不推进”里。全局队列不为空，却没有 worker 执行；worker 数量足够，却全部卡在 future、条件变量或 I/O；任务图理论上有并行度，trace 里却出现长时间空洞。要定位这些问题，运行时不能只记录 task start 和 finish，还要记录 worker 状态。Worker 是执行资源，任务是工作单位，二者的关系随时间变化。只有把关系写进 trace，才能判断慢在队列、运行、阻塞、窃取还是关闭。

\begin{lstlisting}[numbers=none,caption={worker trace 状态}]
Idle
DequeueLocal
DequeueGlobal
StealAttempt
Running
PublishingCompletion
WaitingExternal
Helping
Parked
Waking
Shutdown
\end{lstlisting}

\code{Idle} 表示 worker 暂时没有工作；\code{StealAttempt} 表示它正在寻找别的 deque；\code{Running} 表示它正在执行用户任务；\code{PublishingCompletion} 表示用户函数已经结束，运行时正在发布结果、递减后继 ready count 和唤醒等待者；\code{WaitingExternal} 表示 worker 进入了运行时无法直接推进的阻塞，例如磁盘 I/O、网络等待或外部 mutex；\code{Helping} 表示 worker 在等待某个结果时没有睡死，而是执行运行时中其他 ready work；\code{Parked} 表示没有可执行任务时进入睡眠。

\topic{隐藏阻塞：线程池最容易被自己耗尽}
第 13 章的 stable filter 会产生 count、scan、write、commit 任务。如果 count 任务在 worker 内部等待“所有 count 完成后再继续 scan”，就可能让 worker 全部占在父任务栈上，ready 队列里的后继无人执行。这个事故不需要死锁循环，只要 worker 数等于阻塞父任务数就够了。更隐蔽的版本是任务调用 \code{future.get()}、等待条件变量、同步读文件、请求网络服务、或在锁内执行会阻塞的析构。运行时看到 worker 还没退出，但执行资源已经从 ready 队列中消失。

\begin{lstlisting}[numbers=none,caption={worker 被隐藏阻塞耗尽的 trace}]
t0 worker 0 Running parent_count_group
t1 worker 1 Running parent_count_group
t2 worker 2 Running parent_count_group
t3 worker 3 Running parent_count_group
t4 each parent submits child scan or waits for sibling count
t5 workers enter WaitingExternal or future wait
t6 ready queue has runnable children
t7 no worker returns to DequeueLocal or DequeueGlobal
\end{lstlisting}

这类问题不能靠增加 \code{notify_all} 修复，因为等待的不是条件变量漏通知，而是执行资源被错误占用。修复必须改变等待协议：把后续工作写成 continuation、允许 worker helping、创建补偿 worker、把阻塞 I/O 放进单独的 blocking pool，或干脆禁止 worker 等待同一运行时的 future。

\topic{四种 managed blocking 策略}
Managed blocking 的核心是：运行时知道某个 worker 即将长期不可用，于是采取动作保持执行资源。策略不同，语义和成本也不同。

\begin{center}
\small
\begin{tabular}{@{}p{0.22\linewidth}p{0.34\linewidth}p{0.34\linewidth}@{}}
\toprule
策略 & 适用场景 & 风险 \\
\midrule
禁止 worker wait & 可把依赖全部改成 continuation & API 约束强，调用方需要改结构 \\
continuation & 任务完成后发布后半段 & 状态拆分更细，错误传播要清楚 \\
work helping & 等待者执行其他 ready task & 可能重入运行时，栈和锁顺序要审查 \\
compensation worker & worker 阻塞时临时补一个线程 & 可能过度创建线程，CPU 过度订阅 \\
blocking pool & 把 I/O 或长等待放到专门线程池 & 跨池调度复杂，背压必须明确 \\
\bottomrule
\end{tabular}
\end{center}

禁止 worker wait 是最干净的语义：任务函数不能调用同一运行时 future 的阻塞等待，只能返回、发布 continuation 或把后继节点交给任务图。Continuation 把“等待后继续执行”改成“前驱完成后新任务 ready”，正好对应第 13 章的 stage 状态机。Helping 允许 worker 在等待目标 future 时执行其他 ready task，它能避免线程池耗尽，但必须禁止在持有用户锁或运行时内部锁时进入 helping，否则会把锁顺序变成难以分析的递归图。Compensation worker 像急救措施：当 worker 声明自己将阻塞，运行时临时增加执行线程；它能保护吞吐，但会增加调度成本和内存栈成本。Blocking pool 则承认 I/O 等待和 CPU 任务不是一种资源，用单独线程承接阻塞。

\topic{helping 的边界}
Helping 不是“随便跑点别的任务”。等待者必须声明自己等待的目标、允许执行的任务范围、是否允许执行依赖当前栈帧的后继、以及如何避免重入锁。一个简化协议可以这样写：

\begin{lstlisting}[numbers=none,caption={work helping 的语义边界}]
before helping:
  worker must not hold runtime mutex
  worker must not hold user locks declared non-reentrant
  waited future belongs to the same runtime
  target task is not already terminal

helping loop:
  if target becomes terminal, return its result
  run one ready task that is safe for this worker
  publish completion and trace
  periodically check cancellation and shutdown

exit:
  do not consume the waited result twice
  restore worker state from Helping to Running or Idle
\end{lstlisting}

若 helping 可以执行任意任务，可能出现重入用户代码、递归提交、栈增长和锁顺序反转。若 helping 范围太窄，又可能找不到可执行任务，退化成忙等。实践中常给任务标注执行域：CPU task、blocking task、scheduler internal task、continuation task。等待 CPU future 的 worker 可以帮助 CPU task，但不能在持有某个用户对象锁时执行会回调同一对象的任务。

\topic{park 和 wake 也要有合同}
没有 ready work 时，worker 不能无限自旋；它应该尝试窃取、检查全局队列和关闭状态，然后 park。Park 的本质和条件变量相同：线程睡眠等待谓词变化。谓词通常是“本地队列非空、全局队列非空、某个 victim 可偷、运行时进入关闭状态”。发布 ready 任务时必须先把任务放进队列，再 wake 或 notify。若先 wake 后入队，worker 可能醒来看到空队列并再次睡眠，ready 任务就会滞留到下一次唤醒。

\begin{lstlisting}[numbers=none,caption={ready 发布和 worker park 的顺序}]
publish ready:
  write task fields
  push ReadyItem into a visible queue
  update counters used by park predicate
  wake one or more parked workers

worker park:
  check local queue
  check global queue
  attempt bounded stealing
  if no work and runtime open:
      sleep on condition variable, semaphore, or platform park primitive
  after wake:
      recheck queues and shutdown state
\end{lstlisting}

这正是第 10 章条件变量原则在运行时中的放大版本：通知不是任务，任务在队列和状态表里；醒来不是拥有任务，worker 仍要重新检查。Park/wake 的 bug 往往表现为低概率 hang，原因是丢失唤醒窗口、ready 计数和队列状态不一致，或者关闭状态没有进入等待谓词。

\topic{运行时报告字段}
运行时报告至少要能把“没有任务可跑”和“有任务但没人跑”区分开。前者可能是关键路径限制，后者是调度事故。

\begin{lstlisting}[numbers=none,caption={任务运行时的 trace 和汇总字段}]
runtime.ready_wait_ns.p50
runtime.ready_wait_ns.p99
runtime.run_ns.p50
runtime.run_ns.p99
runtime.blocked_worker_count
runtime.helped_tasks
runtime.steal_attempts
runtime.steal_success
runtime.critical_path_ns
runtime.unfinished_tasks
runtime.ready_queue_depth.max
runtime.park_wake_count
runtime.compensation_workers_created
runtime.external_wait_ns
\end{lstlisting}

\code{ready_wait_ns} 是任务进入 ready 到开始 running 的时间。它高，说明 ready work 等不到 worker，可能是队列竞争、worker 阻塞、窃取失败或线程数不足。\code{run_ns} 是用户函数执行时间，能暴露任务粒度和长尾。\code{blocked_worker_count} 和 \code{external_wait_ns} 能说明 CPU worker 是否被 I/O 或 future 等待吞掉。\code{critical_path_ns} 给出任务图理论下限；若总耗时远大于关键路径，要看调度和阻塞；若总耗时接近关键路径，要改依赖结构而不是继续加线程。

\topic{把第 13 章的 stage 接入运行时}
Stable filter 接入运行时时，不应该让一个父任务顺序调用 count、等待 count、再调用 scan。更好的形状是：每个 count chunk 是独立任务；count 完成时递减 scan 的 \code{remaining_predecessors}；最后一个 count 发布 scan；scan 生成 layout 后发布 write tasks；所有 write verified 后发布 commit。整个过程没有 worker 需要阻塞等待同一 stage。

\begin{lstlisting}[numbers=none,caption={stable filter 的非阻塞任务图账本}]
count[i]:
  reads input chunk i
  writes CountDone block record
  decrements scan.remaining_predecessors

scan:
  runs when every count[i] is terminal success
  builds layout and assigns output ranges
  publishes write[i] tasks

write[i]:
  rewrites its entire output range for its attempt
  verifies records and checksum
  decrements commit.remaining_predecessors

commit:
  chooses unique verified attempts
  publishes manifest
  marks stage Committed or Failed
\end{lstlisting}

这条链把算法合同、恢复合同和运行时合同对齐了。Count 失败时，scan 不会被错误发布；write 失败时，commit 不会读取未验证输出；commit 完成时，下游任务只依赖 manifest，而不依赖 worker 的局部变量。运行时需要做的不是理解 filter 公式，而是执行 ready count、状态发布和失败传播。

\topic{阻塞 I/O 为什么要离开 CPU worker}
第 15 章会进入 I/O。这里先建立边界：CPU worker 适合执行短而可抢占的计算任务，不适合在同步文件读、网络请求或外部服务调用上长期停住。若任务必须读文件，常见做法是提交异步 I/O，I/O 完成事件再发布 continuation；或者把同步 I/O 放到 blocking pool，完成后把 CPU 后继任务放回 CPU runtime。把同步 I/O 直接放在 work stealing worker 里，会让 steal 成功率、ready wait 和 park/wake 数据全部失真，因为 worker 不是真的没工作，而是被外部等待扣住了。

\section{Work Stealing：本地性和负载均衡的折中}

全局队列是最好的第一版 reference。所有任务进入同一队列，worker 从同一队列取任务。它容易证明关闭语义，也容易测试异常和未完成计数。但全局队列有两个明显问题：所有 worker 竞争同一把锁或同一组原子位置；本地生成的子任务可能被任意 worker 拿走，破坏缓存局部性。

Work stealing 的思路是每个 worker 拥有本地 deque。本地 worker 高频 push/pop 自己的一端；空闲 worker 低频从其他 worker 的另一端 steal。这样把常见路径留在本地，把少见路径用于负载均衡。这个不对称所有权，是工作窃取的核心。

\topic{owner LIFO，thief FIFO}
Owner 从本地端按 LIFO 执行新任务，常能保持递归分治或局部数据的 cache 热度。Thief 从另一端偷较老任务，倾向拿到较大的工作块，减少偷取频率。这个策略并不是绝对真理，但它解释了很多经典运行时的设计：常见路径低开销、本地性好；不均衡时再通过窃取把远处工作拉过来。

cache 热度来自时间和地址两个维度。一个 worker 刚处理完父任务，父任务访问的输入块、局部 partial、栈上的小对象和刚生成的子任务元数据，可能还在这个核心的 L1/L2 或共享 LLC 中。owner LIFO 让它立刻执行最新子任务，增加复用概率。thief FIFO 偷较老任务，是为了减少和 owner 争最新 cache 热数据，也更可能拿到粒度较大的上层任务。若任务几乎不共享数据，或者每个任务都只做 I/O 等待，这个策略的优势就会下降。

\topic{Chase-Lev deque 的直觉边界}
Chase-Lev deque 是经典工作窃取队列。它利用单 owner 的事实简化本地 push/pop，让多个 thief 通过另一个端点竞争 steal。最难的地方通常不是“数组怎么放任务”，而是边界状态：deque 只剩最后一个元素时，owner pop 和 thief steal 可能竞争同一个任务；扩容和节点回收还要保证被偷走的任务不会被 owner 同时释放。

Compute Systems Engine 不应一开始就实现无锁 Chase-Lev。更稳的路线是先实现全局队列 reference，再用 mutex 保护每个 worker deque 验证调度策略，最后在 trace 能证明 deque 同步成为瓶颈时再研究无锁队列。这样能把“调度策略是否有效”和“无锁协议是否正确”分开。

\topic{deque 竞争点从 head 和 tail 推出来}
工作窃取 deque 的本地路径之所以能快，是因为 owner 是唯一频繁修改 tail 的线程。owner push 把任务写入数组槽位，再发布新的 tail；owner pop 从 tail 端取最近任务。Thief 只从另一端偷，竞争 head。简化状态可以写成：

\begin{lstlisting}[numbers=none,caption={工作窃取 deque 的关键状态}]
array slots:
  task storage, usually a power-of-two circular buffer

head:
  old end, thieves compete here

tail:
  new end, owner mostly owns this cursor
\end{lstlisting}

当 deque 中有多个任务时，owner pop 和 thief steal 操作不同端点，冲突概率低。真正危险的是只剩一个任务时：owner 从 tail 端 pop，thief 从 head 端 steal，二者可能指向同一个槽位。此时必须用一个线性化点决定任务归谁。常见做法是在最后一个元素竞争时对 head 做 CAS：谁成功推进 head，谁获得任务；失败者必须把状态修正并返回空。

\begin{lstlisting}[numbers=none,caption={最后一个任务竞争的语义伪代码}]
owner pop:
  tail = tail - 1
  if head < tail:
      return slot[tail]
  if head == tail:
      if compare_exchange(head, head, head + 1):
          return slot[tail]
      else:
          restore empty state and return empty

thief steal:
  read head and tail
  if head >= tail:
      return empty
  task = slot[head]
  if compare_exchange(head, head, head + 1):
      return task
  return retry or empty
\end{lstlisting}

这段伪代码省略了很多细节，却保留了关键事实：无锁 deque 的正确性不是来自“数组加两个下标”，而是来自最后一个任务的 CAS 竞争、发布顺序、槽位生命周期和扩容回收协议。C++ 层面的 \code{compare_exchange} 在 x86-64 上常落成 \code{lock cmpxchg} 形态；它会让保存 head 的 cache line 成为 thief 竞争点。正常情况下 steal 频率远低于 owner 本地 pop，所以这条共享线还能接受；若任务太小导致大量偷取，head 的 CAS 争用会迅速暴露。

\topic{发布顺序决定 thief 能否看见任务内容}
owner push 不能先发布 tail，再写 slot。否则 thief 可能看到 tail 已经前进，偷到尚未构造完成的任务。正确顺序是先写任务内容，再用 release 语义发布 tail；thief 看到可偷区间后，用 acquire 语义读取并确认任务内容。使用 mutex 的第一版由锁隐含建立 happens-before；无锁版本必须把这条关系写出来。

\begin{lstlisting}[numbers=none,caption={push 发布顺序的抽象}]
slot[index] = fully constructed task descriptor
publish tail with release ordering

thief observes tail/head with acquire ordering
read slot[index] after successful claim
\end{lstlisting}

任务描述符本身也要有生命周期。若 owner 扩容或回收旧数组，而 thief 正在读取旧 slot，就会出现悬空访问。成熟实现会用受控的数组替换、epoch/hazard 机制、延迟释放或运行时全局 quiescent point。项目第一版使用 mutex deque，并不是逃避底层，而是先避免把调度语义、内存回收和原子协议混在一起。

\topic{无锁 deque 的性能报告要看偷取率}
如果本地 pop 占绝大多数，deque 的热路径主要是 owner 私有 tail 和本地数组，性能很好；如果任务粒度太小、负载分布很不均或 worker 频繁空转，steal 尝试会增加，head CAS、随机远端访问和失败重试会进入成本。报告不能只写吞吐，还要写 local pops、steal attempts、steal successes、failed CAS、empty steals、平均任务耗时和 worker idle 时间。没有这些字段，就无法判断瓶颈是任务太小、负载不均、deque 协议争用，还是任务本身慢。

\topic{窃取成功不一定是好消息}
Steal success 高通常说明负载不均被修复，但它也可能说明任务粒度太小、本地队列长期放不住工作、父任务不断制造跨 worker 迁移，或者 NUMA 本地性被牺牲。一个任务被偷走后，它携带的不只是函数指针，还携带输入 split、partial buffer、字典页、错误摘要和后续提交身份。若这些数据仍在 owner 所在 CPU 或 NUMA 节点附近，thief 执行会把第 3、4 章的地址路径拉远。

\begin{center}
\small
\begin{tabular}{@{}p{0.22\linewidth}p{0.35\linewidth}p{0.32\linewidth}@{}}
\toprule
观测形状 & 可能解释 & 下一步实验 \\
\midrule
idle 高，steal success 低 & 任务分布不可偷、victim 选择差或 park 太早 & 增加任务切分、改变 victim 策略、检查 ready 发布 \\
idle 低，steal success 高，p99 升高 & 窃取减少空闲但破坏本地性或粒度过小 & 记录 stolen task 数据节点，增大粒度或 NUMA 分组 \\
steal attempts 高，CAS 失败高 & head 热点竞争，任务太细或 thief 太多 & 限制偷取频率、批量偷取、调整 park/backoff \\
local pops 高，makespan 仍高 & 关键路径或长任务主导 & 拆长任务、改 DAG 依赖，不继续调 deque \\
global inject 队列长期高 & 外部提交超过本地吸收能力 & 分片 inject queue、背压或直接投递到目标 worker \\
\bottomrule
\end{tabular}
\end{center}

因此，work stealing 报告要同时写调度和地址事实：任务从哪个 worker 生成，被谁执行，输入 split 在哪个 NUMA 节点，partial buffer 谁写，执行前 ready 等了多久，执行后是否触发远端合并。只有调度事实没有地址事实，会把远端访存误判成“任务本身慢”；只有地址事实没有调度事实，又会看不见 worker 空闲和窃取失败。

\topic{数组扩容和悬空引用边界}
固定容量 deque 可以把任务槽位的生命周期绑定到 runtime；可扩容 deque 则要处理数组替换。Owner 可能发现本地 deque 满了，分配新数组，复制旧槽位，发布新数组指针；同时某个 thief 可能已经读取旧数组指针并准备 steal。若旧数组立即释放，thief 会读悬空内存。这个问题和第 12 章的节点回收同构，只是节点变成了任务数组。

第一版运行时可以主动避开这个复杂度：每个 worker 使用固定容量 deque，满时把任务推入全局 inject 队列，或者向上游返回背压。这样做牺牲一部分峰值性能，却让测试集中在任务图、ready count、失败传播和关闭。等 trace 显示固定容量溢出频繁且成为瓶颈，再评估扩容、分段数组、延迟释放或成熟 deque 实现。

\begin{lstlisting}[numbers=none,caption={第一版 work stealing 的容量合同}]
local deque:
  fixed capacity per worker
  owner push may fail when local deque is full

overflow policy:
  push to global inject queue
  or reject submit with backpressure
  or split less aggressively in the parent task

reclamation:
  task slot storage lives until runtime shutdown
  no old deque array is freed while workers are running
\end{lstlisting}

\topic{批量偷取和公平性}
一次偷一个任务的开销在任务很小的时候会很高。批量偷取可以把多个旧任务搬到 thief 本地 deque，减少 head CAS 和 victim 选择成本，也能让 thief 后续执行本地 LIFO。代价是 owner 可能被偷走太多剩余工作，局部性和公平性受影响；若被偷的是同一 NUMA 节点的大量数据，跨节点流量会放大。批量大小应由任务粒度、steal 成功率、idle 时间和 NUMA 证据决定，不应写死成“越大越好”。

\begin{lstlisting}[numbers=none,caption={批量偷取的判读字段}]
steal_batch_size
stolen_tasks_per_batch
stolen_task_run_ns_median
victim_remaining_depth_after_steal
cross_numa_steal_count
owner_idle_after_being_stolen
thief_local_pops_after_batch
\end{lstlisting}

若批量偷取后 thief 很快消化本地任务、owner 没有长期空闲、cross NUMA steal 不高，说明批量有价值；若 owner 被偷空而 thief 又处理远端数据变慢，批量策略应收窄或按 NUMA 分组。调度策略需要证据驱动，不能靠单个吞吐数字决定。

\topic{任务粒度决定调度是否值得}
任务太小，提交、入队、出队、唤醒、trace 和 ready count 成本会吃掉计算；任务太大，最后少数长任务拖住整个 stage，work stealing 也救不了已经不可拆的长尾。一个实用经验是：任务数量要明显多于 worker 数，每个任务又要有足够计算量，使调度固定成本只占小比例。

任务粒度还要和 cache、NUMA 和取消检查点一起选择。按 cache 友好的块切分，有利于顺序访问；按过小粒度切分，有利于负载均衡但增加调度成本；长任务如果没有检查点，取消响应会很慢。项目报告应扫描 block size，而不是凭感觉写一个常数。

粒度选择可以从一个账本估算。若一次 task 调度固定成本约为几百纳秒到数微秒，而任务体只做几十条指令，调度成本会主导；若任务体扫描数 MB 数据，调度成本可以忽略，但长尾和 NUMA 位置会变成主要问题。合理实验应同时输出 task count、平均运行时间、p95/p99 运行时间、ready 等待时间、steal 次数和 worker idle 比例。只看平均耗时，会把“小任务过多”和“大任务长尾”混在一起。

\topic{work helping 和 work stealing 不同}
Work stealing 发生在 worker 空闲时，它去别人的队列里找 ready 任务。Work helping 发生在 worker 即将等待某个 task group 时，它不睡眠，而是帮助执行能推进该等待条件的 ready 任务。前者解决负载不均，后者缓解 worker 内等待导致的资源浪费。二者可以组合，但语义不同。

Work helping 的风险是重入。等待路径执行了另一个任务，另一个任务可能再等待更多任务，异常传播和栈增长都会复杂。Compute Systems Engine 可以先禁止 worker 内等待，再把 helping 作为扩展。若实现 helping，至少要限制在同一 task group 内，并用 trace 标记帮助执行事件。

\topic{work-first 和 help-first}
递归分治调度常讨论 work-first 和 help-first。Work-first 倾向让当前 worker 继续沿着串行执行路径推进，把需要别人帮忙的 continuation 暴露给 thief；help-first 倾向先把子任务交给别人，当前 worker 继续后续逻辑。二者影响关键路径开销、cache 局部性、任务数量和窃取频率。

Compute Systems Engine 不要求复刻 Cilk 理论模型，但必须说明：\code{spawn} 后谁继续执行、谁进入 deque，不是无关细节。它决定本地性、关键路径和调度开销。报告中如果只写“使用 work stealing”，没有说明任务生成策略，结论不完整。

\topic{NUMA、本地性和公平}
在多 NUMA 节点机器上，随机偷取可能把访问某个内存节点的数据搬到远端执行，降低吞吐。更稳的策略是先在同 NUMA 节点内偷取，空闲时间超过阈值后再跨节点偷取。对于小计算任务，跨节点代价可能不明显；对于大数组块、hash table 或 page cache 相关任务，远端访问会成为主要成本。

公平也不是免费。严格公平可能破坏本地性；强优先级可能让低优先级任务长期饥饿。控制面任务，例如取消传播、manifest commit、错误汇总，通常很短但影响关键路径，可能需要高优先级或独立队列。运行时设计是在吞吐、尾延迟、本地性、公平和实现复杂度之间取舍。

\section{取消、异常、关闭和可见性}

可靠运行时的难点往往不在成功路径，而在任务失败、用户取消、关闭过程和结果可见性。一个只在正常输入下跑得快的线程池，不能作为合格实现的终点。真正可用的运行时必须让每个任务进入终态，让每个等待者醒来，让每个错误有归属，让每个输出有提交边界。

\topic{取消是协作式状态转换}
取消不是强杀线程。强行终止 C++ 线程会破坏 RAII、锁状态、对象不变量和外部资源。运行时通常设置 cancel token，任务在安全点检查，例如处理完一个 chunk、完成一次 I/O、写出一个 block、进入下一轮循环之前。Waiting 或 Ready 任务可以直接标记 \code{Canceled}；Running 任务只能请求停止；Succeeded 或 Failed 任务已经是终态，不能再取消。

取消传播要沿任务图发生。若前驱失败，依赖它的后继可能永远不会 ready。运行时必须根据策略把后继取消、跳过、等待重试或转入补偿任务。这个策略不能由每个 worker 临时决定，应由 task graph、stage 或 driver 统一定义。

\topic{异常不能停在 worker 线程}
用户任务抛异常后，worker 不能静默退出，更不能让未完成任务计数泄漏。示例版至少要捕获逃逸异常、记录 failed、减少未完成计数、唤醒等待者。future 版要把异常保存在 shared state 中，让外部 \code{get()} 观察。任务图版还要决定下游如何处理：取消、重试、继续使用部分结果，还是让整个 job 失败。

\begin{lstlisting}[language=C++,caption={错误摘要比日志字符串更适合运行时}]
enum class TaskErrorKind : std::int32_t {
  None,
  UserException,
  Canceled,
  RejectedByShutdown,
  ResourceLimit,
  InternalRuntimeError,
};

struct TaskErrorSummary {
  std::uint64_t task_id = 0;
  std::uint32_t attempt = 0;
  TaskErrorKind kind = TaskErrorKind::None;
  std::uint32_t error_code = 0;
};
\end{lstlisting}

错误分类影响上层决策。用户异常可能让 stage 失败；取消是预期控制流；资源不足可能触发背压；关闭拒绝说明提交时机错误；内部运行时错误可能要求停止整个 runtime。把它们都写成 \code{failed}，driver 就无法做正确恢复。

\topic{Drain shutdown 和 cancel shutdown}
\code{shutdown drain} 表示停止接收新任务，但已经接纳的任务继续完成。它适合作业正常结束。\code{shutdown cancel} 表示未开始任务进入取消，运行中任务协作检查 token，等待者收到取消或失败结果。两者不能混成一个 \code{shutdown()}，否则调用者不知道结果是否应该完成。

关闭期间的 \code{submit} 要有线性化点。若 \code{submit} 先观察到 \code{Open} 并成功入队，任务必须被执行或取消并通知；若 shutdown 先把状态改成 \code{Draining}，后续提交必须失败。偶发“submit 返回成功但任务丢失”的运行时，是关闭协议没有定义清楚。

\topic{未完成计数的语义}
\code{wait_idle} 不能只看队列是否为空，因为 worker 可能已经取出任务正在执行。它也不能只看 worker 是否空闲，因为提交者可能正准备入队。更可靠的做法是维护未终态任务计数：提交成功加一，任务进入 Succeeded、Failed 或 Canceled 终态时减一，计数归零时唤醒等待者。异常路径、取消路径和拒绝路径都必须审查计数是否平衡。

\topic{内存可见性}
使用 mutex 和 condition variable 的第一版，发布关系通常由锁建立：提交者先写完任务对象，再在锁内入队并通知；worker 在锁内取出任务后能看到字段。无锁 ready 队列和工作窃取 deque 会把这个问题暴露出来：生产者必须先构造任务，再用 release 语义发布；消费者必须用 acquire 语义获取，才能看到任务字段。

结果发布也是同样道理。worker 写入结果后把状态设为完成，等待者读取完成状态后必须能看到结果。标准库 future/promise 封装了这些同步；自研 shared state 就要自己建立 happens-before。运行时不是只管调度，它还管结果何时对其他线程可见。

\topic{低开销观测}
Trace 和 counters 是运行时的眼睛，但观测本身不能成为瓶颈。每个任务完成时更新同一个全局原子计数器，在小任务高频场景下会形成共享写热点。更好的做法是 worker 写自己的 \code{WorkerSnapshot} 或 per-worker buffer，周期性汇总。协议状态必须精确，例如未完成任务计数；实验指标可以延迟汇总，例如平均任务耗时和空闲时间。

观测数据也要分冷热。调度热路径只应写少量固定字段，例如 task id、worker id、start/finish 时间戳和终态；字符串格式化、路径名、错误栈和详细上下文应进入冷路径，或者只在失败时采集。per-worker trace buffer 能把写入限制在本 worker 常用 cache line 上，降低共享原子争用；汇总线程再离线计算关键路径、等待分布和 steal 成功率。若 trace 让所有 worker 抢同一个日志锁，测到的就是日志系统，而不是运行时。

\begin{lstlisting}[language=C++,caption={运行时 trace 和 worker 指标的示例形态}]
struct TaskStateRecord {
  std::uint64_t task_id = 0;
  std::uint32_t attempt = 0;
  std::uint32_t worker_id = 0;
  std::uint32_t remaining_dependencies = 0;
  std::uint64_t submit_ns = 0;
  std::uint64_t ready_ns = 0;
  std::uint64_t start_ns = 0;
  std::uint64_t finish_ns = 0;
  TaskErrorKind error_kind = TaskErrorKind::None;
};

struct WorkerSnapshot {
  std::uint32_t worker_id = 0;
  std::uint64_t executed_tasks = 0;
  std::uint64_t local_pops = 0;
  std::uint64_t steal_attempts = 0;
  std::uint64_t steal_successes = 0;
  std::uint64_t idle_ns = 0;
  std::uint64_t running_ns = 0;
};
\end{lstlisting}

这些字段会在分布式章节继续扩展成 task attempt、executor、request id、epoch 和 shuffle block。单机运行时先把身份和状态打稳，后续扩展才不会换一套概念。

\section{工业界设计案例}

工业系统的价值不在于名字，而在于它们在约束下做出的选择。读源码或论文时，报告应回答五个问题：它面对什么负载和失败约束；核心机制是什么；机制购买了什么能力；付出了什么成本；Compute Systems Engine 应该采用、简化还是拒绝哪些部分。

\begin{casebox}
\topic{Cilk 和 work-first 调度}
Cilk 系列运行时把 fork-join 并行的常见路径做得很轻，窃取发生在空闲 worker 上。它的核心原则不是“有一个 deque”，而是把大部分调度开销放到被偷取的少数路径上，让普通串行路径接近原程序执行。这个思想适合递归分治和细粒度并行，但要求任务函数遵守比较清晰的 fork-join 结构。

Compute Systems Engine 可以借鉴它的两个原则：一是本地执行路径要便宜，二是窃取应该暴露较粗工作而不是极小任务。不必一开始实现 Cilk 的完整 continuation 模型，也不必把所有算法都改成递归 spawn。对 stable filter、partition 和 reduce 来说，显式 task table 更利于 trace、取消和恢复。
\end{casebox}

\begin{casebox}
\topic{oneTBB 和任务粒度}
oneTBB 的工程价值在于把任务划分、worker 调度、work stealing 和可组合并行算法组织在一起。它不会神奇消除任务粒度问题：\code{parallel_for} 或 flow graph 的性能仍取决于 range 切分、body 成本、数据局部性和阻塞行为。成熟设计提供机制和默认策略，但无法替业务选择语义。

Compute Systems Engine 借鉴 oneTBB 的方式是：先写 range、blocked range、grain size 和 trace，再决定是否 work stealing。报告要说明任务体是否足够重、是否访问连续内存、是否会阻塞 I/O。若任务只是几十条指令，换成成熟运行时也不会自动变快。
\end{casebox}

\begin{casebox}
\topic{OpenMP task 和依赖表达}
OpenMP task 让程序员声明任务及其依赖，运行时在依赖满足后调度。这说明工业界也不会把所有并行性都塞进线程数量。依赖表达可以提高可读性和调度自由度，但也会引入运行时开销、调试复杂度和数据依赖声明错误的风险。

Compute Systems Engine 不需要复制 OpenMP 语法，但要学习它的抽象：task 不是立即执行的函数调用，而是带依赖和生命周期的工作记录。对 count、scan、write、commit 来说，依赖边比 worker 内等待更清楚。
\end{casebox}

\begin{casebox}
\topic{Folly executor 和执行域}
Folly 等工程库强调 executor 抽象：不同任务应进入合适的执行域。CPU-bound 任务、I/O completion、定时器、后台维护、优先级队列，资源特征不同，不能全部塞进一个 FIFO 池。这个思想会在 I/O completion 和背压里继续展开。

Compute Systems Engine 可以先定义 compute domain 和 blocking I/O domain。Compute worker 只执行 ready 的 CPU 任务；I/O 任务或慢等待进入独立执行域；完成事件再把后继任务放回任务图。这样慢磁盘、慢网络或模拟 I/O 不会占满 compute worker。
\end{casebox}

\begin{casebox}
\topic{Linux workqueue}
Linux workqueue 把“稍后执行的工作项”和“执行它的 worker pool”分开，并处理并发限制、阻塞、救援线程和 CPU 亲和性等工程问题。用户态不应照抄内核实现，但可以学习它的状态分离：工作项有身份和状态，worker pool 有资源和策略，阻塞不能让整个系统无法前进。

源码阅读时不要试图一次读完整个 workqueue。围绕一个问题进入：工作如何入队，worker 如何被唤醒，work item 如何避免重复排队，阻塞路径为什么需要救援机制。把这些问题和 task state、ready queue、shutdown、trace 对照，收获会比背函数名更大。
\end{casebox}

\topic{数据库执行器和分布式调度的同构性}
DuckDB、ClickHouse、Spark 这类系统表面差异很大，但都在处理类似对象：可执行 fragment、依赖、worker、队列、partition、manifest、失败和重试。数据库 vectorized execution 关注批和局部性；Spark DAG scheduler 关注 stage 依赖和 task attempt；本地运行时关注 worker 和 deque。它们的共同原则是：先把工作身份和依赖写清，再谈调度策略。

\section{深度案例：stable filter 如何进入任务运行时}

第 13 章已经给出 stable filter 的算法结构：count、scan、write、commit。现在把它放入任务运行时。朴素实现会在阶段之间使用 barrier：提交所有 count，主线程等待；做 scan；再提交 write；最后等待 commit。这个版本简单、可靠，是第一版 reference。但它也暴露问题：barrier 处可能让 worker 空闲；阶段等待由主线程控制，trace 不足；失败和取消只能粗粒度处理。

任务图版本把每个阶段写成节点和边。所有 count block 初始 ready；最后一个 count 完成后，scan node ready；scan 完成后，所有 write block ready；write 完成后递减 commit 的计数；commit 发布 manifest。worker 不需要阻塞等待下一阶段，运行时能看到每个节点处于 Waiting、Ready、Running 还是终态。

\topic{barrier reference 仍然要保留}
任务图不是为了否定 barrier。Barrier 版本简单，适合作为 reference 和基线起点。它帮助验证输出、manifest、错误处理和基础性能。任务图版本必须和 barrier reference 在语义上对齐：同样输入、同样谓词、同样稳定顺序、同样 manifest。没有 reference，任务图变快或变慢都不好解释。

\topic{失败路径}
若某个 count 失败，scan 不应永远 waiting。运行时应把 scan 和后续 write、commit 标记为 canceled 或让 driver 发起重试。若某个 write 失败，commit 不能发布 manifest。若 write 已经产生临时输出，commit 前崩溃恢复时必须能清理未提交文件。这正是任务运行时和第 15 章 I/O 提交协议的连接点。

\topic{长尾和 work stealing}
如果一个 write block 因输入倾斜变得很慢，work stealing 只能帮助尚未开始或可拆分的工作。已经运行且不可拆的长任务，偷不走。解决方式可能是更细粒度切分、热点 block 特殊处理、动态拆分剩余范围，或在下一轮根据历史统计调整块大小。Work stealing 不是长尾万能药，它需要算法暴露可偷取的剩余工作。

\topic{关键路径判读}
Stable filter DAG 的关键路径大致是最长 count、scan、最长 write、commit 之和。若 trace 显示端到端时间远高于这个估计，检查 ready 队列等待、worker 空闲、窃取失败和 I/O 等待。若接近关键路径，则优化方向是加速 scan、拆分长 write、批量 commit 或改变算法依赖，而不是继续加 worker。

\section{实现路径}

Compute Systems Engine 的任务运行时不要一次写成完整工业框架。正确路线是小步推进，每一步都有 reference、验证边界和运行证据。

\topic{第一步：全局队列 reference}
实现固定 worker 数、全局 FIFO 队列、\code{submit}、\code{wait_idle}、\code{shutdown drain}、异常捕获和未完成计数。测试包括：提交 N 个任务全部执行一次；任务抛异常不会杀死 worker；\code{wait_idle} 在任务终态后返回；shutdown 后 submit 失败；析构不会遗留 joinable thread；多生产者提交不破坏队列。

\topic{第二步：任务图和 trace}
加入 task id、ready count、successor 列表、TaskStateRecord 和错误摘要。先用全局 ready 队列执行任务图。用第 13 章 stable filter 的 count、scan、write、commit 作为 workload。测试并发前驱同时完成时，后继只入队一次；前驱失败时，后继不会永远 waiting；空图和单节点图都能正常终止。

\topic{第三步：本地 deque 和工作窃取}
每个 worker 增加 mutex deque。外部提交进入 global inject queue；worker 产生的本地后继可以进入自己的 deque；空闲 worker 先取全局，再随机或轮询 steal。记录 local pops、steal attempts、steal successes、empty steals、worker idle time 和 stolen task duration。先验证策略收益，再考虑无锁 deque。

\topic{第四步：执行域和背压接口}
区分 compute domain 和 blocking 或 I/O domain。CPU 任务不要阻塞等磁盘或网络；I/O 完成事件通过 CompletionRecord 重新进入任务图。队列容量要有返回值，不能无限堆积。这个步骤和第 15 章衔接。

\begin{lstlisting}[language=C++,caption={项目中 TaskRuntimePlan 的接口草图}]
enum class QueuePolicy : std::int32_t {
  GlobalOnly,
  LocalDeque,
  WorkStealing,
};

struct TaskRuntimePlan {
  std::uint32_t worker_count = 0;
  QueuePolicy queue_policy = QueuePolicy::GlobalOnly;
  bool enable_task_graph = false;
  bool enable_trace = true;
  bool enable_cancellation = true;
};
\end{lstlisting}

这个结构不是为了把运行时配置化成复杂产品，而是为了让机制能逐项打开。全局队列、任务图、work stealing、trace、cancellation 每个复杂度都要有购买理由。若某机制说不出修复哪个反例，就不应该进入最小实现。

\topic{Trace 摘要命令}
项目应提供类似 \cmd{engine trace summarize} 的输出。最低字段包括任务总数、终态分布、最长 ready 等待、最长 running、估计关键路径、worker idle 比例、steal 成功率、取消传播时间和首个错误任务。摘要可以是 key-value 文本，不要求 GUI。关键是能从证据解释运行时，而不是凭感觉判断。

\begin{lstlisting}[numbers=none,caption={任务运行时 trace 摘要示例}]
tasks.total=1536
tasks.succeeded=1536
critical_path_ns=18400000
makespan_ns=23700000
worker.idle_ratio.p95=0.18
steal.attempts=421
steal.successes=73
task.ready_wait_ns.max=2600000
task.running_ns.max=9100000
\end{lstlisting}

若 makespan 远大于 critical path，继续查 ready wait、idle ratio、steal 失败和阻塞事件。若 steal attempts 很多但 successes 很少，victim 选择或任务分布有问题。若 success 很多但吞吐不升，任务可能太小或跨 NUMA 破坏了局部性。

最小综合路径先实现全局队列线程池 reference，再实现任务图版 stable filter。worker 内禁止调用阻塞 \code{future.get()} 等待同池任务。运行证据必须能解释 future 饥饿反例、DAG trace、关键路径估计、取消传播和 shutdown 行为。这样一条路径已经覆盖运行时最关键的边界，不需要额外堆很多分散场景。

\section{运行时实验门禁}

\topic{一、是否从事故推出机制}
所有机制都应能回到 future 饥饿或任务图等待事故。若某段话只是在介绍术语，不能解释哪个失败现象，就应该改成具体反例。

\topic{二、是否区分等待和 ready}
Waiting 表示依赖未满足，Ready 表示可以执行。worker 只应长期执行 Ready 任务。若代码让 worker 睡在 Waiting 条件上，必须有 helping、隔离执行域或明确禁止规则。

\topic{三、是否保留 reference}
全局队列和 barrier 版本不是落后版本，而是语义 reference。work stealing 和 DAG 优化必须能和 reference 对齐输出、错误和 manifest。

\topic{四、是否有失败路径}
只测试成功路径的运行时不合格。至少要测试用户任务异常、上游失败导致后继取消、shutdown 并发 submit、future 饥饿反例和 worker join。

\topic{五、是否解释复杂度购买理由}
本地 deque 购买局部性和低争用；stealing 购买负载均衡；task graph 购买显式依赖；continuation 购买非阻塞等待；trace 购买诊断能力；cancellation 购买资源收束。若购买理由不成立，就不应增加机制。

\topic{六、是否连接 I/O completion}
CPU 任务的等待和完成已经被写成状态机。I/O 会把设备等待也写成提交和 completion。TaskStateRecord、CompletionRecord、背压和 manifest 提交边界应自然衔接。
