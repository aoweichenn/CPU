\chapter{实践卷：并发、运行时和系统合同}

第二册的实践卷分两层。第一层是本册已有实验 \filepath{books/cpu-volume-2/labs/compute_systems}，覆盖并行归约、同步原语、有限队列、wait channel、任务运行时和 futex 切片。第二层是配套代码实践卷 \filepath{books/compute-systems-engine-code}，负责把 reference、manifest、恢复和报告继续做成完整工程。本附录先说明本册实验如何跑、如何查、什么情况下不能把结果写进结论。

\section{实践一：并发代码先验合同}

并发实验不能只看最终数字。有限队列要检查 \code{push}、\code{pop}、\code{close}、等待次数和关闭唤醒；任务运行时要检查 submitted、completed、failed、rejected、unfinished 和 max queue depth；futex 切片要明确用户态快路径和内核等待路径的边界。若只写“多线程跑完了”，没有证明没有丢任务、重复执行或关停遗漏。

\begin{lstlisting}[language=bash,caption={第二册 compute systems 实验构建和测试}]
cmake -S books/cpu-volume-2 -B books/cpu-volume-2/build/compsys-debug -DCMAKE_BUILD_TYPE=Debug
cmake --build books/cpu-volume-2/build/compsys-debug
ctest --test-dir books/cpu-volume-2/build/compsys-debug --output-on-failure
\end{lstlisting}

\section{实践二：每个机制都要有失败样本}

第二册正文里的术语必须落到失败样本上。backpressure 是下游变慢时上游被限制，而不是“队列满了”一句话；wait channel 是等待者和唤醒条件的组合，而不是“睡眠”；attempt 是一次可区分的尝试，重试时不能和旧结果混在一起；epoch 是一段配置或所有权的版本边界，旧 epoch 消息不能覆盖新状态。

实践报告要为每个机制写一个失败样本。有限队列：消费者阻塞后 \code{close} 是否唤醒；任务运行时：同池 future wait 是否会造成 worker 饥饿；futex：锁竞争下计数是否仍稳定；并行归约：不同 worker 数是否和顺序 reference 一致。失败样本不是为了证明代码差，而是为了证明合同足够清楚。
