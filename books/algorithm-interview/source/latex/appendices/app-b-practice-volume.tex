\chapter{实践卷：从 LeetCode 案例到可验证代码}

这一卷不是“答案合集”，而是把刷题过程拆成可以复查的工程过程。入口工程在：

\begin{lstlisting}[language=bash]
books/algorithm-interview/labs/cpp20-examples
\end{lstlisting}

每个案例都按五个阶段推进：先把题目说完整，再写最慢但可信的 reference，再从小例子推到优化规律，然后补边界测试，最后把代码和复杂度验收连起来。reference 在这里指“可信对照实现”：它可以慢，但必须直观、覆盖完整语义，后续优化版要和它对同一批输入给出同一结果。

\section{二、数组、窗口和前缀和案例}

\subsection{案例 1：LeetCode 1 两数之和}

题目描述：给定一个整数数组 \code{nums} 和一个目标值 \code{target}，返回两个不同下标 \code{i,j}，使 \code{nums[i]+nums[j]=target}。注意返回的是下标，不是数值；同一个元素不能用两次。样例 \code{[2,7,11,15]} 和 \code{target=9} 的答案是 \code{[0,1]}。

阶段 0 是合同。函数 \code{two_sum_bruteforce} 和 \code{two_sum_hash_table} 接收同一份数组，返回两个下标；如果工程扩展到“无答案”场景，应返回空数组，而不是返回未初始化下标。

阶段 1 写暴力 reference。双重循环枚举所有 \code{i<j} 的组合，只要有一组满足就返回。它慢在 \complexity{O(n^2)}，但正确性容易看清：所有合法二元组都被覆盖，而且不会重复使用同一个位置。

阶段 2 推到哈希表优化。扫描到位置 \code{i} 时，历史位置里如果存在 \code{target-nums[i]}，就能和当前元素组成答案。关键顺序是“先查再插入当前元素”。如果先插入，\code{nums=[3]}、\code{target=6} 会错误地把同一个 \code{3} 用两次；对 \code{[3,3]}，先查再插入能在第二个 \code{3} 处找到第一个 \code{3}。

阶段 3 补边界测试。\code{tests/algorithm_book_tests.cpp} 已检查普通样例和重复值 \code{[3,3]}。继续扩展时应补：负数、目标值为 0、无答案、多组答案时返回任意合法下标但不能越界。

\subsection{案例 2：LeetCode 11 盛最多水的容器}

题目描述：给定数组 \code{height}，第 \code{i} 根线的高度为 \code{height[i]}。选择两个下标 \code{left<right}，容器面积为
\code{min(height[left],height[right])*(right-left)}，求最大面积。样例 \code{[1,8,6,2,5,4,8,3,7]} 的答案是 \code{49}。

阶段 1 的 reference 是 \code{max_area_bruteforce}：枚举每一对左右边界，直接计算面积。它覆盖所有容器，所以不会漏解。

阶段 2 推导双指针。开始时左右指针放在最宽的位置。若左边高度更小，当前面积受左边短板限制。此时右指针左移会让宽度变小，而短板仍然不可能超过左边高度，所以不可能因为移动右指针得到更大面积；只有移动左指针，才可能遇到更高的短板。右边更短时同理。这不是口号，而是由“面积由较短线决定”和“宽度移动后只会变小”共同推出的剪枝。

阶段 3 的验收不是只跑样例。必须让 \code{max_area_bruteforce} 和 \code{max_area_two_pointers} 在同一批输入上对照：单调递增、单调递减、全相等、只有两根线、包含 0 高度。

\subsection{案例 3：LeetCode 209 长度最小的子数组}

题目描述：给定正整数数组 \code{nums} 和目标值 \code{target}，找一个连续子数组，使其元素和至少为 \code{target}，返回最短长度；不存在则返回 0。样例 \code{target=7}、\code{nums=[2,3,1,2,4,3]} 的答案是 2，因为 \code{[4,3]} 达标且最短。

阶段 1 是 \code{min_subarray_len_bruteforce}：固定左端，右端向右扩展并累加当前和，每次达标就更新最短长度。它是 \complexity{O(n^2)}，但比每个区间重新求和的 \complexity{O(n^3)} 更清楚地复用了一段扫描历史。

阶段 2 才能使用滑动窗口 \code{min_subarray_len_sliding_window}。为什么可以收缩左端？因为题目保证数组元素是正数。右端向右走时窗口和只会增加；当窗口和已经达标，左端右移会让窗口和减少，可以尝试得到更短窗口。正数条件是核心合同。如果数组里有负数，例如 \code{[5,-10,20]}，右端扩展和左端收缩都不再单调，窗口策略会丢掉本该保留的历史。

阶段 3 的测试必须包含：刚好等于目标、无答案、单元素达标、多个答案取最短、全为正数的长数组。若要支持负数，这道题应换成前缀和加数据结构的另一类解法，不能继续套本函数。

\subsection{案例 4：LeetCode 560 和为 K 的子数组}

题目描述：给定整数数组 \code{nums} 和整数 \code{k}，统计连续子数组和恰好等于 \code{k} 的个数。数组可以包含负数和 0。样例 \code{nums=[1,1,1]}、\code{k=2} 的答案是 2，对应两个 \code{[1,1]}。

阶段 1 是 \code{subarray_sum_equals_k_bruteforce}。固定左端，右端向右扩展并维护当前和；每次当前和等于 \code{k} 就计数。它覆盖所有连续段，所以适合做 reference。

阶段 2 推到前缀和哈希。设 \code{prefix[t]} 表示前 \code{t} 个元素之和，区间 \code{[i,j]} 的和是 \code{prefix[j+1]-prefix[i]}。要让它等于 \code{k}，就需要历史里存在 \code{prefix[i]=prefix[j+1]-k}。因此扫描到当前前缀和 \code{cur} 时，答案增加“历史前缀 \code{cur-k} 出现过的次数”，然后把 \code{cur} 记入哈希表。初始时必须有一次前缀和 0，因为从下标 0 开始的区间也要被统计。

阶段 3 解释为什么不能用普通滑动窗口。负数会让窗口和在扩展时变小，在收缩时变大；全零数组 \code{[0,0,0]}、\code{k=0} 有 6 个子数组，靠左右指针很容易漏计重复前缀。测试中已经保留负数和全零用例，用来证明前缀哈希不是模板替换，而是题目条件逼出来的选择。

\subsection{案例 5：LeetCode 3 无重复字符的最长子串}

题目描述：给定字符串 \code{s}，求不含重复字符的最长连续子串长度。样例 \code{"abcabcbb"} 的答案是 3，对应 \code{"abc"}。

阶段 1 的 reference 是 \code{longest_substring_without_repeat_bruteforce}。从每个左端开始向右扩展，用集合记录当前子串中的字符；遇到重复就停止。它慢，但能清楚说明“连续”和“不重复”的边界。

阶段 2 使用窗口和最后出现位置。窗口左边界 \code{left} 表示当前无重复子串的起点。扫描到字符 \code{c} 的位置 \code{right} 时，如果 \code{c} 上一次出现位置在窗口内，就把 \code{left} 移到那个位置之后。左边界只能右移，不能回退；因为回退会把已经确认冲突或过期的字符重新放回窗口，破坏“不重复”不变量。

阶段 3 应测试空串、全重复、全不重复、重复字符出现在窗口外、大小写字符是否区分。若改成 Unicode 字符串，还要重新定义“字符”的单位，不能默认一个 \code{char} 就是一个用户感知字符。

\subsection{扩展案例：LeetCode 53 最大子数组和}

题目描述：给定整数数组 \code{nums}，找一个非空连续子数组，使元素和最大，返回这个最大和。样例 \code{[-2,1,-3,4,-1,2,1,-5,4]} 的答案是 6，对应 \code{[4,-1,2,1]}。

阶段 1 先写 \complexity{O(n^2)} reference：固定左端，右端向右扩展，维护当前区间和并更新最大值。它比 \complexity{O(n^3)} 的“每个区间重新累加”更好，但仍然覆盖所有连续子数组。

阶段 2 从特例推状态。设 \code{best_end_here} 是“必须以前一个位置结尾的最佳连续段”。当处理当前元素 \code{x} 时，只有两个选择：要么让 \code{x} 单独开一段，要么把上一段接到 \code{x} 前面。如果上一段和是负数，接上它会让当前段变小，例如 \code{-5+4<4}，应该丢弃；如果上一段和是正数，接上它会让当前段变大，例如 \code{3+4>4}，应该保留；如果上一段和是 0，接不接对和没有影响，但保留会影响区间起点。于是转移不是背公式，而是：

\begin{lstlisting}[language=C++]
best_end_here = std::max(x, best_end_here + x);
best_overall = std::max(best_overall, best_end_here);
\end{lstlisting}

阶段 3 补测试：全负数组不能返回 0，应返回最大的那个负数；单元素数组直接返回该元素；全 0 返回 0；正负交替要验证没有把不连续的正数拼起来。当前实践工程尚未落地本题函数，补入时建议新增 \code{max_subarray_bruteforce} 和 \code{max_subarray_kadane}，并先用 reference 对照优化版。

\subsection{案例 6：LeetCode 994 腐烂的橘子}

题目描述：网格里 \code{0} 表示空格，\code{1} 表示新鲜橘子，\code{2} 表示腐烂橘子。每一分钟，腐烂橘子会让上下左右相邻的新鲜橘子腐烂。求所有新鲜橘子腐烂所需的最少分钟数；如果有新鲜橘子永远不能腐烂，返回 \code{-1}。

阶段 0 明确这是多源 BFS。初始队列不是一个点，而是所有腐烂橘子。队列中的一层对应同一分钟能传播到的所有位置。

阶段 1 写 \code{oranges_rotting_bfs} 时先扫描网格：把所有 \code{2} 入队，统计新鲜橘子数量。若一开始没有新鲜橘子，答案是 0。每弹出一层，只让新鲜邻居变成腐烂并入队，不能让空格或已经腐烂的位置重复入队。

阶段 2 的时间计数要贴着“层”。如果这一分钟没有新橘子腐烂，就不应该增加时间。测试里已有普通样例和被隔开的新鲜橘子；继续补充时应加入空网格、全空格、全新鲜无腐烂源、单个腐烂橘子。

\subsection{案例 7：LeetCode 200 岛屿数量}

题目描述：给定由 \code{'1'} 和 \code{'0'} 组成的网格，\code{'1'} 表示陆地，\code{'0'} 表示水。上下左右相邻的陆地属于同一座岛，求岛屿数量。斜角相邻不算连通。

阶段 0 明确它不是多源 BFS，而是连通块扫描。外层遍历每个格子；遇到未访问陆地，岛屿数量加一，并从这里启动一次 BFS，把整座岛全部标记。

阶段 1 写 \code{number_of_islands_bfs}。访问标记要在入队时写，而不是出队时写；否则同一个陆地可能被多个邻居重复入队，影响复杂度和报告。可以直接把访问过的陆地改成水，也可以维护 \code{visited} 数组，但要在报告里说明选择。

阶段 2 的测试要体现“连通定义”。全水返回 0，全陆地返回 1，多个岛分开返回多个；对角线相邻的两个 \code{'1'} 仍然是两座岛。若把斜角也算邻接，题目已经变了。

\section{四、回溯扩展案例：LeetCode 679 二十四点游戏}

题目描述：给定 4 个数字，判断能否通过加、减、乘、除和括号得到 24。它考的不是某个公式，而是状态枚举和回退。状态是“当前还剩哪些数”；一步操作是任选两个数 \code{a,b}，把它们替换成 \code{a+b}、\code{a-b}、\code{b-a}、\code{a*b}、\code{a/b}、\code{b/a} 中合法的结果。

阶段 1 写最小 reference：用 \code{std::vector<double>} 保存当前数字。若只剩一个数，检查它和 24 的差是否小于误差。否则枚举两个位置，生成下一轮数字，再继续搜索。

阶段 2 解释“回退”。回退不是神秘动作，而是一次尝试结束后恢复到尝试前的状态。最稳妥的写法是不要原地改原数组，而是为每个选择构造 \code{next} 数组；如果为了减少分配选择原地 \code{erase/push_back}，就必须在返回前把删掉的两个数和插入的结果恢复。

阶段 3 剪枝只在理解完整搜索后再加。加法和乘法满足交换律，\code{a+b} 与 \code{b+a} 不必重复；减法和除法不满足交换律，两个方向都要试；除法要跳过接近 0 的分母。当前工程还没有回溯文件，落地时建议新增 \code{backtracking_patterns.hpp/.cpp} 和对应测试。

\section{五、项目代码分阶段要求}

后续新增任何案例，都按下面的工程顺序进入书稿：

\begin{enumerate}
  \item 阶段 0：写完整题目描述、输入输出合同、LeetCode 编号和至少一个手算样例。
  \item 阶段 1：写 reference 函数，并说明为什么它覆盖所有合法答案。
  \item 阶段 2：写优化函数，从特例和失败样本推到不变量，不能只贴模板。
  \item 阶段 3：在 \filepath{tests/algorithm_book_tests.cpp} 加同一批输入的 reference 对照、边界样例和反例。
  \item 阶段 4：写复杂度解释。复杂度必须从循环次数、队列入队次数、下标移动次数或哈希访问次数推出。
  \item 阶段 5：运行构建和测试命令，把失败样本保留下来。没有测试的题解不进入正文结论。
\end{enumerate}
