\chapter*{常见缩写和术语先导}
\addcontentsline{toc}{chapter}{常见缩写和术语先导}

代码实践卷会把第二册的概念落到工程对象。第一次看到下面这些词时，先把它们理解成“代码里必须有边界的对象”，而不是抽象口号；每个对象都要能在代码、测试、probe 输出或报告里找到证据。英文全称不是背单词用的，它用来防止把相近概念混成一个词。

\begin{description}
  \item[CPU: Central Processing Unit] 中央处理器。它执行指令、访问寄存器和内存，并通过缓存、分支预测、流水线和乱序执行影响程序速度。本卷的实验默认先落在本地 CPU 上，再逐步讨论 SIMD、线程、I/O 和分布式扩展。
  \item[HPC: High Performance Computing] 高性能计算。它关注吞吐、延迟、并行度、内存带宽和资源利用率。本卷提到 HPC 时，不是泛泛追求“更快”，而是要求用 benchmark、PMU 证据、输入规模和正确性校验说明快在哪里。
  \item[PMU: Performance Monitoring Unit] 性能监控单元。现代 CPU 用 PMU 统计 retired instructions、cache miss、branch miss 等硬件事件。实践卷的 \code{model_ch05b_pmu_simd_evidence} 用它提醒读者：只看耗时不够，必须知道程序到底执行了哪些指令、触发了哪些硬件事件。
  \item[SIMD: Single Instruction, Multiple Data] 单指令多数据。一条向量指令同时处理多个 lane。第 7 章模型会统计 vector iterations、scalar tail 和 lane utilization，说明“用了 SIMD”不等于“每个 lane 都被有效利用”。
  \item[ILP: Instruction-Level Parallelism] 指令级并行。它指 CPU 在不改变语义的前提下，让多条独立指令重叠执行的能力。本卷用它解释为什么数据依赖、分支和内存延迟会限制热循环吞吐。
  \item[TLB: Translation Lookaside Buffer] 地址转换后备缓冲。CPU 访问虚拟地址时需要查页表，TLB 缓存最近的虚拟页到物理页映射。第 3 章模型把 cache line、page 和 TLB miss 分开统计，避免把所有内存慢都叫“缓存问题”。
  \item[NUMA: Non-Uniform Memory Access] 非一致内存访问。多路或多节点机器上，本地内存和远端内存访问成本不同。第 4 章模型把 local access、remote access 和 coherence transfer 分开记录。
  \item[SMP: Symmetric Multiprocessing] 对称多处理。多个 CPU 核心共享同一操作系统实例和统一内存视图。它是理解线程调度、锁竞争和共享队列的背景，但 NUMA 机器上的“共享内存”仍可能有远近差异。
  \item[LLC: Last-Level Cache] 最后一级缓存，通常指 L3 或芯片上离内存最近的共享缓存。它影响跨核心共享数据、工作集大小和内存带宽压力。
  \item[L1/L2/L3 Cache: Level 1/2/3 Cache] 一级、二级、三级缓存。L1 最靠近核心、容量小、延迟低；L2 居中；L3/LLC 容量更大、常被多个核心共享。读缓存模型时要区分 cache line 命中、容量 miss 和 TLB miss。
  \item[IPC: Instructions Per Cycle] 每周期退休指令数。它粗略描述 CPU 每个时钟周期完成多少条指令。IPC 低可能来自分支失败、缓存 miss、依赖链、同步等待或 I/O 阻塞，不能单独当成最终答案。
  \item[GFLOPS: Giga Floating-Point Operations Per Second] 每秒十亿次浮点运算。Roofline 模型用它表示计算上限和可达性能；但没有 bytes、bandwidth 和 checksum，GFLOPS 数字没有上下文。
  \item[I/O: Input/Output] 输入输出。文件、网络、管道和设备读写都属于 I/O。本卷强调 submit、completion、commit 不是同一件事：提交请求、收到完成事件、业务确认持久化必须分开建模。
  \item[EOF: End Of File] 文件结束。EOF 不是错误，也不等于短读。读取循环要区分“正常结束”“本次只读到部分数据”和“系统调用失败”。
  \item[CLI: Command-Line Interface] 命令行接口。CLI 用参数、标准输入输出和退出码表达工具行为。本卷 CLI 只拼接输入和输出，不复制 parser、reference 或 split 逻辑。
  \item[CI: Continuous Integration] 持续集成。每次提交后自动构建、测试和检查，防止本地偶然通过的代码进入主线。本卷的最小 CI 思路就是 CMake 构建、CTest、PDF 构建和导出校验。
  \item[JSON: JavaScript Object Notation] 结构化文本格式。适合保存配置、manifest 和机器可读报告。它比随手拼字符串更适合后续由脚本解析。
  \item[CSV: Comma-Separated Values] 逗号分隔文本表格。适合保存 benchmark 结果、trace 摘要和 probe 输出，但字段顺序必须稳定。
  \item[MPMC: Multiple Producer, Multiple Consumer] 多生产者多消费者。多个线程可以同时入队和出队。实践卷的 bounded queue 使用 MPMC 语义，因此要测试容量、等待、关闭和唤醒。
  \item[MPSC: Multiple Producer, Single Consumer] 多生产者单消费者。多个生产者写入，一个消费者读取，常见于日志队列和事件汇聚。它比 MPMC 简单，但消费者成为固定汇聚点。
  \item[SPSC: Single Producer, Single Consumer] 单生产者单消费者。一个生产者、一个消费者，常用于 ring buffer。第 12 章模型用 SPSC ring 固定 push、pop、full 和 empty 边界。
  \item[CAS: Compare-And-Swap] 比较并交换。它读取内存位置，若旧值等于期望值则写入新值，是很多无锁结构的基础。CAS 失败不一定是错误，通常表示其他线程已经抢先修改。
  \item[ABA Problem] ABA 问题。线程看到值从 A 变成 B 又变回 A，单看当前值会误以为没有变化。无锁栈、节点复用和内存回收都要考虑 ABA。
  \item[RCU: Read-Copy-Update] 读-复制-更新。读者不加重锁读取旧版本，写者复制并发布新版本，等待旧读者离开后再回收旧对象。本卷用它帮助理解 epoch reclamation 的动机。
  \item[RAII: Resource Acquisition Is Initialization] 资源获取即初始化。C++ 用对象生命周期管理锁、文件、内存和线程 join 等资源。实践代码应让析构自动释放资源，减少异常路径泄漏。
  \item[RPC: Remote Procedure Call] 远程过程调用。调用者像调用本地函数一样发起远端请求，但工程上必须处理超时、重试、幂等和版本。第 16 到 18 章的 attempt、quorum、checkpoint 都是 RPC/分布式系统的基础问题。
  \item[TCP: Transmission Control Protocol] 传输控制协议。它提供有序、可靠、字节流传输，但不提供业务消息边界、幂等和事务提交。上层仍要定义 framing、retry 和 manifest。
  \item[TSan: ThreadSanitizer] 线程检查器，用于发现数据竞争和部分同步错误。它不能证明并发程序完全正确，但能尽早暴露未受保护的共享访问。
  \item[GTest: GoogleTest] C++ 单元测试框架。本卷用测试固定正常输入、坏行、split 边界、队列关闭和运行时行为。
  \item[GMock: GoogleMock] GoogleTest 配套的 mock 框架。它用于构造可控依赖和交互断言；本卷把它作为工程能力边界保留，即使某些小实验暂时不需要复杂 mock。
  \item[Google Benchmark] C++ benchmark 库，用于重复测量函数成本。本卷 benchmark 只测目标热路径，不把随机输入生成、文件读取和报告打印混入同一次测量。
  \item[reference] 参考实现。它可以慢，但必须清楚、确定、可复查，用来定义正确结果。优化版只有在输出满足 reference 合同后，才有资格谈性能。
  \item[oracle] 判定器。它比较 reference 和待测实现，判断输出是否满足同一语义和误差合同。
  \item[contract] 合同。这里不是法律文件，而是代码边界的明确约定：输入是什么，输出是什么，错误怎样表达，哪些行为不承诺。
  \item[manifest] 发布清单。它记录输入版本、输出位置、校验和、参数和提交事实，让一次运行可以被脚本复查。
  \item[checkpoint] 检查点。保存可恢复状态，让失败后可以从已提交边界继续，而不是从头重跑。
  \item[attempt] 一次物理尝试。同一个逻辑任务可能因为超时或失败重试而产生多个 attempt；最终报告要区分逻辑任务和物理尝试。
  \item[epoch] 版本或轮次编号。它用来区分不同时刻的状态和提交边界，也是很多内存回收和分布式提交协议的基本字段。
  \item[watermark] 水位线。表示某个进度之前的数据已经被观察、处理或提交。流处理和恢复系统用它决定哪些旧状态可以清理。
  \item[quorum] 法定多数或提交多数。分布式副本里，只要达到规定数量的确认就能做出提交判断。第 17 章模型用 replica count 和 acknowledgements 推导 quorum size。
  \item[futex: Fast Userspace Mutex] Linux 的快速用户态互斥原语。无竞争时留在用户态，有竞争时进入内核等待或唤醒。实践卷用 futex probe 说明 wait/wake 不是普通自旋。
  \item[eventfd] Linux 事件计数文件描述符。常用于把线程或内核事件接入 \code{epoll}。它不是消息队列，只是一个可读的计数通知通道。
  \item[epoll] Linux I/O 事件复用接口。它让程序等待多个文件描述符的 readiness 事件。实践卷里要区分 readiness、实际 drain 和业务 completion。
  \item[UTF-8: Unicode Transformation Format 8-bit] Unicode 的常见变长字节编码。实践卷保存 offset 时默认按字节定位；如果 UI 要按字符显示，不能把字节下标直接当字符下标。
\end{description}
