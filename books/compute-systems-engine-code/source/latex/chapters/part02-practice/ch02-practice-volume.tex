\chapter{实践卷：Reference Pipeline 工程化}

上一章定义了输入合同、reference 和报告字段。本章把它落成可构建工程，目录是：

\begin{lstlisting}[language=bash]
books/compute-systems-engine-code/labs/reference_pipeline
\end{lstlisting}

代码使用 C++20，单元测试使用 GTest/GMock，性能入口使用 Google Benchmark。这里的重点不是工具名，而是把输入身份、解析诊断、稳定报告和 manifest 发布格式做成可复查工程。

核心文件如下：

\section{二、阶段 0：输入身份先独立成类型}

\code{RecordId} 描述一条记录从哪里来：

\begin{lstlisting}[language=C++]
struct RecordId {
    std::string input_name;
    std::uint64_t input_generation = 0;
    std::uint64_t byte_offset = 0;
    std::uint64_t line_number = 0;
};
\end{lstlisting}

\code{input_name} 是输入名，\code{input_generation} 是同一输入的版本号，\code{byte_offset} 是该行在原始文本里的字节偏移，\code{line_number} 是行号。generation 不是时间戳装饰，它用于区分“同名输入的第几代”。如果后续做 retry、checkpoint 或恢复，旧 generation 的结果不能覆盖新 generation。

\code{InputCase} 把输入身份和文本放在一起。实践时不要直接传裸字符串，因为裸字符串没有来源，坏行诊断也无法说明是哪份输入、哪一代、哪个位置出错。

\section{三、阶段 1：解析合同从一行开始}

reference pipeline 的输入格式是每行 \code{event,value}。合法行有两个字段：事件名非空，value 是整数。坏行进入 \code{ParseDiagnostic}，不能混入 accepted 计数。

手算样例：

\begin{lstlisting}
view,1
click,1
view,2
\end{lstlisting}

这三行的 \code{records_total} 是 3，\code{accepted} 是 3，\code{counts["view"]} 是 2，\code{counts["click"]} 是 1。

坏行样例：

\begin{lstlisting}
missing
click,1,extra
,2
login,abc
\end{lstlisting}

它们分别对应 \code{missing_field}、\code{extra_field}、\code{empty_event} 和 \code{bad_value}。测试 \code{BadLinesReturnStructuredDiagnostics} 不只检查错误个数，还检查 line number、error kind 和 byte offset。这样后续换 parser 时，不能把诊断精度降级。

\section{四、阶段 2：reference 先可信，再考虑快}

\code{run_reference} 是可信但不追求快的实现。它逐行扫描输入，只有合法记录才更新 \code{counts} 和 checksum。checksum 是结果摘要，用来证明同一输入反复运行稳定，也用于后续优化版对照。

\begin{lstlisting}[language=C++,caption={reference result 的语义字段}]
struct ReferenceResult {
    std::string input_name;
    std::uint64_t input_generation = 0;
    std::uint64_t records_total = 0;
    std::uint64_t accepted = 0;
    std::vector<ParseDiagnostic> diagnostics;
    std::map<std::string, std::uint64_t> counts;
    std::uint64_t checksum = 0;
};
\end{lstlisting}

这里使用 \code{std::map} 是为了让输出 key 顺序稳定。稳定顺序不是小事：报告、checksum、manifest 和回归测试都依赖它。等后续进入热路径优化，可以用哈希表或分片结构提高吞吐，但发布报告前仍要回到稳定顺序。

\section{五、阶段 3：报告和 manifest 是发布合同}

\code{format_report} 面向人读，必须包含输入名、记录数、accepted、checksum、每个事件计数和错误统计。\code{ManifestRow} 面向机器读，字段固定为 input name、generation、records total、accepted、rejected 和 checksum。

manifest 是发布清单。它回答“这次处理发布了哪份输入、哪一代、多少记录、多少接受、多少拒绝、摘要是什么”。后续可靠写、恢复和分布式提交都会依赖它；因此 manifest 格式不能随手改。测试 \code{ReportAndManifestAreDeterministic} 固定了 report 子串和 manifest 前缀。

\section{六、阶段 4：测试覆盖正常、坏行、空输入和诊断质量}

当前测试分五类：

\begin{enumerate}
  \item \code{NormalInputCountsAndChecksumAreStable}：正常五行输入，检查 counts 和 checksum 稳定；
  \item \code{BadLinesReturnStructuredDiagnostics}：四类坏行，检查错误类型和位置；
  \item \code{ReportAndManifestAreDeterministic}：检查报告和 manifest 输出稳定；
  \item \code{EmptyInputHasZeroRecordsButStableChecksum}：空输入没有记录但 checksum 稳定；
  \item \code{DiagnosticDetailNamesTheContract}：诊断 detail 必须说清合同，例如“exactly two fields”。
\end{enumerate}

测试不是附属品。reference pipeline 后续每次优化，都必须先跑这些测试，再用优化结果对照 reference。若优化后报告字段变少、错误位置变粗或 checksum 不稳定，即使吞吐更高，也不能进入书中的“正确实现”。

\section{七、阶段 5：benchmark 只测 reference 主路径}

\code{bench/reference_pipeline_bench.cpp} 只测 \code{run_reference} 在不同记录数下的成本。Debug benchmark 只能说明 target 可运行；正式性能报告必须使用 Release 构建，并记录 CPU、编译器、输入规模、负载、是否开启 CPU scaling、重复次数和异常样本处理。

不要把文件读取、随机输入生成、报告打印和 parser 统计混在同一个 benchmark。一次 benchmark 只回答一个问题：在已给定输入文本的情况下，\code{run_reference} 的成本怎样随记录数增长。

\section{八、下一阶段：input split}

下一步扩展是 input split。split 是把大输入切成多个片段并行处理的单位。它不能破坏 \code{RecordId}：每条记录仍要保留 input name、generation、byte offset 和 line number。

正确实现必须处理“split 边界落在行中间”的情况。不能让前一个 split 和后一个 split 同时处理同一行，也不能让两边都跳过这行。测试应构造一个长输入，把切点放在 \code{event,value} 中间，然后证明总记录数、accepted、诊断行号和 checksum 与整文件 reference 一致。
