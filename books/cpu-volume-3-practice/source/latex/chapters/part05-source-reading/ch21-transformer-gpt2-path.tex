\section{一次可审查的 cached-KV 优化}

优化不能从“感觉慢”开始。LCQI 先用 \code{make cpu3-gpt2-benchmark-compare} 证明 full-prefix 每生成一个 token 都重算前缀，而 cached-KV 已经把前缀复用下来；随后再看 cached path 内部。旧实现每处理一个 token、每经过一层 Transformer，都会重新分配 \code{normed}、\code{query}、\code{key}、\code{value}、\code{attention}、\code{projected}、\code{mlp\_fc}、\code{mlp\_out} 和 attention \code{scores}。同时，生成路径只需要下一个 greedy token，却仍然每步构造完整 \code{Gpt2ForwardResult.logits}。这些问题不是算法语义错误，但会在真实 GPT-2 的 decode loop 里形成额外分配、重复 shape validation、重复边界转换和不必要的结果写回。

本轮优化对应三册里的三条原则。第一册要求从热循环和二进制可观察行为出发，所以 \code{linear\_f32}、lm head 扫描和 attention 内层循环先减少不必要的边界转换，而不是改数学公式。第二册要求减少重复分配和内存扰动，所以新增 \code{Gpt2ForwardWorkspace}，把 hidden、QKV、MLP、scores 和 logits 缓冲区的生命周期提升到整段 cached generation。第三册要求按 decoder runtime 的状态组织代码，所以新增 \code{Gpt2CachedGreedyDecoder}，让 KV cache、workspace 和模型验证跟随一次生成会话，而不是散落在每个 token 的临时对象里。

代码上有三处关键边界。第一，旧的 \code{run\_gpt2\_forward\_cached} 仍然保留完整 logits API，测试和源码阅读可以继续用它和 full-prefix logits 做逐项对照。第二，CLI 的 \code{cached\_kv} 生成路径改用 \code{Gpt2CachedGreedyDecoder::step}，只返回 greedy token；需要完整 logits 的测试使用 \code{step\_with\_logits}。第三，KV cache 对外仍保留 checked \code{key}/\code{value} API，内部 attention 在一次地址和 workspace 校验后使用私有 unchecked span 扫描热路径，避免把“不检查”的能力暴露给普通调用者。

正确性证据在 \filepath{tests/lcqi\_tests.cpp}。tiny GPT-2 同一个 prompt 同时覆盖 full-prefix logits、旧 cached logits、优化 decoder logits 和 logits-free greedy token。它要求 predicted token、logits size、逐项 logit 误差、KV cache filled token 数和 greedy generated ids 全部一致。这样优化后的代码不是“跑得快但不知道还对不对”，而是先通过 tiny golden，再进入真实 GPT-2 benchmark。

性能证据不只看一次运行。新增 \filepath{tools/run\_gpt2\_optimization\_ab.py} 会从指定 baseline commit 建临时 worktree，baseline 和当前工作区都通过 \filepath{books/cpu-volume-3-practice} 的 Release CMake 入口构建，再用同一份 GPT-2 F32 safetensors 模型跑多轮 full-prefix 与 cached-KV。最近一次 2 轮 smoke 摘要写到 \filepath{books/cpu-volume-3/reports/lcqi-gpt2-optimization-ab-summary.md}，原始报告由脚本生成到 \filepath{books/cpu-volume-3/results/lcqi-gpt2-optimization-ab.txt}：cached-KV 生成阶段中位数从 baseline \code{847.353 ms} 到 current \code{764.604 ms}，生成吞吐从 \code{4.85652 token/s} 到 \code{5.23178 token/s}；full-prefix 中位数仍在 \code{1.8 s} 左右波动，说明这轮优化主要改善 cached generation 的常数成本，而不是解决 full-prefix 的算法重算。

这个结论必须保守阅读。真实端到端 GPT-2 benchmark 受调度、频率、温度和共享机器状态影响明显，单次结果可能反向波动。因此报告里必须同时看 \code{median/min/max}、生成文本是否一致、baseline 和 current 是否同一构建入口，以及当前工作区是否 dirty。更大的差距仍然不在这轮小优化里：LCQI 还没有 GGUF 量化导入、packed/quantized lm head、高性能 matmul、线程并行、分页 KV cache 和成熟调度。也就是说，本轮优化把 reference path 从“每步制造大量临时对象”推进到“cached 会话复用工作区”，但还没有把它变成 llama.cpp 级推理引擎。
