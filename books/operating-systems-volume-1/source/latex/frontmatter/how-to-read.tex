\chapter*{怎样阅读本册}
\addcontentsline{toc}{chapter}{怎样阅读本册}

本册必须按顺序阅读。第一部分从单片机裸机、超级循环、中断、定时器、启动和内核初始化开始，解释 OS 的入口和抢占能力怎样长出来；第二部分进入任务、线程、调度器深入、同步、内核锁和死锁；第三部分处理特权级、系统调用、\code{exec}、ELF 装载、虚拟内存、\code{mmap}、page cache 和内核分配器；第四部分把设备驱动、DMA、块层、I/O 调度、文件、page cache、文件系统、socket 入口和持久化连接起来；最后把安全隔离、panic、错误恢复、观测和验收方法收束成工程报告能力。

学习每章时都要落到四个问题：

\begin{enumerate}
\tightlist
\item 状态在哪里：进程表、runqueue、锁等待队列、页表、VMA、fd 表、socket、inode、page cache、日志还是设备队列。
\item 权限在哪里：CPU 特权级、页表权限、文件权限、capability、namespace 还是对象引用。
\item 等待在哪里：runnable 等 CPU、blocked 等事件、mutex 等持有者、page fault 等页面、socket 等缓冲、I/O 等 completion。
\item 提交在哪里：\code{exec} 切换地址空间、状态更新完成、写入 page cache、块设备 flush、目录持久化还是 manifest 生效。
\end{enumerate}

本册的“实践”不是把理论卷变成代码卷，而是要求每个机制能落成纸上状态机、观察命令、实验矩阵或验收报告。若一节讲 fd，就要画出 fd table、open file description、inode 和 page cache 的关系；若一节讲调度，就要写出 runnable、running、blocked、zombie 和 reaped 的状态转换；若一节讲锁，就要画出锁顺序图和等待队列；若一节讲 \code{exec}，就要标出失败回滚边界和提交点；若一节讲持久化，就要写出每个崩溃点后恢复程序应该相信什么。

阅读算法小节时，不要只看伪代码能不能运行。要追问四件事：数据结构是什么，复杂度是多少，维护了哪些不变量，失败路径是否会留下半提交状态。一个教学算法可以比真实内核小，但不能把真实问题讲错。

\begin{warningbox}
不要把工具输出当成结论。工具只是证据来源；结论必须说明哪条 OS 契约被验证，哪条仍然只是合理假设。
\end{warningbox}
