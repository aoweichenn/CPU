\chapter{操作系统地图与契约}

\section{原理}

操作系统的第一性问题不是“提供哪些 API”，而是“怎样让多个不可信、会失败、会竞争的程序共享同一台机器”。硬件只提供有限的 CPU 核、寄存器、物理内存、cache、TLB、页表、中断、总线、设备寄存器、DMA 和特权级。应用希望看到的是进程、线程、虚拟地址、文件、socket、权限、时间和错误码。操作系统处在中间，把硬件事实翻译成可编程契约。

一个契约至少包含四层含义。第一，状态由谁持有，例如进程表、页表、文件描述符表、inode、设备队列。第二，谁可以修改状态，例如系统调用、异常处理、中断处理、内核线程或回收器。第三，失败怎样表达，例如返回错误码、发送信号、杀死进程、回滚事务或留下可恢复日志。第四，外部观察者能看到什么，例如命令输出、trace、统计计数、日志和测试结果。

操作系统同时解决六类问题。抽象让程序不必直接操作物理设备；隔离让一个程序不能随便破坏另一个程序；并发让多个任务共享 CPU 和 I/O；保护让用户态无法绕过页表、特权级和设备边界；持久化让数据在进程退出或机器崩溃后仍有解释；可观测性让系统在出错时有证据，而不是只留下“偶现问题”。

这五类问题不能分开学习。进程调度要理解线程睡眠和唤醒，睡眠和唤醒又依赖同步原语；虚拟内存隔离要理解页表权限，权限错误又通过异常进入内核；文件写入要理解 page cache，page cache 又属于内存管理；可靠提交要理解 \code{fsync}、目录项和崩溃恢复。

本册把操作系统看成一组状态机的组合。进程有生命周期，线程有运行状态，页表有映射状态，fd 有引用状态，文件页有 clean、dirty、writeback 状态，设备请求有 submitted、completed 状态，持久化输出有 prepared、committed 状态。能画出状态机，就能开始解释 OS；只会背 API 名字，无法判断边界。

\begin{keyidea}
本册所有机制都用同一个问题模板分析：谁拥有状态，谁能改变状态，改变时如何同步，失败时如何回滚，外部如何证明它真的发生。
\end{keyidea}

从工程角度看，操作系统的每个功能都同时有“用户可见语义”和“内核维护成本”。例如进程给用户一个独立执行环境，但内核要维护地址空间、fd 表、信号、父子关系和退出状态；虚拟内存给用户一个连续地址幻觉，但内核要维护 VMA、页表、TLB 失效和回收策略；文件给用户一个字节流或目录树，但内核要维护 inode、page cache、块分配、权限和崩溃恢复。只讲语义会太抽象，只讲实现会迷失在细节里，二者必须一起讲。

\section{OS 对象不是名词，而是硬件状态的管理账}

进程、线程、地址空间、文件描述符、inode、socket、设备请求这些词容易被背成名词。更准确地说，它们是内核为了管理硬件状态而建立的账本。CPU 只能执行当前寄存器里的上下文，OS 于是建立 task；MMU 只能按当前页表翻译地址，OS 于是建立 mm 和 VMA；设备只认识描述符和 doorbell，OS 于是建立 request、bio、skb、completion queue；存储设备只完成块写，OS 于是建立 inode、journal 和 fsync 语义。

\begin{longtable}{p{0.2\linewidth}p{0.32\linewidth}p{0.36\linewidth}}
\toprule
OS 对象 & 管理的硬件事实 & 为什么不能只交给应用 \\
\midrule
task/thread & RIP、RSP、寄存器、内核栈、调度状态 & 应用不能自己抢占死循环，也不能保存别人的上下文 \\
address space & CR3、PTE、TLB、VMA 意图 & 应用不能自己修改页表来隔离自己和别人 \\
fd table & 进程内整数句柄到内核对象引用 & 应用需要受控引用资源，而不是直接拿内核指针 \\
file/inode & page cache、块映射、权限、引用计数 & 应用不能直接维护全局目录树和崩溃恢复 \\
socket & NIC 队列、协议状态、收发缓冲区 & 应用不能直接管理网卡 DMA 和 TCP 状态机 \\
request/bio/skb & DMA buffer、队列位置、completion & 设备异步完成，需要内核追踪所有权和失败 \\
\bottomrule
\end{longtable}

这张表背后的规则是：只要某个硬件状态会被多个程序共享，或者失败后需要恢复，就不能让应用直接操作原始硬件。内核必须建立一个对象，给它身份、权限、生命周期、等待队列和可观测证据。对象不是为了“面向对象”而存在，而是为了让硬件状态可命名、可授权、可回收、可诊断。

\section{容易混淆的三层语义}

OS 教材里很多概念看起来相同，是因为它们在不同层次上都叫“完成”“可读”“地址”“上下文”。读源码时必须把用户 API 语义、内核对象语义和硬件事件语义分开。

\begin{longtable}{p{0.18\linewidth}p{0.26\linewidth}p{0.26\linewidth}p{0.18\linewidth}}
\toprule
说法 & 用户 API 层 & 内核对象层 & 硬件层 \\
\midrule
地址 & 指针值 & VMA/PTE 和权限 & TLB、物理地址、MMIO/IOVA \\
可读 & \code{read} 不会立刻阻塞 & buffer/队列中有数据或状态满足 & 中断、completion、DMA 写入 \\
写完成 & 系统调用返回字节数 & page cache/socket buffer/request 状态更新 & 设备队列接受、介质 flush 可能还没发生 \\
上下文 & 用户调用链和线程身份 & task、trap frame、switch context & RIP/RSP/RFLAGS、CR3、PCID \\
错误 & errno 或信号 & 对象状态拒绝、权限失败、生命周期结束 & fault、IRQ error、DMA/IOMMU fault \\
\bottomrule
\end{longtable}

例如“可读”在 pipe、socket、文件和设备上含义不同。普通文件几乎总是可读，因为内核可以从 page cache 或磁盘推进；socket 可读表示接收队列有数据或连接关闭/错误；设备 fd 可读可能表示驱动队列里有 completion。用户只看到 \code{poll} 返回，但内核必须把对象状态和硬件事件翻译成统一接口。

再看“写完成”。写入 pipe 只表示数据进入内核 pipe buffer；写入 TCP socket 只表示数据进入发送路径，不等于对端应用收到；写入普通文件只表示 page cache 被修改，不等于稳定落盘；写入带 \code{O\_DIRECT} 的块设备也可能只表示设备接受了请求，仍要理解 flush/FUA 和错误返回。把这些层混在一起，是很多系统设计错误的源头。

\section{四条完整路径：从硬件事件到内核决策}

下面用四条路径把本章的地图落到具体步骤。它们不是后续章节的替代品，而是读完整本书时不断回看的骨架。

\subsection{路径一：\code{write(fd, buf, len)} 为什么不是一次普通函数调用}

应用看见的是一个 C 库函数，硬件看见的是一次受控特权级转换，内核看见的是多个对象状态更新。

\begin{lstlisting}
user:
  rax = SYS_write
  rdi = fd
  rsi = user_buf
  rdx = len
  syscall

kernel entry:
  save user RIP/RSP/RFLAGS and registers
  switch to trusted kernel stack
  locate current task and fd table
  validate fd refers to writable file/socket/pipe
  validate user buffer range and copy or pin pages
  update target object state
  return byte count or -errno in rax
\end{lstlisting}

\begin{longtable}{p{0.2\linewidth}p{0.36\linewidth}p{0.32\linewidth}}
\toprule
阶段 & 关键检查 & 失败表达 \\
\midrule
系统调用入口 & syscall number 是否存在，当前上下文是否允许 & \code{-ENOSYS}、审计拒绝、seccomp 信号 \\
fd 查找 & fd 是否打开，是否有写权限 & \code{-EBADF}、\code{-EPERM} \\
用户 buffer & 地址是否 canonical、VMA 是否允许读、复制中是否 fault & \code{-EFAULT} 或部分写入 \\
目标对象 & pipe/socket/file 是否可写、是否阻塞 & \code{-EAGAIN}、睡眠、信号中断 \\
提交语义 & 数据进入哪里：page cache、socket buffer、设备队列 & 返回值不一定表示落盘或发出网络包 \\
\bottomrule
\end{longtable}

这条路径说明了 OS 契约的层次。系统调用返回成功，只说明内核接受并完成了该接口定义下的一部分工作。写普通文件可能只是写入 page cache；写 socket 可能只是进入发送缓冲；写终端可能经过驱动队列和中断。若业务需要“崩溃后仍存在”，还要继续追 \code{fsync} 和存储 flush；若业务需要“对端已收到”，还要追协议确认。

\subsection{路径二：一次 page fault 如何在硬件和内核之间分工}

用户态一条 \asmcode{mov rax, qword ptr [rdi]} 可能触发 page fault。硬件做的是检查和转交，内核做的是解释和修复。

\begin{lstlisting}
CPU:
  compute effective address from rdi
  check canonical form
  lookup TLB or walk page table
  if missing or permission denied:
    save fault RIP and error code
    record fault address
    enter page fault handler

kernel:
  find VMA covering fault address
  compare access type with VMA permissions
  inspect PTE state
  allocate zero page, read file page, perform COW, or reject
  install PTE and invalidate stale TLB when needed
  return to faulting RIP or deliver signal
\end{lstlisting}

\begin{longtable}{p{0.22\linewidth}p{0.34\linewidth}p{0.32\linewidth}}
\toprule
硬件看到 & 内核进一步区分 & 结果 \\
\midrule
present=0 & 地址属于匿名 VMA & 分配零页，minor fault \\
present=0 & 地址属于文件映射 & page cache 命中或发起磁盘 I/O \\
write denied & PTE 标记 COW & 复制页并恢复写权限 \\
execute denied & NX 或 VMA 不可执行 & 发送信号，安全拒绝 \\
user bit denied & 用户访问 supervisor 页 & 发送信号或记录攻击/bug \\
\bottomrule
\end{longtable}

缺页机制的关键不是“慢”，而是“可恢复边界”。如果 CPU 不能保存 faulting RIP，COW store 就无法重试；如果内核不能判断 VMA 意图，present=0 就无法区分合法懒分配和非法访问；如果 TLB 失效不正确，PTE 更新后旧权限仍可能生效。

\subsection{路径三：定时器中断如何变成抢占}

用户程序可能永远不主动让出 CPU。定时器中断给内核一个周期性入口，让它能统计时间、处理超时并决定是否切换任务。

\begin{lstlisting}
local APIC timer fires:
  CPU takes interrupt at instruction boundary
  entry saves interrupted RIP/RFLAGS/RSP
  timer handler increments scheduler clock
  expired timers wake sleeping tasks
  current task runtime is charged
  if time slice or policy says preempt:
    set need_resched
  return path calls scheduler before going back to user
  scheduler saves current kernel context and restores next
\end{lstlisting}

这里有两个容易混淆的点。第一，中断 handler 通常不在任意深处直接把用户栈改成另一个任务；它设置调度标志，在安全返回点进入调度。第二，切换任务不是只改一个 current 指针，还可能涉及内核栈、callee-saved 寄存器、地址空间、FPU/SIMD 状态、CPU 本地数据和 TLB 语义。

\begin{longtable}{p{0.22\linewidth}p{0.34\linewidth}p{0.32\linewidth}}
\toprule
动作 & 需要保存什么 & 保存错误的现象 \\
\midrule
中断进入 & 用户 RIP/RSP/RFLAGS 和 scratch 状态 & 返回后用户程序寄存器随机坏 \\
调度统计 & 当前任务运行时间、vruntime、状态 & CPU 时间不公平或饥饿 \\
切换内核上下文 & 内核栈指针、callee-saved 寄存器 & 任务从错误调用链恢复 \\
切换地址空间 & CR3/PCID、必要 TLB 处理 & 任务访问到别人的内存 \\
\bottomrule
\end{longtable}

\subsection{路径四：设备完成如何唤醒一个睡眠任务}

设备 I/O 是异步的。提交请求后，任务可能睡眠；设备通过 DMA 和中断报告完成；内核再把 completion 变成任务可见结果。

\begin{lstlisting}
submit I/O:
  allocate request object
  map buffer for DMA
  fill descriptor ring
  publish descriptor with barrier
  write MMIO doorbell
  if synchronous API:
    put task on wait queue and schedule away

device:
  reads descriptor by DMA
  transfers data
  writes completion entry
  raises MSI-X interrupt

interrupt/bottom half:
  read completion queue
  unmap DMA and mark request done
  store status and byte count
  wake tasks waiting on request
\end{lstlisting}

这条路径解释了为什么 I/O 子系统一定有 request identity。中断到来时，CPU 不知道“哪个用户函数在等它”，只看到某个设备队列有 completion。内核必须通过队列位置、tag、request 指针或 completion entry 找回原请求，撤销 DMA 授权，更新上层对象，再唤醒等待者。没有这层身份映射，异步 I/O 无法可靠对应到发起者。

\section{从组成原理到 OS 抽象}

组成原理解释机器能做什么，操作系统解释这些能力如何被组织成可共享、可隔离、可恢复的服务。把二者分开学时，读者很容易知道“CPU 有特权级”，却不知道系统调用为什么必须切栈；知道“页表做地址翻译”，却不知道 \code{fork} 为什么能 COW；知道“DMA 很快”，却不知道驱动为什么要维护 descriptor ring 和 completion。

\begin{longtable}{p{0.22\linewidth}p{0.34\linewidth}p{0.32\linewidth}}
\toprule
组成原理事实 & OS 抽象 & 关键问题 \\
\midrule
RIP、寄存器和栈可保存 & 线程上下文 & 切走后如何从同一位置恢复 \\
特权级限制敏感指令 & 用户态/内核态 & 用户如何安全请求内核服务 \\
异常入口保存 trap frame & page fault、signal、debug & 错误能否修复并返回原指令 \\
定时器可强制中断 & 抢占式调度 & 不合作任务如何被收回 CPU \\
MMU 检查页表权限 & 地址空间 & 进程如何互相隔离又能共享 \\
cache/TLB 需要维护 & 性能和一致性策略 & 为什么迁移、锁和 unmap 有成本 \\
MMIO 控制设备 & 驱动模型 & 寄存器副作用和顺序如何管理 \\
DMA 异步访问内存 & 高性能 I/O & buffer 何时归 CPU，何时归设备 \\
\bottomrule
\end{longtable}

本册后文所有章节都按这张映射阅读。看到“进程”，不要只想 pid，要想到一组保存的 CPU 上下文、地址空间和内核对象。看到“文件”，不要只想路径，要想到 fd 表、file、inode、page cache、块层和设备队列。看到“权限”，不要只想用户 ID，要想到 CPU 特权级、页表 U/S 位、capability、namespace、LSM 和对象引用。

\section{同一个词在 POSIX、内核和硬件中的落点}

读手册时常见的是 POSIX 词，例如 process、file descriptor、signal、pipe、socket。读内核时看到的是 task、file、inode、skb、wait queue。读硬件时又变成 RIP、CR3、IRQ、DMA、MMIO。三套词不是互相替代，而是同一条路径的不同层。

\begin{longtable}{p{0.18\linewidth}p{0.24\linewidth}p{0.26\linewidth}p{0.22\linewidth}}
\toprule
用户语义 & 内核对象 & 硬件支点 & 关键边界 \\
\midrule
process & task、mm、files、cred & RIP/RSP、CR3、PCID & 身份、地址空间、权限分离 \\
thread & task、kernel stack、thread info & 寄存器、RSP、FPU state & 可调度上下文，不复制全部资源 \\
fd & fd table、struct file & 设备队列、page cache、DMA & 整数句柄不是文件本体 \\
signal & pending signal、signal frame & trap frame、用户栈、\code{iretq} & 异步交付但必须恢复用户现场 \\
socket & socket、skbuff、wait queue & NIC ring、MSI-X、DMA & 协议状态和设备完成分离 \\
timer & hrtimer、timer wheel & local APIC timer、TSC deadline & 未来事件需要硬件唤醒入口 \\
\bottomrule
\end{longtable}

例如 fd 是一个用户态整数，但内核要通过当前进程的 fd table 找到 \code{struct file}，再通过 file operations 调到 pipe/socket/inode/device 的具体实现。若底层是网卡，最终还会走 skb、驱动队列、DMA 和 MSI-X completion。把 fd 当成“文件本体”，就解释不了 \code{dup}、\code{fork} 后共享 offset、socket readiness、设备拔出后旧 fd 返回错误这些现象。

\section{为什么从单片机开始}

单片机不是因为简单才低级，而是因为它暴露了 OS 的原始需求。一个裸机程序拥有全部内存和设备，所有代码同权，主循环靠自觉返回，中断直接修改全局状态。它让我们能看见每个高级机制出现前的失败。

\begin{longtable}{p{0.24\linewidth}p{0.32\linewidth}p{0.32\linewidth}}
\toprule
裸机阶段的问题 & 直接补丁 & 通用 OS 机制 \\
\midrule
函数忙等设备 & 改成状态机或中断 & 阻塞、等待队列、completion \\
任务靠自觉返回 & 任务表和显式 yield & 线程和协作式调度 \\
任务可能不合作 & 加硬件定时器 & 抢占和时间片 \\
全局变量被 ISR 打断 & 关中断临界区 & 锁、原子操作、内存屏障 \\
所有代码同权 & 约定模块边界 & CPU 特权级和系统调用 \\
所有内存同权 & 手工内存布局 & MMU、地址空间、页表权限 \\
CPU 搬每个字节 & DMA 控制器 & 异步 I/O 和驱动队列 \\
\bottomrule
\end{longtable}

这也是本册合并组成原理与 OS 的原因：如果不知道硬件能否强制中断、能否区分用户/内核、能否检查页表权限，就无法判断一个 OS 设计到底是“规范约定”还是“硬件保证”。

\section{实践}

阅读任何 OS 现象时，先写一张“状态和边界表”。

\begin{longtable}{p{0.18\linewidth}p{0.74\linewidth}}
\toprule
问题 & 要填写的内容 \\
\midrule
状态对象 & 进程、线程、页、fd、inode、socket、设备请求、日志记录 \\
所有者 & 用户态对象、内核对象、硬件寄存器、设备控制器 \\
入口 & 系统调用、异常、中断、内核线程、后台回收 \\
权限 & 特权级、页表权限、文件权限、capability、namespace、引用计数 \\
等待 & runqueue、futex、page fault、I/O completion、锁、条件变量 \\
提交 & 状态更新、引用释放、写入 page cache、落盘、manifest 生效 \\
证据 & 返回值、errno、trace、计数器、日志、测试断言 \\
\bottomrule
\end{longtable}

例如一次 \code{write(fd, buf, len)} 可以这样拆：用户态只有一个整数 fd 和一段用户 buffer；内核要先从进程 fd 表找到 open file description，再检查写权限，再复制用户 buffer，再把页标记为 dirty，最后返回写入字节数或错误。这个返回值通常不等于数据已经安全落盘。要证明持久化，还需要 \code{fsync}、目录同步或更高层 manifest。

再看一次 page fault。用户态执行 load/store，硬件发现页表项不存在或权限不符，CPU 进入异常入口，内核根据 VMA 判断这是合法缺页、权限错误还是非法地址。合法缺页可能分配物理页或读取文件页；权限错误可能发送信号；非法访问可能终止进程。这里的核心不是“缺页很慢”，而是虚拟地址访问被拆成硬件检查和内核决策。

\section{契约维度}

一个 OS 机制要讲完整，至少要覆盖五个维度。

\begin{longtable}{p{0.18\linewidth}p{0.34\linewidth}p{0.36\linewidth}}
\toprule
维度 & 追问 & 例子 \\

这五个维度能帮助读者判断一个解释是否完整。比如“socket 可读”只覆盖了等待维度，还要问：哪个 fd 引用它，谁有权限读，缓冲区里数据生命周期如何，读取后 offset 或队列怎样变化，关闭时如何唤醒等待者。

\section{层次与反向依赖}

操作系统经常呈现层次结构：硬件在底层，内核抽象在中间，用户 API 在上层。但真实问题会反向穿透层次。一个 \code{write} 系统调用看似属于文件 API，却可能等待内存回收、块层队列、驱动中断、设备 flush 和调度器唤醒。一个 page fault 看似属于内存管理，却可能读取文件页，触发块设备 I/O。一个安全拒绝看似属于权限系统，却依赖路径解析、namespace、凭据和 LSM。

\begin{lstlisting}
user write()
  -> fd table
  -> file object
  -> page cache
  -> writeback
  -> block layer
  -> driver DMA
  -> interrupt completion
  -> wakeup
\end{lstlisting}

因此阅读本册时，不要把章节理解成互不相干的模块。章节只是学习顺序，真实系统是交叉依赖的状态机网络。

\section{演进路线}

本册后续章节按一条明确路线展开。

\begin{enumerate}
\tightlist
\item 数字电路和 CPU：bit、寄存器、时钟、状态机、取指译码执行。
\item 总线和设备：地址译码、MMIO、设备寄存器、中断和 DMA。
\item 保护硬件：特权级、异常入口、MMU、页表和定时器。
\item 单片机裸机程序：一个主循环直接拥有整台机器。
\item 事件和状态机：阻塞函数被拆成可返回的 step。
\item 中断：外设主动打断 CPU，事件不再只靠轮询。
\item 定时器：系统获得时间基准和抢占入口。
\item 任务上下文：程序计数器、寄存器和栈成为可保存对象。
\item 调度器：多个 runnable 任务按策略共享 CPU。
\item 同步：共享状态需要互斥、等待谓词和唤醒协议。
\item 内存保护：地址空间、页表和权限阻止任务互相破坏。
\item 系统调用：用户程序通过受控入口请求内核操作资源。
\item 文件和 I/O：设备被包装成对象、句柄、队列和 completion。
\item 持久化：数据从“写过”推进到“崩溃后仍可解释”。
\end{enumerate}

这条路线的意义是把 OS 特性从原因中长出来。不是先背“进程、线程、虚拟内存、文件系统”，而是从每个阶段遇到的失败推出下一层机制。

\toprule
失败 & 需要的机制 & 新增代价 \\
\midrule
固定电路不通用 & 存储程序 CPU & 指令编码、取指译码、异常语义 \\
单核状态无法被安全共享 & 多核一致性和原子操作 & cache line 迁移、内存序、锁竞争 \\

\section{测试}

OS 理论卷的测试不是只跑程序，而是验证契约。一个合格测试要覆盖正常路径、错误路径和边界路径，并且能指出是哪条不变量断了。

\begin{rubricbox}
\begin{itemize}
\tightlist
\item 能把一次系统调用拆成用户态入口、内核对象、权限检查、状态更新和返回值。
\item 能区分“函数返回成功”“内核接受请求”“设备完成”“业务提交生效”。
\item 能解释至少一个等待现象：等 CPU、等锁、等页面、等 I/O、等子进程。
\item 能说明至少一个隔离边界：进程地址空间、文件权限、用户/内核态、namespace。
\item 能为任意章节机制写出状态、入口、失败和证据四列。
\end{itemize}
\end{rubricbox}

反例同样重要。若读者把 fd 当成文件本体，把 \code{write} 成功当成落盘，把线程 ready 当成 running，把 \code{malloc} 成功当成物理页已分配，把 \code{notify\_one} 当成消息投递，就说明 OS 契约还没有建立。
