\chapter{编译器 IR、优化 pass 与汇编证据链}

\section{为什么还要专门讲优化器}

前面已经把源码、目标文件、ELF、加载和运行时启动连成一条链。现在还缺中间最容易被误解的一层：优化器。很多工程事故并不是汇编语法读错，而是把 C++ 源码当成 CPU 会照字执行的脚本。优化器看到的不是你屏幕上的缩进和变量名，而是经过语义分析后的中间表示；它按语言规则、别名信息、控制流、数据流和目标机器代价模型改写程序。若源码里包含未定义行为、对象生命周期错误、别名假设错误或不可观察的计算，优化器生成的机器码可能和初学者直觉完全不同。

本章的目标不是让读者背 LLVM 或 GCC 内部所有 pass 名称，而是建立一条可复查的证据链：

\begin{lstlisting}[numbers=none,caption={优化器证据链}]
C++ source
  -> preprocessed translation unit
  -> AST or frontend semantic dump
  -> IR with types, loads, stores, calls and branches
  -> SSA/value graph and control-flow graph
  -> optimization pass before/after
  -> target assembly
  -> object code and debug/unwind side effects
\end{lstlisting}

只要这条链路清楚，读者就能解释三个常见问题：为什么某段代码在 \code{-O0} 和 \code{-O2} 下行为不同；为什么 benchmark 循环消失；为什么看起来“只是改了类型转换”却导致向量化或别名结果变化。

先把本章几个名词压成一个小例子。假设源码是：

\begin{lstlisting}[language=C++,caption={贯穿优化器章节的最小分支}]
extern "C" int choose(int x, int y, bool use_x)
{
    int r = 0;
    if (use_x) {
        r = x + 1;
    } else {
        r = y + 1;
    }
    return r;
}
\end{lstlisting}

AST 适合回答“这个 \code{if} 的 then/else 结构和表达式类型是什么”；IR 适合回答“控制流被拆成哪些基本块，值从哪里流到哪里”；SSA 适合回答“每个虚拟值在哪里被定义，合流处怎样选择值”；pass 适合回答“哪些值能传播、哪些分支能折叠、哪些内存访问能删除”；汇编适合回答“最后目标机器执行哪些指令”。这些名字不是并列术语清单，而是一条从源码语义到机器证据的推理路径。

\section{从源码到 AST：语言语义先于优化}

编译器前端的第一项工作是把文本变成有语义的树。语法树不是性能优化结果，而是语言规则的载体。名字查找、重载解析、模板实例化、常量表达式、对象生命周期、访问控制和异常规范，都在这一层确定。优化器后面能做什么，取决于前端已经证明了什么。

考虑下面的代码：

\begin{lstlisting}[language=C++,caption={看似普通的循环}]
int sum_positive(const int* xs, int n)
{
    int s = 0;
    for (int i = 0; i < n; ++i) {
        if (xs[i] > 0) {
            s += xs[i];
        }
    }
    return s;
}
\end{lstlisting}

先不要急着问它会不会向量化。前端首先要把源码里的词和结构变成语言事实：

\begin{enumerate}
  \item \code{xs} 的类型是 \code{const int*}，所以 \code{xs[i]} 是一次按 \code{sizeof(int)} 缩放的对象访问。
  \item \code{n} 和 \code{i} 都是 \code{int}，所以 \code{i < n} 是有符号比较，不是无符号比较。
  \item \code{xs[i] > 0} 需要先读取 \code{xs[i]}，再按有符号整数解释这个值。
  \item \code{s += xs[i]} 可能发生有符号溢出，而有符号溢出在标准 C++ 中是未定义行为。
  \item \code{xs} 是否为空、\code{n} 是否为负、数组是否足够长，不是这段函数签名自动保证的事实。
\end{enumerate}

最后两点尤其关键。优化器不是“假装机器整数会回绕”，而是可以利用“正确程序不会发生 signed overflow”这个前提做改写。它也不能凭空知道 \code{xs} 指向多少个元素；如果调用者传入非法范围，源码层已经越过语言边界。高质量优化报告必须先写清这些语义前提，否则后面的 IR 和汇编分析没有根。

一个最小 AST 观察可以这样做：

\begin{lstlisting}[language=bash,caption={保存 sum\_positive 的 AST 证据}]
clang++ -std=c++20 -O2 -Xclang -ast-dump -fsyntax-only sum.cpp \
  > reports/sum.ast.txt
rg -n "FunctionDecl|ParmVarDecl|IfStmt|ArraySubscriptExpr|BinaryOperator" \
  reports/sum.ast.txt
\end{lstlisting}

这份证据能支持的结论是：“前端把 \code{xs[i]} 理解为数组下标表达式，把 \code{i < n} 理解为有符号比较，并把 \code{s += xs[i]} 放在 then 分支里。”它不能支持“循环一定存在于最终机器码里”，也不能支持“这个循环一定慢”。那些要继续看 IR、汇编和测量。

因此，AST dump 的价值不只是看树形结构，而是确认编译器理解的类型和语义是否与你一致。高质量报告应保存：

\begin{lstlisting}[numbers=none,caption={AST 证据字段}]
frontend:
  compiler_and_version
  language_standard
  target_triple
  source_file
  preprocessed_file_hash
  ast_dump_path
  warnings_enabled
  sanitizer_configuration
\end{lstlisting}

如果报告里只有最终汇编，没有前端语义，就很难判断某个优化是目标机器选择，还是语言层已经允许了。

\section{IR 是优化器的工作面}

IR 可以粗略理解为“优化器容易分析和改写的程序表示”。真实编译器的 IR 有很多种：Clang/LLVM 常用 LLVM IR，GCC 有 GIMPLE 和 RTL，后端还会有机器 IR。不同 IR 的语法不同，但工程问题相同：它们把源码里的表达式拆成更显式的 load、store、branch、call、phi、metadata 和控制流。

仍然用 \code{choose} 来建立从源码到 IR 的直觉。源码里只有一个 \code{if} 和一个局部变量 \code{r}；IR 更关心的是控制流怎样分叉、每条路径产生哪个值、合流后返回哪个值。因此它会把程序拆成几个 \term{基本块}{basic block}。基本块是一段中间没有跳入跳出的顺序指令，通常以 \code{br} 或 \code{ret} 结尾。这个概念很重要：优化器分析循环、分支和支配关系时，操作对象不是“源码第几行”，而是基本块和块之间的边。

下面用伪 LLVM IR 表达一个最小分支：

\begin{lstlisting}[numbers=none,caption={分支在 IR 中变成基本块和 phi}]
define i32 @select_add(i32 %x, i32 %y, i1 %cond) {
entry:
  br i1 %cond, label %then, label %else

then:
  %a = add nsw i32 %x, 1
  br label %merge

else:
  %b = add nsw i32 %y, 1
  br label %merge

merge:
  %r = phi i32 [ %a, %then ], [ %b, %else ]
  ret i32 %r
}
\end{lstlisting}

\code{entry} 块根据 \code{\%cond} 跳到 \code{then} 或 \code{else}。\code{then} 块定义 \code{\%a}，\code{else} 块定义 \code{\%b}。两个块最后都跳到 \code{merge}。问题来了：\code{merge} 块里返回的值到底是 \code{\%a} 还是 \code{\%b}？源码里用同一个局部变量 \code{r} 表达“不同路径给它赋不同值”，但 SSA 形式要求每个虚拟值只定义一次，不能反复给同一个 \code{r} 赋值。于是 IR 用 \code{phi} 节点表达合流选择：

\begin{lstlisting}[numbers=none]
%r = phi i32 [ %a, %then ], [ %b, %else ]
\end{lstlisting}

这行的意思是：如果控制流从 \code{then} 来，\code{\%r} 取 \code{\%a}；如果从 \code{else} 来，\code{\%r} 取 \code{\%b}。\code{phi} 不是 CPU 指令，它不会在最终汇编里原样出现。后端会根据控制流、寄存器分配和目标机器，把它实现成寄存器选择、移动、条件移动或分支路径上的赋值。

SSA 的核心收益是让“值从哪里来”非常明确。常量传播可以沿定义使用链传播；死代码删除可以看某个虚拟值有没有被使用；循环优化可以识别循环中每次迭代变化的归纳变量；公共子表达式消除可以判断两个计算是否真的相同。读 IR 时要分清：IR 是编译器的证明和改写平面，不是机器最终执行平面。

\begin{warningbox}
不要把 \code{\%a}、\code{\%b}、\code{\%r} 当成真实寄存器。它们是 IR 虚拟值。真实 x86-64 寄存器要等后端寄存器分配后才确定。把 IR 虚拟值和 \register{rax}、\register{rdi} 这类机器寄存器混在一起，会直接读错优化结果。
\end{warningbox}

\subsection{IR 里最先要认的五类东西}

第一次看 IR，不要试图理解所有属性和 metadata。先找五类东西：

\begin{center}
\small
\begin{tabular}{p{0.18\linewidth}p{0.34\linewidth}p{0.34\linewidth}}
\toprule
IR 元素 & 它回答什么 & 不能直接说明什么 \\
\midrule
\code{load} & 从哪个指针读值，读出的虚拟值给谁用 & 数据是否命中 cache \\
\code{store} & 把哪个值写到哪个指针位置 & 写入是否立刻到达内存条 \\
\code{call} & 调用了哪个函数或间接目标 & 函数一定没有被后续内联 \\
\code{br} & 控制流怎样分叉或跳回循环 & 分支预测是否成功 \\
\code{phi} & 合流处的值来自哪个前驱块 & CPU 有一条 phi 指令 \\
\bottomrule
\end{tabular}
\end{center}

这五类元素足够解释很多初学者困惑：源码变量为什么消失、循环为什么被删、分支为什么折叠、load 为什么被提前、函数调用为什么不见。等这些能读稳，再去看 alias metadata、range metadata、vectorization remark 和 target-specific IR。

\subsection{load/store 比变量名更重要}

源码中的局部变量名在优化后可能消失。IR 更关心值流和内存流。若一个局部变量从不被取地址，前端可能先把它放在栈槽，随后 \code{mem2reg} 把它提升成 SSA 值。这样汇编里可能根本没有这个变量的内存地址。

\begin{lstlisting}[numbers=none,caption={mem2reg 前后的思路}]
before:
  %s.addr = alloca i32
  store i32 0, ptr %s.addr
  %old = load i32, ptr %s.addr
  %new = add i32 %old, %x
  store i32 %new, ptr %s.addr

after:
  %s0 = 0
  %s1 = add i32 %s0, %x
\end{lstlisting}

这解释了为什么调试器会显示 \code{optimized out}。变量不是被“弄丢了”，而是源码变量和机器状态之间不再一一对应。优化构建里的调试信息只能尽力把源层变量映射回寄存器、栈槽或表达式；当值被合并、删除、重排或只存在于某些路径上时，调试器无法凭空恢复它。

从零推导一下 \code{mem2reg}。在未优化 IR 里，前端常把局部变量 \code{s} 做成一个栈槽：先 \code{alloca} 一块空间，再用 \code{store} 写入，用 \code{load} 读出。这种形式容易保留源码变量位置，但对优化器不友好，因为每次读写都像是内存访问。若优化器能证明 \code{s} 从未被取地址、没有别名指针能碰到这个栈槽、生命周期只在当前函数内，它就可以把 \code{s} 从“内存里的槽”提升成“SSA 虚拟值”。提升后，许多 load/store 消失，汇编里也可能没有 \code{s} 的栈地址。

这不是为了让调试困难，而是因为机器执行不需要那个栈槽。性能报告如果看到 \code{-O0} 下有很多栈访问，不能直接说“程序本来就频繁访问内存”；要再看 \code{-O2/-O3}，确认这些访问是不是调试构建为了保留变量形态而产生的。

\section{核心 pass：每个 pass 都在证明一类事实}

不要把优化 pass 当成魔法列表。每个 pass 都依赖一类可证明事实，也有明确失效边界。

\begin{center}
\small
\begin{tabular}{p{0.20\linewidth}p{0.36\linewidth}p{0.32\linewidth}}
\toprule
pass & 依赖的事实 & 常见边界 \\
\midrule
常量传播 & 某些值在所有路径上已知 & 输入、volatile、未知调用会打断 \\
死代码删除 & 结果不可观察且无副作用 & I/O、原子、volatile、异常、析构 \\
内联 & 调用目标和体积收益可接受 & ABI 边界、递归、动态派发、代码膨胀 \\
循环不变外提 & 表达式跨迭代不变且移动安全 & 别名、异常、越界、函数副作用 \\
向量化 & 迭代独立且内存访问可证明 & 别名、对齐、tail、浮点语义、数据依赖 \\
\bottomrule
\end{tabular}
\end{center}

\subsection{DCE：不可观察计算会消失}

下面的 benchmark 是坏的：

\begin{lstlisting}[language=C++,caption={结果不被消费的 benchmark}]
void bad_bench(const float* xs, int n)
{
    float s = 0.0f;
    for (int i = 0; i < n; ++i) {
        s += xs[i];
    }
}
\end{lstlisting}

如果 \code{s} 的值没有影响返回值、内存、I/O、volatile、原子或异常路径，优化器可以删除循环。它不是作弊，而是在执行语言规则允许的等价变换。正确实验应该把结果交给 oracle 或 sink，并确保 sink 本身不成为热循环主要成本。

\subsection{LICM：循环外提需要移动安全}

下面的写法看起来 \code{scale()} 每轮都调用：

\begin{lstlisting}[language=C++]
for (int i = 0; i < n; ++i) {
    out[i] = in[i] * scale();
}
\end{lstlisting}

若编译器能证明 \code{scale()} 无副作用、不抛异常、返回值不随迭代变化，并且不与 \code{in/out} 的访问产生别名关系，它可以把调用外提。若 \code{scale()} 可能读全局变量、访问 volatile、抛异常或修改 \code{out} 指向的数据，外提就不安全。报告不能只说“编译器没有优化”，要说明阻止外提的是副作用、异常、别名还是可见调用边界。

\subsection{向量化：不是把标量循环自动乘以寄存器宽度}

向量化需要证明迭代之间没有破坏语义的数据依赖。下面两段循环形状相似，但性质不同：

\begin{lstlisting}[language=C++]
// 容易向量化：每次迭代读写不同元素
for (int i = 0; i < n; ++i) {
    y[i] = a * x[i] + y[i];
}

// 前缀和：后一次依赖前一次
for (int i = 1; i < n; ++i) {
    y[i] += y[i - 1];
}
\end{lstlisting}

前者是典型 map 形状；后者是递推，不能简单按 lane 并行。即使前者也要处理别名：如果 \code{x} 和 \code{y} 指向重叠区域，重排 load/store 可能改变结果。编译器可能生成运行时别名检查，也可能拒绝向量化。高质量报告要保存 vectorization remark、汇编和尾部路径，而不只看耗时。

\section{别名分析：优化器为什么敢或不敢重排}

别名分析回答“两个内存访问是否可能指向同一对象”。这是性能和正确性的交叉点。若编译器知道两个指针不别名，就能更大胆地缓存值、重排 load/store 和向量化；若不能证明，就必须保守。

\begin{lstlisting}[language=C++,caption={别名会阻止简单缓存}]
void add_arrays(float* out, const float* a, const float* b, int n)
{
    for (int i = 0; i < n; ++i) {
        out[i] = a[i] + b[i];
    }
}
\end{lstlisting}

人通常假设 \code{out/a/b} 是三段不同数组，但函数签名没有表达这个事实。C 的 \code{restrict}、某些编译器扩展、\code{span} 加接口约束、运行时检查或更高层 descriptor，都可以提供别名证据。AI tensor runtime 里，memory planner 也必须记录 buffer 区间、lifetime 和 alias set，否则编译器或手写 kernel 都无法安全复用内存。

\subsection{strict aliasing 和 effective type}

严格别名规则不是书面主义，它直接影响机器码。若代码通过不兼容类型的指针读写同一对象，优化器可能假设这些访问不别名，从而缓存旧值或重排访问。例外包括 \code{char/std::byte} 访问对象表示、某些标准允许的类型关系和显式构造的对象生命周期边界。错误代码在 \code{-O0} 下能跑，不代表在优化构建下有语义。

\begin{warningbox}
不要用“我在内存里看到字节就是这样”反驳优化器。C++ 程序首先受语言对象模型约束；机器内存只是实现这个抽象的一种方式。若源码越过对象生命周期、类型访问或对齐边界，优化器可以基于语言前提生成与你直觉不同的机器码。
\end{warningbox}

\section{UB 到汇编变化：报告必须能闭合}

有符号溢出、越界访问、悬空指针、错误别名、未初始化读取和对象生命周期错误，都可能给优化器释放自由度。报告这类问题时，不应只贴“优化后结果错了”，而要沿下面链路闭合：

\begin{lstlisting}[numbers=none,caption={UB 优化事故报告模板}]
ub_optimization_case:
  source_snippet
  language_rule
  compiler_version_and_flags
  sanitizer_result
  ir_before_optimization
  ir_after_optimization
  assembly_difference
  observed_output_O0
  observed_output_O2
  corrected_source
  corrected_ir_or_assembly_summary
\end{lstlisting}

例如 signed overflow 的关键不是“CPU 加法会回绕”，而是 C++ 允许优化器假设正确程序不会发生 signed overflow。若需要回绕语义，应使用无符号整数、显式溢出检查或编译器提供的 checked arithmetic。若只是写 \code{int} 然后依赖二进制补码回绕，源码层已经把自己放在了不稳定地带。

\section{真实证据命令：保存输入和边界}

下面是一组适合第一册实验的命令形状。它们是公开工具链方法，不绑定某台机器。

\begin{lstlisting}[language=bash,caption={保存预处理、IR、汇编和目标文件证据}]
clang++ -std=c++20 -O2 -Wall -Wextra -E add.cpp -o add.ii
clang++ -std=c++20 -O2 -S -emit-llvm add.cpp -o add.ll
clang++ -std=c++20 -O2 -S add.cpp -o add.s
clang++ -std=c++20 -O2 -c add.cpp -o add.o
llvm-objdump -dr add.o > add.objdump.txt
readelf -s add.o > add.symbols.txt
readelf -r add.o > add.relocations.txt
\end{lstlisting}

若使用 GCC，可以保存 GIMPLE dump：

\begin{lstlisting}[language=bash,caption={GCC/GIMPLE 证据形态}]
g++ -std=c++20 -O2 -fdump-tree-all -c add.cpp -o add.o
objdump -dr add.o > add.objdump.txt
readelf -s add.o > add.symbols.txt
\end{lstlisting}

不要把某一次机器输出神化成普遍规律。编译器版本、目标三元组、优化级别、标准库、目标 CPU 和 flags 都会改变结果。报告中的证据应写成“这次构建看到的事实”，并保留边界。

\section{TinyTensor 全链路案例：从 C++ 语义到进程证据}

\code{add\_i32} 适合解释最小 ABI 和重定位，但它太小，暴露不出真实工程里的对象布局、循环优化、别名、异常边界、调试信息和 benchmark 删除问题。第一册还需要一个稍大的闭环案例：\code{TensorViewF32} 加一个 \code{linear\_ref}。这个案例应贯穿后续实验，而不是每章换一个新玩具。

\begin{lstlisting}[language=C++,caption={TinyTensor 的最小源程序骨架}]
struct TensorViewF32 {
    float* data;
    std::uint64_t rows;
    std::uint64_t cols;
    std::uint64_t stride;
};

extern "C" void linear_ref(TensorViewF32 x,
                           TensorViewF32 w,
                           TensorViewF32 y)
{
    for (std::uint64_t m = 0; m < x.rows; ++m) {
        for (std::uint64_t n = 0; n < w.cols; ++n) {
            float acc = 0.0f;
            for (std::uint64_t k = 0; k < x.cols; ++k) {
                acc += x.data[m * x.stride + k] *
                       w.data[k * w.stride + n];
            }
            y.data[m * y.stride + n] = acc;
        }
    }
}
\end{lstlisting}

这段代码要回答的不是“矩阵乘最快怎样写”，而是“同一个 C++ 对象怎样穿过编译器、ABI、对象文件、loader、debugger 和测量”。报告应按下面矩阵闭合。

\begin{center}
\small
\begin{tabular}{p{0.20\linewidth}p{0.36\linewidth}p{0.31\linewidth}}
\toprule
层次 & 必须保存的证据 & 失败案例 \\
\midrule
源码语义 & \code{TensorViewF32} 定义、输入 fixture、checksum oracle & \code{stride < cols} 越界但小输入未崩溃 \\
前端/AST & 预处理文件、AST dump、模板和宏展开边界 & 宏重复求值或头文件配置改变语义 \\
IR/优化 & \code{-O0/-O2} LLVM IR、\code{mem2reg}、DCE、vectorize remark & 输出未消费导致 hot loop 被删除 \\

推荐的命令分三组。第一组固定前端和 IR：

\begin{lstlisting}[numbers=none,caption={TinyTensor 前端和 IR 证据}]
clang++ -std=c++20 -O2 -Wall -Wextra -E tiny_tensor.cpp \
  -o reports/tiny_tensor.ii
clang++ -std=c++20 -O2 -Xclang -ast-dump -fsyntax-only \
  tiny_tensor.cpp > reports/tiny_tensor.ast.txt
clang++ -std=c++20 -O0 -S -emit-llvm tiny_tensor.cpp \
  -o reports/tiny_tensor_O0.ll
clang++ -std=c++20 -O2 -S -emit-llvm tiny_tensor.cpp \
  -o reports/tiny_tensor_O2.ll
\end{lstlisting}

第二组固定目标文件和二进制身份：

\begin{lstlisting}[numbers=none,caption={TinyTensor 对象文件和 ELF 证据}]
clang++ -std=c++20 -O2 -g -c tiny_tensor.cpp \
  -o build/tiny_tensor.o
clang++ -std=c++20 -O2 -g tiny_tensor.cpp \
  -o build/tiny_tensor
sha256sum build/tiny_tensor build/tiny_tensor.o \
  > reports/tiny_tensor.sha256
nm -C build/tiny_tensor.o > reports/tiny_tensor.nm.txt
readelf -h -l -S -s -r build/tiny_tensor \
  > reports/tiny_tensor.readelf.txt
objdump -drwC -Mintel build/tiny_tensor \
  > reports/tiny_tensor.objdump.txt
\end{lstlisting}

第三组固定运行时边界：

\begin{lstlisting}[numbers=none,caption={TinyTensor 运行时和测量证据}]
./build/tiny_tensor --fixture fixtures/linear_2x8x6.json --check
./build/tiny_tensor --fixture fixtures/linear_2x8x6.json \
  --bench --warmup 20 --repeat 200 \
  --csv reports/tiny_tensor_bench.csv
./build/tiny_tensor --print-maps > reports/tiny_tensor.maps.txt
gdb -q ./build/tiny_tensor \
  -ex 'break linear_ref' -ex run -ex 'info registers' \
  -ex 'bt' -ex quit > reports/tiny_tensor.gdb.txt
\end{lstlisting}

这三组命令训练一个原则：同一个结论必须能跨层对齐。例如，“优化器把局部变量放进寄存器”要能在 IR 中看到 \code{alloca} 消失，在汇编中看到内层累加器寄存器，在 GDB 中承认源码变量可能没有稳定地址；“benchmark 测的是 hot loop”要能在反汇编中定位循环，在 CSV 中区分 warmup 与 repeat，在报告中说明 fixture、checksum 和二进制 hash。

\subsection{运行时深水区：同一个案例还要穿过这些边界}

源码、IR、汇编和 ELF 只说明程序怎样被构建；真正运行时还会经过 loader、分配器、虚拟内存、系统调用、线程库、调试信息和异常展开。第一册不要求读者实现这些系统，但要求能把它们从报告里分出来。TinyTensor 的完整证据包应继续扩展为下面几组字段：

\begin{center}
\small
\begin{tabular}{p{0.20\linewidth}p{0.34\linewidth}p{0.32\linewidth}}
\toprule
边界 & 推荐证据 & 不能据此夸大的结论 \\
\midrule
动态链接 & \code{readelf -d/-r}、\code{LD\_DEBUG=libs,bindings}、PLT/GOT 反汇编 & 看到 PLT 不等于每次调用都 lazy bind \\
启动与构造 & \code{\_start}、\code{\_\_libc\_start\_main}、\code{.init\_array}、constructor trace & \code{main} 前成本不能算进 hot kernel \\
TLS & \code{readelf -s} 中 TLS 符号、访问反汇编、线程数对照 & TLS 快慢取决于模型和动态库位置 \\
分配器 & \code{strace brk/mmap/munmap}、\filepath{/proc/<pid>/smaps}、ASan 报告 & 没有 syscall 不代表没有分配成本 \\
缺页与 COW & minor/major fault、maps/smaps、first-touch 对照 & mmap 成功不代表页面已经常驻 \\
syscall/vDSO & \code{strace -T}、\code{clock\_gettime} 路径、errno/partial I/O & syscall 时间不是 CPU kernel 时间 \\
futex/阻塞 & \code{strace -e futex}、context switch、等待谓词 & futex 只证明睡眠路径，不证明同步设计正确 \\
DWARF/unwind & \code{readelf --debug-dump=frames}、GDB backtrace、frame-pointer 对照 & backtrace 好看不等于变量都有稳定地址 \\
异常路径 & RAII cleanup trace、unwind table、异常穿过 wrapper 的策略 & 异常不是普通跳转，手写汇编不能随意穿过 \\
\bottomrule
\end{tabular}
\end{center}

这张表的用法是“为结论选择证据”，不是一次实验全跑。若要解释首次请求慢，优先看动态链接、page fault、allocator 和 cold cache；若要解释 backtrace 断裂，优先看 frame pointer、DWARF CFI 和手写汇编；若要解释 benchmark 抖动，优先看 syscall、futex、page fault 和频率状态。不要把这些边界混成一句“系统开销”。

\begin{lstlisting}[numbers=none,caption={TinyTensor 运行时边界报告模板}]
runtime_boundary_report:
  binary_identity:
    build_id
    sha256
    compiler_and_flags
  loader:
    interpreter
    shared_libraries
    relocations
    lazy_binding_policy
  process_map:
    text_mapping
    heap_mapping
    stack_mapping
    mmap_weight_or_fixture_mapping
  allocator:
    allocation_sites
    brk_mmap_munmap_seen
    first_touch_policy
    rss_before_after
  syscall:
    observed_syscalls
    errno_or_partial_io
    vdso_fast_path_if_relevant
  blocking:
    futex_or_poll_seen
    voluntary_context_switches
    wait_condition
  debug_unwind:
    frame_pointer_policy
    eh_frame_present
    backtrace_quality
  conclusion:

例如，若 \code{linear\_ref} 第一次运行比后续慢，报告不能直接写“cache 冷”。至少要排除三类更基础的解释：第一，是否有首次动态绑定或共享库初始化；第二，输入、输出或权重 buffer 是否首次触页；第三，timed region 是否包含 allocator 和文件 I/O。只有这些边界被记录后，cache/TLB 的解释才有资格进入第二册的性能模型。

\subsection{sanitizer、fuzz 和 CI：第一册就要建立的工程门槛}

底层学习不能等到第三册才补工程验证。第一册的 TinyTensor 至少要有三类自动化门禁。

\begin{lstlisting}[numbers=none,caption={TinyTensor 的最低 CI 矩阵}]
ci_debug_asan_ubsan:
  -O1 -g -fsanitize=address,undefined
  fixtures: empty, 1x1, tail, bad_stride, overflow_shape

ci_release_oracle:
  -O2 -g
  compare reference checksum and candidate checksum
  save binary hash and objdump summary

ci_fuzz_shape:
  random rows, cols, stride, alignment, seed
  reject invalid shape before running kernel
  preserve failing seed as regression fixture
\end{lstlisting}

Sanitizer 的作用是把对象生命周期、越界、未对齐和整数边界尽早变成确定性失败；fuzz 的作用是让 shape、stride、tail、空输入、奇数维度和异常输入持续攻击接口；CI 的作用是保证每次改动都能重新生成同一类证据。性能数字不适合在所有 CI 环境中硬比较，但 correctness、sanitizer、fuzz seed、二进制身份和反汇编摘要可以作为最低门槛。

不要把这些写成“以后再做”。一段底层代码如果没有 sanitizer 和失败 fixture，只能证明它在当前输入上看起来能跑；一段优化代码如果没有 oracle 和二进制身份，只能证明它输出了一个时间数字。第一册的出口报告应把工具链、运行时和验证系统一起交付。

\section{优化报告的最低验收}

一份合格的优化器报告至少回答七个问题：

\begin{enumerate}
  \item 源码的语言语义是什么，是否存在 UB 或实现定义行为。
  \item 前端看到的类型、对象和调用边界是什么。
  \item IR 中有哪些 load、store、call、branch、phi 和 metadata。
  \item 关键 pass 改写了什么，依据是什么。
  \item 汇编是否体现了这些改写。
  \item 调试信息、栈展开或异常路径是否受到影响。
  \item 修复或优化后的代码如何用 sanitizer、oracle 和 benchmark 证明。
\end{enumerate}

\begin{keyidea}
第一册到这里才真正闭合：源码不是机器码，IR 不是汇编，语言 UB 不是硬件异常，优化不是玄学。读者必须能把一个结论从 C++ 规则追到 IR，再追到汇编和运行时证据。
\end{keyidea}

\section{实验与习题}

实验一：选择一个包含分支的小函数，分别保存 \code{-O0} 和 \code{-O2} 的 LLVM IR 与汇编。指出哪些局部变量消失、哪些分支被折叠、哪些 load/store 被提升或删除。

实验二：构造一个 benchmark 循环，先让结果不被消费，再加入 checksum。比较 IR 和汇编中循环是否存在，并解释为什么。

实验三：写一个可能 signed overflow 的函数。用 sanitizer 捕获问题，再改成无符号或显式检查版本。报告源码、IR、汇编和输出差异。

实验四：写两个数组相加函数，一个没有别名约束，一个提供明确 non-alias 证据。保存 vectorization remark 和汇编，说明编译器是否生成运行时别名检查。

验收标准：报告不能只写“编译器优化了”。必须能指出优化器使用的语言前提、IR 证据、pass 结果、汇编变化和未验证边界。
