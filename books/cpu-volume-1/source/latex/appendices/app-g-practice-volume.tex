\chapter{实践卷：从 C++ 代码到机器证据}

第一册的实践卷围绕 \filepath{books/cpu-volume-1/labs/lab00_benchmark_foundation}。这一卷不追求堆很多实验，而是训练一条硬链路：C++ 源码、编译选项、可执行文件、输入规模、计时边界、checksum、CSV 报告和反汇编线索。没有这条链路，任何“这个写法更快”的判断都只是感觉。

\section{实践一：先固定正确性，再测时间}

\code{lab00_tests} 先检查 \code{sum_i32}、\code{sum_f32}、\code{dot_f32}、\code{fill_u8}、\code{copy_u8} 和 benchmark runner 的基本合同。这里的 \code{checksum} 是输出摘要，用来阻止编译器把无用工作优化掉，也用来证明不同实现处理的是同一批输入。若 checksum 不稳定，计时结果没有解释价值。

\begin{lstlisting}[language=bash,caption={第一册实践工程构建和测试}]
cmake -S books/cpu-volume-1 -B books/cpu-volume-1/build/lab00-debug -DCMAKE_BUILD_TYPE=Debug
cmake --build books/cpu-volume-1/build/lab00-debug
ctest --test-dir books/cpu-volume-1/build/lab00-debug --output-on-failure
\end{lstlisting}

\section{实践二：测量报告必须说明边界}

正式测量建议用 Release 构建。报告至少写清 CPU 型号、编译器版本、构建类型、输入规模、预热次数、测量次数、是否有后台负载、是否固定频率。不能只贴“耗时多少纳秒”。同一个循环在 Debug 和 Release 下可能完全不同；同一份 Release 二进制在动态频率、虚拟机或手机远程环境下也可能出现噪声。

\begin{lstlisting}[language=bash,caption={运行 Lab 00 benchmark}]
cmake -S books/cpu-volume-1 -B books/cpu-volume-1/build/lab00-release -DCMAKE_BUILD_TYPE=Release
cmake --build books/cpu-volume-1/build/lab00-release --target lab00_bench lab00_bad_benchmarks lab00_tests
ctest --test-dir books/cpu-volume-1/build/lab00-release --output-on-failure
books/cpu-volume-1/build/lab00-release/bin/cpu-volume-1/lab00/lab00_bench --sizes 1024,65536,1048576
\end{lstlisting}

\section{实践三：从源码回到汇编}

第一册正文讲 C++ 到汇编、ABI 和机器执行。实践卷要求每个性能结论至少给出一个源码到机器证据的回看：热循环里有没有函数调用，是否出现向量指令，是否有意外分支，load/store 是连续还是跨步，编译器有没有把循环删掉。可以用 \cmd{objdump -d}、\cmd{perf} 或编译器生成汇编；若平台没有这些工具，报告要写“未验证”，不能把推断写成事实。
