\chapter*{本册结构}
\addcontentsline{toc}{chapter}{本册结构}

第一册围绕一条基础链路展开：

\begin{center}
\begin{minipage}{0.86\linewidth}
\begin{verbatim}
C/C++ 源码
  -> 编译、汇编、链接
  -> 可执行文件和进程
  -> x86-64 指令、寄存器、内存和控制流
  -> System V AMD64 ABI
  -> C++ 与汇编互操作
  -> 可信性能测量
\end{verbatim}
\end{minipage}
\end{center}

这条链路决定了本册的三部分结构。

\section*{底层模型}

第一部分从程序如何成为进程讲起，再进入 CPU 执行、数据表示、对象布局、指针、虚拟地址空间、Linux 工具和调试证据。这里的重点不是记住工具命令，而是建立一组可验证事实：运行的是哪份二进制，指令改变了哪些架构状态，对象在内存中如何排列，工具输出能证明什么。

\section*{汇编与 ABI}

第二部分把前面的机器状态具体化为 x86-64 汇编。整数指令、寻址模式、flags、条件跳转、循环、函数调用和 \code{switch} 都要同时从 C++ 语义、汇编形态和工具证据三侧解释。System V AMD64 ABI 负责把函数边界变成二进制合同：参数从哪里进入，返回值从哪里出去，栈如何对齐，哪些寄存器由谁保存。C++ 与汇编互操作则把这些规则放进可维护工程：public wrapper、internal C ABI、汇编 kernel、测试、反汇编审计和 benchmark 必须分层。

\section*{可信测量}

第三部分处理性能数字。运行时间本身只是观察，不是结论。可信 benchmark 必须先证明 correctness，再固定输入、计时区域、重复次数、统计口径、环境和二进制身份。第 15 章建立一次测量的基本协议，第 16 章把协议扩展为长期 benchmark suite，使结果可以复现、比较和维护。

\section*{纵向样板：TinyTensor 从源码到机器证据}

第一册需要有一个足够小、但能一路追到底的样板工程。本册使用 \term{TinyTensor}{最小张量 kernel 切片} 作为 MiniLLM-CPU 的前传：它只做一个 \code{[M,K] * [K,N]} 或 \code{[K] * [K,N]} 的线性层，但必须把 C++ 语义、对象布局、二进制、汇编、ABI、调试和测量全部串起来。

\begin{lstlisting}[numbers=none,caption={第一册 TinyTensor 的最小 C++ 切片}]
struct TensorViewF32 {
  float* data;
  std::uint64_t rows;
  std::uint64_t cols;
  std::uint64_t stride;
};

void linear_ref(const TensorViewF32 x,
                const TensorViewF32 w,
                TensorViewF32 y) {
  for (std::uint64_t m = 0; m < x.rows; ++m) {
    for (std::uint64_t n = 0; n < w.cols; ++n) {
      float acc = 0.0f;
      for (std::uint64_t k = 0; k < x.cols; ++k) {
        acc += x.data[m * x.stride + k] *
               w.data[k * w.stride + n];
      }
      y.data[m * y.stride + n] = acc;
    }
  }
}
\end{lstlisting}

这段代码故意不追求最快。它的价值是把第一册所有底层问题压到一个对象上：

\begin{center}
\begin{tabular}{p{0.22\linewidth}p{0.34\linewidth}p{0.30\linewidth}}
\toprule
要观察的事实 & 工具入口 & 必须回答的问题 \\
\midrule
对象布局 & \code{sizeof}/\code{offsetof}/GDB & \code{data/rows/cols/stride} 在对象中偏移多少 \\
函数 ABI & \code{objdump -d}/GDB & 三个 \code{TensorViewF32} 参数怎样传递或拷贝 \\
地址计算 & 反汇编和单步调试 & \code{m * stride + k} 变成哪种寻址与乘法 \\
二进制身份 & \code{readelf}/\code{sha256sum} & benchmark 跑的是哪一个可执行文件 \\
优化影响 & \code{-O0/-O2/-O3} 对照 & 循环、内联、寄存器分配怎样变化 \\
测量可信度 & CSV、环境、重复样本 & 计时区是否只包含 hot loop \\
\bottomrule
\end{tabular}
\end{center}

第一册的学习方式不是“读完语法再做项目”，而是在每章把 TinyTensor 往下一层拆。讲对象时，拆 \code{TensorViewF32} 的布局；讲指针时，拆 \code{x.data[m * stride + k]} 的合法性；讲虚拟内存时，观察三个 buffer 的地址范围；讲汇编时，定位内层循环；讲 ABI 时，观察参数进入寄存器还是栈；讲 benchmark 时，把同一份 kernel 固定为 reference、candidate、fixture、CSV 和报告。

\section*{第一册的实验阶梯}

TinyTensor 在第一册至少应完成六级实验。每一级都要能独立复现，不能只写成读书笔记。

\begin{center}
\begin{tabular}{p{0.12\linewidth}p{0.36\linewidth}p{0.34\linewidth}}
\toprule
级别 & 实验任务 & 验收证据 \\
\midrule
L1 & 编译并运行 \code{linear_ref}，输出 checksum & 编译命令、checksum、输入 fixture \\
L2 & 打印 \code{TensorViewF32} 的大小和成员偏移 & \code{sizeof}/\code{offsetof} 输出与解释 \\
L3 & 用 GDB 单步一个 \code{m,n,k} 点 & 寄存器、局部变量、内存地址截图或日志 \\
L4 & 用 \code{objdump} 标出内层循环 & 汇编片段、源码行、跳转目标对应关系 \\
L5 & 比较 \code{-O0/-O2/-O3} 产物 & 符号、代码大小、核心指令差异 \\
L6 & 写可信 benchmark 报告 & raw CSV、summary、环境、二进制 hash、结论边界 \\
\bottomrule
\end{tabular}
\end{center}

这些实验不要求读者马上写 SIMD，但要求读者能证明自己理解了机器正在执行什么。若连 \code{linear_ref} 的参数传递、地址计算和热循环都说不清，后续第二册谈 cache/SIMD，第三册谈 int8 GEMV，都只会停在名词层。

\section*{第一册常见错误案例}

第一册还要训练读者识别“看起来能跑但结论无效”的情况：

\begin{itemize}
  \item \code{TensorViewF32} 指向的 buffer 已释放，程序偶尔输出正常，但对象生命周期已经错误。
  \item \code{stride < cols}，地址计算越界，benchmark 数字没有意义。
  \item benchmark 没有消费输出，编译器删除 hot loop。
  \item 把文件 I/O、随机输入生成和 kernel 执行放在同一个计时区。
  \item 改了源码却没有确认二进制 hash，实际运行的是旧程序。
  \item 只看 \code{-O0} 汇编就推断优化后行为，忽略寄存器分配、内联和向量化。
\end{itemize}

这些错误比背错一个术语更危险，因为它们会制造“我已经测过”的错觉。第一册的底线是：正确性、二进制身份和测量口径不成立时，不允许讨论性能结论。

\section*{第一册的出口报告}

读完第一册，读者应该能提交一份短报告，题目可以是“TinyTensor linear\_ref 从源码到机器执行”。报告至少包含：

\begin{lstlisting}[numbers=none,caption={第一册出口报告目录}]
1. C++ source and build command
2. TensorView object layout
3. process memory map and buffer addresses
4. ABI boundary of linear_ref
5. annotated assembly of the hot loop
6. correctness reference and checksum
7. benchmark method and raw CSV
8. binary identity and environment
9. failed experiments
10. limits before entering cache/SIMD optimization
\end{lstlisting}

这份报告就是进入第二册的门票。它证明读者不是只知道“代码会运行”，而是能把一段 C++ 热路径追到对象、地址、ABI、指令和可信测量。

\section*{第一册可复现实验命令}

第一册的实验不能停在“建议使用工具”。每个读者都应该能把 TinyTensor 压成下面这组命令。命令中的路径可以随项目调整，但报告字段不能少。

\begin{lstlisting}[numbers=none,caption={TinyTensor 第一册实验命令模板}]
# 1. build three optimization levels
c++ -std=c++20 -O0 -g tiny_tensor.cpp -o build/tiny_o0
c++ -std=c++20 -O2 -g tiny_tensor.cpp -o build/tiny_o2
c++ -std=c++20 -O3 -g tiny_tensor.cpp -o build/tiny_o3

# 2. prove binary identity
sha256sum build/tiny_o0 build/tiny_o2 build/tiny_o3
readelf -h build/tiny_o2
readelf -Ws build/tiny_o2 | grep linear_ref

# 3. inspect assembly
objdump -drwC --no-show-raw-insn build/tiny_o0 > reports/tiny_o0.asm
objdump -drwC --no-show-raw-insn build/tiny_o2 > reports/tiny_o2.asm
objdump -drwC --no-show-raw-insn build/tiny_o3 > reports/tiny_o3.asm

# 4. run correctness and benchmark
./build/tiny_o2 --fixture fixtures/tiny_2x8x6.json --check
./build/tiny_o2 --fixture fixtures/tiny_2x8x6.json --bench \
  --warmup 20 --repeat 200 --csv reports/tiny_bench.csv
\end{lstlisting}

这些命令训练四件事。第一，优化级别不同，汇编形态会不同。第二，报告必须记录二进制 hash，避免跑错程序。第三，correctness 和 benchmark 分开，避免“快但错”。第四，CSV 是原始证据，不是结论本身。

\section*{第一册作业：拆穿一个最小错误}

每个学习者至少要亲手制造并修复三个错误：

\begin{enumerate}
  \item 把 \code{y.stride} 设置成小于 \code{y.cols}，观察 checksum、ASan 或 GDB 怎样暴露越界。
  \item 去掉输出 checksum，让编译器有机会删除 hot loop，对比 \code{objdump} 和 benchmark。
  \item 把 \code{TensorViewF32} 按值传参改成按指针传参，比较 ABI 边界和汇编。
\end{enumerate}

作业报告不要只写“出现错误”。必须写清：错误输入是什么，语言层违反了哪条合同，工具证据在哪里，修复后 correctness 和汇编/benchmark 是否恢复。第一册训练的不是背答案，而是把模糊故障变成可验证事实。

\section*{进入第二册}

第二册会继续展开 cache/TLB、现代乱序核心、SIMD、HPC kernel、线程、NUMA、PMU 和 AI CPU 算子。第一册不提前浅讲这些主题，而是先把它们共同依赖的基础打通：源码语义、对象布局、二进制身份、汇编阅读、ABI 边界和测量证据。没有这些基础，高级优化很容易退化成口号；有了这些基础，后续主题才能落到可以验证的机器事实上。
