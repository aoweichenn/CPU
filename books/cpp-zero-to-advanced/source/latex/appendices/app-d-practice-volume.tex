\chapter{实践卷：从语法到工程切片}

本册新增一个可运行实践工程，目录是：

\begin{lstlisting}[language=bash]
books/cpp-zero-to-advanced/labs/cpp_practice_workbook
\end{lstlisting}

它把前面章节里的值类型、字符串视图、容器、异常、类不变量、RAII 思维、泛型算法、测试和 benchmark 放进同一个小项目。项目不是大而空的综合作业，而是一个文本分析工作簿：解析配置、清洗文本、统计词频、排序输出，并用 LRU 缓存保存最近结果。

\section{实践一：配置解析不是字符串游戏}

入口文件 \filepath{include/cppbook/practice_workbook.hpp} 定义 \code{TextConfig}、\code{Diagnostic} 和 \code{ParseConfigResult}。这里的 \code{Diagnostic} 是结构化诊断，意思是错误不只是一行打印，而是包含行号和消息的对象。读者要先理解三个边界：配置行可以有注释和空白；\code{min_word_length} 必须是正数；\code{keep_numbers} 只能是 \code{true/false} 或 \code{1/0}。

\begin{lstlisting}[language=C++,caption={配置解析的最小合同}]
struct Diagnostic {
    std::int32_t line_number = 0;
    std::string message;
};

struct ParseConfigResult {
    TextConfig config;
    std::vector<Diagnostic> diagnostics;
    [[nodiscard]] bool ok() const;
};
\end{lstlisting}

这个结构刻意不让解析函数直接退出程序。函数返回结果对象，调用者决定是继续、报错还是展示所有诊断。这样能把“库代码”和“命令行程序”分开，也能让 GTest/GMock 测试精确检查坏输入。

\section{实践二：文本统计要拆成可测试步骤}

\code{TextAnalyzer} 的核心不是“统计词频”四个字，而是四步：扫描字符、归一化小写、过滤停用词、稳定排序。每一步都能构造反例。比如 \code{42} 在 \code{keep_numbers=false} 时应被过滤；\code{C++} 会被拆成 \code{c}，若最小长度为 3 就不会保留；两个词频相同时按字典序排序，保证输出稳定。

\begin{lstlisting}[language=bash,caption={C++ 实践工程的测试和 benchmark}]
cmake -S books/cpp-zero-to-advanced -B books/cpp-zero-to-advanced/build/practice-debug -DCMAKE_BUILD_TYPE=Debug
cmake --build books/cpp-zero-to-advanced/build/practice-debug
ctest --test-dir books/cpp-zero-to-advanced/build/practice-debug --output-on-failure
books/cpp-zero-to-advanced/build/practice-debug/labs/cpp_practice_workbook/cppbook_practice_bench --benchmark_min_time=0.01s
\end{lstlisting}

单元测试在 \filepath{tests/practice_workbook_tests.cpp}，使用 GTest/GMock 表达期望，例如 \code{EXPECT_THAT(tokens, ElementsAre(...))}。benchmark 在 \filepath{bench/practice_workbook_bench.cpp}，使用 Google Benchmark 测量 \code{top_words} 在不同文本规模下的成本。Debug 构建的 benchmark 只能证明可执行和相对路径正确，不能当作性能结论；要写性能报告，应另建 Release 构建并记录 CPU、编译器、负载和频率策略。

\section{实践三：LRU 缓存训练对象不变量}

\code{AnalysisCache} 用 \code{std::list} 保存最近访问顺序，用 \code{std::unordered_map} 从 key 找到链表迭代器。这里可以复习“迭代器、引用、生命周期、移动语义”这些术语。迭代器是指向容器位置的对象；哈希表保存迭代器后，链表节点不能随便失效；\code{splice} 能移动链表节点而不复制 \code{WordCount} 向量。

实践报告必须画出三个状态：插入 \code{a,b} 后的顺序；访问 \code{a} 后的顺序；再插入 \code{c} 后谁被淘汰。若解释不清这个状态变化，代码即使通过样例，也不算真正理解 LRU。
