# 第 3 章：x86-64 寄存器、调用约定和 ABI

## 本章目标

本章建立 C++ 与汇编互操作的基础。完成后你应该能：

- 解释 x86-64 通用寄存器和 SIMD 寄存器。
- 掌握 System V AMD64 ABI 的参数传递、返回值、栈对齐。
- 区分 caller-saved 和 callee-saved。
- 写简单 `.S` 汇编函数并从 C++ 调用。
- 用 GDB 观察函数调用时的寄存器和栈。
- 故意制造 ABI bug 并修复。

## ABI 是什么

ABI 是 Application Binary Interface，二进制接口。

API 规定源码层怎么调用。  
ABI 规定二进制层怎么调用。

ABI 回答的问题包括：

- 参数放在哪些寄存器？
- 返回值放在哪里？
- 哪些寄存器由调用者保存？
- 哪些寄存器由被调用者保存？
- 栈如何对齐？
- struct 如何返回？
- C++ 名字如何 mangling？
- 动态库符号如何链接？

你写手写汇编、读函数调用、分析性能开销、理解虚函数和 callback，都离不开 ABI。

## 通用寄存器

x86-64 通用寄存器包括：

```text
rax rbx rcx rdx
rsi rdi rbp rsp
r8  r9  r10 r11
r12 r13 r14 r15
```

其中：

- `rax`：常用于返回值。
- `rsp`：栈指针。
- `rbp`：传统栈帧指针，优化后可能不用。
- `rdi/rsi/rdx/rcx/r8/r9`：System V ABI 下前 6 个整数/指针参数。

32 位子寄存器：

```text
eax ebx ecx edx ...
```

写 32 位寄存器通常会清零高 32 位。例如写 `eax` 会清零 `rax` 高位。

这在读汇编时很重要。

## SIMD 寄存器

常见 SIMD 寄存器：

```text
xmm0-xmm31  128-bit
ymm0-ymm31  256-bit
zmm0-zmm31  512-bit
```

是否有 `xmm16+`、`zmm` 取决于 ISA 和平台支持。

本机当前暴露了：

- `avx`
- `avx2`
- `fma`
- `avx_vnni`

没有暴露：

- `avx512*`
- `amx*`

所以本机适合 AVX2/FMA/AVX-VNNI 实验，不适合直接运行 AVX-512/AMX 实验。

## System V AMD64 参数传递

Linux x86-64 常用 System V AMD64 ABI。

前 6 个整数或指针参数：

```text
1: rdi
2: rsi
3: rdx
4: rcx
5: r8
6: r9
```

第 7 个及之后通常在栈上。

整数返回值：

```text
rax
```

浮点参数通常使用：

```text
xmm0, xmm1, xmm2, ...
```

浮点返回值：

```text
xmm0
```

实验函数：

```cpp
extern "C" long many_args(
    long a, long b, long c, long d, long e, long f, long g);
```

你应该观察到：

- `a-f` 在寄存器。
- `g` 从栈读取。

## caller-saved 和 callee-saved

caller-saved：调用者如果还想保留，调用前自己保存。

常见 caller-saved：

```text
rax rcx rdx rsi rdi r8 r9 r10 r11
```

callee-saved：被调用者如果修改，返回前必须恢复。

常见 callee-saved：

```text
rbx rbp r12 r13 r14 r15
```

如果你手写汇编修改了 `rbx` 却不恢复，调用者可能在后续使用被破坏的值，程序出现诡异错误。

## 栈对齐

System V AMD64 ABI 要求函数调用边界满足栈对齐约定。调用函数前，栈需要按 16 字节对齐。

为什么重要？

- SIMD load/store 可能依赖对齐。
- 被调用函数可能假设栈对齐。
- 标准库或编译器生成代码可能使用对齐指令。

如果你手写汇编里 `push` 了一个寄存器，然后直接 `call` C++ 函数，可能破坏对齐。

## 最小手写汇编

头文件：

```cpp
extern "C" long add_i64(long a, long b);
```

汇编：

```asm
.intel_syntax noprefix
.text
.globl add_i64
.type add_i64, @function
add_i64:
    lea rax, [rdi + rsi]
    ret
```

解释：

- `rdi` 是第一个参数。
- `rsi` 是第二个参数。
- `rax` 是返回值。
- `lea` 可以做地址计算，也常被用作整数加法。

## 汇编中的循环

`sum_i64`：

```asm
.intel_syntax noprefix
.text
.globl sum_i64
.type sum_i64, @function
sum_i64:
    xor rax, rax
    test rsi, rsi
    je .done
.loop:
    add rax, qword ptr [rdi]
    add rdi, 8
    dec rsi
    jne .loop
.done:
    ret
```

假设 C++ 声明：

```cpp
extern "C" long sum_i64(long const* values, std::size_t size);
```

参数：

- `rdi`：指针。
- `rsi`：长度。

循环：

- 从 `[rdi]` 读 8 字节。
- 累加到 `rax`。
- 指针加 8。
- 长度减 1。

## C++ 调汇编的测试

必须写随机测试：

```cpp
for (...) {
    std::vector<long> values = make_random();
    assert(sum_i64_asm(values.data(), values.size()) == sum_i64_ref(values));
}
```

为什么？

手写汇编很容易：

- 漏掉空输入。
- 指针步长错。
- 返回寄存器错。
- 破坏 callee-saved。
- 栈没对齐。

## GDB 观察

编译 Debug：

```bash
cmake -S . -B build/debug -DCMAKE_BUILD_TYPE=Debug
cmake --build build/debug
gdb ./build/debug/your_test
```

常用命令：

```gdb
break add_i64
run
info registers
x/16gx $rsp
stepi
disassemble /m add_i64
```

你要观察：

- 进入函数时参数在哪。
- `rsp` 是多少。
- 返回前 `rax/xmm0` 是什么。
- 修改 callee-saved 后调用者是否出错。

## ABI bug 实验

### bug 1：破坏 `rbx`

错误：

```asm
bad_function:
    mov rbx, 123
    ret
```

如果调用者依赖 `rbx` 保存某个值，就会错。

修复：

```asm
good_function:
    push rbx
    mov rbx, 123
    pop rbx
    ret
```

注意：push/pop 也影响栈对齐。如果中间还要 `call`，必须重新考虑对齐。

### bug 2：返回值放错寄存器

整数返回值必须放 `rax`，不是 `rdi` 或 `rcx`。

### bug 3：浮点返回值放错寄存器

float/double 返回值通常在 `xmm0`。

## 常见误区

- 忘记 `extern "C"`。
- 忘记 `.globl`。
- Intel 和 AT&T 语法混用。
- 修改 `rbx/r12-r15` 不恢复。
- 栈没对齐就 `call`。
- 用 `eax` 改了 `rax` 高位却没意识到。
- float 返回值放到 `rax`。
