# 本书写作标准：经典教材式章节

本文档规定本书所有章节的写作标准。后续新增章节和回填旧章节都必须遵守这里的要求。

本书目标不是写“知识点清单”，也不是写“工具命令集合”，而是写成一本能让刚学完 C/C++ 的读者逐步进入现代计算机体系结构、操作系统、编译器和高性能计算的系统教材。

## 章节必须回答的 8 个问题

每个核心机制都必须回答：

1. 这个机制为什么存在？
2. 如果没有这个机制，会遇到什么问题？
3. 它给程序员提供了什么抽象？
4. 底层大致如何实现这个抽象？
5. C/C++ 代码如何映射到这个机制？
6. 我们如何用工具观察它？
7. 它对性能有什么影响？
8. 初学者最容易误解什么？

例如讲 cache，不能只写“cache 是高速缓存”。必须讲：

```text
CPU 速度和 DRAM 速度差距为什么需要 cache
cache line 为什么存在
地址如何映射到 cache set
load miss 大致如何向下级层级请求数据
C++ 连续数组为什么友好
stride 为什么浪费 cache line
如何用工作集扫描观察 cache 层级
为什么 cache hit 不等于程序一定快
```

## 每章固定结构

每章至少包含以下部分。

### 1. 本章目标

目标不能只是“了解”。必须写成可验证能力：

```text
完成后你应该能解释...
完成后你应该能画出...
完成后你应该能用命令观察...
完成后你应该能写实验验证...
```

### 2. 从一个具体问题开始

不要从定义开始。先给一个读者能理解的问题。

例子：

```text
为什么 main 不是程序真正执行的第一条代码？
为什么 int x = 1; 最后会变成 mov 指令？
为什么同一个循环 -O0 和 -O3 差异巨大？
为什么数组连续访问比链表快？
为什么看到 vaddps 不等于程序一定快？
```

### 3. 逐层抽象

必须按层次讲：

```text
程序员看到的模型
编译器看到的模型
操作系统看到的模型
CPU 执行看到的模型
微架构大致实现模型
```

不是每章都需要五层全讲，但涉及到哪层就必须明确是哪层。

### 4. 最小例子

每个新概念必须配一个最小代码例子。  
例子要短到可以手工推导。

### 5. 从源码到机器的走读

至少选择一个例子，走完整链路：

```text
C/C++ source
  -> compiler command
  -> assembly or IR
  -> binary/tool observation
  -> performance or correctness consequence
```

### 6. 底层实现模型

实现模型不要求达到芯片设计细节，但必须足够解释现象。

例如讲 CPU 执行指令，必须至少讲：

```text
取指
译码
寄存器
执行单元
load/store
提交/退休
```

后续再逐步引入：

```text
pipeline
branch prediction
out-of-order
rename
uops
ports
cache
TLB
```

### 7. 反例和边界

每章必须包含至少两个“看似正确但实际错误”的例子。

例子：

```text
指令少不一定快
branchless 不一定快
inline 不一定快
cache 大不一定快
汇编里函数存在不代表热路径调用它
```

### 8. 实验

实验必须能在本仓库里独立运行或容易复制。每个实验要写：

- 目的。
- 代码。
- 命令。
- 观察点。
- 预期现象。
- 如果结果不同，可能原因。

### 9. 作业

作业不能只问概念。必须包含：

- 解释题。
- 代码题。
- 工具观察题。
- 报告题。

每章至少一个作业要求写报告。

### 10. 验收标准

验收标准要能判断读者是否真的掌握。

错误写法：

```text
理解 CPU 执行。
```

正确写法：

```text
能画出 fetch/decode/execute/load-store/retire 的简化数据流。
能用 objdump 找到 C++ 函数对应机器码。
能解释寄存器值如何作为下一条指令输入。
能说明为什么一次 cache miss 会让后端等待。
```

## 章节深度分层

为照顾刚学完 C/C++ 的读者，每章采用三层深度。

### 第一层：必须掌握

这部分写得最清楚、最慢、例子最多。读者必须能完全理解。

### 第二层：工程理解

这部分解释真实系统如何工作，但允许简化。读者需要能用于分析。

### 第三层：深入方向

这部分指出更深的硬件、OS、编译器细节，为后续学习埋线索。读者暂时可以不完全掌握，但要知道它存在。

## 术语规则

第一次出现术语时必须给出：

- 英文全称。
- 中文解释。
- 它属于哪一层抽象。
- 最小例子。

例如：

```text
TLB 是 Translation Lookaside Buffer，地址翻译后备缓冲。它属于 CPU 内存管理硬件的一部分，用来缓存虚拟页到物理页的翻译结果。
```

## 公式和性能模型规则

出现公式时必须解释每个量的来源。

例如：

```text
AI = FLOPs / bytes moved
```

必须说明：

- FLOPs 怎么数。
- bytes 包括哪些读写。
- 为什么 write allocate 会改变 bytes。
- 这个模型忽略了什么。
- 如何用实验验证。

## 汇编讲解规则

讲汇编时必须先讲语法，再讲语义。

每段汇编至少解释：

- 操作码是什么。
- 源操作数是什么。
- 目的操作数是什么。
- 哪些寄存器被读。
- 哪些寄存器被写。
- 是否访问内存。
- 是否影响 flags。
- 下一条控制流去哪里。

例如：

```asm
add eax, edx
```

必须解释：

```text
eax = eax + edx
读 eax 和 edx
写 eax
更新 EFLAGS
不访问内存
```

## 图示规则

可以使用文本图。复杂机制必须给图。

例如：

```text
virtual address
  -> TLB lookup
  -> page table walk if miss
  -> physical address
  -> cache lookup
  -> data
```

## 对旧章节的回填要求

当前第 1-7 章已经有训练内容，但还需要按本标准回填：

- 第 1 章补“时间测量、编译器优化、统计噪声为什么会出现”。
- 第 2 章补“ELF 和链接器逐层解释”。
- 第 3 章补“汇编语法和 ABI 从零讲解”。
- 第 4 章补“编译器前端、IR、SSA 从为什么存在开始讲”。
- 第 5 章补“pipeline 和 out-of-order 的更基础说明”。
- 第 6 章补“预测器和 speculative execution 的渐进模型”。
- 第 7 章补“SRAM/DRAM、cache line、地址映射的基础解释”。

## 每次写作后的检查

写完一章后检查：

```text
是否从具体问题开始？
是否解释为什么需要这个机制？
是否有程序员模型和底层实现模型？
是否有最小代码例子？
是否有命令能观察？
是否有反例？
是否有作业报告？
是否有验收标准？
```

如果这些问题回答不了，这章还不能算教材章节。
